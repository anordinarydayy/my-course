00Concept · Redis

Why Redis Exists

Your database is fast — until ten thousand people hit the same product page at once, each read waits on a disk, and the whole app crawls. Redis exists for that moment. It is an in-memory data-structure server: it keeps your hot data in RAM and serves it through a handful of purpose-built structures in well under a millisecond. This first lesson builds the mental model — why in-memory, why single-threaded, why “data-structure server” and not merely “cache” — so every later lesson has somewhere to land.

00 The problem

The speed wall

Start with the thing Redis is reacting against. A disk-backed, general-purpose database answers a typical point read in roughly 1 to 10 milliseconds: it parses and plans the query, walks an index, and — when the page it wants isn't already in its buffer pool — waits on an SSD, all across a network round trip. For most pages that is perfectly fine and you should not reach for anything else.

It stops being fine on the hot path: the handful of keys that every request touches. The logged-in user's session. The view counter on a post that just went viral. The top-ten leaderboard the game polls every second. The product JSON behind a flash sale. These are read thousands or millions of times a second, and they are usually the same few keys over and over. Multiply 5 ms by a million reads a second and you have either melted the database or blown your latency budget — often both.

Two numbers make the gap visceral. Reading a value that is already in RAM takes on the order of 100 nanoseconds. Reaching across the network to a database that then touches an SSD takes on the order of milliseconds — a difference of roughly 10,000–100,000×. Even a real Redis call, network round trip included, lands around 0.2–0.5 ms — still tens of times faster than the database query it replaces, and, crucially, it takes that load off the database entirely.

The shape of the problem

Two axes are in play: latency (how long one read takes) and throughput (how many reads per second you can sustain). The hot path needs both at once — sub-millisecond and enormous QPS, on a small set of frequently reused keys. A general-purpose database trades latency away to buy durability and flexible queries. Redis makes the opposite trade: drop most of that generality, keep the working set in memory, and the floor drops by orders of magnitude.

01 The idea

The core insight

Redis gets its speed from three design choices that stack on top of one another, each removing a specific cost.

1 · Keep the data in RAM. Main memory is about a hundred times faster than an SSD and far faster than spinning disk — no seek, no page fault out to storage. That is the whole reason for the speed. The price is capacity (RAM is smaller and dearer than disk) and durability (RAM is volatile — lose power, lose data). Redis has answers for both, which we'll get to, but the headline trade is “spend memory, buy latency.”

2 · Serve it from a single-threaded event loop. This surprises people, but it is deliberate. One thread pulls commands off thousands of connections and executes them one at a time, each to completion. There are no locks, no mutexes, no lock contention, and none of the cross-core cache-coherency traffic that multi-threaded shared state creates. Because exactly one command runs at any instant, every single-command operation is atomic for free — an INCR can't half-apply, and two clients can't interleave inside one command. Throughput stays high because each operation is tiny (memory speed) and the loop multiplexes many connections with epoll/kqueue. (Modern Redis does use extra threads for network I/O and background work, but the command-execution model you reason about is single-threaded.)

3 · Operate on rich data structures, not opaque blobs. The server understands strings, hashes, lists, sets, and sorted sets natively, so it can offer operations with the right complexity: append to a list in O(1), test set membership in O(1), pull the top-N from a sorted set in O(log n). The structure lives server-side, so you mutate it in place with a single command instead of the read–modify–write round trips a dumb cache would force on you.

Put the three together and you get the one sentence that is the product: data in memory + one command at a time + the right structure = O(1)/O(log n) atomic operations at memory speed.

Single-threaded is a feature, not a limitation

Because only one command runs at a time, operations like INCR views, LPUSH q job, or ZADD lb 1850 sam are atomic with zero work from you — no locks, no compare-and-set retry loops, no torn writes. The catch is the flip side: a slow command (KEYS * over a huge keyspace, a giant SORT) blocks everyone, because there is no second thread to pick up the slack. Keep commands O(1)/O(log n) and the model is pure upside. “Why is Redis single-threaded?” is one of its most common interview questions — this paragraph is the answer.

02 Not just a cache

A data-structure server ▸

The word “cache” undersells Redis. A plain cache is a dictionary: opaque bytes in, opaque bytes out, addressed by key. Redis is that and four more containers, each with operations tuned to a job. Choosing the right one is most of using Redis well. The five core types:

  • Strings — the basic key → value (text, JSON, a number, or raw bytes up to 512 MB). Atomic counters live here: INCR, INCRBY, DECR. Use for: cached HTML/JSON, feature flags, view counts, rate-limit counters.
  • Hashes — one key holding a map of field → value, like a row or an object. Update a single field without rewriting the whole blob: HSET user:9 name Mia, HINCRBY user:9 logins 1. Use for: session objects, user profiles, shopping carts.
  • Lists — an ordered sequence with O(1) push/pop at both ends. LPUSH/RPOP is a queue; LPUSH/LPOP a stack; LRANGE a recent-activity feed. Use for: job queues, timelines, “last N events.”
  • Sets — unordered, unique members with O(1) add and membership, plus union / intersection / difference server-side. Use for: unique visitors, tags, “who liked this,” friend-of-friend via SINTER.
  • Sorted sets (ZSET) — a set where every member carries a score and is kept ordered by it. Add or update in O(log n); read a rank range in O(log n + m). This is the leaderboard structure — and the backbone of time-ordered queues, sliding-window rate limiters (score = timestamp), and priority scheduling. Use for: leaderboards, top-N, delay queues.

(There are more — bitmaps, HyperLogLog, geospatial indexes, and streams — built alongside these and covered later. Master the core five first; they carry most of the weight.) The sorted set is the most “magic” of the group, so watch one work: scores change, and the ranking re-sorts itself with no effort on your part.

Pick the structure, get the algorithm

Here is the interview tell: someone reaches for Redis and then does GET → parse JSON → mutate → SET it back. They are using a data-structure server as a dumb cache — and they have re-introduced a race (two clients overwrite each other) plus extra round trips. The fix is almost always “use the right type”: a counter is INCR, a leaderboard is a ZSET, a cart is a Hash, a queue is a List. The server-side operation is both faster and atomic.

03 Cache hygiene

Keys, TTL & eviction

Everything in Redis lives in one flat keyspace: every value has a string key (product:42, session:ab2f, lb:global). Two mechanisms turn that keyspace into a cache that manages its own size: expiry and eviction. They sound alike and get conflated in interviews — they are not the same thing.

Expiry (TTL) is per-key, time-based, and your choice. You attach a time-to-live: SET product:42 ... EX 300 (expire in 300 seconds) or EXPIRE key 60. When the clock runs out, the key is removed. Redis does this two ways: lazily (on access — if a key is found expired, it is dropped and treated as missing) and actively (a background sampler deletes expired keys over time so they don't accumulate). TTL is how a cache stays fresh — stale entries delete themselves — and how you bound how wrong a cached copy is allowed to get.

Eviction is global, space-based, and the policy's choice. Memory is finite, so you set a maxmemory ceiling. When Redis hits it and a new write needs room, it evicts existing keys according to maxmemory-policy. The common policies: allkeys-lru drops the least-recently-used key (the coldest); allkeys-lfu drops the least-frequently-used (better when popularity matters more than recency); volatile-lru only evicts keys that carry a TTL; and noeviction rejects new writes instead of dropping anything. Eviction is what lets Redis run as a fixed-size cache that automatically keeps the hot set and sheds the cold.

So the distinction to lock in: expiry is time-based and per-key — you decided this key should live five minutes — while eviction is space-based and policy-chosen — under memory pressure Redis decides which key dies so a new one can live. A key can leave for either reason; mixing them up is a classic slip.

Eviction policy answers “cache, or store?”

allkeys-lru says “I am a cache — throw away whatever is cold to make room.” noeviction says “I am holding data I can't lose — refuse new writes before you drop anything.” Choosing the eviction policy is really choosing what Redis is for in your architecture. Hold that thought; it returns in §05.

04 The canonical pattern

Cache-aside in practice ▸

The most common way to put Redis in front of a database is the cache-aside pattern (also called lazy loading), and it ties §00–§03 together: an in-memory store, addressed by key, bounded by a TTL, taking read load off the slow path. The key idea is that the application — not the cache — owns the logic: check the cache; on a miss, read the database and populate the cache; let the TTL expire stale entries. Walk a single read through it.

Notice the contract the diagram encodes: Redis here is a performance layer, not the source of truth. Postgres still owns the canonical data. That is deliberate — if Redis goes down, every read simply falls through to the database: slower, but still correct. It also names the failure modes you have to design for, which the worked example below makes concrete.

Worked exampleCache-asidelatency math
Cache-aside latency math

A product page reads one row from Postgres at about 20 ms. Put Redis in front at about 0.3 ms per hit. At a cache hit-rate h, what is the effective read latency — and how much load comes off the database? Assume reads dominate and the cached copy may be slightly stale.

Think it through

Restate

One hot read served two ways: a fast cache hit (~0.3 ms) most of the time, a slow miss (~20 ms) the rest. We want the blended latency and the DB read reduction as functions of h.

The naive path

No cache: every read hits Postgres — ~20 ms each, and the DB absorbs 100% of read QPS. At a million reads a minute that is a million DB reads a minute: the wall from §00.

The cache-aside move

Check Redis first. A fraction h are hits (0.3 ms, DB untouched); the remaining (1−h) miss, pay the DB once, then populate so the next read is a hit.

The math

Effective latency ≈ h·0.3 + (1−h)·20 ms. At h=0.95 → ~1.29 ms (about 15× faster than 20 ms); at h=0.99 → ~0.50 ms (about 40×). DB read load equals the miss rate (1−h): at 95% the DB sees 20× fewer reads, at 99% 100× fewer.

The point

The win scales with hit-rate — and hit-rate comes from the working set fitting in memory and a TTL long enough to stay hot yet short enough to bound staleness. Redis stays the cache; Postgres stays the source of truth.

The code

python

import redis, json
r = redis.Redis()                       # pooled client to Redis

def get_product(pid):
    key = f"product:{pid}"
    hit = r.get(key)                    # ~0.3 ms in-memory lookup
    if hit is not None:                 # HIT: Postgres never sees this read
        return json.loads(hit)
    row = db.fetch_one(                 # MISS: pay the slow path once
        "SELECT * FROM products WHERE id = %s", pid)   # ~20 ms
    r.set(key, json.dumps(row), ex=300)  # populate; expire in 5 min (bounds staleness)
    return row

def update_price(pid, price):
    db.execute("UPDATE products SET price=%s WHERE id=%s", price, pid)
    r.delete(f"product:{pid}")             # invalidate → next read reloads fresh
Reason it through aloud

“I'll frame it as a blended latency. Most reads — say 95% — are cache hits at about 0.3 milliseconds; the other 5% miss and pay Postgres at about 20. So the effective latency is roughly 0.95×0.3 plus 0.05×20 — about 1.3 milliseconds, call it fifteen times faster than going to the database every time. Just as important, only the misses reach the database: at a 95% hit-rate it sees a twentieth of the read traffic, at 99% a hundredth. Redis here is a cache, not the source of truth — Postgres still owns the data, so if the cache is cold or down, reads fall through correctly, just slower.”

Edge cases I'd raise before shipping: stale reads — a cached copy can lag the DB until its TTL expires or I invalidate on write, so I pick the TTL to bound how wrong it can get; cache stampede — when a hot key expires, thousands of misses can hit the DB at once (the “thundering herd”), so I'd add a short lock or early-recompute so one request reloads it; and the source-of-truth line — never store the last copy of data I can't lose, because eviction or a restart can drop it.

05 Surviving a restart

Persistence: cache → store

RAM is volatile — pull the power and it is gone. If Redis only ever lived in memory, a restart would empty it. For a pure cache that is fine: it just refills from the database on the next miss. But it is fatal if Redis holds anything you can't reconstruct. Persistence is what lets the same engine be more than a cache — it can write its in-memory state to disk and reload it on restart. Two mechanisms, usually combined:

  • RDB (snapshotting) — periodically fork the process and dump the whole dataset to a compact binary file (say, every few minutes or after N changes). It is small on disk and fast to load, ideal for backups. The risk: if the server dies between snapshots, you lose everything written since the last one.
  • AOF (append-only file) — log every write command as it happens; on restart, replay the log to rebuild state. A configurable fsync (every write / every second / let the OS decide) trades durability against throughput. More durable than RDB — you can lose as little as ~1 second of writes — but the file is larger and slower to load, so Redis periodically rewrites and compacts it.

Most production setups run both: AOF for point-in-time durability, RDB for fast restarts and tidy backups. The idea to keep for your mental model is that persistence is a dial you choose. Turn it off and Redis is a pure, disposable cache. Turn on AOF (and pair it with replication) and you have a durable, low-latency store. Same data structures; different durability setting.

Durable, not a database

Even with AOF plus replication, treat Redis's durability as “very good cache / fast store,” not “ACID system of record” for money-critical data. People do run it as a primary store — it is designed to allow that — but the safe interview framing is: Redis gives you a durability knob; turn it up when the data matters and back it with replication and backups, while knowing its core strength is latency and the canonical, must-never-lose data usually still belongs in a transactional database. That is exactly the cache-aside contract from §04, stated as a durability choice.

06 The bigger picture

Cache, and a great deal more

Step back and Redis is best understood as “a cache, and a great deal more.” The same in-memory, atomic, structured engine that caches a product page also implements a surprising range of distributed-systems primitives — because each is just a data structure plus an atomic operation:

  • Queues & streams — Lists as simple job queues (LPUSH/BRPOP); Streams (XADD/XREADGROUP) for durable, consumer-group message processing.
  • Rate limiting — an atomic INCR with EXPIRE, or a sorted-set sliding window, counts requests per user with no database in the loop.
  • Distributed locks — SET key val NX PX 30000 hands one holder a time-boxed lock across many app servers.
  • Pub/Sub — PUBLISH/SUBSCRIBE for fan-out messaging and live updates.
  • Leaderboards & top-N — sorted sets, exactly as in §02.
  • The modern platform (Redis 8) — what used to ship as add-on modules is folded into the server: Search (secondary indexes + full-text), JSON (store and query nested documents), probabilistic and time-series types, and vector search over embeddings — so the same engine that caches your pages also backs semantic search and LLM memory.

It helps to fix Redis's place between the two things it is most often compared to:

Memcached

  • Pure in-memory cache; strings only (opaque blobs).
  • Multi-threaded; no persistence; no rich operations.
  • Great when all you need is a fast, simple key → value cache — and nothing more.

Redis

  • In-memory data-structure server; 9+ types with atomic ops.
  • Single-threaded command model; optional persistence (RDB/AOF) & replication.
  • Cache plus queues, locks, leaderboards, pub/sub, search, vectors.

Relational DB

  • On-disk system of record; full SQL, joins, transactions.
  • Strong durability; reads ~1–10 ms.
  • The source of truth — Redis typically sits in front of it.

The trap to avoid is the false binary “Redis vs a database.” They are usually partners: the database owns the durable, queryable truth; Redis owns the hot, latency-critical slice in front of it. Memcached is what you pick when even Redis is more than you need.

★ Review

Summary & self-test

It all compresses to one idea: Redis keeps your hot data in RAM and serves it through purpose-built data structures, one atomic command at a time — so operations a database charges you milliseconds for happen in microseconds. Cache first; reach for the right structure; bound it with TTL and eviction; turn up persistence when the data matters.
