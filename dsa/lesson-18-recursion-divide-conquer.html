Recursion & Divide-and-Conquer

The hardest mental shift in recursion is learning to trust the subcall — assume it works correctly on a smaller input, write the base case that stops it, and let the stack do the rest. Divide-and-conquer sharpens that into a technique: split a problem in half, solve each half recursively, then combine. That one move turns O(n log n) sorting, O(log n) exponentiation, and a range of tree problems from hard into straightforward.

00 The foundation

The recursive leap

A recursive function solves a problem by solving a smaller version of the same problem. That sounds circular until you notice the two pieces that break the circularity: a base case that returns a known answer without any further recursion, and a recursive step that shrinks the input so the base case is eventually reached.

The key mental model is the recursive leap of faith: when writing the recursive step, assume the function already works correctly on anything strictly smaller. Do not trace through the full call stack in your head — trust the subcall. Your only job is to say, "if I had the answer for a smaller slice, how would I combine it into the answer for the current slice?" That is the entire art.

The call stack unwinds depth-first. Each call is pushed onto the stack when it starts and popped when it returns. For an input of size n with a halving recurrence, the deepest call stack is O(log n) frames. For a linear recurrence (tail call or linear scan), it is O(n). The base case always returns first, and results bubble back up through the previously suspended frames.

The three questions to ask every recursive problem

1. What is the base case? An empty input, a single element, an exponent of 0 — something trivially answerable. 2. What is the recursive step? How does the problem shrink? Does it halve, subtract one, or split along a tree edge? 3. How are results combined? After the subcalls return, what arithmetic, comparison, or merge operation assembles the final answer? Write those three before you touch a keyboard.

Worked problemMediumLeetCode 50
Pow(x, n)

Given a floating-point number x and an integer exponent n (which may be negative or very large), compute x raised to the power n efficiently.

Think it through

Restate

I need xn. The brute force multiplies x by itself n times, which is O(n) multiplications — fine for small n, but n can be up to 2 billion on LeetCode, so that is ~10⁹ operations: too slow.

Brute force

A loop: result = 1; for _ in range(n): result *= x. O(n) time. At n = 2,147,483,647 this is clearly off the table.

The waste

Computing x × x × … × x one at a time re-does work that was already done. x² is (x × x); x⁴ is (x²)²; x⁸ is (x⁴)². Each squaring doubles the exponent for one multiplication instead of two. So I can halve the problem at each step.

The move

If n is even: xn = (xn/2)². If n is odd: xn = x × (xn/2)². Recurse with n//2. The base cases are n == 0 (return 1) and negative n (return 1 / pow(x, −n)). At each level I do one squaring and at most one extra multiply, so the depth is log₂n.

Invariant & cost

Each recursive call halves n, so the call tree has depth O(log n) and exactly O(log n) multiplications total. O(log n) time, O(log n) space on the call stack (or O(1) with iteration).

The code

python

def myPow(x, n):
    if n == 0:
        return 1                         # base case
    if n < 0:
        return myPow(1 / x, -n)         # negative exponent
    half = myPow(x, n // 2)            # solve half the problem
    if n % 2 == 0:
        return half * half               # even: just square
    else:
        return x * half * half           # odd: one extra factor
Narrate it to the interviewer

"Restating: I need x to the power n, and n can be up to about 2 billion, so a simple loop doing n multiplications is O(n) — too slow. The key observation is that x to the n/2, once computed, gives me x to the n by squaring it. So I recurse on n//2, get that result, and square it — handling odd n with one extra factor of x. Base case is n=0, which returns 1. Negative n flips the base to 1/x and negates the exponent. Each call halves n, so the depth is O(log n) and I do exactly O(log n) multiplications. Space is O(log n) on the call stack."

Dry-run aloud: x=2, n=10 → half=pow(2,5)=32, answer=32*32=1024. x=2, n=−3 → becomes pow(0.5, 3)=0.125. n=0 → immediate return 1. n=1 → odd branch: half=pow(x,0)=1, answer = x*1*1=x.

What the interviewer asks next

"Can you do this iteratively?" → Yes: use the binary representation of n. Start with result=1 and base=x; while n>0, if n is odd multiply result by base, then square base and halve n. Same O(log n) time but O(1) space. "What about overflow for very large n?" → In Python integers are arbitrary precision so there is no overflow; in Java or C++ you must handle the 32-bit minimum integer specially because −(−2³¹) overflows.

01 The strategy

Divide-and-conquer: split, solve, combine

Divide-and-conquer is the structured application of recursion to problems that have natural split points. The template is always three phases: divide the input into two (or more) roughly equal sub-problems; conquer each sub-problem recursively; combine the results. The payoff is that if dividing and combining each cost O(n) and you make two halves, you get an O(n log n) algorithm instead of O(n²).

The key question is whether the combine step is fast enough to justify the split. In merge sort the combine (merging two sorted halves) is O(n) — that is good enough. In quicksort the combine is O(1) (nothing to do after partition) but the divide is randomized and can degenerate. The analysis in each case comes from writing the recurrence and applying the master theorem.

When divide-and-conquer wins

  • The problem has optimal substructure: the correct answer for the whole can be assembled from correct answers for each half.
  • The combine step is linear or sub-linear: merging two sorted halves in O(n), finding a max across two results in O(1).
  • Classic examples: merge sort, binary search, quick select, closest pair of points, Karatsuba multiplication.

When to reach for DP instead

  • Sub-problems overlap — the same sub-problem appears in many branches. D&C re-solves it; DP memoizes it.
  • You are minimizing/maximizing over choices, not just combining independent answers.
  • Rule of thumb: if the recursion tree has repeated identical nodes, that is DP territory.
02 Merge sort

Merge sort: the canonical example ▸

Merge sort is the textbook divide-and-conquer algorithm. Split the array in half, sort each half recursively, then merge the two sorted halves into one sorted result. The merge step is the key operation: given two sorted sequences, produce one sorted sequence by always picking the smaller of the two front elements. That merge runs in O(n), and because it is called at each of the O(log n) levels of the recursion tree, the total cost is O(n log n).

The brute force comparison sort — trying all permutations to find the sorted one — would be O(n!). Even insertion sort is O(n²). Merge sort is the canonical argument for why divide-and-conquer beats naive repetition by orders of magnitude.

Why merge sort is stable and predictable

Because we always pick the left element when values tie, merge sort preserves the original relative order of equal elements — it is stable. Unlike quicksort it also has no worst-case degeneration: the split is always exactly in half, so O(n log n) is both average and worst case. The trade-off is O(n) auxiliary space for the output array during merge. Python's built-in timsort is a hybrid merge/insertion sort for exactly these reasons.

Worked problemMediumLeetCode 912
Sort an Array (Merge Sort)

Given an array of integers, return it sorted in ascending order. Do not use a built-in sort — implement an O(n log n) algorithm from scratch.

Think it through

Restate

Sort an arbitrary integer array in O(n log n) worst-case time. The constraint rules out O(n²) approaches like bubble sort or insertion sort.

Brute force

Insertion sort: for each element, insert it into its correct position in the already-sorted prefix. Correct but O(n²) — at n = 50,000 that is 2.5 billion comparisons.

The waste

Insertion sort re-compares elements that are already in known sorted order. Once a prefix is sorted, I know everything about it — I should not re-examine it. Merge sort avoids this by combining two fully sorted halves in a single O(n) pass.

The move

If the array has 0 or 1 element, it is already sorted — that is the base case. Otherwise: find the midpoint; recursively sort the left half; recursively sort the right half; merge the two sorted halves into a new array. Return the merged array.

Invariant & cost

At every merge call, both inputs are sorted; after the merge the output is sorted. The recurrence is T(n) = 2T(n/2) + O(n), which solves to O(n log n) time, O(n) space for the auxiliary merge buffer.

The code

python

def sortArray(nums):
    if len(nums) <= 1:
        return nums                   # base case: 0 or 1 element is sorted
    mid = len(nums) // 2
    left  = sortArray(nums[:mid])    # sort left half
    right = sortArray(nums[mid:])    # sort right half
    return merge(left, right)          # combine

def merge(a, b):
    out, i, j = [], 0, 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:              # pick the smaller front element
            out.append(a[i]); i += 1
        else:
            out.append(b[j]); j += 1
    out.extend(a[i:]); out.extend(b[j:])  # drain whichever tail remains
    return out
Narrate it to the interviewer

"I can't use the built-in sort, so I need O(n log n) from scratch. Insertion sort is O(n²) — too slow. The divide-and-conquer approach is merge sort: if the array has at most one element, it is already sorted — that is my base case. Otherwise, I split at the midpoint, sort each half recursively — trusting those subcalls — and then merge the two sorted halves. Merging walks both halves with a pointer each, always taking the smaller front element, and drains any leftover tail. The invariant is that both halves are fully sorted before the merge, so the merge is correct. Recurrence is T(n) = 2T(n/2) + O(n), which is O(n log n) by the master theorem. Space is O(n) for the merge buffer plus O(log n) stack depth."

Dry-run aloud: empty array returns immediately. One element returns immediately. Two elements [3,1] → left=[3], right=[1], merge picks 1 then 3 → [1,3]. All identical elements [5,5,5]: all merges pick from left first when equal, so output is [5,5,5] — stable and correct.

03 Fast power

Fast exponentiation by halving ▸

The Pow(x, n) problem from section 00 is best explained visually as a recursion tree that collapses exponentially. Each level halves the exponent, so the tree depth is exactly ⌊log₂ n⌋. At each node there is at most one squaring and one extra multiplication. The total cost is therefore the depth times O(1) per node — O(log n) multiplications, no matter how large n is.

This is the same structural argument as binary search: by cutting the problem in half at every step, you convert a linear number of operations into a logarithmic one. That halving is the essence of divide-and-conquer on a single branch (no combine step needed — just recursive squaring).

The iterative version (O(1) space)

The same idea works bottom-up without recursion: read the bits of n from least significant to most significant. Start with result = 1 and base = x. For each bit: if the bit is 1, multiply result by base; then square base. After all bits, result holds xn. This is called binary exponentiation or exponentiation by squaring, and it is used everywhere from cryptography (modular exponentiation) to competitive programming.

04 Cost analysis

Recurrence and the master theorem

When you write a divide-and-conquer algorithm, the running time obeys a recurrence relation. For algorithms that split into two halves and do O(n) combining work, the recurrence is:

T(n) = 2 · T(n/2) + O(n)

You can unroll this by hand: at the top level, O(n) combine work. At the next level, 2 subproblems each doing O(n/2) work — still O(n) total. At level k there are 2k subproblems each of size n/2k, still O(n) total per level. There are log₂ n levels before n/2k hits 1. So the total is O(n) per level times O(log n) levels equals O(n log n).

The master theorem makes this mechanical. For T(n) = a · T(n/b) + O(nd): compare logb(a) with d. If they are equal (as in merge sort: a=2, b=2, d=1, so log₂2=1=d) the answer is O(nd log n). The three cases of the master theorem cover the space of outcomes without needing to unroll every time.

Master theorem quick reference

T(n) = a · T(n/b) + O(nd). Let c = logb(a). If c < d: work dominates at the top → O(nd). If c = d: work is balanced across all levels → O(nd log n). If c > d: leaves dominate → O(nc) = O(nlog_b(a)). Merge sort has a=2, b=2, d=1, c=1 — the balanced case → O(n log n). Binary search has a=1, b=2, d=0, c=0 — balanced → O(log n). Knowing which case you are in before you start is a strong interviewer signal.

05 Majority element

Majority element by divide-and-conquer ▸

Majority Element (LeetCode 169) asks for the element that appears more than n/2 times in an array. The well-known O(n)/O(1) solution is Boyer–Moore voting. But the divide-and-conquer solution is a beautiful demonstration of the pattern, and it is often the intended answer when an interviewer asks you to use recursion.

The divide-and-conquer insight: split the array in half. The majority element of the whole array must be the majority element of at least one half (because if it were a minority in both halves, it could appear at most n/2 times total — contradicting its majority status). So solve both halves, get their majority candidates, then count each candidate in the full array to determine which one wins.

The brute force is to count every element's frequency: O(n) with a hash map, but O(n) space. The naive sorting approach is O(n log n) just to find the middle element. The divide-and-conquer approach runs in O(n log n) time and O(log n) space (just the call stack) — and it is a compelling way to demonstrate the pattern to an interviewer who already knows Boyer–Moore.

Worked problemEasyLeetCode 169
Majority Element

Given a non-empty array of integers, find the element that appears more than ⌊n/2⌋ times. You may assume such an element always exists.

Think it through

Restate

One element appears more than half the time. Find it. The key constraint — it always exists — means I never need to handle the "no majority" case.

Brute force

Count frequencies with a dict; return the element with count > n//2. O(n) time, O(n) space. Valid, but O(n) space; the interviewer may push for something more interesting.

The waste

The hash map stores every distinct value. The divide-and-conquer argument shows that I only ever need to compare two candidates — one from each half — and verify the winner. That uses O(1) extra space per recursion level, O(log n) total.

The move

Base case: a single-element array returns that element. Otherwise: split at midpoint. Recursively find the majority candidate in each half. Count both candidates in the full array slice. Return the one with the higher count. The majority of the full array must be the majority of at least one half — so the correct answer is always one of the two candidates.

Invariant & cost

At every level, the returned candidate is the majority of its sub-array. Recurrence: T(n) = 2T(n/2) + O(n) (the O(n) comes from the count step). O(n log n) time, O(log n) space.

The code

python

def majorityElement(nums):
    def helper(lo, hi):
        if lo == hi:
            return nums[lo]               # base case: single element
        mid = (lo + hi) // 2
        left  = helper(lo, mid)           # majority candidate of left half
        right = helper(mid + 1, hi)       # majority candidate of right half
        if left == right:
            return left                   # same candidate: no count needed
        # count both in the full slice [lo..hi]
        lc = nums[lo:hi+1].count(left)
        rc = nums[lo:hi+1].count(right)
        return left if lc >= rc else right

    return helper(0, len(nums) - 1)
Narrate it to the interviewer

"The easiest solution is a frequency dictionary — O(n) time, O(n) space. I can do better on space with a divide-and-conquer approach. The key insight: the majority element appears more than n/2 times, so it must be the majority in at least one half. So I split the array, get the majority candidate from each half recursively, then count both candidates in the current slice and return the one with the higher count. The base case is a single element — it is trivially its own majority. The invariant is that the returned value is always the majority of its sub-array, so one of the two candidates is always correct. The recurrence is T(n) = 2T(n/2) + O(n), giving O(n log n) time and O(log n) stack space."

Dry-run aloud: single-element array [7] → returns 7 immediately. Two identical elements [3,3] → left=3, right=3, same → return 3. Two different elements [4,3] where 4 is majority is impossible since neither exceeds n/2=1 for n=2 — the problem guarantees a majority exists, so this degenerate case cannot arise. Array [3,3,4]: left=helper(0,1) → left=3, right=3, same → 3; right=helper(2,2)=4; count 3 twice vs 4 once → return 3.

What the interviewer asks next

"Can you do O(n) time and O(1) space?" → Yes: Boyer–Moore voting. Maintain a candidate and a count. For each element: if count is 0, set the candidate to this element; if the element equals the candidate, increment count; otherwise decrement count. At the end, the candidate is the majority. "Why does Boyer–Moore work?" → because a majority element survives the cancellation process; every non-majority vote is cancelled by a different value, but the majority has > n/2 votes so it outlasts all cancellations. "What if no majority is guaranteed?" → do a second pass to verify the Boyer–Moore candidate's count actually exceeds n/2.

06 Practice

Practice ladder

Recursion fluency comes from writing it, not reading it. For each problem below, start by writing the three parts — base case, recursive step, combine — before you touch a loop or conditional. State the recurrence and which case of the master theorem it falls into.

EasyFibonacci Numberbasic recursion

Compute the n-th Fibonacci number recursively, then improve it so you never recompute a subproblem.

Idea Naive recursion is O(2^n); add memoization for O(n). State the base cases: F(0)=0, F(1)=1.

EasyMerge Sorted Arraymerge step

Given two sorted arrays, merge them into a single sorted array. This is the combine step of merge sort in isolation.

Idea Two pointers, one per array; always pick the smaller front element. O(m+n) time.

EasyMaximum Depth of Binary Treetree recursion

Find the longest root-to-leaf path in a binary tree, counting edges as depth.

Idea Depth = 1 + max(depth(left), depth(right)); base case is a null node returning 0.

EasyPow(x, n)fast exponentiation

Raise a float to an integer power — positive, zero, or negative — without a loop over all n steps.

Idea Exactly FIG. 3 — halve the exponent each call. O(log n) multiplications.

MediumSort an Arraymerge sort

Sort an array of integers in O(n log n) worst-case time without using any library sort.

Idea Exactly the merge sort from section 02. Recurrence T(n) = 2T(n/2) + O(n) = O(n log n).

MediumMajority Elementd&c vote

Find the element appearing more than n/2 times using the divide-and-conquer approach from section 05.

Idea Two candidates from halves; count both in full slice; return winner. O(n log n), O(log n) space.

MediumKth Largest Element in an Arrayquickselect

Find the k-th largest element without fully sorting the array every time.

Idea Partition around a pivot (like quicksort); recurse only into the half containing rank k. Average O(n), worst O(n²); randomize pivot to minimize worst case.

MediumSearch a 2D Matrix IId&c on grid

Search a matrix where each row and column is sorted. Find a target value in O(m + n) time.

Idea Start at the top-right corner; if the value is too large move left, too small move down. This is converging pointers on a sorted 2D structure.

MediumDifferent Ways to Add Parenthesesd&c on expression

Given a string of numbers and operators, return all possible results from different ways to parenthesize the expression.

Idea Split on each operator; recursively get all results from each side; combine every left result with every right result using the operator.

HardCount of Smaller Numbers After Selfmerge sort + index tracking

For each element, count how many elements to its right are strictly smaller. Return these counts as an array.

Idea Augment the merge step of merge sort: when a right-half element is placed before a left-half element, all remaining left-half elements get their count incremented. O(n log n).

HardThe Skyline Problemd&c merge

Given rectangles representing buildings, produce the outline (skyline) as a list of key points where the height changes.

Idea Divide buildings into two halves; recursively compute each half's skyline; merge the two skylines by scanning with two pointers and tracking the running maximum of each. O(n log n).

HardMedian of Two Sorted Arraysbinary search + d&c

Find the median of two sorted arrays in O(log(m+n)) time — no merging allowed.

Idea Binary search on the smaller array's partition point. At each step, half the remaining search space is eliminated. This is divide-and-conquer on the search space itself.

How to drill recursion for an interview

Every time you write a recursive solution, say three things out loud before coding:
(1) the base case and why it is correct;
(2) the recursive step and why it strictly shrinks the problem;
(3) the combine step and what it assumes the subcalls returned. Then state the recurrence and the resulting complexity. Out loud narration is the skill — interviewers hire for the ability to explain, not just to produce correct code.

★ Review

Summary & self-test

Recursion is the skill of trusting your own function to handle a smaller input correctly and then building the full answer from that smaller result. Divide-and-conquer sharpens this into a competitive algorithm: split the input roughly in half, conquer each half independently by recursion, and combine in O(n) or better. The payoff is that O(n²) brute-force problems collapse to O(n log n), and O(n) problems collapse to O(log n) — as long as the combine step is fast enough. The master theorem quantifies the cost without guessing.
