Graph Representations

A graph is the most general structure in the course — once you know how to encode one, an enormous family of problems (connectivity, shortest paths, dependency ordering, region counting) all become the same set of traversal templates applied to a different shape of data.

00 Foundation

What is a graph?

A graph is a collection of nodes (also called vertices) and edges that connect pairs of nodes. That is the whole definition — and it is deliberately bare, because almost anything can be modelled as a graph once you decide what the nodes and edges represent. Cities and roads, users and friendships, tasks and dependencies, cells in a grid: all graphs in disguise.

Three properties are worth nailing down immediately, because they determine which algorithm you reach for and which representation you choose:

  • Directed vs undirected. In a directed graph (digraph) each edge has an arrow: A → B does not imply B → A. In an undirected graph every edge is a two-way street. Friendships are undirected; web page links are directed.
  • Weighted vs unweighted. Edges can carry a numeric label — a distance, a cost, a capacity. When all weights are equal (or you only care about reachability), the graph is effectively unweighted.
  • Sparse vs dense. A graph with V nodes can have at most V² edges (dense) or just a handful (sparse). This ratio is the single biggest driver of which data structure to use for storage.
Degree vocabulary

In an undirected graph, the degree of a node is its edge count. In a directed graph, it splits into in-degree (arrows arriving) and out-degree (arrows leaving). The handshaking lemma says that in any undirected graph the sum of all degrees equals twice the number of edges — a useful sanity check when building an adjacency list by hand.

01 The workhorse

Adjacency list: building it from edges

The adjacency list is the default representation for interview problems. For each node you keep a list of its neighbours. In Python a dictionary of lists is the natural choice: graph[u] yields every node reachable from u in one edge.

Building from an edge list is a simple loop: for each pair (u, v), append v to graph[u]. For an undirected graph, also append u to graph[v]. Watch the seven insertions below, one edge at a time.

python

from collections import defaultdict

def build_adj_list(n, edges):
    # n = number of nodes (0-indexed); edges = list of (u, v) pairs
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)  # directed: just this line
        graph[v].append(u)  # undirected: also the reverse
    return graph

# example: 5 nodes, 5 edges
edges = [(0,1), (0,2), (1,3), (2,3), (3,4)]
g = build_adj_list(5, edges)
# g[0] = [1, 2],  g[1] = [0, 3],  g[2] = [0, 3]
# g[3] = [1, 2, 4],  g[4] = [3]
The reusable shape

In traversal code (BFS, DFS) the adjacency list is the only thing that changes between problems. The traversal template always looks the same: for neighbour in graph[node]:. Nail the build step once, and all traversal problems reduce to knowing when to visit, what to track, and what to return.

02 The alternative

Adjacency matrix and the trade-off

The adjacency matrix is a V × V grid where mat[u][v] = 1 if the edge (u, v) exists (or the edge weight if weighted). It is immediate to answer "does the edge u–v exist?" in O(1), but it always costs O(V²) space regardless of how many edges there actually are.

Operation Adjacency List Adjacency Matrix
Space O(V + E) O(V²)
Edge lookup: does (u,v) exist? O(deg(u)) O(1)
Iterate all neighbours of u O(deg(u)) O(V) — must scan full row
Add an edge O(1) O(1)
Best for Sparse graphs (most interview problems) Dense graphs, fast edge-existence queries
Default pick in interviews

Unless the problem explicitly involves dense graphs or requires O(1) edge lookups (e.g., checking for specific pairs in a complete graph), always reach for the adjacency list. Most interview graphs are sparse — a grid, a tree, or a graph with V nodes and roughly V edges — and the adjacency list's O(V + E) space keeps BFS and DFS visits proportional to actual work done.

03 Variants

Directed graphs and weighted edges

The build function from section 01 has a one-line difference for directed graphs: skip the reverse insertion. For weighted edges, store a tuple (neighbour, weight) instead of a bare node. Everything else — the dictionary shape, the traversal template — stays identical.

python

from collections import defaultdict

def build_directed_weighted(edges):
    # edges = list of (u, v, weight)
    graph = defaultdict(list)
    for u, v, w in edges:
        graph[u].append((v, w))  # directed: no reverse entry
    return graph

# in-degree / out-degree counters (useful for topological sort)
def compute_indegrees(n, graph):
    indeg = [0] * n
    for u in graph:
        for v, _ in graph[u]:
            indeg[v] += 1
    return indeg
In-degree matters

In-degree is the number of incoming edges. A node with in-degree 0 has no prerequisites — it is a safe starting point for topological sort (Lesson L24). Tracking in-degrees during the build step is cheap (one extra pass) and avoids a full re-scan later.

04 The real skill

Modelling a problem as a graph

Choosing a representation is a mechanical step. The harder skill — and the one interviewers are actually testing — is recognising that a problem is a graph problem and deciding what the nodes and edges represent. There are two common disguises to learn:

  • Grid as a graph. A 2D grid is a graph where every cell (r, c) is a node, and two cells share an edge if they are adjacent (4-directional for most problems, 8-directional occasionally). You never build an explicit adjacency list; you just generate neighbours on the fly with four coordinate deltas.
  • State as a node. In puzzle-style problems (word ladder, shortest transformation sequence) each distinct configuration is a node, and two states are connected if one legal move transforms one into the other. The graph is implicit; you define neighbours by the problem's rules.
Implicit vs explicit graphs

Most grid problems and puzzle problems use an implicit graph: you define a neighbours(node) function rather than building a dictionary upfront. The traversal code is identical — the only difference is where neighbours come from. For small graphs given as edge lists, build explicitly; for grids and states, generate on the fly.

05 See it used

Grid-as-graph: Number of Islands

Number of Islands (LeetCode 200) is the canonical modelling exercise. The grid is a graph. Each '1' cell is a node. Two '1' cells share an edge if they are horizontal or vertical neighbours. An island is a connected component. Count the components.

Watch a BFS frontier expand from a starting cell, marking cells as visited so they are never re-counted. Each time the frontier empties on an unvisited '1', a new island has been found.

python

from collections import deque

def num_islands(grid):
    if not grid: return 0
    rows, cols = len(grid), len(grid[0])
    visited = set()
    islands = 0

    def bfs(r, c):
        q = deque([(r, c)])
        visited.add((r, c))
        while q:
            row, col = q.popleft()
            for dr, dc in [(-1,0),(+1,0),(0,-1),(0,+1)]:
                nr, nc = row+dr, col+dc
                if (0 <= nr < rows and 0 <= nc < cols
                        and grid[nr][nc] == '1'
                        and (nr, nc) not in visited):
                    visited.add((nr, nc))
                    q.append((nr, nc))

    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == '1' and (r, c) not in visited:
                bfs(r, c)
                islands += 1
    return islands
How to narrate this in an interview

"I'm treating the grid as a graph: each land cell is a node, and two cells share an edge if they are 4-directional neighbours. Counting islands means counting connected components. I scan every cell; when I hit an unvisited land cell I launch a BFS that marks the whole component as visited, then I increment the island counter. Each cell is enqueued at most once, so the total work is O(rows * cols) time and space. Edge cases: empty grid returns 0; a single land cell with no neighbours is one island; a grid of all water returns 0."

In-place vs visited set

Many solutions overwrite visited '1' cells with '0' to skip the extra visited set. That is O(1) space but mutates the input — always confirm in an interview whether mutation is allowed. If the grid is immutable or you need to restore it afterward, keep the explicit set.

06 What comes next

Where this shows up next

Graph representations are a prerequisite, not a standalone skill. Every lesson that follows builds directly on the adjacency list (or implicit grid graph) covered here. Once you are comfortable with the build step and the modelling instinct, the pattern lessons slot in naturally.

Study order

Spend a session building graphs by hand from edge lists, then drawing their adjacency lists and matrices. Try modelling three problems as graphs on paper — decide what the nodes are, what the edges are, and whether the graph is directed — before you ever touch traversal code. Modelling fluency is the bottleneck for most learners, not the traversal algorithms themselves.

★ Review

Summary & self-test

A graph is nodes and edges. The adjacency list (dictionary of neighbour lists) is the default for sparse graphs; the matrix is O(1) for edge lookup but wastes O(V²) space. The real interview skill is modelling: recognising that a problem is a graph, deciding what nodes and edges mean in context, and knowing when the graph is implicit (grids, states) vs explicit (given edge list).

01Two representationsAdjacency list: O(V+E) space, fast neighbour iteration. Matrix: O(V²) space, O(1) edge lookup.
02Default: listMost interview graphs are sparse. Use adjacency list unless you need O(1) edge existence checks on a dense graph.
03Directed vs undirectedDirected: insert one direction only. Undirected: insert both. One line of code difference in the build step.
04In-degree mattersFor directed graphs, in-degree = number of incoming edges. Nodes with in-degree 0 are topological sort starting points.
05Implicit graphsGrids and puzzle states: define a neighbours() function instead of building a dictionary. Traversal code is identical.
06Modelling firstName the nodes. Name the edges. State whether directed and weighted. Only then choose a representation.
07Grid = graphcell (r,c) is a node; 4-directional neighbours are edges. Connected land cells = island = connected component.
08Visited setNever re-enqueue a visited node. Mark before enqueuing (BFS) or at entry (DFS) to avoid duplicate work and infinite loops.
