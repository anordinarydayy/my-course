Sliding Window

Whenever a problem asks about a contiguous subarray or substring — the longest, the shortest, the best — your first question should be: can a window expand and shrink its way to the answer in a single left-to-right sweep? The sliding window turns a nest of O(n²) re-scans into a clean O(n) pass by carrying just enough state to move forward without ever looking back.

00 The idea

What problem does it solve?

Here is the canonical opening: given an array of numbers and a window size k, find the maximum sum of any k consecutive elements. The first instinct is to write two nested loops — for every starting index, scan the next k cells and sum them. That works, but it repeats almost all the work on each step: shifting the window by one discards one value and adds one, yet the inner loop recomputes the entire sum from scratch. For n = 100,000 and k = 50,000 that is five billion additions.

The redundancy is the window's interior — all the values between the new left edge and the new right edge are touched again even though they haven't changed. The sliding window trick carries the running total across steps: as the window moves one position right, subtract the element that just left the left edge and add the element that just entered on the right. One subtraction plus one addition per step, regardless of k — making the total cost O(n).

That is the fixed-size case. The more interesting question is: what if the optimal window size is not given, and you must search for it? That leads to the variable window, which is the sibling idea: expand the right edge to include new elements, and shrink the left edge whenever the window's contents violate a constraint, restoring validity before moving on. Because each element enters the window exactly once and leaves exactly once, the total work remains O(n) even though the window size changes at every step.

The two flavors at a glance

Fixed size — k is given. Compute the first window's answer, then slide one step at a time: subtract what leaves, add what enters. Never re-scan the interior. Variable size — no k given; you search for the optimal window. Expand the right edge until the window becomes invalid; shrink the left edge until it is valid again; record the best so far. Each element is processed at most twice: once entering, once leaving.

01 Fixed size

Rolling the window at constant cost

The brute force for "maximum sum of k consecutive elements" re-sums a window of length k from scratch every time it shifts — O(n·k) total. The key observation: when the window moves one step right, only two elements change. Everything in between is unchanged. So we maintain a running sum and perform one subtraction (the departing left element) and one addition (the arriving right element) per step — O(1) per shift, O(n) total.

The pattern is: compute the first window exactly, then loop from k to n-1: win_sum += a[r] - a[r - k]. That single line encodes both updates. Update the best answer after each shift and you are done.

This technique applies to any aggregate that supports efficient "add one element, remove one element" updates — sums, products, counts, character frequencies. The frequency map variant is explored in section 03.

Fixed window (rolling sum)

  • O(n) time — one arithmetic op per step.
  • O(1) space — only the running sum and best-so-far.
  • k must be known upfront; window never changes size.

Brute force (re-sum)

  • O(n·k) time — the full window re-examined at every start.
  • O(1) space — same, but the time cost is unacceptable at scale.
  • Fails the time limit when n and k are both large.
Worked problemEasyLeetCode 643
Maximum Average Subarray I

You have an integer array and a positive integer k. Find the contiguous subarray of length exactly k that has the greatest average value, and return that average as a float.

Think it through

Restate

The window size is fixed at k. Maximize the sum over all windows of that size, then divide by k. Maximizing the sum is enough — dividing by the same k at the end doesn't change which window wins.

Brute force

For each starting index i from 0 to n-k, sum elements i through i+k-1. That is O(k) per window times O(n) windows = O(n·k). For n = 100,000 and k = 50,000 that is five billion operations.

The waste

We recompute the sum of the k-2 middle elements on every step, even though they haven't changed. The new window's sum differs from the old window's sum by exactly two elements: the one that left on the left, and the one that joined on the right.

The move

Sum the first k elements exactly. Then, for each subsequent position, do win_sum += a[r] - a[r - k]. That single expression subtracts the departing element and adds the arriving one. Track the maximum sum seen across all positions and divide at the end.

Invariant & cost

At every moment win_sum holds the exact sum of the k elements ending at index r. The window slides n-k times, each in O(1). Total: O(n) time, O(1) space.

The code

python

def findMaxAverage(nums, k):
    win_sum = 0
    for i in range(k):         # sum the first window exactly
        win_sum += nums[i]
    best = win_sum

    for r in range(k, len(nums)):   # slide right: add new, drop old
        win_sum += nums[r] - nums[r - k]
        if win_sum > best:
            best = win_sum

    return best / k
Narrate it to the interviewer

"The window size is fixed, so this is the fixed sliding window. The brute force sums k elements from scratch at every start — O(n·k). But the only thing that changes when the window moves right by one is: one element exits on the left and one enters on the right, so the sum updates in O(1). I compute the first window exactly, then for each subsequent position I do win_sum += nums[r] - nums[r-k], track the best, and divide by k at the end. Invariant: win_sum always equals the exact sum of the k elements ending at index r. That's O(n) time, O(1) space."

Dry-run aloud: k equals n (one window, first pass gives the answer directly); all identical elements (every window ties, first position wins); negative numbers throughout (the rolling sum still finds the least-negative window correctly).

What the interviewer asks next

"What if there are negative numbers?" — the same code works because we are maximizing a sum, not requiring the sum to be positive. "What if k is larger than the array?" — handle with a guard before the loop or rely on LeetCode's constraint that k ≤ n. "What about an arbitrary aggregate, not just a sum?" — any aggregate that supports O(1) add/remove (sum, product, character count, max-heap) can be carried through a sliding window.

02 Variable size

Expand right, shrink left to restore validity

When the optimal window size is unknown, you search for it. The template is: expand the right edge to include the next element, then — if the window has become invalid — shrink the left edge one step at a time until the window is valid again, and record the best answer before moving on.

Why is this O(n) when the shrink loop can run many times? Because each element enters the window exactly once (when right sweeps over it) and leaves at most once (when left passes over it). The total number of left advances across the entire run is bounded by n, not by n per step. The inner while loop never causes O(n²) behavior — the work is amortized.

Two subtleties of the shrink direction are worth internalizing: (1) you update your running state before checking validity, so the state is always accurate; (2) after shrinking, the window is valid again — you record the best answer once per outer iteration, not once per shrink step.

The reusable shape

Almost every variable-window problem fits this skeleton: add a[R] to state; while window invalid: remove a[L] from state, L++; record best(R - L + 1). The only parts that change between problems are (a) what "state" tracks, (b) what "invalid" means, and (c) whether you record the answer when the window is valid or right before it becomes valid.

Worked problemMediumLeetCode 209
Minimum Size Subarray Sum

Given an array of positive integers and a target value, return the length of the shortest contiguous subarray whose sum is greater than or equal to the target. Return 0 if no such subarray exists.

Think it through

Restate

I need the smallest window whose sum meets or exceeds the target. No fixed size is given — the window must grow and shrink dynamically.

Brute force

Try every pair (i, j) with j ≥ i: sum a[i..j] and check. That's O(n²) pairs times O(n) summation or O(n²) with a prefix sum. For n = 100,000 that is 10¹⁰ or 10¹⁰ operations respectively — both too slow.

The waste

Once the sum of a window reaches the target, any extension to the right only makes the window longer — we should immediately try to shrink from the left. The brute force re-scans the same prefixes because it never uses this monotonicity.

The move

Expand right, accumulating the sum. The moment the sum reaches the target, record the window length and shrink from the left (subtracting a[L] and incrementing L) for as long as the sum stays valid. Because all values are positive, shrinking can only decrease the sum — so we stop as soon as we drop below target again. This finds the minimum valid window for every right-boundary position.

Invariant & cost

At every moment, the sum in the window is exact. Each element is added once (right expands over it) and removed at most once (left passes over it). Total: O(n) time, O(1) space. Note this relies on all values being positive — if negatives were allowed, shrinking might not monotonically reduce the sum.

The code

python

def minSubArrayLen(target, nums):
    L = win_sum = 0
    best = float('inf')          # start with "no window found"

    for R in range(len(nums)):    # expand right
        win_sum += nums[R]
        while win_sum >= target:   # shrink left while valid
            best = min(best, R - L + 1)
            win_sum -= nums[L]
            L += 1

    return 0 if best == float('inf') else best
Narrate it to the interviewer

"This is the variable sliding window. The brute force tries every subarray — O(n²). But because all values are positive, once the window's sum hits the target, extending further right only makes it longer — so I should shrink from the left immediately. I maintain a running sum; right expands one step at a time; whenever the sum meets the target, I record the window length and shrink from the left as far as the sum allows, giving me the minimum valid window for this right boundary. Invariant: win_sum always equals the sum of the current window. Each element is added once and removed at most once — O(n) time, O(1) space. I return 0 if best never updated."

Dry-run aloud: a single-element array equal to target (answer is 1); all elements smaller than target and even their total is under target (return 0); target is 1 and all values are positive (answer is 1, first element).

03 Window + hashmap

Tracking character counts inside the window

Many substring problems ask about which characters are in the window, not just their sum. For those, you replace the running-sum state with a frequency map — a dictionary mapping each character to its count inside the current window. Expanding right increments the count; shrinking left decrements it.

The key insight is that the hashmap update is still O(1) per element (amortized), so the overall complexity stays O(n). The window's validity condition changes — instead of "sum ≥ target" you might check "all required characters are satisfied" or "no character appears more than once" — but the expand-then-shrink skeleton is identical.

Two classic examples: Longest Substring Without Repeating Characters (LeetCode 3) uses a hashmap to track the last index of each character, shrinking left past any repeat; Permutation in String (LeetCode 567) uses two frequency maps and checks whether they match at each fixed-size window position. Both are worked below.

Worked problemMediumLeetCode 3
Longest Substring Without Repeating Characters

Given a string, find the length of the longest contiguous substring in which every character appears exactly once.

Think it through

Restate

Maximize the window length subject to: no character appears more than once inside the window. The window size is not fixed.

Brute force

Try every starting index; from each start, extend right until a repeat is found. Store visited characters in a set. O(n²) pairs, each inner loop potentially O(n) — overall O(n²).

The waste

When we find a repeat at position R, the brute force throws away all the work done building the current set and restarts from i+1. But we know the repeat character is at some position inside [L, R-1]. We only need to advance L past that position — everything else stays valid.

The move

Maintain a set (or dict of last-seen indices) of the characters currently in the window. Expand right: if s[R] is not in the set, add it and record the window length. If it is in the set, shrink L: remove s[L] from the set and increment L — repeat until the duplicate is gone, then add s[R]. A faster variant stores each character's last-seen index; when a repeat is found, jump L directly to max(L, last[s[R]] + 1), adding s[R] in one step.

Invariant & cost

The set always reflects exactly the characters in s[L..R], and the window never contains a duplicate. Each character enters and leaves the window at most once: O(n) time, O(min(n, Σ)) space where Σ is the alphabet size (26 for lowercase, 128 for ASCII).

The code

python

def lengthOfLongestSubstring(s):
    last_seen = {}               # char → its most recent index
    best = 0
    L = 0

    for R, ch in enumerate(s):
        if ch in last_seen and last_seen[ch] >= L:
            L = last_seen[ch] + 1   # jump L past the old occurrence
        last_seen[ch] = R
        best = max(best, R - L + 1)

    return best
Narrate it to the interviewer

"The window must stay duplicate-free. The brute force restarts from i+1 every time — O(n²). Instead, I keep a map of each character's last-seen index. When I see a repeat at R, I jump L directly past that old occurrence — no need to shrink one step at a time. Invariant: every character between L and R appears exactly once. I update the map on every step, and track the best window length. O(n) time, O(alphabet size) space. The guard last_seen[ch] >= L is crucial — without it, a character seen before the current window could incorrectly push L backward."

Dry-run aloud: empty string (return 0); all identical characters like "aaaa" (L catches up to R each step, answer is 1); all unique characters (L never moves, answer is n); a character repeating after a long gap ("abcdefga" — the second 'a' at index 7 pushes L to 1, best is 7).

Worked problemMediumLeetCode 567
Permutation in String

Given two strings s1 and s2, return true if any permutation of s1 appears as a contiguous substring of s2. The check is case-sensitive and considers all 26 lowercase letters.

Think it through

Restate

A permutation of s1 is any rearrangement of its characters. Two strings are permutations of each other if and only if their character frequency maps are identical. I want to know whether any window of length len(s1) in s2 has the same character frequencies as s1.

Brute force

For each starting position in s2, extract the substring of length len(s1) and sort both to compare — O(n · k log k) — or compare frequency maps, still O(n · k) because re-building the map from scratch costs O(k). Both are too slow for large inputs.

The waste

Just like the rolling sum, recomputing the frequency map from scratch for each starting position re-counts the k-2 middle characters on every step. Sliding the window by 1 only changes two characters: one enters, one leaves.

The move

The window size is fixed at len(s1). Build the frequency map for s1 and the first window of s2. At each step, add the new right character and remove the outgoing left character from the window's map. After each update, check if the two maps match. Instead of comparing full maps each time, track a count of "satisfied letters" — a letter is satisfied when its window count matches s1's count. A match is when all 26 letters are satisfied.

Invariant & cost

The window frequency map always reflects exactly the characters in the current window of length k. The "matched" count gives an O(1) validity check. Sliding n-k times: O(n) time, O(1) space (fixed-alphabet frequency maps are constant size).

The code

python

from collections import Counter

def checkInclusion(s1, s2):
    k = len(s1)
    if k > len(s2):
        return False

    need = Counter(s1)                       # target frequencies
    have = Counter(s2[:k])                   # first window
    matched = sum(1 for c in need if have[c] == need[c])

    if matched == len(need):
        return True

    for r in range(k, len(s2)):
        add_ch = s2[r]                       # character entering on the right
        rem_ch = s2[r - k]                   # character leaving on the left

        # add entering character
        have[add_ch] += 1
        if add_ch in need:
            if have[add_ch] == need[add_ch]:
                matched += 1
            elif have[add_ch] == need[add_ch] + 1:
                matched -= 1               # just went over; was satisfied before

        # remove leaving character
        have[rem_ch] -= 1
        if rem_ch in need:
            if have[rem_ch] == need[rem_ch]:
                matched += 1
            elif have[rem_ch] == need[rem_ch] - 1:
                matched -= 1               # just went under; was satisfied before

        if matched == len(need):
            return True

    return False
Narrate it to the interviewer

"Two strings are permutations of each other iff their character frequency maps are identical. So I need: does any fixed-size window of length len(s1) in s2 match s1's frequency map? The brute force rebuilds the map from scratch at each start — O(n·k). The sliding window carries the map: one character enters on the right, one leaves on the left, each in O(1). I avoid comparing the full maps on every step by maintaining a 'matched letters' count — a letter is matched when its window count equals s1's count. When all len(need) letters are matched, I found a permutation. Invariant: have[c] always equals the count of c in the current window. O(n) time, O(1) space since the maps are bounded by the 26-letter alphabet."

Dry-run aloud: s1 longer than s2 (return false immediately); s1 is a single character (window size 1, just scan s2); s1 equals s2 (first window matches); the permutation only appears at the very end of s2.

What the interviewer asks next

"What about Minimum Window Substring?" — that is the harder sibling: variable window, tracking how many of s1's required characters are currently satisfied in the window; expand until all are satisfied, shrink to minimize. The "matched" count trick from Permutation in String carries over directly. "Why O(1) space?" — because both maps have at most 26 keys. If the alphabet were arbitrary Unicode, it would be O(Σ) space.

04 Exactly-k trick

Exactly k occurrences via at-most subtraction

Some problems ask for subarrays with exactly k occurrences of something — exactly k distinct characters, exactly k odd numbers, and so on. Variable windows handle "at most k" naturally: expand right, and whenever the count exceeds k, shrink left until it is k again. "Exactly k" is trickier because validity requires both "not exceeding k" and "reaching k."

The exactly-k trick sidesteps the issue with arithmetic:

count(exactly k) = count(at most k) − count(at most k−1)

Write a helper that counts subarrays with at most k occurrences using the standard variable window. Call it twice — with k and with k-1 — and subtract. This gives the count for exactly k in O(n) time with no special casing.

Worked problemHardLeetCode 992
Subarrays with K Different Integers

Given an array of positive integers and a value k, count the number of contiguous subarrays that contain exactly k distinct integers.

Think it through

Restate

Count subarrays where the number of distinct values is exactly k. We want a count, not the subarrays themselves.

Brute force

Try every subarray (i, j), compute its set of distinct values, count when that size equals k. O(n²) subarrays, O(n) set computation each — O(n²) total, or O(n²) with careful incrementalness still won't avoid quadratic counts.

The waste

A standard variable window tracking "distinct count ≤ k" only handles "at most k" — it can't directly enforce equality, because a window that is valid at-most-k might be smaller than k or exactly k. Trying to add an "at least k" condition breaks the shrink-while-invalid loop.

The move

Write a helper at_most(k) that uses a standard variable window with a frequency map, shrinking whenever the number of distinct keys in the map exceeds k, and sums (R - L + 1) at each step (each new right boundary adds that many valid subarrays ending there). Then return at_most(k) - at_most(k-1).

Invariant & cost

Inside at_most(k), the window always contains at most k distinct values. The sum (R - L + 1) counts all subarrays ending at R that fit. Two linear passes: O(n) time, O(k) space for the frequency map.

The code

python

from collections import defaultdict

def subarraysWithKDistinct(nums, k):
    def at_most(limit):
        freq = defaultdict(int)
        distinct = 0
        L = count = 0
        for R in range(len(nums)):
            freq[nums[R]] += 1
            if freq[nums[R]] == 1:     # new distinct value entered
                distinct += 1
            while distinct > limit:   # too many distinct: shrink left
                freq[nums[L]] -= 1
                if freq[nums[L]] == 0:
                    distinct -= 1
                L += 1
            count += R - L + 1       # subarrays ending at R, at most limit distinct
        return count

    return at_most(k) - at_most(k - 1)
Narrate it to the interviewer

"Directly enforcing 'exactly k distinct' in a sliding window is tricky because the shrink condition needs two-sided bounds. The clean solution uses the identity: exactly k = at-most k minus at-most (k-1). I write a helper that counts subarrays with at most `limit` distinct values — a standard variable window with a frequency map: expand right, shrink left whenever distinct exceeds the limit, accumulate (R - L + 1) at each position. That counts subarrays of all lengths ending at R that have at most `limit` distinct values. Calling it twice and subtracting gives exactly k. O(n) time, O(k) space."

Dry-run aloud: k = 1 on [1,1,1] (every subarray works, answer is 6 = n*(n+1)/2); k greater than array's total distinct count (answer is 0); single element equal to k = 1 (answer is 1).

05 Recognition

Spotting it, and the traps

The sliding window is the right tool when the problem has all three of these properties: the input is an array or string; you want something about a contiguous subarray or substring; and there is a monotonic relationship between window size and validity (larger window = more likely invalid, or always more valid). If any of those is absent, the window may not apply.

Signal phrases that point here:

  • "Contiguous subarray" or "substring" in the problem statement — almost always a window problem.
  • "Longest" or "shortest" subarray/substring satisfying a condition — variable window, record best.
  • "Every window of size k" or "any window of fixed length" — fixed window, rolling aggregate.
  • "Count subarrays with exactly / at most k" — variable window with sum-counting, possibly the at-most trick.
  • "Permutation", "anagram", "all characters of s1 present" — fixed or variable window with a frequency map.

And the bugs that cost candidates the round:

  • Updating the running state after checking validity. Always update state as soon as R expands; check validity after the update. Otherwise the check sees stale data.
  • Not restoring the state when L shrinks. If you add a[R] to the state on expansion, you must remove a[L] from the state on shrink. Forgetting this makes the state drift and produce wrong answers.
  • Recording the answer at the wrong moment. For "longest valid" problems, record after adding R (before possibly shrinking). For "shortest valid" problems, record inside the shrink loop, after each L advance that keeps the window valid.
  • Using a window when the problem is not contiguous. If the problem asks about subsequences (non-contiguous), a window won't work — you need DP or two pointers on sorted data instead.
  • Forgetting the guard last_seen[ch] >= L in the last-seen-index variant of longest-no-repeat. Without it, a character seen before the window starts incorrectly pushes L backward.
The decision in one line

Contiguous + monotonic validity → sliding window. Non-contiguous, or no monotonic signal → DP or two pointers on sorted data. Complement lookup on unsorted data → hash map. The window is the natural sibling of two pointers (L01) — both walk forward once, both amortize cost across the sweep, but the window carries a state aggregate whereas two pointers navigate a sorted structure.

★ Review

Summary & self-test

The whole pattern compresses to one sentence: maintain a window of array indices, expanding the right edge to grow the window and shrinking the left edge to restore a validity condition, so that each element is processed at most twice and the whole scan costs O(n). The three variants — fixed window, variable window, and window with a frequency map — are the same idea with different state and different validity checks.
