Stacks & Queues

A stack and a queue are the two simplest ways to organize deferred work. One hands it back in the reverse order it arrived (LIFO), the other serves it in the same order (FIFO). Mastering these two — and the deque that unifies them — unlocks balanced-bracket matching, call-stack simulation, BFS frontiers, and a dozen other patterns you will reach for constantly in an interview.

Original study notes. Written from scratch for coding-interview prep. Problem names below are well-known identifiers only; every description, diagram, walkthrough, and code sample is original. 00 The idea

What problem do they solve?

Every program manages a list of things it has not dealt with yet. The question is: which one should it handle first? That single decision divides the universe of linear data structures into two camps.

A stack always handles the thing that arrived most recently — last in, first out (LIFO). Think of a stack of plates: you take from the top. A queue always handles the thing that has been waiting longest — first in, first out (FIFO). Think of a queue at a ticket window: the person who arrived earliest goes first.

Both guarantee constant-time access to the next element to process, with no searching. That O(1) guarantee is why they appear everywhere: the CPU call stack is a stack; HTTP request queues, print spoolers, and BFS frontiers are queues. In interview problems they show up whenever you need to defer, reverse, or sequence work.

The roadmap for this lesson

This is a P1 Core Data Structures concept. The pattern lessons that build directly on it are: Monotonic Stack & Queue (L37) — where the stack holds a constraint on decreasing or increasing values; Tree/Graph BFS & DFS (L20/L23) — where a queue drives BFS and the call stack (or an explicit stack) drives DFS; and Backtracking (L19) — where the call stack naturally unwinds to the previous choice point.

01 LIFO

The stack: last in, first out

A stack exposes exactly three operations. Push places a new item on top. Pop removes and returns the top item. Peek looks at the top item without removing it. All three are O(1): there is no searching, no shifting — the top is always known. The price is that you can only see the top; the rest of the stack is opaque until the layers above are removed.

The key insight for interviews is that a stack reverses the order of work. Whatever you defer last is the first thing you revisit. This is exactly what you need when matching nested structures (open brackets, function calls, XML tags) — the most recently opened item must be the first one closed.

Python idiom

Python's built-in list is the idiomatic stack. stack.append(x) is push; stack.pop() is pop; stack[-1] is peek. Both append and pop() (from the end) run in amortized O(1). Never pop from stack[0] — that is O(n) because it shifts every element.

02 Stack in use

Matching pairs: Valid Parentheses

The single most instructive stack problem in the interview canon is bracket matching (Valid Parentheses, LeetCode 20). The task: given a string of brackets, decide whether every opener is closed by the right closer, in the right order.

The stack is exactly the right tool because bracket nesting is LIFO — the most recently opened bracket must be the first one closed. The algorithm scans left to right. When it sees an opener ((, [, {), it pushes it. When it sees a closer, it peeks at the top of the stack: if the top is the matching opener, pop and continue; if not (or if the stack is empty), the string is invalid. At the end, the stack must be empty — any unclosed openers mean failure.

See it usedEasyLeetCode 20
Valid Parentheses

A string contains only the characters ( ) [ ] { }. Return true if and only if every opener is closed by its correct matching closer in the correct nesting order.

Think it through

Restate

Each opener must have a matching closer, and nesting must be consistent — ({[]}) is valid; ({)} is not, even though counts balance.

Brute force

Repeatedly scan the string removing matching adjacent pairs until no more removals are possible; if the string empties, it was valid. This is O(n²) in the worst case and difficult to code cleanly.

The insight

Every closer must match the most recently seen unclosed opener — a LIFO relationship. That is exactly what a stack models. Each step is O(1) and we do at most n steps: O(n) time, O(n) space (worst case: all openers, e.g. (((().

The move

Maintain a dictionary of closer-to-opener pairs. Scan the string: push openers; on a closer, the stack must be non-empty and the top must be the paired opener — pop it; else return false. After the full scan, the stack must be empty.

The code

python

def isValid(s: str) -> bool:
    match = {')': '(', ']': '[', '}': '{'}  # closer → its expected opener
    stack = []
    for ch in s:
        if ch not in match:       # it's an opener — defer it
            stack.append(ch)
        elif not stack or stack[-1] != match[ch]:
            return False           # empty stack or wrong opener on top
        else:
            stack.pop()            # matched pair — consume it
    return not stack               # valid iff no unclosed openers remain
Narrate it to the interviewer

"The brute-force approach repeatedly removes adjacent matching pairs until the string is empty — O(n²). The stack solution is O(n) because bracket nesting is LIFO: the most recently opened bracket must close first. I push every opener and check every closer against the stack top. My invariant is that the stack always holds exactly the unclosed openers, in the order they appeared. At the end the stack must be empty — any remaining opener is unmatched."

Dry-run aloud: empty string (valid — loop never executes, stack empty), a single opener like ( (invalid — stack non-empty at end), a single closer like ) (invalid — stack is empty when we try to pop), and a wrong-order close like ([)] (invalid — stack top is [ but we see )).

The reusable shape

Any time you need to match the most recently seen opener — HTML/XML tags, function call balancing, undo history — this is the same five lines. The only thing that changes is the mapping from closer to opener and the definition of "opener".

03 FIFO

The queue: first in, first out

A queue reverses the stack's philosophy: the item that has waited longest is served first. It exposes enqueue (add to the back) and dequeue (remove from the front), both O(1) when implemented correctly, plus peek (look at the front without removing).

The crucial implementation detail: Python's list is the wrong choice for a queue. Removing from list[0] shifts every remaining element — O(n). Instead, use collections.deque, which is a doubly-linked deque and runs appendleft / popleft / append / pop all in O(1). For BFS problems — by far the most common queue use in interviews — this matters.

python — queue using collections.deque

from collections import deque

q = deque()
q.append(1)        # enqueue at back:  [1]
q.append(2)        # enqueue:           [1, 2]
q.append(3)        # enqueue:           [1, 2, 3]
front = q.popleft() # dequeue from front: front=1, q=[2, 3]
peek  = q[0]       # peek without removing: peek=2, q=[2, 3]
04 Deque

The deque: both ends in O(1)

A deque (double-ended queue, pronounced "deck") is the structure that subsumes both stack and queue. It supports O(1) insertion and removal at either end. You can use it as a pure LIFO stack (push/pop at the right), a pure FIFO queue (push at right, pop at left), or in the more advanced mode where you need to trim from both ends simultaneously.

The sliding-window maximum problem (LeetCode 239) is the canonical deque interview problem. To maintain the maximum in a window of size k as it slides, you keep a deque that stores indices of candidates in decreasing order of value. When a new element arrives, you pop from the back any index whose value is smaller (it can never be a future maximum). You also pop from the front any index that has slipped outside the window. The front of the deque is always the index of the current maximum. Every element is pushed and popped at most once: O(n) total.

Stack (LIFO)

  • Push/pop from one end only.
  • Reverses order — great for nested structure matching.
  • Simulates the call stack; DFS (explicit) uses a stack.
  • Python: list with append / pop().

Queue (FIFO)

  • Enqueue at back, dequeue from front.
  • Preserves order — BFS frontiers, task queues.
  • Python: collections.deque with append / popleft().
  • Not list.pop(0) — that is O(n).
05 Simulation

Simulating a queue with two stacks

A classic interview question asks you to build a queue out of only two stacks (LeetCode 232). It sounds paradoxical — a stack reverses order, so two stacks should reverse it twice, restoring FIFO order. That intuition is exactly right.

Keep an inbox stack (new items arrive here) and an outbox stack (items leave from here). Enqueue always pushes onto the inbox. Dequeue checks: if the outbox is non-empty, pop from it directly. If the outbox is empty, pour all of the inbox into the outbox by popping each element and pushing it to the outbox — that single reversal restores the original arrival order. Then pop from the outbox.

The amortized cost is O(1) per operation: each element crosses from inbox to outbox exactly once over its lifetime, so n enqueues + n dequeues costs 2n moves total — O(1) amortized per operation, O(n) space.

python — queue from two stacks

class MyQueue:
    def __init__(self):
        self.inbox  = []   # new items land here
        self.outbox = []   # items leave from here

    def push(self, x: int) -> None:
        self.inbox.append(x)   # always enqueue to inbox

    def _refill(self) -> None:
        if not self.outbox:                 # outbox empty — pour inbox
            while self.inbox:
                self.outbox.append(self.inbox.pop())  # reversal

    def pop(self) -> int:
        self._refill()
        return self.outbox.pop()   # oldest item now on top

    def peek(self) -> int:
        self._refill()
        return self.outbox[-1]

    def empty(self) -> bool:
        return not self.inbox and not self.outbox
Amortized argument

Each element is pushed once (to inbox) and moved once (inbox to outbox) — two operations total per element, regardless of how many pop calls are made. The total work over n push + n pop calls is 4n operations (push to inbox, move, pop from outbox, call overhead) — O(1) amortized per call. Individual pop calls that trigger a refill are O(k) for that call, but k is borrowed from future calls that will be O(1).

06 What comes next

Where this shows up next

Stacks and queues are primitives — they are almost never the final answer in an interview, but they are the building block inside every answer to the following families of problems. Once you are comfortable with the operations and costs above, move to these pattern lessons.

L37 · Pattern
Monotonic Stack & Queue A stack maintained in strictly increasing or decreasing order. Enables O(n) solutions for "next greater element", "largest rectangle in histogram", and the sliding-window maximum. The deque structure from Section 04 is central.
L20 / L23 · Pattern
Tree & Graph BFS / DFS BFS uses a queue as its frontier — nodes discovered level-by-level. DFS uses the call stack implicitly (recursion) or an explicit stack iteratively. Level-order traversal, shortest-path in unweighted graphs, and connected components all live here.
L19 · Pattern
Backtracking The recursion tree's call stack is a stack of choices. Understanding that "undo the last choice" is a pop helps you see why backtracking is structurally a depth-first search — and why the two-stack trick above generalises to iterative backtracking.
L14 · Pattern
String Simulation Many in-place string problems (evaluate RPN expressions, decode strings, simplify paths) use a stack to defer and then assemble parts. The bracket-matching shape from Section 02 is the template for all of them.
Signal words to recognize in a problem

Stack: "most recently", "undo the last", "nested", "matching pairs", "evaluate expression", "call stack depth", anything involving reversal of order. Queue: "level by level", "process in the order received", "shortest path (unweighted)", "BFS". Deque: "sliding window max/min", "constrained from both ends". When you spot these phrases — reach for the right container first.

★ Review

Summary & self-test

A stack and a queue are the simplest possible ways to organize deferred work, and both offer O(1) access to the next item. The stack's LIFO discipline makes it the right tool for anything involving reversal, nesting, or matching. The queue's FIFO discipline makes it the right tool for anything requiring order preservation or level-by-level processing. The deque combines both, and the two-stack trick shows that one discipline can simulate the other with amortized constant cost.

01LIFO vs FIFOStack returns the newest item; queue returns the oldest. Both O(1) for the critical access.
02O(1) guaranteePush/pop/peek/enqueue/dequeue are all O(1) — no searching, no shifting — when the right structure is chosen.
03Stack = deferral + reversalPushing defers work; popping revisits it in reverse. Perfect for nested matching and expression evaluation.
04Queue = preserve orderFIFO ensures oldest-first service. The foundation for BFS and any level-by-level traversal.
05Use deque in Pythoncollections.deque gives O(1) at both ends. list.pop(0) is O(n) — never use it as a queue.
06Two stacks = queueInbox receives; outbox serves. Each element crosses once: O(1) amortized per operation.
07Deque for sliding windowsMaintain a monotonic deque of candidate indices to answer range-max queries in O(n) total.
08Signal words"Nested / matching" → stack. "Level by level / BFS" → queue. "Sliding window max/min" → deque.
