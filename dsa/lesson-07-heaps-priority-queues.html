Heaps / Priority Queues

A heap is the data structure that makes selection fast: it always knows the minimum (or maximum) in O(1) and lets you add or remove values in O(log n). Problems asking for the k-th largest, the k closest points, or a running median all collapse to a few heap operations once you see the shape.

00 The idea

What problem does it solve?

Imagine you are processing a stream of scores and someone keeps asking "what is the current top-3?" The obvious answer maintains a sorted list and inserts each new score in the right place. That costs O(n) per insert on a list, or O(log n) with a balanced BST — but a BST brings real implementation overhead.

Here is the insight: you do not need everything sorted. You only need the minimum (or maximum) instantly, and you need every push and pop to keep that property intact cheaply. A binary heap does exactly that: O(1) peek at the min/max, O(log n) push, O(log n) pop — and it is nothing more than an ordinary array with a clever index formula.

The family of problems that screams "heap" shares a common shape: repeatedly find or remove the extreme value from a growing or changing dataset. The phrase "k largest", "k smallest", "k closest", "k most frequent", or "merge k sorted lists" is a recognition cue. A heap keeps the work per element at O(log k) instead of O(k) for a linear scan, turning an O(nk) brute force into O(n log k).

Recognition cue

If the problem asks you to repeatedly extract the minimum or maximum, or to maintain a set of the k best/closest/largest/smallest values dynamically, the answer almost certainly involves a heap. The two pattern lessons that build directly on this concept are Top-K & Two Heaps (L28) and Dijkstra & Prim shortest path (L26), where the heap drives the priority queue behind the greedy choice.

01 Structure

Array layout and the heap invariant ▸

A binary heap is a complete binary tree stored directly in an array — no pointers needed. "Complete" means every level is filled left to right. The root lives at index 0, and the parent/child relationships follow a fixed formula:

  • Parent of node i: index (i - 1) / 2 (integer division)
  • Left child of node i: index 2i + 1
  • Right child of node i: index 2i + 2

The heap invariant for a min-heap is simple: every node's value is less than or equal to its children's values. The root is therefore always the global minimum, readable in O(1). A max-heap flips this: every node is greater than or equal to its children, making the root the global maximum.

Why a complete tree?

The completeness guarantee means the tree's height is always floor(log₂ n). Every sift operation (push or pop) travels at most that many levels — which is why all mutations cost O(log n). A heap with 1,000,000 elements needs at most 20 comparisons per operation.

02 Push

Sift-up: inserting a new value ▸

To push a value onto a min-heap, append it to the end of the array (the next available slot in the tree's bottom level), then restore the invariant by sifting up: repeatedly swap the new node with its parent as long as it is smaller than its parent. Each swap moves the node one level up, so at most log n swaps occur.

The critical observation is that a single swap only involves the swapped node and its parent — the rest of the tree is untouched and still satisfies the invariant. You only need to fix the path from the new leaf to the root.

The figure below starts with the min-heap [2, 5, 3, 9, 7, 8, 4] and inserts the value 1. Watch it bubble upward until it reaches the root.

The reusable shape

Sift-up is an upward chain of comparisons, each touching exactly one parent. The work is proportional to the tree height — O(log n) in the worst case (a new minimum bubbles all the way to the root), O(1) in the best case (the new value is already larger than its parent).

03 Pop

Sift-down: removing the minimum ▸

Popping from a min-heap always removes the root — the global minimum. The trick is replacing the root with the last element in the array (the rightmost leaf), then shrinking the array by one. This preserves the complete-tree shape. But the transplanted value is almost certainly larger than its new children, so you sift it down: repeatedly swap it with the smaller of its two children as long as it is larger than either child. At most log n swaps again.

The choice of "smaller child" matters: swapping with the larger child would still leave the invariant violated on the other side. By always promoting the smaller child, you ensure the parent is less than both descendants after each step.

Starting from the same heap, the figure shows a pop: the minimum (root) is extracted, the last element moves to the root position, and sift-down restores order.

Max-heap in Python

Python's heapq module is a min-heap only. To simulate a max-heap, push and pop the negatives of your values: push -x and negate again on pop. For objects, push tuples: (-priority, item). This is a standard interview trick — mention it and your interviewer will nod.

04 Costs

Operation costs at a glance

The table below compares heap operations against the obvious alternatives. The heap's advantage is clear whenever you need repeated minimum extraction: it dominates a sorted array on insertions and dominates an unsorted array on deletions.

Operation Binary heap Sorted array Unsorted array
peek min/max O(1) O(1) O(n)
push O(log n) O(n) O(1) amortized
pop min/max O(log n) O(1) or O(n) shift O(n)
build from n items O(n) heapify O(n log n) sort O(n)
find arbitrary element O(n) O(log n) O(n)
space O(n) O(n) O(n)

Min-heap

  • Root = global minimum, always.
  • Pop gives the next smallest each time.
  • Use when you need the k smallest from a stream, or Dijkstra's greedy pick.
  • Python's heapq is min-heap natively.

Max-heap

  • Root = global maximum, always.
  • Pop gives the next largest each time.
  • Use when you need the k largest — or keep a min-heap of size k so the root is the k-th largest (the boundary value).
  • In Python: push/pop negated values.

The O(n) heapify row deserves mention. Calling heapq.heapify(lst) on an existing list converts it into a valid heap in linear time — faster than pushing each element one by one (which would be O(n log n)). The reason is that sift-down applied bottom-up needs progressively less work per node: the bulk of nodes are leaves and require zero comparisons. The total work sums to O(n).

05 In practice

Heaps in Python and the k-th problem

Python's standard library provides heapq — a min-heap operating on a plain list. The four operations you need for interviews are heapify, heappush, heappop, and nsmallest/nlargest. Here is the idiom for the canonical "k-th largest" problem (LeetCode 215) — a Medium problem that appears in Google, Meta, and Amazon screens in various disguises.

See it usedMediumLeetCode 215
Kth Largest Element in an Array

Given an unsorted array of integers and a positive integer k, find the k-th largest value (not the k-th distinct value) without fully sorting the array.

Think it through

Restate

I want the element that would sit at position k from the right end in a sorted version of the array. If k = 1 that is the maximum; if k = n that is the minimum.

Brute force

Sort the array and index from the end: sorted(nums)[-k]. Correct, but O(n log n) and O(n) extra space for the sort.

The waste

A full sort orders every element, but I only need the boundary between "top k" and "everything else". A min-heap of exactly k elements tracks that boundary: the root is the smallest of the k largest values seen so far, which is precisely the k-th largest overall.

The move

Maintain a min-heap of size k. For each number: push it onto the heap; if the heap grows beyond k elements, pop the minimum. After processing all n numbers, the heap holds the k largest values and the root is the k-th largest. Total: n pushes, each costing O(log k).

Invariant & cost

At every point the heap holds the k largest values seen so far. Its root is their minimum — the k-th largest among those values. When the heap is full and you push a larger value, the minimum is ejected: the invariant is maintained. O(n log k) time, O(k) space. For k << n this is a clear win over sorting.

The code

python

import heapq

def findKthLargest(nums, k):
    # min-heap of the k largest values seen so far
    heap = []
    for x in nums:
        heapq.heappush(heap, x)          # push current value
        if len(heap) > k:
            heapq.heappop(heap)          # evict the current smallest
    # heap[0] is the smallest of the k largest = k-th largest overall
    return heap[0]
Narrate it to the interviewer

"I want the k-th largest element. The brute force is to sort, which is O(n log n). Instead I will keep a min-heap of size k — just the k largest values I have seen so far. Each time I push a new value, if the heap exceeds k I pop the minimum; that minimum was smaller than the k-th largest so it is safe to discard. After the full scan, the root of the heap is the smallest of those k values, which is exactly the k-th largest in the whole array. That is O(n log k) time and O(k) space."

Dry-run aloud: k = 1 (should return the maximum; heap pops everything but the largest); k = len(nums) (should return the minimum; nothing is ever popped); array with duplicate values (the heap holds duplicates correctly since it is a value-based comparison, not identity).

The heapify shortcut

When the whole input is available up front, you can heapify first and then pop k times, each pop costing O(log n). Total: O(n + k log n). For small k this matches the O(n log k) approach; for large k (say k close to n) the two are comparable. In Python:

python

import heapq

def findKthLargest_v2(nums, k):
    # heapify in O(n), then pop k-1 times to reach the k-th largest
    max_h = [-x for x in nums]   # negate for max-heap simulation
    heapq.heapify(max_h)             # O(n) — builds heap in place
    for _ in range(k - 1):
        heapq.heappop(max_h)         # discard the k-1 largest
    return -max_h[0]               # negate back to get the real value
Which version to offer first?

In a streaming scenario (values arrive one at a time), the size-k min-heap is the only option. When the full array is given, either works — the heapify + k pops variant is often cleaner to explain, but the size-k approach generalises better. Mentioning both trade-offs shows depth.

06 Next steps

Where this shows up next

The heap is a building block. Once you understand the invariant and the O(log n) costs, these pattern lessons compose it into complete algorithms:

L26Dijkstra & Prim (Greedy Shortest Path)

Both algorithms use a min-heap as a priority queue to greedily extract the next-cheapest node or edge. The heap replaces the O(n) linear scan for the minimum-distance vertex, driving the algorithm from O(V²) to O((V + E) log V). Understanding heap push/pop is a prerequisite for reading Dijkstra correctly.

L28Top-K & Two Heaps

This is the direct application layer for everything in this concept lesson. Top-K patterns (k largest, k closest, k most frequent) all reduce to maintaining a fixed-size heap. Two-heap problems — like the running median — keep a max-heap for the lower half and a min-heap for the upper half, rebalancing after each insertion to answer the median in O(1). This lesson will feel natural if you are solid on sift-up and sift-down.

L29Merge K Sorted Lists / External Sort

Merging k sorted sequences is a min-heap drill: push the first element of each list onto a heap of size k, then repeatedly pop the minimum and push the next element from the same list. Each of the n total elements is pushed and popped once — O(n log k) total. The same pattern underlies external merge sort when the dataset does not fit in memory.

Study tip

Before moving to those pattern lessons, make sure you can implement a min-heap from scratch: the array representation, the parent/child formulas, sift-up, sift-down, and O(n) heapify. Understanding the mechanism — not just the API — is what lets you adapt when an interviewer asks for a custom comparator or a heap of pairs.

★ Review

Summary & self-test

A binary heap is a complete binary tree stored in a flat array, maintaining the invariant that every node is smaller than (min-heap) or larger than (max-heap) its children. This single rule gives O(1) peek and O(log n) push/pop by limiting all repair work to a path from one node to the root or a leaf. The result is the most efficient general-purpose priority queue for selection problems.

01Array = treeParent at (i-1)/2; children at 2i+1 and 2i+2. No pointers needed.
02Heap invariantEvery node ≤ its children (min-heap). Root is always the global minimum.
03O(1) peekThe min/max is always at index 0 — a direct array access.
04O(log n) pushAppend, then sift-up: swap with parent while smaller. At most log n swaps.
05O(log n) popReplace root with last element, shrink, then sift-down: swap with smaller child. At most log n swaps.
06O(n) heapifySift-down from the middle outward. Faster than n individual pushes.
