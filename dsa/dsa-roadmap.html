Coding Interview DSA: A Pattern-First Path

Organised by the ~20 patterns that actually recur — not textbook-chapter by chapter. Fifteen lessons are ★ Core: master those and you can solve the bulk of LeetCode medium. The rest add depth (DP, advanced graphs) and a long tail you can skip unless you're targeting the hardest loops.

Scope: calibrated for LeetCode medium-to-hard, the band product / super-app / fintech companies in Singapore interview at. On problem names: tags like ex: Two Sum point at well-known public problems for anchoring — they are references, not reproductions. Any generated lesson is original study material. Note: DSA is far more uniform across companies than system design, so company tags here are soft hints about emphasis (e.g. Google/TikTok lean DP-heavy), not about which topics appear.
How to use this path

Do L00–L01 first — the problem-solving framework and complexity analysis are meta-skills every problem reuses. With two years in, treat Phase 1 data structures as a quick refresher. If time-boxed, run the ★ Core 15, then drill company-tagged problems.

Learn the invariant behind each pattern, not the memorised solution. Interviewers change the constraints on purpose; pattern recognition and a clean derivation are what they score. The DP phase (P7) is the highest-variance topic — budget two weeks.

To build a lesson, tell the AI the number — e.g. generate L19 — Backtracking. Each card shows the exact phrase.

The Coding Mock Protocol

The interviewer scores how you get to the answer, not just the answer. Run each ★ Core pattern as a timed ~40-minute mock against this budget and rubric — pattern recognition plus a clean, narrated solve is the whole game.

~40-minute time budget (UMPIRE)
0–3 min · Understand
ClarifyRestate, ask input ranges, edge cases, output format; write 1–2 examples.
3–8 min · Match + Plan
ApproachName the pattern, state a brute force + its O(·), then the optimised idea. Get buy-in before coding.
8–28 min · Implement
CodeClean names, narrate as you go, no premature optimisation.
28–35 min · Review
TestDry-run an example; check empty / single / duplicates / overflow; fix bugs yourself.
35–40 min · Evaluate
WrapState final time/space and discuss follow-ups (scale, variants).
Self-grade rubric (1–5 each, after every mock)
  • Clarified first — pinned constraints and edge cases before writing any code.
  • Brute force → optimised — stated the naive solution and its complexity, then improved it.
  • Communicated the plan — got interviewer buy-in on the approach before implementing.
  • Clean, correct code — compiles in your head, sensible names, no hand-waving.
  • Tested & debugged — traced examples and edge cases and caught your own bugs.
  • Complexity & follow-ups — stated final O(time)/O(space) and handled the "what if" probe.

Below 3 on any line → that's the next rep. Re-attempt the pattern until every line clears 4.

Practice plan — pattern over volume

Don't grind 500 random problems. Master one pattern, then drill ~6–10 problems on it before moving on. A sensible ramp: a curated starter set (Blind 75) → a fuller pattern set (NeetCode 150) → your targets' tagged questions via LeetCode's company filter (Grab, Sea/Shopee, TikTok, Agoda, Stripe…). Roughly two timed problems a day; re-do any you missed after 3–7 days (spaced repetition). One nuance: Stripe-style rounds lean practical and implementation-heavy (build a small working thing) over pure algorithm tricks — practise clean, tested code, not just the optimal Big-O.

P0 Phase Zero

The Coding Interview Operating System

The two meta-skills under every problem: a repeatable method, and the ability to reason about cost.

L00
The Problem-Solving Framework (UMPIRE)

Run every problem the same way: Understand (restate, constraints, edge cases) → Match a pattern → Plan a brute force then optimise → Implement → Review/test → Evaluate. State the brute force out loud before optimising — interviewers want to see the climb, not a memorised jump.

★ CoreFoundationalask ▸ "L00"
L01
Complexity Analysis (Big-O)

Time and space in Big-O, amortised analysis, and the cost of common operations. Justify why a solution is O(n log n), name the bottleneck, and read the target complexity off the input size (n≤20 → exponential is fine; n≤10⁵ → you need n log n).
