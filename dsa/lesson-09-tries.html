09Pattern · Coding Interview

Tries (Prefix Trees)

A trie is a character-keyed tree where every path from root to a marked node spells a word. Instead of scanning N stored strings on each query, you walk at most L characters — the length of the query itself — making prefix search and autocomplete genuinely O(L) regardless of dictionary size.

Original study notes. Written for coding-interview prep covering the pattern of prefix-keyed tree structures. Problem names below are referenced only as well-known identifiers — Implement Trie (208), Word Search II (212), Replace Words (648). 00 The idea

What problem does it solve?

Imagine you have a dictionary of 100,000 English words and you want to know whether any of them starts with the prefix "pre". The first approach everyone reaches for is to loop over every word and call word.startswith("pre"). That is O(N × L) per query where N is the dictionary size and L is the average word length. For N = 100,000 and L = 8, a single query already touches 800,000 characters. An autocomplete box that fires on every keystroke would be sluggish even on a fast machine.

The redundant work is obvious in hindsight: every word that starts with "pre" shares those same three characters. The brute-force scan re-examines that shared prefix for every single word, over and over. A trie compresses all that shared structure so a single path of three hops covers the entire group. Once you follow r → e → e → node, you are standing at the root of every word with that prefix. The query cost becomes O(L) — dependent only on the query length, not on the number of stored words.

  • Prefix search / autocomplete — walk the query path; if it exists, every descendant is a valid completion.
  • Dictionary word lookup — same walk, but also check the isEnd flag at the last node.
  • Word replacement by shortest root — find the shallowest isEnd ancestor and stop early.
  • Board word-search — prune DFS branches the moment the current character sequence doesn't exist in the trie.
Why this pattern matters for interviews

Tries appear at every company that builds search, autocomplete, or text-processing — and interviewers use them to test whether you can design a custom data structure from a node + map, not just string-match your way to a solution. Knowing the node anatomy and the O(L) cost argument cold means you can deliver the answer before the interviewer finishes the problem description.

01 Brute force

Brute force: store all words and scan

Before building a trie, it is worth seeing precisely where the brute-force approach breaks down. Suppose we have a list of N words and we want to answer two kinds of query: "does this exact word exist?" and "does any word start with this prefix?"

The simplest structure is a plain Python list (or set for exact lookup). For exact-word search, a set gives O(L) average via hashing — already fast. The problem arrives with prefix queries: a set cannot answer "does any word start with pre?" without scanning every element, costing O(N × L). With N = 50,000 words and L = 8, each keypress in an autocomplete box runs 400,000 character comparisons. At ten keystrokes a second that is four million comparisons per second just for prefix checking — and the problem gets worse as the dictionary grows.

A sorted list plus binary search brings prefix queries to O(L log N), which is better but still grows with N. A trie removes that dependence entirely: once a word is in the structure, the prefix path is already there, shared with every other word that starts the same way. A query walks at most L nodes and stops, cost O(L) regardless of N.

Hash set / sorted list

  • Exact lookup: O(L) with hashing, O(L log N) with binary search.
  • Prefix query: O(N × L) with a set; O(L log N) with sorted list + scan.
  • Autocomplete: must collect all matches by linear scan — O(N × L).
  • Memory: O(N × L) — every word stored independently, no sharing.

Trie (prefix tree)

  • Exact lookup: O(L) — walk the path, check isEnd.
  • Prefix query: O(L) — same walk; if path exists, prefix is present.
  • Autocomplete: O(L + M) where M is the number of completions returned.
  • Memory: O(total unique characters) — shared prefixes stored once.
Worked problemEasyLeetCode 208
Implement Trie

Design a data structure that supports inserting a word, searching for an exact word, and checking whether any inserted word starts with a given prefix. All three operations must run in O(L), where L is the length of the input string.

Think it through

Restate

Build a prefix-aware dictionary with three methods: insert(word), search(word) that returns true only if the full word was inserted, and startsWith(prefix) that returns true if any inserted word has that prefix.

Brute force

Store words in a Python set. insert and search are O(L) with hashing, but startsWith requires looping over every stored word and calling w.startswith(prefix) — O(N × L) per prefix query. Unacceptable for large dictionaries.

The waste

The hash set stores every word independently, even if thousands of them share a prefix. All that shared character data is re-examined on every prefix query. The trie fuses those shared paths so the prefix is walked once.

The move

Each trie node holds a dictionary mapping a character to its child node, plus a boolean is_end flag. To insert, walk the children dictionary one character at a time, creating missing nodes along the way, and flip is_end at the last node. To search, walk the same path; return true only if every character maps to a child AND the final node has is_end = True. For startsWith, do the same walk but don't check is_end — reaching the end of the prefix without falling off is enough.

Invariant & cost

Every path from root to a node with is_end = True spells an inserted word. The walk for any operation touches exactly L nodes, one per character: O(L) time per operation, O(total characters across all inserted words) space.

The code

python

class TrieNode:
    def __init__(self):
        self.children = {}      # char -> TrieNode
        self.is_end = False    # True if a word ends here

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word: str) -> None:
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()  # create missing node
            node = node.children[ch]
        node.is_end = True                        # mark word boundary

    def _walk(self, s: str):
        # shared helper: returns the node after walking s, or None
        node = self.root
        for ch in s:
            if ch not in node.children:
                return None
            node = node.children[ch]
        return node

    def search(self, word: str) -> bool:
        node = self._walk(word)
        return node is not None and node.is_end  # must reach end AND hit a word

    def startsWith(self, prefix: str) -> bool:
        return self._walk(prefix) is not None  # reaching the node is enough
Narrate it to the interviewer

"Restating: I need insert, exact search, and prefix check, all in O(L). The brute-force approach is a set — search is fine but startsWith becomes O(N × L), not acceptable. The waste is that shared prefixes are re-examined for every word. So I use a trie: each node holds a children map and an isEnd flag. Inserting a word walks the map one character at a time, creating missing nodes, then marks isEnd. Searching walks the same path and checks isEnd at the end. startsWith is the same walk but without the isEnd check — arriving at the node is proof the prefix exists. My invariant is that every root-to-isEnd path spells an inserted word. Time is O(L) per operation; space is O(total characters, shared)."

Dry-run aloud: searching for a word that was never inserted — the walk falls off at the missing child and returns false. Searching for a prefix of an inserted word — the walk succeeds but isEnd is false, so search correctly returns false while startsWith returns true. Empty string insert and search: isEnd flips on the root, which is valid.

02 Node structure ▸

The trie node: a children map and an isEnd flag

Every trie node is exactly two fields: a children map (typically a dictionary mapping a character to another node) and a boolean flag isEnd that marks whether a complete word ends at this node. That is the whole anatomy. There are no values stored in the nodes themselves — the word is reconstructed by reading the characters along the path from root to the flagged node.

The root node is always empty: it represents the state "no characters consumed yet." Every edge label is a single character. Following an edge from node p to node q means "the next character is the label on that edge." When you insert the word "cap", you walk (or create) three edges: c, a, p, and flip isEnd at the third node. If you later insert "cat", the first two nodes already exist — only the t branch is new.

The interactive figure below shows the trie growing as the words "do", "dot", and "dog" are inserted one by one. Watch how inserting "dot" reuses the existing "d" and "o" nodes — only the "t" leaf is new. Nodes with a filled ring carry isEnd = true.

The reusable shape

The insert loop is always: node = root; for ch in word: if ch not in node.children: node.children[ch] = TrieNode(); node = node.children[ch]; node.is_end = True. The search loop is the same walk but returns None on a missing child. Once you know this two-function skeleton, every trie problem is a variation on what you do when you reach a node or when you fall off the tree.

03 Search in O(L) ▸

Walking the path: search and startsWith

Searching in a trie is a character-by-character descent from the root. At each node you ask: "does my children map contain the next character of the query?" If yes, step into that child. If no, the word (or prefix) is not in the trie — return false immediately without examining any other part of the structure. This early-exit property is what makes trie search O(L) rather than O(N × L): the tree's branching structure prunes all irrelevant paths before you even visit them.

The distinction between search (exact word) and startsWith (prefix) is a single extra check at the end of the walk: for an exact search you also verify that the node you land on has isEnd = True. A word like "do" is inserted with isEnd set on the "o" node; searching for "dot" in that trie lands on the "t" node which does not exist yet — false. But startsWith("d") just checks that the "d" node exists, which it does — true, regardless of what words end in that subtree.

The figure below shows a search for the word "dot" and the prefix "do" on the trie built in FIG. 1. Each step highlights the node being visited and shows the current character being matched.

What the interviewer asks next

"What if the alphabet is large — say Unicode?" Use a hash map for children (already shown). For a 26-letter English alphabet you can also use a fixed array of 26 slots, indexed by ord(ch) - ord('a'), which is faster in practice. "Can you delete a word?" Walk to the end, flip isEnd off. Clean up dangling nodes by checking if a node becomes childless and is no longer an end — easiest with a post-order recursive pass. "What about wildcard search like '.' matching any letter?" At each '.' node, recurse into all children — the same DFS you'd use for general tree traversal.

04 Application — Replace Words

Replace Words: shortest root in O(L)

The Replace Words problem (648) provides a list of root words and a sentence. For each word in the sentence, if any root is a prefix of it, you replace the word with the shortest such root. For example, if the dictionary contains "cat" and "cattle" and the sentence has the word "cattle", the replacement is "cat" (the shorter root).

The brute-force fix scans every word in the sentence against every root in the dictionary: O(sentence_words × roots × L). The trie makes this clean. Insert all roots into a trie. For each sentence word, walk the trie one character at a time. The moment you land on a node where isEnd = True, you have found the shallowest (shortest) matching root — return it and stop immediately. You never need to look further down the trie. If you exhaust the query without hitting an isEnd, no root matches and the original word is kept. Each word replacement costs O(L) for the trie walk.

Worked problemMediumLeetCode 648
Replace Words

Given a list of root words and a sentence string, replace every word in the sentence with the shortest root from the list that is a prefix of it. If no root matches, keep the original word. Return the modified sentence.

Think it through

Restate

For each token in the sentence, find the shortest root that the token starts with, and substitute it. If none, keep the token. Rejoin the tokens into a sentence string.

Brute force

For each sentence word, loop over every root and check if the word starts with that root, keeping track of the shortest match. With W sentence words and R roots of average length L, that is O(W × R × L). For W = 1000, R = 500, L = 8 that is four million character comparisons — slow for large inputs.

The waste

The brute-force checks each root independently, even when many roots share a prefix. Building a trie over the roots means a single walk per sentence word, stopping the moment an isEnd is hit — that is exactly the shortest root because trie depth corresponds to word length.

The move

Insert every root into a trie. For each word in the sentence, walk the trie one character at a time. As soon as a node has isEnd = True, the characters walked so far spell the shortest root — return that prefix. If the walk falls off the tree (missing child) before hitting any isEnd, no root matches the word — keep it unchanged.

Invariant & cost

Because trie depth equals string length, the first isEnd encountered is always the shortest match. Building the trie: O(total root characters). Replacing each word: O(word_length). Overall: O(total root chars + total sentence chars) time and space — linear in the input size.

The code

python

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

def replaceWords(dictionary, sentence):
    # build trie over the root dictionary
    root = TrieNode()
    for word in dictionary:
        node = root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True

    def shortest_root(token):
        # walk trie; stop at first isEnd (shallowest root)
        node = root
        for i, ch in enumerate(token):
            if ch not in node.children:
                break              # no root matches
            node = node.children[ch]
            if node.is_end:
                return token[:i + 1]  # shortest matching root
        return token           # no match found — keep original

    return " ".join(shortest_root(t) for t in sentence.split())
Narrate it to the interviewer

"Restating: for each sentence word, find the shortest dictionary root that is a prefix, and swap it in. The brute-force is O(W × R × L) — checking each root separately for each sentence word. The waste is that roots often share prefixes, so many checks are redundant. My fix: insert all roots into a trie, then for each sentence token walk the trie character by character. The moment I hit a node with isEnd set, that depth in the trie equals the length of the shortest root — I return that prefix immediately. If I fall off the tree first, no root matches. The invariant is that the trie's isEnd nodes appear at the exact depth of each root word, so the first one encountered is always the shortest. Building the trie is O(total root characters); processing each sentence word is O(word length). Total is linear in the input."

Dry-run aloud: a sentence word that is itself a root — it should be returned unchanged (isEnd found on the final character). A sentence word with no matching root — the walk falls off at some missing child and the original word is returned. An empty dictionary — no isEnd is ever set, every word kept unchanged.

05 Application — Word Search II ▸

Board DFS pruned by a trie

Word Search II (212) gives you a 2D grid of letters and a list of target words. You must find all words that can be formed by tracing a path of adjacent cells — no cell reused in a single word. A standard DFS from every cell explores every path from every starting position, with the per-cell fanout multiplied across the full grid. That is expensive: in the worst case, O(rows × cols × 4^max_word_length).

The trie cuts a huge chunk of that search space. Rather than trying to match each target word independently (which would multiply the DFS cost by the number of words), you build one trie over all target words and then run a single DFS across the board, carrying a pointer into the trie. At each cell, you check whether the current board character exists as a child of the current trie node. If it does not, you prune immediately — no path starting with this prefix can ever spell a target word. If it does, step the trie pointer down and continue. When you reach a node with isEnd = True, you have found one of the targets.

The interactive below shows the DFS visiting cells and the trie pointer advancing in lock-step, with pruning highlighted when a character is not found in the current node's children.

The reusable shape

The pattern is: build a trie from the word list; DFS from each grid cell; at each step move to an adjacent unvisited cell only if its character exists in node.children; mark visited cells with a sentinel (e.g. '#') while recursing and restore on backtrack; collect words when node.is_end is true; set node.is_end = False after collecting to avoid duplicates. This DFS-plus-trie combo is a standard Hard pattern once you have the trie basics down cold.

Worked problemHardLeetCode 212
Word Search II

Given a grid of single characters and a list of words, return all words from the list that can be traced by moving to adjacent cells (up, down, left, right) on the grid, without revisiting any cell within a single word's path.

Think it through

Restate

Return every target word that can be spelled along a connected, non-self-intersecting path of grid cells. A word is found if a contiguous sequence of adjacent cells, each letter matching in order, exists on the board.

Brute force

For each word in the list, run a standalone DFS from every grid cell looking for that specific word. Cost: O(words × cells × 4^L) where L is the max word length. With 30 words on a 12×12 grid, this is already millions of nodes visited per second — and many paths are explored identically across different target words. O(W × R × C × 4^L) overall.

The waste

Each per-word DFS explores the same board paths that other words also explore. If ten words all start with "ca", the path starting with c-a is explored ten times independently. The trie merges those common prefixes so one DFS step covers all ten at once.

The move

Build a trie from all target words. DFS from every grid cell; carry the current trie node pointer. At each adjacent cell, check if its letter is a child of the current node — if not, prune immediately. If yes, step into that child and recurse. When the node's is_end is true, record the found word (and clear is_end to avoid duplicates). Mark each visited cell with a sentinel during recursion; restore on backtrack.

Invariant & cost

The trie ensures every DFS step only continues along valid word-prefix paths. Any branch diverging from all target words is pruned at the first character. Space: O(total characters in word list). Time: O(R × C × 4^L) in the worst case, but in practice the trie pruning makes this far faster than the brute force for any real word list.

The code

python

class TrieNode:
    def __init__(self):
        self.children = {}
        self.word = None          # stores the full word at an end node

def findWords(board, words):
    root = TrieNode()
    for w in words:              # build trie over all target words
        node = root
        for ch in w:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.word = w

    rows, cols = len(board), len(board[0])
    found = []

    def dfs(r, c, node):
        ch = board[r][c]
        nxt = node.children.get(ch)
        if nxt is None: return     # prune: char not in trie
        if nxt.word:                  # found a complete word
            found.append(nxt.word)
            nxt.word = None           # prevent duplicate collection
        board[r][c] = '#'            # mark visited
        for dr, dc in [(-1,0),(1,0),(0,-1),(0,1)]:
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols and board[nr][nc] != '#':
                dfs(nr, nc, nxt)
        board[r][c] = ch              # restore cell on backtrack

    for r in range(rows):
        for c in range(cols):
            dfs(r, c, root)
    return found
Narrate it to the interviewer

"Restating: find all words from the list that can be traced along adjacent board cells without repeating a cell. The brute force runs a per-word DFS from every cell — O(W × cells × 4^L) — and repeats huge chunks of work for words that share prefixes. The trie merges those common prefixes. I build a trie over all target words, storing the full word at each end node. Then I run one DFS from each cell, carrying the current trie pointer. At each adjacent cell, I try moving the pointer one level down; if the character is missing in children, I prune immediately — no word with this path prefix exists. When I hit a node with a stored word, I collect it and clear the field to prevent duplicates. I mark each cell visited during recursion and restore it on backtrack. Space is O(total word characters). Time is O(cells × 4^L) but with heavy pruning from the trie in practice."

Dry-run aloud: a word list with a single character — the DFS starts on every cell with that character and succeeds immediately. A word that appears twice in the list — deduplicated because we clear node.word after the first collection. An empty board or empty word list — the outer loops simply don't fire.

★ Review

Summary & self-test

The trie pattern compresses to a single sentence: when a set of strings shares prefixes, store them in a character-keyed tree so any path query costs O(L) — the length of the query — instead of O(N × L) where N is the number of stored strings. Master the node anatomy, the insert and search loops, and the three application shapes — shortest root, prefix aggregation, board DFS pruning — and you can handle every trie problem in the practice ladder from first principles.
