Backtracking

When a problem asks for every valid arrangement, subset, or path through a constrained space, backtracking is the pattern: build a candidate incrementally, one choice at a time, and the moment the partial candidate violates the rules, abandon it and try the next option. It is systematic, exhaustive search — but with pruning that skips entire subtrees of dead branches before they grow.

00 The idea

What problem does it solve?

Imagine you are packing a bag from a shelf of items and you want to list every possible bag you could fill. There are 2n possible subsets of n items. You cannot avoid that cost — the output alone is exponential. But generating them carelessly, by trying every number from 0 to 2n−1 and using binary digits to decide which items to include, is hard to extend the moment the rules become richer (e.g., "the bag can hold at most weight W").

Backtracking reframes the search as a decision tree. At each level you pick one choice from the available options. After making a choice you recurse deeper. If at any point the partial candidate cannot possibly lead to a valid answer, you prune — skip that branch entirely and undo the choice. If you reach a leaf with a complete candidate, record it. The mental model is a tree where each node is a partial solution and each edge is one choice; backtracking explores this tree depth-first, cutting branches early.

The universal skeleton looks like this:

  • Choose — select the next element to add to the partial candidate.
  • Explore — recurse into the sub-problem with that choice made.
  • Un-choose — undo the choice so the next option can be tried cleanly.
The reusable shape

Every backtracking solution is the same three-line skeleton: choose, explore, un-choose. What changes across problems is (a) what the choices are, (b) when to record a result, and (c) what counts as a dead branch worth pruning. Internalize the skeleton and you can apply it to subsets, permutations, combinations, grid paths, and constraint puzzles like N-Queens.

01 Subsets

Include or exclude each element

The subset problem asks for every distinct sub-collection of an array. The brute-force approach iterates from 0 to 2n−1 and uses bitmasking to decide membership. That works in O(n · 2n) but is awkward to read, hard to prune, and falls apart the moment a constraint is added.

The backtracking formulation is cleaner. Think of each element as a binary decision: do I include it in the current subset? At index i, the recursive call has two branches — include nums[i] and recurse to i+1, or skip it and recurse to i+1. Every call to the function records the current partial list as a valid subset (including the empty list at the root). When i reaches the end of the array, the recursion bottoms out. No un-choosing is needed between the include and the skip branches because the same index is simply not revisited: the immutable index advance acts as the implicit undo.

Below, step through the decision tree for the array [1, 2, 3]. The blue highlight shows which element is being decided; the readout accumulates the current partial subset.

The key insight

Because every partial list at every depth is itself a valid subset, we record the current path before making any choice — at the very top of the recursive call. This means all 2n subsets (including the empty set) are captured without any special leaf check.

Worked problemMediumLeetCode 78
Subsets

Given an integer array with no repeated values, return every possible subset (including the empty set). The order of subsets in the output does not matter.

Think it through

Restate

I need all 2n subsets of a distinct-element array. The output size is inherently exponential, so I cannot beat O(n · 2n) in total work — I just need to generate each subset exactly once.

Brute force

Iterate over all bitmasks 0 to 2n−1; for each mask, collect the elements whose bit is set. Correct, at O(n · 2n), but the bitmask approach doesn't generalize to problems with constraints or variable-length choices.

The waste

The bitmask approach doesn't express the decision structure, so there is nowhere to hang a pruning condition when constraints appear. The backtracking tree is the same asymptotic cost but extends naturally.

The move

At each index start, record the current path, then branch: include nums[start] and recurse to start+1, then remove nums[start] and recurse to start+1 again (skip). The two branches correspond to include and exclude. Recursion terminates when start == len(nums).

Invariant & cost

Each of the 2n nodes in the decision tree is visited exactly once, and copying the current path at each node costs O(n) in the worst case. Total: O(n · 2n) time, O(n) stack space (ignoring output).

The code

python

def subsets(nums):
    result = []

    def bt(start, path):
        result.append(list(path))    # every partial list is a valid subset
        for i in range(start, len(nums)):
            path.append(nums[i])     # choose: include nums[i]
            bt(i + 1, path)          # explore: recurse past i
            path.pop()               # un-choose: undo for next branch

    bt(0, [])
    return result
Narrate it to the interviewer

"I need every subset of a distinct-element array. The output alone is 2n entries, so I can't do better than O(n · 2n). The clean way to generate them is a backtracking decision tree: at each call I record the current path as a subset, then loop from start to the end — for each index I append the element (choose), recurse past it (explore), then pop it (un-choose). Starting from index start instead of 0 ensures each element is considered only going forward, so no subset is duplicated. The invariant is that after each pop the path is exactly what it was before, so the next branch starts clean. O(n · 2n) time, O(n) stack."

Dry-run aloud: empty input (returns [[]]), single element (returns [[], [x]]), and check the un-choose restores the path correctly at every level.

What the interviewer asks next

"What if the array has duplicates?" (Subsets II, LeetCode 90) — sort the array first, then inside the loop skip nums[i] == nums[i-1] when i > start. This skips sibling branches that would produce the same subset. "What about power set size?" — confirm it is exactly 2n and explain why: each element has two choices, independently. "Can you do it iteratively?" — yes, start with [[]], and for each element extend every existing subset with it; that is the bottom-up equivalent.

02 Permutations

Tracking which elements have been used

A permutation uses every element exactly once, in some order. The brute force is to generate all n! orderings of the indices — already that is the size of the output. The backtracking tree has n branches at the first level (any element can go first), n−1 branches at the second (any unused element), and so on. The tree has n! leaves and n · n! nodes total.

The mechanism requires knowing which elements are already in the current partial permutation. The classic tool is a boolean used[] array of length n: used[i] = True means element i is already committed somewhere above in the path. On each recursive call, loop over all indices; skip those already used; mark, recurse, unmark.

An alternative is in-place swapping: move the chosen element to the front by swapping it with the current position, recurse, then swap back. This avoids the extra array and is slightly faster, but it produces permutations in a different order.

used[] boolean array

  • Clear and easy to debug: print used at any step.
  • O(n) extra space for the boolean array.
  • Works naturally when elements have indices (distinct values).

In-place swap

  • O(1) extra space; no auxiliary array needed.
  • Harder to reason about for duplicates or ordered output.
  • Each swap must be undone: swap(a[i], a[pos]) then recurse, then swap again.
Worked problemMediumLeetCode 46
Permutations

Given an array of distinct integers, return every possible ordering of its elements. Each element must appear exactly once in each permutation.

Think it through

Restate

Return all n! orderings of a distinct-element array. Output size is n!, so no algorithm can do better than O(n · n!) total work — I just need to generate each ordering exactly once without duplicates.

Brute force

Enumerate all n! permutations with a nested loop for each position — that is itertools.permutations under the hood. It works, but it doesn't generalize to "permutations with constraints" like "element A must precede B."

The waste

Nested loops know nothing about which elements are already used — they need the deduplication to happen at the end. The backtracking tree knows the used set at every node, so it never builds a path that reuses an element.

The move

Maintain a used[] boolean array. At each level, loop over all indices; if used[i] is False, append nums[i] to the path and set used[i] = True, recurse, then pop and set used[i] = False. When the path length equals n, record it.

Invariant & cost

used[i] is True if and only if nums[i] is somewhere in the current path. After the un-choose step, the state is identical to before the choose — the next sibling starts clean. Total: O(n · n!) time, O(n) stack + O(n) used array.

The code

python

def permute(nums):
    n = len(nums)
    result = []
    used = [False] * n

    def bt(path):
        if len(path) == n:
            result.append(list(path))   # complete permutation — record it
            return
        for i in range(n):
            if used[i]:
                continue               # skip elements already in the path
            used[i] = True
            path.append(nums[i])        # choose
            bt(path)                    # explore
            path.pop()                  # un-choose
            used[i] = False

    bt([])
    return result
Narrate it to the interviewer

"I need all n! orderings of a distinct array. The output alone is n! permutations of length n, so the work is at least O(n · n!). I'll use a backtracking tree with a boolean used array. At each level I loop through all indices; if an element is already in the current path I skip it. Otherwise: mark it used, append it, recurse, pop it, unmark it. When the path length hits n, I've got a full permutation — I record it. The invariant is that used[i] is true precisely when nums[i] is in the live path, and the un-choose step restores it exactly. O(n · n!) time, O(n) extra space."

Dry-run aloud: n=1 (returns [[x]]), n=2 (returns [[a,b],[b,a]]), and verify the used array is all-false at the root of every sibling branch.

What the interviewer asks next

"What if there are duplicate values?" (Permutations II, LeetCode 47) — sort the array, then inside the loop add: if i > 0 and nums[i] == nums[i-1] and not used[i-1]: continue. This skips a sibling branch that would build the identical permutation via a different index. "Can you do it in-place?" — swap nums[pos] with nums[i], recurse with pos+1, then swap back. No extra array, same complexity. "Why not use a set instead of used[]?" — fine for integers, but O(n) lookup vs O(1) with a boolean array indexed by position.

03 Combinations

A start index prevents duplicate sets

Combinations are like subsets with a fixed size, and Combination Sum adds the twist that the same element may be reused as many times as needed — but you must not count the same multiset twice. The danger is generating [2, 3] and [3, 2] as separate results when they represent the same combination.

The fix is the start index: the recursive call is always passed an index equal to or greater than the current element's position in the sorted input. This means the search only moves forward through the candidates, and since the candidates are sorted you can also apply a powerful pruning rule: if the remaining target is less than the smallest remaining candidate, the entire sub-branch is dead — cut it.

The decision tree for Combination Sum on candidates [2, 3, 5] with target 7:

Start index vs used[]

Use a start index when the answer is a combination (order doesn't matter, or elements can repeat from the same position onward). Use used[] when the answer is a permutation (order matters and each element may be used once). The two guards are the main structural difference between subset/combination backtracking and permutation backtracking.

Worked problemMediumLeetCode 39
Combination Sum

Given a list of distinct positive integers and a target, find every unique list of numbers from the candidates that sum to the target. Each candidate may be used an unlimited number of times in a single combination.

Think it through

Restate

Find all multisets of candidates (repetition allowed) that sum to target. Two lists with the same elements in different orders count as one combination, so I must not emit duplicates.

Brute force

Try every ordered sequence via permutations and deduplicate with a set of frozensets. This over-generates wildly — O(target/min_candidate)n — and the deduplication overhead is expensive.

The waste

The brute force generates [2,5] and [5,2] separately and then discards one. The start index guard prevents this: by only moving forward in the sorted candidate list, each combination appears in exactly one canonical (non-decreasing) order.

The move

Sort candidates. At each call pass a start index and the remaining target. Loop from start to the end. If candidates[i] > remaining, break (sorted, so all later candidates are also too big — prune). Otherwise: append, recurse with the same i (reuse allowed) and reduced target, then pop.

Invariant & cost

Every combination appears in the output in sorted order, exactly once. The pruning from the sorted-break cuts the tree substantially: no branch past an element larger than the remaining target. Worst-case still O(nT/min) time where T is the target and min is the smallest candidate, but real inputs see far fewer paths. Stack depth is O(T/min).

The code

python

def combinationSum(candidates, target):
    candidates.sort()       # sort enables the pruning break below
    result = []

    def bt(start, remaining, path):
        if remaining == 0:
            result.append(list(path))   # exact hit — record
            return
        for i in range(start, len(candidates)):
            if candidates[i] > remaining:
                break                   # sorted: rest are even bigger — prune
            path.append(candidates[i])
            bt(i, remaining - candidates[i], path)   # i (not i+1): reuse allowed
            path.pop()

    bt(0, target, [])
    return result
Narrate it to the interviewer

"I need all multisets of the candidates that hit the target — repeats allowed, but order doesn't create a new combination. I sort the candidates so I can enforce non-decreasing order via the start index, which prevents [2,5] and [5,2] from both appearing. At each level I pass the index I'm sitting on — using i, not i+1, lets me reuse the same element. When the remaining target hits zero, I record the path. When a candidate exceeds the remaining target, I break — sorted order guarantees nothing later will help either. That break is the key pruning move that keeps the tree manageable. Worst-case depth is T divided by the minimum candidate; each node does O(n) work. O(n) stack space."

Dry-run aloud: no solution exists (empty result), target equals a single candidate (one result), and reuse scenario [2,2,3] summing to 7 to confirm the i-not-i+1 choice.

What the interviewer asks next

"What if each candidate can only be used once?" (Combination Sum II, LeetCode 40) — pass i+1 instead of i, and after processing skip over equal siblings: if i > start and candidates[i] == candidates[i-1]: continue. "What about Combination Sum III (exactly k numbers summing to n, digits 1-9)?" — add a size check: prune when the path length exceeds k. The pruning levers always come from the constraints of the specific problem.

04 Constraint search

N-Queens: place, recurse, undo

N-Queens is the canonical constraint-satisfaction backtracking problem. You must place n queens on an n×n chessboard so no two threaten each other — no two queens share a row, a column, or a diagonal. The brute force is to try every permutation of columns for the n rows: n! arrangements, checking each for conflicts. For n=8 that is 40,320 arrangements, but with n=12 it is over 479 million.

Backtracking prunes far earlier. Place queens one row at a time. At row r, try each column c: if placing a queen at (r, c) is safe given what is already on the board, place it and recurse to row r+1. If no column is safe, backtrack to the previous row and try the next column there. The three conflict checks are:
(1) another queen in the same column;
(2) another queen on the same main diagonal (row − col is constant);
(3) another queen on the same anti-diagonal (row + col is constant). Keep three sets for O(1) lookup.

Worked problemHardLeetCode 51
N-Queens

Place n queens on an n×n chessboard so that no two queens threaten each other (same row, column, or diagonal). Return all valid board configurations as lists of strings.

Think it through

Restate

One queen per row (since no two can share a row). I'm assigning a column to each row such that no two queens share a column or diagonal. Return every valid assignment as a formatted board.

Brute force

Try every permutation of columns [0…n-1] assigned to rows [0…n-1]. There are n! permutations; for each one, check all C(n,2) pairs for diagonal conflicts. O(n! · n²) — not practical for n ≥ 10.

The waste

The brute force places the entire board before checking conflicts. Backtracking checks at each step: if placing the queen at (r, c) is immediately illegal, the entire subtree below it is skipped without being explored.

The move

Maintain three sets: cols, diag (row − col), anti (row + col). Recurse row by row from row 0. At row r, try each column c; if c, r-c, and r+c are all absent from their respective sets, place a queen: add the three values, set board[r][c] = 'Q', recurse to row r+1, then undo — remove the three values and reset the cell.

Invariant & cost

At any point in the search, the queens placed in rows 0 to r−1 are guaranteed conflict-free — each add/remove step maintains this. The number of valid placements is at most n! (usually far fewer). Practical complexity: O(n!) time, O(n) space for the three sets and the board (O(n²) for the board, but that is the output).

The code

python

def solveNQueens(n):
    cols, diag, anti = set(), set(), set()
    board = [['.' ] * n for _ in range(n)]
    result = []

    def bt(r):
        if r == n:                    # all rows placed — record the board
            result.append([''.join(row) for row in board])
            return
        for c in range(n):
            if c in cols or r - c in diag or r + c in anti:
                continue              # conflict — skip this column
            # choose: place queen at (r, c)
            board[r][c] = 'Q'
            cols.add(c); diag.add(r - c); anti.add(r + c)
            bt(r + 1)               # explore: next row
            # un-choose: remove queen
            board[r][c] = '.'
            cols.discard(c); diag.discard(r - c); anti.discard(r + c)

    bt(0)
    return result
Narrate it to the interviewer

"One queen per row — since no two can share a row — so I'm assigning a column to each row. The brute force tries all n! column permutations and checks diagonals after the fact — that's O(n! · n²). Backtracking checks at each step: if placing at (r, c) conflicts immediately, I skip that column entirely. I track three sets: column, main diagonal (r minus c), anti-diagonal (r plus c). Adding and removing from these sets in constant time is the key — it makes the conflict check O(1). When I reach row n, I've placed n non-attacking queens and I record the board. The invariant is that every queen placed so far is conflict-free with all earlier queens. Time is O(n!), space is O(n) for the sets plus O(n²) for the board."

Dry-run aloud: n=1 (trivially [[Q]]), n=2 and n=3 (no solution — the loop exhausts all columns and backtracks every time), n=4 (two solutions). Trace the diagonal sets for the first placement to confirm the add/remove logic.

What the interviewer asks next

"Can you output just the count of solutions?" (N-Queens II, LeetCode 52) — trivial: replace the append with a counter increment. "What about Word Search?" (LeetCode 79) — grid backtracking: at each cell try all four directions; mark the cell as visited (in-place: temporarily overwrite it), recurse, then restore. The structure is identical: choose a direction, explore, un-choose. "Why is n=1 and n=2/3 interesting?" — 1 has exactly one solution; 2 and 3 have zero — good edge cases to mention unprompted.

05 Pruning & cost

When to cut, and what it costs

The core cost of any backtracking algorithm is O(branchdepth) — an exponential whose base is the branching factor and whose exponent is the depth of the tree. For subsets it is 2n; for permutations it is n!; for combinations it depends on the target-to-minimum-element ratio. This is unavoidable when the output itself is that large. What pruning does is cut subtrees early so the explored tree is much smaller than the worst-case tree.

There are two families of pruning:

  • Constraint pruning — the partial candidate already violates a rule, so no extension can be valid. Examples: the running sum exceeds the target in Combination Sum; a queen conflicts with an earlier one in N-Queens; a cell character mismatches in Word Search. These cuts happen at the "before we recurse" check.
  • Symmetry / duplicate pruning — two branches would produce the same output. Examples: skipping equal siblings in Subsets II and Combination Sum II after sorting the input; skipping unused[i-1] in Permutations II. Sorting is usually a prerequisite for this kind of pruning.

With pruning

  • Sort the input first if duplicate-skipping is needed.
  • Check the constraint before recursing, not after.
  • Break (not continue) when candidates are sorted and one is too large — all subsequent ones are also too large.
  • Use O(1) lookups (sets, booleans) for conflict checks, not O(n) scans.

Without pruning

  • Explores every path to a leaf before checking validity.
  • Generates duplicate results if elements repeat.
  • Tree size is the absolute worst case: n! or 2n.
  • Often times out on medium or hard test cases.

The rule of thumb for interview time limits: if the input size n is at most 12–15, an O(n!) or O(2n) backtracking solution is fast enough. If n is larger, look for a DP recurrence that computes the same result without re-exploring.

When to reach for DP instead

If the problem asks "how many" or "is it possible" rather than "list all," dynamic programming usually works: the exponential search collapses to a polynomial count. For example, counting combinations that sum to a target is a classic unbounded knapsack DP — O(n · target) — while listing them all requires the O(n · 2branch) backtracking tree. Always ask: do I need the actual enumeration, or just the count/yes-no?

How to drill backtracking problems

Before coding each problem, say aloud:
(1) What are the choices at each node?
(2) What is the depth?
(3) Do I need a start index (combination order doesn't matter) or a used[] array (permutation order matters)?
(4) What is the pruning condition?
(5) When do I record a result — at every call or only at leaves? Then write only the skeleton, run it on a tiny example by hand, and add pruning after confirming correctness. Pruning that is added incorrectly silently discards valid results — always check with a known full enumeration first.

★ Review

Summary & self-test

Backtracking is depth-first search through a decision tree, combined with a pruning rule that aborts dead branches before they finish. Every backtracking problem shares the same three-move skeleton — choose, explore, un-choose — and the skill lies in wiring up the right guard (start index or used[] or conflict sets) and the right pruning condition for the problem's constraints.

01Three-move skeletonChoose, explore, un-choose. The un-choose step is what makes each branch start with a clean slate.
02SubsetsRecord at every call; loop from start to avoid revisiting earlier elements. 2n results.
03Permutationsused[] boolean tracks committed elements; record only at depth = n. n! results.
04CombinationsStart index prevents order-flipped duplicates; pass i (not i+1) if reuse is allowed.
05Constraint pruningCheck the constraint before recursing; break on sorted candidates that overshoot; use O(1) conflict sets.
06Duplicate pruningSort first; skip nums[i] == nums[i-1] when i > start (combinations) or when used[i-1] is False (permutations).
07Cost is branchdepthExponential but unavoidable when the output is that large. DP is the alternative when only a count or yes/no is needed.
08Grid backtrackingMark visited in-place; undo on return. Prune on character mismatch or boundary violation.
