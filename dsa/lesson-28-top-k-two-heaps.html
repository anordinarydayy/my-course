Top-K & Two Heaps

A heap is a machine for tracking an extreme value under insertions and deletions. Keep it deliberately small — bounded to size k — and you can scan an entire data stream while only ever touching its k most relevant elements. Pair two heaps back-to-back and you maintain an exact median over a live stream with O(log n) per insertion. These are the core tricks behind a family of interview problems that sort can solve but a heap solves faster.

00 The idea

What a heap gives you

A heap is a binary tree kept in a special shape: every parent is more extreme than its children (in a min-heap, the parent is smaller; in a max-heap, the parent is larger). That one invariant means the most extreme element always sits at the root, reachable in O(1). Insertions and removals take O(log n) time. Python's heapq module is a min-heap by default — to simulate a max-heap, negate the values.

Three interview patterns exploit this:

  • Bounded top-k — cap the heap at size k so you keep exactly the k largest (or smallest) elements without sorting all n. One pass, O(n log k).
  • K-way merge — drain k sorted sequences simultaneously using a size-k heap of their current heads. O(N log k) for N total elements.
  • Two-heap median — a max-heap of the lower half plus a min-heap of the upper half, kept balanced, so the median is always at one or both roots. O(log n) per element.
The core trade-off

Sorting gives you the k-th largest element in O(n log n) time, but it throws away the structure — you can never efficiently extend it when new elements arrive. A bounded heap of size k lets you maintain the answer in O(log k) per new element, which matters enormously when the stream is large and k is small.

01 Bounded heap

A min-heap of size k for the k largest

The brute force for "find the k largest elements in a stream" sorts everything: O(n log n) time, O(n) space. That is wasteful — sorting elements that are clearly too small to matter.

The insight is to keep only a min-heap of the k largest seen so far. The root of this heap is the smallest of those k elements — call it the "guard". For each new element x:

  • If the heap has fewer than k elements, push x unconditionally.
  • If x is greater than the root (the current k-th largest), pop the root and push x.
  • Otherwise x is too small to be in the top k — discard it.

After processing all n elements the heap holds exactly the k largest. The k-th largest is at the root. Time: O(n log k). Space: O(k).

Full sort

  • O(n log n) time, O(n) space.
  • Works offline on a static array.
  • Cannot extend when new elements arrive.

Bounded min-heap (size k)

  • O(n log k) time, O(k) space.
  • Handles an online stream naturally.
  • Root is always the k-th largest right now.
Worked problemEasyLeetCode 215
Kth Largest Element in an Array

You receive an unsorted integer array and an integer k. Return the value that would sit at position k if the array were sorted in descending order. Do not sort the entire array.

Think it through

Restate

Find the k-th largest distinct-position value (not distinct value) — equivalently, the element at index k-1 in a fully sorted descending copy of the array.

Brute force

Sort descending and return index k-1. Correct and simple, but O(n log n) time and O(1) or O(n) space depending on the sort. Interviewers expect you to name this and then improve it.

The waste

Sorting orders every element, but I only need to know which one is at rank k. Elements smaller than the k-th largest are sorted relative to each other for no benefit.

The move

Maintain a min-heap of exactly k elements. For each element: if the heap has fewer than k items push it; else if it beats the current root pop the root and push the new value; else discard. When done, the root is the k-th largest.

Invariant & cost

The heap always holds the k largest values seen so far, with the smallest of them at the root. Each push or pop is O(log k), and we do at most n of them: O(n log k) time, O(k) space. If k ≈ n, this does not improve on sort; the gain is when k is small relative to n.

The code

python

import heapq

def findKthLargest(nums, k):
    heap = []                           # min-heap of size k
    for x in nums:
        heapq.heappush(heap, x)
        if len(heap) > k:
            heapq.heappop(heap)         # evict the smallest; root stays k-th largest
    return heap[0]                     # root is the k-th largest
Narrate it to the interviewer

"The problem asks for the k-th largest, not a full sort. Sorting works but costs O(n log n) and I can do better. I'll keep a min-heap bounded to exactly k elements: the root at any moment is the smallest of the k largest seen — which is the k-th largest overall. For each new value I push it; if the heap grows past k I pop the root to evict the smallest. After the full scan the root is the answer. Time is O(n log k), space O(k). The invariant is that the heap always holds the k largest elements encountered so far."

Dry-run aloud: k = 1 (just track the maximum); k = n (the whole array fits in the heap, returns the minimum); a single-element array; all equal values.

02 Frequency counts

Top-K frequent elements

Sometimes you don't want the k largest values — you want the k values that appear most often. The setup changes slightly because you first need to count frequencies, and then pick the k largest counts.

The brute force is: count with a hash map in O(n), then sort the (value, count) pairs in O(d log d) where d is the number of distinct values, and take the top k. Total: O(n + d log d). When d ≈ n that is O(n log n).

The heap improvement: after counting, run the same bounded-heap trick, this time using a min-heap keyed on the count. Push each (count, value) pair; if the heap grows past k, pop the smallest-count entry. Final size is k. Time: O(n + d log k). When k is much smaller than d, that is a genuine win over a full sort of the pairs.

An even faster O(n) alternative for this specific problem is bucket sort by frequency: an array of n+1 buckets where bucket index = count. The answer is the last k non-empty entries scanned from the right. This works because counts are bounded by n, so the bucket array fits in O(n) space.

Worked problemMediumLeetCode 347
Top K Frequent Elements

Given an array of integers and an integer k, return the k values that appear most frequently. Your algorithm must be better than O(n log n) time. Order within the result does not matter.

Think it through

Restate

Count each value's frequency, then extract the k values with the largest frequencies. The result set must be better-than-sort in the worst case.

Brute force

Count with a dict in O(n), sort by count descending in O(d log d), slice the first k. Correct, but O(n log n) when d ≈ n.

The waste

Sorting all distinct counts is unnecessary when k is small. Alternatively, frequencies are bounded by n, making bucket sort possible.

The move — bucket sort variant

Build an array of n+1 buckets indexed by count. Place each value in the bucket whose index matches its count. Then scan the bucket array from right to left, collecting values until k are gathered. This is O(n) time and O(n) space, and it satisfies the "better than O(n log n)" requirement without any heap at all.

Invariant & cost

Buckets are indexed from 0 to n; count of any element is in that range; scanning right gives the highest counts first. Total: O(n) time, O(n) space. The heap approach is O(n + d log k) which is also fine when k is small.

The code

python

from collections import Counter

def topKFrequent(nums, k):
    count = Counter(nums)               # {value: frequency}
    buckets = [[] for _ in range(len(nums) + 1)]
    for val, freq in count.items():
        buckets[freq].append(val)       # place val in its frequency bucket
    result = []
    for freq in range(len(buckets) - 1, 0, -1):
        for val in buckets[freq]:
            result.append(val)
            if len(result) == k:
                return result          # stop as soon as k elements collected
    return result
Narrate it to the interviewer

"The prompt says better than O(n log n), so sorting by count is out. I'll use bucket sort: first I count frequencies with a hash map in O(n). Then I create an array of n+1 buckets — index is the frequency. Each value goes into the bucket matching its count. Finally I scan right to left, collecting values, and stop after k. Since the bucket array has at most n+1 entries and we scan it once, total time and space are both O(n). The invariant is that values in higher-indexed buckets appear more frequently."

Dry-run aloud: k equals the number of distinct values (return all); all elements are identical; k = 1 (the single most frequent value); ties between values with the same count (any answer is accepted).

03 K-way merge

Merge K sorted lists with a heap of heads

Merging two sorted lists is a well-known two-pointer exercise. Merging k sorted lists by repeatedly taking the minimum across all k current front elements is the natural extension — but with a naive linear scan, picking the minimum from k candidates costs O(k) per element and the total is O(N k) for N total elements. That is very slow when k is large.

The heap fix: keep a min-heap of the k current list heads. Each heap entry stores the current value and which list it came from (so you know where to pull the next element). On each step: pop the global minimum from the heap, append it to the output, and push that list's next node (if any). The heap always has at most k entries, so each pop/push costs O(log k). Total: O(N log k) for N elements across all lists.

Worked problemHardLeetCode 23
Merge K Sorted Lists

You are given an array of k linked-list heads, where each list is sorted in ascending order. Merge all lists into a single sorted linked list and return its head. All k lists may have different lengths.

Think it through

Restate

Drain k sorted sequences simultaneously in sorted order, stitching them into one linked list.

Brute force

Collect all values, sort them, rebuild the list. O(N log N) time, O(N) space. Correct, but ignores the fact that each individual list is already sorted.

The waste

A flat sort discards the pre-sorted structure of each list. Scanning all k heads to find the minimum every step costs O(k) per pick, totaling O(Nk). For large k that is much worse than O(N log k).

The move

Seed the min-heap with each list's first node: push (value, list_index, node). On each step: pop the heap minimum (O(log k)), attach that node to the output list, then push the popped node's .next if it exists. Repeat until the heap is empty.

Invariant & cost

The heap always holds exactly one node per non-exhausted list (at most k entries). The heap root is the globally smallest unprocessed value. Each of N total nodes is pushed once and popped once: O(N log k) time, O(k) space for the heap (O(1) extra beyond the output).

The code

python

import heapq

def mergeKLists(lists):
    dummy = ListNode(0)               # sentinel head for the output list
    tail = dummy
    heap = []
    for i, node in enumerate(lists):
        if node:
            heapq.heappush(heap, (node.val, i, node))  # (value, index, node)
    while heap:
        val, i, node = heapq.heappop(heap)
        tail.next = node                # stitch the min node onto the output
        tail = tail.next
        if node.next:
            heapq.heappush(heap, (node.next.val, i, node.next))
    return dummy.next
Narrate it to the interviewer

"The naive approach collects all values and sorts them — O(N log N) — but that ignores the existing per-list order. The other naive approach scans all k heads linearly each time — O(Nk) — which is slow when k is large. The heap approach keeps exactly one node per active list in a min-heap. Pop the global minimum, stitch it onto the output, push its successor. Each node is pushed and popped once: O(N log k) total, O(k) space for the heap. The invariant is that the heap always holds the current front of each non-exhausted list."

Dry-run aloud: some lists are empty (skip them at initialization); k = 1 (degenerate: return the single list); all lists have one element; two lists with interleaved values.

04 Two heaps

A streaming median with two heaps

The median of a dataset is the middle value when the data is sorted. For a static array you can sort in O(n log n) and index the middle. But when elements arrive one by one and you need the median after every insertion, sorting from scratch each time is O(n² log n) total — clearly too slow.

The key observation: you don't need the full sorted order. You only need to know the boundary between the lower half and the upper half. A max-heap on the left (the lower half) keeps its largest at the top. A min-heap on the right (the upper half) keeps its smallest at the top. The median is either the top of the larger half, or the average of both tops when the sizes are equal.

Insertion and rebalancing protocol:

  • Always push the new value to the max-heap (left) first. Then move the max-heap's top to the min-heap (right) to enforce left ≤ right at the boundary.
  • Rebalance sizes: if the min-heap has more elements than the max-heap, move its minimum back to the max-heap. The invariant is |left_size − right_size| ≤ 1.
  • If sizes are equal, the median is the average of both roots. If one side is larger by 1, the median is that side's root.
Worked problemHardLeetCode 295
Find Median from Data Stream

Design a class that accepts integers one at a time and, after each addition, can return the median of all values inserted so far. Both addNum and findMedian must be faster than O(n) amortized.

Think it through

Restate

Maintain an online sorted partition so that the middle can be read in O(1), with each insertion in O(log n).

Brute force

Keep a sorted list; binary search for the insertion point; insert at that index. Insertion into a sorted list is O(n) in the worst case (shifting elements). findMedian is O(1). Over n insertions: O(n²) total.

The waste

The sorted list maintains far more information than needed — the complete order of every element. I only need the split point: the largest of the lower half and the smallest of the upper half.

The move

Use a max-heap (lo) for the lower half and a min-heap (hi) for the upper half. Push to lo, then push lo's max to hi; if hi grows larger than lo, rebalance by moving hi's min back to lo. Keep len(lo) ≥ len(hi) so the median is always -lo[0] (the negated max-heap top) when sizes differ, or (-lo[0] + hi[0]) / 2 when equal.

Invariant & cost

Every element in lo is ≤ every element in hi; sizes differ by at most 1. Each addNum does at most 2 heap operations: O(log n). findMedian reads two roots: O(1). Space: O(n).

The code

python

import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap (negate values): the lower half
        self.hi = []   # min-heap: the upper half

    def addNum(self, num):
        heapq.heappush(self.lo, -num)           # push to lower half
        # enforce lo's max <= hi's min
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        # keep lo size >= hi size
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]               # odd total: extra is in lo
        return (-self.lo[0] + self.hi[0]) / 2.0  # even total: average of both tops
Narrate it to the interviewer

"To get the median I only need the boundary between the lower and upper halves, not a full sorted order. I use two heaps: lo is a max-heap of the lower half, hi is a min-heap of the upper half. On each insertion I push to lo, then move lo's max to hi — this enforces that everything in lo is smaller than everything in hi. Then if hi has grown larger than lo, I move hi's minimum back. The invariant is that lo's size is at most one more than hi's size, so the median is either the top of lo or the average of both tops. Python's heapq is a min-heap; I negate values to simulate a max-heap."

Dry-run aloud: single element (median = that element); two elements; all equal values; negative numbers; the sequence 1, 2, 3 to verify median progresses 1 → 1.5 → 2.

What the interviewer asks next

"What if the stream has only values in range [0, 100]?" → maintain 101 frequency buckets plus running prefix counts; median in O(1). "What if you need the median after removing elements?" → add lazy deletion with a hash map tracking removed elements, cleaning them from the heap on demand. "What if memory is tight and you can tolerate approximate medians?" → reservoir sampling or a quantile sketch (P2 algorithm). These all hinge on stating the invariant clearly before coding.

05 Recognition

When to reach for a heap, and the traps

Train your ear for these signals in a problem statement:

  • "Find the k largest / k smallest / k-th largest" in a stream or an unsorted array.
  • "Merge k sorted" sequences — lists, arrays, or iterators.
  • "Running median", "sliding window median", or "percentile over a stream".
  • A cost that is "always the cheapest available task" — scheduling problems often hide a min-heap.
  • A graph shortest path when you can't use BFS (weighted edges) — Dijkstra's algorithm is a min-heap application.

And the bugs that cost candidates the round:

  • Wrong heap direction. For the k largest use a min-heap of size k (the root is the k-th largest guard). Reaching for a max-heap stores the wrong thing.
  • Forgetting to negate for max-heap in Python. heapq is always a min-heap. Push -value and read -heap[0].
  • Tie-breaking in the k-merge heap. If two nodes have equal values, Python will try to compare the second tuple element. If that is a ListNode, it will crash. Always include a tie-breaker integer (list index) in the tuple: (val, idx, node).
  • Off-by-one in the two-heap invariant. After every push, rebalance so len(lo) ≥ len(hi). If you accidentally allow len(hi) > len(lo), the median formula breaks.
  • Using a heap where a deque would do. Sliding window maximum/minimum is better served by a monotonic deque in O(n) than a heap in O(n log n).
The decision tree in one line

You need a k-th extreme and the data is a stream or unsorted → bounded min-heap of size k. You need to merge k pre-sorted sequences → size-k heap of current heads. You need an exact median under insertions → two heaps. You need sliding window extremes → monotonic deque, not a heap. You need all-pairs shortest paths on a sparse graph → Dijkstra (priority queue).

06 Practice

Practice ladder

Work top to bottom. For every card: state the brute force and its big-O, name the heap move, code it, then dry-run the edge cases listed on the card. Aim for easy drills in under 15 minutes and medium/hard in under 30.

Design a class initialized with a list of numbers and an integer k. For each new number added, return the k-th largest element in all values seen so far.

Idea Maintain a min-heap of exactly k elements; the root is always the answer. O(log k) per query.

EasyLast Stone Weightmax-heap · LC 1046

Each round, smash the two heaviest stones together; the heavier minus the lighter survives. Return the weight of the last stone (or 0 if none remain).

Idea Max-heap: pop the two largest, compute the remainder, push back if nonzero. O(n log n).

EasyKth Largest Element in an Arraybounded heap · LC 215

Given an unsorted array and integer k, find the value at position k in a descending sort without fully sorting.

Idea Min-heap of size k; root after full scan is the answer. O(n log k).

MediumTop K Frequent Elementsfrequency + heap · LC 347

Return the k values that appear most often in an array; beat O(n log n).

Idea Count with a hash map, then bucket sort by frequency; scan from the end. O(n).

MediumTop K Frequent Wordsfrequency + heap · LC 692

Return the k words with the highest frequencies from a list of strings; break ties alphabetically.

Given a list of 2D points, return the k that are closest to the origin. Exact order among the k does not matter.

MediumTask Schedulermax-heap + counting · LC 621

A CPU runs tasks from a list, with a cooldown n between any two identical task types. Return the minimum total time needed to finish all tasks.

Idea Greedy with a max-heap of counts: always schedule the most frequent eligible task. O(t log d) where t = total tasks, d = distinct types.

Rearrange the characters of a string so that no two adjacent characters are identical. Return any valid rearrangement, or an empty string if impossible.

MediumMerge K Sorted Listsk-way merge · LC 23

HardFind Median from Data Streamtwo heaps · LC 295

Support inserting integers one at a time and querying the current median efficiently.

Idea Max-heap for lower half, min-heap for upper half; rebalance after each insert so sizes differ by at most 1. O(log n) insert, O(1) median.

HardSliding Window Mediantwo heaps + lazy deletion · LC 480

Given an array and window size k, return the median of each sliding window as the window advances one step at a time.

HardIPOgreedy + two heaps · LC 502

You have initial capital w and can complete at most k projects; each project has a profit and a required minimum capital. Maximize total capital after completing k projects.

Idea Sort projects by capital; use a max-heap of profits of affordable projects; greedily take the most profitable affordable project k times. O(n log n).

How to drill these problems

For every card:
(1) state the brute force and its big-O before touching the keyboard;
(2) name the heap direction (min or max), the invariant, and the bound on heap size;
(3) code it — use Python's heapq with negation for max-heaps;
(4) narrate the time complexity — n elements, each costing O(log k) or O(log n);
(5) dry-run at least two edge cases aloud — empty input, k equals the array length, all-equal elements. In Singapore SWE loops, explaining the heap invariant clearly is often more valued than getting to a working solution silently.

★ Review

Summary & self-test

The whole pattern compresses to two ideas: a bounded heap keeps only the k most relevant elements in O(n log k) time, and two heaps back-to-back maintain the exact median by tracking the boundary between halves. Both are O(log n) per operation and both handle streaming data that a static sort cannot efficiently extend.

01Bounded min-heapFor k largest: keep a min-heap of size k; root is the k-th largest guard. O(n log k), O(k) space.
02Heap direction mattersk largest → min-heap of size k. k smallest → max-heap of size k. Negate values in Python for max-heap.
03Frequency + heapCount first (O(n)), then heap or bucket sort on counts. Bucket sort hits O(n) when counts are bounded by n.
04K-way mergeSeed a min-heap with one node per list; pop min, push its successor. O(N log k) for N total elements.
05Two-heap medianMax-heap (lower half) + min-heap (upper half); rebalance after every insert; sizes differ by ≤ 1.
06Median readsIf sizes equal: average of both roots. If left is larger: left root. O(1) per query after O(log n) insert.
07Tuple tie-breakingIn Python heaps holding nodes, always include an integer tie-breaker to avoid comparing non-comparable objects.
08Heap vs dequeSliding window extremes → monotonic deque O(n). Streaming k-th extreme → bounded heap O(n log k).
