Math & Number Theory Essentials

A surprising number of coding-interview problems reduce to a handful of classical number-theory tools: the Euclidean algorithm for greatest common divisors, the Sieve of Eratosthenes for prime generation, modular arithmetic for keeping results in range, and binary exponentiation for raising numbers to large powers efficiently. Recognizing which tool applies — and why it is fast — is the whole skill.

00 The idea

Why number theory shows up in interviews

Pure number-theory problems are not the most common interview topic, but they appear often enough — especially at companies that care about correctness proofs and mathematical reasoning — that gaps here cost candidates. More importantly, several everyday patterns secretly rest on number-theory ideas. Repeating-cycle detection, string tiling, hash design, combinatorial counting, and cryptographic primitives all reach for these same building blocks.

There are four tools worth owning completely. Each one removes a bottleneck from a naive brute-force:

  • Euclid's GCD — shrinks the pair (a, b) by remainder repeatedly; finds the greatest common divisor in O(log min(a, b)) steps rather than trying every candidate divisor.
  • Sieve of Eratosthenes — marks all primes up to n in O(n log log n) time rather than testing each number individually with trial division at O(sqrt(n)) per number.
  • Modular arithmetic — lets you do addition, multiplication, and subtraction on huge numbers without overflow, keeping every intermediate result inside a safe range.
  • Binary (fast) exponentiation — computes x^n in O(log n) multiplications by repeatedly squaring, rather than n multiplications one by one.
How to spot a math problem

The problem mentions "divisors", "multiples", "prime", "GCD", "modulo", "power", or "factorial" — or you notice the answer involves a huge number that must be returned "mod 10^9+7". Those are your trigger words. State the brute-force complexity first, then name the tool that removes the bottleneck. That narrative arc is exactly what interviewers reward.

01 GCD & LCM

Greatest common divisor via Euclid

The greatest common divisor of two positive integers a and b is the largest integer that divides both without a remainder. The naive approach checks every integer from 1 up to min(a, b) and keeps the largest hit — O(min(a, b)) steps, which is hopelessly slow when the inputs are large.

Euclid's key insight: if d divides both a and b, it also divides a mod b. So replacing the larger value with the remainder does not change the GCD. Repeat until the remainder is zero — whatever is left is the answer. The recurrence is:

gcd(a, b) = gcd(b, a mod b)      gcd(a, 0) = a

Each application at least halves the larger value (because a mod b < b, and if a mod b < b/2 we are done in one step, otherwise b itself halves in the next step). So the pair shrinks to zero in at most O(log min(a, b)) steps.

Once you have GCD, the least common multiple (LCM) falls out for free: lcm(a, b) = a * b // gcd(a, b). Be careful of overflow: compute GCD first, then divide before multiplying.

Euclid (recursive / iterative)

  • O(log min(a, b)) steps; fits in a two-line function.
  • Works on arbitrarily large integers (Python handles big ints natively).
  • LCM is free: a * b // gcd(a, b) — divide first to avoid overflow.

Trial division for GCD

  • O(min(a, b)) — visits every candidate divisor one by one.
  • Fine for very small inputs (up to a few thousand).
  • Becomes prohibitive when inputs are in the millions or billions.
Worked problemEasyLeetCode 1071
Greatest Common Divisor of Strings

You are given two strings s and t. A string x is said to "divide" a string y if y is just x repeated some whole number of times. Return the longest string that divides both s and t, or an empty string if no such string exists.

Think it through

Restate

I need the longest prefix of both strings that tiles each of them perfectly — a "string GCD".

Brute force

Try every prefix length from min(len(s), len(t)) down to 1. For each candidate prefix p, check whether s and t are both assembled by repeating p. That is O(n · (n + m)) — checking n candidate lengths, each requiring O(n + m) work to validate — where n = min(len(s), len(t)).

The waste

We are repeatedly re-examining prefixes that share the same divisibility structure. The key insight is the direct analogy with integer GCD: if there is a common tiling unit, the answer's length must be gcd(len(s), len(t)).

The move

First verify that a common tiling can exist at all: if s + t equals t + s, both are constructed from the same tile (if they weren't, concatenating in either order would expose a mismatch). Then the answer is simply s[:gcd(len(s), len(t))].

Invariant & cost

The concatenation check is O(n + m) in time and O(n + m) for the new strings; the gcd call is O(log min(n, m)). Overall O(n + m) time, O(n + m) space (for the concatenation). The reduction from a length-scan down to a single gcd call is the whole win.

The code

python

from math import gcd

def gcdOfStrings(s, t):
    # If a common tile exists, both orderings of concatenation are equal.
    if s + t != t + s:
        return ""
    # The tile's length must equal gcd of the two string lengths.
    length = gcd(len(s), len(t))
    return s[:length]
Narrate it to the interviewer

"Restating it: I need the longest string that tiles both inputs perfectly — a string analogue of GCD. The brute force tries every prefix length, O(n · (n+m)). But I notice that if a common tile exists, then appending s to t must give the same result as appending t to s — because both are just different-length repeats of the same tile. So I check s + t == t + s first; if that fails I return empty. Otherwise the tile's length has to divide both string lengths, so the longest one is at position gcd(len(s), len(t)) in s. O(n+m) time."

Dry-run aloud: s = "AB", t = "ABABAB" — "AB" + "ABABAB" vs "ABABAB" + "AB" — both are "ABABABAB"; gcd(2, 6) = 2; s[:2] = "AB". Also test s = "A", t = "B" — "AB" != "BA", return "". And s = t = "X" — gcd(1,1)=1, s[:1]="X".

What the interviewer asks next

"What if the strings are not ASCII — does your check still work?" → Yes, Python string equality is always character-by-character. "Can you do this without the math import?" → Yes: write two lines of Euclid iteration: while b: a, b = b, a % b. "What is LCM useful for?" → Finding when two cycles synchronize, e.g. finding the first time two counters with different periods both reset to zero. lcm(a, b) = a // gcd(a, b) * b — divide before multiplying to avoid overflow in languages with fixed-width integers.

02 Primes

Primes and the Sieve of Eratosthenes

A prime is a positive integer greater than 1 that has no positive divisors other than 1 and itself. Testing a single number n for primality by trial division takes O(sqrt(n)) time — for each candidate divisor d from 2 to sqrt(n), check whether d divides n. That is because any non-trivial factor must be at most sqrt(n).

But when you need all primes up to n — as Count Primes (LC 204) requires — trial division applied to every number in 1…n costs O(n · sqrt(n)), which is roughly 3 × 10^10 operations for n = 10^6. Far too slow.

The Sieve of Eratosthenes flips the problem. Instead of asking "is this number prime?" for each number individually, it asks "which multiples of a prime can I immediately cross off?" Start with all numbers from 2 to n marked as potentially prime. Take 2 — it is prime — and cross off every multiple of 2 starting at 4 (2×2). Move to the next unmarked number (3), cross off every multiple of 3 starting at 9. Continue. Any number that reaches the scan still unmarked is prime: nothing smaller divided it. You only need to sieve primes up to sqrt(n), because any composite number ≤ n has a prime factor ≤ sqrt(n).

The total work is a harmonic-series sum: n/2 + n/3 + n/5 + …, which converges to O(n log log n) — effectively linear for practical n.

Worked problemMediumLeetCode 204
Count Primes

Given an integer n, return the count of prime numbers strictly less than n. So for n = 10 the answer is 4 (the primes 2, 3, 5, 7).

Think it through

Restate

Count every prime in the range [2, n-1]. I need to generate all of them, not just test one.

Brute force

For each number k from 2 to n-1, trial-divide by every integer up to sqrt(k). Correct, but costs O(n · sqrt(n)) — around 10^9 operations when n = 10^6. That is over the limit.

The waste

Trial division re-examines the same small primes for every single number. The sieve avoids this entirely by crossing off composites in bulk the moment their prime factor is identified.

The move

Allocate a boolean array is_prime of size n, all True. Mark 0 and 1 as not prime. For each p from 2 to sqrt(n): if is_prime[p] is still True, mark every multiple of p starting at p*p as False (smaller multiples were already cleared by earlier primes). Finally, count the True entries.

Invariant & cost

When the outer loop reaches p, every composite number below p has already been crossed off by one of its prime factors. Starting at p*p (not 2*p) saves work — everything below p*p was handled by smaller primes. Total cost O(n log log n) time, O(n) space for the boolean array.

The code

python

def countPrimes(n):
    if n < 2:
        return 0
    is_prime = [True] * n      # indices 0 .. n-1
    is_prime[0] = is_prime[1] = False
    p = 2
    while p * p < n:            # only sieve up to sqrt(n)
        if is_prime[p]:
            # cross off p*p, p*p+p, p*p+2p, ... (all < n)
            for multiple in range(p * p, n, p):
                is_prime[multiple] = False
        p += 1
    return sum(is_prime)         # True counts as 1
Narrate it to the interviewer

"I need to count every prime below n. The brute-force tests each number by trial division — O(n sqrt(n)), which is too slow at n = 10^6. The sieve does it differently: I allocate a boolean array of size n, mark 0 and 1 as not prime, then for every prime p from 2 up to sqrt(n), I cross off all multiples of p starting at p-squared — smaller multiples are already handled. Whatever remains True is prime. I just return the sum. Time is O(n log log n), space is O(n)."

Dry-run aloud: n = 0 or 1 (return 0 immediately). n = 2 (no inner loop, is_prime is [False, False], sum = 0). n = 3 (index 2 is True, sum = 1). For the p*p start: when p = 3, 3*2 = 6 is already crossed by 2; p*p = 9 is the first one only 3 knows about.

What the interviewer asks next

"How do you test a single number for primality?" → Trial divide up to sqrt(n): O(sqrt(n)). Handle 2 and 3 specially, then check 6k±1 patterns to save work. "What about very large n — can you still sieve?" → For n up to about 10^7 a flat boolean array is fine; beyond that, a segmented sieve splits the range into chunks that fit in cache. "Can the sieve be made faster in practice?" → Use a bit-array instead of a byte array (8× smaller), skip even numbers from the start (sieve only odds), and start marking at p*p.

03 Modular arithmetic

Keeping huge numbers small

Many combinatorics and dynamic-programming problems produce answers that are astronomically large — think the number of ways to tile a grid or the sum of all subset products. Rather than return the raw answer (which might have millions of digits), interviewers ask you to return it modulo 10^9 + 7 (a common prime, written MOD). This keeps every number representable in 32 or 64 bits.

The rules that make this safe are straightforward: for any integers a, b, and a positive modulus M:

  • (a + b) mod M = ((a mod M) + (b mod M)) mod M
  • (a × b) mod M = ((a mod M) × (b mod M)) mod M
  • (a - b) mod M = ((a mod M) - (b mod M) + M) mod M  (add M to avoid a negative)

Division (and therefore combination numbers) is trickier: you cannot simply do (a / b) mod M. Instead you need a modular inverse — a number inv_b such that b * inv_b ≡ 1 (mod M). When M is prime (as 10^9 + 7 is), Fermat's little theorem gives inv_b = b^(M-2) mod M, computed with fast exponentiation in O(log M).

The reusable rule

Apply % MOD after every addition and every multiplication in your DP loop — not just at the end. Forgetting a single intermediate mod can let a value overflow a 64-bit integer inside Python (which handles big ints for you), but will silently corrupt answers in C++ or Java where overflow wraps around. Python's pow(base, exp, mod) built-in computes modular exponentiation in O(log exp) and is the cleanest way to compute modular inverses.

Worked problemMediumLeetCode 50
Pow(x, n)

Implement a function that raises a floating-point number x to an integer power n, where n can be negative or zero. Do not use a library power function.

Think it through

Restate

Compute x^n. n can be any 32-bit integer including negative values and zero. I cannot call ** or math.pow.

Brute force

Multiply x by itself n times in a loop. That is O(n) multiplications. When n = 2^31 − 1 (about two billion), that is impossible in any reasonable time.

The waste

Multiplying x a billion times repeats identical subcomputations. x^8 = (x^4)^2 = ((x^2)^2)^2 — only 3 multiplications, not 8. Squaring halves the exponent each time, so the total is O(log n).

The move

Binary exponentiation (fast power): maintain a running base and a running result. Inspect the bits of the exponent from least significant to most significant. If the current bit is 1, multiply result by base. Always square base and shift the exponent right by 1. Repeat until the exponent hits 0. For negative n, compute the positive case then take the reciprocal.

Invariant & cost

At every iteration, result × base^remaining_exponent equals the original x^n. When remaining_exponent = 0, result is the answer. Each iteration halves the exponent: O(log |n|) time, O(1) space. Edge case: n = −2^31 is the most negative 32-bit integer; negating it naively overflows in 32-bit languages, so convert exponent to a Python int or use abs carefully.

The code

python

def myPow(x, n):
    if n < 0:
        x, n = 1 / x, -n    # negative exponent: flip base, negate n
    result = 1.0
    base = x
    while n > 0:
        if n & 1:              # lowest bit of n is 1
            result *= base
        base *= base           # square the base each round
        n >>= 1               # shift exponent right (divide by 2)
    return result
Narrate it to the interviewer

"x^n via O(n) repeated multiplication is obvious but prohibitively slow for n near 2^31. The key is binary exponentiation: I look at the bits of n from right to left. If bit k is 1, I multiply the result by base; regardless, I square base and shift n right. This works because n in binary tells me exactly which powers of 2 I need to multiply together. I handle negative n by inverting x and negating n at the start. O(log n) multiplications, O(1) space."

Dry-run aloud: x=2, n=10 — binary 1010. Bit 0 = 0 (skip), base becomes 4, n=5. Bit 0 = 1, result = 4, base becomes 16, n=2. Bit 0 = 0, base becomes 256, n=1. Bit 0 = 1, result = 4*256 = 1024, done. x=2^10=1024 confirmed. Also test x=2, n=0 (loop never runs, result = 1.0), and x=2, n=-1 (x becomes 0.5, n becomes 1, result = 0.5).

04 Fast exponentiation

Binary exponentiation step by step

The worked problem above described binary exponentiation in prose; this figure makes the bit-inspection loop visual. We trace x = 3, n = 13 (binary: 1101). Each step inspects one bit of the exponent from the least significant end, optionally folds the current base into result, then squares base for the next round.

Observe how result only updates when a bit is 1: it accumulates 3^1, 3^4, and 3^8 (the bits set in 13 = 1 + 4 + 8), for a product of 3 × 81 × 6561 = 1594323 = 3^13. Only 4 multiplications for an exponent of 13 instead of 13 multiplications.

The reusable shape

The same binary-exponentiation skeleton works under any modulus: replace every * with * ... % MOD. Python's pow(base, exp, mod) does exactly this in C speed, so prefer it in contests. In C++ or Java you must be careful that base * base does not overflow 64-bit integers before the mod — cast to long long or use __int128.

What the interviewer asks next

"How would you compute x^n mod M?" → Same loop, add % M after each multiplication of result and base. "What if x is a matrix?" → Matrix exponentiation: same binary skeleton, but base is a matrix and multiplication is matrix multiplication. Costs O(k^3 log n) for k-by-k matrices; used to compute Fibonacci in O(log n). "Is the recursive or iterative version cleaner?" → Both have the same O(log n) complexity; the iterative avoids call-stack overhead and is easier to reason about under overflow constraints.

05 Combinatorics

A nod to combinatorics

Combinatorics problems — "in how many ways can you…" — often reduce to choosing k items from n (C(n, k), read "n choose k") and returning the answer modulo a prime. The tools from the previous sections wire together naturally here.

Pascal's triangle is the simplest route for small n: build the triangle row by row, applying mod at every addition. For larger n (say n up to 10^5), precompute factorials and modular inverses of factorials up to n, then answer each query in O(1) with C(n, k) = fact[n] * inv_fact[k] % MOD * inv_fact[n-k] % MOD.

The precomputation pattern

Any problem that asks for many C(n, k) queries under a prime modulus should precompute two arrays of length N+1: fact[i] = i! mod MOD (built by multiplying incrementally) and inv_fact[i] = pow(fact[i], MOD-2, MOD) (using fast exponentiation, then built backwards with inv_fact[i-1] = inv_fact[i] * i % MOD). Each query then takes O(1) with three array lookups and two multiplications.

06 Practice

Practice ladder

Number theory problems reward pattern recognition more than raw cleverness. For each problem below, say the brute-force approach first, name the bottleneck it hits, then name the tool that removes it. Code from scratch — the mechanics of Euclid, the sieve, and binary exponentiation all have to be automatic.

EasyGreatest Common Divisor of StringsGCD / Euclid

Two strings are given; find the longest string that tiles both of them perfectly by repetition.

Idea Verify s+t == t+s; return s[:gcd(len(s), len(t))]. O(n+m) time.

EasyFind GCD of ArrayGCD / Euclid

Given a list of positive integers, find the GCD of the minimum and maximum values.

Idea Single pass to find min and max, then one gcd(min, max) call. O(n) with O(log min) for the GCD.

EasyNumber of Days Between Two Datesmath

Given two calendar dates as strings, return the number of days between them, handling leap years.

Idea Convert each date to a day count from a fixed epoch; subtract. Leap year: divisible by 4 and (not 100 or divisible by 400).

MediumCount Primessieve

Count how many prime numbers exist strictly below a given integer n.

Idea Sieve of Eratosthenes: boolean array, cross off multiples starting at p*p. O(n log log n), O(n) space.

MediumPow(x, n)binary exponentiation

Compute x raised to the integer power n (positive, negative, or zero) without a library power function.

Idea Inspect bits of n: bit=1 folds base into result; always square base; shift n right. O(log |n|) multiplications.

MediumSuper Powmodular exponentiation

Compute a^b mod 1337 where b is given as a very large integer encoded as an array of digits.

Idea Use the identity a^(bcd) = (a^(bc))^10 * a^d mod M, processing digits one by one with modular fast exponentiation.

MediumUgly Number IIprime factors / merge

The "ugly numbers" are those whose only prime factors are 2, 3, or 5. Return the n-th one.

Idea Three-pointer DP: maintain the next ugly as min of 2*dp[i2], 3*dp[i3], 5*dp[i5]; advance whichever pointer produced the min.

MediumBulb Switcherperfect squares / math

n bulbs start off; in round k you toggle every k-th bulb. After n rounds, how many bulbs are on?

Idea A bulb stays on iff it is toggled an odd number of times — which happens exactly for perfect squares (divisors come in pairs except for the square root). Answer: floor(sqrt(n)).

MediumSmallest Value After Replacing With Sum of Prime Factorsprime factorization

Replace a number with the sum of its prime factors repeatedly until it stabilizes; return the final value.

Idea Trial-divide by 2, then by odd divisors up to sqrt(n) to collect prime factors; sum them; repeat. Converges quickly because the sum of factors < the number for composites > 4.

HardNth Magical NumberGCD / LCM + binary search

A "magical" number is divisible by either integer a or b. Return the n-th smallest magical number, modulo 10^9 + 7.

Idea Binary search on the answer x: count = x//a + x//b - x//lcm(a,b) (inclusion-exclusion). Find the smallest x where count >= n. O(log(n * min(a,b))).

HardCount Ways to Build Good Stringscombinatorics / DP mod

Count strings of a given length range constructed by appending 0s or 1s in fixed-length chunks, modulo 10^9 + 7.

Idea DP where dp[i] = number of strings of length exactly i; transition from dp[i-zero] and dp[i-one], mod at every step. Sum dp for lengths in [lo, hi].

HardSuper Egg Dropbinary search + math

Given k eggs and n floors, find the minimum number of trials needed in the worst case to determine the highest safe floor.

Idea Instead of "given trials, find floors", ask "given t trials and k eggs, how many floors can I cover?" — that value has a closed form C(t,1)+C(t,2)+...+C(t,k). Binary-search on t until the sum >= n. O(k log n).

How to drill these effectively

For each problem:
(1) state the brute-force approach and its exact big-O;
(2) identify the number-theory tool that removes the bottleneck;
(3) code it from memory with no autocomplete;
(4) dry-run at least two edge cases out loud — n = 0 or 1 for sieve problems, n negative for power problems, strings of length 1 for GCD problems. The narration matters: interviewing is a communication task. State your invariant before you start writing code, and say it again after you finish.

★ Review

Summary & self-test

The through-line of this lesson is replacing an exhaustive search with a mathematical shortcut: Euclid's remainder chain finds the GCD in logarithmic steps; the sieve crosses off composites in one batch pass; modular arithmetic keeps huge numbers bounded; binary exponentiation halves the exponent at every step. Recognizing which tool applies — and explaining why the brute force is inadequate — is the whole skill.
