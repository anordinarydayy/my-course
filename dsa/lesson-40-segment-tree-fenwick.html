Segment Tree & Fenwick (BIT)

When an array changes and you must answer range queries in between each update, prefix sums rebuild in O(n) time — far too slow. Segment trees and Fenwick trees (Binary Indexed Trees) both bring that cost down to O(log n) per update and O(log n) per range query, covering the full family of mutable-array, interval-aggregation problems that appear at the hardest tier of coding interviews.

00 The motivation

Why prefix sums break under updates

The prefix-sum array is a beautiful trick: build it in O(n), then answer any range query sum(l, r) in O(1) as pre[r+1] - pre[l]. But the moment the underlying data changes, the whole prefix array is invalid. Every value at or after the changed index must be recomputed — O(n) work per update.

That trade-off is fine when queries dominate and updates are rare. The moment you have a mix of updates and queries — say, n calls of each interleaved — you need something else. The brute-force alternative is to skip precomputation entirely and answer every range query by summing from scratch: O(1) per update, O(n) per query. For n queries that is O(n²) total — back to slow.

The insight is that both extremes are wasteful. You don't need to recompute the entire prefix sum on every update, and you don't need to scan the entire range on every query. You need a structure that precomputes some overlapping sub-range sums, so updates touch only O(log n) of them and queries combine only O(log n) of them. That is the core idea behind both the segment tree and the Fenwick tree.

The interview signal

Any problem that says "you will receive a series of updates and range queries" is asking for one of these two structures. If the query combines values with a function that has an inverse (sum, XOR), a Fenwick tree is usually leaner and faster to code. If the query is non-invertible (min, max, GCD), a segment tree is the only clean answer.

01 Segment tree — build

Storing range sums in a complete binary tree

A segment tree maps the input array onto the leaves of a complete binary tree. Each internal node stores the aggregate (here, the sum) of the sub-array covered by that subtree. The root covers the entire array; every node at depth d covers half the range of its parent.

The construction trick is to use a flat array of size 4n and address nodes by index: the root is at index 1, the left child of node i is at 2i, and the right child is at 2i+1. This avoids allocating any explicit tree objects. Building the tree is a single post-order traversal: fill the leaves, then every parent receives the sum of its two children. Build time is O(n).

Why 4n slots?

A complete binary tree with n leaves has at most 2n - 1 nodes, but when n is not a power of two the flat-array addressing with 2i / 2i+1 may use up to 4n slots before the last level fills. Using 4n is the safe over-allocation that avoids any bounds check.

Worked problemMediumLeetCode 307
Range Sum Query — Mutable

You are given an integer array. Two operations arrive in any order: update(i, val) sets arr[i] to val, and sumRange(l, r) returns the total of all elements from index l to r inclusive. The operations can be interleaved many thousands of times.

Think it through

Restate

Support point updates and range-sum queries on a mutable array, in the most efficient way for repeated mixed operations.

Brute force

Either recompute the prefix sum on every update — O(n) per update, O(1) per query — or skip precomputation and scan the range on every query — O(1) per update, O(n) per query. Both are O(n) per mixed call; with n calls that is O(n²) total.

The waste

Prefix sum recomputes every suffix on each update even though most of the prefix array stays valid. Brute-force scanning recomputes overlapping sub-ranges that could be cached. Both extremes avoid learning from prior work.

The move

Build a segment tree over the array. Each node stores the sum of its covered range. To update index i: walk from the matching leaf up to the root, recomputing each ancestor's sum — at most O(log n) nodes. To query [l, r]: recurse down the tree, combining at most O(log n) non-overlapping nodes whose union exactly covers the target range.

Invariant & cost

Every node's stored sum always reflects the current state of its sub-array. Build is O(n); each update and each query is O(log n); space is O(n) for the tree.

The code

python

class NumArray:
    def __init__(self, nums):
        self.n = len(nums)
        self.tree = [0] * (4 * self.n)  # flat array representation
        self._build(1, 0, self.n - 1, nums)

    def _build(self, node, lo, hi, nums):
        if lo == hi:                          # leaf: store the value itself
            self.tree[node] = nums[lo]
            return
        mid = (lo + hi) // 2
        self._build(2*node, lo, mid, nums)
        self._build(2*node+1, mid+1, hi, nums)
        self.tree[node] = self.tree[2*node] + self.tree[2*node+1]

    def update(self, i, val):
        self._update(1, 0, self.n - 1, i, val)

    def _update(self, node, lo, hi, i, val):
        if lo == hi:
            self.tree[node] = val
            return
        mid = (lo + hi) // 2
        if i <= mid:
            self._update(2*node, lo, mid, i, val)
        else:
            self._update(2*node+1, mid+1, hi, i, val)
        self.tree[node] = self.tree[2*node] + self.tree[2*node+1]

    def sumRange(self, l, r):
        return self._query(1, 0, self.n - 1, l, r)

    def _query(self, node, lo, hi, l, r):
        if r < lo or hi < l:              # no overlap: return identity
            return 0
        if l <= lo and hi <= r:           # fully contained: use stored sum
            return self.tree[node]
        mid = (lo + hi) // 2              # partial: recurse into both halves
        return (self._query(2*node, lo, mid, l, r) +
                self._query(2*node+1, mid+1, hi, l, r))
Narrate it to the interviewer

"Restating: I need a structure that handles both point updates and range queries efficiently when they're interleaved. Prefix sums give O(1) queries but O(n) updates; brute-force scanning gives O(1) updates but O(n) queries — both degrade to O(n²) for n mixed operations. So I'll build a segment tree: a flat array of size 4n whose entries represent node sums in a complete binary tree. Build is O(n). Each update walks the path from the changed leaf to the root — at most log n nodes — recomputing sums along the way. Each range query recursively splits the query range, using stored sums whenever a node's range is fully contained and recursing only when there's partial overlap. Both operations are O(log n). My invariant is that every tree node always stores the correct sum of its sub-array range."

Dry-run aloud: a single-element array (tree is just the root = leaf), and a query equal to the full range (should return tree[1] directly). Also verify the three cases in _query: fully outside returns 0, fully inside returns tree[node], partial recurses to both children.

02 Range query combining

Walking down to combine O(log n) nodes

The query algorithm is where the cleverness lives. When you ask for sumRange(1, 4) on an 8-element array, the tree already has a node that covers exactly [0,3] and another that covers [4,7]. Your range [1,4] overlaps both but fits neither exactly — so you recurse into each, and within the left subtree you pick up [1,3] (which is the right child of the root's left child), and from the right you pick up just the leaf at index 4. In total you touch at most 4 nodes per level, and there are only log n levels, so the whole query visits O(log n) nodes.

The three-case rule is the heart of every segment tree query: if the query range and the node's range do not overlap at all, return the identity (0 for sum); if the query range fully contains the node's range, return the stored value; otherwise recurse into both children. The second case is the short-circuit that makes the log-n bound work — once a node is fully covered, you never look deeper into it.

Segment tree

  • Works for any associative, not-necessarily-invertible function: min, max, GCD, product.
  • Query and update both O(log n); build O(n); space O(n).
  • Supports lazy propagation for range updates (covered next).
  • More code to write in an interview — about 30 lines in Python.

Fenwick (BIT)

  • Only works when the query function has an inverse (sum, XOR).
  • Same asymptotic cost: O(log n) update and prefix query; O(n) build; O(n) space.
  • Much leaner: the whole structure fits in ~10 lines of code.
  • Range-min/max queries are awkward or impossible without extra structure.
03 Range updates

Lazy propagation: deferring the work

Plain segment trees handle point updates cheaply, but what about adding a constant to every element in a range — say, increment a[2..6] by 5? Without lazy propagation you would call single-element update O(r - l + 1) times — potentially O(n) work. Lazy propagation fixes this.

The idea is to annotate each node with a pending delta that has been applied to this node's stored value but not yet pushed down to its children. When a range update fully covers a node, update that node's stored value and record the pending delta there. When you later need to visit a child (for a query or a narrower update that crosses the node), you first push down the pending delta: apply it to both children and clear it from the current node. This lazy push ensures every visited node has accurate data before you read it.

When to reach for lazy propagation

If the problem says "add x to every element in a range" or "set every element in a range to x", you need lazy propagation. Range-sum + range-update problems — like range add then range query — appear at Hard level. For interviews at most companies, a plain segment tree without lazy is sufficient unless the problem explicitly requires range updates.

What the interviewer asks next

"What if the range update sets all values to x instead of adding to them?" — this is a range assign lazy tag, which overwrites any existing tag rather than combining. The push-down logic must treat the two tag types differently. "Can you handle both range add and range set in the same tree?" — yes, but the tag composition rule needs careful thought; a set tag should overwrite any pending add. These edge cases are the distinguishing test for senior candidates.

04 Fenwick tree

The low-bit trick: a leaner alternative

The Fenwick tree (Binary Indexed Tree, or BIT) achieves the same O(log n) prefix-query + O(log n) point-update performance as a segment tree, but with a cleverer encoding that makes it half the code. The price: it only works when the aggregate function has an inverse (subtraction for sum, XOR for XOR). Range-min queries are not possible without augmentation.

The BIT uses a 1-indexed array bit[1..n] where each index i stores the sum of a specific range of length lowbit(i) = i & -i. That is the value of the lowest set bit of i in two's-complement. For example, lowbit(6) = lowbit(110₂) = 010₂ = 2, so bit[6] stores the sum of two elements ending at position 6.

To query the prefix sum up to index i: start at i, add bit[i], then jump to i - lowbit(i), and repeat until i reaches 0. Each step strips the lowest set bit, so the loop runs at most O(log n) times. To update index i by delta: start at i, add delta to bit[i], then jump to i + lowbit(i), and repeat until i exceeds n. Each step advances to the next covering index, and there are at most O(log n) such covering indices.

Worked problemHardLeetCode 315
Count of Smaller Numbers After Self

Given an integer array, return an answer array where answer[i] is the count of elements to the right of nums[i] that are strictly smaller than it.

Think it through

Restate

For each position i, count how many elements in the suffix nums[i+1..n-1] are strictly less than nums[i].

Brute force

For each i, scan every j > i and count nums[j] < nums[i]. That is O(1) per pair and O(n²) pairs: O(n²) total — too slow for n = 100,000.

The waste

The brute force recounts the suffix from scratch for every index. If we process the array from right to left and maintain a frequency structure over the values we have already seen, each "count smaller" question becomes a prefix-sum query over the value space — exactly what a BIT answers in O(log n).

The move — coordinate compression

The values can be large negative or positive integers. Map them to ranks 1..n by sorting unique values and replacing each with its rank. This shrinks the BIT's domain to size n.

The move — right-to-left BIT scan

Process indices from right to left. For each element: (1) query the BIT for the prefix sum up to rank(nums[i]) - 1 — that is the count of already-inserted values strictly smaller than nums[i]; (2) update the BIT at rank(nums[i]) by +1, recording that this value has been seen. Store the query result as the answer for index i.

Invariant & cost

At each step, the BIT contains the frequency of every element to the right of the current index. Coordinate compression is O(n log n); BIT has n update + query pairs each costing O(log n): O(n log n) time, O(n) space.

The code

python

def countSmaller(nums):
    # coordinate compression: map values to ranks 1..n
    sorted_unique = sorted(set(nums))
    rank = {v: i+1 for i, v in enumerate(sorted_unique)}
    n = len(nums)
    bit = [0] * (len(sorted_unique) + 2)

    def update(i):                     # add 1 at position i
        while i < len(bit):
            bit[i] += 1
            i += i & -i               # i += lowbit(i)

    def query(i):                      # prefix sum [1..i]
        s = 0
        while i > 0:
            s += bit[i]
            i -= i & -i               # i -= lowbit(i)
        return s

    result = []
    for v in reversed(nums):          # process right to left
        r = rank[v]
        result.append(query(r - 1))  # count values seen so far < v
        update(r)                     # record v as seen
    result.reverse()                   # collected right-to-left; reverse to match input order
    return result
Narrate it to the interviewer

"Restating: for each index I need the count of elements to its right that are strictly smaller — that is a suffix-counting problem. The O(n²) brute force scans the whole suffix for each element. My insight is that if I process right to left, every query 'how many smaller values have I seen so far' becomes a prefix count on the value space — perfect for a BIT. I first coordinate-compress the values to ranks 1..n, because BIT indices must be small positive integers. Then for each element from right to left I query the BIT for the sum up to rank-1 (elements strictly smaller), record that as the answer, and update the BIT at the element's rank. The invariant is that at each step the BIT holds the exact frequency of every element to the right of the current position. Coordinate compression is O(n log n), and n BIT operations each cost O(log n), so the total is O(n log n) time, O(n) space."

Dry-run aloud: [5, 2, 6, 1]. Ranks: 1→1, 2→2, 5→3, 6→4. Process right-to-left: 1 (rank 1): query(0)=0, update(1). 6 (rank 4): query(3)=1 (the 1), update(4). 2 (rank 2): query(1)=1 (the 1), update(2). 5 (rank 3): query(2)=1 (the 2, not the 1 or 6). Results collected right-to-left: [0,1,1,1]. Reverse: [1,1,1,0]. Expected: [2,1,1,0] — wait, let me recheck: for 5, values to right are [2,6,1], smaller are 2 and 1 → answer 2. Ranks: 1→1, 2→2, 5→3, 6→4. Right-to-left: index 3 (val 1, rank 1): query(0)=0 →res=[0], update(1). Index 2 (val 6, rank 4): query(3)=1 →res=[0,1], update(4). Index 1 (val 2, rank 2): query(1)=1 →res=[0,1,1], update(2). Index 0 (val 5, rank 3): query(2)=1+1=2 →res=[0,1,1,2], update(3). Reverse: [2,1,1,0]. Correct. Edge cases: all equal (all answers 0), single element (answer [0]), fully decreasing array (each answer 0).

05 Recognition

Spotting it, and the traps

Both structures solve the same family of problems. Learn to map the problem's language to the right tool:

  • Any prompt mentioning "update" and "query range" on a mutable array — reach for a segment tree or BIT.
  • Query function is sum or XOR (invertible) — BIT is leaner; both work.
  • Query function is min, max, GCD, product (not invertible) — segment tree only.
  • "Count of elements smaller / larger / equal" over a changing multiset — BIT over coordinate-compressed values (as in LeetCode 315).
  • Range updates (add to a range, not just a point) — lazy-propagation segment tree.
  • Order statistics (k-th smallest, rank of an element) — BIT with binary search, or a segment tree over value space.

And the traps that bite candidates:

  • 0-indexed vs 1-indexed: BITs are naturally 1-indexed (index 0 is never used); always add 1 when mapping from 0-indexed input. Segment trees work with either convention, but mixing them causes off-by-one errors in the range logic.
  • Forgetting the base case in _query: if the query range and node range are completely disjoint, return the identity immediately — for sum that is 0, for min that is infinity, for max that is -infinity. Returning the wrong identity corrupts the result.
  • Coordinate compression with duplicates: if values repeat, each distinct value gets one rank. Use sorted(set(nums)), not sorted(nums).
  • BIT size: the BIT array needs to be large enough for the largest rank value. Allocate len(unique_values) + 2 to be safe — the +2 avoids off-by-one bounds errors when the update loop overshoots.
  • Confusing query direction in the BIT: the prefix-query loop subtracts the low bit (i -= i & -i); the update loop adds the low bit (i += i & -i). Swapping them silently corrupts the tree.
The two-minute decision

In an interview: if the aggregate is invertible (sum, XOR) and updates are point updates, code the BIT — it is eight lines and hard to get wrong. If the aggregate is non-invertible, or you need range updates, write the segment tree. State your choice out loud before coding: "I'm going to use a Fenwick tree because sum is invertible and I only need point updates."

How to drill these structures

Write the BIT from memory first — just the 8-line template — until you can do it in under 3 minutes without looking. Then drill the segment tree's three functions: _build, _update, _query. For each drill problem:
(1) name the aggregate and whether it is invertible;
(2) state your structure choice;
(3) code it;
(4) dry-run with a small example;
(5) state the invariant and the complexity. The BIT low-bit trick and the segment tree's three-case query are the two things worth memorizing verbatim — everything else follows from the structure.

★ Review

Summary & self-test

Both structures solve the same core problem — interleaved point updates and range queries in O(log n) each — by precomputing a tree of overlapping sub-range aggregates. The segment tree is more general (works for any associative function, supports lazy range updates) but costs more to write. The Fenwick tree is leaner code and faster in practice, but requires an invertible aggregate. Pick the simpler one when you can; reach for the general one when you must.
