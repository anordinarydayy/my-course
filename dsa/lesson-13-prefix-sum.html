Prefix Sum & Difference Array

Every repeated range-sum query on a fixed array does redundant addition. A one-time O(n) build of a prefix table turns every subsequent query — no matter how many, no matter how wide the range — into a single subtraction taking O(1). That same idea powers fast subarray counting, 2-D region queries, and bulk range updates, making it one of the most broadly useful precomputation tricks in competitive and interview programming.

00 The idea

What problem does it solve?

Imagine you have an array of n numbers and a client that will ask you thousands of questions of the form: "what is the total of all elements from index l to index r?" The naive answer re-adds all the elements in that range for each query — up to O(n) work per question. With 10,000 queries on an array of length 10,000 you are looking at 10⁸ operations: right on the edge of a one-second time limit, and almost certainly over it.

The redundant work is obvious once you see it: every query re-walks a segment of the array that overlaps with segments from previous queries. Prefix sum (also called a cumulative sum or running total) is the precomputation trick that eliminates all that overlap. Build one auxiliary array — call it P — where P[i] holds the sum of everything from the start up to index i − 1. Then any range sum collapses to P[r+1] − P[l]: one subtraction, always O(1), regardless of range width.

The same idea extends in three useful directions covered in this lesson:

  • Prefix + hashmap — count subarrays whose sum equals a target k, in O(n) without a nested loop.
  • 2-D prefix sums — answer rectangle region queries on a matrix in O(1) with a similar inclusion-exclusion trick.
  • Difference array — flip the precomputation: instead of fast reads, get O(1) range updates on an array that you only need to read at the end.
The core formula

Define P[0] = 0 and P[i] = P[i−1] + a[i−1] for i from 1 to n. Then the sum of a[l..r] (both ends inclusive, 0-indexed) is exactly P[r+1] − P[l]. The off-by-one shift — the sentinel zero at P[0] — is what makes the formula work cleanly even when l = 0.

01 Build & query

Building the prefix array, then querying O(1)

The brute force re-adds up to n elements for every range query — O(n) per query, O(n·Q) total for Q queries. The redundant work is that every overlapping pair of queries shares a prefix they both compute from scratch.

The fix is to compute that shared prefix exactly once. Scan left to right, maintaining a running total: P[i] = P[i-1] + a[i-1]. This takes O(n) time and O(n) extra space and happens once. After that, sum(l, r) = P[r+1] − P[l] — one subtraction, always O(1), no matter how many queries arrive or how wide their ranges are.

The invariant is that P[i] always equals the sum of the first i elements of a. Because P[0] = 0, the formula handles a range starting at index 0 without any special case. Watch the prefix row fill up, then see how a highlighted range resolves to a single subtraction.

Prefix sum

  • O(n) build, O(1) per query.
  • O(n) extra space for the prefix array.
  • Array must be static — rebuilding on each update costs O(n).

Brute force

  • O(1) "build" (no precomputation).
  • O(n) per query — catastrophic for many queries.
  • Works fine if Q is tiny (one or two queries total).
Worked problemEasyLeetCode 303
Range Sum Query — Immutable

Design a class that takes an integer array at construction time and then answers multiple range-sum queries, each returning the sum of elements from index l to r inclusive. The array never changes between queries.

Think it through

Restate

One build phase, many query calls. I want each query to be as fast as possible — O(1) is the target.

Brute force

In each sumRange call, loop from l to r summing elements. Correct, but O(n) per query. For Q queries that is O(n·Q) total — unacceptable when both n and Q are large.

The waste

Every query re-walks a prefix of the array it shares with prior queries. If I precompute all prefixes once, the shared work is never repeated.

The move

During __init__, build a prefix array P of length n+1 where P[0] = 0 and P[i] = P[i-1] + nums[i-1]. In sumRange(l, r), return P[r+1] − P[l].

Invariant & cost

P[i] always equals the sum of the first i elements. Build is O(n) time, O(n) space; each query is O(1) time, O(1) space.

The code

python

class NumArray:
    def __init__(self, nums):
        n = len(nums)
        self.P = [0] * (n + 1)          # P[0] = 0 sentinel
        for i in range(n):
            self.P[i + 1] = self.P[i] + nums[i]

    def sumRange(self, l, r):
        return self.P[r + 1] - self.P[l]  # one subtraction
Narrate it to the interviewer

"Restating: a static array, many queries. The naive answer loops inside every query — O(n) per call, which blows up for large Q. The waste is re-summing shared prefixes. I'll precompute once: build a prefix array P of length n+1, where P[0] is a sentinel zero and P[i] is the sum of the first i elements. Then sumRange(l, r) is just P[r+1] minus P[l] — one subtraction, always O(1). My invariant is that P[i] equals the running total of the first i elements; the sentinel makes the l=0 case fall out without a special case. Build is O(n) time and space; every query is O(1)."

Dry-run aloud: an empty array (P = [0], queries are vacuously 0); a single-element array; a query where l = 0 (P[r+1] − P[0] = P[r+1], correct); a query spanning the full array.

What the interviewer asks next

"What if the array can be updated between queries?" → A plain prefix array must be rebuilt from the update point — O(n) per update. For mixed update+query workloads, a Fenwick tree (binary indexed tree) gives O(log n) for both operations. "Why is P one element longer than the input?" → The sentinel at P[0] = 0 removes the need for a special case when l = 0; P[r+1] − P[0] simplifies to P[r+1] which is the correct sum from the start.

02 Prefix + hashmap

Counting subarrays that sum to K

The brute force for "how many contiguous subarrays sum to exactly k?" tries every pair of endpoints, computes the range sum (already O(1) with a prefix array), and counts — O(n²) pairs. Even with the prefix trick for individual queries, the enumeration itself is quadratic.

The insight comes from rewriting the condition. A subarray a[i..j] sums to k if and only if P[j+1] − P[i] = k, which rearranges to P[i] = P[j+1] − k. So as you scan and compute each new prefix P[j+1], you need to know how many previous prefix values equal P[j+1] − k — a number you can look up in O(1) from a hash map that you built as you went. One forward pass, maintaining both a running prefix and a frequency count of past prefixes: O(n) time, O(n) space.

The invariant is that at each position the hash map already contains the frequency of every prefix value from P[0] through P[current], so the lookup is always complete. Seed the map with {0: 1} before the scan to count subarrays that start at index 0.

Worked problemMediumLeetCode 560
Subarray Sum Equals K

Given an integer array (which may contain negatives) and a target integer k, return the count of contiguous subarrays whose elements sum to exactly k.

Think it through

Restate

Count every contiguous segment with sum k. The array can have negatives, so a sliding window won't work — the sum isn't monotone as the window grows.

Brute force

Fix every left endpoint l, extend right computing the running sum; increment the answer whenever it equals k. O(n²) time. For n = 20,000 that's 400 million checks — too slow.

The waste

The nested loop recomputes prefix sums redundantly. Converting to the condition P[i] = prefix − k lets us ask "how many times have I seen this value before?" — answered in O(1) with a hash map.

The move

Initialize a map freq = {0: 1}. Scan with a running prefix total. At each step: add freq[prefix − k] to the answer (how many subarrays ending here sum to k), then record freq[prefix]++.

Invariant & cost

At every position, freq holds the count of each prefix value seen so far, including the seed {0:1}. The lookup freq[prefix − k] correctly counts all valid left endpoints. O(n) time, O(n) space.

The code

python

from collections import defaultdict

def subarraySum(nums, k):
    freq = defaultdict(int)
    freq[0] = 1          # empty prefix has sum 0
    prefix = 0
    count = 0
    for x in nums:
        prefix += x
        count += freq[prefix - k]  # subarrays ending here with sum k
        freq[prefix] += 1         # record this prefix for future lookups
    return count
Narrate it to the interviewer

"Restating: count subarrays summing to k, including negatives. A sliding window fails here because the sum isn't monotone. The brute force is O(n²). Instead, I reformulate: a subarray from i to j sums to k iff prefix[j+1] minus prefix[i] equals k, which means I need prefix[j+1] minus k to have appeared in previous prefixes. So I walk forward, maintaining a running prefix and a frequency map seeded with {0:1}. At each step I look up how many times the value 'prefix minus k' appeared — that's exactly how many valid subarrays end here — then I record the current prefix in the map. My invariant is that the map always contains the frequency of every prefix value up through the current position. O(n) time, O(n) space."

Dry-run aloud: k = 0 (the seed {0:1} catches the case where the entire prefix equals 0); all negatives; an array where k never appears (count stays 0); and an array of all zeros where k = 0 (every prefix equals 0, so each step finds many matches — verify with [0,0,0], k=0: answer should be 6).

What the interviewer asks next

"Why seed with {0: 1}?" → Without it, a subarray starting at index 0 that sums to k would never be counted, because there is no earlier prefix equal to prefix − k = 0. The seed pretends you visited a zero-prefix before the array started. "Does this work with negative numbers?" → Yes — the prefix approach has no monotone assumption, unlike a sliding window. "What if you need the actual subarrays, not just the count?" → Store a list of start indices in the map instead of a count, then reconstruct.

03 Two dimensions

2-D prefix sums and matrix region queries

The brute force for "sum all elements in a rectangle from (r1,c1) to (r2,c2)" visits every cell inside the rectangle — up to O(m·n) per query for an m×n matrix. Repeated queries make this O(m·n·Q): impractical for large grids.

The 1-D idea extends to two dimensions with a simple inclusion-exclusion formula. Build a 2-D prefix table P of size (m+1)×(n+1), where P[i][j] is the sum of all cells in the rectangle from the top-left corner (0,0) to (i−1,j−1). Fill it with: P[i][j] = P[i−1][j] + P[i][j−1] − P[i−1][j−1] + a[i−1][j−1]. The subtraction removes the double-counted top-left overlap.

Any query rectangle from (r1,c1) to (r2,c2) (0-indexed) then resolves to: P[r2+1][c2+1] − P[r1][c2+1] − P[r2+1][c1] + P[r1][c1]. Four table lookups and three arithmetic operations — always O(1), regardless of how large the rectangle is.

Worked problemMediumLeetCode 304
Range Sum Query 2D — Immutable

Design a class that accepts a 2-D integer matrix at construction and then answers multiple rectangle-sum queries, each returning the sum of all elements inside the rectangle from (r1,c1) to (r2,c2) inclusive. The matrix does not change between queries.

Think it through

Restate

Static 2-D matrix, many rectangle queries — O(1) per query is the goal. Directly extends the 1-D prefix idea.

Brute force

Loop over every cell inside the rectangle for each query: O(m·n) per query. For Q queries that is O(m·n·Q) — far too slow.

The waste

Cells near the top-left are summed repeatedly by many overlapping queries. Precomputing all prefix rectangles from the origin eliminates the shared work.

The move

Build P[i][j] = a[i−1][j−1] + P[i−1][j] + P[i][j−1] − P[i−1][j−1] with sentinel row and column of zeros. Query: P[r2+1][c2+1] − P[r1][c2+1] − P[r2+1][c1] + P[r1][c1].

Invariant & cost

P[i][j] always equals the sum of the sub-matrix from the top-left corner to row i−1, column j−1. Build is O(m·n) time, O(m·n) space; each query is O(1).

The code

python

class NumMatrix:
    def __init__(self, matrix):
        m, n = len(matrix), len(matrix[0])
        # P is (m+1)×(n+1), all zeros by default
        self.P = [[0] * (n + 1) for _ in range(m + 1)]
        for i in range(1, m + 1):
            for j in range(1, n + 1):
                self.P[i][j] = (matrix[i-1][j-1]
                                + self.P[i-1][j]
                                + self.P[i][j-1]
                                - self.P[i-1][j-1])

    def sumRegion(self, r1, c1, r2, c2):
        return (self.P[r2+1][c2+1]
                - self.P[r1][c2+1]
                - self.P[r2+1][c1]
                + self.P[r1][c1])
Narrate it to the interviewer

"This is the 2-D extension of the range-sum query pattern. The brute force visits every cell in each query rectangle — O(m·n) per call. I'll precompute a 2-D prefix table P where P[i][j] holds the sum of the top-left sub-matrix ending at row i−1, column j−1. The build recurrence is: take the cell value, add the prefix above and to the left, then subtract their shared corner to avoid double-counting. A query then uses four table reads and three arithmetic operations — always O(1). Build is O(m·n) time and space, each query O(1)."

Dry-run aloud: a 1×1 matrix (P is 2×2, P[1][1] = a[0][0], query is just that cell); a single-row matrix (reduces to 1-D prefix); and a query that covers the entire matrix (should equal P[m][n]).

What the interviewer asks next

"Derive the build recurrence on the fly." → Draw a 2×2 sub-table and show that P[i][j] = cell + top + left − top-left corner. "Why subtract P[i−1][j−1]?" → The row strip P[i−1][j] and the column strip P[i][j−1] both include the top-left rectangle P[i−1][j−1] — it gets counted twice, so you subtract it once. This is the 2-D inclusion-exclusion principle.

04 Difference array

Fast range updates, single read-out at the end

Consider the inverse problem: you start with an array of zeros and receive a batch of instructions of the form "add v to every element from l to r." The brute force touches each element in the range: O(n) per update. If you receive Q such instructions, that is O(n·Q) to process them all before printing the final array.

The difference array D flips the prefix idea. D[i] stores the change in value between a[i] and a[i−1], so that the original array a can always be recovered by taking a prefix sum of D. An operation "add v to a[l..r]" becomes just two point updates: D[l] += v and D[r+1] -= v. After processing all Q instructions in O(Q) time, a single O(n) prefix-sum pass on D reconstructs the final array. Total cost: O(n + Q) instead of O(n·Q).

The invariant is that taking a prefix sum of D at any moment gives you the current state of a. The two-point update preserves this invariant because the +v propagates to all positions from l onward, and the −v at r+1 cancels it beyond r.

Difference array

  • O(1) per range update, O(n) reconstruct at the end.
  • O(n) space for D.
  • Ideal when updates come in a batch and you only need to read a once at the very end.

Direct update

  • O(n) per range update — each cell in [l, r] must be visited.
  • O(1) reads at any time — the array always holds the current value.
  • Better when reads interleave with updates and ranges are tiny.
Worked problemMediumLeetCode 1094 (variant)
Car Pooling — Range Addition Pattern

A bus has a seat capacity. A list of trips each describe: passenger count, pickup stop, and drop-off stop. Trips may overlap. Determine whether all passengers can be accommodated without exceeding the seat limit at any stop.

Think it through

Restate

Each trip adds passengers for a range of stops and removes them at the end. I need to know whether the maximum occupancy ever exceeds capacity. This is "apply range updates, then find the maximum."

Brute force

For each trip, loop through the stop range and add the passenger count. Then scan for the max. O(stops · trips) time. For large stop counts and many trips this is slow.

The waste

The trips form a batch of range additions. I only need the final snapshot after all updates — a classic difference array scenario.

The move

Build a difference array D over the stop range. For each trip (passengers, from, to): D[from] += passengers, D[to] -= passengers. Prefix-sum D to get occupancy at each stop. Check if any exceeds capacity.

Invariant & cost

After the prefix-sum pass, D[i] holds the occupancy exactly at stop i. Processing all trips is O(T), reconstruction is O(N) (N = number of stops). Total: O(T + N) time, O(N) space.

The code

python

def carPooling(trips, capacity):
    # stop indices go up to 1000 per constraints
    D = [0] * 1002
    for passengers, frm, to in trips:
        D[frm] += passengers    # board at frm
        D[to]  -= passengers    # alight at to (exclusive end)
    occupancy = 0
    for v in D:
        occupancy += v          # running prefix sum
        if occupancy > capacity:
            return False
    return True
Narrate it to the interviewer

"Restating: multiple trips add and remove passengers at stops; I need to know whether occupancy ever exceeds capacity. The brute force loops over every stop in each trip — O(stops × trips). The key observation is that each trip is a range addition followed by a range subtraction — a difference array handles that in O(1) per trip. I build a difference array D over the stop range, add passengers at the pickup stop, subtract at the drop-off (passengers leave, so the effect ends there). Then I take one forward prefix-sum sweep of D, checking the running total at each step against capacity. O(T + N) total, O(N) space."

Dry-run aloud: a single trip that spans the entire route (D[0] += p, D[end] -= p; occupancy is p throughout); two trips that overlap at exactly one stop (confirm the boarding/alighting boundaries); capacity = 0 with any trip (should return False immediately); no trips (always True).

What the interviewer asks next

"When does the difference array beat direct range updates?" → When updates come in a batch and reading only happens after all updates are done. If reads interleave arbitrarily with updates, use a segment tree or Fenwick tree instead. "What if the range is huge (10⁹)?" → Use a hash map (or a sorted event list) instead of an array — sparse difference representation, same O(1)-per-update principle.

05 Recognition

Spotting it, and the traps

Train your eye for these signals in a problem statement:

  • "Multiple queries" on a fixed (immutable) array or matrix for range sums or rectangle sums — prefix sum for O(1) per query.
  • Counting subarrays with a specific sum (possibly with negatives) — prefix + hashmap for O(n).
  • A batch of range increment/decrement operations, where the final array is only needed at the end — difference array for O(Q + n).
  • The phrase "how many consecutive elements sum to k" or "number of subarrays where" — almost always a prefix-frequency approach.
  • Any problem asking you to check a balance across a range — e.g., equal 0s and 1s in a subarray, which converts to "subarray summing to 0" after replacing 0→−1.

Common bugs to watch for:

  • Off-by-one on the prefix index — the sentinel is at P[0]; the sum from l to r is P[r+1] − P[l], not P[r] − P[l-1]. Memorize the formula and dry-run with l=0 to confirm.
  • Forgetting to seed {0: 1} in the prefix-hashmap — subarrays starting at index 0 won't be counted. Always initialize freq[0] = 1 before the loop.
  • Wrong boundary in difference array — if passengers leave at stop to (exclusive), do D[to] -= v, not D[to+1] -= v. Read the problem carefully for inclusive vs exclusive endpoints.
  • Using prefix sum on a mutable array without rebuilding — every update invalidates the prefix table. For mixed update+query workloads, reach for a Fenwick tree.
The decision tree

Static array, range sum queries → prefix sum. Count subarrays summing to k (with negatives) → prefix + hashmap. Static matrix, rectangle queries → 2-D prefix sum. Batch range updates, read-at-end → difference array. Mixed updates + queries with large n → Fenwick tree (Pattern 27).

How to drill effectively

For every problem:
(1) restate it in one sentence;
(2) give the brute force and its big-O;
(3) name which prefix variant applies and why;
(4) state the invariant — what P[i] or freq or D[i] represents at all times;
(5) code it without looking at notes;

★ Review

Summary & self-test

The whole family reduces to one sentence: precompute running totals once so that any future range sum, subarray count, rectangle query, or range update collapses to a constant number of arithmetic operations. The four variants are the same core insight applied in slightly different directions — forward prefix, prefix+map, 2-D prefix, and backward via difference array.
