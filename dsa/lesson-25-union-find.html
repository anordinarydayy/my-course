Union-Find (Disjoint Set)

Dozens of graph problems reduce to one question repeated thousands of times: are these two nodes already in the same group? Union-Find answers that question in near-constant time by maintaining a forest of trees where finding a group means following parent pointers to a root, and merging two groups means linking one root to the other. With path compression and union by rank, each operation costs O(α(n)) — so close to O(1) that it is effectively free.

00 The idea

What problem does it solve?

Imagine a social network: as people friend each other, you need to answer queries like "are Alice and Bob in the same friend circle?" at any point in time. The edges arrive one at a time — the graph is dynamic. With a standard BFS or DFS you would re-traverse the whole graph after every new edge to answer each query, which is O(V + E) per query. For 10,000 people and 1,000,000 friendships, that is a billion operations before a single useful answer.

Union-Find (also called Disjoint Set Union, or DSU) is a data structure purpose-built for this dynamic connectivity problem. It maintains a partition of a set of elements into non-overlapping groups and supports exactly two operations in near-O(1) amortised time:

  • find(x) — return the canonical representative (the root) of the group containing x.
  • union(x, y) — merge the groups containing x and y into one. If they are already in the same group, nothing changes.

To check connectivity, just ask: find(x) == find(y). The key insight is that we never need the full group membership list — we only need to agree on a single representative per group, and we can get there in essentially constant time.

When to reach for Union-Find

The signal is dynamic connectivity: edges or memberships arrive over time and you must repeatedly answer "same component?" after each arrival. Static graphs (edges all given up front, single query) are better handled by DFS/BFS. Dynamic graphs with connectivity queries are the Union-Find sweet spot. Common interview triggers: connected components, detecting a cycle in an undirected graph, merging groups of items, Kruskal's MST.

01 Core operations

Find & Union — the two primitives ▸

The data structure is just two arrays: parent[i] stores the parent of node i (each node starts as its own parent — a singleton), and rank[i] stores a rough measure of the tree's depth. Both operations trace parent pointers upward until they hit a root (a node whose parent is itself).

find(x) walks up the chain: x → parent[x] → parent[parent[x]] → … until it reaches a root. Without any optimisation, a chain of n nodes costs O(n).

union(x, y) finds the roots of both nodes, then links the root of the smaller-rank tree to the root of the larger-rank tree (union by rank). This prevents the structure from degenerating into a long chain and keeps tree height bounded at O(log n) even before path compression.

Union by rank

  • Attach the shorter tree under the root of the taller one.
  • Only increment rank when two equal-rank trees merge.
  • Guarantees tree height stays O(log n) even without path compression.

Union by size

  • Attach the smaller group under the root of the larger group.
  • Track actual component size instead of an approximate height.
  • Equally effective; also lets you query component size in O(1).
Worked problemMediumLeetCode 547
Number of Provinces

You are given an n×n adjacency matrix where isConnected[i][j] = 1 means cities i and j are directly connected. A province is a maximal group of directly or indirectly connected cities. Return the total number of provinces.

Think it through

Restate

Count the number of connected components in an undirected graph given as an adjacency matrix.

Brute force

BFS or DFS from every unvisited node, marking components. Correct at O(n²) time (must read the full matrix), O(n) space for visited array. Nothing wrong with this for a static graph.

The waste

BFS/DFS explores the same node structure every call. With Union-Find we can process each edge exactly once and the data structure maintains component membership automatically — also O(n²) to read the matrix but with a smaller constant and trivially extensible if edges arrive dynamically later.

The move

Initialise Union-Find with n nodes (one per city), each in its own province. Scan the upper triangle of the matrix; whenever isConnected[i][j] == 1, call union(i, j). After all edges, count how many nodes are their own parent — each is a province root.

Invariant & cost

After every union, all members of the merged province share the same root. The province count equals the number of distinct roots. Reading the matrix is O(n²); each union/find is O(α(n)) ≈ O(1). Total: O(n²) time, O(n) space.

The code

python

def findCircleNum(isConnected):
    n = len(isConnected)
    parent = list(range(n))      # each city starts as its own province
    rank   = [0] * n

    def find(x):
        while parent[x] != x:
            parent[x] = parent[parent[x]]  # path halving (light compression)
            x = parent[x]
        return x

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx == ry:
            return
        if rank[rx] < rank[ry]:
            rx, ry = ry, rx         # ensure rx is the taller root
        parent[ry] = rx             # attach shorter under taller
        if rank[rx] == rank[ry]:
            rank[rx] += 1

    for i in range(n):
        for j in range(i + 1, n):   # upper triangle only
            if isConnected[i][j]:
                union(i, j)

    return sum(1 for i in range(n) if find(i) == i)
Narrate it to the interviewer

"Restating: I need to count connected components in a graph given as an adjacency matrix. I can do it with BFS per unvisited node in O(n²), and that's perfectly correct. But I'll show Union-Find instead, because it generalises naturally to dynamic edge arrival and costs the same asymptotically here. I initialise parent[i] = i for each city. Then I scan just the upper triangle of the matrix — since the graph is undirected — and union(i, j) whenever isConnected[i][j] is 1. After all edges, I count nodes where find(i) == i; those are province roots. Invariant: every node in the same province shares a single root. Time O(n²) to read the matrix, each union is near-O(1). Space O(n)."

Dry-run aloud: n=1 (single city, answer 1); a fully connected matrix (answer 1); an identity matrix (no edges, answer n).

02 Counting

Tracking connected components

One elegant extension of Union-Find is tracking the number of connected components as edges arrive, without a separate counting pass at the end. The trick is a single counter: start it at n (every node is its own component) and decrement it by 1 every time a union call actually merges two distinct components. If find(x) == find(y) before the union, the nodes are already connected — don't decrement.

This makes the component count available at any point during edge insertion, which is exactly what problems like making a network connected or number of operations to make network connected ask for.

The reusable shape

Any problem that hands you a list of edges and asks for the number of connected groups — or asks how many more edges are needed to fully connect something — follows this template: count = n; for each edge (u, v), if find(u) != find(v) then union(u, v); count -= 1. Return count. One loop, near-O(1) per edge.

03 Cycle detection

When union fails, a cycle closes ▸

Union-Find detects cycles in an undirected graph with a single elegant rule: process edges one at a time. For each edge (u, v), check find(u) == find(v). If the roots are the same, u and v are already connected by a path that doesn't use this edge — adding this edge creates a cycle. If the roots differ, the edge is a tree edge; call union(u, v) safely.

The brute-force equivalent is DFS after every edge insertion — O(V + E) per check. Union-Find does it in O(α(n)) per edge with no graph traversal at all. This is also exactly how Kruskal's MST algorithm works: sort edges by weight, then greedily add each edge if it doesn't form a cycle (i.e., if its endpoints are in different components).

Worked problemMediumLeetCode 684
Redundant Connection

You are given a list of edges that form an undirected graph that started as a tree and had exactly one extra edge added. Return the last edge in the list that, if removed, would restore the graph to a tree (i.e. the edge that closes the first cycle found when processing in order).

Think it through

Restate

Find the first edge whose endpoints are already in the same component — that is the redundant edge that created a cycle.

Brute force

For each candidate edge, remove it and check with DFS/BFS whether the remaining graph is still connected. O(E × (V + E)) — quadratic in the worst case, and awkward to code cleanly.

The waste

We are re-traversing the graph for every candidate. But we only need to know one thing per edge: were the two endpoints already connected before this edge was added? Union-Find answers that in O(α(n)).

The move

Process edges left to right. For each (u, v), call find(u) and find(v). If they share a root, this edge is the answer — return it immediately. Otherwise, union(u, v) and continue. The problem guarantees exactly one such edge exists.

Invariant & cost

Before processing edge i, the Union-Find reflects exactly the connectivity induced by edges 0 … i-1. So a same-root check tells us precisely whether this edge is redundant. Each edge costs O(α(n)). Total: O(E · α(n)) ≈ O(E) time, O(V) space.

The code

python

def findRedundantConnection(edges):
    n = len(edges)
    parent = list(range(n + 1))  # nodes are 1-indexed
    rank   = [0] * (n + 1)

    def find(x):
        while parent[x] != x:
            parent[x] = parent[parent[x]]  # path halving
            x = parent[x]
        return x

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx == ry:
            return False           # same component: cycle detected
        if rank[rx] < rank[ry]:
            rx, ry = ry, rx
        parent[ry] = rx
        if rank[rx] == rank[ry]:
            rank[rx] += 1
        return True

    for u, v in edges:
        if not union(u, v):        # union returned False → cycle
            return [u, v]

    return []                      # problem guarantees an answer exists
Narrate it to the interviewer

"Restating: I need the first edge in the list that connects two nodes already reachable from each other — that edge is the one that created a cycle. The brute force removes each candidate and re-runs BFS, costing O(E²) in the worst case. That's wasteful because I'm re-traversing a graph I've already partially explored. Instead, I process edges left to right with Union-Find. Before adding each edge (u, v), I check find(u) == find(v). If they share a root, they were already connected — this edge closes a cycle, so I return it. Otherwise I union them and move on. Invariant: at step i, the structure reflects only edges 0 through i-1, so the check is exact. Near-O(1) per edge, O(V) space."

Dry-run aloud: the minimal cycle case — three nodes in a triangle [[1,2],[2,3],[1,3]]; third edge (1,3) shares a root after the first two are union-ed. Also check self-loops: edge [u, u] — find(u)==find(u) immediately returns it as redundant.

What the interviewer asks next

"What about a directed graph?" — Union-Find is for undirected graphs only; directed cycle detection needs DFS with a recursion-stack coloring (white/grey/black). "What if there are multiple redundant edges?" — the problem guarantees one, but in general you'd collect all edges where find(u)==find(v) and return the last one in the input order. "How does Kruskal use this?" — sort edges by weight, iterate, and call union; the first E-V+1 accepted edges are the MST edges; any rejected edge would create a cycle so it's discarded.

04 Path compression

Flattening chains to near-O(1) ▸

Without optimisation, union-by-rank alone can still leave trees of depth O(log n). Path compression is the second trick that collapses those chains. When find(x) walks up to the root, it simultaneously re-attaches every node on the path directly to the root. On the next find call for any of those nodes, the walk is just one step.

There are two variants. Full path compression (recursive find) re-attaches every node on the path directly to the root in one call. Path halving (iterative) makes each node point to its grandparent: parent[x] = parent[parent[x]]; x = parent[x] — halves the chain length with a single pass and no recursion stack. Both yield the same O(α(n)) amortised bound.

Why the combined bound is O(α(n))

Union by rank alone gives O(log n) per operation. Path compression alone gives O(log n) amortised. Together, the two optimisations yield O(α(n)) amortised — where α is the inverse Ackermann function, which is ≤ 4 for any n that could fit in physical memory. In practice this is indistinguishable from O(1). The proof (by Tarjan) is non-trivial, but the implementation is just four lines of code.

05 Real application

Merging accounts

The power of Union-Find shines in problems where the "nodes" are not integers but arbitrary keys — like email addresses — and you must group them by shared identity. The challenge is mapping arbitrary keys to integer ids, then running standard Union-Find on those integers.

The classic problem: you have a list of accounts; each account has a name and a list of emails. Two accounts belong to the same person if they share at least one email. Merge all accounts belonging to the same person and return the consolidated list sorted by email. The graph edges are implicit: within each account, every email is connected to every other email (they all belong to the same account).

The approach has three phases:

  • Index emails: build a map from each unique email to an integer id. Also record the owner's name for each email id.
  • Union within each account: for each account, union all its email ids together. The first email in the account acts as the anchor — union every other email to it.
  • Collect by root: group email ids by their root in the Union-Find. For each group, sort the emails and prepend the owner name.
Worked problemHardLeetCode 721
Accounts Merge

Each account is a list where the first element is a name and the rest are email addresses. Accounts belong to the same person if they share any email. Merge all accounts belonging to the same person and return a sorted, deduplicated list of emails with the name prepended.

Think it through

Restate

Group accounts by shared email. Two accounts merge if their email sets overlap. Return merged groups sorted by email with the name prepended.

Brute force

For every pair of accounts, check if their email sets intersect; if so, merge them and repeat until no more merges happen. Correct but O(A² × E) per round, and you need multiple rounds until convergence — very slow for large inputs.

The waste

The pairwise scan re-examines already-merged accounts repeatedly. Union-Find processes each email exactly once and propagates group membership automatically.

The move — phase 1

Assign each unique email an integer id. Also store email_owner[id]. Union-Find runs on these integer ids.

The move — phase 2

For each account, call union(id_of_email_0, id_of_email_k) for every email k in that account. After all accounts, emails that share a root belong to the same person.

Invariant & cost

After phase 2, find(e) returns the same root for all emails of the same person. Group by root, sort each group, prepend name. Total: O(A⋅E⋅α(A⋅E)) time, O(A⋅E) space — effectively linear in the total number of emails.

The code

python

from collections import defaultdict

def accountsMerge(accounts):
    parent = {}
    owner  = {}

    def find(x):
        if parent[x] != x:
            parent[x] = find(parent[x])  # full path compression
        return parent[x]

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx != ry:
            parent[rx] = ry

    # phase 1: register each email
    for acc in accounts:
        name = acc[0]
        for email in acc[1:]:
            if email not in parent:
                parent[email] = email   # each email is its own root initially
            owner[email] = name

    # phase 2: union all emails within the same account
    for acc in accounts:
        anchor = acc[1]               # first email acts as group anchor
        for email in acc[2:]:
            union(anchor, email)

    # phase 3: collect emails by root, sort, prepend name
    groups = defaultdict(list)
    for email in parent:
        groups[find(email)].append(email)

    result = []
    for root, emails in groups.items():
        result.append([owner[root]] + sorted(emails))
    return result
Narrate it to the interviewer

"Restating: I need to merge accounts that share at least one email. The brute force checks every pair of accounts repeatedly until convergence — that's quadratic or worse. Union-Find handles this in three clean phases. Phase one: every unique email becomes its own node in a Union-Find that uses email strings as keys directly (Python dicts make this easy). Phase two: for each account, I union the first email with every other email in that account — after this, all emails in the same person's accounts share one root. Phase three: I group emails by their root, sort each group, and prepend the owner's name. Invariant: after phase two, same root means same person. Effectively linear in total emails."

Dry-run aloud: a single account with one email (no union needed, output is that account); two accounts with identical emails (they merge into one); two accounts with the same name but no shared email (they stay separate — name alone doesn't merge them).

What the interviewer asks next

"What if accounts have millions of emails?" — the approach scales; each email is touched once per account it appears in. "How would you also track component size?" — maintain a size array alongside parent; increment size[root] during union. "Does order of union calls matter?" — the final connectivity is the same regardless of union order; only the intermediate tree shapes differ.

Idea Classic online Union-Find: for each new land cell, start count++; check all four neighbours; if a neighbour is already land and in a different component, union and count--. Return count after each addition.

How to drill Union-Find problems

The template is always:
(1) declare parent and rank/size;
(2) write find with path compression;
(3) write union with rank/size balancing;
(4) process the input. For each drill, say out loud what the "nodes" are (integers, strings, coordinates), what the "edges" mean (shared attribute, adjacency), and what the answer extracts from the final structure (root count, cycle detection, component size). ★ Review

Summary & self-test

Union-Find is a specialised data structure for a specialised problem: maintaining disjoint sets under dynamic merges while answering "same group?" queries in effectively constant time. Memorise the template once, and the pattern repeats across every problem in this family — the only variation is what the nodes represent and what you do with the final component structure.

01Dynamic connectivityEdges arrive over time; connectivity must be answerable after each arrival. BFS/DFS is O(V+E) per query; Union-Find is O(α(n)) per query.
02find(x)Walk parent pointers to the root. Path compression re-attaches nodes directly to the root so future calls are one step.
03union(x, y)Link root of smaller tree under root of taller tree (by rank/size). Keeps tree depth at O(log n) before compression.
04Cycle detectionIf find(u)==find(v) before union(u,v), the edge (u,v) closes a cycle. This is the only test needed — no DFS required.
05Component countingStart counter at n; decrement only on a successful union (two distinct roots). Counter is always correct.
06String keysMap arbitrary identifiers (emails, names) to integer ids first, or use a dict as the parent store directly.
07O(α(n)) amortisedUnion by rank + path compression together. α(n) ≤ 4 for all realistic n. Treat it as O(1) in interview complexity analysis.
08Kruskal teaserSort edges by weight; greedily add each edge if union succeeds (no cycle). Accepted edges form the MST. Same Union-Find template, one extra sort.
