BST Patterns

A Binary Search Tree is not just a tree with some ordering rule bolted on — the invariant left < root < right runs through every node and turns dozens of hard problems into focused O(h) descents. Once you see how to exploit that ordering for validation, ranked selection, and range queries, you stop treating a BST like a generic tree and start reaching for its structure on sight.

00 The invariant

The ordering invariant — left < root < right

A BST stores one value per node with a strict rule: every value in a node's left subtree is smaller than the node, and every value in its right subtree is larger. That rule applies recursively at every level, not just between a parent and its immediate children.

That subtle point — every descendant, not just the direct child — is where most validation bugs live. A node can have a left child that is smaller than it, yet still violate the BST property if that same left child is larger than a grandparent further up. The correct model is a propagating window: each node inherits a valid range from its ancestors, and its value must land strictly inside that window.

Once you internalize this, three families of problems become straightforward:

  • Structural queries — is this tree a valid BST? Does a given value exist? — answered by a range-guided descent, O(h) time per query.
  • Ordered queries — what is the k-th smallest element? What is the closest value to a target? — answered by the fact that an inorder traversal of a BST yields a perfectly sorted sequence.
  • Ancestor queries — what is the lowest common ancestor of two nodes? — answered by comparing both values to the current root and deciding which subtree to descend into.
Why this pattern now

Trees appear in almost every SWE interview loop. Most candidates know DFS and BFS on generic trees; the BST pattern layer on top is what separates an average answer from a strong one. The moment you spot a sorted-tree structure, you should be reaching for an O(h) descent or an inorder traversal — not a full O(n) scan — and stating that trade-off out loud.

01 Validation

Validate with a propagating (min, max) window

The naive validation attempt compares each node only to its direct parent: left child must be less than parent, right child must be greater. This fails silently on the classic trap: a right child of the root that has a left child smaller than the root itself — a local check passes, but the BST invariant is broken at the global level.

The correct approach threads a valid range down the tree. Start at the root with the range (−∞, +∞). When you recurse into the left subtree of a node with value v, you tighten the upper bound: the new range is (current_min, v). When you recurse into the right subtree, you tighten the lower bound: the range becomes (v, current_max). At every node, check that the node's value falls strictly inside its inherited window. If any node violates its window, the tree is invalid — stop immediately.

This single recursive pass checks the whole tree in O(n) time and O(h) stack space, and it is correct because every ancestor's constraint is encoded in the inherited window.

The reusable shape

Pass bounds down, not up. Any problem that asks "does this subtree satisfy a constraint that depends on its ancestors" follows this template: thread the constraint into the recursive call as parameters. For BST validation the constraint is a numeric window; for other problems it might be a running max or a parity flag — but the shape is identical.

Worked problemMediumLeetCode 98
Validate Binary Search Tree

You are given the root of a binary tree. Determine whether it satisfies the BST property — that is, for every node, all values in its left subtree are strictly less than it and all values in its right subtree are strictly greater.

Think it through

Restate

Check whether the full tree, including all indirect descendants, obeys left < root < right at every node.

Brute force

For each node, collect all values in its left and right subtrees and check them — but that revisits nodes repeatedly. O(n²) in the worst case on a skewed tree.

The waste

We keep re-collecting the same subtree values. The constraint each node imposes can instead be encoded in the recursive call as a (min, max) window, so each node is visited exactly once.

The move

Recurse with a helper that carries a (lo, hi) window. At each node, check that lo < node.val < hi. Recurse left with the window (lo, node.val) and right with (node.val, hi). Start with lo = −∞, hi = +∞ at the root.

Invariant & cost

Every node receives the tightest possible window from all its ancestors. A violation is caught the instant we reach the offending node. Each node is visited once: O(n) time, O(h) space for the call stack (O(n) worst case on a skewed tree).

The code

python

def isValidBST(root):
    def check(node, lo, hi):
        if node is None:
            return True             # empty tree / leaf child is valid
        if node.val <= lo or node.val >= hi:
            return False            # violates inherited (lo, hi) window
        return (check(node.left,  lo,       node.val) and
                check(node.right, node.val, hi))

    return check(root, float('-inf'), float('inf'))
Narrate it to the interviewer

"Restating: I need to verify the BST property for every node, not just parent-child pairs. The naive route visits the whole subtree for each node — O(n²). The insight is to carry the constraint downward as a (lo, hi) window. I call a recursive helper that receives lo and hi bounds. If the node is null I return True. If its value falls outside (lo, hi) I return False immediately. When I recurse left I tighten the upper bound to the current value; when I recurse right I tighten the lower bound. Starting with (−∞, +∞) at the root ensures the first node gets no restriction. Each node is touched once, so this is O(n) time, O(h) space."

Dry-run aloud: a single node (trivially valid), a two-node tree where the root has a right child equal to the root (must be strictly less — return False), and the classic trap where a node is locally correct but globally wrong (e.g., node 12 in FIG. 1).

What the interviewer asks next

"What if values can repeat?" → The standard BST definition requires strict inequality; check with the interviewer, then adjust <= vs < accordingly. "Can you do it iteratively?" → Use a stack with (node, lo, hi) tuples and simulate the recursion. "What if the tree is huge and mostly invalid?" → The early-return on the first violation already short-circuits the traversal — the O(n) is a worst-case bound for a tree that turns out to be valid.

02 Inorder traversal

Inorder traversal yields a sorted sequence

There is a beautiful duality at the heart of BSTs: an inorder traversal (visit left subtree, then the current node, then right subtree) visits every node in strictly ascending order if and only if the tree is a valid BST. This is not a coincidence — it is a direct consequence of the ordering invariant applied recursively.

That duality turns any problem of the form "do something with the BST's elements in sorted order" into a straightforward inorder walk. You do not need to collect all values first; you can act on each one as you visit it, which is how you get early-exit patterns like kth-smallest (Section 03).

Equivalently, if an inorder traversal produces a sequence that is not strictly ascending, the tree is not a valid BST. This gives a second validation algorithm: run inorder, track the previous value, and fail if the current value is not greater than the previous.

Recursive inorder

  • Concise — three lines of recursion.
  • O(h) stack depth; risky on very skewed trees.
  • Easy to add an early-exit (e.g., stop at k-th).

Iterative inorder (stack)

  • Explicit stack instead of call stack.
  • Safer for production (no stack-overflow risk).
  • Needed when the interviewer says "no recursion".
Worked problemMediumLeetCode 230
Kth Smallest Element in a BST

Given the root of a BST and a positive integer k, return the k-th smallest value in the tree. You may assume k is always valid (1 ≤ k ≤ number of nodes).

Think it through

Restate

Find the k-th element in the sorted order of the BST's values without explicitly sorting.

Brute force

Collect all node values into a list, sort it, return result[k-1]. This is O(n log n) time and O(n) space — and it ignores the BST structure entirely.

The waste

Sorting an already-ordered structure is wasteful. An inorder traversal is the sorted sequence — we just need to count off k steps and stop.

The move

Run inorder traversal, decrementing a counter at each node visited. When the counter hits zero, the current node's value is the answer. Return immediately — do not continue traversing the rest of the tree.

Invariant & cost

Inorder visits the k-th smallest after exactly k steps. We touch at most k + h nodes before returning. O(k + h) time, O(h) space. For a balanced tree h = O(log n), making this O(k + log n).

The code

python

def kthSmallest(root, k):
    count = [k]          # mutable so the inner function can modify it
    result = [None]

    def inorder(node):
        if node is None or result[0] is not None:
            return          # already found — stop recursing
        inorder(node.left)
        count[0] -= 1
        if count[0] == 0:
            result[0] = node.val
            return
        inorder(node.right)

    inorder(root)
    return result[0]
Narrate it to the interviewer

"The brute force collects all values and sorts them — O(n log n) — but it ignores that inorder traversal of a BST is already sorted. So I run inorder traversal and count nodes as I visit them. When the count reaches k, I record that value and stop immediately — I don't need to see the rest of the tree. The cost is O(k + h) time — I descend to the leftmost leaf in O(h) steps and then count up k values — and O(h) space for the call stack. On a balanced tree that is O(k + log n)."

Dry-run aloud: k = 1 (should return the leftmost node), k = n (should return the rightmost), and a single-node tree where k = 1.

03 Kth smallest

Stop-early inorder for the k-th element

The FIG. 2 inorder traversal visits all n nodes. For the k-th smallest query, that is too much. The key observation is that inorder visits nodes in ascending order — so you only need to travel as far as the k-th node. Stop there, and the rest of the tree is irrelevant.

The challenge in a recursive implementation is signaling "stop" back up the call stack once the answer is found. One clean approach is to use a mutable counter and a mutable result slot (as in the worked problem above). An iterative approach using an explicit stack avoids the communication problem entirely: just break out of the loop when the counter hits zero.

The reusable shape

Any "find the k-th in order" query on a BST follows this template: inorder traversal with a counter that fires an early return at k. The same pattern handles "find the k-th largest" by running reverse inorder (right, root, left). The stop-early character is what makes this O(k + h) rather than O(n).

What the interviewer asks next

"What if the tree is frequently modified and you need many k-th-smallest queries?" → Augment each node with a left_size count (number of nodes in its left subtree). Then each query is a guided O(h) descent: at each node, compare k to left_size to decide whether to go left, return the current node, or recurse right with an adjusted k. Updates maintain left_size in O(h) per insert/delete. This is the classic "order-statistic tree" and worth mentioning as a follow-up even if you do not code it.

04 LCA & ranges

Lowest Common Ancestor and range queries

The Lowest Common Ancestor (LCA) of two nodes p and q is the deepest node that has both p and q as descendants (where a node is considered a descendant of itself). In a generic binary tree, finding the LCA requires a post-order traversal — O(n) time and O(h) space. In a BST, the ordering invariant lets you find the LCA in a single guided descent.

The key insight: at each node with value v, if both p and q are less than v, the LCA is in the left subtree. If both are greater, it is in the right subtree. If they split — one is less and one is greater, or one of them equals v — then the current node is the LCA. This is a simple comparison at each step, no backtracking needed.

Range queries

A range query asks: "list all values in the BST that lie between lo and hi (inclusive)." In a generic tree you would scan everything — O(n). In a BST, you can prune entire subtrees: if the current node's value is less than lo, its entire left subtree is also less than lo (skip it). If it is greater than hi, its entire right subtree is also greater (skip it). Only nodes that fall inside the range are visited fully.

The worst case is still O(n) when the range covers all values, but in practice range queries on a BST are O(k + h) where k is the number of values inside the range — a dramatic improvement when the range is tight.

Worked problemEasyLeetCode 235
Lowest Common Ancestor of a BST

Given the root of a BST and two nodes p and q that are guaranteed to exist in the tree, return their lowest common ancestor — the deepest node that is an ancestor of both.

Think it through

Restate

Find the deepest node that has p and q in its subtrees (or is one of them). Use the BST ordering to guide the descent.

Brute force

On a generic binary tree: do DFS, find the path from the root to each target, then find the last common node in those two paths. O(n) time, O(n) space.

The waste

For a BST we do not need two full path-finding traversals. The ordering tells us at every node whether both targets are to one side or whether they split — and splitting means we are at the LCA.

The move

Start at the root. If both p.val and q.val are less than the current node, recurse left. If both are greater, recurse right. Otherwise (they straddle or one equals the current node) the current node is the LCA — return it.

Invariant & cost

At each step we move exactly one level closer to the answer, or we have found it. We travel at most h levels. O(h) time, O(h) space recursive (O(1) with an iterative loop).

The code

python

def lowestCommonAncestor(root, p, q):
    node = root
    while node:
        if p.val < node.val and q.val < node.val:
            node = node.left   # both targets are in the left subtree
        elif p.val > node.val and q.val > node.val:
            node = node.right  # both targets are in the right subtree
        else:
            return node        # they split → this node is the LCA
    return None                # unreachable if p and q are guaranteed present
Narrate it to the interviewer

"On a generic tree the LCA needs O(n) to find paths to both nodes and trace back. But in a BST, I can decide at each node by comparing both target values. If both p and q are smaller than the current node, the LCA must be in the left subtree — I move there. If both are larger, I move right. The moment they split — one on each side, or the current node equals one of them — that is the LCA. I wrote this iteratively to use O(1) space instead of O(h) stack. The descent is at most h steps, so O(h) time, O(1) space."

Dry-run aloud: p and q are the same node (the node is its own ancestor — the else branch fires immediately); p is an ancestor of q (the comparison equals the ancestor's value, else branch fires); both in the left subtree; both in the right subtree.

05 Modification

Insert, delete, and why balance matters

Insert into a BST is a guided descent: compare the new value to the current root, go left if smaller, go right if larger, and place the new node when you hit a null pointer. Each step eliminates half the remaining tree on a balanced BST, giving O(log n) for a balanced tree and O(n) in the worst case for a degenerate (fully skewed) one.

Delete has three cases, in increasing complexity:

  • Leaf node — simply remove it; no structural repair needed.
  • One child — splice the node out by connecting its parent directly to its single child.
  • Two children — find the inorder successor (the smallest node in the right subtree), copy its value into the node to be deleted, then delete the inorder successor from the right subtree (which is the simpler one-child or leaf case).

Why balance matters

If you insert values in sorted order into a plain BST — say, 1, 2, 3, 4, 5 — the tree degenerates into a linked list. Every operation (search, insert, delete) is O(n) in this worst case, not O(log n). This is why production systems use self-balancing BSTs like AVL trees or Red-Black trees, which maintain a height of O(log n) through rotations. Python's sortedcontainers.SortedList and Java's TreeMap/TreeSet are backed by such structures.

For an interview, you do not need to implement rotations — but you should know when to say "a plain BST degenerates on sorted input; a balanced BST (like a Red-Black tree) maintains O(log n) guarantees." That sentence alone shows structural understanding.

Balanced BST — O(log n) ops

  • Search, insert, delete all O(log n).
  • Inorder gives sorted sequence in O(n).
  • Suitable for large dynamic sorted sets.

Degenerate BST — O(n) ops

  • Worst case: sorted or reverse-sorted insertions.
  • Height = n; every op degrades to linear scan.
  • Prevented by AVL / Red-Black / treap / skip list.
What the interviewer asks next

"Implement BST insert/delete" → Code insert first (simpler); for delete, handle the three cases explicitly. "Why not just use a sorted array?" → Sorted arrays give O(log n) search via binary search but O(n) insert/delete (shifting). BSTs give O(h) for all three, and in a balanced BST that is O(log n) — better for frequent modifications. "What is an inorder successor?" → The leftmost node in the right subtree — descend right once, then go as far left as possible.

06 Practice

Practice ladder

Work top to bottom. For every problem, state the brute force and its complexity first, then name the BST technique you will apply (range descent, inorder traversal, LCA split, etc.) and code it. Target the easy tier in under 15 minutes, the medium tier in under 30.

EasySearch in a BSTrange descent

Given the root of a BST and a target value, return the subtree rooted at the node containing that value, or null if it is not present.

Idea At each node compare the target to the node's value and descend left or right. O(h) time, O(1) space iteratively.

EasyLowest Common Ancestor of a BSTLCA split

Find the deepest node that is an ancestor of two given nodes in a BST.

Idea Exactly FIG. 4 — if both targets are left, go left; if both right, go right; else the current node is the split and the LCA. O(h) time.

EasyRange Sum of BSTrange pruning

Given a BST and two integers lo and hi, return the sum of all node values that lie within the inclusive range [lo, hi].

Idea DFS with pruning: skip the left subtree when the current node is <= lo; skip the right subtree when the current node is >= hi. O(n) worst case but O(k + h) when the range is tight.

EasyMinimum Absolute Difference in BSTinorder adjacency

Find the minimum difference between any two distinct node values in a BST of non-negative integers.

Idea Inorder traversal; track the previous node; minimum difference is always between adjacent nodes in sorted order. O(n) time, O(h) space.

MediumValidate Binary Search Treewindow descent

Determine whether a binary tree satisfies the full BST invariant — every ancestor constraint, not just parent-child.

Idea Exactly the worked problem in section 01 — thread (lo, hi) down the recursion. O(n) time.

MediumKth Smallest Element in a BSTstop-early inorder

Return the k-th smallest value in a BST without sorting all values first.

Idea Exactly FIG. 3 — inorder traversal with a counter; stop and return when counter reaches k. O(k + h) time.

MediumConvert Sorted Array to BSTdivide & conquer

Given a sorted array, build a height-balanced BST from it.

Idea The midpoint of the array is the root; recurse on the left half and the right half. By always picking the midpoint the tree stays balanced. O(n) time.

MediumInsert into a BSTguided descent

Insert a value into a BST and return the root of the modified tree, preserving the BST property.

Idea Descend left or right until you find a null child; insert there. Recursively: if node is null create it; else recurse into the correct subtree and connect the result. O(h) time.

MediumDelete Node in a BSTthree-case delete

Remove a node with a given value from a BST and return the root. If the value is not present, return the tree unchanged.

Idea Three cases: leaf (remove), one child (splice), two children (swap with inorder successor, then delete the successor). O(h) time.

MediumRecover Binary Search Treeinorder anomaly

Exactly two nodes in a BST have been swapped by mistake, breaking the ordering. Find and fix them without changing the tree's structure.

Idea Inorder traversal; find the two spots where the sequence decreases (the first decrease gives the first bad node, the last decrease gives the second). Swap their values. O(n) time, O(1) extra space with Morris traversal.

HardSerialize and Deserialize BSTpreorder + bounds

Design a way to convert a BST to a string and reconstruct it exactly. Optimize the encoding to be smaller than a generic binary tree serialization.

Idea Preorder traversal gives the root first; during deserialization, use the inherited (lo, hi) window to place each value in the correct subtree without null markers. Shorter encoding than generic trees because the structure is recoverable from the values alone.

HardCount of Smaller Numbers After SelfBST rank tracking

For each element in an array, count how many elements to its right are strictly smaller. Return all counts.

Idea Process right to left; insert each value into a BST augmented with left-subtree sizes. The count of smaller elements equals the rank of the inserted value, accumulated during the descent. O(n log n) average on a balanced BST.

How to drill BST problems effectively

For each problem, follow this five-beat rhythm out loud:
(1) name the brute force and its complexity;
(2) identify which BST property you will exploit (ordering invariant, inorder = sorted, or LCA split);
(3) state what you will pass down or track (bounds, a counter, a previous value);
(4) code the solution;
(5) dry-run the edge cases — an empty tree, a single node, a fully left-skewed tree (degenerate case), and k = 1 or k = n for ranked queries. Practicing on a shared editor with no autocomplete mirrors the real interview environment.

★ Review

Summary & self-test

The BST pattern family compresses to three moves: pass bounds down for validation and structural queries; run inorder to exploit sorted order for ranked or sequential queries; compare both targets at the current node for ancestor queries. Every other BST problem is a variant of these three. The cost of each is O(h) or O(k + h), not O(n) — and naming that distinction out loud is the signal interviewers look for.
