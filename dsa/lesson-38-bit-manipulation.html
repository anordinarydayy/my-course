Bit Manipulation

Integers are just sequences of bits, and the right bitwise operation can replace an entire loop, a hash set, or even a conditional branch. XOR tricks, masking, and bit counting are the backbone of a small but high-signal category of interview questions — the ones where the intended answer is a one-liner that looks like magic until you see why it works.

00 Foundation

Bit basics: the four operators

Every integer in memory is a row of bits. The four bitwise operators work on those bits in parallel, one column at a time, with no carry. That is what makes them fast and what makes their results sometimes surprising.

  • AND (&) — a bit is 1 only when both inputs are 1. Use it to mask (isolate a subset of bits) or to test a specific bit.
  • OR (|) — a bit is 1 when at least one input is 1. Use it to set a bit unconditionally.
  • XOR (^) — a bit is 1 when the inputs differ. XOR is the toggle operator: it flips whichever bits differ between the two operands. Its crucial identity is a ^ a = 0 and a ^ 0 = a — pairs cancel.
  • Left/right shift (<<, >>) — slide all bits left or right by k positions, equivalent to multiplying or floor-dividing by 2k.

Single-bit operations on bit i

The mask 1 << i has exactly one 1-bit at position i. Four idioms follow directly:

  • Test bit i: (n >> i) & 1 — is the i-th bit set?
  • Set bit i: n | (1 << i) — force that bit to 1.
  • Clear bit i: n & ~(1 << i) — force that bit to 0 (the bitwise NOT flips the mask so everything is 1 except position i).
  • Toggle bit i: n ^ (1 << i) — flip that bit: 0 becomes 1, 1 becomes 0.
Why interviews test bit manipulation

Bit tricks are not used everywhere, but when they fit they reduce O(n) hash-set solutions to O(1) extra space, or collapse counting loops into a single expression. The question set is small and well-defined, so a candidate who has genuinely internalized the identities can answer in seconds. That signal-to-noise ratio is why recruiters keep reaching for them.

01 XOR tricks

Paired values cancel; the loner survives

XOR has three properties that combine into a superpower. First, a ^ a = 0 — a value XORed with itself disappears. Second, a ^ 0 = a — XOR with zero is the identity. Third, XOR is commutative and associative, so the order of operations does not matter. Together these mean: if you XOR a whole list of numbers, every value that appears an even number of times cancels, and what is left is the XOR of all the values that appear an odd number of times.

The classic application is finding the single element in a list where every other element appears exactly twice. The brute-force approach uses a hash map to count occurrences — O(n) time and O(n) space. XOR does the same work with O(1) extra space: fold the entire list with XOR, and the paired duplicates vanish, leaving the loner intact.

The reusable shape

Any time a problem says "every element appears twice except one" or "find the missing number from 0..n", reach for XOR. For the missing-number variant, XOR all indices 0..n with all values in the array — the ones that pair cancel and the missing index survives.

Worked problemEasyLeetCode 136
Single Number

You are given a non-empty array of integers where every value appears exactly twice except for one element that appears exactly once. Return that singleton. Your solution must run in O(n) time and use O(1) extra space.

Think it through

Restate

All pairs cancel; I need the one survivor. Extra space must be constant — no counting dictionary or set.

Brute force

Build a hash map of counts; scan for the key whose value is 1. Correct, but O(n) space — the constraint forbids it.

The waste

The hash map is storing counts, but I do not need individual counts — I only need to know whether a value has been seen an even or odd number of times. XOR tracks exactly that parity, for all values simultaneously, in a single integer accumulator.

The move

Initialize acc = 0. XOR every element into acc. When a value appears for the first time it enters acc; when it appears again it cancels itself out (v ^ v = 0). After the whole array, only the singleton remains because it never paired up.

Invariant & cost

After processing index k, acc equals the XOR of all elements seen so far that have appeared an odd number of times. At the end, exactly one element qualifies. O(n) time, O(1) space.

The code

python

def singleNumber(nums):
    acc = 0
    for v in nums:
        acc ^= v    # paired values cancel: v ^ v = 0; loner survives
    return acc
Narrate it to the interviewer

"The naive answer uses a hash map to count occurrences, but that's O(n) space and the constraint says O(1). The insight is that I don't need the counts — I only need parity. XOR of a value with itself is zero, and XOR with zero is the identity, so folding the array with XOR makes every pair vanish and leaves the singleton. One pass, one integer accumulator — O(n) time, O(1) space."

Dry-run aloud: single-element array (no pairs, answer is that element); all elements the same except one (confirm pairs cancel in pairs, not just with their neighbors — order doesn't matter due to commutativity).

What the interviewer asks next

"What if each element appears three times instead of two, except one?" That breaks the XOR trick because three is odd, not even. The standard fix uses a 32-bit counter per bit position: count how many times each bit is set across all numbers, take each count modulo 3, and reconstruct — O(32n) = O(n) time, O(1) space. "Can you do it without any arithmetic at all?" That requires bit-counting registers (the general version), which is the same idea.

02 Counting set bits

Brian Kernighan's trick: strip the lowest set bit

The brute-force way to count the 1-bits in an integer is to loop over all 32 positions, test each one with (n >> i) & 1, and sum. That is always 32 iterations regardless of the actual value. Brian Kernighan's observation cuts the work to only as many iterations as there are 1-bits in the number.

The key identity is n & (n - 1). Subtracting 1 from n flips the lowest set bit to 0 and sets all the lower bits to 1. ANDing with n then clears exactly those positions, leaving a number with exactly one fewer 1-bit than n. Repeating until the result is zero gives a loop that runs exactly popcount(n) times. For sparse numbers (few 1-bits) this is dramatically faster than the 32-step approach.

Naive 32-pass loop

  • Always 32 iterations (or 64 for 64-bit numbers).
  • Simple to remember: right-shift and AND with 1 each time.
  • Predictable cost regardless of the value.

Kernighan's n & (n-1)

  • Iterations = number of set bits, which can be much fewer than 32.
  • Works on any integer size; the pattern is the same.
  • Doubly useful: the n & (n-1) == 0 check tests power-of-two in O(1).
Worked problemEasyLeetCode 191
Number of 1 Bits

Given a 32-bit unsigned integer, return the number of 1-bits (also called the Hamming weight or popcount). For example, the number 11 is 0...01011 in binary and has three 1-bits.

Think it through

Restate

Count the set bits in a 32-bit value. I need the Hamming weight — how many positions carry a 1.

Brute force

Loop 32 times: right-shift by i, AND with 1, accumulate. That is always 32 iterations — O(32) = O(1) in theory, but every iteration is wasted work when the high bits are zero.

The waste

Scanning positions that are already 0 contributes nothing. I want to hop directly from one set bit to the next and stop as soon as no bits remain.

The move

Use Kernighan's identity: n & (n - 1) clears the lowest set bit. Loop while n != 0, apply n &= (n - 1), and increment a counter each time. The loop runs exactly as many times as there are 1-bits — no wasted iterations over zero-bit positions.

Invariant & cost

After each iteration the number of set bits in n has decreased by exactly one, and the counter has increased by one. When n == 0 the counter holds the original popcount. O(k) time where k is the popcount — at most O(32) = O(1), and often much less. O(1) space.

The code

python

def hammingWeight(n):
    count = 0
    while n:
        n &= n - 1    # strip the lowest set bit
        count += 1
    return count
Narrate it to the interviewer

"I want to count the 1-bits. The obvious loop tests all 32 positions, but that wastes iterations on zero bits. Kernighan's identity tells me that n & (n-1) clears exactly the lowest set bit — subtracting 1 turns that bit off and lights up all the lower ones, then AND with the original turns those lower bits off too. So I loop while n is nonzero, apply that one operation, and increment a counter. The loop runs exactly as many times as there are set bits. O(k) time, O(1) space, and k is at most 32 for a 32-bit integer."

Dry-run aloud: n = 0 (zero iterations, returns 0); n = all ones (32 iterations); n = a power of two (exactly one iteration).

What the interviewer asks next

"How would you compute this for all numbers from 0 to n at once?" That is the Counting Bits problem (LeetCode 338), covered in the next section. "What is the Hamming distance between two numbers?" XOR them to get a mask of differing bits, then popcount the mask — two operations. "Is there a built-in?" Python has bin(n).count('1'); mention it but still explain the algorithmic approach because that is what is being tested.

03 DP on bits

Building a popcount table with a recurrence

Now the problem scales up: instead of counting bits in one integer, produce an array ans[0..n] where ans[i] is the popcount of i. The brute force runs Kernighan's algorithm on each of the n+1 numbers — O(n log n) total. But there is a recurrence that fills the table in exactly one pass.

The key observation: right-shifting a number by 1 drops the least-significant bit, so i >> 1 has either the same number of set bits as i (if bit 0 of i was 0) or one fewer (if bit 0 was 1). In both cases: dp[i] = dp[i >> 1] + (i & 1). When you fill the table from 0 upward, dp[i >> 1] is always already computed because i >> 1 < i. One pass, O(1) extra work per entry, O(n) time and space total.

Worked problemEasyLeetCode 338
Counting Bits

Return an array of length n+1 where the element at index i equals the number of 1-bits in the binary representation of i. Your solution must run in O(n) time and not use any built-in popcount function.

Think it through

Restate

Fill an array of n+1 popcounts in one linear pass — no repeated work, no library functions.

Brute force

For each i in 0..n, run Kernighan's loop to count bits. O(k) per number where k ≤ 32, so O(n log n) total — or O(32n) for 32-bit integers. Correct but not the O(n) the problem asks for.

The waste

Kernighan's loop recomputes the bit count of each number from scratch. But most numbers are almost identical to a smaller number: drop the lowest bit and you get a number whose popcount I have already computed. I should reuse that stored result.

The move

The recurrence dp[i] = dp[i >> 1] + (i & 1) does exactly this. Right-shifting i by 1 gives a number strictly smaller than i (for i > 0), so its entry is already in the table. Adding the least-significant bit of i (0 or 1) accounts for whether i is odd or even. Fill left to right; every lookup hits a computed cell.

Invariant & cost

At the time dp[i] is computed, dp[i >> 1] is already correct because i >> 1 < i for all positive i. Each entry is computed in O(1). The result array itself is the only extra space used. O(n) time, O(n) space (the output array).

The code

python

def countBits(n):
    dp = [0] * (n + 1)          # dp[0] = 0 by default
    for i in range(1, n + 1):
        dp[i] = dp[i >> 1] + (i & 1)   # drop LSB + check if LSB was set
    return dp
Narrate it to the interviewer

"The brute force runs Kernighan's bit-count on each of the n+1 numbers, giving O(n log n). To get O(n) I use a DP recurrence: dp[i] = dp[i >> 1] + (i & 1). Right-shifting by one drops the least-significant bit and produces a smaller index that is already computed. Adding the LSB accounts for whether i is odd. I fill the table left to right in one pass — O(n) time and O(n) space for the result array. The base case dp[0] = 0 is set by initialization."

Dry-run aloud: n = 0 (returns [0]); n = 1 (returns [0, 1]); verify dp[6] = dp[3] + 0 = dp[1] + 1 + 0 = 2, which matches 6 = 0b110 having two set bits.

04 Bitmask enumeration

Every subset as an integer

A set of n elements has exactly 2n subsets. Each subset can be represented as an n-bit integer where bit i being 1 means element i is included. This is the bitmask enumeration pattern, and it converts subset problems into integer iteration.

To enumerate all subsets of an n-element collection: loop mask from 0 to (1 << n) - 1. For each mask, loop i from 0 to n-1 and check whether bit i is set using (mask >> i) & 1. The loop over all 2n masks times n-bits inspection is O(n · 2n) — exponential, but unavoidable for exhaustive subset enumeration. What the bitmask buys you is a clean, compact representation and fast set operations: union is OR, intersection is AND, complement is NOT, difference is AND-NOT.

When to reach for bitmask enumeration

The signal is small n: if n ≤ 20 and the problem asks you to try all subsets or all partitions of a small set, a bitmask loop is exactly right. Classic examples include the Traveling Salesman Problem on small graphs, tiling problems, and bit DP where the state is a subset. At n = 20, 220 ≈ one million — workable in a few seconds. At n = 30, 230 ≈ one billion — too slow without further pruning.

Worked problemMediumLeetCode 371
Sum of Two Integers

Add two integers a and b without using the + or - operators at any point in your solution. The inputs may be negative.

Think it through

Restate

Compute a + b using only bitwise operators and bit shifts. This forces me to think about how binary addition actually works in hardware.

Brute force

There is no brute force here — the constraint removes the obvious tool. The question is entirely about understanding binary addition.

The waste

Binary addition works in two pieces: XOR gives the sum bits that do not carry, and AND shifted left gives the carry bits. The waste to avoid is trying to handle all bit positions at once — iterating one round of XOR + carry at a time converges because each carry shifts left, so eventually there is no carry left.

The move

Compute carry = (a & b) << 1 (positions where both bits are 1, shifted left because a carry propagates to the next position) and a = a ^ b (sum without carry). Repeat with b = carry until carry is zero. In Python, integers are arbitrary precision and can grow negative carry bits forever, so we must mask to 32 bits: work within mask = 0xFFFFFFFF.

Invariant & cost

Each iteration, at least one of carry's set bits is one position higher than before (or the count decreases). The loop terminates in at most 32 iterations for 32-bit integers. O(1) time, O(1) space.

The code

python

def getSum(a, b):
    mask = 0xFFFFFFFF    # 32-bit mask (Python ints are unbounded)
    while b & mask:
        carry = (a & b) << 1
        a = a ^ b
        b = carry
    # if bit 31 is set, the 32-bit result is negative — sign-extend
    if (a >> 31) & 1:
        return ~(a ^ mask)
    return a & mask
Narrate it to the interviewer

"Binary addition splits into two parts: the XOR of a and b gives all the bits that sum without carrying, and the AND of a and b shifted left gives the carry into the next position. I repeat — new a = XOR, new b = carry — until the carry is zero. In Python I need to mask to 32 bits because Python integers are unbounded and carry can grow without limit. After the loop, if the 31st bit is set I sign-extend to get the correct negative integer. O(1) time and space — the loop runs at most 32 times."

Dry-run aloud: a = 0, b = 5 (answer 5, no carry ever); a = -1, b = 1 (answer 0, carry propagates until all bits clear); a = b = 1 (sum = 2, one carry round).

05 Power checks & lowest bit

Two identities that appear constantly

Two more identities show up in so many bit-manipulation problems that they deserve their own section.

Power-of-two check: n & (n - 1) == 0

A power of two has exactly one set bit. Subtracting 1 flips that bit to 0 and turns all lower bits on. ANDing the two results zeros everything — so n & (n - 1) == 0 if and only if n is a power of two (and n > 0). This is O(1), no loop needed.

Isolate the lowest set bit: x & -x

In two's complement, -x is the bitwise NOT of x plus one. The NOT flips all bits; adding one then ripples through the trailing zeros and flips the lowest set bit back on, leaving all higher bits inverted. ANDing the original x with -x keeps only the lowest set bit and clears everything else. This one-liner extracts the LSB and is the core of Fenwick-tree (BIT) implementations.

Putting it all together: the bit-manipulation toolkit

Six idioms cover the vast majority of interview bit problems:
(1) test/set/clear/toggle a bit with masks;
(2) XOR folding to cancel pairs;
(3) n & (n-1) to strip the lowest set bit and count bits;
(4) dp[i] = dp[i >> 1] + (i & 1) for a linear popcount table;
(5) bitmask enumeration over subsets when n is small;
(6) n & -n to isolate the lowest set bit for Fenwick trees and similar. Learning these six patterns — not just memorizing but understanding why each one works — is what allows you to reconstruct any bit trick from scratch under pressure.

Practice ladder

Return the Hamming weight (count of set bits) of a 32-bit unsigned integer.

Idea Loop: n &= (n-1) strips one set bit per iteration; count how many rounds it takes.

EasyCounting BitsDP on bits

Return an array where entry i holds the popcount of i, for i from 0 to n.

Idea dp[i] = dp[i>>1] + (i&1); fill left to right in O(n).

EasyPower of Twon & (n-1)

Decide whether a given integer is a power of two, in O(1) time.

Idea n > 0 and n & (n-1) == 0 iff n is a power of two — a single expression.

EasyMissing NumberXOR folding

An array contains n distinct integers drawn from 0..n with exactly one missing. Find the missing value in O(n) time and O(1) space.

Idea XOR all indices 0..n with all values; paired indices and values cancel, the missing index survives.

EasyReverse Bitsbit construction

Reverse the 32 bits of a given unsigned integer and return the result as an unsigned 32-bit integer.

Idea Loop 32 times: shift the result left, OR in the LSB of n, then right-shift n.

MediumSum of Two IntegersXOR + carry

Add two integers a and b without using the + or - operators anywhere in your code.

Idea XOR gives the no-carry sum bits; AND shifted left gives the carry; iterate until carry is zero. Mask to 32 bits in Python.

MediumSingle Number IIbit counting mod 3

Every element appears three times except one which appears once. Return the singleton in O(n) time and O(1) space.

Idea Count each bit position across all numbers, take count modulo 3; reconstruct the result bit by bit.

MediumSingle Number IIIXOR + partition

Two elements appear once; all others appear twice. Return both singletons in O(n) time and O(1) space.

Idea XOR all elements to get xor_ab = a ^ b; find any set bit in xor_ab, use it to partition numbers into two groups, XOR each group to recover a and b separately.

MediumHamming DistanceXOR + popcount

The Hamming distance between two integers is the count of bit positions where they differ. Compute it.

Idea XOR the two integers to mark differing positions; popcount the result via Kernighan's n & (n-1) loop.

MediumSubsetsbitmask enumeration

Given a set of unique integers of size n (n ≤ 12), return all 2n subsets.

Idea Loop mask from 0 to 2^n - 1; for each mask collect elements whose index bit is set — O(n * 2^n) total.

HardMaximum XOR of Two Numbers in an Arraytrie + greedy bits

Find the maximum XOR value achievable from any pair of elements in an integer array.

Idea Insert all numbers into a binary trie; for each number greedily choose the opposite bit at each level of the trie to maximize XOR — O(32n) = O(n).

How to drill bit manipulation

Before writing any code, draw the bits on paper. State which identity you are using and why it applies. Narrate the brute-force solution first, then explain the bit trick as an optimization — interviewers want to see the reasoning chain, not just the one-liner. Dry-run every solution on zero, one, the maximum value, a power of two, and an all-ones value (e.g., 0xFFFFFFFF). The bugs in bit code almost always live at the boundaries: off-by-one shifts, wrong mask size, and sign-extension in Python where integers are unbounded.

★ Review

Summary & self-test

Bit manipulation problems are compact by design: the gap between the naive solution and the intended one is almost always a single identity. The whole pattern collapses to knowing six idioms cold — test/set/clear/toggle with masks; XOR folding to cancel pairs; Kernighan's n & (n-1) to strip bits and count them; the linear DP recurrence for popcount tables; bitmask enumeration for small-n subset problems; and x & -x to isolate the lowest set bit.
