Fast & Slow Pointers

Run two pointers at different speeds through a linked list and their relative motion reveals hidden structure — whether the list loops, where the loop starts, and what node sits at the midpoint — all without any extra memory.

Original study notes. Written from scratch for coding-interview prep. Problem names below are bare identifiers only; every description, diagram, and line of code is original. Contains three interactive diagrams (press Play or step through with Next) and three fully worked LeetCode problems — each with the reasoning chain, the code, and a script for narrating it to the interviewer.
00 The idea

What problem does it solve?

A linked list gives you no random access and no known length. When you need to answer questions like "does this list loop back?", "which node is the midpoint?", or "where does a cycle begin?" the obvious answer is to walk the list once and keep a record — a hash set of every node address you visit. Correct, but that is O(n) extra space, and the technique is mostly bookkeeping rather than insight.

The fast & slow pointer pattern, sometimes called Floyd's tortoise and hare, removes the set entirely. Send one pointer (the tortoise, slow) forward one node per tick and a second pointer (the hare, fast) two nodes per tick. Their relative speed is the trick: inside a cycle the hare laps the tortoise and they must eventually share a node; in a straight list the hare simply falls off the end. No memory beyond the two pointers is needed — O(1) space.

The same muscle generalises further. Advance the fast pointer by n nodes before both move together, and slow will be sitting exactly n positions behind when fast reaches the tail. Or just let fast race ahead and when it runs out, slow is at the midpoint. Three different questions, one mechanical technique.

When to reach for it

The pattern is the right tool whenever you see: a singly linked list with a question about a cycle, the midpoint, or the n-th from the end; a sequence where "have I visited this state before?" is the core question; or any problem that says O(1) extra space on a structure you can only traverse forward. The tell is that the brute force wants to memorise every position it has seen.

01 Cycle detection

Does the list loop? — tortoise & hare ▸

The brute force stores every visited node in a hash set. When a node you are about to visit is already in the set, you have found a cycle — O(n) time, O(n) space. The set is the bottleneck.

The insight: if a cycle exists, the fast pointer will eventually lap the slow one inside the loop. Picture two runners on a circular track — the faster one always catches up. Inside the cycle the gap between fast and slow shrinks by exactly one node each tick (fast gains two, slow gains one), so after at most c ticks (where c is the cycle length) they share a node. If the list is acyclic, fast simply falls off the tail into None and you return false.

The invariant: if a meeting occurs it must be inside a cycle, because that is the only structure that prevents fast from running off the end. Conversely if there is a cycle, fast's lapping guarantees a meeting. Detection in O(n) time, O(1) space.

Fast & slow (O(1) space)

  • O(n) time — at most n steps before fast exits or they meet.
  • O(1) space — only two pointer variables regardless of list length.
  • Cannot tell you where the cycle starts — that needs phase 2 (next section).

Hash-set approach

  • O(n) time — walk once and query the set.
  • O(n) space — one entry per visited node.
  • Naturally gives you the entry node on first revisit — simpler to read, but uses memory.
Worked problemEasyLeetCode 141
Linked List Cycle

Given the head of a singly linked list, decide whether the list contains a cycle. Return true if any node's next pointer eventually points to a previously visited node, false otherwise. Use O(1) extra memory.

Think it through

Restate

I need a yes/no answer about whether the list loops, and I cannot use extra memory proportional to the list's length.

Brute force

Walk the list and add each node to a hash set; if a node is already in the set a cycle exists. O(n) time, O(n) space. Space is the problem.

The waste

The hash set is just recording "have I been here?" The fast pointer already encodes that signal through speed: if the structure loops, it will catch the slow pointer inside the loop without storing anything.

The move

Start both slow and fast at the head. Each tick: slow = slow.next, fast = fast.next.next. Guard fast's advance with fast and fast.next — if either is None, the list ends and there is no cycle. If slow is fast at any point, return true.

Invariant & cost

Inside a cycle the gap between fast and slow shrinks by one per tick, so they meet in at most c ticks (cycle length). Overall at most n steps before one outcome fires: O(n) time, O(1) space.

The code

python

def hasCycle(head):
    slow = fast = head
    while fast and fast.next:   # fast falls off the end → no cycle
        slow = slow.next
        fast = fast.next.next
        if slow is fast:          # met inside the loop
            return True
    return False
Narrate it to the interviewer

"Restating: does this linked list cycle back? The constraint is O(1) extra space, so I cannot use a hash set. My approach is fast and slow pointers — the tortoise and hare. Both start at the head; each tick slow moves one node and fast moves two. If fast ever hits null, the list is finite and there is no cycle. If slow and fast land on the same node, a meeting inside a cycle is the only possible explanation. The invariant is that the gap between them shrinks by exactly one per tick inside any loop, so they must meet in at most c steps where c is the cycle length. Overall O(n) time, O(1) space."

Dry-run aloud: empty list (fast is None immediately, return false); single node with no self-loop (fast.next is None, return false); single node pointing to itself (first tick slow == fast, return true).

02 Cycle start

Finding where the loop begins ▸

Detection tells you a cycle exists but not where it starts. The brute approach — hash set again — gives the entry for free (the first revisited node), at O(n) space. Staying in O(1) space requires a small argument about distances.

Let a = the number of nodes from the head to the cycle entry, and c = the length of the cycle. When slow and fast first meet, slow has traveled some distance a + b inside the cycle (where b is the distance from entry to meeting point). Fast has traveled a + b + k·c for some integer k because it has looped at least once. Since fast always covers exactly twice slow's distance: 2(a + b) = a + b + k·c, which simplifies to a = k·c − b. That means the distance from the head to the entry (a) equals the distance from the meeting point forward to the entry, modulo c. So: reset one pointer to the head, keep the other at the meeting point, and advance both one step at a time — they arrive at the entry simultaneously.

Why the math works

The key equation is a = k·c − b. The distance from head to entry (a) equals some whole number of cycle lengths minus the distance already walked inside the cycle. Walking a steps from the meeting point puts you exactly at the entry — either on the first loop or after wrapping around — and walking a steps from the head puts you there by definition. Both pointers arrive together.

Worked problemMediumLeetCode 142
Linked List Cycle II

Given the head of a singly linked list that may contain a cycle, return the node where the cycle begins. If there is no cycle return None. Do not modify the list and use O(1) extra memory.

Think it through

Restate

Detect the cycle and, if one exists, pinpoint the exact node at which the tail links back — all without extra memory.

Brute force

Walk the list and store each node in a hash set; the first node encountered that is already in the set is the entry. O(n) time, O(n) space — the space budget is violated.

The waste

The hash set is doing nothing that the two-pointer mechanism can't derive from the meeting-point distance argument. Once fast and slow have met, the distances encode exactly where the entry is.

The move — phase 1 (detect)

Run slow += 1, fast += 2 until fast exits (None) or they share a node. If fast exits, return None.

The move — phase 2 (locate)

Reset slow to the head; keep fast at the meeting point. Advance both one node at a time. Where they next share a node is the cycle entry — provably correct by the distance argument above.

Invariant & cost

Phase 1 takes at most n steps; phase 2 takes at most a ≤ n steps. Combined: O(n) time, O(1) space.

The code

python

def detectCycle(head):
    slow = fast = head
    # phase 1 — find meeting point inside cycle
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            break
    else:
        return None               # fast exited → no cycle
    # phase 2 — walk from head and from meeting point at speed 1
    slow = head
    while slow is not fast:
        slow = slow.next
        fast = fast.next
    return slow                    # the cycle entry node
Narrate it to the interviewer

"I need the entry node of the cycle in O(1) space. Hash set gives it trivially but at O(n) space, so I'll use two-phase Floyd's. Phase one: slow moves one, fast moves two — if fast hits null there is no cycle. When they meet, I know a cycle exists and the meeting point encodes the entry via the distance equation: the head-to-entry distance equals the meeting-to-entry distance mod the cycle length. Phase two: reset slow to the head, keep fast at the meeting node, walk both one step at a time — they converge on the entry. Invariant: the distance formula is always satisfied, so the meeting in phase 2 is always the entry. Overall O(n) time, O(1) space."

Dry-run aloud: null head (return None immediately); a self-loop at the head (a = 0, both pointers meet at the head right away in phase 2 — returns head, which is correct); a tail that loops back to itself only (single-node cycle at the very end).

03 Midpoint

Middle of the list in one pass ▸

Finding the middle node of a list without knowing its length ahead of time: the brute force counts the length with one full pass, then walks length / 2 steps in a second pass — O(n) time, O(1) space, but two passes. Can you do it in one?

Yes. The same fast & slow trick with a slightly different stopping condition: advance fast two nodes and slow one node per tick, but stop when fast reaches the end of the list (fast is None or fast.next is None). At that moment fast has covered roughly twice as much ground as slow, which means slow sits at the halfway mark — the midpoint. One pass, one pair of pointers.

For a list with an odd number of nodes slow lands on the exact middle. For an even count slow lands on the second of the two middle nodes. Many problems accept either convention; when one is required (e.g., reorder list wants the first middle) use the condition fast.next and fast.next.next so fast stops one tick earlier.

Worked problemEasyLeetCode 876
Middle of the Linked List

Given the head of a non-empty singly linked list, return the middle node. If two middle nodes exist (even-length list) return the second one. Do it in a single pass without counting the length first.

Think it through

Restate

Return the node at position n // 2 (0-indexed from the head) in a single traversal, without knowing n upfront.

Brute force

Traverse once counting nodes, then walk length // 2 steps from the head a second time. O(n) time, O(1) space, but two passes. The problem implicitly asks for one.

The waste

The first counting pass is redundant. Slow's position relative to fast is already a real-time measurement of how far we are through the list — no separate count needed.

The move

Start both at the head. While fast and fast.next are not None, advance slow one node and fast two. When the loop exits, slow is the middle node.

Invariant & cost

At all times slow is at roughly half the index of fast. When fast reaches the tail, that ratio pins slow at the midpoint. O(n) time, O(1) space, one pass.

The code

python

def middleNode(head):
    slow = fast = head
    while fast and fast.next:   # fast exits when the list ends
        slow = slow.next
        fast = fast.next.next
    return slow                  # slow is now the midpoint
Narrate it to the interviewer

"The naive approach counts the length and then does a second walk — two passes, O(1) space. I can get to O(1) space and one pass with fast and slow. Both start at the head; fast moves two nodes per tick and slow moves one. When fast can no longer advance — either it is on the last node or its next is null — slow has covered exactly half the list and sits at the middle. For an even-length list this lands on the second of the two middle nodes, which matches the problem's spec. One pass, O(n) time, O(1) space."

Dry-run aloud: a single node — fast is non-null but fast.next is null, loop does not execute, return head (correct); a two-node list — one tick moves slow to the second node while fast tries to move two steps but exits, returning the second node (correct second-middle behaviour).

04 Fixed gap

Nth node from the end

A close relative of the midpoint trick: find (or delete) the n-th node counting from the tail. Without knowing the list length the brute force counts first then walks back — two passes. With fast and slow you can do it in one by creating a fixed n-node gap between the two pointers.

Advance fast by n steps from the head with slow stationary. Once the gap is established, move both one node at a time until fast.next is None. At that point slow is sitting on the (n + 1)-th node from the end — the node just before the one you want to remove (or slow.next is the target). This collapses two passes into one.

The reusable shape

The fixed-gap technique is:
(1) advance fast by n steps;
(2) if fast is already None the list has exactly n nodes and the head is the target;
(3) otherwise advance both until fast.next is None — slow is one behind the target. The pattern generalises to "stop fast at a landmark, then move together" — you will see this in palindrome checks and reorder-list problems.

Worked problemMediumLeetCode 19
Remove Nth Node From End of List

Given the head of a singly linked list and an integer n, remove the node that is n positions from the tail and return the updated head. Do it in a single pass.

Think it through

Restate

Delete the node at offset length − n from the head (1-indexed from the tail), one pass only.

Brute force

Count the length L with one traversal, then walk to node L − n − 1 (the predecessor) in a second traversal and unlink its next. O(n) time, O(1) space, two passes.

The waste

The counting pass is redundant if we can maintain a real-time "how far from the end" measure — which a fixed n-node gap between fast and slow provides.

The move

Use a dummy node before the head so deletion of the first node needs no special case. Advance fast by n + 1 steps from the dummy (or n steps then check for None). Then advance both until fast is None. slow.next is the node to remove: set slow.next = slow.next.next.

Invariant & cost

The n-node gap between slow and fast is maintained throughout the joint walk. When fast exits, slow is exactly at the predecessor of the target. O(n) time, O(1) space, one pass.

The code

python

def removeNthFromEnd(head, n):
    dummy = ListNode(0, head)      # sentinel avoids head-removal edge case
    slow = fast = dummy
    for _ in range(n + 1):         # advance fast by n+1 steps from dummy
        fast = fast.next
    while fast:                     # move together until fast exits
        slow = slow.next
        fast = fast.next
    slow.next = slow.next.next      # unlink the target node
    return dummy.next
Narrate it to the interviewer

"The brute force is two passes: count the length, walk to the predecessor. I'll do it in one pass with a fixed gap. I create a dummy node before the head so I never need a special case for removing the first node. I advance fast by n + 1 steps from the dummy, creating a gap of n between slow and fast. Then I walk both together until fast becomes null — at that point slow is right behind the target node, and I unlink it. The invariant is that slow is always n + 1 nodes behind fast, so when fast exits, slow.next is exactly the node to remove. O(n) time, O(1) space, one pass."

Dry-run aloud: a one-node list with n=1 — fast advances once to the real node, again to null; slow stays at the dummy; dummy.next (the only node) is unlinked, returning None. A two-node list with n=2 — removing the head — verify the dummy absorbs the edge case cleanly.

05 Number sequences

Cycle detection without a linked list

The fast & slow idea applies anywhere you can define a "next" function — not just linked lists. A happy number is defined by iteratively replacing a number with the sum of the squares of its digits. The sequence either reaches 1 (the number is happy) or falls into a repeating cycle that never reaches 1.

To detect the cycle, treat the sequence as an implicit linked list: the "next pointer" of a number is its digit-square sum. Run fast at two steps per tick and slow at one — if they ever share a value inside a cycle they must meet (the same lapping argument applies). If slow reaches 1, the number is happy. Because the sequence is deterministic and bounded (it eventually enters a small fixed cycle or hits 1), the pointers will always meet or reach 1 in bounded time.

Worked problemEasyLeetCode 202
Happy Number

A number is happy if repeatedly replacing it with the sum of the squares of its digits eventually produces 1. Determine whether a given positive integer n is happy. Use O(1) extra space.

Think it through

Restate

Apply the digit-square function repeatedly. Either the sequence hits 1 (return true) or it loops forever (return false) — and I need to detect the cycle without storing every value seen.

Brute force

Put each result in a hash set; if a value repeats before 1 is reached, there is a cycle and the number is unhappy. O(log n) time per step, O(k) space where k is the number of steps before cycling (bounded in practice but still extra memory).

The waste

The hash set is cycle detection — the same role the slow & fast pointers perform on a linked list. The "next pointer" here is the digit-square-sum function, not a struct field, but the logic is identical.

The move

Define next_val(x) as the sum of the squares of x's digits. Run slow = next_val(slow) and fast = next_val(next_val(fast)) each tick. Stop when slow == fast (met in a cycle) or slow == 1 (reached the happy terminus).

Invariant & cost

The sequence is deterministic and bounded — it always either reaches 1 or enters a fixed cycle — so the pointers always meet or terminate. O(log n) per step, O(1) space.

The code

python

def isHappy(n):
    def nxt(x):                          # digit-square sum
        total = 0
        while x:
            x, d = divmod(x, 10)
            total += d * d
        return total

    slow, fast = n, nxt(n)
    while fast != 1 and slow != fast:   # stop at 1 or cycle
        slow = nxt(slow)
        fast = nxt(nxt(fast))
    return fast == 1                     # true iff terminated at 1
Narrate it to the interviewer

"The problem is really cycle detection on an implicit sequence: each number's 'next' is its digit-square sum. The brute force stores every value in a set — O(extra space). I'll apply fast and slow instead: slow takes one step per tick, fast takes two. If fast reaches 1, the number is happy. If slow and fast meet, we are in a cycle that never reaches 1, so it's unhappy. The stopping condition checks both at once. The invariant is the same as on a linked list: if a cycle exists, the lapping argument guarantees a meeting; if the sequence terminates at 1, fast gets there first and we return true. O(log n) per step, O(1) space."

Dry-run aloud: n=1 — next(1) is 1, so fast starts as 1 and the loop body never executes, returning fast==1 which is true. n=2 — walk a few steps confirming slow and fast converge at 4 (or later in the cycle) and we return false.

How to drill this pattern

Every fast & slow problem rewards the same four-step narration:
(1) state the brute force and why it costs extra space;
(2) name the pattern and which variant — detection, cycle entry, midpoint, or fixed gap;
(3) write the two-pointer loop, verifying that the guard conditions handle null correctly;
(4) dry-run the edge cases aloud — empty list, single node, self-loop, and even vs odd length for midpoint problems. Code it without an IDE at least once per problem; the actual loop is very short and easy to get wrong on null-pointer guards.

★ Review

Summary & self-test

The whole pattern compresses to: run slow at speed 1 and fast at speed 2; their relative speed is the signal. Inside a loop they must meet; at the tail they diverge; a fixed gap lets one measure how far the other is from the end. Four techniques share the same two pointers and O(1) space.

01Kill the hash setWhenever the brute force stores every visited node, fast & slow can replace it with two pointer variables.
02Cycle detectionslow +1, fast +2; they meet iff a cycle exists. The gap closes by one per tick inside any loop.
03Cycle entryAfter meeting, reset one pointer to the head; walk both at speed 1 — they converge on the entry node.
04MidpointWhen fast exits the tail, slow is at the midpoint. Even lists land on the second middle node.
05Fixed gap (nth from end)Advance fast by n steps first, then move both together until fast exits — slow is n behind the tail.
06Implicit listsAny deterministic "next" function (digit squares, array-as-pointer) can be treated as a linked list for cycle detection.
07Guard conditions matterCheck fast and fast.next before each double-step to avoid null-pointer errors — the most common bug.
08Always O(1) spaceTwo pointer variables only — no sets, no arrays, no recursion stack required.
Self-test before you move on

Cover the notes and answer aloud:
(1) Why are fast and slow guaranteed to meet inside a cycle — what is the invariant, and how many ticks does it take in the worst case?
(2) After meeting in phase 1, why does resetting one pointer to the head and walking both at speed 1 find the cycle entry — sketch the distance argument.
(3) For the midpoint, what stopping condition gives the first middle vs the second middle of an even-length list — and which matters for Palindrome Linked List?
(4) How do you find the n-th node from the end in one pass — what gap do you create and between which two pointers?
(5) Happy Number uses no linked list struct — why can you still apply fast & slow, and what is the "next pointer" function?
(6) Write the two-pointer guard condition for a double-step — why is fast.next checked as well as fast? Any fuzzy answer: re-read that section, replay its diagram, and re-code the matching drill from scratch.
