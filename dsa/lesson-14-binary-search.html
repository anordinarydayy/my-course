Binary Search

Every time you halve the search space you turn a linear problem into a logarithmic one. Binary search is the engine behind that transformation — and its real power lies not in the textbook case of "find a value," but in a reusable off-by-one-proof template that unlocks boundaries, rotated arrays, and invisible search spaces.

00 The idea

What problem does it solve?

The most direct way to find a value in an array is to check every element from left to right. For an array of a million elements, that is up to a million comparisons — O(n). Binary search collapses that to about 20 comparisons — O(log n) — by exploiting a single precondition: the array must be sorted, or more generally, there must be a monotonic predicate that partitions the space into a "left half where the answer cannot be" and a "right half where it might be."

The idea is to look at the midpoint of the current search range and ask: which half can I eliminate entirely? Sortedness guarantees an answer: if the value at the midpoint is too large, everything to its right is also too large — discard the entire right half in one step. If it is too small, discard the left half. Each comparison halves the remaining space, and log₂(10⁶) ≈ 20, so a million-element array needs at most 20 steps.

The bottleneck binary search removes is re-examining elements that are provably useless. A linear scan looks at element 500,001 even after seeing that element 1,000,000 is already too large. Binary search never does.

Why this unlocks more than "find a value"

The textbook version searches for an exact target. But the same halving logic applies to any question of the form "find the leftmost position where some condition first becomes true." That single generalization covers: first and last occurrences, the insertion point for a new element, the minimum in a rotated array, and even problems where the "array" is an implicit range of integers — all with O(log n) time and O(1) space.

01 The template

The canonical lo / hi / mid template ▸

Every implementation of binary search lives or dies by two choices: the loop condition and the mid formula. Get either wrong and you get an infinite loop, a missed element, or an integer overflow. Here is the version that eliminates all three traps.

The mid formula is mid = lo + (hi - lo) // 2, never (lo + hi) // 2. When lo and hi are both large (near 2³¹ - 1 in a 32-bit context), their sum overflows. Subtracting first keeps the arithmetic inside range. In Python integers don't overflow, but the idiom is worth internalizing for language-agnostic interviews.

The loop invariant is: if the target exists in the array, it lies within a[lo .. hi] inclusive. Each iteration either finds the target at mid, or provably eliminates mid and moves one boundary past it. The loop runs while lo <= hi — the moment the range becomes empty (lo > hi), the target is not present.

while lo <= hi (exact match)

  • Searches for one specific value. Returns its index or -1.
  • Loop terminates when lo > hi — the range is truly empty.
  • Boundary updates: lo = mid + 1 or hi = mid - 1 — always skip past mid.

while lo < hi (boundary search)

  • Finds the leftmost or rightmost position satisfying a predicate.
  • Loop terminates when lo == hi — a single candidate remains.
  • One boundary keeps mid (e.g. hi = mid); check the answer after the loop.
Worked problemEasyLeetCode 704
Binary Search

You have an integer array sorted in ascending order and a target value. Return the index of the target if it exists, or -1 if it does not. Your solution must run in O(log n) time.

Think it through

Restate

The array is sorted; I need the exact index of one target value, or -1 if it is absent. The constraint explicitly rules out a linear scan.

Brute force

Scan left to right: for each index, check if a[i] == target. Correct, but O(n) — and the problem forbids it.

The waste

The linear scan ignores sortedness. If a[mid] < target, every element at indices 0 through mid is also less than the target and can never be the answer — a linear scan rechecks them all anyway.

The move

Maintain lo = 0, hi = n - 1. Each iteration compute mid = lo + (hi - lo) // 2. If a[mid] == target, return mid. If a[mid] < target, set lo = mid + 1 (left half is useless). If a[mid] > target, set hi = mid - 1 (right half is useless). If the loop ends without a hit, return -1.

Invariant & cost

At every step, if the target is in the array, it sits within a[lo .. hi]. Each iteration halves the range, so the loop runs at most ⌈log₂ n⌉ times: O(log n) time, O(1) space.

The code

python

def search(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = lo + (hi - lo) // 2     # safe mid: avoids (lo+hi) overflow
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            lo = mid + 1               # target is in the right half
        else:
            hi = mid - 1               # target is in the left half
    return -1                          # lo > hi: range is empty, not found
Narrate it to the interviewer

"The constraint says O(log n), so binary search. My invariant: if the target is present, it is always within a[lo .. hi]. I compute mid as lo + (hi - lo) // 2 — the subtraction-first form avoids potential overflow in languages with fixed-width integers. If mid is the target, done. If mid is too small, the entire left half is eliminated, so lo = mid + 1. If mid is too large, the right half is eliminated, so hi = mid - 1. The loop runs while lo <= hi — once lo > hi the range is empty and the target is absent. That is O(log n) time, O(1) space."

Dry-run aloud: single-element array where the element is the target; single-element array where it is not; target smaller than the entire array; target larger than the entire array.

What the interviewer asks next

"Why not (lo + hi) // 2?" — In Python it is fine because integers are arbitrary-precision, but in Java/C++ with 32-bit ints, lo + hi overflows when both are near INT_MAX. Name that trade-off. "What if there are duplicates?" — The above returns any matching index, not necessarily the first or last. That leads directly to the lower-bound and upper-bound variants in Section 02.

02 Boundary search

First and last occurrence: lower / upper bound ▸

When duplicates are allowed, "find the value" is not enough — an interviewer will often ask for the first occurrence (lower bound) or the last occurrence (upper bound). Both use a slightly different template: the loop runs while lo < hi, and instead of returning immediately on a match, you remember the match and keep narrowing.

The key difference is which boundary you move when you see a match, and whether you adjust by 1 or not. For the lower bound: when a[mid] == target, the first occurrence might still be to the left, so set hi = mid (keep mid in range, do not skip it). For the upper bound: when a[mid] == target, the last occurrence might be further right, so set lo = mid + 1. After the loop, check whether a[lo] actually equals the target — if not, it was absent.

The reusable shape

Both bounds share the same structure — only the direction of the "on match" move differs. Lower bound: if a[mid] < target: lo = mid + 1 else: hi = mid — loop ends when lo == hi, that is the answer slot. Upper bound: if a[mid] <= target: lo = mid + 1 else: hi = mid — lo - 1 after the loop is the last occurrence. Memorize the direction, not the full code.

Worked problemMediumLeetCode 34
Find First and Last Position of Element in Sorted Array

A sorted integer array may contain duplicates. Return the starting and ending index of a given target value as a two-element list. If the target is not present, return [-1, -1]. Your algorithm must be O(log n).

Think it through

Restate

Find the index of the leftmost copy of the target (lower bound) and the index of the rightmost copy (upper bound). Two separate binary searches on the same sorted array.

Brute force

Scan left to right; record the first and last hit. Correct, but O(n) — a million elements with one target buried near the end would scan the whole thing.

The waste

A linear scan touches every duplicate in between the first and last. Binary search can locate the boundary positions directly by adjusting which direction it continues after a match.

The move — lower bound

Run binary search with lo < hi. When a[mid] < target, set lo = mid + 1. Otherwise (includes a[mid] == target), set hi = mid — the first occurrence might still be at mid or to its left, so do not skip it. When the loop ends, check if a[lo] == target; if not, the target is absent.

The move — upper bound

Same structure, but when a[mid] <= target, set lo = mid + 1 — a match means we might go further right. Otherwise set hi = mid. After the loop, lo - 1 is the last occurrence (if the lower bound confirmed the target is present).

Invariant & cost

Each search halves the window. Two independent O(log n) searches give O(log n) total, O(1) space. The loop terminates because each step strictly reduces hi - lo.

The code

python

def searchRange(nums, target):
    def lower_bound():
        lo, hi = 0, len(nums)       # hi = n so range [0, n) covers all slots
        while lo < hi:
            mid = lo + (hi - lo) // 2
            if nums[mid] < target:
                lo = mid + 1
            else:
                hi = mid             # mid might be the first hit — keep it
        return lo

    def upper_bound():
        lo, hi = 0, len(nums)
        while lo < hi:
            mid = lo + (hi - lo) // 2
            if nums[mid] <= target:
                lo = mid + 1        # mid matched but last hit might be further right
            else:
                hi = mid
        return lo - 1               # last occurrence is one before the insertion point

    first = lower_bound()
    if first == len(nums) or nums[first] != target:
        return [-1, -1]
    return [first, upper_bound()]
Narrate it to the interviewer

"I need two boundary positions, so I run two independent binary searches. For the lower bound, I use lo < hi and set hi = mid on any match — keeping mid in range because the first occurrence might still be there or further left. When the loop closes, lo is the insertion point, which is also the first occurrence if the value exists. For the upper bound, on a match I set lo = mid + 1 — the last occurrence could be further right. After the loop, lo - 1 is the last hit. I confirm the target is actually present by checking the lower-bound result before returning. Two O(log n) searches — O(log n) total, O(1) space."

Dry-run aloud: target not in the array; entire array is the same value as target; single-element array matching the target; target is smaller than all elements.

03 Rotated array

Search in a rotated sorted array

A common interview twist: take a sorted array and rotate it — cut it at some pivot and move the left chunk to the right end. For example, [3, 5, 8, 11, 14, 17] rotated at index 3 becomes [11, 14, 17, 3, 5, 8]. The array is no longer globally sorted, so vanilla binary search fails.

The key observation is that after computing mid, at least one of the two halves is always sorted. You can tell which half by comparing a[lo] with a[mid]: if a[lo] <= a[mid], the left half [lo, mid] is sorted (the rotation happened entirely in the right half). Otherwise the right half [mid, hi] is sorted.

Once you know which half is sorted, you can check in O(1) whether the target falls within that half's range. If it does, narrow to that half. If it does not, narrow to the other half. Each step still halves the space, preserving O(log n).

Worked problemMediumLeetCode 33
Search in Rotated Sorted Array

An array of distinct integers was sorted in ascending order, then rotated at some unknown pivot index. Given the rotated array and a target, return the index of the target, or -1 if it is not present. You must solve this in O(log n) time.

Think it through

Restate

Sorted then rotated at an unknown pivot — find the target's index in O(log n). Integers are distinct (no duplicates, which simplifies the rotation check).

Brute force

Linear scan: check each element. Works, but O(n) — fails the time constraint.

The waste

A linear scan again ignores structure. The array is not globally sorted, but after computing mid, one of the two halves is always a contiguous sorted block — sortedness is preserved locally even if not globally.

The move

Standard lo <= hi loop. After computing mid, check if the left half is sorted: a[lo] <= a[mid]. If yes, check if target is in [a[lo], a[mid]) — if it is, set hi = mid - 1; otherwise set lo = mid + 1. If the left half is not sorted, the right half must be sorted; check if target is in (a[mid], a[hi]] — if yes, set lo = mid + 1; otherwise set hi = mid - 1.

Invariant & cost

At each step, the range halves because we always move a boundary past mid. The target is never in the discarded half because the range check for the sorted portion is exact. O(log n) time, O(1) space.

The code

python

def search(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = lo + (hi - lo) // 2
        if nums[mid] == target:
            return mid
        if nums[lo] <= nums[mid]:         # left half [lo..mid] is sorted
            if nums[lo] <= target < nums[mid]:
                hi = mid - 1           # target is in the sorted left half
            else:
                lo = mid + 1           # target is in the right half
        else:                             # right half [mid..hi] is sorted
            if nums[mid] < target <= nums[hi]:
                lo = mid + 1           # target is in the sorted right half
            else:
                hi = mid - 1           # target is in the left half
    return -1
Narrate it to the interviewer

"The array is rotated, so I can't apply vanilla binary search directly. The key insight: after picking mid, at least one of the two halves is always sorted. I check which one by comparing a[lo] with a[mid] — if a[lo] <= a[mid], the left half is a clean sorted segment. I then check if the target falls in that segment's range; if it does, I search the left half, otherwise the right. I flip the logic when the right half is the sorted one. Every step eliminates one half, so it remains O(log n)."

Dry-run aloud: no rotation at all (pivot at 0); pivot at the last element; target is the smallest element; target larger than all elements; single-element array.

What the interviewer asks next

"What if there are duplicates?" — When a[lo] == a[mid], you cannot tell which half is sorted (both could have the same value on a repeated sequence). The safe fallback is lo++ to skip the ambiguity, which degrades worst-case to O(n) but is still correct. "Find the rotation pivot index itself?" — Binary search for the minimum: the minimum is the only element smaller than its left neighbor, and it is always in the unsorted half.

04 2D matrix

Search a 2D matrix as a flattened array

A classic variant gives you an m × n matrix where each row is sorted in ascending order and the first element of each row is greater than the last element of the previous row. The matrix is essentially a sorted array bent into rows — so you can run a single binary search over the virtual index range [0, m × n - 1) and convert the virtual index to (row, col) with simple division.

Given virtual index k in an m × n matrix: row = k // n and col = k % n. The conversion costs O(1) arithmetic, so the search is still O(log(m × n)) = O(log m + log n).

The reusable shape

Whenever a 2D structure has the property that "row i ends before row i+1 begins," treat it as a 1D sorted array. The virtual index trick keeps the code almost identical to plain binary search — only the indexing into the "array" changes. This also applies to any structure that can be serialized into a monotonically increasing sequence.

Worked problemMediumLeetCode 74
Search a 2D Matrix

You have an m × n integer matrix where each row is sorted in ascending order and the first integer of each row is strictly greater than the last integer of the previous row. Given a target, determine whether it exists in the matrix. Solve this in O(log(m × n)) time.

Think it through

Restate

The matrix is a sorted 1D array folded into rows. I need to find one target in O(log(m × n)).

Brute force

For each row, scan each column. That is O(m × n) — reading every cell.

The waste

A double loop ignores the global sorted order. Any element to the right of and below a "too large" cell is also too large — binary search can eliminate entire halves of the matrix in one step.

The move

Set lo = 0, hi = m * n - 1. Compute mid in the virtual index space. Map it to the matrix: row = mid // n, col = mid % n, and read matrix[row][col]. The rest is identical to 1D binary search.

Invariant & cost

The virtual flat array is sorted, so the standard invariant holds. Each step halves the virtual range: O(log(m × n)) time, O(1) space.

The code

python

def searchMatrix(matrix, target):
    m, n = len(matrix), len(matrix[0])
    lo, hi = 0, m * n - 1
    while lo <= hi:
        mid = lo + (hi - lo) // 2
        val = matrix[mid // n][mid % n]   # map virtual index → (row, col)
        if val == target:
            return True
        elif val < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return False
Narrate it to the interviewer

"The matrix is a sorted 1D array bent into rows — the first integer of each row follows the last of the previous, so the entire matrix is monotonically increasing in row-major order. I treat it as a flat array by searching the virtual index range [0, m*n - 1]. At each step I convert the virtual mid to a real cell with mid // n for the row and mid % n for the column, then compare normally. The invariant and termination are identical to 1D binary search. O(log(m × n)) time, O(1) space."

Dry-run aloud: 1×1 matrix (target matches; target does not match); target smaller than all elements (lo immediately passes hi); target larger than all elements.

05 Recognition

Recognizing it, and the traps

Binary search shows up in more places than just array lookups. Train your eye for these signals:

  • The input is sorted and the problem asks for a specific value or a boundary position.
  • The problem involves an implicit sorted space — a range of integers where you are checking a monotonic condition (e.g., "find the minimum speed such that you finish a task in time").
  • The answer is a number and you can frame the question as "is X enough?" — if the predicate is monotone (once true, always true), binary search the answer space.
  • The word "rotated sorted" — the standard rotated-array template applies.
  • A 2D grid where each row and column is sorted — either the flatten trick or a staircase search from a corner.

And the bugs that end coding rounds:

  • Off-by-one on the boundary update — forgetting to add or subtract 1 from mid when updating lo or hi. Missing it causes the loop to stall at a two-element subarray where mid == lo forever.
  • Using the wrong loop condition — lo <= hi for exact match; lo < hi for boundary search. Mixing them produces either an extra iteration or a missed element.
  • Not checking the result after a lower/upper bound search — the loop always terminates; you must verify that a[lo] actually equals the target before declaring success.
  • The infinite loop in the two-element case — with lo < hi and hi = mid, mid floor-divides toward lo, so the loop shrinks. But if you accidentally write hi = mid - 1 on a match, you might skip the answer. Draw a two-element example by hand whenever unsure.
The decision in one line

If the question is "does value X exist?" — exact-match binary search with lo <= hi. If the question is "what is the leftmost / rightmost position satisfying a condition?" — lower/upper bound with lo < hi. If the input is not sorted but the answer space is monotone — binary search the answer space, not the input. If the array is rotated — add the sorted-half check.

How to drill binary search effectively

Binary search fails in interviews because candidates remember the idea but not the exact boundary rules. For every problem, write the loop condition and the boundary updates before writing any other code. Then:
(1) trace a two-element array by hand — the most common spot for off-by-one bugs;
(2) trace the case where the target is absent;
(3) state the invariant explicitly. Once that is clean, the rest of the code follows mechanically. Aim for less than 20 minutes per medium problem.

★ Review

Summary & self-test

Binary search compresses to one principle: if you have a sorted (or monotone) space and can discard half of it after each comparison, you can find any answer in O(log n) steps. The template has two flavors — exact match with lo <= hi and boundary search with lo < hi — and the choice between them determines the entire shape of the code including how you update the boundaries.
