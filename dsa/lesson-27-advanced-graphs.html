Advanced Graphs — MST & More

Once you can traverse a graph, the next tier of problems asks you to build something optimal over it — the cheapest tree that connects every node, or a proof that the graph's nodes can be split into two conflicting groups. Minimum spanning trees, bipartite checks, and their cousins are the graph patterns that separate solid candidates from strong ones.

Original study notes. Written from scratch for coding-interview prep. Problem names (Min Cost to Connect Points, Is Graph Bipartite?, Swim in Rising Water) are referenced only as standard identifiers; every description, diagram, walkthrough, and line of code here is original. Contains three interactive diagrams and three fully worked problems — each with the reasoning chain, entity-escaped Python, and a narration script for the interview.
00 The idea

What does this pattern solve?

Standard BFS and DFS visit every reachable node — they answer "can I get there?" Once that foundation is solid, a cluster of harder questions appears: What is the cheapest way to wire up all the cities? Can I schedule tasks without a conflict? Is this road network safe even if one bridge is destroyed? Each of these is still a graph question, but the interesting work is about choosing or checking structure, not just traversal.

Three sub-problems show up most often in interviews:

  • Minimum spanning tree (MST) — connect all V nodes with exactly V−1 edges, minimising total weight. Two classical algorithms: Kruskal (sort edges, union-find) and Prim (grow from a seed with a heap).
  • Bipartite checking — decide whether the graph's vertices can be 2-coloured so no two neighbours share a colour. Equivalent to asking whether any odd-length cycle exists.
  • MST as a min-bottleneck path — several disguised problems (including "swim in rising water") reduce to finding the path whose maximum edge weight is minimised, which is the same as finding the MST's critical edge.
P5 — Graph Patterns (Stretch)

This lesson sits in phase 5 of the roadmap — the stretch tier. It assumes comfort with BFS/DFS (Lesson 23) and union-find (Lesson 24). If either feels shaky, review those first. MST problems appear in medium-to-hard slots at companies like Google, Meta, and the Singapore tech offices; bipartite checks are a clean medium that interviewers use to probe your graph fundamentals.

01 MST via edges

Kruskal's algorithm: sort edges, union-find

A spanning tree of a connected graph with V vertices uses exactly V−1 edges and reaches every node — there is no shorter tour. A minimum spanning tree picks those V−1 edges with the smallest total weight. The brute-force approach tries every subset of V−1 edges and scores it — exponential in the number of edges and completely impractical.

Kruskal's key observation is the cut property: given any partition of the vertices into two non-empty groups, the cheapest edge that crosses the cut is always in some MST. You can grow an MST greedily by sorting all edges cheapest-first and adding each one unless it would form a cycle. A cycle would mean both endpoints are already in the same connected component — and union-find checks that in near-constant time.

The algorithm in three lines of intent:

  • Sort all edges by weight ascending.
  • For each edge (u, v, w): if find(u) ≠ find(v) (different components), add the edge and call union(u, v).
  • Stop when V−1 edges have been added.

The invariant is that the edges collected so far form a forest that is a subset of some MST — no cycle ever enters, and only safe cheap edges are admitted. Time: O(E log E) for the sort, plus near-O(E α(V)) for the union-find operations (α is the inverse-Ackermann function, effectively constant). Space: O(V) for the union-find structure.

Kruskal

  • Sorts edges first — great when E is small or the graph is sparse.
  • Union-find gives near-O(1) cycle detection.
  • Natural for problems that hand you an explicit edge list.

Prim (next section)

  • Grows from one vertex with a heap — great when V is small or graph is dense.
  • Never needs to see all edges sorted globally.
  • More natural when the graph is given as an adjacency list.
Worked problemMediumLeetCode 1584
Min Cost to Connect Points

You are given n points on a 2-D plane. The cost of connecting two points is their Manhattan distance. Return the minimum total cost to connect all points (so that every point is reachable from every other point).

Think it through

Restate

Build a fully connected graph where the edge weight between any two points is their Manhattan distance, then find its MST.

Brute force

Try every spanning tree of n nodes — that is an exponential number of subsets. Or at least try every permutation of edges: still O(n² log n²) = O(n² log n) just for sorting all n² edges, which is feasible at small n but the approach is identical to Kruskal.

The waste

We do not need all n² edges in memory at once. Prim's algorithm picks the cheapest unvisited edge from the current frontier at each step, so it only ever looks at the neighbours of already-included nodes — but here the graph is complete, so both Kruskal and Prim see the same O(n²) edges. The real saving is the greedy structure: we never reconsider a node already in the MST.

The move

Generate all n(n−1)/2 edges (pairs of points with their Manhattan distance), sort ascending, then run Kruskal with union-find: add an edge if the two points belong to different components. Stop when n−1 edges are added.

Invariant & cost

Every accepted edge is the cheapest crossing some cut, so the greedy choice is globally safe. We collect exactly n−1 edges. Generating edges: O(n²). Sorting: O(n² log n). Union-find: near-O(n²). Total: O(n² log n) time, O(n²) space.

The code

python

def minCostConnectPoints(points):
    n = len(points)
    if n == 1: return 0

    # Build edge list: (cost, i, j) for every pair
    edges = []
    for i in range(n):
        for j in range(i + 1, n):
            dist = abs(points[i][0] - points[j][0]) + abs(points[i][1] - points[j][1])
            edges.append((dist, i, j))
    edges.sort()                          # cheapest edges first

    # Union-Find with path compression + rank
    parent = list(range(n))
    rank   = [0] * n

    def find(x):
        while parent[x] != x:
            parent[x] = parent[parent[x]]  # path halving
            x = parent[x]
        return x

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx == ry: return False
        if rank[rx] < rank[ry]: rx, ry = ry, rx
        parent[ry] = rx
        if rank[rx] == rank[ry]: rank[rx] += 1
        return True

    total, added = 0, 0
    for cost, i, j in edges:
        if union(i, j):            # different components: safe to add
            total += cost
            added += 1
            if added == n - 1: break  # MST complete
    return total
Narrate it to the interviewer

"This is a minimum spanning tree problem in disguise. I have n points and the cost between any two is their Manhattan distance — so I'm building a complete weighted graph and want its MST. The naive approach tries every subset of n−1 edges, which is exponential. Instead I'll use Kruskal's: generate all O(n²) edges, sort cheapest-first, then greedily add each edge if the two endpoints are in different union-find components. My invariant is the cut property — the cheapest edge crossing any partition is always in some MST — so each greedy pick is globally safe. I stop after n−1 edges are added. That's O(n² log n) for the sort, near-O(n²) for the union-find: the sort dominates."

Dry-run aloud: a single point (return 0 immediately); two points (one edge, cost is their distance); three collinear points at (0,0), (1,0), (2,0) — MST picks edges of cost 1 and 1, total 2, not the 2-length direct edge between the extremes.

What the interviewer asks next

"Can you do it without generating all edges?" → Yes: Prim's algorithm with a lazy heap processes each candidate edge once, and for dense complete graphs the O(n²) Prim variant (no heap, just a dist[] array scanned linearly) is actually faster than the sort. "Why is the greedy choice safe?" → the cut property: if you partition the already-connected component from the rest, the minimum-weight crossing edge can't be skipped without increasing total cost. "What changes for directed graphs?" → directed MST is called minimum arborescence and requires Edmonds' algorithm — significantly harder.

02 MST via vertices

Prim's algorithm: grow a tree with a heap

Kruskal thinks in terms of edges — it processes the global sorted list. Prim thinks in terms of vertices — it grows a single connected tree one node at a time. At each step it picks the cheapest edge that leads out of the current tree into an unvisited node. A min-heap on (weight, node) makes that "cheapest out-edge" lookup O(log V).

The setup: start by pushing the seed node with cost 0 into the heap. Mark it visited when you pop it. For each unvisited neighbour, push (edge_weight, neighbour). Pop the minimum — if already visited, skip it (a stale heap entry). Otherwise mark it visited, add its edge weight to the MST cost, and push its unvisited neighbours. Stop when V nodes are visited.

The invariant is that every node inside the tree is reachable from the seed via already-accepted MST edges, and the heap always holds the cheapest known crossing edge for every fringe node. Time: O(E log V) with a binary heap. Space: O(V + E).

Minimax path: Prim's idea on a grid

A natural variant replaces "cheapest edge to add to the tree" with "path whose worst (maximum) edge weight is smallest." Instead of accumulating a sum in the heap priority, you track max(running_cost, new_edge). This is how Swim in Rising Water (and Path With Minimum Effort) works — the first time you pop the destination cell, its priority is the minimum possible bottleneck over all paths from start.

Dense vs sparse: choosing the right MST algorithm

For dense graphs (E ≈ V²), Kruskal's O(E log E) sort dominates because E is large; Prim with a Fibonacci heap achieves O(E + V log V) and wins. For sparse graphs (E ≈ V), Kruskal is simpler to implement and the sort is only O(V log V). In interview problems — especially on grids or complete point clouds — Prim's O(V²) array variant (scan a dist[] array instead of using a heap) is sometimes the cleanest solution.

Worked problemHardLeetCode 778
Swim in Rising Water

An n×n grid holds distinct integers 0..n²−1, each representing the elevation at that cell. Water rises uniformly over time — at time t, all cells with elevation ≤ t are flooded. You start at the top-left (0,0) and want to reach the bottom-right (n−1,n−1). Return the minimum time t at which a path of flooded cells connects start to finish.

Think it through

Restate

Find a path from (0,0) to (n−1,n−1) that minimises the maximum elevation along the path. The answer is that maximum, because t must be at least as large as every cell you cross.

Brute force

Binary search on t, and for each candidate t run BFS to check connectivity. That's O(n² log n²) = O(n² log n). Correct but not the cleanest insight.

The waste

Binary search + BFS rediscovers the same paths repeatedly. The real structure is: we want the path whose bottleneck (maximum edge elevation) is minimised — the classic minimax path. The MST has a special property: the path between any two nodes in the MST is the minimax path in the original graph. So we just need the MST's path from (0,0) to (n−1,n−1), and the bottleneck on that path is the answer.

The move

Use a min-heap (Prim-like / Dijkstra-like). Priority is the elevation of the cell, not cumulative sum. Pop the minimum-elevation unvisited cell, mark it visited, push its 4-directional unvisited neighbours with their elevations. The answer is the elevation of the last popped cell when we first pop (n−1,n−1). We also need to wait for the start cell: the answer is at least grid[0][0].

Invariant & cost

Every popped cell has its true minimax distance from (0,0) — the heap always gives the cell whose maximum-elevation path from start is smallest. Time: O(n² log n) for the heap operations over all n² cells. Space: O(n²).

The code

python

import heapq

def swimInWater(grid):
    n = len(grid)
    visited = [[False] * n for _ in range(n)]
    # heap entry: (max_elev_on_path_so_far, row, col)
    heap = [(grid[0][0], 0, 0)]
    visited[0][0] = True

    while heap:
        t, r, c = heapq.heappop(heap)
        if r == n - 1 and c == n - 1:
            return t                # first time we reach the goal
        for dr, dc in ((0,1),(0,-1),(1,0),(-1,0)):
            nr, nc = r + dr, c + dc
            if 0 <= nr < n and 0 <= nc < n and not visited[nr][nc]:
                visited[nr][nc] = True
                cost = max(t, grid[nr][nc])   # bottleneck: max elevation seen
                heapq.heappush(heap, (cost, nr, nc))
    return -1                          # unreachable (shouldn't happen)
Narrate it to the interviewer

"The key insight is that the answer equals the minimum possible bottleneck — the maximum elevation on any path from start to finish. I want to minimise that maximum, which is the minimax path problem. The MST solves minimax paths in general, but on a grid I can implement it directly with a min-heap on (max_elevation_so_far, row, col). I pop the cheapest unvisited cell, update the running maximum when pushing neighbours, and the first time I pop the destination cell, that running max is my answer. The invariant is that a popped cell's priority is the true minimax distance from (0,0). This runs in O(n² log n) time — the heap holds at most n² entries."

Dry-run aloud: a 1×1 grid (return grid[0][0] directly); a 2×2 grid where the only path must go through a high cell — confirm the heap correctly picks the lower of two adjacent paths.

What the interviewer asks next

"Could you binary-search instead?" → Yes, binary search on t and BFS for connectivity also gives O(n² log n) — the minimax-heap approach is cleaner and avoids the outer loop. "Why not use Dijkstra's standard cost?" → Standard Dijkstra accumulates a sum; here we want the maximum along the path, so we use max(current_cost, neighbour_elevation) as the priority. This is sometimes called a "modified Dijkstra" or "Prim-style" traversal.

03 2-colouring

Bipartite check: spread two colours with BFS

A graph is bipartite if you can assign every vertex one of two colours such that no two neighbouring vertices share the same colour. This is equivalent to saying the graph contains no odd-length cycle. Problems that ask "can we divide into two groups with no conflicts?" — scheduling, graph colouring with two colours, or detecting negative cycles in certain graph types — often reduce to this check.

The brute-force approach would enumerate all possible 2-colourings — exponential. The insight is that once you colour one vertex, its neighbours are forced: they must get the opposite colour. And their neighbours are forced in turn. So you run BFS (or DFS) and greedily propagate the opposite colour. The moment you try to colour a vertex that is already coloured with the same colour you intended, you have found an odd cycle — the graph is not bipartite. If BFS completes without contradiction, it is.

The invariant: at every BFS step, the colour of the current node is fixed, and all its unvisited neighbours will receive the opposite colour. Time: O(V + E). Space: O(V) for the colour array.

Worked problemMediumLeetCode 785
Is Graph Bipartite?

You receive a graph as an adjacency list with n nodes (0-indexed). Determine whether the graph can be coloured with exactly two colours such that no edge connects two nodes of the same colour. The graph may be disconnected.

Think it through

Restate

Try to 2-colour the graph so no edge is monochromatic. Return true if it is possible, false if any odd-length cycle exists.

Brute force

Try every 2^n assignment of colours and check all edges. That's O(2^n · E) — completely infeasible even for n = 30.

The waste

Once you fix the colour of one node, all other colours are forced by the graph's edges — there is no real choice. The brute-force explores choices that the graph structure determines for you.

The move

BFS from every unvisited node (handle disconnected components). Assign colour 0 to the source, push it into the queue. For each neighbour: if uncoloured, assign the opposite colour and enqueue; if already coloured with the same colour as the current node, return False immediately — odd cycle found. If BFS drains without conflict, return True.

Invariant & cost

Every queued node has a colour consistent with all of its already-coloured neighbours. The BFS visits every node and edge at most once. O(V + E) time, O(V) space for the colour array and queue.

The code

python

from collections import deque

def isBipartite(graph):
    n = len(graph)
    colour = [-1] * n         # -1 = unvisited

    for start in range(n):
        if colour[start] != -1:
            continue            # already coloured by a previous BFS
        colour[start] = 0
        q = deque([start])
        while q:
            node = q.popleft()
            for nei in graph[node]:
                if colour[nei] == -1:
                    colour[nei] = 1 - colour[node]  # flip colour
                    q.append(nei)
                elif colour[nei] == colour[node]:
                    return False    # same colour on an edge → odd cycle
    return True
Narrate it to the interviewer

"I need to 2-colour the graph so no edge is monochromatic — that's equivalent to checking for odd-length cycles. The exponential approach tries all 2^n assignments; but once you colour one node its neighbours are forced. So I run BFS: assign colour 0 to the source, and every unvisited neighbour gets the opposite colour. If I ever reach a neighbour that's already the same colour as its parent, I found an odd cycle — not bipartite. I also need to handle disconnected graphs by running BFS from every unvisited node. The invariant is that every coloured node has a consistent colour with all its processed neighbours. O(V + E) time, O(V) space."

Dry-run aloud: a single node with no edges (trivially bipartite); a triangle (0-1-2-0, odd cycle → False); a 4-cycle (0-1-2-3-0, even cycle → True); a disconnected graph where one component is an odd cycle.

What the interviewer asks next

"Does it matter whether you use BFS or DFS?" → No, either works; DFS is slightly shorter to write, BFS is easier to reason about visually. "What does bipartite mean for real-world problems?" → It means two disjoint sets with edges only between sets — think users vs items in a recommendation graph, or students vs exam slots in a scheduling conflict graph. "What about directed graphs?" → Bipartiteness is defined for undirected graphs; for directed ones you'd check for cycles using DFS coloring (white/grey/black states), which is a different check.

04 Beyond MST & bipartite

Bridges, articulation points, and max-flow

The advanced graph toolkit extends well beyond MST and bipartite checks. Two more topics appear in stretch-level interviews — you need to know the concepts and the key algorithm names even if you only fully implement them in senior-level rounds.

Bridges and articulation points

A bridge is an edge whose removal disconnects the graph. An articulation point (cut vertex) is a node whose removal disconnects it. Both are found in a single DFS using Tarjan's algorithm, which tracks two values per node:

  • disc[u] — the DFS discovery time of node u.
  • low[u] — the earliest discovery time reachable from the subtree rooted at u (including back-edges).

An edge (u, v) is a bridge if low[v] > disc[u] — meaning the subtree below v cannot reach u or any ancestor of u via a back-edge. If it could, removing (u,v) wouldn't disconnect anything. Tarjan's runs in O(V + E).

Maximum flow (the augmenting-path idea)

Max-flow asks: given a directed graph with capacities on each edge, what is the maximum flow you can push from a source to a sink? The fundamental theorem (max-flow = min-cut) says the answer equals the weight of the smallest edge-cut that separates source from sink.

The simplest algorithm is Ford-Fulkerson: repeatedly find any augmenting path from source to sink in the residual graph (which includes reverse edges for flow that can be cancelled), push as much flow as the path's bottleneck allows, and update residual capacities. Stop when no path exists. With BFS to find shortest augmenting paths (the Edmonds-Karp variant), the complexity is O(V · E²). Max-flow rarely appears as a direct interview question, but the min-cut framing does — recognising that a "minimum number of edges to disconnect A from B" problem is min-cut is the key skill.

When these come up in interviews

Bridges and articulation points: expect them in problems framed as "critical connections in a network" (LeetCode 1192) or "minimum nodes whose removal disconnects the graph." Max-flow: mostly at senior or staff-level rounds, or disguised as "minimum cuts" in network design. For the vast majority of coding interviews, MST and bipartite are the graph patterns that actually test you — treat bridges and max-flow as knowledge to name confidently rather than implement from memory.

05 Recognition

Spotting the pattern and the traps

The hardest part of advanced graph questions is identifying which structure the problem needs. Train your ear for these signals:

  • MST signals: "minimum cost to connect all", "cheapest network", "reduce total weight while keeping connected." If the graph is given as an edge list, reach for Kruskal. If given as adjacency list or a grid, Prim is usually cleaner.
  • Bipartite signals: "divide into two groups", "no conflict between neighbours", "can you schedule without overlap", "possible 2-colouring." Any even-vs-odd or friend-vs-enemy structure.
  • Minimax-path signals: "minimum of the maximum", "lowest bottleneck path", "minimum time at which a path exists." This is MST / modified Dijkstra with max instead of sum.
  • Bridge / cut signals: "critical edge", "single point of failure", "what removal disconnects the network." Think Tarjan's.

Common traps:

  • Forgetting disconnected components in bipartite checks — you must run BFS/DFS from every unvisited node, not just node 0.
  • Confusing MST cost with shortest path — MST minimises total edge weight to span all nodes; Dijkstra minimises path cost to a specific target. They answer different questions and should not be mixed up.
  • Union-find without path compression in Kruskal — without compression, find() can degrade to O(n) per call, blowing up the total runtime.
  • Using sum instead of max in minimax-path problems — the heap priority must be max(running_cost, new_edge), not the sum. Using sum gives the wrong answer.

Practice ladder

Idea Min-heap on (max_elevation_so_far, row, col); pop destination when it first appears. O(n² log n).

HardCritical Connections in a NetworkTarjan bridges

Idea Tarjan's bridge algorithm: DFS with disc[] and low[] arrays. Edge (u,v) is a bridge when low[v] > disc[u]. O(V + E).

On a 2-D grid of heights, find a path from top-left to bottom-right that minimises the maximum absolute height difference between consecutive cells.

Idea Same minimax-heap as "Swim in Rising Water" but the edge cost is |height[nr][nc] − height[r][c]|. Priority = max(running_max, abs_diff). O(mn log(mn)).

How to drill advanced graph problems

Advanced graph questions demand a two-step narration: first name the reduction ("this is an MST / bipartite / minimax-path problem because…"), then explain the algorithm. Practice that framing out loud before touching the keyboard. Always state:
(1) the graph model (nodes, edges, weights/direction);
(2) the algorithm and its invariant;
(3) time and space complexity;
(4) the edge cases — disconnected graph, single node, all edges equal weight. Interviewers at this level are testing whether you can recognise the structure, not just whether you can implement union-find.

★ Review

Summary & self-test

The advanced graph pattern compresses to one decision tree: connect all nodes cheaply → MST (Kruskal or Prim); 2-colour without conflict → bipartite BFS; minimise the worst edge on a path → minimax-heap; find critical structure → Tarjan's. Each algorithm has a single clean invariant and a matching union-find, BFS, or heap skeleton.

01MST = greedy on cutsThe cut property guarantees each cheapest crossing edge is safe — that makes the greedy choice globally optimal.
02KruskalSort edges, union-find to reject cycles. O(E log E). Best for sparse graphs or explicit edge lists.
03PrimGrow from one node via a min-heap on crossing edges. O(E log V). Best for dense graphs or adjacency lists.
04Bipartite = no odd cycleBFS propagates opposite colours; same-colour conflict on any edge → odd cycle → not bipartite.
05Minimax path = MST pathUse max instead of sum in the heap priority. The MST contains every minimax path.
06Union-find must compressAlways use path compression (and union by rank) or find() degrades to O(n) per call.
07Handle disconnected graphsAlways outer-loop over all nodes for bipartite/BFS; check if MST has V−1 edges for connectivity.
08Name the reduction firstSay "this is an MST problem because…" before coding. That framing is what senior interviewers evaluate.
Self-test before you move on

Cover the notes and answer aloud:
(1) What is the cut property and why does it make Kruskal's greedy choice safe?
(2) Walk through Prim's algorithm from a seed node — what does the heap store and when do you skip a popped entry?
(3) A problem says "find the minimum time at which a path exists." How do you recognise this as minimax-path and not shortest-path, and what changes in the heap priority?
(4) You run bipartite BFS on a graph and reach a node that's already coloured with the same colour. What exactly does that prove and why?
(5) Kruskal on a disconnected graph finishes with only k < V−1 edges added. What does that tell you and how do you report it?
(6) You must find all "critical connections" in a network. Name the algorithm, the two values it tracks per node, and the condition for identifying a bridge.
