Linked Lists

A linked list is pointer surgery: instead of a contiguous block of memory, each node carries its value and a reference to the next one. Mastering it means learning the three rewiring moves that power dozens of interview problems — the dummy-node trick, in-place reversal, and two-pointer traversal.

00 The structure

Nodes & structure

An array stores elements at consecutive memory addresses, so jumping to index k costs O(1) — you add k * element_size to the base address. A linked list gives that up entirely. Each node is a small object containing a value and a next pointer (or two pointers for a doubly linked list). The nodes can sit anywhere in memory; the chain is stitched together by pointers, not by address proximity.

The list itself is represented by a reference to its head node. If you lose that reference, you lose the whole list. The last node's next pointer is null (or None in Python), marking the tail. A doubly linked list adds a prev pointer so you can walk backwards, at the cost of an extra pointer per node and slightly more care when rewiring.

Why linked lists appear in interviews

They are not the best data structure for most production use cases — a resizable array (Python list, Java ArrayList) is almost always faster in practice. But linked lists expose the core skill of pointer manipulation: the ability to rewire references in your head without losing a node. That is what interviewers test. The operations that look trivial on arrays (insert at head, delete a middle node without a second scan) become the difficult moves here, and the bugs (losing the rest of the list by setting a pointer too early) are the exact mistakes that reveal inexperience.

01 Costs

Costs & trade-offs

Understanding why each operation costs what it costs is the foundation for every technique in this lesson. The key asymmetry is this: linked lists make insertion and deletion cheap wherever you already have a pointer, and make random access expensive.

Linked list

  • Insert / delete at head or tail: O(1) — adjust one or two pointers.
  • Insert / delete at a known node: O(1) — same pointer surgery, no shifting.
  • Access by index: O(n) — must walk from head, no shortcut.
  • Search: O(n) — linear scan; no binary search without a sorted structure.
  • Space: O(n) total, but each node pays extra for its pointer(s).

Dynamic array

  • Insert / delete at end: O(1) amortized (occasional resize).
  • Insert / delete in the middle: O(n) — must shift elements.
  • Access by index: O(1) — direct address calculation.
  • Search (sorted): O(log n) — binary search is possible.
  • Space: O(n) compact; better cache locality than a linked list.

The split is clean: if your workload is heavy on insertions and deletions at arbitrary positions and you already have a pointer to the relevant node, a linked list wins. If you need fast random access or binary search, an array wins. Real-world structures like queues and LRU caches often use a linked list precisely for O(1) head/tail operations.

02 The trick

The dummy-node trick

The most common source of messy linked-list code is the head deletion edge case. Deleting any middle node is straightforward: you have a pointer to its predecessor, so you set pred.next = pred.next.next. But if the node to delete is the head, there is no predecessor — you need a special branch that reassigns the head variable. You end up with if node == head: head = head.next; else: pred.next = pred.next.next, duplicating logic and creating a common bug site.

The fix is to create a dummy node — a sentinel node with no meaningful value, whose next pointer is set to the real head. Now every real node, including the original head, has a predecessor (the dummy). The uniform deletion rule works everywhere: pred.next = pred.next.next. You return dummy.next at the end as the (possibly new) head of the list.

When to reach for a dummy

Any time the result could have a different head than the input — a deletion, a merge, a filter, or a partition. Create the dummy at the start (dummy = ListNode(0); dummy.next = head), build the result chain by threading dummy.next through, and return dummy.next at the end. This pattern alone eliminates an entire class of off-by-one and null-pointer bugs.

python — see it in code

# Delete all nodes whose value equals `target`
def removeElements(head, target):
    dummy = ListNode(0)         # sentinel before the real head
    dummy.next = head
    pred = dummy                # pred always points to the node before cur
    while pred.next:
        if pred.next.val == target:
            pred.next = pred.next.next   # unlink: works even if pred is dummy
        else:
            pred = pred.next
    return dummy.next           # the (possibly new) head
03 Reversal

In-place reversal

Reversing a linked list is the canonical pointer-surgery exercise. The brute-force approach collects all values into an array and rebuilds the list — O(n) time and O(n) extra space. In place needs only three local variables and O(1) space. The key insight is that you cannot move forward and rewire at the same time without saving the next pointer first — that save step is where most bugs hide.

Keep three variables: prev (initially None, the new tail's end), cur (the node being rewired), and nxt (saved reference so you can advance after the rewire). Each iteration does exactly three things in order: save nxt = cur.next, flip the arrow (cur.next = prev), advance both pointers (prev = cur; cur = nxt). When cur runs off the end, prev is the new head.

The invariant

After processing node cur, the sublist from the original head through cur is fully reversed and prev points to its (new) first node. The sublist from cur.next onwards is still in its original order. This invariant holds at every step and terminates when cur is None — at which point prev is the complete reversed head.

See it usedEasyLeetCode 206
Reverse Linked List

Given the head of a singly linked list, return a new head after reversing all the links in place — no extra list or array allowed.

The code

python

def reverseList(head):
    prev = None                    # will become the new tail's terminator
    cur = head
    while cur:
        nxt = cur.next           # 1. save: don't lose the rest of the list
        cur.next = prev          # 2. flip: point backward
        prev = cur               # 3. advance prev
        cur = nxt                # 4. advance cur
    return prev                  # cur is None; prev is the new head
Narrate it to the interviewer

"I need to reverse all the next-pointers in place. The naive approach collects all values and rebuilds — O(n) space — but I only need three local variables. prev starts as None (the reversed list's terminal null), cur starts at head. Each step: I save cur.next as nxt before I destroy it, flip cur.next to prev, then slide both pointers forward. When cur hits None, prev is sitting on the new head. O(n) time, O(1) space."

Dry-run edge cases: empty list (return None immediately), single node (prev=None, cur=head, one iteration sets head.next=None, returns head — correct). Two nodes: verify both arrows flip correctly.

04 Merging

Merging two sorted lists

Given two sorted singly linked lists, the goal is to produce a single sorted linked list using the existing nodes — no new node allocation, O(1) extra space. The naive approach would collect all values, sort them, and rebuild — O((m+n) log(m+n)) time and O(m+n) space — but we can do O(m+n) time and O(1) space by exploiting the sortedness directly.

Keep a pointer into each list. Whichever node has the smaller value gets appended next to the growing merged list. Advance only that list's pointer. When one list is exhausted, splice the remainder of the other directly — no element-by-element copying needed, because it is already sorted and already a linked chain. A dummy node at the front of the merged list eliminates the initial-head edge case.

python — merge two sorted lists

def mergeTwoLists(l1, l2):
    dummy = ListNode(0)         # sentinel head for the merged list
    tail = dummy
    while l1 and l2:
        if l1.val <= l2.val:
            tail.next = l1
            l1 = l1.next
        else:
            tail.next = l2
            l2 = l2.next
        tail = tail.next
    tail.next = l1 if l1 else l2   # splice remaining chain
    return dummy.next
Why the splice is safe

When one pointer reaches None, the other pointer still points to the front of a sorted chain. Because that chain is already sorted and already linked, you can splice the entire remaining chain in one step — tail.next = l1 if l1 else l2 — rather than walking it node by node. The dummy node meant we never had to special-case what the merged head would be.

05 Fast & slow

Finding the middle

You have a linked list and no length. How do you find the middle node in a single pass, without counting? You cannot binary-search into a linked list, so any approach that first computes the length and then re-walks to n/2 takes two passes. A fast-and-slow pointer pair does it in one.

Advance a slow pointer one node per step and a fast pointer two nodes per step. When fast runs off the end (or reaches the last node for an odd-length list), slow sits at the middle. For an even-length list there are two middle candidates; advancing while fast and fast.next both exist returns the first middle, which is the convention most problems expect. The same fast-and-slow pair underlies cycle detection (Lesson 11) — here we are seeing its simpler cousin.

python — find the middle node

def middleNode(head):
    slow = fast = head
    while fast and fast.next:   # fast needs 2 steps; guard both
        slow = slow.next
        fast = fast.next.next
    return slow                  # slow is now at the middle
Three questions with the same fast/slow muscle

Find the middle — stop when fast hits the end; slow is there. Detect a cycle — if slow and fast ever land on the same node, a cycle exists (they can only meet inside a loop). Find the n-th node from the end — advance fast n steps first, then move both together until fast hits null; slow is at the target. The same pair of pointers, three different stopping conditions.

★ Review

Summary & self-test

A linked list trades O(1) random access for O(1) insertion and deletion at any position where you hold a pointer. The three core techniques — the dummy node, the prev/cur/nxt reversal, and the fast/slow pair — handle the vast majority of linked-list interview problems when combined with a clear picture of what each pointer invariant guarantees at every step.
