Tree DFS & BFS Patterns

Trees look intimidating until you realize almost every tree problem is one of two questions: what value does each subtree aggregate upward (DFS), or what is true about each horizontal layer (BFS)? Master those two traversal shapes and a handful of close relatives, and you hold the key to an entire class of medium-to-hard interview problems.

00 The idea

What problem does it solve?

Almost every tree question asks you to compute something at every node and combine the results. Without a traversal strategy, you end up recomputing heights, sums, or depths from scratch at each level — that compounds to O(n²) in the worst case. The two canonical traversals each eliminate that redundant work in a different direction.

Depth-first search (DFS) recurses all the way down to the leaves, then collects answers on the way back up. It is the natural fit for anything that asks about a subtree's aggregate — height, sum of all nodes, diameter — because you need the children's answers before you can answer the parent. The call stack implicitly remembers where you are, so the code is usually five to ten lines of recursion with no auxiliary data structure.

Breadth-first search (BFS) visits all nodes level by level, using an explicit queue. It is the natural fit for anything that asks about horizontal layers — level averages, the right-side view, zigzag order — because you process every node at depth d before any node at depth d+1. The queue keeps track of the frontier; you drain it one level at a time.

  • Post-order DFS — children answer first, then the parent aggregates. The workhorse for height, diameter, path sum, LCA.
  • Pre-order DFS — the parent acts first, then visits children. Good for propagating a running value downward (path prefix, depth tag).
  • BFS with a level loop — drain all nodes at the current depth before enqueuing the next level. The workhorse for anything level-specific.
The decision in one line

If the answer at a node depends on its children's answers, use post-order DFS. If the answer at a node depends only on what its ancestors passed down, use pre-order DFS. If the answer is about an entire horizontal layer, use BFS. Almost every hard tree problem is a combination of exactly two of these.

01 Post-order aggregate

Recursion that bubbles answers upward

The most common DFS shape asks each node to return a value that its parent needs to compute its own answer. Height is the clearest example: to know how tall a subtree is, you first ask both children for their heights, then take the maximum and add one for the current node. The brute force — computing the height of each subtree from scratch whenever you need it — re-traverses overlapping paths and costs O(n²) on a balanced tree.

The recursive DFS solves it in O(n) by combining exactly this structure: each call handles one node, the two recursive calls handle its children, and the return value carries the answer up. No node is visited more than once. The call stack is O(h) deep, where h is the tree height (O(log n) balanced, O(n) worst-case skewed).

The reusable shape

Post-order recursion always follows the same skeleton: if not node: return base then left = dfs(node.left), right = dfs(node.right), then return combine(left, right, node.val). Height, sum, count, diameter, and LCA are all this shape — only base and combine differ.

Worked problemEasyLeetCode 104
Maximum Depth of Binary Tree

Given the root of a binary tree, return the number of nodes on the longest root-to-leaf path. An empty tree has depth 0.

Think it through

Restate

Count the height of the tree — the maximum number of edges on any root-to-leaf path, plus one (since depth counts nodes, not edges). An empty tree returns 0; a single node returns 1.

Brute force

Walk every root-to-leaf path, measure each length, keep the max. That visits every node at each ancestor — for a path of length h, the root is re-visited h times: O(n·h), which is O(n²) for a skewed tree.

The waste

The brute force recomputes each subtree's height from scratch. Post-order DFS computes each subtree exactly once and returns the answer directly: no node is visited more than once.

The move

If the node is null, return 0. Otherwise ask the left subtree for its height and the right subtree for its height (two recursive calls), take the max, and add 1 for the current node. That is the complete answer.

Invariant & cost

Every call handles exactly one node and makes two calls. The total work is proportional to the number of nodes: O(n) time, O(h) space for the call stack (O(log n) balanced, O(n) skewed).

The code

python

def maxDepth(root):
    if root is None:          # base case: empty subtree has height 0
        return 0
    left_h  = maxDepth(root.left)   # post-order: ask left child first
    right_h = maxDepth(root.right)  # then right child
    return max(left_h, right_h) + 1 # combine: taller subtree + this node
Narrate it to the interviewer

"I restate it as: find the height of the tree — the longest root-to-leaf node count. The naive path enumeration is O(n²) because it re-traverses overlapping segments. Instead I use post-order DFS: each call handles one node, recursing into both children first, then combines their heights. The base case for a null node is 0. The combine is max(left, right) + 1. Every node is visited exactly once, so it's O(n) time, O(h) space for the stack — O(log n) balanced, O(n) skewed."

Dry-run aloud: an empty tree (returns 0), a single node (left=0, right=0, returns 1), a skewed chain of 3 nodes (returns 3). Confirm the recursion bottoms out at null, not at leaves.

What the interviewer asks next

"Can you do it iteratively?" Yes — use a stack of (node, depth) pairs in a loop; push right then left child with depth+1; track the running max. Same O(n)/O(h) cost. "What if it's an n-ary tree?" The same shape: the base case is still not node → 0, and the combine is max(dfs(child) for child in node.children) + 1. "What about minimum depth?" Careful — the minimum is measured to the nearest leaf, so you must not count a path that ends at a node with only one child (that node isn't a leaf).

02 Direction of flow

Top-down vs. bottom-up: two sides of the same recursion

Post-order DFS gathers information from the bottom up — children answer first, then the parent combines. But some problems need information to flow in the opposite direction: a parent passes its own accumulated state down to its children so they can use it. These two directions correspond to the two natural recursive orderings, and recognizing which one a problem wants is often the key insight.

Bottom-up (post-order)

  • Children return a value; the parent aggregates.
  • Use when the answer at a node depends on what's below it: height, diameter, subtree sum, number of leaves.
  • Pattern: left = dfs(node.left); right = dfs(node.right); return combine(left, right).
  • The function's return value IS the answer for that subtree.

Top-down (pre-order)

  • Parent passes context; children use it.
  • Use when the answer at a node depends on what's above it: path prefix sum, depth tag, "is this a valid BST given the bounds from the parent".
  • Pattern: dfs(node.left, state + node.val); dfs(node.right, state + node.val).
  • The function writes into a shared result or checks a condition.
Worked problemMediumLeetCode 543
Diameter of Binary Tree

Given the root of a binary tree, return the length of the longest path between any two nodes. The path does not have to pass through the root. Length is measured in edges.

Think it through

Restate

The diameter is the longest path in the tree — measured in edges, not nodes. It might or might not pass through the root. At each node, the longest path through that node is left_height + right_height (the two longest arms reaching downward). I need the maximum of that across all nodes.

Brute force

For every node, compute left_height and right_height separately by re-running a height DFS each time. That's O(n) height calls each costing O(n) = O(n²) total.

The waste

Each height calculation re-traverses the same subtrees. If I compute height bottom-up and record the candidate diameter at each node in the same pass, I touch every node once.

The move

Carry a nonlocal variable best. In the post-order DFS, after computing left and right heights of the children, update best = max(best, left + right) — that is the path length through this node. Then return max(left, right) + 1 so the parent gets the correct height.

Invariant & cost

Every node is visited once. At each node, the candidate diameter (left + right) is recorded. The running max is the answer. O(n) time, O(h) space.

The code

python

def diameterOfBinaryTree(root):
    best = [0]                         # nonlocal via list (avoids global)

    def height(node):
        if node is None:
            return 0
        left  = height(node.left)
        right = height(node.right)
        best[0] = max(best[0], left + right)  # path through this node
        return max(left, right) + 1             # height upward to parent

    height(root)
    return best[0]
Narrate it to the interviewer

"I restate it as: at every node, the longest path through that node is left_height + right_height in edges. I need the global maximum across all nodes. The brute force computes height separately per node — O(n²). Instead I piggyback on a single post-order height traversal: after I get left and right heights from my children, I update a running best with left + right, then return max(left, right) + 1 to my parent. One pass, O(n) time, O(h) space."

Dry-run aloud: an empty tree (returns 0), a single node (left=0, right=0, best stays 0, correct), a root with left chain of 2 and right chain of 3 (best = 2+3 = 5). Confirm the path does not have to go through root — at some non-root node the two arms may add up to more.

03 BFS level-order

Expanding the frontier level by level

When you need something about entire rows of the tree — the values in each level, the average at each depth, whether two nodes are on the same row — DFS loses the level boundary information as it recurses depth-first. BFS preserves it naturally: you process every node at depth d completely before touching depth d+1, because the queue gives them to you in that order.

The trick for separating levels is the inner loop. At the start of each outer iteration, snapshot level_size = len(queue). The inner loop drains exactly that many nodes — all of depth d — and any children they enqueue belong to depth d+1. After the inner loop, one complete level has been processed. This adds zero extra cost; it is just a measurement of the queue at the right moment.

The reusable shape

Every BFS level-order solution is the same frame: queue = deque([root]), then while queue: size = len(queue); level = []; for _ in range(size): node = queue.popleft(); level.append(...); enqueue children. The outer while iterates once per level; the inner for drains exactly that level. Everything else is just what you record in level.

Worked problemMediumLeetCode 102
Binary Tree Level Order Traversal

Given the root of a binary tree, return a list of lists: each inner list contains the values of all nodes at one depth level, from top to bottom, left to right within each level.

Think it through

Restate

Group nodes by depth and return the groups in order. Within each group, preserve the left-to-right order. The output is a 2D list where result[d] holds all values at depth d.

Brute force

Run a DFS and store (node.val, depth) pairs, then sort or bucket by depth. The bucket approach is actually fine — O(n) — but the code is more tangled than BFS and it does not naturally preserve left-to-right order within a level without extra tracking.

The waste

DFS can reach a node at depth 3 before finishing all nodes at depth 2. BFS gives you depth-2 nodes in left-to-right order before any depth-3 node, matching the output exactly — no sorting, no bookkeeping.

The move

Use a deque. Seed it with the root. Each outer iteration snapshots size = len(queue), runs an inner loop that drains exactly size nodes (all at the current level), collects their values, and enqueues their children. Append the collected level list to the result after each outer iteration.

Invariant & cost

At the start of each outer iteration the queue holds exactly all nodes at depth d, in left-to-right order. Each node is enqueued and dequeued once: O(n) time, O(w) space where w is the maximum level width (O(n) in the worst case for the last row of a full tree).

The code

python

from collections import deque

def levelOrder(root):
    if root is None:
        return []
    result = []
    queue  = deque([root])
    while queue:
        size  = len(queue)          # snapshot: all nodes at current depth
        level = []
        for _ in range(size):
            node = queue.popleft()
            level.append(node.val)
            if node.left:  queue.append(node.left)
            if node.right: queue.append(node.right)
        result.append(level)        # one complete level collected
    return result
Narrate it to the interviewer

"I restate it as: group nodes by depth, left to right within each group. BFS gives me exactly that — the queue holds each layer in order. I seed the queue with the root, then in the outer loop I snapshot the current size, run an inner loop that drains exactly those nodes (collecting values and enqueuing their children), then append the collected list. The invariant is that at the top of the outer loop, the queue holds all nodes at depth d in left-to-right order. Each node is enqueued and dequeued once, so it's O(n) time, O(w) space — where w is the widest level."

Dry-run aloud: an empty tree (returns []), a single node (returns [[val]]), a perfect 7-node tree with 3 levels (returns [[root], [l, r], [ll, lr, rl, rr]]). Also check that if a node has only a left child, right children are not enqueued.

What the interviewer asks next

"Return the level order in reverse?" — collect levels exactly the same way, then reverse the result list at the end, or prepend each level to a deque. "Right side view?" — from each level, keep only the last element. "Zigzag order?" — alternate whether you append left-to-right or right-to-left; a simple boolean flag flipped each outer iteration handles it. All three are one-line changes on the same BFS frame.

04 Finding where paths cross

Lowest common ancestor

The lowest common ancestor (LCA) of two nodes p and q in a binary tree is the deepest node that has both p and q among its descendants (a node is considered its own descendant). The naive approach is O(n²): for every node, trace both p's and q's ancestor chains and find the first node that appears in both. You re-walk the tree for every candidate.

The elegant solution is a single post-order DFS pass. Each call returns a value that carries three possible meanings: "I found p", "I found q", or "I found their meeting point (the LCA)". The key insight is that the LCA is the first node where one subtree contains p and the other contains q — you know this because the left and right recursive calls each return a non-null value at that point.

Worked problemMediumLeetCode 236
Lowest Common Ancestor of a Binary Tree

Given the root of a binary tree and two nodes p and q that are guaranteed to exist in it, return the deepest node that is an ancestor of both p and q (a node counts as its own ancestor).

Think it through

Restate

Find the deepest node such that both p and q are in its subtree (counting itself). This node is either the first fork point between p and q, or it is p or q itself (if one is an ancestor of the other).

Brute force

Walk from root to p, record the ancestor chain. Walk from root to q, record its chain. Find the last common element. Two O(n) walks: O(n) time but also O(n) space for storing both ancestor chains. The code is also non-trivial to get right.

The waste

We re-traverse the tree twice. A single post-order pass can locate both p and q simultaneously and identify where their subtrees meet.

The move

Each call returns the node itself if it is p or q; otherwise it returns whatever non-null value its children found. If both children return non-null, the current node is the fork point — return it as the LCA. If only one child returned non-null, propagate that value upward (the LCA is somewhere below, already identified).

Invariant & cost

The LCA is returned from the exact node where it occurs and propagates up unchanged. Every node is visited once: O(n) time, O(h) space for the call stack.

The code

python

def lowestCommonAncestor(root, p, q):
    if root is None:
        return None
    if root is p or root is q:   # this node is one of the targets
        return root
    left  = lowestCommonAncestor(root.left,  p, q)
    right = lowestCommonAncestor(root.right, p, q)
    if left and right:             # p in left subtree, q in right (or vice versa)
        return root                # this node is the fork → LCA
    return left if left else right # propagate the non-null hit upward
Narrate it to the interviewer

"I restate it as: find the deepest node that has both p and q in its subtree — a node counts as its own ancestor. The brute force records two ancestor chains in O(n) space. Instead I use a single post-order DFS: if the current node is p or q, I return it immediately — any deeper nodes won't matter for this branch. After recursing left and right, if both return non-null, this is the fork point and I return this node as the LCA. If only one is non-null, the LCA is already resolved below, and I bubble it up. One pass, O(n) time, O(h) space."

Dry-run aloud: p is an ancestor of q (when we reach p, we return it before finding q — correct since the problem says a node is its own ancestor); p and q on opposite sides of the root (both children return non-null at root — root is the LCA); p == q (the first encounter returns p immediately).

What the interviewer asks next

"What if p or q might not exist in the tree?" You need to track whether each target was found — return a (node, found_p, found_q) tuple instead of a single node. "What about a BST?" Exploit the ordering: if both p.val and q.val are less than root.val, descend left; if both are greater, descend right; otherwise root is the LCA. That's O(h) instead of O(n). "What if it's an n-ary tree?" Iterate over all children and count how many return non-null; if two or more do, the current node is the LCA.

05 Path problems

Root-to-leaf paths and the maximum path sum

A common extension of post-order DFS is the path problem: instead of a single aggregate value per subtree, you need the value of the best path that passes through or starts at each node. "Best" might mean longest, maximum sum, or matching a target sum.

Two shapes dominate. Root-to-leaf paths use top-down DFS — the prefix sum (or prefix path) is accumulated as a parameter passed down, and whenever you hit a leaf you check the full path value. Maximum path sum through any node uses bottom-up DFS — each call returns the best single arm extending downward, and updates a global best using left_arm + node.val + right_arm before returning the longer single arm to the parent.

Why arms, not full paths

A path can only go in one direction — it cannot branch. So each call returns the best single arm (max(left_arm, right_arm) + node.val), which the parent can extend. But locally, the two arms can be combined: left_arm + node.val + right_arm is a valid path through this node, and you check it against your global best before discarding the shorter arm.

Worked problemHardLeetCode 124
Binary Tree Maximum Path Sum

Given the root of a binary tree where node values can be negative, return the maximum sum of any non-empty path. A path is any sequence of nodes where each adjacent pair is connected by an edge; it does not have to pass through the root or touch any leaf.

Think it through

Restate

A path is any connected chain of nodes (no branching). I want the maximum sum over all such paths. Paths may be entirely in a subtree, may pass through the root, or may be a single node. Node values can be negative.

Brute force

Enumerate all O(n²) pairs of nodes and find the maximum-sum path between each pair. Computing one path costs O(n), so the total is O(n³).

The waste

Most paths are re-examined repeatedly. A single post-order pass, where each node returns its best contribution-to-a-parent path, computes the answer for every possible through-node in O(1) extra work per node.

The move

Each call returns the best single arm: max(0, left_arm, right_arm) + node.val. The zero floor means we discard a negative arm (it's better to start fresh at this node). Before returning, update best with left_arm + node.val + right_arm — the full path through this node using both arms (both floored at 0).

Invariant & cost

Each node is visited once. The running best captures the maximum path through every node. O(n) time, O(h) space. The zero floor correctly handles all-negative trees: the best single-node path has no arms, and the global best is the largest single node value.

The code

python

def maxPathSum(root):
    best = [float('-inf')]          # track global maximum

    def arm(node):
        if node is None:
            return 0
        left  = max(0, arm(node.left))   # discard negative arms
        right = max(0, arm(node.right))
        best[0] = max(best[0], left + node.val + right)  # path through node
        return max(left, right) + node.val               # best single arm up

    arm(root)
    return best[0]
Narrate it to the interviewer

"I restate it: find the maximum-sum non-branching path anywhere in the tree. Negative values make this non-trivial. My key insight is that each node should contribute to a parent path only if doing so increases the sum — so I floor each arm at 0. In a post-order DFS, each call gets the best non-negative arm from each child, records left + node.val + right as the candidate path through this node, then returns max(left, right) + node.val — the best single arm extending upward. The global best captures the largest through-path candidate from every node. One pass: O(n) time, O(h) space."

Dry-run aloud: all-negative tree (flooring at 0 means no arms are used; the best path is the single least-negative node — the initial value of best is −inf, so the first candidate replaces it); a tree with a single node (arm returns node.val, best is set to 0 + node.val + 0 = node.val, correct); a path that passes through neither the root nor any leaf (the algorithm still finds it, since it checks every node as a potential through-point).

06 Practice

Practice ladder

The patterns here repeat across many problems — height, diameter, LCA, level order, path sum. Work through these drills in order. For each one: state the traversal direction first (top-down or bottom-up? DFS or BFS?), then the return value and invariant, then code it. Easy problems should take under 15 minutes; medium under 30.

EasyMaximum Depth of Binary Treepost-order DFS

Return the length of the longest root-to-leaf path, measured in nodes.

Idea Post-order: return max(left, right) + 1; base case null returns 0. O(n) time.

EasySame Treesimultaneous DFS

Decide whether two binary trees are structurally identical with the same node values at every position.

Idea Recurse both trees in lockstep: if both null → true; if one null or values differ → false; else recurse left and right pairs. O(n).

EasySymmetric Treemirror DFS

Check whether a binary tree is a mirror image of itself around its center.

Idea Write a helper isMirror(left, right) that checks left.val == right.val and recurses isMirror(left.left, right.right) and isMirror(left.right, right.left). O(n).

EasyInvert Binary Treepre-order DFS

Flip a binary tree around its vertical axis so every left child becomes a right child and vice versa.

Idea Swap node.left and node.right, then recurse into both. Pre-order or post-order both work; O(n) time.

EasyPath Sumtop-down DFS

Given a root and a target integer, decide whether any root-to-leaf path has values summing exactly to that target.

Idea Pass remaining = target - node.val downward; at a leaf, check remaining == 0. O(n) time, O(h) space.

MediumBinary Tree Level Order TraversalBFS

Return a 2D list where each inner list holds all node values at one depth level, top to bottom, left to right.

Idea BFS with inner loop: snapshot size = len(queue), drain size nodes per outer iteration, enqueue children. O(n).

MediumDiameter of Binary Treepost-order DFS

Return the number of edges on the longest path between any two nodes (the path need not pass through the root).

Idea Single post-order pass: track best = max(best, left + right) at each node; return max(left, right) + 1. O(n).

MediumLowest Common Ancestor of a Binary Treepost-order DFS

Given two guaranteed-present nodes p and q, return their deepest shared ancestor.

Idea Return node if it matches p or q; if both children return non-null, this node is the LCA; otherwise propagate the non-null child upward. O(n).

MediumCount Good Nodes in Binary Treetop-down DFS

A node is "good" if its value is greater than or equal to every ancestor on the path from the root. Count such nodes.

Idea Pass the running max downward; at each node, if node.val >= running_max, it is good — increment count and update max. O(n).

HardBinary Tree Maximum Path Sumpost-order DFS

Node values may be negative. Return the maximum sum of any non-empty, non-branching path anywhere in the tree.

Idea Each call returns max(0, left_arm, right_arm) + node.val; update global best with left + node.val + right before returning. O(n).

Idea Pre-order DFS: serialize as "val,null,null,..." with null markers; deserialize by consuming tokens from a queue left-to-right. O(n) both directions.

How to drill these problems

For every problem:
(1) decide the traversal direction first — does the answer flow up (post-order), down (pre-order), or across (BFS)? Name it out loud.
(2) State the return value and what it means for one node.
(3) Write the base case (null node) first.
(4) Write the recursive case, combining children.
(5) State the invariant — what is always true when the function is called?
(6) Call out edge cases before you run: empty tree, single node, all-negative values (path sum), a perfect balanced tree vs. a skewed chain. Skipping step 1 is the most common mistake — pattern recognition before coding.

★ Review

Summary & self-test

Tree problems reduce to a small number of traversal frames. Post-order DFS handles anything that depends on subtree aggregates — height, diameter, LCA, maximum path sum — because children answer before the parent combines. Pre-order DFS handles anything that depends on ancestor context — path prefix, depth tag, BST bounds — because the parent injects state before the children are called. BFS handles anything level-specific — layer values, right-side view, zigzag — because the inner-loop trick guarantees each level is completely drained before the next begins.

01Post-order defaultIf the answer at a node requires its children's answers first, write bottom-up post-order DFS.
02Pre-order for contextIf children need something from their ancestors (running max, depth), pass it as a parameter top-down.
03BFS for layersLevel-specific questions need BFS with the inner-loop size snapshot to separate each depth band.
04Return value is the interfaceIn post-order, the function's return value is the subtree's answer. Decide what it means before writing a line of code.
05LCA: both sides non-nullThe LCA is the unique node where one child returned p and the other returned q — detected by both-non-null.
06Path sum: arms not full pathsA path cannot branch — return the best single arm upward, but evaluate the full through-path (both arms) locally.
07Zero-floor negative armsFor max path sum, floor each arm at 0 before combining. A negative arm always hurts — start fresh at this node instead.
08ComplexityMost tree DFS/BFS: O(n) time, O(h) space for stack (O(log n) balanced, O(n) skewed). BFS: O(w) space where w is max level width.
