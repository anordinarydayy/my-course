Greedy

A greedy algorithm makes the locally best choice at each step and never revisits it. The whole challenge — and the entire interview skill — is not picking the right choice; it is arguing that the locally optimal choice is also globally safe, so greed never costs you the best solution.

00 The idea

What does "greedy" mean — and how do you trust it?

Suppose you are planning a road trip and each city has a sign telling you how far you can drive from there. You want to know: can you reach the last city at all? A greedy approach says: at every city, just update the furthest point you know you can reach. You never backtrack to reconsider a city. The question is — is that safe?

The classical proof technique is the exchange argument. You compare your greedy solution to any optimal solution and show that swapping any optimal choice for the greedy one never makes things worse. If you can always make that swap, the greedy solution must be at least as good as the optimal — which means it is optimal. Greedy interviews live or die on whether you can sketch this argument out loud, not just code the loop.

When does greedy fail? When the locally best choice now closes off a better path later — a classic trap is making change with arbitrary coin denominations. With coins {1, 3, 4} and a target of 6, greedy takes 4 first, then two 1s = three coins; but {3, 3} uses only two. Greedy failed because grabbing the biggest coin was locally optimal but globally wasteful. Whenever you sense that a current choice might "use up" a resource needed later, you likely need dynamic programming instead.

Use Greedy when

  • You can rank choices by a single key and the best current choice never hurts future choices.
  • The exchange argument goes through: swapping greedy ↔ optimal leaves the total cost the same or better.
  • Subproblems are independent — the choice at step i does not change what is possible at step i+1 in a harmful way.

Switch to DP when

  • The best choice depends on what comes after — you cannot decide locally without global information.
  • Counterexamples exist: a small greedy simulation shows a case where greedy mis-picks.
  • The problem involves counting ways, not just finding the best value; or coin/cut problems with non-standard sets.
The two-sentence template for any greedy answer

After stating the choice: "This greedy choice is safe because: if any optimal solution made a different choice here, I could swap in the greedy choice and the result would be no worse — so the greedy solution is also optimal." Practice saying that out loud before the interview.

01 Reachability frontier

Jump Game — can you reach the end?

Jump Game (LeetCode 55): each cell holds the maximum number of steps you can jump forward from it. Starting at index 0, can you ever reach the last index?

The brute force tries every possible sequence of jumps via recursion or BFS, which is O(n²) or worse in the worst case. It is wasteful because it re-explores paths that clearly can't matter once we know how far we can already reach.

The insight: you do not need to track which exact path gets you there. You only need to track the furthest index you can reach so far, call it frontier. Walk left to right. At each index i, if i > frontier you are stuck — you can never get here. Otherwise update frontier = max(frontier, i + nums[i]). If frontier ever reaches or passes the last index, return true. This is greedy because at each step you pick the most optimistic reachability, and the exchange argument holds: any path that reaches a cell contributes the same update to the frontier as any other path — you can never do better than the maximum, so you always take it.

Why the frontier update is globally safe

At cell i, the furthest we can possibly reach from here is i + nums[i]. Any path that previously brought us to cell i — no matter its history — gives us exactly this opportunity. Taking the maximum over all reachable cells is therefore never wrong: the frontier is a tight lower bound on "how far could any valid sequence of jumps reach." If it ever hits the last index, some sequence of valid jumps achieves it.

Worked problemMediumLeetCode 55
Jump Game

You are given an array of non-negative integers. Each value is the maximum number of steps you may jump forward from that position. Starting at index 0, decide whether it is possible to reach the last index.

Think it through

Restate

Can any valid sequence of jumps get us from index 0 to index n-1? We only need a yes/no answer, not the actual path.

Brute force

Try every combination of jumps using DFS/recursion. From each index we try every jump length from 1 to nums[i]. This is O(n²) in the worst case (e.g. all cells equal to n), and it revisits the same reachability facts repeatedly.

The waste

We re-explore subproblems we already know the answer to. The only thing that matters is: what is the furthest index any valid prefix of jumps can reach? That single number summarises all paths simultaneously.

The move

Maintain frontier = 0. For each index i from 0 to n-1: if i > frontier, we can never reach here — return False immediately. Otherwise, frontier = max(frontier, i + nums[i]). If frontier reaches n-1, return True.

Invariant & cost

frontier is always the maximum index reachable by any valid jump sequence seen so far. Every index is visited once: O(n) time, O(1) space.

The code

python

def canJump(nums):
    frontier = 0              # furthest index any jump chain can reach
    for i in range(len(nums)):
        if i > frontier:       # can't reach this cell — stuck
            return False
        frontier = max(frontier, i + nums[i])
        if frontier >= len(nums) - 1:
            return True
    return True               # loop finished without getting stuck
Narrate it to the interviewer

"I need to decide if any sequence of valid jumps reaches the last index. The brute force tries every path — O(n²) — but it's wasteful because multiple paths converge to the same reachability fact. So I maintain a single number: frontier, the furthest index any valid jump chain can reach so far. I walk left to right; if I'm ever at a cell past the frontier, I'm stuck, return False. Otherwise I update the frontier. The exchange argument: taking the max is always optimal — no path can reach farther than i + nums[i] from cell i, so I'm not leaving any potential on the table. O(n) time, O(1) space."

Dry-run aloud: single-element array (already at the end — True); all zeros except index 0 (frontier = nums[0], check if it covers n-1); a zero in the middle that blocks progress — make sure i > frontier catches it.

What the interviewer asks next

"What if nums[i] is the exact number of steps, not the maximum?" → then we need dynamic programming because each cell offers only one option, not a range; the greedy overestimate would be wrong. "What if we want the path, not just True/False?" → store a parent array while scanning and reconstruct backwards. "Can the array contain negative values?" → the problem guarantees non-negatives; with negatives the whole model breaks because jumping backwards isn't defined.

02 Fewest jumps

Jump Game II — the minimum number of jumps

Jump Game II (LeetCode 45) adds a count: find the minimum number of jumps to reach the last index, given that it is always reachable. The brute force is BFS on the jump graph — each node is an index, each edge is a valid jump — which runs in O(n²) in the worst case because of the dense edge set.

The greedy insight treats this like a BFS with variable-width layers. At each jump count jumps, the "current layer" is the range of indices we can reach in exactly that many jumps. The end of the current layer is the rightmost index we can be at after jumps jumps. The next layer's end is the furthest index reachable from anything in the current layer — computed greedily as we scan. When we exhaust the current layer (index passes cur_end), we need one more jump to move to the next layer, so we increment jumps and slide cur_end to the best frontier found so far.

The exchange argument: within the current layer, you always want to extend the next layer's reach as far as possible. Using a shorter reach would only make future layers narrower — it can never help. So the greedy choice (always prefer the furthest reachable point) is globally optimal.

Worked problemMediumLeetCode 45
Jump Game II

Same jump array as before, but now you must return the minimum number of jumps needed to reach the last index. The last index is always reachable.

Think it through

Restate

Reachability is guaranteed. I want the count of jumps, minimised. Think of it as BFS on an implicit graph — each level is one more jump — but done greedily instead of storing a queue.

Brute force

BFS: enqueue index 0 at level 0; from each index enqueue all reachable successors at level+1; stop when the last index is dequeued. Correct, but building and traversing the edge set is O(n²) in the worst case.

The waste

BFS wastes time storing all indices in each level individually. Because jumps are ranges (from index i you can go anywhere in [i+1, i+nums[i]]), entire layers collapse into a single interval — just track the interval boundary.

The move

Keep three variables: jumps = 0, cur_end = 0 (right edge of the current layer), farthest = 0 (right edge of the next layer, updated greedily). Walk every index i. Update farthest = max(farthest, i + nums[i]). When i == cur_end and we haven't reached the last index yet, we must jump: jumps += 1; cur_end = farthest.

Invariant & cost

After processing each layer, cur_end is the furthest index reachable in jumps jumps. We never miss a cell, and we never process a cell twice: O(n) time, O(1) space.

The code

python

def jump(nums):
    jumps = 0
    cur_end = 0      # rightmost index we can reach in 'jumps' jumps
    farthest = 0    # best right edge of the next layer
    for i in range(len(nums) - 1):   # last index: already there, no jump needed
        farthest = max(farthest, i + nums[i])
        if i == cur_end:              # exhausted this layer — must jump
            jumps += 1
            cur_end = farthest
            if cur_end >= len(nums) - 1:
                break
    return jumps
Narrate it to the interviewer

"I model this as BFS by jump count, but I collapse each BFS level into a single boundary value instead of a queue. cur_end is the right edge of the cells I can stand on after jumps jumps; farthest is the best right edge of the next layer, updated greedily as I scan. When I exhaust the current layer — when index i reaches cur_end — I increment jumps and set the new layer boundary to farthest. The greedy choice is safe: within a layer you always want the widest possible reach for the next layer; a narrower choice can never help. O(n) time, O(1) space."

Dry-run: single element (0 jumps); nums = [1,1,1,1] (each layer width 1, 3 jumps); nums = [4,0,0,0,1] (one jump from 0 covers everything). Watch the loop stop at len-2 — you don't need to jump away from the last index.

What the interviewer asks next

"What if some cells are unreachable?" → the loop would exit with cur_end never reaching n-1; return -1 or handle it like Jump Game I (check frontier). "Is this the same as BFS?" → yes, conceptually: each layer is one BFS level; we just avoid building an explicit queue by tracking only the layer boundary. "Can you do it with DP?" → yes, dp[i] = min jumps to i, filled left to right; it's O(n²) because each dp[i] scans backwards. The greedy is strictly faster.

03 Interval scheduling

Pick the most intervals — the "earliest end first" rule

Classic interval scheduling: given a list of intervals, select the maximum number of non-overlapping intervals. The brute force is exponential — try all subsets. Even sorting and DP is O(n²) if done naively.

The greedy insight: always pick the interval that ends earliest. Why? If an optimal solution does not include the earliest-ending interval, we can swap in the earliest-ending interval for whatever the optimal picks first — and the swap can only free up more time for future intervals (the new first interval ends no later than the one we removed). The exchange argument goes through: every swap preserves or improves the count. Repeat until greedy equals optimal — greedy is optimal.

Algorithm: sort intervals by end time. Walk through them keeping a variable last_end. For each interval, if its start is at or after last_end, take it (it doesn't overlap with the last selected interval) and update last_end. Otherwise skip it — it overlaps. Total: O(n log n) for the sort, O(n) for the scan.

The reusable shape: earliest end picks the most room

The earliest-ending interval leaves the most timeline open for future intervals. Any other first pick would end later, blocking more of the future. This generalises: whenever you need to greedily "leave the most room for what comes next," sort by end time and take the earliest-ending compatible item. This same logic underlies task scheduling, meeting room allocation, and activity selection problems.

What the interviewer asks next

"Minimum rooms for non-overlapping meetings?" → count how many intervals overlap at any point; use a min-heap of end times. "Merge overlapping intervals?" → sort by start; merge if start[i] ≤ end[i-1]. "What if intervals have weights?" → greedy fails; need weighted interval scheduling DP, O(n log n) with binary search. These are the three variants the interviewer cycles through on this topic.

04 Circular greedy

Gas Station — finding the one valid start

Gas Station (LeetCode 134): you have n stations arranged in a circle. Station i provides gas[i] litres and costs cost[i] to travel to the next station. Starting with an empty tank, find the unique starting station from which you can complete the full circle, or return -1 if impossible.

The brute force tries every starting station and simulates the trip: O(n²). The waste is in re-simulating prefixes we have already checked. The key observations are two greedy facts working together:

  • Global feasibility: if the total gas minus total cost is negative, no solution exists. If the total is non-negative, a solution always exists and is unique.
  • Starting point: walk the stations in order, tracking tank (the running surplus). Whenever tank goes negative, all stations from the current candidate start up to the current station are invalid starters — the candidate start could not have accumulated enough surplus. Reset the candidate to the next station and reset tank to 0.

The exchange argument: any starting point before the reset point can be ruled out because simulating from there would accumulate the same negative prefix — it's the same as the current candidate plus some additional deficits before it, which only makes things worse. So skipping to the next station after a failure is safe.

Worked problemMediumLeetCode 134
Gas Station

An array of n gas stations is arranged in a circle. Station i provides gas[i] fuel and costs cost[i] to travel to station (i+1) % n. Your tank starts empty and has unlimited capacity. Return the index of the station from which you can complete a full loop, or -1 if no such starting point exists. If a solution exists, it is unique.

Think it through

Restate

Find a starting station (or declare -1) such that circling all n stations leaves the tank non-negative at every intermediate point. Solution is unique if it exists.

Brute force

For each of the n starting stations, simulate the full circuit. If the tank ever goes negative, that start fails. This is O(n²) — too slow for n up to 10⁵.

The waste

When we fail starting from station k at station j, we repeat the k → j work for every subsequent start between k and j — but any start in that range inherits the deficit we just measured. We can skip the whole block at once.

The move — phase 1 (check feasibility)

If sum(gas) < sum(cost), return -1 immediately. The total surplus being negative means no circular trip is possible regardless of starting point.

The move — phase 2 (find start)

Set tank = 0, start = 0. Walk each station; add gas[i] - cost[i] to tank. If tank < 0, the current candidate start is invalid — reset start = i + 1 and tank = 0. The first valid candidate that survives to the end is the answer.

Invariant & cost

After the feasibility check guarantees a solution exists, the greedy scan finds it. Every station is visited once: O(n) time, O(1) space.

The code

python

def canCompleteCircuit(gas, cost):
    # Phase 1: global feasibility
    if sum(gas) < sum(cost):
        return -1
    # Phase 2: find the valid start
    tank, start = 0, 0
    for i in range(len(gas)):
        tank += gas[i] - cost[i]
        if tank < 0:          # start..i block is all invalid
            start = i + 1   # try starting after the failure point
            tank = 0
    return start
Narrate it to the interviewer

"Brute force tries each of n starts — O(n²). The insight is two-part. First: if total gas minus total cost is negative, it is globally impossible — return -1. Second: I walk the stations tracking a running surplus. When the surplus goes negative, every station in the current candidate window is proven invalid as a start — they would accumulate the same deficit. So I reset the candidate start to the next station and clear the tank. Because the total is non-negative, the greedy scan always lands on the unique valid start. O(n) time, O(1) space."

Dry-run: single station with gas = cost (start = 0, surplus = 0, round trip works); all stations at a deficit except one (find the one surplus station); total exactly 0 (greedy still finds start correctly).

05 Match-up greedy & limits

Assign Cookies — and when greedy breaks down

Assign Cookies (LeetCode 455): each child has a "greed factor" — the minimum cookie size that satisfies them. Each cookie has a size. Assign at most one cookie per child to satisfy as many children as possible.

The greedy move: sort both children and cookies. Use two pointers — one for the smallest unsatisfied child, one for the smallest unused cookie. If the current cookie satisfies the current child, assign it and advance both; otherwise advance only the cookie pointer (the cookie is too small for this child, but maybe for none — discard it). This is the "match the smallest sufficient resource to the smallest demand" pattern.

Exchange argument: suppose an optimal solution assigns a large cookie to a small child while leaving a smaller cookie for a larger child. Swapping the cookies (give the small cookie to the small child, the large to the large) satisfies the same children — or more, because the large cookie can still satisfy the large child. So sorting and matching greedily never loses anything. O(n log n + m log m) sort, O(n + m) scan.

When greedy fails — the DP boundary

The question every interviewer will eventually ask: "could you have solved this greedily?" Here is how to diagnose it quickly at the board:

  • Try a small counterexample. If you can construct even one case where the greedy choice gives a suboptimal result, greedy is wrong.
  • Check for "choice now restricts options later." In coin change with denominations {1, 3, 4} targeting 6, choosing 4 (greedy max) blocks you from the {3, 3} solution. The choice now permanently changes what remains.
  • Check subproblem independence. In Jump Game, knowing the frontier is enough — history doesn't matter. In 0/1 Knapsack, what you took earlier changes what capacity remains, so each choice is not independent.
  • Can you write the exchange argument? If swapping the greedy choice with any other choice and claiming "it's no worse" feels false, greedy is likely wrong.
The diagnostic in one question

Ask yourself: "Does using option A now make option B strictly worse later, in a way that option B alone cannot compensate for?" If yes — DP. If no — greedy may work. Build a two-element or three-element counterexample before committing; counterexamples are faster than proofs to find and equally decisive.

06 Practice

Practice ladder

Each card is a well-known interview problem described in original words. For every one: state the brute force and its big-O first, identify the waste, then name the greedy move and argue in one sentence why it is safe. Code it only after you can narrate all three. Aim for easy cards in under 15 minutes, medium under 30.

EasyAssign CookiesLeetCode 455 · match smallest

Satisfy as many children as possible by assigning at most one cookie to each; a cookie satisfies a child only if the cookie's size meets the child's minimum.

Idea Sort both arrays; match smallest sufficient cookie to each child in order. O(n log n).

EasyLemonade ChangeLeetCode 860 · greedy cash

Customers pay with $5, $10, or $20 bills; each lemonade costs $5. Decide if you can always make exact change, given bills arrive in order.

Idea When making change for $20, prefer to use a $10+$5 rather than three $5s (keeps $5s for future $10 customers). O(n).

EasyMaximum Units on a TruckLeetCode 1710 · unit rate

Each box type has a count and a unit count per box; a truck holds at most a fixed number of boxes. Maximise total units loaded.

Idea Sort boxes by units per box descending; take as many of the highest-density type as possible before moving down. O(n log n).

MediumJump GameLeetCode 55 · frontier

Given a jump array, determine if the last index is reachable.

Idea Track the furthest reachable index; if you ever scan past it, return False. O(n).

MediumJump Game IILeetCode 45 · BFS layers

Return the minimum number of jumps to reach the last index (always reachable).

Idea Collapse BFS levels into layer boundaries; increment jump count when you exhaust a layer. O(n).

MediumGas StationLeetCode 134 · circular reset

Find the unique starting station for a circular gas-station tour, or return -1 if impossible.

Idea Check global feasibility first; then reset the candidate start every time the running surplus goes negative. O(n).

MediumNon-overlapping IntervalsLeetCode 435 · earliest end

Given a set of intervals, find the minimum number to remove so no two overlap.

Idea Count how many you can keep (earliest-end greedy from FIG. 3); answer = n − kept. O(n log n).

MediumMeeting Rooms IILeetCode 253 · min-heap

Find the minimum number of conference rooms needed to schedule all meetings without conflicts.

Idea Sort by start; min-heap of end times — if the earliest-ending meeting finishes before the next one starts, reuse the room. O(n log n).

MediumTask SchedulerLeetCode 621 · frequency greedy

Schedule CPU tasks with a cooldown period n between identical tasks; minimise total time including idle slots.

Idea Always schedule the most frequent remaining task first; the answer depends on whether the most frequent task forces idles. O(m) with counting.

MediumBoats to Save PeopleLeetCode 881 · two ends

Each boat carries at most two people subject to a weight limit; minimise boats needed.

Idea Sort; try pairing the heaviest with the lightest. If they fit, use one boat for both; otherwise the heaviest goes alone. O(n log n).

HardIPOLeetCode 502 · two heaps

You can complete at most k projects; each has a capital requirement and a profit. Maximise total capital after k projects.

Idea Min-heap of capitals (unlock projects you can afford); max-heap of profits (always take the highest-profit affordable project). O(n log n + k log n).

HardCandyLeetCode 135 · two passes

Distribute the minimum number of candies to children in a row; each must get at least one, and a child with a higher rating than a neighbour must get more.

Idea One left-to-right pass for the left-neighbour constraint; one right-to-left pass for the right-neighbour constraint; take the max at each position. O(n).

How to drill greedy problems

The interview trap is jumping straight to the greedy move without justifying it. Train these five beats on every problem:
(1) state the brute force and its big-O;
(2) name the waste;
(3) state the greedy choice;
(4) give the exchange argument in one sentence;
(5) code it and dry-run the edge cases — empty input, a single item, all items identical, the boundary where a reset or layer-change happens. If you cannot complete beat 4, the greedy might be wrong — build a small counterexample before coding.

★ Review

Summary & self-test

Greedy compresses to one discipline: make the locally optimal choice, then prove it is globally safe with the exchange argument. The code is almost always a short scan with one variable tracking the greedy state. The intellectual work — and the interview substance — is the argument, not the loop.

Self-test before you move on

Cover the notes and answer aloud:
(1) What is the exchange argument — state it as a general procedure, not just for one problem?
(2) In Jump Game, what does the variable frontier represent, and why is updating it greedily (taking the max) globally safe?
(3) How does Jump Game II collapse an O(n²) BFS into O(n) — what are the two boundary variables and when does the jump count increment?
(4) For interval scheduling, why is "earliest end" the correct sort key rather than "earliest start" or "shortest duration"?
