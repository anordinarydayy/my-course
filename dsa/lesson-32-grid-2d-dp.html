Grid & 2D DP

A grid cell's answer depends only on what came from above and from the left — that single observation converts a family of exponential search problems into a table you fill once, row by row, in polynomial time.

Original study notes. Written for coding-interview preparation. Problem names such as Unique Paths and Maximal Square are referenced only as standard identifiers; every description, diagram, walkthrough, and line of code here is original. Covers P7 · Dynamic Programming in the roadmap. Contains three interactive diagrams (press Play or step with Next) and four fully worked problems — each with the reasoning chain, Python code, and an interviewer narration script.
00 The idea

State over a 2-D table

Grid DP problems share a very specific shape: you have an m × n grid and you want some quantity for the bottom-right corner (or for every cell). The naive approach — enumerate every possible path or configuration — grows exponentially with grid size. The saving observation is that the answer for any cell depends only on a tiny neighbourhood: usually just the cell directly above and the cell directly to the left. That is the recurrence.

Define dp[i][j] as the answer for cell (i, j). Base cases are the top row and the left column, which each have only one neighbour to draw from. Then sweep left-to-right, top-to-bottom and every cell is resolved in O(1) work. Total cost: O(m × n) time, O(m × n) space — or O(n) space with a rolling row.

The reusable shape

Every problem in this lesson is a variation on dp[i][j] = f(dp[i-1][j], dp[i][j-1], grid[i][j]). The function f changes — sum for counting paths, min for cost, something cleverer for squares — but the sweep order and the base-case discipline are identical every time.

01 Counting routes

Unique paths — counting with addition

A robot sits in the top-left corner of an m × n grid. It can only move right or down on each step. How many distinct routes lead to the bottom-right corner?

The brute-force answer is to enumerate every route recursively — but the number of routes is combinatorially huge, growing as C(m+n-2, m-1). With memoization, the recursion becomes the same O(m × n) DP, so we might as well think about the table directly.

The recurrence is clean: dp[i][j] = the number of distinct routes to reach (i, j). The robot arrived either from above ((i-1, j)) or from the left ((i, j-1)), and those paths are disjoint, so dp[i][j] = dp[i-1][j] + dp[i][j-1]. Every cell on the top row and every cell on the left column has exactly one route to it (go straight across or straight down), so those base cases are all 1.

Worked problemEasyLeetCode 62
Unique Paths

A robot starts at the top-left of an m × n grid and can only move right or down one cell at a time. Count the number of distinct paths that reach the bottom-right cell.

Think it through

Restate

Count routes from (0,0) to (m-1, n-1) using only right/down moves — a classic combinatorics question phrased as a grid.

Brute force

Recursively try every right/down split at each cell. The recursion tree has depth m+n-2 with branching up to 2, giving O(2^(m+n)) calls before memoization. For a 20×20 grid that is astronomical.

The waste

The recursion recomputes the count for every cell from scratch whenever multiple paths converge on it. If we build the table bottom-up, each cell is filled exactly once — the overlap disappears.

The move

Allocate a 2-D table dp[m][n]. Seed the entire top row and left column with 1 (only one straight path reaches each). Then for i from 1 to m-1 and j from 1 to n-1: dp[i][j] = dp[i-1][j] + dp[i][j-1]. Return dp[m-1][n-1].

Invariant & cost

At the start of row i, every cell in rows 0 through i-1 holds the correct count. Each of the m×n cells is written once: O(m×n) time, O(m×n) space — reducible to O(n) with a single rolling row.

The code

python

def uniquePaths(m, n):
    # build the DP table row by row
    dp = [[1] * n for _ in range(m)]   # base: top row all 1
    for i in range(1, m):
        for j in range(1, n):
            dp[i][j] = dp[i-1][j] + dp[i][j-1]
            # left col already seeded to 1; inner cells use top+left
    return dp[m-1][n-1]
Narrate it to the interviewer

"I need to count routes using only right and down moves. The brute-force recursion recomputes every cell from scratch when multiple paths converge on it — that is the wasted work. So I build a 2-D DP table. Every cell in the top row and left column has exactly one path to it, so those start at 1. For every other cell, the robot must have arrived from above or from the left, and those are disjoint sets of paths, so I simply add: dp[i][j] = dp[i-1][j] + dp[i][j-1]. I sweep the table once and read off the bottom-right cell. Time is O(m×n), space is O(m×n) — and I can cut it to O(n) by keeping only the current row."

Dry-run aloud: a 1×1 grid (one path, the robot is already there — return 1); a 1×n grid (only one path, straight right — return 1). Both fall out of the initialization with no special casing.

What the interviewer asks next

"Can you do it in O(n) space?" — yes, maintain a single row and update it in place: row[j] += row[j-1]. "What if moves are also allowed up or left?" — that breaks the DAG structure and the simple recurrence; you'd need BFS/DFS or a different DP formulation. "What is the closed-form answer?" — it is the binomial coefficient C(m+n-2, m-1), which can overflow for large inputs — the DP table avoids that risk if done with integers.

02 Adding a cost

Minimum path sum — optimization over the same table

Now each cell carries a non-negative cost. You still move only right or down, and you want the route from top-left to bottom-right whose total cost is smallest.

The structure is identical to the counting problem — the only thing that changes is how we combine neighbours. Instead of adding counts, we take the minimum incoming cost and add the current cell's weight: dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1]). The top row must be seeded as prefix sums (you can only come from the left), and so must the left column (you can only come from above).

Worked problemMediumLeetCode 64
Minimum Path Sum

Given an m × n grid of non-negative integers, find a path from the top-left to the bottom-right using only right and down moves that minimises the sum of all cell values along the route. Return that minimum sum.

Think it through

Restate

Find the cheapest right/down path from (0,0) to (m-1, n-1) in a cost grid. We need the minimum total weight, not just a count.

Brute force

Try every right/down path by recursion. The same exponential blowup applies — O(2^(m+n)) without memoization — for exactly the same reason: overlapping subproblems.

The waste

Every cell reachable from multiple directions gets recomputed. The minimum cost to reach (i,j) depends only on the minimum cost reaching (i-1,j) and (i,j-1) — both already stable when we fill left-to-right, top-to-bottom.

The move

Seed the top row as cumulative sums (dp[0][j] = dp[0][j-1] + grid[0][j]) and the left column similarly. Then dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1]) for all interior cells. The answer is dp[m-1][n-1]. You may modify the grid in-place to save the allocation.

Invariant & cost

After processing row i, every cell in that row holds the true minimum cost from the top-left. Each cell is touched once: O(m×n) time, O(1) extra space if we reuse the grid.

The code

python

def minPathSum(grid):
    m, n = len(grid), len(grid[0])
    # seed top row: only way is straight right
    for j in range(1, n):
        grid[0][j] += grid[0][j-1]
    # seed left col: only way is straight down
    for i in range(1, m):
        grid[i][0] += grid[i-1][0]
    # fill interior: take the cheaper neighbour
    for i in range(1, m):
        for j in range(1, n):
            grid[i][j] += min(grid[i-1][j], grid[i][j-1])
    return grid[m-1][n-1]
Narrate it to the interviewer

"This is the same structure as counting unique paths — the only change is the recurrence. Instead of summing counts I take the minimum incoming cost and add the local weight. The top row and left column can only be reached in one direction, so they become prefix sums. Interior cells pick the cheaper neighbour: grid[i][j] += min(above, left). I reuse the grid in-place so extra space is O(1). After the sweep the bottom-right holds the minimum total cost. O(m×n) time."

Dry-run aloud: a 1×1 grid (return the single cell's value — the loop bodies never run and grid[0][0] is returned unchanged); a grid of all zeros (answer is 0, the loop fills it all correctly); a grid where the top row is cheaper than every column path — the algorithm picks it naturally.

03 Blocked cells

Obstacles — zeroing out blocked positions

Unique Paths II (LeetCode 63) adds a twist: some cells are marked as obstacles that cannot be entered. A path through an obstacle is invalid, so the count for that cell is zero — and anything downstream that can only be reached through it is also zero.

The recurrence is almost identical. Before filling a cell, check the obstacle grid: if obs[i][j] == 1, write 0 and continue. Otherwise apply the usual dp[i][j] = dp[i-1][j] + dp[i][j-1], but treat a neighbour that is blocked as contributing 0. The top row and left column need careful seeding: as soon as you hit an obstacle in the top row while initializing, every cell to its right must also be 0 (there is no way around it).

Worked problemMediumLeetCode 63
Unique Paths II

An m × n grid has some cells blocked by obstacles. Count the distinct right/down paths from the top-left to the bottom-right that avoid all obstacles. Return 0 if no such path exists.

Think it through

Restate

Same as Unique Paths but some cells are off-limits. We need a count of valid paths only.

Brute force

Recursive DFS trying every right/down move, skipping blocked cells. Correct but revisits the same cells repeatedly — O(2^(m+n)) without memoization.

The waste

Exactly the overlapping-subproblems issue from before. The count for each cell depends on at most two neighbours and those neighbours don't change once filled — so bottom-up DP eliminates re-work entirely.

The move

Initialize a dp grid with all zeros. If the start is blocked, return 0 immediately. Seed dp[0][0] = 1. Fill the top row: carry 1 forward until an obstacle, then everything to the right stays 0. Fill the left column the same way. For interior cells: if blocked, dp[i][j] = 0; else dp[i][j] = dp[i-1][j] + dp[i][j-1]. Return dp[m-1][n-1].

Invariant & cost

Each row is complete and correct before the next is started. Each cell is touched once: O(m×n) time, O(m×n) space — or O(n) with a rolling row (same trick as before).

The code

python

def uniquePathsWithObstacles(obstacleGrid):
    m, n = len(obstacleGrid), len(obstacleGrid[0])
    if obstacleGrid[0][0] == 1: return 0   # start is blocked
    dp = [[0] * n for _ in range(m)]
    dp[0][0] = 1
    # top row: carry forward until an obstacle stops it
    for j in range(1, n):
        dp[0][j] = 0 if obstacleGrid[0][j] else dp[0][j-1]
    # left col: same idea going downward
    for i in range(1, m):
        dp[i][0] = 0 if obstacleGrid[i][0] else dp[i-1][0]
    # interior cells
    for i in range(1, m):
        for j in range(1, n):
            if obstacleGrid[i][j]:
                dp[i][j] = 0
            else:
                dp[i][j] = dp[i-1][j] + dp[i][j-1]
    return dp[m-1][n-1]
Narrate it to the interviewer

"This is Unique Paths with one extra rule: if a cell is blocked, its count is zero and nothing downstream that depends solely on it can be reached. I check if the start itself is blocked — that immediately returns zero. Then I seed the top row and left column, stopping the propagation of 1s as soon as I hit an obstacle. Interior cells either get zero (blocked) or sum their two live neighbours. O(m×n) time, O(m×n) space — same rolling-row trick cuts it to O(n)."

Dry-run aloud: obstacle at the start (return 0); obstacle at the end (return 0); obstacle in the middle of the top row (everything right of it in that row becomes 0 and paths must route below). All three are handled by the seeding loop without any extra checks.

04 Different recurrence

Maximal square — a cleverer combination rule

So far, every cell has combined its neighbours with a simple arithmetic operation. Maximal Square (LeetCode 221) shows that the neighbours can encode more structure.

Given a binary grid of '0's and '1's, find the largest all-1 square sub-grid. The naive check — for each cell, try every possible square size — runs in O(m×n×min(m,n)²). The DP insight cuts it to O(m×n).

Define dp[i][j] as the side length of the largest all-1 square whose bottom-right corner is at (i,j). If the cell itself is '0', dp[i][j] = 0 (no square ends here). Otherwise: a square of side s ending at (i,j) requires squares of side at least s-1 ending at the three adjacent corners — top, left, and top-left. So dp[i][j] = 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]). The answer is max(dp[i][j])² across all cells.

Worked problemMediumLeetCode 221
Maximal Square

Given an m × n grid of '0' and '1' characters, find the area of the largest square region that contains only '1's. Return the area (side × side).

Think it through

Restate

We want the side length s of the largest all-ones square sub-grid, then return s². A square is a fixed-size block, not a path.

Brute force

For each cell, try every side length from 1 upward, checking all cells in the candidate square. For an m×n grid, that is O(m×n×min(m,n)²) — far too slow for large grids.

The waste

Checking a large square re-examines rows and columns that were already verified by smaller squares. If we record, for each cell, the largest square whose bottom-right corner sits there, each step needs only three already-computed values.

The move

dp[i][j] = 0 if the cell is '0'. Otherwise dp[i][j] = 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]). The minimum of the three neighbours is the bottleneck: a square of side s needs all three corners to support side s-1. Track the running maximum. Return max_side².

Invariant & cost

After processing cell (i,j), dp[i][j] holds the exact largest square side anchored there. Each of the m×n cells is filled once: O(m×n) time, O(m×n) space — reducible to O(n) with a single previous-row array.

The code

python

def maximalSquare(matrix):
    if not matrix: return 0
    m, n = len(matrix), len(matrix[0])
    dp = [[0] * (n + 1) for _ in range(m + 1)]  # 1-indexed; row/col 0 = sentinel 0
    best = 0
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if matrix[i-1][j-1] == '1':
                dp[i][j] = 1 + min(dp[i-1][j],
                                    dp[i][j-1],
                                    dp[i-1][j-1])  # min of three neighbours
                best = max(best, dp[i][j])
    return best * best
Narrate it to the interviewer

"The brute force re-checks entire square regions from scratch. The key insight is: if I know the largest all-one square ending at each neighbour — top, left, and top-left diagonal — I can extend by one only when all three can support it. So dp[i][j] is the side of the largest square ending here, and it is 1 plus the minimum of those three values. I use a 1-indexed table with a sentinel zero-row and zero-column to avoid bounds checks. I track the running maximum and return the square of that. O(m×n) time and space."

Dry-run aloud: an all-zero grid (best stays 0, return 0); a grid of all '1's (the diagonal grows to min(m,n) correctly); a single '1' cell isolated by zeros (returns 1 × 1 = 1). Check that the sentinel row/column means dp[1][1] = 1 + min(0,0,0) = 1 for a '1' cell at position (0,0).

What the interviewer asks next

"Why min and not max?" — a square of side s fails if any direction can only support s-1; the weakest neighbour is the constraint. "Can you do O(n) space?" — keep only the previous row plus one extra variable for the diagonal predecessor; update right-to-left or track the previous diagonal value separately. "What about maximal rectangle (not square)?" — that is a harder variant (LeetCode 85) solved with a histogram-based stack approach row by row.

05 Optimisation & beyond

Space compression and the two-string teaser

Every problem in this lesson has an O(m×n) table, but most can be compressed to a single row of length n. The trick is that row i only ever reads from row i-1 — once you finish a row you never go back. So you update a single 1-D array left-to-right, overwriting each entry as you go:

# rolling-row unique paths: O(n) space
dp = [1] * n
for i in range(1, m):
    for j in range(1, n):
        dp[j] += dp[j-1]   # dp[j] holds old row[i-1][j]; dp[j-1] = new row[i][j-1]
# dp[n-1] is the answer

Maximal Square's three-neighbour rule needs a tiny extra variable to hold the previous row's diagonal value before it is overwritten. That is a common interview follow-up.

Two strings as a 2-D grid

Once you are comfortable with grid DP, the same 2-D table pattern appears for pairs of strings. The cell dp[i][j] now represents the answer for the first i characters of string A and the first j characters of string B. Classic examples:

  • Longest Common Subsequence (LeetCode 1143) — dp[i][j] = dp[i-1][j-1] + 1 when the characters match, else max(dp[i-1][j], dp[i][j-1]).
  • Edit Distance (LeetCode 72) — dp[i][j] = dp[i-1][j-1] on a match, else 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) for delete/insert/replace.
  • Distinct Subsequences (LeetCode 115) — counts ways to embed one string in another; same left/top dependency, different combination rule.

The sweep is the same: fill row by row, reading only the previous row and the current row's already-settled cells. Master the grid version first — the two-string problems are the same skeleton dressed in a different story.

Full 2-D table O(m×n) space

  • Easiest to code and debug — you can print the entire table.
  • Required when you need to reconstruct the actual path (traceback).
  • Safe default in an interview; optimize only if asked.

Rolling row O(n) space

  • One row replaces the whole table; overwrite as you go.
  • Maximal Square needs one extra diag variable for the diagonal predecessor.
  • Cannot reconstruct the path from this form alone.
Spotting the grid DP signal

Look for: "robot moves right/down"; "minimum/maximum cost path"; "count distinct paths"; "largest all-ones rectangle or square"; and (for two-string variants) "edit distance", "common subsequence", or "ways to form one string from another." If the answer at position (i,j) depends only on (i-1,j) and (i,j-1) — and those are already settled when you need them — it is grid DP.

06 Practice

Practice ladder

Work top to bottom. For each problem: state what dp[i][j] means, write out the base cases, write the recurrence, then code it. Every problem below has a correct O(m×n) DP solution and most have an O(n) space follow-up.

EasyUnique Pathsbasic grid DP

Count how many distinct right/down routes connect the top-left to the bottom-right of an m×n grid.

Idea dp[i][j] = dp[i-1][j] + dp[i][j-1]; base cases all 1. O(mn) time, reducible to O(n) space.

EasyClimbing Stairs1-D warm-up

You can climb 1 or 2 steps at a time; find the number of ways to reach the top of an n-step staircase.

Idea dp[i] = dp[i-1] + dp[i-2] — same addition structure as unique paths but in 1-D; good warm-up for the pattern.

MediumMinimum Path Sumcost grid

In a grid of non-negative integers, find the right/down path from top-left to bottom-right with the smallest total cost.

Idea dp[i][j] = grid[i][j] + min(above, left); seed top row and left col as prefix sums. Modify in-place for O(1) space.

MediumUnique Paths IIobstacles

Same as Unique Paths but some cells are blocked; a path must avoid all obstacles.

Idea Zero out blocked cells; stop propagating 1s along the top row or left col as soon as an obstacle appears.

MediumTrianglevariable-width grid

Given a triangle of numbers where each row is one element wider, find the minimum-sum path from the apex to any element in the base row.

Idea Process bottom-up; dp[j] = triangle[i][j] + min(dp[j], dp[j+1]) — a rolling row naturally fits the variable width.

MediumMaximal Squaremin-of-3-neighbours

In a binary grid of '0's and '1's, find the area of the largest square containing only '1's.

Idea dp[i][j] = 1 + min(top, left, diag) when cell is '1'; track running max. Return max_side².

MediumLongest Common Subsequencetwo-string grid

Given two strings, find the length of their longest subsequence that appears in both (characters need not be contiguous).

Idea dp[i][j] = dp[i-1][j-1]+1 on a character match, else max(dp[i-1][j], dp[i][j-1]). Same sweep, different combination.

MediumEdit Distancetwo-string grid

Count the minimum number of single-character insert, delete, or replace operations needed to transform one string into another.

Idea dp[i][j] = dp[i-1][j-1] on match; else 1 + min(delete, insert, replace) — the three-neighbour min is the same instinct as Maximal Square.

MediumCoin Change IIunbounded knapsack

Count the number of combinations of coin denominations that sum to a target amount, using any number of each denomination.

Idea A 2-D table over (coin index, amount) with the same row-dependency structure; the rolling-row trick applies directly.

HardMaximal Rectanglegrid + stack

In a binary grid, find the area of the largest all-'1' rectangle (not just a square).

Idea Build a histogram of 1-heights row by row, then apply the largest-rectangle-in-histogram stack algorithm on each row. O(mn) total.

HardDistinct Subsequencestwo-string grid

Count the number of ways the characters of a shorter string appear as a subsequence of a longer string.

Idea dp[i][j] = dp[i-1][j] + (dp[i-1][j-1] if chars match); watch for overflow on large inputs.

HardCherry Pickuptwo-pass grid

Collect the maximum cherries on a round trip from top-left to bottom-right and back, with each cherry vanishing after the first visit.

Idea Simulate two simultaneous forward paths using a 3-D DP over (step, col1, col2); a clever reformulation avoids the backward pass entirely.

How to drill effectively

For every problem:
(1) write out what dp[i][j] means in plain English before touching code;
(2) state the base cases — top row, left column, and any sentinel;
(3) write the recurrence and check whether it reads from above, left, or both;
(4) code it, then ask yourself if a rolling row reduces the space;
(5) dry-run on a 2×2 or 3×3 example by hand and compare with your code's output. Pattern recognition is the goal — once you see "path on a grid" or "match two strings character by character," the table sketch should be automatic.

★ Review

Summary & self-test

Every problem in this lesson reduces to the same skeleton: define dp[i][j] as the answer for position (i,j), seed the top row and left column as base cases, sweep the interior left-to-right and top-to-bottom, and combine each cell from its settled neighbours. The combination rule — add, min, or min-of-three — is the only thing that changes between problems.
