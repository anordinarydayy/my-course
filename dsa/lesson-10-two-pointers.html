Two Pointers

Place two indices into an array — one at each end, or both racing forward at different speeds — and the classic O(n²) nested scan collapses into a single O(n) pass. Two pointers is the pattern that makes pair sums, in-place edits, cycle detection, and three-way partitioning all feel like the same idea wearing different clothes.

00 The problem

What does it solve?

Consider the most common interview warm-up: given a sorted array of integers, find any pair that adds up to a given target. The blunt answer is two nested loops — pick every index i, then scan every index j > i looking for target − a[i]. The logic is airtight, but for an array of length n you make roughly n²/2 comparisons. At n = 100,000 that is five billion operations — well past the one-second budget of a typical online judge.

The waste is obvious once you see it: when the array is sorted and the current sum is already too large, there is no need to try smaller right-hand values — they can only make the sum smaller. Conversely, when the sum is too small, nothing to the right of the current left pointer can help, because those values are even smaller. The whole nested scan is redundant; what you actually need is a way to eliminate one candidate per comparison. Two indices that converge from the ends do exactly that, finishing in at most n steps.

The same observation powers four distinct sub-patterns. Each one replaces a different bottleneck — a nested search, an O(n) auxiliary array, a visited-node hash set, or a multi-pass sort — with a single linear sweep.

  • Converging — one pointer at each end, walking toward the middle. Works on sorted data where moving either end changes the quantity (sum, product, area) monotonically.
  • Read & write — both pointers start at the front; a fast one scans while a slow one builds the answer in place. The go-to for O(1)-space in-place edits.
  • Fast & slow — both advance forward but at different rates. The trick for linked-list cycles, midpoints, and n-th from the end.
  • Three-pointer — extends converging or read/write to three indices that together carve the data into settled regions plus one shrinking unknown.
Why this pattern first

Two pointers is the minimal upgrade from brute force: it adds just two variables and one decision rule per step, yet it cuts complexity from quadratic to linear. In an interview, the move that signals fluency is saying the brute force first — "try every pair, O(n²)" — then immediately naming the signal that makes two pointers valid: sorted order gives a monotonic direction to move. That single sentence demonstrates you understand when a pattern applies, not just how to execute it.

01 Converging

Pointers that close in from both ends

The converging variant places L at index 0 and R at the last index. On each step you look at a[L] + a[R]. Because the array is sorted, the sum is monotonically controlled by which end you move:

  • Sum too small — even the largest remaining value can't bring a[L] up to the target, so a[L] cannot be in any solution. Discard it: L++.
  • Sum too large — even the smallest remaining value can't shrink a[R] enough, so a[R] cannot be in any solution. Discard it: R--.
  • Sum equals target — done.

The invariant is that any valid pair must have both its indices inside the current window [L, R]. Each move only ever discards a value that has been proved useless for all remaining candidates, so the search is complete and nothing is missed. The pointers converge, and the whole job takes at most n steps.

Two pointers (sorted)

  • O(n) time after sorting; O(1) extra space.
  • Returns actual indices, generalizes to triplets or quadruplets.
  • Requires sorted input — costs O(n log n) to sort first if not already.

Hash set (unsorted)

  • O(n) time; O(n) extra space.
  • For each value, look up target − value in the set.
  • Best when the data is unsorted and you only need a yes/no answer.
Worked problemEasyLeetCode 167
Two Sum II — Input Array Is Sorted

You receive an integer array already in non-decreasing order and a target value. Return the 1-based indices of the unique pair of elements that sums exactly to the target. You may not use the same element twice.

Think it through

Restate

Sorted array, guaranteed exactly one pair that hits the target — give back their positions as 1-based indices. No extra allocation permitted.

Brute force

Try all pairs: for each i, scan every j > i. If a[i] + a[j] == target, return them. Correct, but O(n²) — completely ignores the sorted order.

The waste

When the current sum is too small, every j beyond the current right pointer would only make things worse — we would never find a match by scanning smaller values on the right. The sorted structure tells us exactly which direction to push, making the nested scan pointless.

The move

Place L at index 0, R at the last index. Each step: if the sum is below target, bump L++ to grow it; if above, pull R-- to shrink it; if equal, the answer is [L+1, R+1]. The invariant guarantees the valid pair always lies within the current window.

Invariant & cost

Every step either advances L or retreats R, so the window shrinks by one. We make at most n iterations. O(n) time, O(1) space.

The code

python

def twoSumSorted(numbers, target):
    L, R = 0, len(numbers) - 1
    while L < R:
        s = numbers[L] + numbers[R]
        if s == target:
            return [L + 1, R + 1]   # convert to 1-based
        if s < target:
            L += 1                  # sum too small: move left boundary up
        else:
            R -= 1                  # sum too big: pull right boundary down
    return []                       # guaranteed hit; defensive guard
Narrate it to the interviewer

"Restating: the array is already sorted, I need the one pair that hits the target, returned as 1-based positions. The brute force tries all O(n²) pairs, but that ignores the sorted property entirely. With two pointers I can eliminate one candidate per comparison. I put L at the front and R at the back and check their sum. If it's below target I move L up — because a[L] can never pair with anything to make the target, even with the largest remaining element on the right. If it's above target I move R down by the same logic. My invariant is that the valid pair, if any, always sits between the two pointers. Every step shrinks the window, so I finish in O(n) time and O(1) space."

Dry-run aloud: a two-element array where the pair is the only option, and the case where L and R start adjacent — confirm the L < R guard prevents using a single element twice.

What the interviewer asks next

"What if the array isn't sorted?" — either sort it first and two-point in O(n log n), or use a hash set in O(n) time and O(n) space: name the trade-off aloud. "Extend to 3Sum (LeetCode 15)?" — fix one element with an outer loop from left to right, then run converging pointers on the remaining suffix; when a triple is found, skip equal values on both sides to avoid duplicate triples. That gives O(n²) total. "Could L and R ever cross?" — the while L < R guard prevents it; if they meet, every pair has been considered.

02 Read & write

A slow writer and a fast reader

The second variant keeps both pointers at the left and sends them rightward — but at independent speeds. One pointer acts as a read head: it visits every element. The other acts as a write head: it only advances when it places a value that belongs in the final answer. The invariant is that everything from index 0 up to (but not including) the write pointer is already the correct, compacted prefix.

This is the O(1)-space answer for any in-place "keep only the elements that pass a test" problem. The interviewer's tell: the prompt says in place, asks for no extra allocation, or says to modify the array and return a length. The code skeleton is essentially always the same five lines — only the keep condition changes.

Below: compacting an array so that each non-zero value occupies a write slot, then filling the remainder with zeros (Move Zeroes, LeetCode 283). Watch the write pointer hold still while the read pointer skips zeros, then lurch forward the moment it finds a non-zero.

The reusable shape

Almost every in-place "keep what passes" problem is this five-line template:
slow = 0; for fast in range(n): if keep(a[fast]): a[slow]=a[fast]; slow++.
The answer length is slow. Remove Element, Remove Duplicates from Sorted Array, and Move Zeroes all share this skeleton — only the keep predicate differs. Recognizing that identity is the skill the interviewer is probing.

Worked problemEasyLeetCode 283
Move Zeroes

Given an integer array, move every zero element to the tail while preserving the relative order of the non-zero values. Do it in place, without allocating a second array.

Think it through

Restate

Compact all non-zeros to the front in their original order; fill the rest with zeros. In place, so I cannot allocate an auxiliary array.

Brute force

Build a new list of non-zero values, pad it with zeros, copy everything back. Correct, but O(n) extra space — which the problem forbids.

The waste

The second array is unnecessary: the non-zeros only need to slide into the front of the same array. This is exactly the read/write pattern — keep is just "not zero".

The move

slow starts at 0 and is the next write slot for a non-zero. fast iterates through every index. When a[fast] != 0, copy it to a[slow] and advance slow. After the scan, everything from slow to the end needs to be set to 0.

Invariant & cost

a[0 .. slow-1] always contains the non-zeros seen so far, in their original relative order. Each element is touched a constant number of times. O(n) time, O(1) space.

The code

python

def moveZeroes(nums):
    slow = 0                          # write index for the next non-zero
    for fast in range(len(nums)):     # fast reads every element
        if nums[fast] != 0:           # keep predicate: is it non-zero?
            nums[slow] = nums[fast]
            slow += 1
    for i in range(slow, len(nums)):  # fill the tail with zeros
        nums[i] = 0
Narrate it to the interviewer

"Restating: compact the non-zeros to the front in their original order, then zero out the tail — in place, O(1) extra space. The naive approach builds a second array, which costs O(n) space the problem forbids. So I use the read/write pattern: slow is the write head for non-zeros, fast reads every cell. Whenever fast sees a non-zero it copies it to slow and bumps slow. After the loop, everything from slow onward gets zeroed. My invariant is that a[0..slow-1] is always the non-zeros in their original order. Every element gets touched once in the read pass and at most once in the fill pass — O(n) time, O(1) space. A one-pass swap variant also exists: swap a[slow] and a[fast] instead of copying, which avoids the second loop."

Dry-run aloud: an empty array (both loops do nothing), all zeros (slow stays 0, entire array gets zeroed), no zeros at all (slow tracks fast the whole way, fill loop runs zero iterations).

03 Different speeds

Fast and slow on a linked list

Linked lists block direct indexing, so the converging and read/write variants don't apply. A different problem surfaces instead: does this list contain a cycle? The safe, obvious answer visits every node and stores it in a hash set, returning true when a node appears for the second time. That's O(n) time and O(n) space.

The O(1)-space answer uses two pointers advancing at different rates. A slow pointer moves one node per tick; a fast pointer moves two. If there is no cycle, fast eventually steps off the end into null. If there is a cycle, fast keeps lapping the track — and the gap between the pointers shrinks by one each tick, so they must eventually land on the same node. A meeting can only happen inside a cycle, so it is both necessary and sufficient proof that a loop exists.

What the interviewer asks next

"Find the exact start of the cycle (LeetCode 142)" — after slow and fast meet, move one pointer back to the head and advance both one step at a time. They meet at the cycle's entry. The distance argument: if the head-to-entry length is a, and the meeting node is distance b into the cycle of length c, then a ≡ c − b (mod c), so walking a steps from head and from the meeting point arrive at the same entry node simultaneously. "Find the middle of a list" — when fast reaches the end, slow is at the midpoint. "Remove the k-th node from the end" — advance fast by k first, then walk both until fast hits the tail; slow is then one behind the target.

Worked problemMediumLeetCode 142
Linked List Cycle II

Given the head of a singly linked list, return the node at which the cycle begins, or null if there is no cycle. You may not modify the list and must use O(1) extra space.

Think it through

Restate

Detect a cycle and, if one exists, return the exact node where the loop begins. No extra memory, no mutation of the list.

Brute force

Walk the list, storing each visited node in a hash set. The first node that appears a second time is the cycle's entry. O(n) time but O(n) space — the problem requires O(1).

The waste

The hash set remembers every node only to check for re-visits. Two pointers at different speeds can detect re-entry into the cycle without storing anything. A follow-up distance argument then pins the entry node.

The move — phase 1 (detect)

slow advances one step per tick, fast advances two. If fast ever hits null, no cycle exists. If they land on the same node, a cycle was confirmed.

The move — phase 2 (locate)

Move one pointer back to the head, keep the other at the meeting node. Advance both one step at a time. They meet at the cycle's entry. This follows from the distance equation: head‑to‑entry = c − b (mod c), where b is how far into the cycle they met.

Invariant & cost

Inside any cycle, the gap between the two pointers closes by exactly one per tick — they must meet in at most c ticks after slow enters the cycle. Total: O(n) time, O(1) space, no mutation.

The code

python

def detectCycle(head):
    slow = fast = head
    # phase 1: find a meeting point inside the cycle
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:              # met inside a cycle
            slow = head               # phase 2: reset one pointer
            while slow is not fast:  # step both by 1 until they meet
                slow = slow.next
                fast = fast.next
            return slow              # cycle entry node
    return None                      # fast hit the end, no cycle
Narrate it to the interviewer

"Restating: detect a loop and return the node where it starts, in O(1) space, without touching the list. The hash-set answer is O(n) space, so I use Floyd's two-pointer approach. Phase one is detection: slow moves one step, fast moves two. If fast steps off the end, no cycle. If they share a node, a cycle exists — a meeting can only happen inside the loop. Phase two locates the entry: I move one pointer back to the head and walk both at the same speed. They converge at the entry node, which falls out of the distance math — the head-to-entry distance equals the number of steps from the meeting point to the entry. Overall O(n) time, O(1) space."

Dry-run aloud: a null head (return null immediately), a single node pointing to itself (slow=fast=head, fast.next=head, so slow and fast meet on the first tick — then both are at head, which is the entry — correct), a two-node list with no cycle (fast.next.next is null — returns null).

04 Generalization

Three-pointer partition

Two pointers can stretch to three. The Dutch national flag problem — sort an array containing only 0, 1, and 2 in a single pass — is the canonical example. It appears frequently in interviews precisely because the obvious solutions (counting sort, library sort) are either two-pass or overkill, while the three-pointer answer is one pass with O(1) space and requires a non-trivial invariant argument to get right.

Keep three pointers: low marks the end of the confirmed 0-region, high marks the start of the confirmed 2-region, and mid is the scan cursor. For each element at mid:

  • a[mid] == 0 — swap it down to low; advance both low and mid. Safe to advance mid because the value swapped in from low was already classified (it was in the 0- or 1-region).
  • a[mid] == 2 — swap it up to high and shrink high. Do not advance mid: the value pulled from high has never been inspected.
  • a[mid] == 1 — already in the right place. mid++ only.

Stop when mid > high — the unknown region has been exhausted.

Worked problemMediumLeetCode 75
Sort Colors

An array holds only the values 0, 1, and 2. Rearrange it in place so all 0s come first, then all 1s, then all 2s. Do it in a single pass — no counting sort, no library sort.

Think it through

Restate

In-place sort of a ternary-valued array in one scan, O(1) extra space.

Brute force

Count how many 0s, 1s, and 2s appear, then overwrite the array with that many of each. Two passes, O(1) space, correct — but the interviewer has specifically asked for a single pass. A generic sort would be O(n log n) and would ignore the ternary structure entirely.

The waste

With only three values, I don't need to count then rewrite — I can directly partition into three settled regions while scanning. That's the three-pointer extension of two-pointer converging.

The move

low=0 (end of 0-region), high=n-1 (start of 2-region), mid=0 (scan cursor). If nums[mid]==0: swap with nums[low]; both low and mid advance. If nums[mid]==2: swap with nums[high]; high retreats — do not advance mid, because the swapped-in value from the high boundary is unseen. If nums[mid]==1: mid advances. Stop when mid > high.

Invariant & cost

a[0..low) all 0s, a[low..mid) all 1s, a[mid..high] unknown, a[high+1..] all 2s. Every step either advances mid or retreats high, so the unknown window empties in ≤ n steps. O(n) time, O(1) space, single pass.

The code

python

def sortColors(nums):
    low, mid, high = 0, 0, len(nums) - 1
    while mid <= high:
        if nums[mid] == 0:
            nums[low], nums[mid] = nums[mid], nums[low]
            low += 1; mid += 1      # swapped-in value already classified
        elif nums[mid] == 2:
            nums[mid], nums[high] = nums[high], nums[mid]
            high -= 1               # do NOT advance mid: new value is unclassified
        else:                        # nums[mid] == 1: already in place
            mid += 1
Narrate it to the interviewer

"There are only three distinct values, so I can sort with a three-way partition instead of a general sort. The two-pass counting approach works but the problem asks for one pass. My three pointers create four regions: confirmed 0s on the left, confirmed 1s just inside, an unknown window in the middle, and confirmed 2s on the right. Scanning with mid: a 0 swaps down to low and I advance both — the value that came up from low is known to be 0 or 1 (it's been classified), so mid is safe to move. A 2 swaps up to high and I shrink high — but I do not advance mid, because the value swapped in from high has never been inspected. A 1 just steps mid forward. I stop when mid exceeds high, meaning the unknown window is empty. One pass, O(n) time, O(1) space."

Dry-run aloud: all-same array (e.g., all 1s — mid advances from 0 to n-1 in a single pass, low=0, high=n-1 throughout, no swaps), and a two-element array [2,0] — mid=0 is 2: swap with high=1, high becomes 0, mid stays 0; now mid=0 is 0: swap with low=0 (no-op), low=1, mid=1; mid=1 > high=0, stop. Array is [0,2]. Correct.

05 Recognition

Spotting it, and the traps

The skill isn't executing the pattern — it's recognizing it in time to use it. Train yourself to hear these signals:

  • The input is sorted (or sortable) and you need a pair, triple, or closest sum.
  • The phrase "in place" or "O(1) extra space" on an array problem where you need to compact or reorder.
  • Palindrome checks, or reversals — compare or swap from both ends inward.
  • Anything about a linked list mentioning a cycle, the middle node, or the k-th element from the end.
  • Geometry problems over an array — most water, trapping rain — where you move the constraining side inward.
  • Merging two sorted sequences — one pointer per sequence, walk both forward simultaneously.

And the bugs that cost candidates the round:

  • Forgetting to move any pointer in some branch — the loop hangs forever. Audit every if branch: at least one pointer must advance or retreat.
  • Off-by-one on the loop guard — L < R prevents pairing an element with itself; L <= R allows processing the middle element as its own pair. Know which you need before you type the loop header.
  • Applying converging pointers to unsorted data — there is no monotonic reason to choose which end to move, so the algorithm cannot be proved correct. If the data isn't sorted and you can't sort it, reach for a hash map or a sliding window instead.
  • Skipping the duplicate-avoidance step in 3Sum — after recording a valid triple, advance both inner pointers past all equal values, or the output will contain repeated triples.
  • In the Dutch flag algorithm, advancing mid after swapping with high — the value that just arrived from the right end has never been classified; treating it as classified is the most common bug in this problem.
The decision in one sentence

If the data is sorted and you're hunting a pair or aggregate across two ends — converging. If you need in-place compaction of a prefix — read/write. If you're on a linked list checking cycles or distances — fast and slow. For everything else where two pointers feel forced: an unsorted pair-complement lookup belongs to a hash map, and a best-subarray-under-a-constraint belongs to the sibling pattern, the sliding window.

★ Review

Summary & self-test

Two pointers condenses to one idea: maintain two indices and let the current state dictate which one to move, so every element is touched a constant number of times. The pattern has three well-defined variants — converging, read/write, and fast/slow — each requiring a specific precondition that gives a monotonic direction for the decision. The Dutch-flag three-pointer algorithm is the same idea with one extra boundary. All four reduce a quadratic or linear-space solution to linear time and constant space.

01Kills nested loopsConverts O(n²) all-pairs scanning into one O(n) pass by discarding one candidate per step.
02ConvergingSorted data; L and R close in; each comparison eliminates one end permanently.
03Read & writeFast scans, slow writes keepers. In-place compaction in O(1) space — only the keep condition changes per problem.
04Fast & slowDifferent advance rates detect linked-list cycles and pinpoint their entry — no hash set, no mutation.
05Precondition firstYou must have a monotonic reason to pick which pointer moves. Without it, two pointers cannot be proved correct.
06State the invariantThe valid answer always lies within the current window (or region). That invariant is the proof of correctness.
07Three pointersDutch national flag: four regions, unknown shrinks each step. Don't advance mid after swapping a 2 with high.
08When to skip itUnsorted complement lookups need a hash map. Best contiguous subarray under a constraint needs a sliding window.

(1) Why can a converging two-pointer search never miss a valid pair? State the invariant and explain why each pointer move is safe.
(2) In the read/write variant, what does each pointer represent, and what is always true of the prefix to the left of the slow pointer?
(3) Why must the tortoise and hare meet if and only if a cycle exists? Once they meet, how do you find where the loop begins — and what is the distance argument?
(4) In Sort Colors, what are the four regions, and why does mid not advance after a swap with high?
(5) Given an unsorted array and a target pair-sum, which technique do you reach for and why — and what does it cost?
(6) Name four signals in a problem statement that point toward two pointers, and two signals that point away from it.
