The Problem-Solving Framework (UMPIRE)

Every pattern lesson in this course reuses the same six-step loop: Understand, Match, Plan, Implement, Review, Evaluate. Internalise this method before the patterns and you arrive at each problem with a repeatable structure rather than a blank stare.

Original study notes. Written from scratch for coding-interview prep. Problem names are referenced only as well-known identifiers; every description, diagram, and line of code here is original. Contains three interactive diagrams (press Play or step through with Next) and one worked walkthrough of Two Sum (LeetCode 1) carried through all six phases. This is a concept lesson — no practice ladder; instead, the final section points to the pattern lessons that apply this loop at every step.
00 Motivation

Why you need a method

The most common failure mode in a coding interview is not a lack of knowledge — it is a lack of structure. A candidate who knows two pointers, BFS, and dynamic programming but charges straight at the code without restating the problem or thinking about edge cases will still freeze, miss constraints, and write bugs they cannot locate. The interviewer sees this as noise, not signal.

UMPIRE is a six-letter mnemonic for the order of operations every strong candidate applies, consciously or not, on every problem. It is not a magic formula — it is a scaffold that keeps you moving forward and gives the interviewer something clear to follow. Think of it as the operating system your pattern knowledge runs on.

The core discipline

In an interview, silence is fine during Understand and Plan — most interviewers actively want you to think aloud. What kills candidates is the opposite: jumping to code in the first thirty seconds, then going quiet when the code breaks. UMPIRE keeps your narration continuous and your logic defensible at every step.

01 The method

The six phases ▸

Each phase has a specific deliverable. Skipping one costs you more time than it saves, because the work doesn't disappear — it just surfaces later as a bug or a missed edge case.

U
Understand
Restate the problem in your own words. Ask clarifying questions. Pin the input/output contract and any hard constraints. Never assume.
M
Match
Map the problem to a known pattern. What is the input shape? Does it look like a sorted-array, graph, or DP problem? Name the target complexity.
P
Plan
State the brute-force approach and its cost first. Then describe your optimised plan — the key data structure, the invariant, the loop bound.
I
Implement
Write clean code that matches the plan exactly. Keep variable names meaningful. Speak each step aloud as you type it.
R
Review
Dry-run the code on the provided examples, then on at least two edge cases: empty input, a single element, all-equal values, and any constraint boundaries.
E
Evaluate
State the final time and space complexity. Name any remaining trade-offs. Offer one possible follow-up optimisation if one exists.
The most skipped phase: Understand

Candidates who score poorly almost always begin coding before they have confirmed the output format, whether indices or values are expected, how to handle ties or duplicates, and what the size constraints are. Spending 90 seconds on Understand and Match typically saves 10 minutes of debugging at the end of the round.

02 Input size

Reading input size to pick target complexity

The M (Match) phase has a mechanical sub-task: look at the constraint on n and immediately rule out any algorithm that is too slow. Interviewers set constraints deliberately — they are a hint to the intended complexity. Train yourself to read them as a budget.

n (input size) Acceptable worst-case Typical approach Danger zone
n ≤ 20 O(2ⁿ) or O(n!) Backtracking, permutations, bitmask DP —
n ≤ 500 O(n²) All-pairs scanning, O(n²) DP O(n³) likely TLE
n ≤ 10,000 O(n log n) or O(n²) if tight Sort + scan, two pointers, heap O(n²) may TLE
n ≤ 100,000 O(n log n) Sort, BFS/DFS, segment tree, two pointers O(n²) will TLE
n ≤ 1,000,000 O(n) or O(n log n) Hash map, sliding window, linear DP O(n log² n) risky
n ≤ 10⁹ O(log n) or O(1) Binary search, math, closed-form Anything linear will TLE

At the Match phase you do not need to know which algorithm to use yet — you just need to know which complexity class you are aiming for. That rules out half the wrong approaches before you have thought of them. For Two Sum (LeetCode 1) with n up to 10⁴, you can see immediately that O(n²) will pass but O(n) is the expected target.

The two-pointers connection

Once you know you need O(n) or O(n log n) on an array, the pattern shortlist almost writes itself: hash map for unsorted lookups, two pointers for sorted arrays or in-place edits, sliding window for contiguous subarrays, prefix sums for range queries. Pattern 01 (Two Pointers) is the first tool that exploits this — see Lesson 01.

For the Review phase, always test these four categories of edge case regardless of the problem:

  • Empty input — an empty array, a null pointer, an empty string. Does your loop execute zero times correctly?
  • Single element — does your algorithm behave sensibly when n = 1?
  • All identical values — duplicate inputs stress conditions that check for equality.
  • Constraint boundaries — the minimum and maximum allowed values. Look for integer overflow when values can reach 10⁹.
03 Live walkthrough

UMPIRE on Two Sum (LeetCode 1) ▸

The best way to absorb a framework is to watch it applied once, in full, on a real problem. Two Sum (LeetCode 1) is ideal: it is short enough to carry through all six phases without losing the thread, and it has two distinct approaches that illustrate the Match and Plan phases cleanly.

Our problem (in our own words): given an array of integers and a target integer, return the indices of the two elements that add up to the target. You may assume exactly one valid pair exists, and you cannot use the same element twice.

The full code from phase I

Below is the implementation that comes out of the Plan phase. The brute-force (all pairs, O(n²)) is shown first so the waste is clear, then the one-pass hash-map solution.

python

# --- brute force: try every pair --- O(n²) time, O(1) space
def two_sum_brute(nums, target):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] + nums[j] == target:
                return [i, j]
    return []                          # guaranteed a pair exists; guard anyway

# --- optimised: one-pass hash map --- O(n) time, O(n) space
def two_sum(nums, target):
    seen = {}                          # value -> index of values we have passed
    for i, val in enumerate(nums):
        complement = target - val      # what we need to pair with val
        if complement in seen:
            return [seen[complement], i]
        seen[val] = i                  # record val only after checking, so i != j
    return []
Narrate it to the interviewer (phase I + E script)

"Restating: I need the indices of any two elements that sum to the target — exactly one valid pair, can't reuse an element. The brute-force checks all pairs in O(n²) time. The waste is that for each element I already know what its partner needs to be — target minus the current value — so instead of searching for it again I can just look it up in a table I built as I went. I'll walk the array once, and for each element I check whether its complement is already in a hash map of values I have seen. If yes, I return both indices immediately; if no, I record this element and move on. The key detail is that I record only after I check, so I never pair an index with itself. Final cost: O(n) time, O(n) space."

Dry-run aloud: [3, 7, 2, 9] target 11 → seen empty, see 3, complement is 8 (not in seen), store {3:0}; see 7, complement is 4 (not in seen), store {3:0, 7:1}; see 2, complement is 9 (not in seen), store {3:0, 7:1, 2:2}; see 9, complement is 2 — yes, return [2, 3]. Edge: nums=[5, 5], target=10 → see first 5, complement 5 not in seen yet, store {5:0}; see second 5, complement 5 is in seen at index 0, return [0,1]. Correct because we check before we store.

04 Clarifying questions

The Understand phase in practice ▸

The Understand phase is a conversation, not a monologue. Experienced candidates have a mental checklist of categories to probe before they touch the whiteboard. A good clarifying question either (a) resolves an ambiguity that would force a completely different approach, or (b) confirms an assumption so the interviewer can correct it now rather than after you have spent ten minutes on the wrong path.

The stepper below maps a category of question to the reason it matters and the approach it unlocks or rules out.

Questions that change your pattern

  • "Is the input sorted?" — yes unlocks two pointers; no pushes you to a hash map.
  • "Can values repeat?" — duplicates can break converging pointers unless you skip equal elements.
  • "Do I return indices or values?" — critical for hash-map approaches where you store different things.
  • "What should I return if there is no answer?" — affects your base case and guards.

Questions that change your complexity budget

  • "How large can n get?" — tells you your target complexity class directly from the input size table.
  • "Is extra space allowed?" — O(1) space rules out hash maps and forces in-place techniques.
  • "Can I modify the input?" — sorting in place is fine only if allowed; otherwise O(n log n) requires extra space.
  • "Are there integer overflow concerns?" — sum of two 10⁹ values overflows a 32-bit int.
05 What comes next

Where this shows up next

UMPIRE is not a one-off exercise — it is the operating loop for every pattern lesson in this course. Here is how the six phases map to the content of each pattern lesson you will study:

L01Two PointersU+M+P dominant

The Match phase locks in "sorted array or in-place edit"; the Plan phase articulates which of the three two-pointer flavors applies; Review tests empty, single-element, and all-equal cases. The worked problems each carry their own UMPIRE narration script.

UMPIRE link Reading input size (L00 §2) tells you whether O(n) is required — two pointers delivers it.

L02Sliding WindowP+I dominant

The Plan phase must identify the monotonic condition that lets the window shrink — that is the invariant you state before coding. Review always checks the window at its minimum size (size 1 or 0).

UMPIRE link The Understand phase distinguishes "exact length" from "at most / at least" windows, which changes the implementation entirely.

L03Binary SearchM+P dominant

Match asks: is there a predicate that turns the search space into a monotone true/false sequence? Plan then sets the precise loop invariant (lo and hi meaning) before a single line of code. Getting this wrong is the most common binary-search bug.

UMPIRE link Evaluate must state O(log n) time and justify it — the input size table (L00 §2) is the quick check.

L20–L22Dynamic ProgrammingP+I+R dominant

DP is where UMPIRE earns its keep most visibly. The Plan phase must define the state, the recurrence, and the base cases — all three, before touching code. The Review phase dry-runs the table on a small example to catch off-by-one errors in the recurrence.

UMPIRE link The brute-force recursion (P phase) is the jumping-off point; memoisation and tabulation are the moves that remove repeated work.

L23–L25Graph Traversal (BFS/DFS)U+M dominant

The Understand phase must confirm: is the graph directed or undirected? Can there be cycles? Is edge weight relevant? The Match phase picks BFS (shortest path, level-order) vs DFS (connectivity, topological order) based on the answers.

UMPIRE link Evaluate must account for O(V+E) time and O(V) space — never forget the visited set in the space count.

The universal reuse rule

Every pattern lesson you study is structured around a mini UMPIRE loop for each worked problem: the "Think it through" block is U+M+P; the code is I; the narration script is R+E. Once you recognise this, the format becomes second nature and you can apply it with the same fluency in a live coding round.

★ Review

Summary & self-test

UMPIRE compresses to a single rule: never write code until you have restated the problem, matched it to a pattern and complexity class, and narrated a plan including the brute force and the invariant. The last three phases are execution and verification. Every pattern lesson in this course is an application of this loop on a specific family of problems.

01Understand firstConfirm input/output contract and constraints before anything else. Silence during this phase is fine.
02Match = read nThe input size tells you the target complexity. Rule out too-slow approaches immediately.
03Brute force out loudAlways state the naive O(n²) or O(2ⁿ) approach first — it anchors the complexity and shows the interviewer your baseline.
04Invariant before codeName the property your algorithm maintains at every step. That is the proof that it is correct.
05Review with edgesEmpty input, single element, all-equal, and boundary values — test all four before claiming done.
06Evaluate both costsState time AND space. An O(n) time answer that uses O(n) space may have a follow-up asking for O(1) space.
07Ask to unlock patternOne clarifying question about sortedness or extra-space allowance can immediately resolve which pattern applies.
08Narrate continuouslyGoing silent is the worst signal. If you are stuck, say what you know and what you are trying — that is itself useful signal for the interviewer.
Self-test before you move on

Cover the notes and answer aloud:
(1) What does each letter of UMPIRE stand for, and what is the one deliverable you must produce in each phase before moving to the next?
(2) If the problem says n can reach 200,000, which complexity classes are acceptable and which will almost certainly time out?
(3) Why should you always state the brute-force approach first, even if you already know a better one?
(4) Name three clarifying questions you would ask at the start of any new problem, and explain which phase is forced by the answer to each one.
(5) In the Two Sum walkthrough, why does the hash-map approach record a value only after checking for its complement? What bug does that ordering prevent?
(6) What does "stating the invariant" mean in the Plan phase, and why does it matter for proving correctness?
