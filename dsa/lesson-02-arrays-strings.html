02Concept · Coding Interview

Arrays & Strings

Before any pattern clicks, you need a precise mental model of the structure it runs on. Arrays and strings are the substrate — the flat, contiguous blocks of memory that show up in the majority of interview problems. Understanding their cost profile is what lets you argue, quickly and correctly, why your O(n) solution is better than a naive O(n²) one.

00 Foundation

The substrate everything else runs on

An array is a block of memory where all elements are the same size and sit side by side. The index is not magic — it is arithmetic: the address of element i is just base_address + i × element_size. That formula is computed in a single instruction, which is why array reads and writes at a known index are O(1) regardless of array size. Nothing else you will learn in this course is quite so fast at random access.

A string is an array of characters wearing a slightly different hat. In languages like Java and Python, strings are immutable — once created, the bytes cannot be changed in place. In C++, std::string is mutable. That one distinction produces some of the worst hidden-cost traps in interview code, so it gets its own section later.

Most interviews hand you an array (or its character-array cousin, a string) and ask you to manipulate it under constraints. Understanding the cost of each operation — not just time, but the reason for that cost — is what lets you choose the right technique and explain it persuasively to the interviewer.

Why the memory model matters

Most higher-level arrays (Python lists, Java ArrayLists) are dynamic arrays under the hood — they allocate a bigger backing block than they need, so that appending to the end is cheap. The amortised O(1) append cost comes from a doubling strategy: whenever the block fills, a new block twice the size is allocated and the contents are copied. Infrequent but expensive copies average out over many appends. Knowing this explains why appending is fast while inserting in the middle is not.

01 Cost table

Operation costs and why they are what they are

Knowing a big-O is only useful if you know the reason. Below is a cost table — study the "why" column; that is what you say to the interviewer. The interactive diagram beneath it walks through three representative operations on a concrete array so the motion matches the cost.

Operation Cost Why
Read / write at index i O(1) Direct address calculation — one multiply and one memory access.
Append to the end (dynamic array) O(1) amortised Backing buffer usually has free space. Occasionally the buffer doubles (O(n) copy), but averaged over many appends the per-operation cost is constant.
Insert or delete at the front O(n) Every existing element must shift one position to make room (insert) or close the gap (delete). At n = 100,000 that is 100,000 moves.
Insert or delete in the middle at index i O(n) All elements from i onward must shift. On average half the array moves, so the constant is 0.5n — still O(n).
Slice a[i:j] O(j − i) A slice copies the selected elements into a new array — proportional to the slice length, not the total array size.
Search (unsorted) O(n) Must scan every element in the worst case; the target might be last or absent.
Search (sorted, binary) O(log n) Each comparison halves the remaining candidates.
The interview implication

Whenever your solution inserts or deletes at the front or middle of an array inside a loop, you have an O(n) operation nested inside an O(n) loop — that is O(n²) total, which will likely TLE on large inputs. The fix is usually to work from the back (so shifts are avoided), or to use the read/write two-pointer pattern (which never shifts anything — it only copies keepers forward once).

02 In-place tricks

In-place reversal and index scanning

Many array problems want the answer produced without allocating a second array of the same size. The swap-from-both-ends reversal is the simplest in-place operation, and it is the building block for the three-reversal rotate trick in the next section.

The pattern: place one pointer at the left end and one at the right. While they have not crossed, swap a[L] and a[R] and move both inward. After at most n/2 swaps, the array is reversed — no extra memory, O(n) time. The invariant is that at every moment the already-reversed prefix and suffix are correct; only the middle window is unprocessed.

python — in-place reversal helper

def reverse_range(arr, lo, hi):
    # Swap arr[lo..hi] in place — O(hi-lo) time, O(1) space.
    while lo < hi:
        arr[lo], arr[hi] = arr[hi], arr[lo]
        lo += 1
        hi -= 1

In-place (swap)

  • O(n) time, O(1) extra space.
  • Works when you are allowed to mutate the input.
  • No allocation — fast in practice; cache-friendly.

Copy-and-build

  • O(n) time, O(n) extra space.
  • Cleaner to reason about; original is preserved.
  • Useful when output must be a new container (e.g., immutable strings).
03 Rotate by k

The three-reversal rotate (Rotate Array, 189)

Rotate Array (LeetCode 189) asks you to shift every element of an array k positions to the right, in place, using O(1) extra space. Moving elements one slot at a time takes O(n · k) in the worst case — quadratic when k ≈ n/2 and n is large. Allocating a second array is O(n) space and the problem asks you not to.

The elegant O(n)-time, O(1)-space answer uses exactly three reversals of the helper from section 02:

  1. Reverse the entire array. The elements that should end up at the front are now at the back, but in the wrong order within their group.
  2. Reverse the first k elements. Those elements are now in the right order at the front.
  3. Reverse the remaining n − k elements. The back section is corrected.

Dry-run on [1, 2, 3, 4, 5, 6, 7] with k = 3: after step one you get [7, 6, 5, 4, 3, 2, 1]; after step two (reverse first 3) [5, 6, 7, 4, 3, 2, 1]; after step three (reverse last 4) [5, 6, 7, 1, 2, 3, 4] — which is exactly right.

Worked problemMediumLeetCode 189
Rotate Array

You are given an integer array and a non-negative integer k. Rotate the array to the right by k positions — so the last k elements wrap around to the front. Do it in place with O(1) extra space.

Think it through

Restate

Shift every element k slots to the right, wrapping around; mutate the input directly and use only a fixed amount of extra memory.

Brute force

Perform k single-slot right rotations one at a time. Each rotation saves the last element, shifts everything right by one, and places the saved element at index 0. That is O(n) work per rotation, O(n · k) total — O(n²) when k is near n/2.

The waste

Each brute-force rotation moves every element, but most elements end up somewhere they just came from a step ago. The three-reversal approach moves every element exactly twice — once in each of the two sub-reversals — so the total is 2n moves, still O(n) but with a tiny constant.

The move

Clamp k to k %= n (if k equals n, no rotation is needed). Then: reverse the whole array, reverse a[0 … k-1], reverse a[k … n-1]. Each reversal is the O(1)-space helper from section 02. Three passes is still O(n).

Invariant & cost

After each reversal step the portions touched are locally correct; after all three the whole array is in the right order. Total: O(n) time, O(1) space. Watch the edge case k = 0 (no-op) and k = n (same, handled by k %= n).

The code

python

def rotate(nums, k):
    n = len(nums)
    k %= n                        # k = n is a no-op; k > n wraps
    if k == 0:
        return

    def rev(lo, hi):              # reverse nums[lo..hi] in place
        while lo < hi:
            nums[lo], nums[hi] = nums[hi], nums[lo]
            lo += 1; hi -= 1

    rev(0, n - 1)               # step 1: reverse entire array
    rev(0, k - 1)               # step 2: reverse first k elements
    rev(k, n - 1)               # step 3: reverse the remaining n-k
Narrate it to the interviewer

"Restating: right-rotate by k in place, O(1) space. The naive approach does k separate single-slot rotations — O(n·k), quadratic when k is large. The key insight is that three reversals achieve the same result in O(n). First I reverse the whole array; now the last k elements are at the front, but reversed within their group. A second reversal on just those k elements fixes their order. A third reversal on the remaining n-k elements fixes the back. Every element moves at most twice across the three reversals — O(n) total, O(1) space. I clamp k with k %= n first so k = n or k = 0 are both handled as no-ops."

Dry-run aloud: k = 0 (no-op via the early return), k = n (same, from k %= n), a one-element array (rev does nothing), and k = 1 on a two-element array to confirm the reversal boundaries are correct.

04 String traps

Immutable strings and the O(n²) concatenation trap

Strings look like arrays but carry a hidden constraint in most languages: they are immutable. In Python, Java, and JavaScript, every time you write s = s + c inside a loop, you are not modifying the string in place — you are allocating a brand-new string of length len(s) + 1, copying every character, and discarding the old one. Do that n times and you copy 1 + 2 + 3 + … + n characters: O(n²) total work for what looks like "just building a string".

python — the bad pattern vs. the fix

# Bad: O(n²) — each += allocates and copies the whole string so far
def build_slow(chars):
    result = ""
    for c in chars:
        result += c          # new string every iteration
    return result

# Good: O(n) — collect into a list, join at the end
def build_fast(chars):
    parts = []
    for c in chars:
        parts.append(c)      # O(1) amortised — no copy
    return "".join(parts)   # one O(n) allocation at the end
Language-specific notes

In Python: strings are immutable; always accumulate into a list and join. In Java: use StringBuilder — it maintains a mutable char buffer and only converts to a String when you call .toString(). In C++: std::string is mutable and has push_back, so naive concatenation is not always O(n²), but + with temporaries still allocates. In JavaScript: strings are immutable; accumulate into an array and call .join(''). The safe habit in any language: if you are building a string character by character, use a mutable buffer and commit once.

Slicing is also a copy

In Python, s[i:j] creates a new string of length j − i. That is O(j − i), not O(1). If your algorithm takes many slices in a loop, it may be secretly quadratic. The fix is to track index boundaries without slicing and only extract the slice when you actually need to return it — the two-pointer trick applied to strings.

05 What comes next

Where this shows up next

Arrays and strings are not a pattern themselves — they are the ground every pattern runs on. Once you are fluent with the cost table and the in-place manipulation idioms, the following lessons each add one algorithmic idea that exploits the array's properties:

L10Two Pointerspattern lesson

Two indices scan the same array under a monotonic rule — converging from both ends, or read/write from the front. Relies on the O(1) random access and the ability to swap in place (both covered here).

Key dependency In-place swap (section 02 above) is the primitive both converging and read/write variants use.

L12Sliding Windowpattern lesson

A contiguous subarray of variable or fixed length moves across the array, maintaining an invariant about its contents. The amortised O(1) cost of advancing each end is what makes the whole sweep O(n).

Key dependency Understanding why end + 1 and start + 1 are O(1) (no shifting, just pointer arithmetic) is the prerequisite for the O(n) argument.

L13Prefix Sumspattern lesson

Precompute a running total across the array so that any subarray sum can be answered in O(1) by subtraction. The precomputation is a single O(n) scan; queries are then trivially O(1) random access.

Key dependency The O(1) index access that makes prefix[j] − prefix[i] a constant-time operation is exactly the property from section 01.

The recurring shape

Notice the theme: every pattern lesson reduces some repeated or nested work down to a single O(n) scan by exploiting a property of the array — either its index arithmetic (random access) or its in-place mutability (swap/overwrite without allocation). Knowing which property a technique is using makes it easier to remember and easier to explain.

★ Review

Summary & self-test

Arrays and strings underpin almost every interview problem. Their defining property — contiguous, same-size elements accessible in O(1) — is both their great strength (fast reads) and the source of the two most common interview bugs: hidden O(n²) from middle inserts, and hidden O(n²) from naive string concatenation. Internalise the cost table and the three-reversal trick, and you have the vocabulary every subsequent pattern lesson assumes.

01Random access is O(1)Address = base + i × element_size. One multiply, one load. The whole array is irrelevant.
02Append is O(1) amortisedDoubling strategy: occasional O(n) copies average out over many appends. Middle insert is always O(n).
03Shift = O(n)Inserting or deleting in the middle forces every element after the point to move. Common hidden cost inside loops.
04Slice = O(length)A slice copies selected elements into a new object — O(j − i), not O(1). Track indices instead of slicing in tight loops.
05Swap in-place = O(1)No allocation — just two assignments. The reversal helper uses exactly ⌊n/2⌋ swaps.
06String += is O(n²)Each concatenation copies the entire string built so far. Use a list and join once at the end.
07Three reversals → rotateReverse all, reverse first k, reverse the rest. O(n) time, O(1) space — no extra array needed.
08Index arithmetic beats allocationWhen tracking a subarray boundary, store two integers — not a slice. Allocate only when you must return a result.
