Monotonic Stack & Queue

A surprisingly large family of array problems — "what is the next bigger value?", "how tall a rectangle fits here?", "what is the maximum in this sliding window?" — all share a hidden bottleneck: repeatedly re-scanning backward through already-examined elements. A monotonic stack or monotonic deque eliminates that scan entirely by maintaining a data structure whose contents stay in sorted order at every step, letting you answer range queries in amortized O(1).

Problem names below are referenced only as well-known identifiers; every description, diagram, walkthrough, and line of code is original. 00 The idea

What problem does it solve?

Consider the question: for each position in an array, find the next element to the right that is larger than the current one. The obvious answer is to nest loops — for every index i, walk right until you find something bigger. On an array of length n that is O(n²) comparisons in the worst case. For n = 100,000 on a warm server that still feels slow; for n = 10⁶ it is impossible within a 2-second limit.

The waste is easy to see: every time we look right from a new position, we re-examine elements we already visited. The insight is that we only ever need to keep track of the candidates — elements that might still be the "next greater" for some future position to the left. When a new, larger element arrives, it instantly resolves all waiting candidates that are smaller than it. A stack is the right structure for this: it naturally remembers a sequence of candidates, and we pop them as each is answered.

The invariant that the stack must maintain depends on the query:

  • Next greater / smaller element — keep a decreasing (or increasing) stack of indices. When a new value beats the top, pop and record the answer.
  • Widest rectangle that fits below a height boundary — same decreasing stack; the pop event tells you the width of the rectangle ending here.
  • Sliding-window maximum — a decreasing deque of indices; evict from the left when the oldest index falls out of the window, pop from the right when a new value is at least as big as the back.
The reusable shape

Every monotonic-stack problem has the same skeleton: iterate left to right; for each new element, pop the stack while the top violates the monotone property (and record the answer for that popped element); then push the current index. Because each index is pushed exactly once and popped at most once, the total work across all iterations is O(n) regardless of how many pops happen at any single step.

01 Core mechanic

Next greater element with a monotonic stack ▸

The brute-force approach for "find the next greater element to the right of each position" costs O(n²): for each index i, scan every index j > i until finding a larger value. When n is large, the inner scan re-examines elements that earlier iterations already visited — that repeated work is the bottleneck.

The insight: whenever a new element v arrives, it is the "next greater" for every waiting candidate whose value is less than v. We keep a decreasing stack of indices. Before pushing index i, pop every index j whose value a[j] < a[i] — at that pop moment, a[i] is the answer for position j. Anything left in the stack at the end of the scan has no next greater (record −1 or ∞).

The invariant: values of the indices currently on the stack are non-increasing from bottom to top. Every element is pushed once and popped once, so the total cost across the entire array is O(n) time, O(n) space for the stack and result array.

Decreasing vs increasing stack

Push raw indices onto the stack (not values), so you can write both the answer and the position at pop time. Use a decreasing stack (pop when the new value is greater) for "next greater" queries. Flip to an increasing stack (pop when the new value is smaller) for "next smaller." The code changes by exactly one comparison operator.

Worked problemMediumLeetCode 496
Next Greater Element I

You have two arrays of distinct integers, nums1 and nums2, where every element of nums1 also appears in nums2. For each element in nums1, find the first value to its right in nums2 that is strictly larger, or −1 if none exists.

Think it through

Restate

Pre-compute "next greater in nums2" for every element, then look up each query from nums1 in O(1). The output array has the same length as nums1.

Brute force

For each element in nums1, find its position in nums2, then scan right until a larger value appears. That is O(m · n) where m = |nums1| and n = |nums2| — O(m · n) in the worst case.

The waste

We re-scan a suffix of nums2 for every query. If we precompute the next-greater for every position in nums2 once, every lookup is O(1). The precomputation is exactly the monotonic-stack sweep from FIG. 1.

The move

Walk nums2 left to right with a decreasing stack of indices. Before pushing index j, pop every index k where nums2[k] < nums2[j], and store nxt[nums2[k]] = nums2[j]. Remaining stack entries get −1. Then for each element of nums1 just return nxt[x].

Invariant & cost

Stack stays non-increasing in value; each of the n elements is pushed and popped at most once. Precomputation: O(n). Lookup for m queries: O(m). Total: O(n + m) time, O(n) space.

The code

python

def nextGreaterElement(nums1, nums2):
    nxt = {}                        # value -> its next-greater value in nums2
    stk = []                        # decreasing stack of values waiting for answer

    for v in nums2:
        while stk and stk[-1] < v:   # v beats the top → it's the answer
            nxt[stk.pop()] = v
        stk.append(v)

    for v in stk:               # anything left has no next-greater
        nxt[v] = -1

    return [nxt[x] for x in nums1]
Narrate it to the interviewer

"The brute force scans a suffix of nums2 for every query — O(m · n). The waste is repeating that scan. So I'll precompute the next-greater for every element in nums2 once with a monotonic stack, then each query is just a hash-map lookup. I walk nums2 left to right with a decreasing stack of values. Whenever a new value beats the top, the top's answer is that new value — I pop and store it. Anything still in the stack at the end has no next-greater, so I record −1. That's O(n) precomputation and O(m) lookup, O(n + m) total with O(n) space."

Dry-run aloud: nums2 is already descending (no pops — all answers are −1); nums2 is ascending (every element pops all prior ones); nums1 has a single element that is the largest in nums2.

What the interviewer asks next

"What if nums2 is circular — elements wrap around?" → Simulate two full passes over nums2 (indices 0 to 2n−1 modulo n) with the same stack; elements popped on the second pass have their circular next-greater answered. "What if you need the next smaller instead?" → flip the comparison: pop while stk[-1] > v. "What if values are not distinct?" → store indices on the stack instead of values, so duplicate values each get their own answer slot.

02 Index distances

Daily temperatures: waiting for a warmer day ▸

The "daily temperatures" problem is the same monotonic-stack mechanic, but the answer is not the larger value itself — it is the distance (number of days) until that larger value arrives. This is why we store indices on the stack rather than values: at pop time the distance is simply current_index − popped_index.

Brute force scans right from each day until finding a warmer reading — O(n²) in the worst case (e.g., temperatures sorted descending). The monotonic-stack approach handles every day in amortized O(1): each index is pushed once and popped at most once, yielding O(n) total.

Worked problemMediumLeetCode 739
Daily Temperatures

Given a list of daily temperatures, return an array where each position holds how many days you must wait for a strictly warmer temperature. If no warmer day ever comes, that position is 0.

Think it through

Restate

For each index i, find the smallest j > i with T[j] > T[i] and record j − i. Output 0 if no such j exists.

Brute force

For each day scan rightward until a warmer day appears. That is a nested loop: O(n²) in the worst case — a monotonically decreasing temperature sequence forces the inner scan to reach the end every time.

The waste

The inner scan revisits indices that the outer loop already processed. We want each index answered the moment its warmer day arrives — without re-scanning. The monotonic stack catches arriving "warmer days" and dispatches all waiting cooler days at once.

The move

Maintain a decreasing stack of indices. For each new index i: while T[i] > T[stk[-1]], pop index j from the stack and record ans[j] = i − j. Then push i. Anything remaining in the stack after the full scan stays at 0 (no warmer day).

Invariant & cost

The values of the stacked indices decrease from bottom to top (a new index is pushed only after popping all smaller-temperature indices). Each index enters once and exits once: O(n) time, O(n) space.

The code

python

def dailyTemperatures(temps):
    n = len(temps)
    ans = [0] * n
    stk = []                        # decreasing stack of indices

    for i in range(n):
        while stk and temps[i] > temps[stk[-1]]:
            j = stk.pop()
            ans[j] = i - j          # days waited = distance between indices
        stk.append(i)
    # remaining indices in stk have no warmer day → stay 0
    return ans
Narrate it to the interviewer

"The answer for position j is the index of the first warmer day minus j. Brute force is O(n²) — the inner scan re-examines every earlier day. The fix is a decreasing stack of indices. Each day I check whether the current temperature beats the top of the stack. If it does, the current day is the answer for that popped index, and the wait is the difference in indices. I keep popping until the stack is empty or the top is at least as warm. Then I push the current index. After the scan, anything still on the stack has no warmer day so its answer stays 0. O(n) time, O(n) space — each index pushed once, popped once."

Dry-run aloud: a single element (stack empties, answer [0]); a strictly increasing sequence (each new day pops all prior, answers are all 1 except the last which is 0); a strictly decreasing sequence (nothing ever pops, all answers are 0).

03 Width extension

Largest rectangle in a histogram

The histogram rectangle problem asks: given an array of non-negative bar heights, find the area of the largest axis-aligned rectangle that fits within the histogram. The brute force considers every pair of left and right boundaries and finds the minimum height in between — O(n²) pairs, each needing O(n) for the minimum in the naive version, or O(n²) with a running minimum. Either way, quadratic in n.

The key insight: a rectangle's height is constrained by its shortest bar. We want to know, for each bar of height h[i], how far left and right we can extend before hitting a shorter bar. The pop event on a decreasing stack tells us exactly that: when we pop index j because h[i] < h[j], the current index i is the first shorter bar to the right of j, and the element now on top of the stack after the pop is the first shorter bar to the left of j. The width of the rectangle rooted at height h[j] is thus i − stk[-1] − 1 (or just i if the stack is empty). Appending a sentinel height of 0 at the end forces any remaining bars to be popped and measured.

The sentinel trick

Appending a zero-height bar at the end (index n) guarantees the loop pops every remaining index in the stack before finishing. Without it you would need a second pass or special-casing. Similarly, prepending a sentinel at the beginning (or pushing index −1 as a base) handles the case where a bar extends all the way to the left edge.

Worked problemHardLeetCode 84
Largest Rectangle in Histogram

An array of non-negative integers represents a histogram where each bar has width 1. Find the area of the largest rectangle that can be drawn within the outline of the histogram.

Think it through

Restate

For each pair of left/right boundaries, the rectangle height is the minimum bar in that range and the area is height × (right − left + 1). Find the maximum such area.

Brute force

Try every pair of left and right boundaries. Track a running minimum across the range as we expand. That is O(n²) pairs, each O(1) with the running min — O(n²) time.

The waste

We re-examine pairs we already know are sub-optimal. We want to jump straight to: "for bar i acting as the height constraint, how wide is the best rectangle?" The monotonic stack answers that exactly at pop time.

The move

Push a sentinel index −1 (height 0) first. Walk left to right; push each index while bars are non-decreasing. When h[i] < h[stk[-1]], pop index j — h[j] is the height, width is i − stk[-1] − 1, area is their product. Update the global max. Append a height-0 sentinel at the end so all remaining indices get popped in the final iterations.

Invariant & cost

Stack contains indices whose heights are non-decreasing from bottom to top. Each of the n bars is pushed once and popped at most once. O(n) time, O(n) space.

The code

python

def largestRectangleArea(heights):
    heights = heights + [0]         # sentinel flushes remaining bars
    stk = [-1]                       # sentinel base for width calculation
    best = 0

    for i, h in enumerate(heights):
        while stk[-1] != -1 and h < heights[stk[-1]]:
            ht = heights[stk.pop()]
            width = i - stk[-1] - 1   # gap between new top and current i
            best = max(best, ht * width)
        stk.append(i)

    return best
Narrate it to the interviewer

"The brute force is O(n²) — for every pair of boundaries I compute area with a running minimum. The insight is: for each bar acting as the height limiter, I want to know how far left and right I can stretch before hitting something shorter. The pop event on a decreasing stack tells me exactly that. I push a base sentinel −1 and a trailing 0. When bar i is shorter than the stack top, I pop that top — its height is the rectangle height, the width is i − new_top − 1, and I update the global maximum. Everything is pushed once and popped once, so it is O(n) time and O(n) space."

Dry-run aloud: a single bar (area = that bar's height); all equal heights (one large rectangle across all bars); strictly increasing (no pops until the trailing sentinel); strictly decreasing (each new bar pops all prior ones).

What the interviewer asks next

"How does this extend to a 2D grid of 0s and 1s?" (Maximal Rectangle, LeetCode 85) → treat each row as a histogram built from the cumulative column heights above it; run the 1D histogram algorithm on each row. That gives O(m · n) overall. "Can you solve it without a sentinel?" → yes, add a second while-loop after the main scan to drain the stack, using n as the right boundary. The sentinel is just cleaner.

04 Stack variant

Trapping rain water

In the classic "trapping rain water" problem, an array encodes elevation heights. After a rainfall, water pools wherever a lower bar is bounded on both sides by taller bars. The brute-force approach computes, for each position, the maximum height to its left and the maximum height to its right, then the water at that position is min(leftMax, rightMax) − height[i]. Finding those two maxima naively via scans from each position is O(n²).

A monotonic stack solves this in O(n) by computing water horizontally in layers. We maintain a non-increasing stack of indices. When a taller bar arrives at index i, the bar just popped (index mid) is the bottom of a water pocket. The water width is i − stk[-1] − 1 (the gap between the current bar and the new stack top). The water height is min(height[i], height[stk[-1]]) − height[mid]. We accumulate this area and keep popping until the stack can accept the new bar.

Two-pointer alternative

There is a well-known two-pointer solution (from the two-pointers lesson) that runs in O(n) time and O(1) space by tracking running left-max and right-max as two pointers converge. The monotonic-stack approach uses O(n) space but is worth knowing because it generalises to 2D "trapping" and pairs naturally with the histogram variant on the same problem sheet.

Worked problemHardLeetCode 84 variant
Trapping Rain Water (stack approach)

An array of non-negative bar heights represents an elevation map. Compute how many units of water are trapped between the bars after it rains, using a monotonic stack approach.

Think it through

Restate

Water sits in "pits" — places where a lower bar is flanked by taller bars on both sides. Sum the water volume across all pits.

Brute force

For each position, scan left and right for the tallest bar on each side, then add min(leftMax, rightMax) − height[i]. That is O(n²) with the naive scan (or O(n) with precomputed prefix max arrays — but both add O(n) space; the stack approach earns its place because it generalises).

The waste

The prefix-max precomputation visits every element twice. The stack approach interleaves the discovery of boundaries with the accumulation of water — no separate precomputation pass.

The move

Maintain a non-increasing stack of indices. When height[i] > height[stk[-1]], pop mid. If the stack is now empty, there is no left wall — no water. Otherwise, the water height is min(height[i], height[stk[-1]]) − height[mid] and the width is i − stk[-1] − 1. Accumulate and continue popping.

Invariant & cost

Stack holds non-increasing heights; each index pushed and popped once. O(n) time, O(n) space.

The code

python

def trapWithStack(height):
    stk = []
    water = 0

    for i in range(len(height)):
        while stk and height[i] > height[stk[-1]]:
            mid = stk.pop()
            if not stk:
                break                      # no left wall → no water pocket
            h = min(height[i], height[stk[-1]]) - height[mid]
            w = i - stk[-1] - 1
            water += h * w
        stk.append(i)

    return water
Narrate it to the interviewer

"I'll think of water as horizontal layers. Whenever a new taller bar arrives, it forms the right wall of a potential water pocket. The bar I pop is the bottom of that pocket, and the new stack top is the left wall. The water height is min(left_wall, right_wall) − bottom and the width is the gap between the two walls. I keep popping because the same right wall may close multiple pockets. If the stack empties mid-pop there is no left wall — no water there. Each bar is pushed once and popped at most once: O(n) time, O(n) space."

Dry-run aloud: a flat array (no water); a V-shape (one pop at the peak, water fills the valley); bars all descending then one tall bar at the end (the tail bar pops everything and captures all water at once).

05 Deque upgrade

Sliding-window maximum with a monotonic deque ▸

The sliding-window maximum asks: given an array and a window of size k, report the maximum value in the window as it slides from left to right. The naive answer — scan all k elements in every window — is O(n · k). A sorted data structure (like a heap) can get it to O(n log k), but a monotonic deque achieves O(n) — the maximum is always at the front of the deque in O(1).

The key difference from a stack: we need to evict old elements from the front when they fall out of the current window, while still popping dominated elements from the back. That bidirectional access is exactly what a deque (double-ended queue) provides.

The invariant: the deque holds indices in left-to-right order, and their corresponding values are strictly decreasing from front to back. Before appending index i, pop from the back every index j with a[j] ≤ a[i] — those indices can never be a future window maximum (the newer, at least equally large value a[i] will always be in the window too, and for longer). Also, discard from the front any index that has slid out of the window. The front is always the index of the current window's maximum.

Monotonic deque

  • O(n) time — each index enters and leaves the deque exactly once.
  • O(k) extra space for the deque at most.
  • Front always holds the window maximum in O(1).
  • Works for any aggregate that is a running max or min.

Max-heap (priority queue)

  • O(n log k) time — each push and lazy-pop costs O(log k).
  • O(k) space for the heap; needs lazy deletion for stale entries.
  • More flexible: supports min, median, or k-th largest with small tweaks.
  • Better choice when the aggregate is something other than max/min.
Worked problemHardLeetCode 239
Sliding Window Maximum

Given an integer array and a window size k, return an array of the maximum value in each window as the window slides one step at a time from left to right.

Think it through

Restate

For each starting index i from 0 to n − k, report max(arr[i … i + k − 1]). Output has n − k + 1 values.

Brute force

For each window, scan all k elements and find the max. Across all n − k + 1 windows that is O(n · k) — for large k this is effectively O(n²).

The waste

When the window slides one step, only one element enters and one leaves. We re-scan k − 1 elements we just saw. The monotonic deque keeps the window's maximum alive at its front without re-scanning.

The move

Use a collections.deque storing indices. For each new index i:
(1) pop from the back any index j with arr[j] ≤ arr[i] — they are dominated and can never win a future window.
(2) Push i onto the back.
(3) Pop from the front any index that is outside the current window (dq[0] < i − k + 1).
(4) If the window is full (i ≥ k − 1), record arr[dq[0]] as the current window's maximum.

Invariant & cost

Values at indices in the deque are strictly decreasing from front to back. Each index enters and leaves the deque at most once. O(n) time, O(k) space.

The code

python

from collections import deque

def maxSlidingWindow(nums, k):
    dq = deque()                    # stores indices; front = max of current window
    result = []

    for i in range(len(nums)):
        # remove from back: new value dominates these
        while dq and nums[dq[-1]] <= nums[i]:
            dq.pop()
        dq.append(i)

        # remove from front: index slid out of window
        if dq[0] < i - k + 1:
            dq.popleft()

        # window is full: record the maximum
        if i >= k - 1:
            result.append(nums[dq[0]])

    return result
Narrate it to the interviewer

"The naive approach scans all k elements per window — O(n · k). The insight is that when a new element enters the window, any prior element that is smaller or equal is dominated: it can never be the maximum of any future window, because this new element will be in the window too and for at least as long. So I maintain a decreasing deque of indices. For each new index I pop dominated values from the back, then push. I also pop from the front if that index has slid out of the window. The front is always the window max in O(1). Each index is pushed once and popped at most once: O(n) time, O(k) space."

Dry-run aloud: k = 1 (every window is a single element, deque always holds exactly one index); k = n (one window, answer is the global max, popped only at index n − 1); all equal elements (back-pop fires every step but the front never changes).

What the interviewer asks next

"Can you do sliding window minimum?" → flip the comparison: pop from the back while nums[dq[-1]] ≥ nums[i]. The deque becomes increasing. "What about sliding window median?" → two heaps (a max-heap for the lower half, a min-heap for the upper half), not a deque — different tool for a harder aggregate. "How does this relate to the next-greater pattern from section 01?" → both maintain a monotone invariant on the stack/deque; the difference is that the deque also tracks a left boundary (the window), so it needs pops from both ends.

How to drill these effectively

Work top to bottom. For each problem:
(1) say the brute force out loud and give its big-O;
(2) name the bottleneck the monotonic structure removes;
(3) state the invariant — what property the stack or deque maintains at all times;
(4) code it from scratch in a plain editor, no autocomplete;
(5) dry-run at least three edge cases: an empty or single-element input, a monotonically increasing or decreasing sequence, and all-equal values. Pay special attention to the sentinel trick for the histogram problems — drawing a pop event on paper before coding saves time in a real interview.

★ Review

Summary & self-test

Every problem in this lesson shares one root cause: a nested scan that re-examines elements already processed. The monotonic stack or deque eliminates that re-examination by maintaining a sorted invariant — when a new element enters, it pops all elements it renders useless, answering their queries in the same move. Because each element is pushed and popped at most once, the total work across the entire array is O(n) regardless of how many pops occur at any single step.

01Root causeAn inner loop re-visits elements already seen — the monotonic structure eliminates that scan.
02Decreasing stackPop when new value is greater. Records "next greater" at pop time. Keeps candidates whose queries are still open.
03Store indicesPush indices, not values, so pop time gives both the answer value and the position for distance / width calculations.
04Sentinel barsA height-0 sentinel at the end (and index −1 at the base) flushes remaining stack entries and simplifies the width formula.
05Monotonic dequeAdds left-boundary eviction (popleft) to handle sliding windows; front = window max, back = dominated candidates.
06Amortized O(n)Each index enters once and exits once. The total pop count across all steps is ≤ n, so the overall loop is O(n) even with inner while-loops.
07Flip for minimumSwap the comparison: increasing stack for "next smaller"; increasing deque for sliding-window minimum. Same code, one character change.
08Stack vs dequeUse a plain stack when there is no left-boundary constraint. Upgrade to a deque only when old elements must also be evicted from the front.
