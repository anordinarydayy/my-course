Intervals

Most interval problems share a single setup move: sort by start time, then sweep left to right. From there, one of four jobs follows — collapsing overlaps, stitching in a new interval, counting how many are alive at once, or greedily picking the maximum non-overlapping subset. Recognizing which job the question is asking for is the whole skill.

Problem names below (Merge Intervals, Insert Interval, Meeting Rooms II, Non-overlapping Intervals) are referenced only as well-known identifiers; every description, diagram, walkthrough, and line of code here is original. 00 The idea

Sort by start, then sweep

An interval is a pair [start, end] representing a closed range on a number line — a meeting time, a task window, a gene segment. The central challenge is that the input arrives in arbitrary order, which makes it hard to reason about which intervals touch which others. The opening move for nearly every interval problem is the same: sort by start time. Once the intervals are sorted, the sweepline invariant kicks in: any interval that can possibly overlap the current one must come immediately after it in the sorted order, so a single left-to-right pass is enough to find all interactions.

After sorting, the logic branches depending on what the problem asks:

  • Merge — extend the current interval's end as long as the next one's start is at or before the current end. Emit when no more overlap.
  • Insert — copy non-overlapping intervals before the new one, merge everything that touches it, copy the rest.
  • Count concurrent — track how many intervals are open at every moment; the peak is the answer.
  • Schedule — greedily pick intervals by their earliest end time to leave as much room as possible for future ones.
The sorting cost is paid once

Every algorithm below costs O(n log n) for the sort and then O(n) for the sweep — total O(n log n). If the input is already sorted (stated in the problem, or you built it that way), the sort is free and the whole job is O(n). Always confirm the sort assumption out loud.

01 Merge

Collapsing overlapping intervals ▸

After sorting by start, maintain a current interval — the candidate for the next output block. For each incoming interval: if its start is greater than the current end, the two cannot overlap (because everything after is also further right), so emit the current interval and replace it with the incoming one. Otherwise — next.start <= current.end — they overlap, and you extend the current end to max(current.end, next.end). The max is important because the incoming interval might be entirely contained inside the current one, in which case the end should not shrink. Repeat until all intervals are consumed, then emit the final current.

The brute-force approach — for each interval, scan all others to find overlapping ones and merge repeatedly — runs in O(n²). The sort-and-sweep above does it in a single left-to-right pass after sorting: O(n log n) time, O(n) space for the output list.

Worked problemMediumLeetCode 56
Merge Intervals

You receive a list of intervals, each [start, end], in no particular order. Return a new list in which all overlapping intervals have been merged so that no two intervals in the output share any point. Two intervals that share an endpoint count as overlapping.

Think it through

Restate

Collapse all touching or overlapping intervals into the fewest non-overlapping blocks that cover the same points on the number line.

Brute force

For each interval, scan every other to find overlaps, merge, repeat until stable. This is O(n²) in the worst case and complicated to implement correctly.

The waste

The brute force does not exploit order. If we sort by start, any interval that overlaps the current one must come immediately after it — we never need to look back. That shrinks the scan to a single pass.

The move

Sort intervals by start. Initialize cur = intervals[0]. For each subsequent interval nxt: if nxt[0] <= cur[1] they overlap — set cur[1] = max(cur[1], nxt[1]). Otherwise emit cur and set cur = nxt. After the loop, emit the final cur.

Invariant & cost

At every point in the sweep, cur is the merged interval of all intervals seen so far that have not yet been emitted. Each interval is processed exactly once. O(n log n) time (dominated by the sort); O(n) space for the output.

The code

python

def merge(intervals):
    intervals.sort(key=lambda x: x[0])   # sort by start
    out = [intervals[0]]                  # seed with the first interval
    for start, end in intervals[1:]:
        if start <= out[-1][1]:             # overlaps current tail
            out[-1][1] = max(out[-1][1], end)  # extend — don't shrink!
        else:
            out.append([start, end])        # gap: emit new interval
    return out
Narrate it to the interviewer

"The problem is: collapse all overlapping intervals into the minimal non-overlapping set. The brute force checks every pair — O(n²). But if I sort by start, any interval that can overlap mine must be the very next one, so a single sweep suffices. I keep the last output interval as my current candidate; if the next interval's start is at or before my current end, they overlap and I extend the end to whichever is larger — I use max because the incoming interval might be entirely contained inside mine. If there's a gap, I append the current candidate and start a new one. The invariant is that the output list is always a valid non-overlapping set of merged intervals. O(n log n) for the sort, O(n) for the sweep."

Dry-run aloud: a single interval (returns it unchanged), all intervals identical (one merged result), intervals sorted in reverse (sort fixes it), and a fully contained interval like [1,10] followed by [3,5] — end must not shrink to 5.

What the interviewer asks next

"What if the intervals are given as tuples (immutable)?" — convert to lists on input or store start/end separately. "What if you get intervals one at a time as a stream?" — maintain a sorted structure (e.g., a balanced BST or sorted list) and merge lazily. "Do endpoints touching count as an overlap?" — yes per the standard problem statement; confirm this aloud before coding.

02 Insert

Stitching a new interval into a sorted list

Here the input is already a sorted, non-overlapping list of intervals, and you are handed one new interval to slot in. The brute approach appends the new interval, re-sorts, and runs the full merge — which works but discards the fact that the list is already sorted, costing an unnecessary O(n log n). Because the list is pre-sorted, you can identify the new interval's position in a single left-to-right pass and do the job in O(n) time.

The pass has three natural phases:

  • Before — copy every existing interval that ends strictly before the new one starts. These cannot possibly overlap.
  • Overlap — while the current existing interval's start is at or before the new interval's end, the two overlap. Extend the new interval to cover both (expand start and end).
  • After — copy every remaining interval whose start is strictly after the merged result's end.

The key condition for the overlap phase is existing.start <= new.end. As long as this holds, the existing interval touches or overlaps the working new interval, so you absorb it: new.start = min(new.start, existing.start) and new.end = max(new.end, existing.end). When the condition fails for the first time, you insert the absorbed result and switch to copying.

Worked problemMediumLeetCode 57
Insert Interval

You have a sorted list of non-overlapping intervals and a single new interval [newStart, newEnd]. Insert the new interval into the list, merging with any intervals it overlaps, and return the result in sorted order. The input list is already sorted by start and contains no overlaps — do not re-sort it.

Think it through

Restate

The list is already sorted and clean. Slot in the new interval, merging anything it touches, without disturbing the existing order.

Brute force

Append the new interval and run the full Merge Intervals algorithm. Works, but costs O(n log n) when an O(n) solution exists because the list is already sorted.

The waste

Re-sorting throws away the pre-existing order. A single left-to-right scan can handle the three-phase structure directly.

The move

Use an index i. Phase 1: while intervals[i][1] < new[0], append intervals[i] to output and advance i. Phase 2: while i < n and intervals[i][0] <= new[1], expand: new[0] = min(new[0], intervals[i][0]), new[1] = max(new[1], intervals[i][1]), advance i. Append the merged new. Phase 3: append all remaining intervals[i:].

Invariant & cost

After phase 1, everything emitted ends before the new interval begins — no overlap possible. After phase 2, new covers every interval it could touch. Phase 3 starts beyond the new interval's end — again no overlap possible. Each interval is visited exactly once: O(n) time, O(n) space.

The code

python

def insert(intervals, newInterval):
    out = []
    i, n = 0, len(intervals)
    # phase 1: copy intervals that end before the new one starts
    while i < n and intervals[i][1] < newInterval[0]:
        out.append(intervals[i]); i += 1
    # phase 2: merge all intervals that overlap the new one
    while i < n and intervals[i][0] <= newInterval[1]:
        newInterval[0] = min(newInterval[0], intervals[i][0])
        newInterval[1] = max(newInterval[1], intervals[i][1])
        i += 1
    out.append(newInterval)         # emit the merged result
    # phase 3: copy the remaining intervals
    out.extend(intervals[i:])
    return out
Narrate it to the interviewer

"The list is already sorted, so I can do better than re-sorting. I'll walk left to right in three phases. Phase one: while an existing interval's end is strictly before my new interval's start, there's no possible overlap — I copy it straight to output. Phase two: while an existing interval's start is at or before my new interval's current end, they touch — I absorb it by taking the min start and max end. Phase three: everything left comes after my merged interval, so I copy it unchanged. Invariant: after phase one, everything emitted is strictly to the left; after phase two, the new interval covers every interval it can possibly reach. O(n) time, O(n) space."

Dry-run aloud: empty list (returns [newInterval]), new interval before all existing (only phase 3 fires), new interval after all existing (only phase 1 fires), and new interval spanning the entire list (all absorbed in phase 2).

03 Count concurrent

Maximum simultaneous intervals (Meeting Rooms) ▸

The question is: at the moment of peak activity, how many intervals are open at the same time? Think of it as a staffing question — you need one resource per concurrent interval, so the peak count is the minimum resource requirement.

The brute force would check every point on the axis, counting how many intervals contain it — O(n × range) or O(n²). Two cleaner approaches both cost O(n log n):

  • Sweep line / events: create a list of events — each interval contributes a +1 event at its start and a -1 event at its end. Sort all events by time (breaking ties by putting ends before starts, if open endpoints do not overlap). Scan and accumulate; the running maximum is the answer.
  • Two sorted arrays: sort the starts and ends separately. Walk two pointers — when the next start is before the next end, a new interval opened: count++. When the next end comes first, one closed: count--. Track the maximum count seen.

The interactive diagram below demonstrates the two-sorted-arrays sweep, which is the cleaner approach for an interview setting because both arrays come from a single sort step and the pointer logic is transparent.

Worked problemMediumLeetCode 253
Meeting Rooms II

You receive a list of meeting time intervals, each as [start, end]. Find the minimum number of conference rooms needed so that every meeting can run without sharing a room. Two meetings that share an endpoint cannot share a room.

Think it through

Restate

Determine how many meetings are running simultaneously at the peak — that is the minimum number of rooms required.

Brute force

Try every pair of time points in the axis, count how many meetings contain each point, return the maximum. That is O(n × max_time), which becomes enormous for large times.

The waste

Only start and end times matter — everything between two adjacent events is unchanging. Collapsing to just those events reduces the work to O(n log n) for sorting and O(n) for the sweep.

The move

Extract all starts into one array and all ends into another, then sort both. Use two pointers s and e starting at 0. While s < n: if starts[s] < ends[e], a new meeting has opened — count++, s++. Otherwise, a meeting has closed — count--, e++. Track the running maximum of count.

Invariant & cost

count is always the exact number of meetings open at the moment represented by starts[s]. When we process a start, we know at least one meeting has opened without a corresponding end having passed. O(n log n) time, O(n) space.

The code

python

def minMeetingRooms(intervals):
    starts = sorted(iv[0] for iv in intervals)
    ends   = sorted(iv[1] for iv in intervals)
    s, e = 0, 0
    count = max_count = 0
    while s < len(starts):
        if starts[s] < ends[e]:  # next event is an opening
            count += 1
            max_count = max(max_count, count)
            s += 1
        else:                    # a meeting ends before or as next starts
            count -= 1
            e += 1
    return max_count
Narrate it to the interviewer

"The minimum number of rooms equals the peak overlap — how many meetings are alive at the same time. Checking every point on the axis is too slow. Instead I extract starts and ends into two separate sorted arrays, then sweep with two pointers. When the next start is strictly less than the next end, a new meeting has opened before any old one closed — one more room is needed, so count goes up. Otherwise, a meeting ends first — a room is freed, count goes down. I track the maximum count seen. The invariant is that count always matches the number of currently open meetings. O(n log n) from the sorts, O(n) space."

Dry-run aloud: a single meeting (returns 1), non-overlapping meetings (returns 1), all meetings at the same time (returns n), and two meetings sharing only an endpoint — confirm the strict < ensures they need different rooms.

What the interviewer asks next

"Can you use a min-heap instead?" — yes: sort by start, push each interval's end onto a min-heap. For each new meeting, if the earliest-ending meeting has finished (heap top <= new start), reuse that room (pop and push); otherwise add a new room (just push). The heap size at the end is the answer. Both approaches are O(n log n) — name the trade-off: the sweep is constant-space after sorting, the heap uses O(n) extra but generalizes more naturally if you need to assign specific rooms.

04 Schedule

Maximum non-overlapping intervals ▸

The final variant is a scheduling problem: given n intervals, select as many as possible such that no two overlap. Equivalently, remove the minimum number to make the rest non-overlapping — these are two sides of the same coin (answer for one = n − answer for the other).

The brute force tries every subset — O(2ⁿ). A dynamic-programming approach exists in O(n²), but the greedy argument gives O(n log n) with no extra thought: always pick the interval that ends earliest. Why? Because picking an early-ending interval leaves the maximum time available for future choices. Any other choice of "first interval" produces a solution that is no better — you can always swap in the earliest-ending one without losing any future intervals.

The algorithm: sort by end time. Walk left to right. Keep track of the end time of the last interval you selected (last_end, initialized to −∞). For each candidate: if its start is at or after last_end, select it and update last_end. Otherwise skip it (one removal, for the minimum-removal variant).

Worked problemMediumLeetCode 435
Non-overlapping Intervals

Given a collection of intervals, return the minimum number of intervals you must remove so that the remaining intervals do not overlap. Two intervals that share only an endpoint are not considered overlapping.

Think it through

Restate

Find the maximum number of non-overlapping intervals you can keep. The number to remove is n − (number kept). Two intervals touching at a single point do not overlap.

Brute force

Try every subset, check if it is pairwise non-overlapping, track the largest valid one. That is O(2ⁿ) — clearly untenable even for n = 20.

The waste

We do not need to examine every combination. The greedy insight is that among all intervals that start at or after the current point, the one that ends earliest is always the best pick — it leaves the most room for future choices, and there is never a reason to prefer a later-ending interval at the same position.

The move

Sort by end time. Initialize last_end = -inf, kept = 0. For each interval [s, e]: if s >= last_end, this interval does not overlap the last kept one — select it: kept++, last_end = e. Otherwise skip it (one removal). Return n - kept.

Invariant & cost

last_end is always the end time of the most recently selected interval. Every interval we see with start >= last_end is safely non-overlapping with everything selected so far. O(n log n) time (sort); O(1) space beyond the sort.

The code

python

def eraseOverlapIntervals(intervals):
    intervals.sort(key=lambda x: x[1])  # sort by END time
    kept = 0
    last_end = float('-inf')
    for start, end in intervals:
        if start >= last_end:            # no overlap with last kept
            kept += 1
            last_end = end
        # else: overlaps — skip (count as one removal)
    return len(intervals) - kept
Narrate it to the interviewer

"Minimum removals equals n minus the maximum number of non-overlapping intervals I can keep. I need the maximum independent set of intervals. The greedy insight: sort by end time and always take the interval that finishes earliest, as long as it doesn't overlap the last selected one. Any swap of the chosen interval for a later-ending one can only worsen or tie the result — never improve it. So I walk the end-sorted list, track the end of the last selected interval, and skip anything whose start is before that end. Each step I make at most one removal. O(n log n) for the sort, O(1) extra space beyond that."

Dry-run aloud: all non-overlapping (0 removals), all identical intervals (n−1 removals), and intervals sharing only an endpoint — confirm that start >= last_end (not strictly >) treats touching as non-overlapping.

The reusable shape: earliest-deadline-first

Sorting by end and greedily picking non-overlapping intervals is a classical result in combinatorial optimization, sometimes called "activity selection." It generalizes: if intervals have weights, the greedy argument breaks and you need DP. But for unweighted maximum selection — the most common interview variant — the greedy is optimal and provable by an exchange argument. State the exchange argument out loud if asked why greedy works.

05 Practice

Practice ladder

Work top to bottom. For each problem: say the brute force and its complexity, name the interval technique, state the invariant, code it, and dry-run the edges. The merge and sweep patterns repeat across many domains beyond scheduling — calendars, gene intervals, CPU burst allocation.

EasyMeeting Rooms Isort + adjacent check

Given a set of meeting intervals, decide whether one person can attend all of them without any conflicts.

Idea Sort by start. If any interval starts before the previous one ends, return false. O(n log n).

EasySummary Rangesadjacent grouping

A sorted, unique integer array is given. Condense consecutive runs into range strings like "1->4" and isolated values into plain strings like "7".

Idea Walk left to right tracking the start of the current run; emit when the next value is not consecutive.

MediumMerge Intervalssort + sweep

Collapse a list of possibly overlapping intervals into the minimal non-overlapping set that covers the same points.

Idea Sort by start; extend current end while next.start <= current.end. O(n log n).

MediumInsert Interval3-phase linear insert

A sorted, non-overlapping interval list receives one new interval; insert and merge in a single pass.

Idea Copy before, absorb overlap, copy after. O(n) because the list is pre-sorted.

MediumMeeting Rooms IIsweep / min-heap

Compute the minimum number of conference rooms for a set of meetings.

Idea Two sorted arrays + two pointers; track peak concurrent count. O(n log n).

MediumNon-overlapping Intervalsgreedy by end

Find the fewest intervals to remove so no two remaining intervals overlap.

Idea Sort by end; greedy keep earliest-ending non-overlapping. Answer = n − kept. O(n log n).

MediumMinimum Number of Arrows to Burst Balloonsgreedy by end

Balloons are intervals on a horizontal axis; an arrow at position x bursts every balloon whose range covers x. Find the minimum number of arrows.

Idea Same greedy as activity selection: sort by end, advance last_end on each non-overlapping group. O(n log n).

MediumCar Poolingevents sweep

A car with capacity K picks up and drops off passengers at stops; decide if the trip is possible without exceeding capacity at any point.

Idea Create +passengers at pickup and -passengers at dropoff events, sort by position, check running sum never exceeds K. O(n log n).

MediumTask Schedulerinterval counting

CPU tasks labeled A–Z must have at least n cool-down cycles between repetitions of the same task. Find the minimum total cycles needed to complete all tasks.

Idea Compute idle slots from the most-frequent task; the answer is tasks + idle, or total tasks if no idle is needed. O(26) = O(1).

HardData Stream as Disjoint Intervalssorted structure + merge

Design a class that receives integers one at a time from a stream and reports the current set of disjoint intervals formed by all values seen so far.

Idea Maintain a sorted list of disjoint intervals; on each add, binary-search for neighbors and merge as needed. O(n) insert per call or O(log n) with a sorted container.

HardEmployee Free Timemerge across lists

Multiple employees each have a sorted list of working intervals. Find the list of finite time intervals when all employees are free simultaneously.

Idea Merge all working intervals from all employees into one sorted list, then identify gaps between merged blocks. O(n log n) with a heap or flat sort.

How to drill these problems

For each problem:
(1) state the brute force and its big-O before touching a keyboard;
(2) identify which of the four interval jobs it is — merge, insert, count, or schedule;
(3) name the sort key (start or end) and why;
(4) state the invariant for your sweep;
(5) code it and verify correctness on at least three edge cases — empty input, all identical intervals, and two intervals sharing only an endpoint. In a live interview, writing the sort line first signals that you know the pattern.

★ Review

Summary & self-test

The whole intervals pattern compresses to three steps: sort by start (or by end for scheduling), then sweep left to right, then apply the rule for your specific job — extend-or-emit for merge, three-phase for insert, running peak for count, greedy for schedule. The sort costs O(n log n) and pays for everything; the sweep is always O(n).

02Merge ruleOverlap iff next.start ≤ current.end; extend end to max(current.end, next.end).
03Insert: 3 phasesBefore (copy), overlap (absorb into new), after (copy). O(n) because input is pre-sorted.
04Count peakTwo sorted arrays of starts and ends; advance the start pointer when starts[s] < ends[e]. The peak count is the answer.
05Schedule greedilySort by end, pick each interval whose start ≥ last_end. Min removals = n − max kept.

Self-test before you move on

Cover the notes and answer aloud:
(1) Why does sorting by start make overlapping intervals adjacent? What would go wrong if you skipped the sort?
(2) In the merge sweep, why do you use max(current.end, next.end) rather than just next.end? What input breaks the naive assignment?
(3) In the Insert Interval three-phase approach, what is the precise condition that moves from phase 1 to phase 2, and from phase 2 to phase 3?
(4) For the Meeting Rooms II two-pointer sweep, when you see starts[s] < ends[e], what exactly does that tell you about the current moment in time?
(5) Why does sorting by end time make the greedy activity-selection correct — what breaks if you sort by start instead?
(6) Two intervals [1,5] and [5,8] — do they overlap or not? How does your answer change the code?
