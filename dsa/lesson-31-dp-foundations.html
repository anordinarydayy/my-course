DP Foundations (1D)

Dynamic programming is the technique of breaking a problem into overlapping subproblems, solving each subproblem exactly once, and assembling the final answer from those saved results. On one-dimensional sequences — arrays, strings, indices — it converts exponential recursion into clean O(n) or O(n²) solutions that interviewers see as a major skill signal.

Original study notes. Written from scratch for coding-interview prep. Problem names below are referenced only as well-known identifiers (Climbing Stairs LeetCode 70, House Robber 198, Maximum Subarray 53, Coin Change 322); every prompt, diagram, walkthrough, and line of code here is original. Contains three interactive diagrams and three fully worked problems — each with the reasoning chain, correct Python code, and a script for narrating it out loud to the interviewer.
00 The idea

Two properties that make DP possible

Not every problem can be solved with DP. Before reaching for a table, you must confirm that the problem has two structural properties. If either is missing, DP will give a wrong or suboptimal answer.

  • Overlapping subproblems. The naive recursive solution calls itself on the same arguments more than once. If every recursive call is distinct — like in merge sort — there is nothing to cache and DP does not help. The tell: draw the recursion tree and look for repeated nodes.
  • Optimal substructure. An optimal solution to the whole problem can be built from optimal solutions to smaller versions of the same problem. In plain English: the best answer for size n depends only on the best answers for smaller sizes, not on the specific path you took to get there.

When both hold, you get to trade exponential time (naive recursion re-explores the same subproblems again and again) for linear or polynomial time, by saving results in a memo table or a DP array.

The DP workflow — four steps

1. Define the state. What does dp[i] mean in English? 2. Write the recurrence. Express dp[i] in terms of earlier dp values. 3. Set the base cases. The smallest subproblems whose answers you can state without recursion. 4. Decide the order. Compute smaller problems before larger ones (bottom-up tabulation), or let recursion do it and cache results (top-down memoisation). Every 1-D DP problem fits this mold.

01 State & recurrence

Climbing Stairs: define the shape ▸

The first skill is naming the state and deriving the recurrence before writing any code. Consider counting the number of distinct ways to climb a staircase of n steps, when you can advance either 1 or 2 steps at a time.

The brute-force recursion tries every possible sequence of moves: from step k, branch to step k+1 and to step k+2. That gives a recursion tree with depth n and branching factor 2, for a naive O(2n) time. For n = 40 that is over a trillion calls — completely unusable.

The waste is clear: the same subproblem "how many ways from step 3?" gets recomputed every time any higher step branches through it. Once you have the answer for step 3, there is no need to redo it.

Define the state: dp[i] = the number of distinct ways to reach step i. The recurrence is immediate: to arrive at step i you must have come from step i−1 (one step up) or from step i−2 (two steps up). So dp[i] = dp[i−1] + dp[i−2]. The base cases are dp[1] = 1 (only one way to reach the first step) and dp[2] = 2 (either two singles or one double).

This is exactly the Fibonacci shape — and it teaches the central DP habit: name what the cell means, then derive how it depends on earlier cells.

The Fibonacci shape in interviews

Any time a problem says "from position i you can jump to i+1 or i+2" — or more generally "from state i you can transition to a small fixed set of next states" — the recurrence is a sum of a constant number of previous dp values. That structure always yields O(n) time and at most O(n) space (often reducible to O(1) with rolling variables).

Worked problemEasyLeetCode 70
Climbing Stairs

You are at the bottom of a staircase with n steps. Each move you can go up exactly 1 or 2 steps. How many distinct sequences of moves lead you to the top?

Think it through

Restate

Count the distinct ways to reach step n, where each move is +1 or +2. Answer is an integer; no path needs to be listed.

Brute force

Recurse: f(n) = f(n-1) + f(n-2), base cases f(1)=1, f(2)=2. This is correct but the recursion tree has exponential size — O(2n) calls. For n = 45 that is tens of billions.

The waste

The subtree rooted at f(k) is re-expanded from scratch every time any node calls it. If we store the result the first time it is computed, every future call is O(1).

The move

Build a dp array left to right. dp[i] = ways to reach step i. dp[1]=1, dp[2]=2. For i from 3 to n: dp[i] = dp[i-1] + dp[i-2]. The answer is dp[n]. Only two previous values are ever needed, so we can keep just two variables instead of a full array.

Invariant & cost

After processing index i, dp[i] holds the exact count for step i. Every cell is filled exactly once: O(n) time, O(1) space with rolling variables.

The code

python

def climbStairs(n: int) -> int:
    if n <= 2:
        return n                     # base cases: 1 way for n=1, 2 ways for n=2
    prev2, prev1 = 1, 2            # dp[1], dp[2]
    for _ in range(3, n + 1):
        curr = prev1 + prev2         # dp[i] = dp[i-1] + dp[i-2]
        prev2, prev1 = prev1, curr   # slide the window forward
    return prev1
Narrate it to the interviewer

"The brute force is plain recursion — f(n) = f(n-1) + f(n-2) — but it is O(2^n) because the same sub-answer is recomputed repeatedly. So I'll use DP: define dp[i] as the number of ways to reach step i. The recurrence is dp[i] = dp[i-1] + dp[i-2] because the only ways to land on step i are from step i-1 or step i-2. Base cases are dp[1]=1 and dp[2]=2. I only ever look back two cells, so I keep two rolling variables and avoid the O(n) array entirely. Final complexity: O(n) time, O(1) space."

Dry-run aloud: n=1 (return 1), n=2 (return 2), n=3 (prev2=1, prev1=2, one iteration: curr=3, return 3). Check: three ways — {1,1,1}, {1,2}, {2,1}.

What the interviewer asks next

"What if you could take 1, 2, or 3 steps?" — extend the recurrence to dp[i] = dp[i-1] + dp[i-2] + dp[i-3] and add the base case dp[3]=4. You now need three rolling variables instead of two. "What if step sizes are arbitrary?" — that generalises to Coin Change (LeetCode 322) with a min-cost formulation; same DP skeleton, different operator.

02 Top-down

Memoisation: cache as you recurse

Tabulation is not the only way to avoid redundant work. You can keep the recursive structure and simply add a dictionary that stores results the first time they are computed. On any subsequent call with the same argument, return the cached value immediately — O(1) instead of re-expanding the subtree.

Top-down memoisation and bottom-up tabulation are two implementations of the same idea. The right choice depends on three factors:

Top-down (memoised recursion)

  • Code mirrors the recurrence directly — often easier to write correctly in a first pass.
  • Only computes subproblems actually reachable from the root; large sparse state spaces pay less.
  • Recursion stack overhead; may hit Python's recursion limit for very large n.
  • Use @functools.lru_cache or a dict; Python's decorator handles the boilerplate.

Bottom-up (tabulation)

  • Iterative — no stack overhead; scales to very large inputs without hitting recursion limits.
  • Forces you to think about the order of computation explicitly, which clarifies the recurrence.
  • Computes every cell in the table, even subproblems that are never needed by the answer.
  • Easier to space-optimise: once a row or column is consumed, discard it.
Python shortcut for memoisation

Decorate a recursive function with @functools.lru_cache(maxsize=None) (Python 3.9+ alias: @cache). The decorator wraps every call in a dict lookup automatically. In an interview, name it explicitly — "I'm adding an lru_cache decorator for memoisation; this turns the O(2^n) recursion into O(n) time and O(n) space" — to show you understand the mechanism, not just the trick.

Worked problemMediumLeetCode 322
Coin Change

Given a list of coin denominations and a target amount, find the minimum number of coins needed to make that amount. You have unlimited copies of each coin. Return -1 if the amount cannot be made.

Think it through

Restate

For a target total T and a set of coin values, find the smallest count of coins (repetitions allowed) whose sum equals T. The order of coins does not matter; only the count does.

Brute force

Try every combination of coins recursively: for each coin c, recurse on T − c and take the minimum. Uncached, many subtotals are recomputed. Worst case is O(Sn) where S is the number of coin types — exponential, completely unacceptable.

The waste

The sub-answer "minimum coins for amount k" is the same regardless of which path of larger coins led us here. Cache it once — overlapping subproblems, optimal substructure — both DP properties hold.

The move

Define dp[a] = minimum coins for amount a. Base case: dp[0] = 0. For each amount from 1 to T, try every coin: if a − coin >= 0, then dp[a] = min(dp[a], dp[a − coin] + 1). Initialise all entries to infinity so only reachable amounts converge to a finite answer. Answer is dp[T], or -1 if still infinity.

Invariant & cost

After processing amount a, dp[a] holds the provably-minimum coin count for that amount. The outer loop runs T times; the inner loop runs at most S times: O(T × S) time, O(T) space.

The code

python

def coinChange(coins: list[int], amount: int) -> int:
    INF = float('inf')
    dp = [INF] * (amount + 1)
    dp[0] = 0                           # zero coins needed for amount 0
    for a in range(1, amount + 1):
        for c in coins:
            if c <= a:
                dp[a] = min(dp[a], dp[a - c] + 1)
    return dp[amount] if dp[amount] != INF else -1
Narrate it to the interviewer

"The brute force is exponential — trying every combination. Both DP conditions hold: a lot of subtotals get recomputed, and the optimal answer for amount T only depends on optimal answers for smaller amounts. I define dp[a] as the minimum coins for amount a, initialise everything to infinity except dp[0]=0, then for each amount I try subtracting each coin and take the best. The invariant is that when I process amount a, all smaller amounts are already solved. Total: O(T times S) time, O(T) space. If dp[T] is still infinity at the end, the amount is unreachable, so I return -1."

Dry-run aloud: coins=[2], amount=3 — dp[1]=inf, dp[2]=1, dp[3]=inf, return -1. Coins=[1,2,5], amount=11 — dp[5]=1, dp[10]=2, dp[11]=3 (10+1). Also check amount=0 directly.

03 Bottom-up

House Robber: tabulate the take-vs-skip choice ▸

House Robber (LeetCode 198) is the canonical "take or skip" DP. You have an array of houses, each with a certain cash amount inside. You can rob any subset of houses, but you cannot rob two houses that are directly next to each other. Maximise the total loot.

The brute force is to try every subset: 2n possibilities, which is hopeless for large n. The key observation is that the optimal choice for house i depends on only two earlier results — the best total if you stop at i−1 (skip house i) or the best total at i−2 plus what house i holds (rob house i).

Define dp[i] = maximum loot considering houses 0 through i. The recurrence is:

dp[i] = max( dp[i-1], dp[i-2] + a[i] )

Base cases: dp[0] = a[0] (only one house, take it); dp[1] = max(a[0], a[1]) (best of the first two). The answer is dp[n-1].

Worked problemMediumLeetCode 198
House Robber

An array of non-negative integers represents money in a row of houses. You may rob any subset of houses but cannot rob two adjacent houses. Return the maximum total you can collect.

Think it through

Restate

Choose a subset of indices — no two consecutive — that maximises the sum of the selected elements.

Brute force

Try every valid subset: there are 2n subsets and checking each for the adjacency constraint takes O(n), giving O(n × 2n). Hopeless beyond n = 30.

The waste

Whether or not to rob house i depends only on the best outcomes for i−1 and i−2, not on the specific houses chosen in those prefixes. The sub-answer for each prefix length is computed over and over in the naive recursion.

The move

Build the dp table left to right. At each house i, compare: (a) skip it — inherit dp[i-1], or (b) rob it — take a[i] plus the best up through i-2. Pick the larger. Only the two previous values are needed, so replace the array with two variables.

Invariant & cost

When computing dp[i], all smaller indices are already finalised. Every house is visited once: O(n) time, O(1) space with rolling variables.

The code

python

def rob(nums: list[int]) -> int:
    if not nums:
        return 0
    if len(nums) == 1:
        return nums[0]
    prev2 = nums[0]                  # dp[0]
    prev1 = max(nums[0], nums[1])  # dp[1]
    for i in range(2, len(nums)):
        curr = max(prev1, prev2 + nums[i])  # skip or rob
        prev2, prev1 = prev1, curr
    return prev1
Narrate it to the interviewer

"The brute force is exponential — trying all subsets. Both DP properties hold: the same prefix best-values appear in many branches, and the optimal for house i only needs the optimal for i-1 and i-2. I define dp[i] as the most I can earn from the first i+1 houses. At each step I decide: skip house i and keep dp[i-1], or rob it and get dp[i-2] plus the current house's value — take the max. I only need the last two cells so I use two rolling variables. O(n) time, O(1) space."

Dry-run: empty array (return 0), one house (return it), two houses (return the larger), and an all-equal array — confirm the result is every-other-one times the value.

What the interviewer asks next

"What if the houses form a circle — the first and last are adjacent?" (LeetCode 213) — run the same algorithm twice: once on nums[0…n-2] and once on nums[1…n-1], then return the larger result. The adjacency between head and tail means they can never both be in an optimal solution, so splitting covers both cases. "What about a binary tree of houses?" (LeetCode 337) — extend to tree DP: each node returns a pair (rob_me, skip_me) and you combine children recursively.

04 Space optimise

Rolling variables: drop the array ▸

Once a tabulation is working, ask: which earlier cells does the recurrence actually read? If dp[i] only looks back one or two positions, you do not need to store the whole array — keep just those positions in plain variables. This is the rolling-variable (or space-optimisation) step, and it turns O(n) space into O(1).

The Climbing Stairs and House Robber recurrences both read dp[i-1] and dp[i-2], so they need exactly two variables. Coin Change reads dp[a - c] for arbitrary coin values, so the full O(T) array is required — you cannot reduce it further.

This stepper shows the two rolling variables tracking the house-robber solution in real time:

When space optimisation is possible

Look at the recurrence: identify the furthest lookback. If it is a fixed constant k (1, 2, 3, …), you need only k rolling variables. If the recurrence reads dp[i - coin] for a variable coin, you need the whole array. For 2-D DP (like edit distance or longest common subsequence) you can usually drop all but the current and previous rows, turning O(n²) space into O(n).

05 Running best

Kadane's algorithm: maximum subarray

The Maximum Subarray problem (LeetCode 53) asks for the contiguous subarray with the largest sum. It looks different from Climbing Stairs and House Robber, but it is still 1-D DP with a clean recurrence.

Brute force: try every pair of indices (i, j), compute the sum for each window. O(n²) time, O(1) space. For n = 105, that is 1010 additions — far too slow.

The key insight is: at each index i, the maximum subarray ending exactly at i is either (a) just the single element a[i], or (b) the element a[i] appended to the best subarray ending at i-1. If the best subarray to the left has a negative sum, it only drags the total down, so we discard it and start fresh. This gives:

curr[i] = max( a[i], curr[i-1] + a[i] )

The global answer is the running maximum of all curr[i] values. Both curr and the running best collapse to two variables — O(1) space. This is Kadane's algorithm.

Worked problemMediumLeetCode 53
Maximum Subarray

Given an integer array (which may contain negatives), return the sum of the contiguous subarray with the largest possible total. The array contains at least one element.

Think it through

Restate

Find the maximum sum over all contiguous subarrays a[i…j] with 0 <= i <= j < n. The answer is a single integer.

Brute force

For each pair (i, j), compute the window sum. With a prefix-sum array, each query is O(1), but there are O(n²) pairs: O(n²) time, O(n) space. For n = 105 that is 1010 operations.

The waste

When the best subarray ending at i-1 is negative, including it only hurts the subarray ending at i. There is no point carrying a negative running total — just restart. This observation collapses the 2-D brute force to a 1-D pass.

The move

Track curr = maximum subarray sum ending at the current index. At each step: curr = max(a[i], curr + a[i]). If curr + a[i] is less than starting fresh at a[i], restart. Update a global best = max(best, curr) at each index. Return best.

Invariant & cost

At index i, curr is the largest sum of any subarray ending exactly at i; best is the largest seen so far across all indices 0…i. Single pass: O(n) time, O(1) space.

The code

python

def maxSubArray(nums: list[int]) -> int:
    curr = nums[0]       # best sum ending at the current index
    best = nums[0]       # global maximum seen so far
    for i in range(1, len(nums)):
        curr = max(nums[i], curr + nums[i])  # extend or restart
        best = max(best, curr)
    return best
Narrate it to the interviewer

"The brute force is O(n²) — check every pair of endpoints. The DP insight is that at each index i, the best subarray ending at i is either just a[i] by itself, or a[i] tacked onto the best subarray ending at i-1. If the running total goes negative, it only drags future values down, so I reset. I track two values: curr for the best ending here, and best for the global max. Single pass, O(n) time, O(1) space — this is Kadane's algorithm."

Dry-run: all-negative array like [-3, -1, -2] — curr and best both start at -3, curr becomes max(-1,-4)=-1, best=-1, curr becomes max(-2,-3)=-2, best=-1. Correct: the least-bad single element. Also check single-element array and all-positive array.

What the interviewer asks next

"Return the actual subarray, not just the sum." — track start, end, and temp_start indices; when curr resets, update temp_start to i; when best updates, record start=temp_start and end=i. "What if the array must be circular — you can wrap around?" (LeetCode 918) — the circular maximum is either the normal maximum subarray, or the total sum minus the minimum subarray (the non-selected middle). Run Kadane's twice, once for max and once for min, then compare. "What about a 2-D grid?" — Kadane extends to 2-D by fixing the top and bottom row and running 1-D Kadane on the column sums; O(n² × m) for an n-by-m grid.

06 Practice

Practice ladder

DP fluency comes from drilling the four steps on many problems: define the state, write the recurrence, handle the base cases, optimise space. Work through each rung by narrating the brute force first, then deriving the DP table, then coding it. Aim for the easy tier in under 20 minutes each and the medium tier under 35.

EasyClimbing Stairsfibonacci shape

Count distinct ways to reach the top of an n-step staircase taking 1 or 2 steps at a time.

Idea dp[i] = dp[i-1] + dp[i-2]; two rolling variables; O(n) time O(1) space.

EasyFibonacci Numberbase case

Return the n-th Fibonacci number. Identify why naive recursion is exponential and what caching buys you.

Idea Same rolling-two-variables trick as Climbing Stairs; an excellent warm-up for memoisation vs tabulation.

EasyMin Cost Climbing Stairscost recurrence

Each step has a cost; you pay to step off it. Find the minimum total cost to exit the staircase (you can start from step 0 or 1).

Idea dp[i] = cost[i] + min(dp[i-1], dp[i-2]); answer is min(dp[n-1], dp[n-2]).

EasyMaximum Subarraykadane

Find the contiguous subarray with the largest sum in an array that may contain negatives.

Idea Kadane: curr = max(a[i], curr + a[i]); best = max(best, curr); O(n) time O(1) space.

MediumHouse Robbertake-or-skip

An array of house values; you cannot rob two adjacent houses. Maximise total loot.

Idea dp[i] = max(dp[i-1], dp[i-2] + a[i]); two rolling variables; O(n) time O(1) space.

MediumCoin Changeunbounded knapsack

Given coin denominations and a target amount, find the fewest coins that sum to the target; return -1 if impossible.

Idea dp[a] = min over coins c of dp[a-c]+1; O(T * S) time O(T) space.

MediumJump Gamereachability dp

Each element is your max jump distance from that index. Return whether you can reach the last index.

Idea DP or greedy: track the furthest index reachable so far; if it ever falls behind the current index, return false.

MediumJump Game IImin-jumps dp

Same setup as above; return the minimum number of jumps needed to reach the last index.

Idea dp[i] = minimum jumps to reach i; or BFS-style greedy tracking the current jump's reach window.

MediumDecode Waysstring dp

A string of digits encodes letters (1→A, 26→Z). Count the total number of valid decodings.

Idea dp[i] depends on dp[i-1] (single digit) and dp[i-2] (two digits if 10-26); base cases are the empty prefix and the first character.

MediumHouse Robber IIcircular array

Same as House Robber, but the houses form a circle — the first and last are adjacent.

Idea Run House Robber on nums[0..n-2] and again on nums[1..n-1]; return the larger of the two results.

HardMaximum Product Subarraymin & max tracking

Find the contiguous subarray with the largest product. A negative times a negative can flip the sign.

Idea Track both the running max and running min ending here; a negative element swaps them. O(n) time O(1) space.

HardJump Game VIIsliding window dp

A binary string; from each '0' you can jump between minJump and maxJump steps. Can you reach the last index?

Idea Boolean dp[i] plus a sliding window count of reachable positions in the window; O(n) time O(n) space.

How to drill DP problems

For every problem, follow these steps aloud: (1) State the brute-force time complexity. (2) Identify the overlapping subproblems — draw the recursion tree if it helps. (3) Define dp[i] in a single English sentence. (4) Write the recurrence and the base cases before touching code. (5) Code the tabulation; then space-optimise by asking "how far back does the recurrence look?" (6) Dry-run at least three cases: an empty or single-element input, a case where all values are equal, and one involving the all-negative path (Kadane). Any fuzzy answer → re-read the section and re-code from scratch.

★ Review

Summary & self-test

Dynamic programming converts exponential recursion into polynomial time by ensuring each overlapping subproblem is solved exactly once. The full workflow is: check the two properties (overlapping subproblems, optimal substructure) → define the state and recurrence → handle base cases → choose top-down memo or bottom-up tabulation → space-optimise where the lookback is bounded. On 1-D problems, this almost always means an O(n) loop and O(1) to O(n) space.
