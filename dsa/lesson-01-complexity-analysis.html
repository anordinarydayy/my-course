Complexity Analysis (Big-O)

Before you can choose the right algorithm, you need a language for comparing them. Big-O gives you that language — a way to describe how an algorithm's time or memory demand grows as the input grows, independent of hardware or language. It is the single most-used tool in every coding interview, and every pattern lesson that follows builds on it.

00 The idea

What Big-O actually measures

Suppose you write a function that loops over an array of length n once. On your laptop it runs in 3 ms. On a server with a different CPU it might run in 1 ms. Those numbers say nothing about the algorithm — they just reflect hardware. What does scale with the algorithm is the number of elementary operations it performs as n grows.

Big-O notation strips away constants and lower-order terms and answers the question: as n grows without bound, how does the operation count grow? That is its entire job. A loop that runs 3n + 12 comparisons is O(n) — the 3 and the 12 vanish because at large n they become negligible relative to the dominant term. Two nested loops over the same array do about n² / 2 comparisons — written O(n²) because the constant one-half disappears.

Three common confusions to clear up now:

  • Big-O is an upper bound, not an exact count. Saying O(n²) means "grows no faster than n²" — the actual coefficient might be 0.5 or 5. It tells you the shape of growth, not the wall-clock time.
  • It describes worst-case by default in interview contexts (unless you say "average case" explicitly). For a linear search the worst case is that the target is the last element: O(n) comparisons.
  • n is the size of the input. For an array, n is its length. For a graph, it might be V (vertices) or E (edges) or both. Always name what n means before giving a complexity.
The one-line rule

Write your algorithm, count the loops and how they nest, keep the fastest-growing term, drop all constants. That is Big-O. Everything else in this lesson is applying that rule to specific structures and patterns.

01 The ladder

The growth ladder ▸

The classes you will encounter in every interview — from fastest to slowest — form a strict ordering. Knowing this ordering by heart lets you instantly judge whether an algorithm is fast enough for a given input size.

The ladder's key insight is how sharply the classes separate. At n = 10 everything looks similar. By n = 1 000, O(n²) is a million operations while O(n log n) is just ten thousand. By n = 40, an O(2ⁿ) algorithm does over a trillion operations — no hardware in the world saves it.

Fast classes

  • O(1) — constant: hash lookup, array index, stack push.
  • O(log n) — halves the search space each step: binary search, heap operations, balanced BST lookup.
  • O(n) — one pass: linear scan, two-pointer sweep, BFS/DFS on a list.
  • O(n log n) — merge sort, heap sort, most sorting-based solutions.

Slow classes (avoid at scale)

  • O(n²) — nested loops over the same data: naive pair search, insertion sort, bubble sort.
  • O(2ⁿ) — exponential: brute-force recursion on subsets, naive Fibonacci.
  • O(n!) — factorial: brute-force permutations, travelling salesman brute force.
The simplification rules — memorise these

Drop multiplicative constants: 7n → O(n). Keep only the dominant term: n² + n → O(n²). Products of independent loops multiply: an outer loop of n with an inner loop of m → O(nm). A loop whose size halves each iteration contributes O(log n), not O(n).

02 Space

Space complexity

Every interview has two complexity questions: time and space. Space complexity counts the extra memory your algorithm allocates beyond the input — called auxiliary space. The input itself does not count unless you copy it.

The analysis technique is the same as for time. Count the data structures you create. A single variable is O(1). An array of size n is O(n). A recursion stack of depth n is O(n) — each call frame sits on the stack until it returns. A 2-D table of n×n values is O(n²). Most interview problems expect an O(1) or O(n) auxiliary answer; anything quadratic in space is almost never acceptable.

The space trap: recursion stacks

Candidates often claim O(1) space for a recursive solution and forget the call stack. A recursive depth-first search on a graph of n nodes uses O(n) space on the stack even if it allocates no extra variables. The iterative equivalent using an explicit stack uses O(n) space too, but at least you control it. Always account for the maximum recursion depth.

03 Amortised

Amortised analysis: the dynamic-array append ▸

Some operations are occasionally expensive but cheap on average. If you spread the cost of the expensive operations over all the cheap ones, you get the amortised cost per operation — often what matters in practice.

The clearest example is Python's list.append (or Java's ArrayList). Internally it holds a buffer of some capacity. Most appends just write into the next free slot: O(1). But when the buffer is full, it allocates a new buffer of double the size and copies every existing element across — an O(n) event. Is append O(n)? No, because doubling means the next copy won't happen until another n appends pile up. That O(n) copy is paid for by n cheap appends that preceded it — each one "pre-pays" O(1) toward the next resize.

The formal argument: start with capacity 1. After k doublings the capacity is 2k and the total copy work done is 1 + 2 + 4 + … + 2k-1 = 2k − 1 < 2k. So n appends cause at most n total copy operations, giving an amortised cost of O(1) per append. The lesson: a single O(n) event spread over n operations costs O(1) amortised.

When amortised reasoning matters in interviews

Interviewers ask: "What is the time complexity of your append?" A good answer is: "O(1) amortised — individual appends are O(1) except when the buffer resizes, which is O(n) but happens so rarely that the average cost per append stays constant." The same argument applies to hash table insertions when the load factor triggers a rehash.

04 Cost table

Data structure operation costs

Every pattern lesson assumes you know these on sight. Study the table until you can recite it without looking — an interviewer who asks "what is the time complexity of inserting into a heap?" expects an immediate answer, not a derivation.

Time vs space trade-off

  • A hash map turns an O(n) search into O(1) at the cost of O(n) extra space — the classic time-space swap.
  • Sorting first (O(n log n)) unlocks binary search (O(log n)) for repeated queries.
  • A heap gives O(log n) insert/delete vs O(n) for a sorted array insert — useful when the dataset changes frequently.

What to say out loud

  • "Hash map lookup is O(1) average — worst case O(n) if all keys collide, but that's adversarial and rare in practice."
  • "Heapify is O(n), not O(n log n) — it's the classic insight for building a heap from an existing array."
  • "Sorting is a one-time O(n log n) investment; if I'll query many times it pays off."
05 Read n

Read n off the constraints, pick your target ▸

In a real interview, the problem gives you the input size constraint (n ≤ 105, for instance). That number is your budget. You work backwards: given the time limit (typically 1–2 seconds, roughly 108 simple operations), what is the most expensive Big-O class you can afford?

These thresholds assume Python (about 107–108 basic ops/sec) and typical contest/interview time limits. If you are coding in C++ the constants are ~10× better, which sometimes lets an O(n²) solution pass at n = 104. Always check whether Python is the expected language before committing to a complexity class.

The interviewer test

When an interviewer says "n can be up to 105" and you are considering an O(n²) solution, say out loud: "n = 105 makes n² = 1010 operations — that's too slow by about two orders of magnitude. I need an O(n log n) or O(n) approach." This single sentence shows you understand Big-O at a practical level and are actively ruling out bad solutions before writing code.

06 Before & after

Before & after complexity analyses

The best way to cement Big-O is to apply it to concrete algorithms — first the naive version, then the optimised version, and name exactly what changed. Here are three that recur across almost every interview pattern.

Analysis 1 — Pair-sum search

Complexity walkthroughEasypattern: Two Pointers
Unsorted pair-sum

Given an integer array of length n, do any two distinct elements sum to a given target? Return true or false.

Before (brute force)

Algorithm

For every index i, scan every index j > i and check if a[i] + a[j] == target.

Time

The outer loop runs n times; the inner loop runs up to n−1 times. Total: about n(n-1)/2 iterations → O(n²).

Space

No extra allocation. O(1) auxiliary space.

After (hash set)

Algorithm

Walk the array once. For each a[i], check if target - a[i] is already in a hash set. If yes, return true; otherwise insert a[i] into the set.

Time

One pass of n iterations; each hash-set operation is O(1) average. Total: O(n).

Space

The hash set holds at most n values. O(n) auxiliary space.

Trade-off

We traded O(n) space to drop time from O(n²) to O(n). If the array is sorted, two converging pointers achieve O(n) time and O(1) space — the best of both worlds when the sorted precondition holds.

python — hash set version

def pair_sum_exists(nums, target):
    seen = set()                      # O(n) space
    for x in nums:                     # O(n) iterations
        if target - x in seen:        # O(1) average lookup
            return True
        seen.add(x)
    return False

Analysis 2 — Finding the k-th largest

Complexity walkthroughMediumpattern: Heap / Quickselect
K-th largest element (Kth Largest Element in an Array, LeetCode 215)

Find the k-th largest value in an unsorted array of n integers.

Before (sort)

Algorithm

Sort the array descending; return index k−1.

Time

Sorting: O(n log n). Access: O(1). Total: O(n log n).

Space

Merge sort: O(n); in-place quicksort: O(log n) stack.

After (min-heap of size k)

Algorithm

Maintain a min-heap of exactly k elements. For each new value: if the heap has fewer than k elements, push it. Otherwise if the new value exceeds the heap's min, pop the min and push the new value. The heap's root is always the k-th largest seen so far.

Time

n iterations; each heap operation is O(log k) because the heap stays at size k. Total: O(n log k). When k is small (k << n) this is much better than O(n log n).

Space

The heap holds at most k elements. O(k) auxiliary space.

Trade-off

O(n log k) vs O(n log n). The heap approach shines when k is small; for k ≈ n/2 both are similar. Quickselect achieves O(n) average time but O(n) worst case with a poor pivot.

python — min-heap of size k

import heapq

def kth_largest(nums, k):
    heap = []                           # min-heap, size ≤ k
    for x in nums:                     # O(n) iterations
        heapq.heappush(heap, x)        # O(log heap_size)
        if len(heap) > k:
            heapq.heappop(heap)        # keep only k largest seen
    return heap[0]                    # root = k-th largest

Analysis 3 — Detecting a duplicate

Complexity walkthroughEasypattern: Hashing / Sorting
Contains duplicate (Contains Duplicate, LeetCode 217)

Given an integer array, return true if any value appears more than once.

Before (sort, then scan)

Algorithm

Sort the array. Walk it and check if any adjacent pair is equal.

Time

Sort: O(n log n). Linear scan: O(n). Total: O(n log n).

Space

In-place sort: O(log n) stack.

After (hash set)

Algorithm

Insert each value into a hash set. If an insertion finds the value already present, return true immediately.

Time

One pass, O(1) hash operations. Total: O(n) average.

Space

Hash set: O(n) in the worst case (no duplicates).

Trade-off

O(n) time at the cost of O(n) space vs O(n log n) time and O(log n) space. If memory is tight and the array can be sorted in place, sorting is better on space.

python — hash set

def has_duplicate(nums):
    seen = set()
    for x in nums:
        if x in seen:
            return True
        seen.add(x)
    return False
07 Next steps

Where this shows up next

Big-O is not a pattern by itself — it is the measuring stick every other pattern uses. Every lesson that follows will state a brute-force complexity, name the insight that improves it, and give the final time and space. Once you have this lesson down, those statements become instantly meaningful.

Two PointersPattern 01

Converts O(n²) nested-loop pair searches into a single O(n) sweep. The key question is always whether you can name a monotonic reason to choose which pointer to move.

Complexity O(n) time, O(1) space — the first big win after learning this lesson.

Sliding WindowPattern 02

Avoids recomputing a contiguous subarray sum from scratch each step — brings O(n²) down to O(n) by maintaining a running total across a moving window.

Complexity O(n) time, O(1) or O(k) space depending on the constraint tracked.

Binary SearchPattern 03

Each iteration halves the search space — a textbook O(log n) algorithm. Understanding the growth ladder makes it obvious why log n is so much faster than n at large inputs.

Complexity O(log n) time, O(1) space for iterative; O(log n) space for recursive.

Top-K & K-th LargestPattern 10

The heap cost table entry (O(log n) insert/delete, O(1) peek) is the entire explanation for why keeping a heap of size k solves top-k in O(n log k) instead of O(n log n).

Complexity O(n log k) time, O(k) space.

Dynamic ProgrammingPatterns 18–25

DP transforms exponential brute-force recursion (O(2ⁿ)) into polynomial time (typically O(n²) or O(n)) by memoising overlapping subproblems. The Big-O comparison is what makes the optimisation self-evident.

Complexity varies; classic knapsack is O(nW) time, O(nW) or O(W) space.

Graph Traversal (BFS/DFS)Pattern 23

BFS and DFS each visit every vertex and edge once: O(V + E) time, O(V) space for the frontier or recursion stack. Knowing this lets you immediately evaluate any graph algorithm against the constraint.

Complexity O(V + E) time, O(V) space.

★ Review

Summary & self-test

Big-O is the universal language of algorithm efficiency. It strips away hardware, constants, and lower-order terms to reveal the shape of growth. The growth ladder O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!) defines what is feasible for each input size, and every data-structure cost table entry is a consequence of how that structure is built. Amortised analysis explains why O(1) append is honest even when some appends are expensive.

03Space tooCount auxiliary memory: extra arrays are O(n), deep recursion is O(depth), constants are O(1).

06Heap = O(log n)Insert and delete-min are O(log n); peek is O(1); heapify from an array is O(n).

Self-test — cover the notes and answer aloud

(1) What is the Big-O of an algorithm with operation count 5n² + 3n log n + 42? Name the dominant term and explain why the others disappear.

(2) A function calls itself recursively on each half of an array (like binary search). What is its time complexity, and what is its space complexity?

(3) The problem says n ≤ 10⁵. Your current approach is O(n²). How many operations is that, roughly? Is it acceptable in Python under a 1-second time limit? What class do you need instead?

(4) A sorted array insert is O(n) even though binary search finds the position in O(log n). Why?

(5) You append 1 000 items to an empty Python list (which starts with capacity 1 and doubles). How many total copy operations happen across all resizes? What is the amortised cost per append?

(6) Name one algorithm whose brute-force complexity is O(2ⁿ) and explain what technique brings it into polynomial time. State the resulting complexity.
