03Concept · Coding Interview

Hash Maps & Sets

The single most-used lookup structure in coding interviews: a table that turns any key into an O(1) average-time slot, reducing a loop-within-a-loop to one clean scan.

00 The idea

What are hash maps and sets?

Arrays give you O(1) lookup by position: a[3] is instant because the index is just an offset. The trouble with real data is that keys are rarely tidy consecutive integers — they are strings, coordinates, words, whatever the problem throws at you. A hash map (Python dict, Java HashMap, C++ unordered_map) generalises array indexing: it turns any key into a small integer (the hash), uses that integer as an array index, and stores your value there. The result is a data structure that gives you O(1) average-time lookup, insertion, and deletion regardless of what the keys look like.

A hash set is the same idea without storing a value — just membership. You ask "have I seen this key before?" and get an O(1) yes/no.

This is why hash maps come up in nearly every interview: the brute force for almost any "have I seen X?" or "how many times did X appear?" question is an O(n) scan for each query, giving O(n²) overall. Swapping that inner scan for a hash map lookup drops it to O(n).

Map vs Set — pick by what you need back

Use a hash map when each key carries a payload you need to retrieve (a count, an index, a list of words). Use a hash set when you only need to answer "was this seen?" — it uses less memory and the intent is clearer in code. Both cost O(1) average per operation.

01 Under the hood

How hashing works

A hash function converts a key into an integer, then the table divides that integer by the number of buckets (N) and uses the remainder as the slot address. Crucially, the same key always produces the same hash — that is what makes lookups deterministic and O(1).

The diagram below walks through inserting six integer keys into a small 7-bucket table. Watch the keys land in different slots: the index of each slot is simply key mod 7. When two keys map to the same slot — a collision — the table chaining stores both in a list at that bucket.

The invariant of a hash table

If the load factor (items / buckets) stays below about 0.75, the average chain length stays near 1 and every operation costs O(1). When you insert past that threshold the table resizes — it allocates roughly double the buckets and rehashes every key. Resizing is O(n), but it happens so rarely (each key triggers at most one resize per doubling) that the amortised cost per insertion stays O(1).

02 Operations

Core operations and their costs

Here is the full operation table — average case with a good hash function, and worst case (everything collides into one chain, so every operation degrades to a linear scan of that chain).

Operation Average Worst Notes
map[key] = val (insert/update) O(1) O(n) Triggers resize when load factor > threshold
map[key] (lookup) O(1) O(n) KeyError if absent; use .get(key, default)
key in map (membership) O(1) O(n) Same for set: key in s
del map[key] (delete) O(1) O(n) Marks the slot as a tombstone internally
Iteration over all keys/values O(n) O(n) Must visit every bucket, including empty ones
Minimum / maximum key O(n) O(n) No ordering — use a sorted structure if you need this

Hash map / set

  • O(1) average lookup by key — the whole point.
  • Keys can be any hashable type (string, int, tuple in Python; objects that implement hashCode() in Java).
  • No guaranteed ordering of keys (Python 3.7+ insertion-ordered but not sorted).
  • Extra memory: typically 1.3–2× the number of stored keys.

Sorted map (BST-backed)

  • O(log n) per operation — slower, but ordered.
  • Supports floor/ceiling, range queries, predecessor/successor.
  • Use when you need "nearest key" or sorted iteration.
  • Python: SortedDict (sortedcontainers); Java: TreeMap.

The working rule: reach for a hash map by default, switch to a sorted structure only when you need the keys in order or need range queries.

03 The idioms

The four patterns every interviewer expects

Once you have O(1) lookup, four shapes recur constantly across interview problems. Recognise the shape and the code almost writes itself.

Frequency counting

Walk through a sequence and count how many times each value appears. Each step is one insertion or increment — O(n) total.

python

freq = {}
for ch in s:
    freq[ch] = freq.get(ch, 0) + 1  # or: Counter(s)

Grouping (anagram buckets)

Use a computed signature as the key and collect all items sharing that signature into a list. Classic for grouping anagrams: the sorted characters of a word are a unique fingerprint for the group.

python

groups = {}
for word in words:
    key = "".join(sorted(word))   # signature: sorted letters
    if key not in groups:
        groups[key] = []
    groups[key].append(word)

Seen-set (deduplication / cycle detection)

Maintain a set of everything visited so far. Each new element is either fresh (add it) or a repeat (react immediately). O(1) per check.

python

seen = set()
for x in sequence:
    if x in seen:
        return x         # first duplicate
    seen.add(x)

Complement lookup

Store what you have already seen (or its index). For each new element, check whether its "partner" (whatever completes the goal) was seen before. This is the engine behind Two Sum and dozens of similar problems.

python

seen = {}
for i, x in enumerate(nums):
    complement = target - x
    if complement in seen:
        return [seen[complement], i]
    seen[x] = i             # store index so we can return it

When to reach for each idiom

Frequency counting — any "how many times", "most common", or "at least K" question. Grouping — any "group X by property Y" question; the signature is your map key. Seen-set — duplicates, intersection, "first repeat", cycle detection in number sequences. Complement lookup — any "two things that combine to a target" question; store the first element, check for its partner as you go.

04 See it used

Two Sum: complement lookup in action

Two Sum (LeetCode 1) is the canonical complement-lookup problem and a near-universal warm-up question. Given an array of integers and a target, find the two indices whose values add up to the target. Every number is used at most once, and exactly one solution is guaranteed.

See it usedEasyLeetCode 1
Two Sum

Given an array of integers and a target value, return the indices of the two entries that add up to the target. Each entry may be used at most once, and exactly one valid pair exists.

Think it through

Restate

For each element x at index i, I need to know if target - x appears somewhere earlier in the array. If yes, return both indices. If no, remember x so a future element can find it.

Brute force

Try every ordered pair: for each i, scan every j > i looking for a[j] == target - a[i]. Correct, but that is O(n²) — about 10⁹ operations for n = 50,000.

The waste

The inner scan searches the whole remaining array for a single value — exactly what a hash map replaces with O(1). We are repeating work we could cache.

The move

Walk the array once. Before recording x, check if its complement (target - x) is already in the map. If it is, the answer is [seen[complement], i]. If not, store seen[x] = i and continue. One pass, O(1) per step.

Invariant & cost

At each index i, the map contains exactly the values seen at indices 0 .. i-1 — so any complement found in the map must be at a strictly earlier index, which means we never pair an element with itself. O(n) time, O(n) space.

The code

python

def twoSum(nums, target):
    seen = {}                          # value -> index
    for i, x in enumerate(nums):
        complement = target - x
        if complement in seen:           # O(1) lookup
            return [seen[complement], i]
        seen[x] = i                    # store after check: avoids self-pairing
    return []                          # guaranteed a hit; guard for safety
Narrate it to the interviewer

"My restate: for each value I ask 'did I already see its complement?' The brute force is O(n²) nested loops, which doesn't use the fact that I can cache what I've seen. I'll walk the array once, and at each step I check the map for target minus current. If found, I return both indices. If not, I record the current value and its index. I store after the check so I never use the same index twice. One pass, O(n) time, O(n) space."

Dry-run aloud: two-element array (check complement immediately), all negative numbers (complement math still works), duplicate values like [3, 3] with target 6 (the second 3 finds the first 3 in the map).

Common follow-up questions

"What if there are multiple valid pairs?" — collect all of them instead of returning on the first hit; same O(n) loop. "What if the array is already sorted?" — two pointers (Lesson 10, L01 in the pattern lessons) become O(1) space vs the map's O(n); name the trade-off. "What about 3-sum?" — fix one element with an outer loop, run complement lookup (or two pointers) on the rest: O(n²) time, O(n) space.

05 Limits

When does it degrade?

The O(1) guarantee is an average, not a promise. Three scenarios push the worst case toward O(n):

  • A malicious or colliding key set. If every key hashes to the same bucket (hash flooding attack), all lookups scan a single chain of length n. Production runtimes defend against this with hash randomisation; interview code rarely needs to worry about it.
  • A bad hash function. If you implement a custom key type and give it a trivial hash (return 1 for everything), you get a degenerate O(n) table. Always implement hash functions that distribute keys broadly.
  • High load factor without resizing. A fixed-size table that never grows will fill up; once load factor approaches 1 the average chain length grows and lookups become slower. Built-in hash maps resize automatically — only a concern if you implement your own.

In interview settings, the practical rule is: use built-in hash maps (they have well-tested hash functions and auto-resize) and state "O(1) average" in your complexity analysis — that is what the interviewer expects.

Keys must be hashable

In Python, only immutable types can be dictionary keys: integers, strings, tuples, and frozen sets. A list cannot be a key — use a tuple instead. In Java, any object whose hashCode() and equals() are correctly implemented can serve as a key. Mutable keys that change after insertion break the invariant and lead to "lost" entries — a subtle bug worth knowing about.

06 What comes next

Where this shows up next

Hash maps and sets are infrastructure, not a pattern by themselves. Almost every intermediate pattern lesson plugs them in to unlock O(1) lookups that replace an inner loop. Here is where you will use what you learned today:

L10 · Two Pointers

The unsorted alternative to sorted two pointers

When the array is not sorted, the complement-lookup idiom (section 03 here) replaces the converging pointer approach. You store each value as you go and ask for its partner — the same pattern as Two Sum above, but you will apply it to harder variants in L10.

L12 · Sliding Window

Frequency map inside a moving window

Sliding window keeps a hash map of the current window's character or value frequencies. Adding or removing an element from the window is a single O(1) map update — without the map, maintaining those counts would cost O(window size) per step, wrecking the overall complexity.

L13 · Prefix Sum

Complement lookup on running sums

Prefix sum problems (like counting subarrays that sum to a target) use a map from prefix-sum value to how many times it has appeared. That is exactly the frequency-counting idiom from section 03, applied to a different key type. Knowing this lesson makes the prefix-sum pattern feel immediate.

L28 · Top-K / Frequency

Count first, then select

Top-K element problems always start with a frequency map (count every value in O(n)), then select the K highest-frequency entries with a heap or bucket sort. The counting step is pure hash-map idiom from today's section 03.

The decision in one line

If the problem asks about relationships between elements — "have I seen X", "what is its complement", "how many times" — and a nested scan would give you O(n²), reach for a hash map. You will use this tool in almost every pattern lesson that follows.

★ Review

Summary & self-test

A hash map or set gives you O(1) average-time lookup on any key, turning quadratic "find a partner" problems into one clean linear scan. It does this by converting keys into array indices through a hash function, accepting the occasional O(n) collision in the worst case while keeping the amortised cost near constant.
