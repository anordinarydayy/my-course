Tree Construction & Serialization

A surprising number of hard tree questions reduce to one of two moves: rebuild a tree from the fingerprints left by its traversals, or flatten a tree into a string and recreate it faithfully. Both operations sit at the crossroads of recursion, index tracking, and the fundamental relationship between a node and its children — master them and an entire class of problems becomes a template.

Original study notes. Written from scratch for coding-interview prep. Problem names below (Construct Binary Tree from Preorder and Inorder Traversal, Serialize and Deserialize Binary Tree, Convert Sorted Array to BST) are cited only as well-known identifiers; every explanation, diagram, and line of code is original. 00 The idea

Why tree construction matters

A binary tree is defined by its shape and the values at each node, but a single traversal loses one or both. Print a preorder sequence and you know the root of every subtree, but not where its left child ends and its right begins. Print an inorder sequence and you see the left/right split perfectly, but you no longer know which node was the root. The insight that powers this whole lesson: combining two complementary traversals recovers the tree completely — one traversal pins the root, the other draws the boundary.

Serialization is the mirror problem: given a live tree in memory, produce a string a partner program can decode back into an identical tree. The naive approach — just write the values — loses structural information as soon as a node has only one child. The fix is elegantly simple: emit a sentinel (a null marker) for every absent child. With that one addition, a preorder walk becomes fully invertible.

  • Preorder + inorder — rebuild any binary tree whose values are distinct. (LeetCode 105)
  • Sorted array to BST — build a height-balanced BST by always picking the midpoint as root. (LeetCode 108)
  • Serialize / deserialize — encode a tree as a string and decode it exactly. (LeetCode 297)
The reusable shape

Every variant in this lesson uses the same recursive skeleton: pick a root, figure out which values belong to the left subtree and which to the right, then recurse on both halves. The only thing that changes per variant is how you identify the root (first in preorder / middle of sorted array / next non-null token) and how you draw the boundary (inorder index lookup / halving / consuming tokens left-to-right).

01 Two traversals

Rebuilding a tree from preorder and inorder

Start with the brute force: for each candidate root, scan the inorder array to find where the left subtree ends. That is O(n) per level and O(n²) overall on a skewed tree. The redundant work is the search. Pre-build a hash map from value → inorder index and every lookup drops to O(1), giving O(n) total.

The algorithm keeps two pointers into the preorder array. A single integer pre_idx marches left to right, claiming the next root on every recursive call — this is the key insight that avoids slicing arrays. It passes in_lo and in_hi bounds into each call to identify which slice of the inorder array belongs to the current subtree.

  • Root: preorder[pre_idx]. Advance pre_idx.
  • Boundary: look up the root's position m in the inorder map. Left subtree spans inorder indices [in_lo, m-1]; right spans [m+1, in_hi].
  • Base case: if in_lo > in_hi, this subtree is empty — return None.
  • Order matters: recurse left before right, because preorder fills left subtrees before right subtrees.
Why the preorder index must be shared

If you pass a fresh slice of preorder on each recursive call, both the time and space blow up to O(n²) on a skewed tree. Sharing a single mutable index (or a Python list holding one integer) lets you consume values left-to-right across all recursion frames — O(n) total allocations. The inorder bounds, by contrast, are just integers passed as arguments and cost O(1) per frame.

Worked problemMediumLeetCode 105
Construct Binary Tree from Preorder and Inorder Traversal

You are given the preorder and inorder traversals of a binary tree whose node values are all distinct integers. Reconstruct the original tree and return its root.

Think it through

Restate

Preorder visits root before its children; inorder visits left subtree, then root, then right. I need to recover the unique tree consistent with both.

Brute force

For each recursive call, scan the inorder array linearly to find where the root sits. That's O(n) per call and O(n) calls at each level, giving O(n²) overall on an unbalanced tree.

The waste

The linear scan is repeated for every node even though inorder positions never change. A single hash map built once makes every lookup O(1).

The move

Pre-build pos = {value: index} from the inorder array. Maintain a shared index pi into preorder. Each call: root = preorder[pi++]; m = pos[root.val]; recurse left over inorder [in_lo, m-1] then right over [m+1, in_hi]. Return None when in_lo > in_hi.

Invariant & cost

Every node is claimed exactly once from preorder via the shared index. Recursion depth is O(h) where h is tree height. Total: O(n) time, O(n) space (hash map + call stack).

The code

python

def buildTree(preorder, inorder):
    pos = {v: i for i, v in enumerate(inorder)}  # value → inorder index
    pi = [0]                                        # shared mutable index into preorder

    def build(in_lo, in_hi):
        if in_lo > in_hi:
            return None                              # empty subtree
        val = preorder[pi[0]]; pi[0] += 1          # claim next root
        node = TreeNode(val)
        m = pos[val]                                  # where root sits in inorder
        node.left  = build(in_lo, m - 1)            # left subtree first!
        node.right = build(m + 1, in_hi)
        return node

    return build(0, len(inorder) - 1)
Narrate it to the interviewer

"Restating: preorder gives the root of every subtree in the order I'll visit them; inorder tells me what's in the left versus right subtree once I know the root. Brute force is O(n²) because I'd scan inorder linearly for each node. The fix is to pre-build a hash map of value-to-index so each lookup is O(1). I share a single index into preorder across all recursive calls — that's the key move — so I never slice arrays. Each call claims the next preorder value as the root, looks up its inorder position, and recurses left before right. The invariant is that the shared index always points to the root of the current subtree. O(n) time and O(n) space."

Dry-run aloud: a single-node tree (preorder = inorder = [5], returns a leaf); a right-skewed tree (left recursion always hits the base case immediately); duplicate values (prompt says distinct, but mention the hash map breaks if they're not).

What the interviewer asks next

"What if you had postorder instead of preorder?" — the root is now the last element, and you recurse right before left. "Can you do it iteratively?" — yes, with an explicit stack that tracks the current node and whether you've finished its left child; it's fiddly but worth knowing exists. "What if values repeat?" — you'd need to pass index ranges into both arrays, not just values, which bumps complexity without the hash map trick.

02 Sorted input

Building a balanced BST from a sorted array

Brute force: pick the first element as root, recurse on the rest — that creates a completely right-skewed tree of height n, valid but maximally unbalanced (O(n) lookup). The wasted opportunity is the sortedness: the array already tells us the correct root for a height-balanced BST — it is the midpoint.

Think of it as binary search turned inside-out. In binary search you pick the midpoint and discard one half. Here you pick the midpoint as the root and recurse on both halves. The resulting tree has the minimum possible height ⌈log₂(n+1)⌉, which is optimal. The algorithm is three lines: compute mid = (lo + hi) // 2; make that the root; left subtree from [lo, mid-1]; right subtree from [mid+1, hi].

Midpoint as root

  • Splits array evenly at every level.
  • Height O(log n) — lookups stay fast.
  • O(n) time, O(log n) stack space.

First element as root

  • Degenerates to a right-linked list.
  • Height O(n) — lookup worst-case O(n).
  • Still O(n) time, but O(n) stack (dangerous).
Worked problemEasyLeetCode 108
Convert Sorted Array to BST

Given an array of integers sorted in ascending order, build a height-balanced binary search tree from it and return its root. A height-balanced tree has subtrees whose heights differ by at most one at every node.

Think it through

Restate

I have a sorted array. I want a BST where every node satisfies the BST property and no subtree is disproportionately tall.

Brute force

Insert elements in sorted order one by one into a plain BST. Each insert hits the same rightmost path, producing a skewed tree of height n with O(n) lookup — technically O(n) time overall but a terrible tree.

The waste

Inserting in sorted order ignores the structure the sorted order already encodes. The element that should be the root of a balanced tree is sitting right in the middle of the array — no search required.

The move

Pick mid = (lo + hi) // 2 as the root. Its left subtree comes from [lo, mid-1] and its right from [mid+1, hi] — both halves have nearly equal length, so every split is even. Recurse until lo > hi (empty subarray → None).

Invariant & cost

At every level the two children receive arrays that differ in length by at most one, so the resulting tree's height is ⌈log₂(n+1)⌉. Each of the n elements is created as a node once: O(n) time, O(log n) space (stack depth).

The code

python

def sortedArrayToBST(nums):
    def build(lo, hi):
        if lo > hi:
            return None                # empty slice
        mid = (lo + hi) // 2       # midpoint is the balanced root
        node = TreeNode(nums[mid])
        node.left  = build(lo, mid - 1)
        node.right = build(mid + 1, hi)
        return node

    return build(0, len(nums) - 1)
Narrate it to the interviewer

"The array is sorted, so the element that gives the most balanced BST is the midpoint — pick it as root, recurse on the left half for the left subtree and the right half for the right. The base case is an empty slice. Every split is even, so the tree height is O(log n), and each of the n nodes is created exactly once — O(n) time, O(log n) space for the call stack. Inserting in order would give a skewed tree, which is technically correct for BST property but catastrophic for height."

Dry-run aloud: a single-element array (one node, both subtrees empty); a two-element array (mid = 0, root = first element, right = second, left = None — or mid = 1 depending on convention; confirm both produce valid balanced BSTs); an empty array (return None immediately).

03 Flattening

Serializing a tree with preorder + null markers

The naive approach: write only the node values in some traversal order. The problem is structural ambiguity — the string "3 9 7" could describe a root with two children or a root with only a right child named 9 whose own right child is 7. Without knowing which children are absent, decoding is impossible.

The fix is to emit a sentinel — here the string "N" — for every absent child. A preorder walk with null markers produces a string that is fully invertible: every possible tree shape maps to a distinct token sequence, and the sequence determines the shape completely. The rule during serialization is simple: visit the current node; if it is null, emit N and return; otherwise emit its value then recurse left then right.

Why preorder, not inorder?

Inorder with null markers is not uniquely decodable: given the tokens left to right, you cannot tell when the left child ends and the right begins without already knowing the tree structure. Preorder is decodable because the first non-null token is always the root — you know it immediately and can recurse left, then right, in a single left-to-right pass. Level-order also works (covered in section 05) but requires a queue during decoding.

Worked problemHardLeetCode 297
Serialize and Deserialize Binary Tree

Design a codec that converts a binary tree to a string and converts that string back to an identical tree. Any integer values may appear, including negatives. Your encoding scheme must handle any tree shape, including trees with only one child at some nodes.

Think it through

Restate

I need a serialize(root) -> str and deserialize(s) -> TreeNode that are inverses. Any lossless encoding that handles arbitrary shapes works — I'll choose preorder with null markers because deserialization is a clean left-to-right token scan.

Brute force

Write only non-null values. Fails because you cannot distinguish "right child only" from "left child only" without knowing the full shape. Information is lost.

The waste

The missing information is the location of absent children. Emitting a sentinel for each null child adds at most n+1 tokens to an n-node tree — a constant factor. That small overhead buys complete decodability.

The move — serialize

Preorder DFS: if node is None, append "N" and return; otherwise append str(node.val), recurse left, recurse right. Join tokens with a comma separator.

The move — deserialize

Split on comma into a token list, wrap in an iterator. DFS: pull the next token; if it is "N", return None; otherwise create a node with that value, assign its left from a recursive call, then its right, return the node.

Invariant & cost

Each node produces exactly one value token and each absent child produces one "N" token. The iterator advances exactly once per token. O(n) time and O(n) space for both serialize and deserialize.

The code

python

class Codec:
    def serialize(self, root):
        tokens = []
        def dfs(node):
            if node is None:
                tokens.append("N"); return
            tokens.append(str(node.val))
            dfs(node.left); dfs(node.right)     # preorder: root first
        dfs(root)
        return ",".join(tokens)

    def deserialize(self, data):
        it = iter(data.split(","))           # left-to-right token iterator
        def build():
            tok = next(it)
            if tok == "N": return None    # null sentinel
            node = TreeNode(int(tok))
            node.left  = build()              # consume left subtree first
            node.right = build()
            return node
        return build()
Narrate it to the interviewer

"I'll serialize with a preorder DFS, emitting the value for each real node and 'N' for each absent child, joined by commas. Preorder is decodable left-to-right because the first token is always the root — I don't need to know the shape in advance. Deserializing splits the string into tokens, wraps them in an iterator, and runs the same DFS structure: pull the next token; if it's 'N' return None; otherwise create the node and assign left then right from recursive calls. The iterator advances exactly once per token — O(n) time and space for both directions. The null markers are the critical ingredient; without them I'd lose structural information for nodes with only one child."

Dry-run aloud: empty tree (serialize returns "N", deserialize returns None); a single node with value 5 (serialize "5,N,N", deserialize reads "5", creates node, reads "N" twice); a tree with only a left child (null right child gets its own "N" token, distinguishing it from a tree with only a right child).

04 Inverting the codec

Deserializing: consuming tokens to rebuild

Deserialization is where the design pays off. The token sequence produced by preorder serialization has a remarkable property: you can reconstruct the tree by reading tokens strictly left to right, one per recursive call, with no backtracking and no lookahead. The structure emerges automatically from the recursion pattern.

The key data structure is an iterator (or equivalently, a pointer into the token list) shared across all recursive frames. Each call pulls exactly one token: if it sees "N" it returns None immediately (the null leaf); otherwise it creates a node and immediately recurses for the left subtree, then for the right. The iterator position after a subtree is deserialized is always exactly at the start of the next sibling subtree — no fencing or count-checking needed.

Preorder codec

  • Deserialization is a single left-to-right pass.
  • No queue, no level bookkeeping.
  • String length: 2n + 1 tokens worst case (leaf-heavy).

Level-order codec

  • Matches LeetCode's own tree display format.
  • Needs a queue during both encode and decode.
  • Shorter output for wide, shallow trees.
What the interviewer asks next

"Can you serialize an n-ary tree?" — yes, add a child-count token after each node value so the decoder knows how many children to pull. "Can you handle arbitrary graphs?" — no, trees are acyclic; a graph with cycles would loop forever. You'd need a visited set and adjacency list encoding. "What if the values can be very large integers?" — the string representation scales naturally; no bit-packing concerns here.

05 Alternative encoding

Level-order encoding

The BFS (level-order) approach encodes the tree exactly as LeetCode displays it: nodes level by level, with null for absent children. It is worth knowing both because interviewers sometimes specify level-order, and because the two decoders exercise different skills — DFS recursion versus BFS queue management.

Serialization uses a queue initialized with the root. At each step: dequeue a node; if it is null, emit a null sentinel; otherwise emit its value and enqueue both children (even if they are null — but a common optimization is to stop once the queue contains only nulls). Deserialization reads the first token as the root, uses a queue of nodes waiting for children, and assigns tokens pair-wise as left and right children.

Optimization: trim trailing nulls

Real-world serializers (and LeetCode's own format) stop emitting null tokens once the queue contains only nulls — leaf nodes have no children to enumerate. This saves space for leaf-heavy trees. The trade-off is a slightly more complex decoder that must check whether a null's "children" tokens are even present. For interviews, emitting all nulls is simpler and perfectly acceptable — just state the trade-off.

How to drill this pattern

For every problem:
(1) identify which traversal anchors the root — preorder, postorder, or sorted midpoint;
(2) name the boundary mechanism — inorder index map, token iterator, or array halving;
(3) state the shared state across recursive calls — the mutable index or iterator;
(4) write the base case first;
(5) code it and dry-run on a three-node tree with a missing left child. The missing-child case catches most bugs because it forces a null token or an empty subtree slice.

★ Review

Summary & self-test

Every problem in this pattern shares a single recursive skeleton: identify the root from the encoding, determine what belongs to the left subtree and what to the right, then recurse. The encoding dictates how you identify the root (first in preorder, last in postorder, middle of a sorted array, next non-null token) and how you draw the boundary (inorder index, array halving, null sentinel, or child count). Internalize those three moving parts and any variant becomes a template fill-in.

01Preorder gives rootsThe first element of a preorder sequence is always the root of its subtree — consume it first, recurse after.
02Inorder splitsEverything left of the root's inorder position is the left subtree; everything right is the right. Hash positions for O(1) lookup.
03Share the indexAvoid slicing arrays. A single mutable index into the preorder (or token) sequence, shared across all frames, keeps rebuild at O(n).
04Midpoint = balanceFor a sorted array, the midpoint is the root of the optimal height-balanced BST. Recurse on both halves symmetrically.
05Null markers make it invertibleA preorder walk alone is lossy. Emitting "N" for every absent child makes the sequence uniquely decodable.
06Preorder deserializationA left-to-right token iterator with no lookahead reconstructs the tree: non-null creates a node and recurses; "N" returns None.
07Two encodingsPreorder: DFS, simpler decoder. Level-order: BFS, matches LeetCode's format, needs a queue. Know when to use each.
08Postorder mirrors preorderRoot is last, not first; recurse right before left; consume from the end of the sequence.
