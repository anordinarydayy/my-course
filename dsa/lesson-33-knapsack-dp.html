Knapsack DP

A surprising number of medium and hard DP problems are really the same question in disguise: given a set of items and a capacity, can you fill it exactly — and if so, how many ways, or using how few items? Learning to spot the 0/1 knapsack and the unbounded knapsack beneath their costumes is the single biggest unlock in the dynamic programming phase of interview prep.

Problem names below (Coin Change, Partition Equal Subset Sum, Coin Change II) are referenced only as well-known identifiers; every description, diagram, walkthrough, and line of code here is original. 00 The core idea

What problem does it solve?

The classical knapsack problem goes like this: you have a bag with a weight limit and a pile of items each with a weight. Which subset of items can you pack without busting the limit? At first glance this looks like a search problem — try every subset, pick the ones that fit. With n items there are 2n subsets; that is 230 ≈ one billion for a modest input, way too slow.

The rescue is a classic DP insight: the answer for capacity c using the first i items depends only on the answer for capacity c and c − weighti using the first i − 1 items. Each sub-problem is computed once and stored in a one-dimensional array indexed by capacity. Two decisions define the entire family:

  • 0/1 knapsack — each item may be used at most once. Disguises: subset sum, equal-partition, 0/1 target fill.
  • Unbounded knapsack — each item may be reused as many times as needed. Disguises: coin change (fewest coins), coin combinations (count ways).
The reusable shape

In almost every knapsack problem the DP array is indexed by remaining capacity (or target), and the recurrence asks: "what is the best answer if I include this item?" vs "what if I skip it?" Recognizing capacity as the dimension to DP over — and the item as the outer loop driving state transitions — is the pattern's core muscle. Everything else is a twist on that frame.

01 0/1 knapsack

Each item used at most once ►

The brute-force answer tries all 2n subsets of items and checks whether each subset's total weight is at most the capacity — exponential time, completely impractical for large inputs.

The key redundancy: many subsets share a common prefix of items and a common remaining capacity. Rather than recomputing from scratch, store the result for every (items-so-far, capacity) pair. Because the items dimension only appears as "first i items", you can compress it away with a careful update order, leaving a single array dp of length capacity + 1.

The recurrence is: for each item with weight w, iterate capacity from the top down — from capacity down to w — and set dp[c] = max(dp[c], dp[c - w] + value). Sweeping downward prevents using the same item's update twice in the same pass (which would effectively use it more than once). After processing all items, dp[capacity] holds the maximum value achievable without exceeding the limit.

Worked problemMediumLeetCode 416
Partition Equal Subset Sum

You are given an array of positive integers. Decide whether you can split it into two groups such that both groups have the same sum. Return true or false.

Think it through

Restate

If the total is odd, no equal split exists — return false immediately. Otherwise the target for each half is total / 2. The question becomes: is there a subset that sums to exactly that target? This is a classic 0/1 knapsack feasibility check.

Brute force

Enumerate every subset, check its sum against the target. With n elements that is 2n subsets — O(2n) time. At n = 30 this is a billion operations, far too slow.

The waste

Many subsets share common elements and common partial sums. The same partial sum is recomputed repeatedly. We need to cache "can we reach sum s using elements considered so far?" — that is exactly the 0/1 knapsack DP over the capacity dimension.

The move

Set target = total // 2. Allocate dp, a boolean array of length target + 1, with dp[0] = True (zero sum is always reachable). For each number x in the array, sweep c from target down to x and set dp[c] |= dp[c - x]. The downward sweep is mandatory — it prevents x from contributing to its own update in the same pass (which would let it be used twice).

Invariant & cost

After processing element x, dp[c] is true iff the subset of elements seen so far can sum to exactly c. Total work is O(n × target) time and O(target) space — the item axis is compressed away.

The code

python

def canPartition(nums):
    total = sum(nums)
    if total % 2 != 0:
        return False           # odd total → impossible
    target = total // 2
    dp = [False] * (target + 1)
    dp[0] = True              # empty subset sums to 0
    for x in nums:
        for c in range(target, x - 1, -1):  # sweep HIGH → LOW (0/1)
            dp[c] = dp[c] or dp[c - x]
    return dp[target]
Narrate it to the interviewer

"The question is really 'does a subset sum to total/2?' — that is 0/1 knapsack feasibility. The brute force is exponential, but DP over the capacity axis gives O(n × target). I keep a boolean array dp where dp[c] is true iff some subset of elements seen so far sums to exactly c. For each number I sweep capacity from high to low — the downward direction prevents the same number from being used in the same pass. My invariant is that after element x is processed, dp correctly reflects all reachable sums. Return dp[target]."

Dry-run aloud: [1, 1] → target 1, dp after first 1: dp[1] = True → answer true. [1, 2, 3, 5] → total 11, odd → false immediately. [2, 2, 1, 1] → target 3, check dp[3] is reached via 2+1.

What the interviewer asks next

"Why sweep downward for 0/1 but upward for unbounded?" → in the downward pass, dp[c - x] was set in a previous item's pass, so using it adds x exactly once. Upward would let the same item appear again in the same pass. "Can you reconstruct which subset was chosen?" → after the DP, backtrack: for each item from last to first, if dp[c] is true and dp[c - w] was also true, the item was not taken; if dp[c - w] was true but only after the item, it was taken — you need a 2-D table to do this cleanly.

02 Unbounded knapsack

Each item reusable without limit ►

The brute force for the unbounded case is still exponential: how many ways can you pick items (with repetition) so that the total weight equals a target? Enumerate all combinations — again, hopeless for large inputs.

The only change from 0/1 knapsack is the sweep direction. Sweep capacity from low to high. When you compute dp[c] = max(dp[c], dp[c - w] + value) with an upward pass, dp[c - w] may already have been updated in the current pass — meaning the item at weight w has been included again. That is exactly what we want for unlimited reuse.

This single flip — high-to-low becomes low-to-high — is what separates the two variants. Everything else (the outer loop over items, the recurrence itself) stays the same.

Worked problemMediumLeetCode 322
Coin Change

You have a collection of coin denominations and an integer amount. Return the fewest number of coins that add up to the amount exactly, or −1 if no combination can reach it.

Think it through

Restate

Coins may be used as many times as needed (unbounded). Minimize the coin count. If unreachable, return −1. This is the canonical unbounded knapsack minimization problem.

Brute force

Recursive enumeration: for each amount, try subtracting each coin and recurse. Without memoization this is O(Sk) where S is the amount and k is the number of coin types — explosive for large S.

The waste

The same sub-amount is solved repeatedly on different branches of the recursion tree. Memoize the result for each sub-amount, collapsing the tree into a table. Bottom-up, this is exactly the unbounded DP: compute answers for every amount from 0 up to S.

The move

Initialize dp[0] = 0 (zero coins to make amount zero); all other entries to infinity. For each coin c, sweep capacity from c up to amount. Set dp[a] = min(dp[a], dp[a - c] + 1). The upward direction lets the same coin be used again. After all coins, if dp[amount] is still infinity, return −1; otherwise return it.

Invariant & cost

After processing all coins up to amount a, dp[a] holds the minimum coins needed. O(S × k) time, O(S) space.

The code

python

def coinChange(coins, amount):
    INF = float('inf')
    dp = [INF] * (amount + 1)
    dp[0] = 0                    # base case: 0 coins to make 0
    for coin in coins:
        for a in range(coin, amount + 1):  # sweep LOW → HIGH (unbounded)
            if dp[a - coin] != INF:
                dp[a] = min(dp[a], dp[a - coin] + 1)
    return dp[amount] if dp[amount] != INF else -1
Narrate it to the interviewer

"This is unbounded knapsack — each coin can be reused. The brute-force recursion has exponential branching but massive repeated sub-problems; DP over the amount axis fixes that. I initialize dp[0] = 0 and everything else to infinity, then for each coin I sweep the amount upward: dp[a] = min(dp[a], dp[a - coin] + 1). The upward direction is the key — it allows the same coin to contribute to dp[a - coin] which was already updated in this very pass, so the coin gets reused for free. Invariant: dp[a] is the minimum coins to reach exactly a, using any subset of coins with repetition. If dp[amount] is still infinity at the end, no combination works: return −1."

Dry-run aloud: amount = 0 → return 0 (dp[0] = 0). Coins = [2], amount = 3 → dp[3] stays INF → return −1. Coins = [1, 2, 5], amount = 11: dp[11] = min via 5+5+1 = 3 coins.

What the interviewer asks next

"What if coins are not sorted — does order matter?" → no; the outer loop is over coins (any order) and the inner loop is over amounts. "Can you make the outer loop amounts and inner loop coins?" → yes, that gives the same result for both variants with the same sweep logic. "What does dp[0] = 0 represent?" → it is the base case: zero coins are needed to form amount zero, which bootstraps all other transitions.

03 The disguises

Subset sum and equal partition

Real interview problems almost never announce themselves as "this is 0/1 knapsack". Instead they frame it differently. Here is the translation dictionary:

  • Subset Sum: given an array of integers, does any subset sum to exactly T? → 0/1 knapsack feasibility, target = T, dp is boolean.
  • Equal Subset Partition (LC 416): split array into two equal-sum groups → subset sum with target = total / 2. Odd total = instant false.
  • Last Stone Weight II (LC 1049): split array to minimize the difference of two group sums → find the largest subset sum ≤ total / 2 using 0/1 DP, then answer is total − 2 × that sum.
  • Target Sum (LC 494): assign + or − to each element to reach a target → reducible to a subset sum once you do the algebra: the "plus" group sums to (total + target) / 2, which is a fixed integer (or infeasible if odd / negative).
Initialization tells you the objective

The base case at capacity 0 encodes the goal: set dp[0] = True for feasibility, dp[0] = 0 for minimization (zero cost to reach zero), dp[0] = 1 for counting (one way to reach zero: the empty selection). Everything else starts at False / ∞ / 0 respectively. Getting the initialization wrong is the #1 source of subtle bugs in knapsack problems.

Worked problemMediumLeetCode 416
Partition Equal Subset Sum

Given a non-empty array of positive integers, determine whether you can divide the array into two groups such that both groups have an identical total.

Think it through

Restate

Can a subset of the array sum to exactly half the grand total? If the total is odd, no equal split can exist. This is 0/1 subset-sum with target = total / 2.

Brute force

Try all 2n subsets. O(2n), way too slow for n up to 200.

The waste

The same partial sum is reached via many different subset orderings. Cache "is sum s reachable?" for each s up to target.

The move

Build a boolean dp[0..target]. dp[0] = True. For each number x (0/1 → sweep target down to x): dp[c] |= dp[c - x].

Invariant & cost

dp[c] is true iff some subset of processed numbers sums to c. O(n × target) time, O(target) space.

The code

python

def canPartition(nums):
    total = sum(nums)
    if total % 2 != 0:
        return False
    target = total // 2
    dp = [False] * (target + 1)
    dp[0] = True
    for x in nums:
        for c in range(target, x - 1, -1):  # 0/1: high → low
            dp[c] = dp[c] or dp[c - x]
    return dp[target]
Narrate it to the interviewer

"I restate it as: can any subset sum to total/2? Odd total → impossible. I track reachable sums in a boolean dp array. For each number I sweep capacity downward — that is the 0/1 guard — updating dp[c] with dp[c - x]. Invariant: dp[c] is true iff some subset of numbers seen so far sums to c. The answer is dp[target]. Time O(n × target), space O(target)."

Dry-run: [1, 5, 11, 5] → total 22, target 11. After 1: dp[1]. After 5: dp[5], dp[6]. After 11: dp[11] → true. Return true.

04 Coin Change (min)

Fewest coins to reach a target ►

The brute force for "fewest coins" is a recursion that tries every coin at every remaining amount. Without caching, the recursion tree branches k ways at each of the S levels — O(kS) work. Memoization collapses the tree to O(S × k); the bottom-up DP version of that is exactly the upward sweep shown in FIG. 2.

A key subtlety: the order of the loops can be outer-coins / inner-amounts or outer-amounts / inner-coins and you get the same minimum-cost result either way. The sweep direction within the inner loop is what matters for correctness (upward = unbounded, downward = 0/1). The interactive figure below (FIG. 4) shows the sweep in slow motion for coins [1, 3, 4] and target 6, where the answer is 2 coins (3 + 3).

Worked problemMediumLeetCode 322
Coin Change

You have an unlimited supply of coins with given denominations. Return the minimum number of coins needed to total a given amount exactly, or −1 if it is impossible.

Think it through

Restate

Unbounded supply (each coin reusable), minimize count. Classic unbounded knapsack minimization.

Brute force

Recursive branching: at each amount, try every coin. Without memoization: O(kS) where k is the number of coin denominations. Explodes quickly.

The waste

Subproblems are repeated massively. dp[a] is computed fresh on every branch that passes through amount a. Cache it once — bottom-up DP over all amounts.

The move

dp[0] = 0, rest = ∞. For each coin, sweep amounts LOW to HIGH (unbounded). Set dp[a] = min(dp[a], dp[a - coin] + 1) whenever dp[a - coin] is finite.

Invariant & cost

dp[a] = fewest coins to exactly form a. O(S × k) time, O(S) space.

The code

python

def coinChange(coins, amount):
    INF = float('inf')
    dp = [INF] * (amount + 1)
    dp[0] = 0                        # 0 coins to make 0
    for coin in coins:
        for a in range(coin, amount + 1):  # low → high: unbounded
            if dp[a - coin] < INF:
                dp[a] = min(dp[a], dp[a - coin] + 1)
    return dp[amount] if dp[amount] < INF else -1
Narrate it to the interviewer

"Unlimited coins, minimize count — unbounded knapsack. Brute recursion is exponential; DP over the amount axis is O(S × k). Base: dp[0] = 0. For each coin, sweep upward so each coin can be used again. Invariant: dp[a] is the minimum coins to reach exactly a. If it stays infinity, that amount is unreachable."

Dry-run: coins=[2], amount=3 → dp[2]=1, dp[3] stays INF → return -1. Coins=[1,2,5], amount=11 → dp[11] = min via 5+5+1 = 3.

05 Count the ways

Counting combinations: Coin Change II

The next twist is counting instead of deciding or minimizing. "How many different ways can you form the target?" is the question in Coin Change II (LC 518). The coins are still unbounded (reusable), but the objective changes: instead of taking the minimum, you take the sum of all ways.

The only change from the minimization version is the update: dp[a] += dp[a - coin] instead of min(dp[a], dp[a - coin] + 1). The base case also shifts: dp[0] = 1 (there is exactly one way to reach zero: choose nothing). All other entries start at zero. The sweep remains upward (unbounded).

Combinations vs permutations

The outer loop over coins (with inner loop over amounts) counts combinations — distinct multisets of coins, regardless of order. Reversing the loops (outer = amounts, inner = coins) counts ordered sequences (permutations, or "climbing stairs" style). Interviewers sometimes test whether you know the difference. Coin Change II asks for combinations, so coins go in the outer loop.

Worked problemMediumLeetCode 518
Coin Change II

You have coins of various denominations in unlimited quantity. Return the number of distinct combinations of coins that sum to a given amount. Combinations differing only by order count as the same.

Think it through

Restate

Count distinct (unordered) multisets of coins summing to amount. Unbounded supply. This is the counting variant of unbounded knapsack.

Brute force

Generate all ordered selections of coins (each of length up to amount / min-coin) and count those summing to amount. Truly exponential in the worst case.

The waste

The number of ways to reach sub-amount a is recomputed for every larger amount that passes through it. Cache it. Also, generating ordered sequences then deduplicating is wasteful — putting coins in the outer loop naturally avoids duplicated combinations by only ever picking coins ≥ the current coin denomination.

The move

dp[0] = 1 (one way to reach 0). For each coin, sweep amounts LOW to HIGH: dp[a] += dp[a - coin]. The coin-outer-loop structure ensures every combination is counted once: you only combine coins in left-to-right denomination order.

Invariant & cost

After coin c is processed, dp[a] = number of combinations using only coins processed so far that sum to a. O(amount × k) time, O(amount) space.

The code

python

def change(amount, coins):
    dp = [0] * (amount + 1)
    dp[0] = 1               # 1 way to reach 0: the empty selection
    for coin in coins:      # coin in outer loop → counts combinations
        for a in range(coin, amount + 1):
            dp[a] += dp[a - coin]
    return dp[amount]
Narrate it to the interviewer

"Count unordered combinations of coins summing to amount. The key decision: coin in the outer loop means each combination is counted once — the same set can only appear in sorted-coin order. dp[0] = 1 seeds it (one way to make 0). For each coin, sweep amounts upward (unbounded) and add: dp[a] += dp[a - coin]. Invariant: dp[a] is the number of combinations using coins seen so far. If I reversed the loops, I'd count ordered sequences, not combinations."

Dry-run: amount=5, coins=[1,2,5]. After coin 1: dp = [1,1,1,1,1,1]. After coin 2: dp[2]=2(1+1 or 2), dp[3]=2, dp[4]=3, dp[5]=3. After coin 5: dp[5]=4 (add dp[0]=1 → 3+1=4). Answer: 4 ways (1+1+1+1+1, 1+1+1+2, 1+2+2, 5).

06 Sweep direction

Why iteration order is the whole trick

The single most important thing to memorize about knapsack DP is the sweep direction for the inner (capacity) loop, and why it is what it is. Get it wrong and the answer is silently incorrect — no runtime error, just a wrong number.

0/1 knapsack — sweep HIGH → LOW

  • Each item may be used at most once.
  • When computing dp[c], you pull from dp[c - w].
  • Sweeping downward means dp[c - w] was set in the previous item's pass — so the item cannot contribute to its own update.
  • If you swept upward, dp[c - w] might already have the current item included → item used twice.

Unbounded knapsack — sweep LOW → HIGH

  • Each item may be used any number of times.
  • Sweeping upward means dp[c - w] might already reflect the current item in this pass → item counted again.
  • That re-use is exactly what we want.
  • If you swept downward, each item would be limited to one use per pass — effectively 0/1 behaviour.

The mnemonic: 0/1 = once, sweep from the top; unbounded = unlimited, sweep from the bottom. The word "unlimited" starts with the same letter as "low".

Common bugs and interview traps

Wrong base case: for minimization, initializing dp[0] to 1 instead of 0 will add one phantom coin. For counting, initializing dp[0] to 0 instead of 1 yields zero for every amount because the empty-selection base is missing.

Skipping the impossibility check: after the minimization DP, if dp[amount] is still ∞, you must return −1, not the infinity value. Always guard.

Target computed before the odd-total check: in equal-partition, if total is odd, total//2 rounds down and you will test an incorrect target. Do the parity check first.

Combinations vs. permutations mix-up: if the interviewer wants distinct ordered arrangements, coins go in the inner loop; if they want unordered sets, coins go in the outer loop. Name the distinction out loud.

07 Practice

Practice ladder

Work through every card top to bottom. For each: identify the variant (0/1 or unbounded), name the dp array semantics, state the sweep direction and why, write the code, and dry-run the edge cases. Aim for the easy tier in under 20 minutes each and the medium tier in under 35.

EasyClimbing Stairsunbounded / count

You can climb 1 or 2 steps at a time. How many distinct ways are there to reach the top of an n-step staircase?

Idea Unbounded knapsack counting over "steps": dp[i] = dp[i-1] + dp[i-2]; dp[0]=1, dp[1]=1. O(n) time, O(1) space.

EasyHouse Robber0/1 variant

Houses are arranged in a row; robbing adjacent houses triggers an alarm. Find the maximum amount you can steal.

Idea dp[i] = max(dp[i-1], dp[i-2] + nums[i]). Conceptually "choose this house or skip it" — the 0/1 choice applied along a linear chain. O(n) time, O(1) space.

EasyPerfect Squaresunbounded / min

Given a positive integer n, return the minimum number of perfect squares (1, 4, 9, 16, ...) that sum to n.

Idea Unbounded knapsack minimization; coins = [1, 4, 9, ...] up to sqrt(n). dp[0]=0, sweep upward. O(n × sqrt(n)).

MediumPartition Equal Subset Sum0/1 / feasibility

Split an array of positive integers into two groups with identical sums, or report that it cannot be done.

Idea Target = total/2; 0/1 boolean DP, sweep high to low. Return dp[target]. O(n × target).

MediumCoin Changeunbounded / min

Unlimited coins, various denominations — find the fewest coins summing to the target, or −1 if impossible.

Idea Unbounded minimization; dp[0]=0, rest=∞, sweep low to high. Guard the return value. O(S × k).

MediumCoin Change IIunbounded / count

Unlimited coins — count the number of distinct (unordered) combinations that sum to the target.

Idea Unbounded counting; dp[0]=1, rest=0, coins in outer loop, sweep low to high. O(amount × k).

MediumTarget Sum0/1 / count

Assign a + or − sign to each integer in an array. Count the number of ways to reach a given target.

Idea Algebra: P - N = target, P + N = total, so P = (total + target) / 2. Count subsets summing to P — 0/1 counting DP, sweep high to low. O(n × P).

MediumLast Stone Weight II0/1 / max subset

You can smash any two stones together; the survivor has weight equal to their absolute difference. Find the smallest possible final weight.

Idea Minimize |S1 - S2| = total - 2*S1; maximize S1 ≤ total/2 via 0/1 DP. Answer = total - 2 * dp_max_reachable. O(n × total/2).

MediumCombination Sum IVunbounded / ordered count

Integers can be used multiple times in any order. Count the number of ordered sequences (not just sets) that sum to a target.

Idea Outer loop = amounts, inner loop = coins (reversed loop order from Coin Change II). dp[0]=1. O(target × k).

HardOnes and Zeroes2-D 0/1 knapsack

A list of binary strings, each costing some number of 0s and some number of 1s. Find the largest subset that fits within a budget of m zeros and n ones.

Idea 2-D 0/1 knapsack: dp[i][j] = max strings using at most i zeros and j ones. Sweep both dimensions downward. O(L × m × n) where L = number of strings.

HardProfitable Schemes2-D DP

Each crime requires some gang members and yields some profit. Count the number of schemes using at most k members that generate at least minProfit total profit.

Idea 2-D DP: dp[members][profit]. Both constraints handled together; profit dimension is capped at minProfit (anything beyond the minimum counts). O(n × k × minProfit).

How to drill knapsack problems

Every time you sit down with a new problem, ask four questions before writing code:
(1) Is each item used once (0/1) or many times (unbounded)?
(2) Am I deciding feasibility, minimizing a count, or counting all ways?
(3) What is dp[0] and what does dp[c] mean in plain English?
(4) Which direction does the inner loop sweep, and why? If you can answer all four in under 30 seconds, you are ready to code. Narrate the reasoning out loud — the interviewer is as interested in your thinking process as in the correct answer.

★ Review

Summary & self-test

The knapsack family compresses to one line: keep a 1-D array indexed by capacity, process items one by one in the outer loop, and let the sweep direction of the inner loop decide whether items can be reused. High-to-low for 0/1; low-to-high for unbounded. Change the update and initialization for minimization or counting. Every problem in this family is that frame plus a disguise.

01Two variants0/1 (each item once) and unbounded (reuse freely); everything else is a remix.
02Sweep direction0/1 sweeps capacity high→low; unbounded sweeps low→high. That single difference is the entire distinction.
03Initialization = objectivedp[0]=True/0/1 for feasibility/minimization/counting. Everything else starts False/∞/0.
04Equal partitionTarget = total/2; odd total → instant false. Reduces to 0/1 subset-sum.
05Coin Change (min)Unbounded, minimize. dp[0]=0, rest=∞. Sweep up. Guard the return for ∞.
06Coin Change II (count)Unbounded, count combinations. dp[0]=1. Coins in outer loop to avoid counting permutations.
07The disguisesSubset sum, equal partition, target sum, last stone weight — all are 0/1 DP after a simple algebraic reduction.
082-D extensionsTwo constraint dimensions (weight + volume, 0s + 1s) just add a second dp dimension; still sweep both downward for 0/1.
