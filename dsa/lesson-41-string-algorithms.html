String Algorithms — KMP & Rabin-Karp

Searching for one string inside another sits at the heart of editors, search engines, DNA analysis, and intrusion detection — and the naive double loop is always the first answer and almost never the right one. KMP removes backtracking by precomputing how much of the pattern is already matched when a mismatch occurs; Rabin-Karp turns the match into an arithmetic check by hashing a sliding window. Together they give you two complementary O(n+m) tools for linear pattern matching.

00 The baseline

The cost of naivety

Given a text T of length n and a pattern P of length m, the straightforward substring search tries every alignment: for each starting position i in the text, check whether T[i..i+m-1] matches P character by character. On a mismatch you reset the pattern index to zero and shift the text index by one. That loop is at most n × m comparisons.

Why is that bad? For text length n = 10,000 and pattern length m = 1,000 the worst case is ten million comparisons — already slow — and you hit that worst case when the text is something like aaaa…aaab and the pattern is aaa…ab: you almost match on every alignment, restart, and crawl forward by just one character each time. With DNA sequences or large log files, the text can be hundreds of megabytes long.

The bottleneck is that a mismatch throws away every character comparison made so far. But those comparisons revealed something: we know the prefix of the pattern that matched before the failure, and the structure of that prefix may mean that some shorter prefix of it also occurs at the current text position. KMP exploits that. Rabin-Karp takes a completely different angle: use a hash to compare the whole window at once, and only do character-by-character verification on a hash hit.

Two tools, one goal

KMP precomputes a failure table from the pattern alone (O(m)), then uses it to skip redundant comparisons during the scan (O(n)). The text pointer never backtracks. Rabin-Karp reduces each comparison to a hash arithmetic check (O(1) per window slide) and only does character work on a hash collision. KMP is deterministically O(n+m); Rabin-Karp is expected O(n+m) with a small constant but O(nm) in the adversarial worst case. Pick KMP for worst-case guarantees; pick Rabin-Karp for multi-pattern search or when hash-arithmetic simplicity matters more.

01 KMP phase 1

The prefix function (failure table)

The prefix function — sometimes called the failure function — is the heart of KMP. For each index j in the pattern, fail[j] stores the length of the longest proper prefix of P[0..j] that is also a suffix. "Proper" means it can't be the whole string itself. This single array encodes everything about how the pattern overlaps with itself.

For example, with pattern ABCAB:

  • fail[0] = 0 — a single character has no proper prefix.
  • fail[1] = 0 — "AB" has no prefix that is also a suffix.
  • fail[2] = 0 — "ABC": still nothing.
  • fail[3] = 1 — "ABCA": the prefix "A" equals the suffix "A".
  • fail[4] = 2 — "ABCAB": the prefix "AB" equals the suffix "AB".

Computing it: maintain a pointer k that tracks the length of the current matching prefix. Walk j from 1 to m−1. If P[k] == P[j], both extend: fail[j] = k+1, advance both. If they differ and k > 0, fall back: set k = fail[k−1] and retry without advancing j. If they differ and k == 0, the answer is 0 and move j forward. This runs in O(m) because the total number of times k can fall back is bounded by the number of times it advanced.

What the table says

When KMP detects a mismatch at pattern position j, fail[j−1] tells it how many characters of the pattern are already confirmed at the current text position — so it jumps j back to fail[j−1] instead of 0. The text pointer never retreats. That single non-backtracking property is why the full match runs in O(n) after the O(m) table build.

Worked problemHardLeetCode 1392
Longest Happy Prefix

Given a string s, find its longest prefix that is also a suffix (the prefix and suffix must not overlap the whole string — a string is not a prefix-suffix of itself). Return the empty string if none exists.

Think it through

Restate

Find the longest proper prefix of s that equals a proper suffix — exactly what the KMP prefix function computes for the last character.

Brute force

Try every prefix length from len(s)−1 down to 1; for each, compare that prefix with the matching suffix character by character. That is O(n²) in the worst case (e.g., aaa…a).

The waste

Each brute-force comparison re-checks characters that the previous comparison already validated. The prefix function avoids this by building the table in one forward sweep: every comparison is either consumed or falls back to a previously computed value.

The move

Build the KMP prefix function for s. The value at the last index, fail[n−1], is exactly the length of the longest proper prefix that is also a suffix. Return s[:fail[n−1]].

Invariant & cost

fail[j] always holds the correct longest-prefix-suffix length for s[0..j]; the pointer k can only fall back as many times as it advanced, so the table builds in O(n) time, O(n) space for the table.

The code

python

def longestPrefix(s: str) -> str:
    n = len(s)
    fail = [0] * n          # fail[j] = len of longest proper prefix-suffix for s[0..j]
    k = 0                   # length of current matching prefix
    for j in range(1, n):
        while k > 0 and s[k] != s[j]:
            k = fail[k - 1]   # fall back to the previous border
        if s[k] == s[j]:
            k += 1
        fail[j] = k
    return s[:fail[n - 1]]   # the last entry gives us the answer length
Narrate it to the interviewer

"I need the longest string that is simultaneously a proper prefix and a proper suffix. Brute force tries all prefix lengths and compares — O(n²). But I recognize this as exactly what the KMP failure function computes: for each position j, it stores the length of the longest prefix that also ends at j. So I build that table in one pass with the standard fallback loop, and the answer is simply the last value in the table. O(n) time and space, one pass over the string."

Dry-run: "aaaa" → fail = [0,1,2,3] → return "aaa" (length 3). Empty string → loop never runs, fail[-1] = fail[0] for a 1-char input means fail[0]=0, return "". Single character → n=1, loop doesn't execute, fail[0]=0, return "". All distinct → every fail value is 0, return "".

02 KMP phase 2

The match — a text pointer that never retreats

With fail[] in hand, the actual search is a single left-to-right scan of the text. Keep two indices: i walks the text (0 to n−1), and j tracks how many characters of the pattern have been matched so far. On each step, compare T[i] with P[j]:

  • Match: both characters agree. Advance j. If j == m, you found an occurrence at i − m + 1; to keep searching, set j = fail[j−1].
  • Mismatch with j > 0: don't touch i. Instead set j = fail[j−1] — this says "the first fail[j−1] characters of the pattern are already known to match, so pick up from there." Then re-examine T[i] against P[j].
  • Mismatch with j == 0: no partial match at all; just advance i.

The key insight is that i never moves backward. Each character in the text is compared at most twice — once during a potential match, once as part of the fallback that adjusts j — so the total work is O(n) comparisons over the text. The complete algorithm is O(n + m): O(m) to build the table, O(n) to scan.

KMP

  • Deterministically O(n + m) — no worst-case blowup.
  • O(m) extra space for the failure table.
  • Text pointer never backtracks — great for streaming input.
  • Works for any alphabet; no hash collisions to worry about.

Naive search

  • O(nm) worst case — brutal on repetitive inputs.
  • O(1) extra space.
  • Fine in practice for short patterns or random text.
  • Simpler to write from memory in a time-pressured interview.
Worked problemEasyLeetCode 28
Find the Index of the First Occurrence in a String

Given a text haystack and a needle, return the index of the first occurrence of needle inside haystack. Return −1 if it doesn't appear.

Think it through

Restate

Substring search — find the leftmost position in haystack where needle starts, or report absence.

Brute force

For each starting index in haystack, compare up to len(needle) characters. Correct, but O(n·m) in the worst case — hit when haystack is "aaaa…a" and needle is "aaa…ab".

The waste

On a mismatch after matching several characters, the brute force discards everything it learned. The prefix function captures the overlap structure of the needle so we can shift intelligently instead of resetting to zero.

The move

Build fail[] for the needle in O(m). Then scan the haystack with two pointers i (text) and j (pattern). A mismatch sets j = fail[j−1] without retreating i. A full match (j == m) returns i − m + 1.

Invariant & cost

j encodes how many prefix characters of the needle are confirmed at the current text position; since i never retreats, each text character is processed in amortized O(1): O(n + m) time, O(m) space.

The code

python

def strStr(haystack: str, needle: str) -> int:
    n, m = len(haystack), len(needle)
    if m == 0: return 0          # empty needle always found at 0

    # --- build the failure table for the needle ---
    fail = [0] * m
    k = 0
    for j in range(1, m):
        while k > 0 and needle[k] != needle[j]:
            k = fail[k - 1]
        if needle[k] == needle[j]:
            k += 1
        fail[j] = k

    # --- scan the haystack ---
    j = 0                            # characters of needle matched so far
    for i in range(n):
        while j > 0 and haystack[i] != needle[j]:
            j = fail[j - 1]          # slide pattern; don't touch i
        if haystack[i] == needle[j]:
            j += 1
        if j == m:
            return i - m + 1         # full match found
    return -1
Narrate it to the interviewer

"Restating: find the first occurrence of needle in haystack, return -1 if absent. Brute force tries all alignments — O(n·m), and the edge case where haystack is all-same characters destroys it. The KMP approach removes the backtracking. Phase one: build a failure table for the needle — for each position, what's the longest prefix of the needle that also ends there? That runs in O(m). Phase two: one left-to-right scan of the haystack. My invariant is that j tells me how many needle characters are already matched at the current position; on a mismatch I don't retreat i, I just slide j back using the table. Full match when j reaches the needle length. Total: O(n + m) time, O(m) space."

Dry-run: empty needle → return 0 immediately. Needle longer than haystack → never reaches j==m, return -1. Needle equals haystack → returns 0. haystack = "aaa", needle = "aab" → never matches, return -1.

03 Rabin-Karp

Rolling hash — update the window in O(1)

Rabin-Karp approaches substring search from a hashing angle. Instead of comparing characters, compute a numeric hash of the pattern and compare it against hashes of every m-length window in the text. If the hashes match, do a full character comparison to confirm (in case of a collision). If they differ, the window definitely does not match — no comparison needed.

The clever part is the rolling hash. Treat the window's characters as digits in a base-b number modulo a large prime MOD. When the window slides one step to the right, update the hash in O(1): subtract the contribution of the character leaving on the left, multiply by the base, and add the character entering on the right. No need to recompute the hash of all m characters from scratch.

Concretely, if the current window hash is h and the character sliding out is T[i] with precomputed power b^(m-1) mod MOD:

h_new = (h − T[i] · b^(m-1)) · b + T[i+m] (all mod MOD)

This gives O(n + m) expected time across all window positions. The risk is hash collisions — two different strings with the same hash — but with a large prime modulus the probability is tiny, and verifying on a hit keeps correctness guaranteed.

Why Rabin-Karp shines for multiple patterns

If you need to find k different patterns in the same text, KMP requires running k separate scans (or building an Aho-Corasick automaton). Rabin-Karp lets you store all pattern hashes in a set and check every window against the set in O(1) — a single O(n) pass handles all k patterns simultaneously. That multi-pattern trick is why Rabin-Karp still earns its place even when KMP is the safer single-pattern choice.

Worked problemMediumLeetCode 459
Repeated Substring Pattern

Given a non-empty string s, decide whether it can be formed by repeating some shorter substring two or more times. The repeated unit must divide the length of s evenly.

Think it through

Restate

Does some string p (with len(p) < len(s) and len(s) % len(p) == 0) satisfy p * (len(s)//len(p)) == s?

Brute force

Try every divisor d of len(s) from 1 to len(s)//2, extract the first d characters as the candidate, and check whether repeating it fills s. Each check is O(n), and there are up to O(√n) divisors, so total is O(n√n).

The waste

We can do this in a single pass using a KMP insight. If s is made of repeating unit of length d, then the KMP prefix function of s at the last position equals n − d. So if fail[n−1] > 0 and n % (n − fail[n−1]) == 0, the answer is yes.

The move (KMP trick)

Build the KMP failure table for s. Let last = fail[n−1]. The candidate period length is n − last. If that period divides n evenly, then s is periodic. Alternative (string rotation trick): s has a repeating unit iff s is a substring of s+s at a position other than 0 and n — find that with a single strStr call.

Invariant & cost

Building the failure table is O(n) time, O(n) space. The string rotation alternative also uses O(n) for the concatenation and O(n) for the search, same total.

The code

python

def repeatedSubstringPattern(s: str) -> bool:
    n = len(s)

    # Build the KMP failure table for s itself
    fail = [0] * n
    k = 0
    for j in range(1, n):
        while k > 0 and s[k] != s[j]:
            k = fail[k - 1]
        if s[k] == s[j]:
            k += 1
        fail[j] = k

    last = fail[n - 1]            # length of longest proper prefix-suffix
    period = n - last               # candidate repeating-unit length
    return last > 0 and n % period == 0   # period must divide n evenly
Narrate it to the interviewer

"I need to know whether the string is made of copies of a shorter unit. The brute-force approach tries every divisor of the length — O(n√n). I can do it in O(n) using the KMP failure table. The key observation: if s has repeating unit of length d, then the longest prefix of s that is also a suffix has length n−d. So I build the failure table, check the last value, call it 'last'. The candidate period length is n−last. If that divides n evenly and last > 0, the string is periodic. O(n) time and space."

Dry-run: "abab" → fail = [0,0,1,2], last=2, period=2, 4%2==0 → True. "aba" → fail = [0,0,1], last=1, period=2, 3%2≠0 → False. "abcabc" → fail=[0,0,0,1,2,3], last=3, period=3, 6%3==0 → True. Single character → last=0, return False immediately.

What the interviewer asks next

"Walk me through a hash collision in Rabin-Karp." → when two different strings hash to the same value you do a full O(m) character check to confirm or reject. In the worst case (adversarial input where every window collides) this degrades to O(nm). In practice, using a large prime modulus and a random base makes collisions astronomically unlikely. "Can you use double hashing?" → yes: maintain two hashes with independent bases and primes; a collision in both is essentially impossible. "When would you pick KMP over Rabin-Karp?" → KMP is deterministically O(n+m) with no collision risk; Rabin-Karp's advantage is multi-pattern search and its clean arithmetic structure when the alphabet is large.

04 Z-function

The Z-function and what it solves

The Z-function is a close cousin of the KMP prefix function. For a string s, Z[i] is the length of the longest substring starting at s[i] that matches a prefix of s. By definition, Z[0] is conventionally set to 0 (the whole string is not usually counted). For every other index, Z[i] tells you how far the current position "looks like the start of s."

Computing it: maintain a window [l, r] that is the rightmost Z-box (the rightmost known match of a prefix). For each new i:

  • If i < r, the position is inside the current Z-box. Use the previously computed value at the mirror position i − l as a lower bound: Z[i] = min(Z[i−l], r−i+1).
  • Then extend greedily while characters match, and update [l, r] if the new match extends further right.

This runs in O(n) because r only ever moves right.

Repeated Substring via Z

If a string s of length n is made of copies of unit of length d, then Z[d] == n − d (the suffix starting at position d is a prefix of length n − d). So iterate over divisors of n, compute the Z-function, and check.

Pattern Search via Z

Concatenate P + "$" + T (the sentinel "$" must not appear in either string) and compute the Z-function. Any position i in the T part where Z[i] == m marks an occurrence of P in T. This gives the same O(n + m) result as KMP with a slightly different implementation path.

KMP vs Z — pick your tool

Both compute essentially the same information in O(n), just encoded differently. The KMP failure table is a backward pointer: "if I fail here, resume there." The Z-array is a forward extent: "how long does the match starting here go?" Competitive programmers often prefer Z because its definition is more direct. In an interview, naming either and explaining why it solves the problem in O(n) is sufficient — examiners care more that you know a linear method exists than which one you implement.

05 Recognition

Spotting it, and the traps

The signals that point toward KMP or Rabin-Karp:

  • The problem asks for a substring occurrence or all occurrences and the naive O(nm) approach would time out.
  • The string has a strongly repetitive structure and you need to detect or exploit that ("is this made of repeating units?", "does this rotation equal the original?").
  • You need the longest prefix that is also a suffix — directly the last value in the failure table.
  • You need to find multiple different patterns in a single text — Rabin-Karp with a hash set of patterns, or Aho-Corasick for the full solution.
  • The problem involves a sliding window of fixed length where you are comparing content — rolling hash is the natural O(1) update mechanism.

The bugs that trip candidates at this level:

  • Off-by-one in the failure table: fail[j−1], not fail[j], is used when a mismatch occurs at pattern position j. Confusing the two is the single most common KMP bug.
  • Forgetting the modular arithmetic in Rabin-Karp: every hash operation must stay within the modulus, including the subtraction (Python handles negative modulo correctly, but C++ and Java do not — add MOD before taking mod).
  • Skipping the verification step in Rabin-Karp: a hash match is only probable, not guaranteed. Always do a full character-by-character check on a hash hit.
  • Computing fail[] on the text instead of the pattern: the failure table is a property of the pattern only. The text is never touched during table construction.
  • Not handling the empty needle: conventionally returns 0 (found at index 0). Most implementations miss this edge case.
Decision flow

Is it a fixed-length window comparison with possible hash collisions tolerated? → Rabin-Karp. Is it one or more patterns in a text where worst-case guarantees matter? → KMP (single pattern) or Aho-Corasick (multiple). Is the question about structure within a single string (longest prefix-suffix, periodicity, shortest palindrome)? → KMP failure table or Z-function directly on the string itself.

How to drill string algorithms

String problems reward one specific habit: build the failure table by hand before writing code. On paper, trace through the two-pointer build for a small example — write out each value of fail[j] and why it got that value. Once you can do that in 60 seconds without thinking, the code follows mechanically. Then code the full KMP from scratch (no notes), dry-run with the edge cases above, and aim for the easy tier in under 20 minutes. The rolling-hash for Rabin-Karp is worth memorizing as a formula: subtract the leaving character's contribution, multiply, add the entering character, take mod.

★ Review

Summary & self-test

The whole pattern compresses to this: naive substring match is O(nm) because every mismatch discards all previous work. KMP breaks that with a precomputed failure table that tells the pattern pointer where to land on a mismatch — so the text pointer never retreats and the full scan is O(n). Rabin-Karp breaks it by converting m-character comparisons into O(1) hash arithmetic, sliding the window in constant time. The failure table also solves structural string questions directly: longest prefix-suffix, periodicity, and palindrome extension are all single table lookups.
