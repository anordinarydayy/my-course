State-Machine & Interval DP

Two related ideas live under this heading: model a sequence of decisions as transitions between named states and step through it day by day, or think of the answer for a range [i, j] as the best split made at some final action inside that range. Both approaches turn exponential brute-force search into polynomial DP by caching the right subproblem.

00 The idea

What problem does it solve?

Consider the classic stock-trading setup: you are given a list of daily prices and you want to know the maximum profit you can earn by buying and selling shares. With no constraints the greedy answer is easy. Add a cooldown day after every sale and the problem mutates — a naive recursion fans out into exponentially many branches, re-computing the same (day, last-action) pairs over and over.

The pattern that kills the repetition is to name your positions explicitly. State-machine DP says: at every step you are in exactly one of a small set of states (e.g., holding, just sold, resting). The value of each state on day d depends only on a constant number of state values from the day before. So the whole recurrence resolves in a single left-to-right sweep with O(1) or O(k) extra memory.

Interval DP handles a different shape: questions about a contiguous sub-range [i, j] where the optimal answer depends on which action happens last inside that range, splitting it into two cheaper sub-ranges. You fill the DP table by increasing interval length, so shorter sub-problems are always ready when you need them.

The reusable shape

Both patterns work because there is a small, enumerable set of subproblems. In the state machine each (day, state) pair is computed once. In interval DP each (i, j) pair is computed once. Memoisation or bottom-up table-filling turns the overlapping recursion into a polynomial sweep.

01 State transitions

Stock state machine: hold, sold, rest

The brute-force approach to the cooldown stock problem tries every possible buy/sell sequence recursively. With n days there are O(2ⁿ) candidate sequences — far too slow.

The insight: on any given day there are exactly three positions you can be in.

  • hold — you own a share right now.
  • sold — you just sold today (this triggers the cooldown).
  • rest — you own nothing and are not in cooldown; free to buy tomorrow.

Write hold[d], sold[d], rest[d] for the best cash balance achievable while ending day d in each state. The transitions are:

  • hold[d] = max(hold[d-1], rest[d-1] - price[d]) — keep holding, or buy from rest.
  • sold[d] = hold[d-1] + price[d] — sell what you're holding.
  • rest[d] = max(rest[d-1], sold[d-1]) — stay idle, or emerge from cooldown.

The answer is max(sold[n-1], rest[n-1]) — you never want to end while holding. Because each state on day d depends only on day d-1, the whole table collapses to three running variables and the sweep runs in O(n) time, O(1) space.

State machine mental model

Draw a circle for each state and a directed edge for each valid transition. Then every edge becomes one line of the recurrence. If you can draw the diagram in an interview, you can write the recurrence from the picture — no memorisation required.

Worked problemMediumLeetCode 309
Best Time to Buy and Sell Stock with Cooldown

You have an integer array of daily stock prices. On any day you may buy one share or sell the share you hold, but after selling you must skip the very next day (cooldown) before buying again. Find the maximum total profit.

Think it through

Restate

Maximise cash when the sequence ends, given the buy-sell-cooldown constraint on the day sequence.

Brute force

Recurse over every subset of buy/sell days. Each day can be in or out of a transaction, giving roughly O(2ⁿ) calls with enormous overlap.

The waste

Subproblems repeat: the same (day, current-state) pair gets re-computed from many different call paths.

The move

Name the three positions: hold, sold, rest. Express each as a recurrence over the previous day's values. Iterate left to right, carrying only the three variables.

Invariant & cost

At every day d, hold[d] is the best profit if I own a share; sold[d] if I just sold; rest[d] if I own nothing and am free. O(n) time, O(1) space.

The code

python

def max_profit_cooldown(prices):
    if not prices:
        return 0
    # hold = best profit while owning a share
    # sold = best profit right after selling (triggers cooldown)
    # rest = best profit while free to act (no share, no cooldown)
    hold = -prices[0]   # buy on day 0
    sold = 0
    rest = 0
    for p in prices[1:]:
        hold, sold, rest = (
            max(hold, rest - p),   # keep holding OR buy from rest
            hold + p,              # sell what we hold today
            max(rest, sold),       # idle OR emerge from cooldown
        )
    return max(sold, rest)   # never end while holding
Narrate it to the interviewer

"The brute-force recursion has O(2ⁿ) branches with huge overlap. I'll model the three positions I can be in — holding, just-sold, resting — and write one recurrence per transition. Each day's values depend only on yesterday's, so I carry three variables through a single O(n) sweep."

Dry-run: empty list returns 0. Single price returns 0 (can't sell what you haven't bought). Prices [5,1,5,1,5]: hold=-5, then rest/sold update correctly; the peak-valley structure gives 8. Confirm: hold starts negative (spent money to buy).

02 Fee & transaction variants

Transaction fee and other stock variants

The cooldown model extends cleanly to other constraints because you only need to adjust which edges exist in the state diagram.

With a transaction fee (LeetCode 714)

There is no cooldown, so sold and rest collapse into a single free state (not holding). The fee is paid at the moment of selling. The two states become:

  • hold[d] = max(hold[d-1], free[d-1] - price[d])
  • free[d] = max(free[d-1], hold[d-1] + price[d] - fee)

Again O(n) time, O(1) space. The fee just shifts the sell transition by a constant.

Worked problemMediumLeetCode 714
Best Time to Buy and Sell Stock with Transaction Fee

Given daily prices and a fixed fee charged once per transaction at the time of sale, find the maximum total profit. You may hold at most one share at a time, and there is no cooldown.

Think it through

Restate

Maximise profit over unlimited buy-sell cycles where each sale costs a fixed fee.

Brute force

Try every subset of buy/sell pairs: O(2ⁿ) subsets, each taking O(n) to evaluate.

The waste

The same (day, holding-or-not) state is reachable via many paths but always has the same optimal value.

The move

Two-state machine: hold and free. Sweep prices left to right, updating both in constant time. No cooldown means sold immediately becomes free — merge the two into one variable.

Invariant & cost

free is always the best cash you can have without owning a share; hold is the best cash while owning one. O(n) time, O(1) space.

The code

python

def max_profit_fee(prices, fee):
    # hold: best profit while owning a share
    # free: best profit while not holding (can buy next)
    hold = -prices[0]
    free = 0
    for p in prices[1:]:
        hold, free = (
            max(hold, free - p),      # keep holding OR buy today
            max(free, hold + p - fee), # stay free OR sell (subtract fee)
        )
    return free   # best profit when not holding at the end
Narrate it to the interviewer

"Without cooldown I only need two states: hold and free. The only change from the basic unlimited-trade problem is that selling costs a fee, so I subtract it from the sell transition. This is O(n) time and O(1) space."

Dry-run: prices=[1,3,2,8,4,9], fee=2. hold=-1, free=0. Day 1 (p=3): hold=-1, free=max(0,-1+3-2)=0. Day 2 (p=2): hold=max(-1,0-2)=-1, free=0. Day 3 (p=8): free=max(0,-1+8-2)=5. Day 4 (p=4): hold=max(-1,5-4)=1. Day 5 (p=9): free=max(5,1+9-2)=8. Answer=8. Correct: buy at 1 sell at 8 (profit 5), buy at 4 sell at 9 (profit 3).

03 At most k transactions

Transaction-limited stock DP

When you can make at most k full buy-sell transactions (LeetCode 188), the two-variable trick must grow to track how many transactions remain. The brute force tries all O(n²ᵏ) ways to choose k buy-sell pairs — infeasible for large inputs.

The key observation: define dp[t][d] as the best profit using at most t transactions through day d. The recurrence is:

  • dp[t][d] = max(dp[t][d-1], max over j<d of (price[d] - price[j] + dp[t-1][j-1]))

A standard trick rewrites the inner maximum as a running best-buy variable best_buy_t = max(best_buy_t, dp[t-1][j-1] - price[j]), reducing the O(n) inner scan to O(1). Total complexity: O(k·n) time, O(k) space.

Special cases: if 2k ≥ n, there is no effective limit — every profitable day pair can be captured — so use the greedy O(n) solution instead.

Interviewer follow-up

Once you have the k-transaction solution, the interviewer may ask for k=1 (just track running min price) or k=infinity (greedy sum of positive differences). Show that the state-machine framework unifies all of them, and state which edge cases trigger the 2k≥n greedy shortcut.

04 Interval DP

Interval DP: pick the last action in [i, j]

For some problems, the cost of a range [i, j] depends on picking one element inside that range as the last to be processed, and then paying a cost that depends on what surrounds the remaining pieces. The key structural insight is: if k is the last element processed in [i, j], then the subproblems [i, k-1] and [k+1, j] were fully processed before k — so we know their costs from the table.

The brute-force recursion considers all n! orderings of elements: O(n!). The DP collapses this because every subrange [i,j] is computed exactly once. Filling by increasing length (all length-1 ranges first, then length-2, …) guarantees every needed sub-result is already in the table.

  • Enumerate interval lengths from 2 to n.
  • For each (i, j) pair of that length, try every split point k in [i, j].
  • Take the minimum (or maximum) over all choices of k.

The template is O(n³) time and O(n²) space — one entry per (i,j) pair, one k loop per entry.

The reusable shape

The hallmark of interval DP is a recurrence of the form dp[i][j] = min/max over k in [i,j] of f(dp[i][k-1], dp[k+1][j], cost(i,j,k)). The "last balloon to burst" framing in Burst Balloons is perhaps the clearest example of why you pick the last rather than the first: if k is last, the boundaries i-1 and j+1 are still present when k is processed, so the cost is exactly nums[i-1]*nums[k]*nums[j+1].

Worked problemHardLeetCode 312
Burst Balloons

You have a row of balloons each holding a positive integer. Bursting balloon k earns coins equal to the product of its value and its two current neighbours (boundary neighbours are treated as 1). Choose a bursting order to maximise total coins.

Think it through

Restate

Find the bursting order of n balloons that maximises the total coins earned, where each burst earns product of the balloon and its live neighbours at that moment.

Brute force

Try every permutation of burst order: O(n!) orderings, each O(n) to score.

The waste

The same sub-range of remaining balloons can be reached via many different orderings of bursts outside it — yet the optimal value for that sub-range is always the same regardless of how we got there.

The move

Pad the array with sentinel 1s at both ends. Define dp[i][j] as the max coins from bursting every balloon strictly between padded indices i and j (balloons i and j themselves are already gone or are sentinels). Pick k as the last balloon to burst in [i+1..j-1]. When k is burst last, its neighbours are exactly nums[i] and nums[j], giving a clean cost: nums[i]*nums[k]*nums[j] + dp[i][k] + dp[k][j]. Fill by increasing interval length.

Invariant & cost

dp[i][j] is always the optimal coins for the open interval (i,j), with i and j acting as fixed boundaries. O(n³) time, O(n²) space.

The code

python

def max_coins(nums):
    # pad with sentinel 1s on both ends
    a = [1] + nums + [1]
    n = len(a)
    dp = [[0] * n for _ in range(n)]

    # fill by increasing interval length
    for length in range(2, n):          # length of open interval (i,j)
        for i in range(0, n - length):
            j = i + length
            for k in range(i + 1, j):   # k is LAST burst in (i,j)
                coins = a[i] * a[k] * a[j]
                dp[i][j] = max(dp[i][j],
                               coins + dp[i][k] + dp[k][j])
    return dp[0][n - 1]
Narrate it to the interviewer

"My first instinct is to try every burst order — but that is O(n!) which is hopeless. The key reframe is to think about which balloon bursts last in a range. If k is last, its neighbours are exactly the boundaries i and j, so the cost for k is nums[i]*nums[k]*nums[j] plus the optimal sub-ranges on each side. I pad with sentinel 1s, define dp[i][j] as max coins for the open interval, fill by length, and read dp[0][n-1]. That is O(n³) time, O(n²) space."

Dry-run: nums=[3,1,5,8]. Padded a=[1,3,1,5,8,1]. dp[0][2]: k=1, coins=1*3*1=3; dp[0][2]=3. dp[1][3]: k=2, coins=3*1*5=15; dp[1][3]=15. dp[2][4]: k=3, coins=1*5*8=40; dp[2][4]=40. dp[0][3]: k=1: 1*3*5+0+15=30; k=2: 1*1*5+3+40=48; dp[0][3]=48. Continue to dp[0][5]=167. (Matches expected output.)

05 Table fill order

Interval DP table filling by length

The mechanics of filling by interval length are easy to get wrong under pressure. The table is upper-triangular: row i, column j where j ≥ i. The diagonal (length 0 or 1) holds base cases. Then you sweep length 2, length 3, …, up to length n. For each length you sweep all valid (i,j) pairs with j = i + length.

This fill order ensures that when you compute dp[i][j] you look up dp[i][k] and dp[k][j] — both strictly smaller intervals — which are already filled from an earlier pass. Getting the loop order backwards (sweeping by i then j) silently reads uncomputed cells; this is one of the most common interval DP bugs.

Matrix-chain multiplication — same template

Matrix-chain multiplication (LeetCode 1039 is a related variant; the classic textbook problem is CLRS chapter 14) uses exactly the same fill order. dp[i][j] = minimum multiplications for matrices i through j. The "last split" is the matrix boundary where the final multiplication happens. Recognising both Burst Balloons and matrix chain as instances of the same template saves significant thinking time in an interview.

06 Practice

Practice ladder

EasyBest Time to Buy and Sell Stockstate machine · 1 tx

Given a price list, find the single buy-sell pair that maximises profit (at most one transaction).

Idea Track running min price seen so far; profit = price - min. O(n) time, O(1) space.

EasyBest Time to Buy and Sell Stock IIgreedy · unlimited tx

Same setup but you may make as many transactions as you like (no cooldown, no fee). Maximise total profit.

Idea Sum every positive day-to-day price difference. O(n) time. This is also the 2k≥n special case of the k-transaction DP.

MediumBest Time to Buy and Sell Stock with Cooldown3-state machine

Cooldown of one day after every sale. Maximise total profit.

Idea Three states (hold / sold / rest), single left-to-right sweep. O(n) time, O(1) space.

MediumBest Time to Buy and Sell Stock with Transaction Fee2-state machine

Fixed fee subtracted whenever you sell. Maximise total profit over unlimited transactions.

Idea Two states (hold / free); subtract fee in the sell transition. O(n) time, O(1) space.

MediumBest Time to Buy and Sell Stock IIIk=2 transactions

At most two complete buy-sell transactions. Maximise total profit.

Idea Run the k-transaction DP with k=2; or explicitly track four running variables (buy1/sell1/buy2/sell2). O(n) time, O(1) space.

MediumMinimum Cost Tree from Leaf Valuesinterval DP · min product

Given leaf values, build a binary tree so the sum of each non-leaf node (product of max leaves in its left and right subtrees) is minimised.

Idea Interval DP: dp[i][j] = min cost tree from leaves i..j; split at last merge point k. O(n³) time, O(n²) space.

MediumStrange Printerinterval DP · string

A printer can print a contiguous sequence of the same character per turn and each new turn overwrites what is already there. Find the minimum turns to print a given string.

Idea Interval DP on the string: if s[i]==s[j], dp[i][j] = dp[i][j-1]; otherwise min over all splits k. O(n³) time.

HardBest Time to Buy and Sell Stock IVk transactions

At most k complete transactions. Maximise total profit.

Idea If 2k≥n use greedy. Else dp[t][d] = max profit with at most t transactions through day d; maintain best-buy variable to keep each inner loop O(1). O(k·n) time.

HardBurst Balloonsinterval DP · last burst

Given a row of numbered balloons, bursting balloon k (with current live neighbours) earns their product. Choose the order to maximise total coins.

Idea Reframe as "last to burst" in each interval; pad with sentinel 1s; fill dp by length. O(n³) time, O(n²) space.

HardRemove Boxesinterval DP · 3D

Remove groups of identical boxes in any order; removing k consecutive identical boxes scores k² points. Find the maximum points.

Idea 3D interval DP: dp[i][j][k] = max points from boxes i..j with k extra copies of boxes[i] attached on the left. O(n⁴) time — the hardest in this family.

HardZuma Gameinterval DP · insertion

A row of coloured balls; insert balls to form groups of 3 or more of the same colour to clear them. Find the minimum insertions to clear the board.

Idea Interval DP with memoisation; track current run length. O(n⁴) in the worst case. Related to Remove Boxes in spirit.

★ Review

Summary & self-test

State-machine DP and interval DP both tame exponential recursion by caching a small, well-defined set of subproblems. The state-machine flavour sweeps left to right with a constant number of named positions; each position's value on day d is a max/min over a tiny set of day-d-1 values. Interval DP fills an upper-triangular table by increasing interval length, with the crucial framing of picking the last action inside each range rather than the first.

02O(1) per dayStock state-machine DP needs only three (or two) running variables; no array required.
03Sell never endsYou never want to end while holding; the answer is max(sold, rest) at the last day.
04Adjust edges for variantsCooldown, fee, and k-limit each just change which transitions exist — same template.
05Last action framingInterval DP works by choosing which action happens last in [i,j]; boundaries are known then.
06Fill by lengthSweep lengths 2, 3, …, n. Wrong loop order (by i then j) silently reads uncomputed cells.
07O(n³) / O(n²)Standard interval DP: n² subproblems each needing O(n) split choices.
08Sentinel paddingBurst Balloons needs 1-sentinels at both ends so boundary balloons have defined neighbours.
