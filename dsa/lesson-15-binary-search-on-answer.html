Binary Search on the Answer

When every candidate answer above a threshold is feasible and every one below it is not, the answer space itself becomes a sorted boolean sequence — and you can binary-search it to find the exact boundary in O(log R) feasibility checks instead of trying every value.

00 The idea

What problem does it solve?

Some problems ask you to find a number — a minimum speed, a maximum load, a smallest group size — rather than a specific element. The brute force is to try every candidate value one by one. If the range of candidates is huge (say, 1 to 10⁹), that is too slow.

The trick exploits a hidden monotonicity. Define a predicate feasible(x): "is the value x good enough to satisfy the constraint?" In many problems this predicate has a clean structure: there is some threshold x* such that feasible(x) is false for all x < x* and true for all x ≥ x*. The answer space looks like this:

Because the predicate is monotone, we can binary-search the boundary in O(log(hi − lo)) iterations. Each iteration calls feasible(mid) once. If the predicate takes O(n) to evaluate, the total cost is O(n log R) where R is the size of the value range — often far cheaper than the naïve O(n·R) scan.

Two flavours

Minimise the maximum — find the smallest x such that feasible(x) is true. Template: start with lo = smallest_possible, hi = largest_possible, and search for the leftmost true. When feasible(mid) is true, record mid as a candidate and try smaller: hi = mid - 1. When false, lo = mid + 1. Result: lo after the loop. Maximise the minimum is the mirror: the predicate flips in the other direction.

01 The predicate

The feasibility predicate & the monotonic cue ▸

Before writing a single line of binary search, you need two things: a lower bound and an upper bound for the answer, and a feasible(x) function that is monotone over that range. The cue that tells you the answer is a value rather than an array index is usually the phrase "minimum possible maximum" or "maximum possible minimum" — or a constraint like "all tasks must finish within D days."

To check feasibility you almost always do a greedy simulation: simulate what happens if you fix the answer at x and see whether the constraint is met. For example, "can a monkey eat all bananas in h hours at speed x?" → sum up the ceiling-division of each pile by x and check if the total ≤ h. That simulation is linear in the input size, so each feasibility check costs O(n).

The standard loop template: while lo <= hi: mid = (lo + hi) // 2; if feasible(mid): ans = mid; hi = mid - 1; else: lo = mid + 1. After the loop, ans holds the smallest feasible value. Notice the loop always terminates: each iteration either raises lo or lowers hi.

Setting lo and hi

Choosing the right bounds is half the problem. lo should be the smallest value that could possibly be the answer; hi should be the largest value that could possibly be needed. For speed/capacity problems: lo = 1 (or the largest single element, since the answer cannot be smaller); hi = sum(array) (one trip, ship everything). Tight bounds mean fewer iterations — but correctness matters more than tightness.

Worked problemMediumLeetCode 875
Koko Eating Bananas

A monkey sits by n piles of bananas. Each hour she eats at most k bananas from one pile (any leftover in that pile must wait for a future hour). The guards return after exactly h hours. Find the minimum integer eating speed k such that she can finish all the piles before the guards arrive.

Think it through

Restate

Given pile sizes and a time budget h, find the smallest integer k ≥ 1 such that summing ceil(pile / k) over all piles gives a total ≤ h.

Brute force

Try every speed from 1 upward until the first one that works. For each speed the check costs O(n). Worst case, the answer is max(piles) ≈ 10⁹, giving O(n · max) — far too slow.

The waste

Once we find a speed that works, every faster speed also works. Once a speed fails, every slower speed also fails. The answer space is monotone — exactly the binary-search-on-answer shape.

The move

Binary-search over k from 1 to max(piles). For a given mid, compute hours = sum(ceil(p / mid) for p in piles). If hours <= h, the speed works — record it and try slower (hi = mid - 1). If not, try faster (lo = mid + 1).

Invariant & cost

At every iteration the optimal speed lies in [lo, hi]. The loop runs at most log(max(piles)) ≈ 30 times; each iteration costs O(n): O(n log M) time, O(1) space where M = max pile size.

The code

python

import math

def minEatingSpeed(piles, h):
    lo, hi = 1, max(piles)       # speed range: 1 banana/hr to clear a whole pile in 1 hr
    ans = hi                       # fallback: eat the largest pile in one hour
    while lo <= hi:
        mid = (lo + hi) // 2
        hours = sum(math.ceil(p / mid) for p in piles)
        if hours <= h:             # feasible — record and try slower
            ans = mid
            hi = mid - 1
        else:                      # too slow — must go faster
            lo = mid + 1
    return ans
Narrate it to the interviewer

"Restating the problem: I want the smallest integer speed such that eating every pile at that rate fits within h hours. Trying each speed one by one is O(n · max) — too slow. But higher speeds are always at least as good as lower ones, so the feasibility function is monotone over the speed range. That's the binary-search-on-answer shape. I set lo to 1 and hi to max(piles) — the slowest conceivable speed that still finishes in one hour per pile. At each mid I compute total hours by summing ceil(pile/mid) across all piles; if that fits in h hours I record mid as a candidate and try to go slower, otherwise I need to go faster. My invariant is that the answer always lies in [lo, hi]. After the loop, ans is the smallest feasible speed. Time: O(n log M) where M is the largest pile; space O(1)."

Dry-run aloud: piles = [1], h = 1 → lo = hi = 1, answer is 1. piles = [3,6,7,11], h = 8 → answer is 4. Verify: ceil(3/4)+ceil(6/4)+ceil(7/4)+ceil(11/4) = 1+2+2+3 = 8 ≤ 8. Check that when all piles are equal the loop still terminates correctly.

What the interviewer asks next

"What if h < n?" → Impossible to finish: she can't even visit all piles. The constraint usually guarantees h ≥ n, but name this edge case. "Can the speed exceed the max pile?" → No — eating more than the pile in a single hour just wastes time. Setting hi = max(piles) is tight and correct. "Why lo = 1 and not 0?" → Speed 0 is undefined (infinite hours). Always choose lo as the smallest valid candidate.

02 Minimise the maximum

Seeing the bisection on a real input ▸

The pattern "minimise the maximum" appears whenever a single actor (a worker, a ship, a packer) must handle items and you want the smallest capacity that finishes the job in a fixed number of rounds. The feasibility check is always a greedy simulation: given the fixed capacity, greedily assign items until a round overflows, start a new round, and count rounds.

Below, we trace the bisection on the array [5, 8, 3, 7, 4, 6] searching for the minimum capacity that ships all packages in ≤ 3 trips. The stepper highlights which cells contribute to the current trip and shows whether mid passes or fails.

Feasibility check (greedy)

  • Walk items left to right; keep a running load.
  • If adding the next item exceeds capacity, start a new trip.
  • Count trips at the end; compare against the budget.
  • O(n) per check — simple and correct.

Binary search outer loop

  • lo = max(items) — smallest single-trip capacity needed.
  • hi = sum(items) — one trip handles everything.
  • O(log(sum − max)) iterations, each O(n).
  • Total: O(n log S) where S = sum of items.
Worked problemMediumLeetCode 1011
Capacity To Ship Packages Within D Days

You have a list of packages with given weights, arranged in a fixed order. A conveyor belt ships them in that order: each day it loads packages until adding the next one would exceed its capacity, then it stops. Find the minimum integer capacity that ships all packages within exactly d days.

Think it through

Restate

Given weights and a day budget d, find the smallest capacity c such that a greedy packing (fill a day's load, start new day when the next package would overflow) uses ≤ d days.

Brute force

Try every capacity from max(weights) upward, simulate the packing for each, return the first that works. O(n · S) total where S is the sum of all weights — could be up to 5 × 10⁸. Too slow.

The waste

If a capacity of c works, then c+1 also works (a larger ship can only do better). If c fails, so does c-1. The space is monotone — the leftmost true is the answer.

The move

Binary-search from lo = max(weights) (must fit the heaviest single package) to hi = sum(weights) (one day, everything fits). For each mid, greedily simulate: accumulate a day's load; if adding the next weight exceeds mid, start a new day. Count days and compare to d.

Invariant & cost

At every step the minimum feasible capacity lies in [lo, hi]. The loop runs O(log S) times; each simulation costs O(n): O(n log S) time, O(1) space.

The code

python

def shipWithinDays(weights, d):
    lo, hi = max(weights), sum(weights)
    ans = hi                             # safe upper bound
    while lo <= hi:
        mid = (lo + hi) // 2
        days, load = 1, 0
        for w in weights:
            if load + w > mid:           # overflow: start a new day
                days += 1
                load = 0
            load += w
        if days <= d:                    # feasible — try smaller capacity
            ans = mid
            hi = mid - 1
        else:                            # need more capacity
            lo = mid + 1
    return ans
Narrate it to the interviewer

"Restating it: packages come in a fixed order and I need the smallest capacity that ships them in d days using a greedy fill-and-start-new-day rule. Scanning every capacity is O(n · S) — too slow given the constraints. The key insight is that capacity is monotone: if c works, c+1 works too. So I binary-search between max(weights) — the smallest possibly valid capacity — and sum(weights) — the trivial one-day answer. For each midpoint I simulate the packing in O(n): walk the weights, accumulate a running load, start a new day when the next package would overflow. If I finish in ≤ d days I record that mid and try smaller; otherwise I need a bigger capacity. The invariant is that the optimal answer is always in [lo, hi]. Total cost O(n log S), constant space."

Dry-run aloud: a single package (lo = hi = that weight, one day). All packages identical weight w and d = n (one per day) → answer = w. A package heavier than sum/d — note lo starts at max(weights), so this is automatically handled. Verify the greedy starts with days = 1 (not 0) so a single day is still counted.

03 The greedy simulation

Why the greedy check is always correct

Both Koko and the ship problem use the same inner feasibility check: a greedy scan through the items. It is worth pausing to understand why that greedy is optimal. Consider shipping packages with capacity c. The greedy rule — keep adding to the current day until the next package would overflow, then start a new day — always uses the minimum possible number of days for the given capacity.

Proof sketch: at each decision point the greedy packs as much as possible into the current day. Any other strategy that starts a new day earlier must also start a new day for everything that follows, using at least as many days total. This is the classic "greedy stays ahead" argument. The greedy is tight, so counting its trips gives an exact answer for each candidate capacity.

This is a recurring theme in binary-search-on-answer problems: the feasibility check is always a greedy simulation, and that greedy is provably optimal by a "stays-ahead" argument. Your job in an interview is to state this argument briefly: "The greedy packs as much as possible each time, so any other packing for the same capacity uses at least as many trips."

Worked problemMediumLeetCode 1011
Capacity To Ship Packages — edge-case drill

The same ship problem, but focus on the boundary setting: why is lo = max(weights) the right lower bound, and what breaks if you set lo = 1?

Think it through

Restate

We need the smallest integer capacity that ships all weights in order within d days.

Brute force

Start from lo = 1 and scan upward. Correct but O(n · S) time — slow. Also wastes iterations on obviously infeasible capacities below max(weights).

The waste

Any capacity smaller than the heaviest package is immediately infeasible — the ship cannot even carry that package alone. So lo = 1 forces the binary search to spend log(max(weights)) extra iterations ruling out a known-infeasible region. Set lo = max(weights) instead.

The move

lo = max(weights) is tight: it is the smallest possible valid capacity. hi = sum(weights) is tight on the other side: with that capacity the ship carries everything in one day. Binary-search this range.

Invariant & cost

The loop invariant holds: the answer is in [lo, hi] at all times. With tight bounds, the loop runs at most log(sum − max) ≈ 30 times. O(n log S) time, O(1) space.

The code

python

def shipWithinDays(weights, d):
    # lo: must fit the single heaviest package
    # hi: one-day total — trivially feasible
    lo, hi = max(weights), sum(weights)
    while lo < hi:              # loop ends when lo == hi (the answer)
        mid = (lo + hi) // 2
        days, load = 1, 0
        for w in weights:
            if load + w > mid:
                days += 1; load = 0
            load += w
        if days <= d:
            hi = mid              # mid is feasible; push hi down (keep mid as candidate)
        else:
            lo = mid + 1         # mid is infeasible; lo must exceed it
    return lo                   # lo == hi == answer
Narrate it to the interviewer

"This is the same ship problem, but I want to highlight the bound-setting and the loop variant. Setting lo = 1 is wasteful — any capacity below max(weights) cannot carry the heaviest package, so it is provably infeasible. I set lo = max(weights) for a tight lower bound, and hi = sum(weights) for the trivially feasible upper bound. I use the while lo < hi variant: when mid is feasible I set hi = mid (keeping mid as a candidate); when infeasible I set lo = mid + 1. When the loop exits, lo equals hi, which is the answer. This variant avoids maintaining a separate ans variable and is a clean alternative to the while lo <= hi form."

Dry-run aloud: weights = [5], d = 1 → lo = hi = 5, loop never executes, returns 5. weights = [1,1], d = 2 → lo = 1, hi = 2, mid = 1, check: 2 days = d, hi = 1, lo = hi = 1, return 1.

04 Split into k parts

Minimising the largest group sum ▸

The hardest variant in this family asks you to split an array into exactly k contiguous subarrays and minimise the largest subarray sum. The direct approach — dynamic programming — runs in O(n²k) time and uses O(nk) space. Binary-search-on-answer knocks this down to O(n log S).

The insight: if we could ask "is it possible to split the array into at most k parts where no part exceeds a given budget x?", that is a greedy check — pack elements greedily into a part; when the next element would push the sum over x, start a new part; count parts. If the count ≤ k, the answer is yes. And this predicate is monotone in x: bigger budget → easier to satisfy. So we binary-search x from max(array) to sum(array).

Worked problemHardLeetCode 410
Split Array Largest Sum

Given an array of non-negative integers, split it into exactly k non-empty contiguous subarrays. Find the minimum possible value of the largest subarray sum across all valid splits.

Think it through

Restate

Partition the array into k consecutive pieces; minimise the maximum piece-sum. Equivalently, find the smallest budget x such that the array can be divided into at most k pieces each with sum ≤ x.

Brute force

Try every possible set of k−1 split points (C(n−1, k−1) choices); compute the max piece-sum for each; return the minimum. O(n^k) time — completely intractable for n = 1000, k = 50.

The waste

If a budget of x allows a valid k-way split, then x+1 also allows it. If x fails, x−1 fails too. The feasibility is monotone over x — binary-search territory. The DP approach is O(n²k) and works, but the binary-search approach is simpler to implement and faster.

The move

Binary-search x over [max(array), sum(array)]. For each x, greedily scan: keep a running part-sum; when adding the next element would exceed x, start a new part and count it. If parts ≤ k at the end, x is feasible. Record x and try smaller (hi = mid − 1). Else try larger (lo = mid + 1).

Invariant & cost

The answer always lies in [lo, hi]. Each feasibility check is O(n); the loop runs O(log S) times: O(n log S) time, O(1) space. The DP alternative is O(n²k) — for large k the binary-search approach wins decisively.

The code

python

def splitArray(nums, k):
    def feasible(budget):
        parts, cur = 1, 0
        for x in nums:
            if cur + x > budget:   # overflow: start a new part
                parts += 1
                cur = 0
            cur += x
        return parts <= k

    lo, hi = max(nums), sum(nums)
    ans = hi
    while lo <= hi:
        mid = (lo + hi) // 2
        if feasible(mid):          # valid split exists — try smaller budget
            ans = mid
            hi = mid - 1
        else:                      # budget too tight — need more room
            lo = mid + 1
    return ans
Narrate it to the interviewer

"Restating it: I want the minimum possible largest-piece sum when splitting into k parts. The brute-force enumeration of all split points is exponential; the DP is O(n²k). But the feasibility — can I split into ≤ k parts each summing to at most x? — is monotone in x: higher budget makes it easier. So I binary-search x from max(nums) to sum(nums). For each midpoint I run a greedy O(n) check: scan left to right, accumulate a running sum, start a new part whenever adding the next element would overflow the budget, and count parts. If parts ≤ k, I record this mid and try a tighter budget; otherwise I need more room. Invariant: the answer is always in [lo, hi]. Total cost O(n log S), O(1) space."

Dry-run aloud: nums = [1], k = 1 → lo = hi = 1, loop runs once, ans = 1. nums = [7,2,5,10,8], k = 2: lo = 10, hi = 32; mid = 21 → feasible([7+2+5+10=24 no, 7+2+5=14 yes then 10+8=18 → 2 parts ≤ 2]) hmm, let me check mid=21: 7+2+5=14, 14+10=24 > 21 so new part, 10+8=18 ≤ 21: parts = 2 ≤ 2 → feasible, ans = 21, hi = 20; mid = 15 → 7+2+5=14, 14+10=24 > 15, parts = 2; 10+8=18 > 15, parts = 3 > 2 → infeasible, lo = 16; mid = 18 → 7+2+5=14, 14+10=24 > 18, parts = 2; 10+8=18 ≤ 18 → parts = 2 ≤ 2, ans = 18, hi = 17; lo=16 > hi=17 → return 18.

What the interviewer asks next

"Why not just use DP?" → The DP is O(n²k) and for k near n that's O(n³) — much worse than O(n log S). For large inputs, binary search wins. Also, the binary-search solution is about 15 lines versus a double nested loop with a min-over-split-point inner loop. "What if k = 1?" → lo = hi = sum(nums), loop exits immediately, return that value. "What if k = n?" → Each element is its own part; the largest piece sum is max(nums); binary search still converges to the right answer since lo starts there. Both edge cases fall out of the same code — no special-casing needed.

05 Recognition

Spotting it, and the traps

The pattern recognition question is: "Is the thing I'm searching for a value, and is the feasibility function monotone?" Train your ear for these signals:

  • The problem says "minimum possible maximum" or "maximum possible minimum" — these exact phrases are almost a direct tell.
  • You need to find a threshold — a speed, capacity, radius, time — that satisfies a constraint.
  • The constraint involves distributing items into groups subject to a budget: workers, days, ships, servers.
  • The answer lies in a large integer range (1 to 10⁹) and you cannot enumerate it.
  • You can write a simple greedy simulation that checks whether a given value works.

And the mistakes that cost candidates the round:

  • Wrong bounds. Setting lo too low wastes iterations or misses the answer. Setting hi too low (e.g., max instead of sum for the ship problem) causes incorrect results. Justify both bounds out loud.
  • Off-by-one in the loop. The while lo <= hi form with ans tracking, and the while lo < hi form that converges to lo == hi, are both correct — but mixing their update rules produces infinite loops. Pick one form and be consistent.
  • A non-monotone predicate. If the feasibility function is not monotone (some capacities work, then a range fails, then works again), binary search gives a wrong answer. Always verify monotonicity before coding.
  • Integer overflow. In languages with fixed-width integers, (lo + hi) // 2 can overflow. Use lo + (hi - lo) // 2 in C/Java. Python has arbitrary-precision integers, so this is not a concern there.
  • Forgetting ceil in the predicate. For problems like Koko, dividing pile sizes by speed needs ceiling division: ceil(pile / speed). In Python, -(-pile // speed) or math.ceil(pile / speed). Using floor division underestimates the hours needed.
The decision in one line

If you can write a function feasible(x) that is false for small x and true for large x (or vice versa), and the answer is the exact boundary — binary-search the answer space. If the array itself is sorted and you want an index — plain binary search on the array. If the answer space is not monotone — rethink your feasibility definition.

How to drill

For every problem, say aloud before coding:
(1) what is the answer range — what are the tightest valid lo and hi?
(2) What does feasible(x) check — state it as a predicate, then verify it is monotone by reasoning about what happens when x increases.
(3) Is the goal to find the leftmost true or the rightmost false?
(4) Write the feasibility check first as a standalone function, test it on a small example, then wrap the binary-search loop around it.
(5) Dry-run the edge cases: lo = hi at the start (answer is forced), k = 1 or k = n for split problems, empty or single-element arrays.

★ Review

Summary & self-test

The whole pattern compresses to one rule: when the answer is a value and the feasibility function is monotone, binary-search the value space. The feasibility check is almost always a greedy simulation that costs O(n); the binary search halves the range in O(log R) iterations. The result — O(n log R) — is usually the fastest possible for these problems and dramatically beats either a linear scan or a dynamic-programming approach.
