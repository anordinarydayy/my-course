Sorting & Custom Comparators

A surprisingly large class of problems that look complex at first become straightforward the moment you sort the input — or sort it in an unusual order defined by a custom comparator. Recognizing that the bottleneck is disorder, and knowing how to eliminate it with O(n log n) work, is one of the most broadly useful moves in a coding interview.

00 The idea

When sorting unlocks a problem

Many problems hide a crucial insight: two elements that need to be compared or grouped are spread arbitrarily across the input. The brute force compares every pair — O(n²) work. But once the data is sorted, the relevant neighbors are already adjacent, and a single linear scan does the rest.

The cost of this unlock is always O(n log n) for the sort. That is the price you pay once. Everything after — scanning, grouping, merging, deduplicating — runs in O(n). The final complexity is O(n log n) dominated by the sort, which is acceptable for most problem sizes.

Sorting unlocks a problem when at least one of these signals is present:

  • You need to group equal or equivalent elements together (sort brings them adjacent).
  • You need to merge overlapping ranges (sort by start; then scan once).
  • You need to find the closest pair, a missing element, or detect a duplicate (sorted neighbors reveal gaps and repeats).
  • You need a greedy decision that only works if elements are processed in a particular order.
  • You need to arrange elements to maximize or minimize a combined value (sort defines which order maximizes it).
The O(n log n) price is almost always worth it

In most interview constraints (n ≤ 10⁵ or 10⁶), O(n log n) is fully acceptable. The moment you see that a sorted order would make the rest trivial, commit to the sort. State it explicitly: "I'll pay O(n log n) to sort, then the rest is linear." That sentence alone signals pattern fluency.

01 Keys & multi-criteria

Sorting by a key and breaking ties

Python's sorted() and list .sort() accept a key= function. Whatever that function returns becomes the value used for comparison — the original elements are never directly compared. This is the cleanest way to transform sorting into a form the problem needs.

The brute force of most grouping problems is to scan every element for every other element — O(n²). Using a key function, you pay O(n log n) once and get the entire structure for free.

The key function can return a tuple to sort by multiple criteria. Python compares tuples lexicographically: the first element decides unless there is a tie, then the second decides, and so on. This gives you total control over priority without any extra logic.

The reusable shape

Any time you need a custom priority, reach for a key tuple: sorted(items, key=lambda x: (primary(x), secondary(x))). Negate numeric fields to reverse their order (-freq sorts highest first). String fields sort alphabetically as-is. This pattern handles the vast majority of multi-criteria problems without writing a comparator function.

Worked problemEasyLeetCode 905
Sort Array By Parity

Given an array of non-negative integers, rearrange it so that all even numbers come before all odd numbers. Any valid ordering within each group is acceptable; return the result.

Think it through

Restate

Partition the array so every even integer precedes every odd integer. Relative order within each group does not matter.

Brute force

Scan once to collect evens, scan again to collect odds, concatenate — O(n) time but O(n) space for the two extra lists.

The waste

The two-list approach allocates memory proportional to n. A key-based sort sidesteps the allocation. Even better: an in-place two-pointer approach achieves O(1) extra space, but sorting with a key is the clearest statement of intent and is accepted in interviews.

The move

Sort using key=lambda x: x % 2. For even numbers x % 2 == 0, for odd numbers x % 2 == 1. Sorting in ascending key order puts all evens (key 0) before all odds (key 1) — exactly the required arrangement.

Invariant & cost

After the sort, every element with key 0 precedes every element with key 1. Within each key group, Python's stable sort preserves the original relative order. O(n log n) time, O(n) space for the output array (or O(1) if sorted in place is acceptable).

The code

python

def sortArrayByParity(nums):
    # key=0 for even, key=1 for odd; evens sort first
    return sorted(nums, key=lambda x: x % 2)

# In-place O(1)-space alternative: two-pointer partition
def sortArrayByParityInPlace(nums):
    lo, hi = 0, len(nums) - 1
    while lo < hi:
        if nums[lo] % 2 == 0:   # lo is even — already in place
            lo += 1
        else:                   # lo is odd — swap with hi
            nums[lo], nums[hi] = nums[hi], nums[lo]
            hi -= 1
    return nums
Narrate it to the interviewer

"The ask is to move all evens before all odds. The brute force allocates two lists — linear space. The key-sort version uses x % 2 as the key: evens get key 0, odds get key 1, so ascending sort puts the groups in the right order — O(n log n) time. If they want O(1) space, I switch to a two-pointer partition: lo at the front, hi at the back. If lo is even it is already in the right zone and I advance lo. If lo is odd I swap it with hi and shrink hi. The invariant is that everything before lo is even and everything after hi is odd. One pass, O(n) time, O(1) space."

Dry-run aloud: all evens (nothing to swap), all odds (every lo gets swapped immediately), and a single-element array.

What the interviewer asks next

"What if relative order within each group must be preserved?" → The two-pointer approach does not preserve order; use the stable key-sort or a stable partition. "What about sorting by multiple properties — say, frequency then alphabetical?" → Return a tuple from the key function; Python compares tuples element by element. "Can you do it in O(n) time?" → Only if the domain is small enough for counting sort or a pigeonhole argument (see Section 04).

02 Custom comparator

Defining order with a comparator

Some problems require a sort order that cannot be expressed as a simple key function — the order depends on a relationship between two elements, not on a property of each element alone. For these, you write a comparator: a function that takes two elements a and b and returns a negative number if a should come first, zero if they are equal, and a positive number if b should come first.

In Python 3, you pass a comparator to sorted() by wrapping it with functools.cmp_to_key(). The comparator must be transitive: if a beats b and b beats c, then a must beat c. Without transitivity, the sort order is undefined and the output is wrong.

The canonical example is Largest Number (LC 179): given a list of non-negative integers, arrange them so their concatenation forms the largest possible number. The natural integer sort does not work — the right question is: does placing a before b (i.e., the string str(a) + str(b)) produce a larger number than str(b) + str(a)? That comparison is the comparator.

Proving transitivity for the ab/ba comparator

Let a, b, c be number strings. The concatenation order defines a total order if it is transitive: if ab > ba and bc > cb, then ac > ca. This can be shown by treating the concatenations as numbers in base 10: ab = a · 10^|b| + b. The relation ab ≥ ba expands to a · 10^|b| + b ≥ b · 10^|a| + a, which rearranges to a(10^|b| − 1) ≥ b(10^|a| − 1). This is an inequality on non-negative integers that chains transitively, so the comparator is valid for sorting.

Worked problemMediumLeetCode 179
Largest Number

Given a list of non-negative integers, arrange them in an order such that their concatenated string representation is numerically as large as possible. Return that string.

Think it through

Restate

Choose a permutation of the integers such that concatenating their string forms is lexicographically (and numerically) maximal.

Brute force

Try all permutations, concatenate each, return the maximum. That is O(n!) permutations — completely impractical even for n = 12.

The waste

The brute force re-examines global permutations when only pairwise decisions matter. The greedy insight: between any two numbers a and b, the better one to put first is whichever concatenation — str(a)+str(b) or str(b)+str(a) — is larger. This pairwise relation is transitive, so it defines a total order that we can sort by.

The move

Convert all integers to strings, then sort them with a comparator: cmp(a, b) = -1 if a+b > b+a else 1 (negative means a comes first). Concatenate the sorted result. Handle the all-zeros edge case: if the largest element is "0", the answer is "0".

Invariant & cost

After the sort, for any adjacent pair, the left element wins the pairwise race — transitivity guarantees this extends to the global arrangement. O(n log n) time for the sort (each comparison takes O(L) string time where L is the max digit length — a constant in practice). O(n) space for the string conversion.

The code

python

from functools import cmp_to_key

def largestNumber(nums):
    strs = [str(n) for n in nums]   # convert to strings first

    def cmp(a, b):
        # a before b if str(a)+str(b) is lexicographically larger
        if a + b > b + a:
            return -1              # negative → a comes first
        elif a + b < b + a:
            return 1               # positive → b comes first
        return 0

    strs.sort(key=cmp_to_key(cmp))
    result = "".join(strs)
    return "0" if result[0] == "0" else result  # all-zeros edge case
Narrate it to the interviewer

"The brute force is all permutations — O(n!) — so that's out. The key insight is that I only need to compare elements in pairs: given two numbers a and b, which concatenation is bigger, a followed by b, or b followed by a? That pairwise relation is transitive — I can prove it using the base-10 expansion — so it defines a total order I can sort by. I convert everything to strings, sort with a custom comparator that compares ab versus ba, then join the result. The only edge case is all zeros: if the biggest string is '0', the whole answer is '0'. That's O(n log n) for the sort and O(n) space."

Dry-run aloud: [3, 30, 34, 5, 9] → sorted as 9, 5, 34, 3, 30 → "9534330". Also check [0, 0] → "0", and [10] → "10".

What the interviewer asks next

"Prove your comparator is transitive." → Walk through the base-10 expansion above. "What if there are very large numbers?" → The integers can be large, but once converted to strings, string comparison is still character-by-character, which is O(L) per comparison. "Could you use a key function instead?" → For this problem, no: the ordering of "9" vs "30" depends on both values, not a property of either alone, so a comparator is required.

03 Sort-then-scan

Sort-then-scan and sort-then-greedy combos

Sorting is almost never the whole solution — it is the preprocessing step that makes a subsequent scan or greedy pass correct. These combinations are among the most common medium-difficulty interview problems because they require two clean insights: recognizing that sort unlocks the problem, and then executing the correct follow-up pass.

Sort-then-scan: detecting overlaps and merging

After sorting intervals by start time, any interval that overlaps the previous one has a start that falls within the previous interval's range. A single left-to-right pass handles it: keep a running "current" interval and either extend it when the next one overlaps, or flush it and start fresh when there is a gap.

The brute force checks every pair of intervals for overlap — O(n²). After sorting — O(n log n) — the scan is O(n). The invariant of the scan: everything in the output so far is already fully merged, and the current interval represents the running extent of the latest cluster.

Sort-then-greedy: decisions that only work in order

Some greedy algorithms are only correct when elements are processed in a specific order. Sorting is what delivers that order. A common structure: sort by one criterion, then greedily pick or skip elements in that order, extending some running quantity. The correctness argument is always: given the sort order, the greedy choice at each step is locally optimal and cannot be improved by any later decision.

Worked problemMediumLeetCode 56
Merge Intervals

Given a list of intervals where each interval is a [start, end] pair, merge all overlapping intervals and return the resulting list of non-overlapping intervals, in sorted order.

Think it through

Restate

Collapse overlapping or touching intervals into single merged intervals; return the non-overlapping cover.

Brute force

For each interval, scan every other interval to check for overlap; merge on the spot and repeat until no overlaps remain. The repeated passes are O(n²) in the worst case, and it is tricky to implement correctly without extra bookkeeping.

The waste

Two non-adjacent intervals in the unsorted list can only overlap if there is a chain of overlapping intervals between them. Once sorted by start, overlapping intervals are always adjacent — the check collapses to a single comparison with the previous interval.

The move

Sort intervals by start time. Walk left to right maintaining a current interval (initialized to the first). For each subsequent interval: if its start is at or before current's end, it overlaps — extend current's end to the max of the two ends. Otherwise, flush current to the output and start a new current.

Invariant & cost

Everything in the output is fully merged; current holds the running extent of the ongoing cluster. Each interval is visited once. O(n log n) time (sort dominates), O(n) space (output list).

The code

python

def mergeIntervals(intervals):
    if not intervals:
        return []
    intervals.sort(key=lambda x: x[0])   # sort by start time
    merged = [intervals[0]]               # seed with the first interval

    for start, end in intervals[1:]:
        if start <= merged[-1][1]:         # overlaps → extend
            merged[-1][1] = max(merged[-1][1], end)
        else:                               # gap → flush and start fresh
            merged.append([start, end])
    return merged
Narrate it to the interviewer

"Without sorting, detecting overlaps needs every pair — O(n²). Once I sort by start, overlapping intervals are adjacent by definition: any interval that starts before the current one ends must overlap it. So I sort, seed a result list with the first interval, then walk forward: if the next start is at or before the running end I extend it; otherwise I flush and start a new current. The invariant is that everything already in the result is fully merged. The sort is O(n log n); the scan is O(n). I do watch for the edge case where one interval is completely contained inside another — the max handles that."

Dry-run aloud: a single interval, all intervals identical, completely non-overlapping, and one interval fully swallowing all others.

04 Stability & linear sorts

Stable sort and the O(n) counting/radix note

A sort is stable if elements with equal keys retain their original relative order. Python's sorted() and .sort() are both guaranteed stable (they use Timsort). Stability matters whenever you sort in multiple passes — for example, sort by secondary key first, then by primary key; a stable sort on the primary key preserves the secondary order among ties.

The O(n log n) lower bound applies only to comparison-based sorts. When the values come from a small, known range — say, integers in [0, k) — you can use counting sort (or its generalization, radix sort) to sort in O(n + k) time. This is effectively O(n) when k is a constant or bounded by n.

Comparison sort (Timsort)

  • Works on any comparable type.
  • O(n log n) time, O(n) space (stable).
  • No domain restriction; handles strings, tuples, custom objects.
  • Uses a key function or comparator — flexible.

Counting / radix sort

  • Restricted to integer keys in a bounded range.
  • O(n + k) time, O(k) space; stable in standard implementations.
  • No comparisons — pure bucket arithmetic.
  • Required when the problem forbids O(n log n) or says "one pass".
When to reach for counting sort

If the problem says the values are small non-negative integers (e.g., digits 0–9, grades 0–100, colors 0–2), counting sort is a signal. It also appears implicitly whenever the problem asks you to sort in "one pass" and n is large — for example, Sort Colors (LC 75) is really a three-way counting sort with three buckets.

05 Practice

Practice ladder

The pattern only becomes fluent through repetition. For every problem below, say the brute force and its complexity first, then name the sort key or comparator, then code the follow-up scan. Aim for easy rungs in under 15 minutes, medium in under 30.

EasySort Array By Paritykey sort

Rearrange an array so all even numbers come before all odd numbers; any relative order within each group is fine.

Idea Key x % 2 → evens (key 0) sort before odds (key 1); O(n log n) sort or O(n) two-pointer partition.

EasyRelative Sort Arraykey sort

Given two arrays, sort the first array so elements appearing in the second array come in the order they appear there, with leftover elements appended in ascending order.

Idea Build a rank map from the second array; key = (rank if present else (n, x)); single sort.

EasySort Array by Increasing Frequencykey tuple

Sort an array so elements with lower frequency come first; break ties by placing larger values first.

Idea Count frequencies; key = (freq[x], -x); a single sorted() call handles both criteria.

EasyFind Minimum in Rotated Sorted Arraysort signal

An originally sorted array has been rotated at some unknown pivot; find the minimum element in O(log n).

Idea A sort-aware observation — binary search; but if O(n) is fine, a linear scan also works.

MediumLargest Numbercustom comparator

Arrange non-negative integers so their concatenation forms the numerically largest possible value.

Idea Comparator: cmp(a,b) = -1 if a+b > b+a; wrap with cmp_to_key; handle all-zeros edge case.

MediumMerge Intervalssort-then-scan

Merge all overlapping [start, end] intervals and return the resulting non-overlapping list.

Idea Sort by start; walk and extend or flush; invariant: output is always fully merged.

MediumInsert Intervalsort-then-merge

Insert a new interval into an already-sorted, non-overlapping list of intervals and merge any resulting overlaps.

Idea No re-sort needed; three phases — copy before, merge with, copy after the new interval.

MediumSort Colorsthree-bucket counting

Sort an array containing only 0, 1, and 2 in place, in a single pass, without a library sort.

Idea Three-pointer Dutch national flag — this is a counting sort with three buckets done in place.

MediumWiggle Sort IIsort-then-interleave

Rearrange an array so that it alternates between smaller and larger values: a[0] < a[1] > a[2] < …

Idea Sort; then interleave the smaller half and the larger half in reverse to avoid equal neighbors at the boundary.

MediumTask Schedulersort-then-greedy

Given tasks with types and a cooldown n, find the minimum time to finish all tasks obeying the cooldown.

Idea Sort by frequency descending; the most frequent task governs the frame count — a greedy formula from the sorted counts.

HardMaximum Gappigeonhole / bucket sort

Find the maximum difference between successive elements in a sorted form of the array, in O(n) time and space.

Idea Pigeonhole: with n elements in range [min, max], the answer is at least (max-min)/(n-1); place elements into n-1 buckets, then scan adjacent buckets.

HardCount of Smaller Numbers After Selfmerge sort / BIT

For each element in an array, count how many elements to its right are strictly smaller.

Idea Modified merge sort that counts inversions during the merge step; O(n log n) time and space.

How to drill this pattern

For every problem:
(1) say the brute force and its big-O;
(2) identify the sort unlock — what property becomes visible after sorting?;
(3) name the key, comparator, or sort criterion;
(4) state the follow-up pass and its invariant;
(5) code it;
(6) dry-run the edge cases — empty input, single element, all equal values, all identical intervals, or all-zeros for Largest Number. The interviewer is watching whether you naturally move from brute force to insight to code, not just whether you remember the answer.

★ Review

Summary & self-test

The whole pattern compresses to one diagnostic question: what becomes visible or easy once the data is in a particular order? If the answer is "overlaps become adjacent," "the greedy choice is well-defined," or "equal elements group together," then sorting is your move. Pay the O(n log n) price, then execute a linear follow-up with a clear invariant.
