Service Optimization

This is Phase 6 · Inference Optimization — Latency & Cost, and it opens with a claim that sounds too good: you can make the same model 3–5× faster and cheaper without touching a single weight. Not a smaller model, not a quantized one, not a distilled one — the identical checkpoint, served better. The wins don't come from the math the GPU does; they come from the memory it moves and the work it repeats. Decode is memory-bandwidth-bound, and the KV cache is the real ceiling on how many requests you can run at once. This lesson is the toolbox that pushes both limits: the KV cache, paged attention, continuous batching, speculative decoding, and prompt/response caching.

00 The framing

The GPU is 30% busy and the bill is huge

You shipped a good model. Now the pager goes off: under load, time-to-first-token spikes, tokens-per-second sags, and the cost per request keeps climbing. You open nvidia-smi expecting a pegged accelerator — and the H100 is sitting at 30% utilization. Something is deeply wrong, but it is not the model. It is how the model is being served. A naive server — one request at a time, or a fixed batch that stalls until its slowest member finishes, recomputing attention over the whole sequence at every step — leaves most of the most expensive hardware you own doing nothing.

Two facts anchor everything that follows, and getting them straight is most of the lesson. First, inference has two phases with opposite bottlenecks. Prefill — the single forward pass that reads the whole prompt at once — is compute-bound: it is one giant batch of matrix multiplies, and the GPU's raw FLOPs are the limit. Decode — generating the answer one token at a time — is memory-bandwidth-bound: each step produces exactly one token but must reload all the model weights plus the growing KV cache from HBM to do it. The ceiling is how fast you can move bytes, not how many multiplies you can do. Decode dominates latency and cost for typical generations, so every technique here attacks memory movement and repeated work, not arithmetic.

Second, after the weights, the KV cache is the biggest thing in GPU memory — and memory, not compute, is what usually caps concurrency. Teams size a batch by FLOPs, then OOM on the KV cache the moment context gets long or traffic gets bursty. So the plan is: understand the cache (why it exists, why it hurts), manage its memory so you can pack more requests in, keep the GPU saturated across requests, generate more than one token per expensive pass, and stop paying for work you've already done. Same weights, radically different economics.

The one distinction to keep straight

Prefill is compute-bound; decode is memory-bandwidth-bound. That single asymmetry explains the whole lesson: paged attention and continuous batching pack more requests into the memory-bound decode phase; speculative decoding gets more tokens out of each memory-bound pass; prefix caching skips redundant compute-bound prefill. If a technique doesn't touch bandwidth or repeated work, it won't move serving cost much.

01 Why decode is even possible

The KV cache ▸

Start with the thing that makes autoregressive decoding tractable at all. To produce token t, self-attention needs the keys and values of every token before it. The obvious — and disastrous — implementation recomputes all of them from scratch on every step: emit token 1, then re-read tokens 1–2 to emit token 2, then re-read 1–3, and so on. That is O(t) work per step and O(n²) to generate n tokens, and it gets slower with every token. Step through the contrast — the naive path on top, the cached path on the bottom:

The cache is the whole trick: compute each token's K and V once, keep them in HBM, and let the new token attend against the stored pair. Every step becomes O(1) and the full generation is O(n). Without it, long-context decode would be hopeless. But the bill for that speed is memory. The cache size is exact and unforgiving: per block it is 2 (K and V) × block_size × num_kv_heads × head_size × dtype_bytes, and it grows linearly with batch size and with sequence length at once. As a rough mid-2026 anchor, an 80 GB H100 holding BF16 KV has room for roughly ~32K tokens of cache after the weights; drop the KV to FP8 and you roughly double that to ~64K. That is why long-context or high-concurrency workloads hit the wall first — and why the next three sections are all, at heart, about spending that memory well.

The ceiling is memory, not FLOPs

The most common serving mistake is to size the batch by compute and then crash on the KV cache under real traffic. Budget KV memory explicitly — it scales with batch × context — and treat every technique that follows as a way to raise that ceiling: paging removes waste, FP8/NVFP4 KV halves the bytes, and offloading spills cold blocks off the GPU. Concurrency is a memory problem wearing a compute costume.

02 Manage the memory

Paged attention: the KV cache as virtual memory

If the cache is the ceiling, the first win is to stop wasting it. The classic implementation reserved one big contiguous slab of KV memory per request, sized for the longest output the request might produce. That is a memory allocator's nightmare: you over-reserve for sequences that finish early (internal fragmentation), you can't fit a new request into the gaps between old ones (external fragmentation), and short requests hoard space they never use. Estimates put the waste at 60–80% of KV memory — most of your concurrency ceiling, gone to bookkeeping.

PagedAttention, the idea that launched vLLM, borrows the fix operating systems have used for decades: paging. Instead of one contiguous slab, the cache is carved into small fixed-size blocks (vLLM's default is 16 tokens), and a request's logical sequence maps to a scattered set of physical blocks through a block table — exactly like virtual memory maps pages to physical frames. A sequence grows one block at a time, on demand, so there is no over-reservation and no need for contiguity. Waste drops from that 60–80% to under ~4%, and every reclaimed byte converts directly into a larger concurrent batch — which, since decode is memory-bound, is throughput.

Contiguous KV (naive)

  • One big slab per request, pre-sized for the max possible length.
  • Internal fragmentation (early finishers) + external fragmentation (gaps).
  • ~60–80% of KV memory wasted → small batches, low throughput.
  • Sharing a prefix means copying it per request.

Paged KV (PagedAttention)

  • Fixed 16-token blocks, allocated on demand, mapped via a block table.
  • No contiguity needed; grow one block at a time.
  • <~4% waste → far larger concurrent batches, higher throughput.
  • Blocks are shareable → the substrate for prefix caching (Section 05).

One caveat so this lesson doesn't age badly: the original PagedAttention CUDA kernel is now something of a legacy artifact. Modern vLLM runs FlashAttention- and FlashInfer-based kernels over the same paged block layout — faster attention math, same memory abstraction. Learn the abstraction (fragmentation-free block management, and blocks you can share and reference-count), not the specific kernel. That block layout is precisely what makes the later tricks possible: shared blocks give you prefix caching, and evictable blocks give you cross-request offload.

Why paging is load-bearing

Paging isn't just a memory optimization — it's the data structure the rest of serving is built on. Continuous batching needs to admit and evict sequences cheaply (blocks, not slabs). Prefix caching needs two requests to share the same physical blocks. KV offloading needs to move cold blocks to CPU or disk. None of that is practical over one contiguous reservation per request.

03 Keep the GPU full

Continuous batching ▸

Paging lets you fit more requests; continuous batching keeps them working. Under old-style static batching, you gather N requests, run them together, and can't touch the batch until every member is done. Mix a 20-token reply with a 2,000-token essay and the whole batch runs at the pace of the essay while the short request's slot sits idle for the rest — the GPU is "busy" and mostly wasted. Continuous batching (also called in-flight or iteration-level batching) fixes this by scheduling at the granularity of a single decode step. Step through a few timesteps and watch the batch stay saturated:

The engine runs a tight schedule → forward → postprocess cycle every iteration: pick which sequences run this step, do one decode pass, then evict anyone who just emitted a stop token and admit waiting requests into the freed capacity. Because a request can join or leave mid-flight, a short reply never waits behind a long one. Under the hood the batch is flattened into a single long "super-sequence," with position offsets and masking so each sequence attends only to its own tokens — which also kills the right-padding waste static batching needed to make lengths line up. A modern scheduler can even mix prefill and decode in the same step. This is table stakes now: vLLM, SGLang, and TensorRT-LLM all do it, and combined with paging and chunked prefill it is a big slice of that 3–5× over a naive PyTorch generate loop.

Where the win is biggest

Continuous batching pays off most when request lengths are mixed — exactly what real traffic looks like. If every request were the same length, static and continuous batching would nearly tie. The gain is proportional to how much variance there is between your shortest and longest generations, because that variance is precisely the idle time static batching bakes in.

04 More tokens per pass

Speculative decoding ▸

Paging and batching pack the GPU; speculative decoding gets more output from each expensive pass. The insight is subtle. A decode step is memory-bound: verifying a token costs about the same as generating one, because either way you reload all the weights. So what if a cheap model guessed several next tokens, and the big model verified all of them in a single pass? A small draft model proposes k tokens; the big target model checks all k in one parallel forward pass; every guess the target would have made anyway is accepted for free. Step through one round:

The magic word is lossless. Because acceptance uses rejection sampling against the target's own probabilities, the tokens you keep are distributed exactly as if you had sampled them one at a time from the target — not "approximately," not "usually." The speedup (~2–3× typically) is pure latency, no quality tax. The variants differ only in where the draft comes from: an independent small model; n-gram / prompt-lookahead (no draft model at all — it matches n-grams already in the prompt/history, superb for code and RAG where text repeats); Medusa (extra decoding heads on the target, but not strictly lossless under non-greedy sampling because it relaxes acceptance); and the EAGLE family, which drafts from the target's own hidden states. EAGLE-3 (2025) is current SOTA — a dynamic draft tree over fused multi-layer features, reporting ~3.0–6.5× at low batch. vLLM ships n-gram, EAGLE, and Medusa.

Not a free lunch at scale

Speculative decoding's speedup is entirely a function of acceptance rate, and it can underperform plain decode at high concurrency: when the batch is already large enough to saturate compute, the extra verification work costs more than the memory-bound step it was hiding. It shines at low batch (~1–4) and on predictable text — code, structured output, RAG. And don't over-draft: pushing past ~4–5 proposed tokens usually reduces throughput, because acceptance falls the further out you guess.

05 Stop redoing work

Cache the prompt, cache the answer

The cheapest token is the one you never compute. Two very different caches live at this layer, and conflating them is a classic source of wrong reasoning about correctness. The first is prefix / prompt caching: reuse the computed KV of a shared prefix — a long system prompt, a stack of few-shot examples, a big document — across requests, so its prefill is done once and skipped forever after. This reuses computation, so it is exact and lossless: byte-identical prefix in, byte-identical KV out. vLLM turns on automatic prefix caching by default, hashing tokens into 16-token chunks and doing a find-longest-cache-hit against stored blocks. SGLang generalizes it with RadixAttention — a radix tree over all cached sequences, not just one system prefix — reporting up to ~6× on prefix-heavy RAG and multi-turn chat.

Anthropic exposes the same mechanism as a managed API feature, and the pricing is the part worth memorizing because the multipliers are durable even as model names and dollar prices drift. A cache write costs 1.25× the base input price for the default 5-minute TTL (2.0× for the optional 1-hour TTL); every cache read costs just 0.10× base input — a 90% discount. So a 5-minute cache pays for itself after one hit, a 1-hour cache after two. The catch is that the shared prefix must be byte-identical and above the model's minimum cacheable size (often ~1,024 tokens): one changed token, reordered tool definitions, a toggled thinking mode, or an inserted image invalidates it. The rule that falls out is simple — put the stable content first (system prompt, tools, few-shot, long doc) and the volatile content (the user's turn) last, or you pay full price every time.

The second cache is response / semantic caching, and it is a different beast: it reuses output, not computation. A response cache returns a stored answer for an exact-repeat query; a semantic cache (GPTCache, a Redis vector cache) embeds the incoming query and returns a stored answer when similarity to a past query exceeds a threshold. The savings are large — 60–73% off the bill on repetitive traffic — but this cache is lossy and genuinely risky. A false hit serves a wrong answer, because embedding models happily match texts that share words but mean opposite things.

Prefix / prompt cache

  • Reuses KV (computation) of a shared prefix.
  • Exact & lossless — same result as no cache.
  • Cuts TTFT and input cost (up to ~90% cost / ~85% TTFT on long shared prefixes).
  • Fails safe: a miss just means full price, never a wrong answer.

Response / semantic cache

  • Reuses the output for repeat / near-duplicate queries.
  • Approximate & lossy — a false hit returns a wrong answer.
  • Big bill cuts (~60–73%) on repetitive traffic.
  • Needs threshold tuning, a cross-encoder re-rank, a TTL, and per-tenant namespacing.

Semantic caches also go stale: LRU eviction without a TTL will happily serve yesterday's correct-but-now-wrong answer about a price, a policy, or live data forever. So bound the blast radius — tune the similarity threshold (too loose = wrong answers, too tight = no savings), re-rank the top candidates with a cross-encoder before you trust a hit, attach a TTL and invalidate on content or model changes, and namespace by user tier, model version, and system-prompt hash. Prefix caching you turn on and forget; semantic caching you turn on and monitor.

Don't conflate the layers

Prefix caching reuses computation and is always exact — the worst case is a cache miss. Semantic caching reuses output and is approximate — the worst case is a confident wrong answer to your user. When someone says "we cache the LLM," ask which: the correctness risk is entirely different, and only one of them can hurt you.

06 Put it together

Build the fix

Back to the pager. Same symptom you opened with: GPUs at 30% utilization, TTFT spiking under load, cost per token too high — and a model you're not allowed to change. Here is the build, in the order you'd actually turn the knobs, each lever aimed at a specific one of the bottlenecks from Section 00.

Build walkthrough30% util · TTFT spikes · high $/token
From a naive generate loop to a saturated, cached, speculative server

The workload: a support assistant with a large shared system prompt (policies + tools + few-shot, ~3K tokens) and short, bursty user turns. The current server runs one request at a time and re-prefills that whole system prompt on every call. You can't touch the weights. You can change how it's served — and three stacked levers turn 30% utilization into a saturated GPU that's both faster and cheaper.

The serving config — three levers on one checkpoint

serve config — tune the SERVING, never the weights# same checkpoint; three serving levers stacked on one H100
vllm serve meridian/atlas-70b-instruct \
  --kv-cache-dtype fp8              # halve KV bytes → ~2× cache capacity
  --enable-prefix-caching          # reuse the shared system-prompt prefill (lossless)
  --enable-chunked-prefill         # cap prefill chunks so decode TTFT stays flat
  --max-num-seqs 256             # continuous-batching width — bounded by KV memory, not FLOPs
  --gpu-memory-utilization 0.92  # how much HBM the KV cache may claim
  --speculative-config '{"method":"eagle","num_speculative_tokens":4}'
                                # small draft proposes 4; target verifies in one pass
Lever 1 — paged server + continuous batching

Move off the naive loop onto a paged-attention server with continuous batching. Paging cuts KV waste from ~60–80% to <4%, so far more requests fit in memory; iteration-level scheduling then admits and evicts them every step so the GPU stops idling behind the slowest request. This is what fixes the 30% utilization directly — it attacks the memory-bound decode phase where your traffic actually lives.

Lever 2 — prefix cache the system prompt

Turn on prefix caching. That 3K-token system prompt is byte-identical on every request, so its prefill is computed once and reused — cutting TTFT (no re-prefill) and input cost (you're billed a fraction for the cached prefix). It's lossless, so there's no correctness cost. Put the stable system prompt first and the volatile user turn last, or a single reordered tool breaks the hit.

Lever 3 — speculative decoding

Add a small draft model. It proposes ~4 tokens; the target verifies them in one parallel pass, so accepted tokens are effectively free — cutting TPOT (time per output token) ~2–3×, and losslessly thanks to rejection sampling. Because your traffic is low-batch and fairly predictable, this is exactly the regime where speculation wins; if load ever spikes into a saturated batch, expect the gain to shrink.

The binding constraint — the KV ceiling

Why not just crank --max-num-seqs to 4,000 and call it done? Because you'll OOM on the KV cache, not on FLOPs. Concurrency is capped by batch × context worth of KV, so the honest levers are the ones that buy KV headroom: paging removes waste, --kv-cache-dtype fp8 halves the bytes (~32K → ~64K cacheable tokens), and offload spills cold blocks to CPU. Chunked prefill is the tie-breaker that keeps a long prompt's prefill from stalling everyone's decode.

Say it back in one breath

“The weights are fixed, so I fix the serving: a paged-attention engine with continuous batching to saturate the memory-bound decode phase, prefix caching so the shared system prompt is prefilled once, and a small speculative draft so each expensive pass yields several tokens — losslessly. I can't just raise the batch forever because the KV cache is the ceiling, so I buy headroom with paging, FP8 KV, and offload, and protect TTFT with chunked prefill.”

Sanity-check knobs, in order: is prefix caching actually hitting (watch the hit rate and keep the prefix byte-stable)? is max-num-seqs KV-limited or FLOP-limited? does speculation still help at your real batch size, or is it time to disable it under load?

That build is the durable core, but the 2025–2026 frontier is worth knowing so you can reach for the heavier tools when the config knobs run out. Prefill/decode disaggregation is now a mainstream production pattern, not just research: run prefill and decode on separate GPU pools so a long prompt's compute-bound prefill can't stall another request's memory-bound decode, and tune TTFT independently of TPOT with different parallelism per phase. vLLM ships it (still labeled experimental) and NVIDIA Dynamo makes disaggregated serving plus KV-aware routing first-class — but it adds a KV-transfer hop between pools, so 2025–2026 work (TaiChi and the "aggregation vs disaggregation" goodput papers) argues the right answer is often a hybrid that switches on load. Chunked prefill — the lever you already flipped — is the lighter default: it splits a long prefill into token-capped chunks interleaved with ongoing decodes so decodes aren't blocked, but it mitigates rather than eliminates the interference, which is the residual argument for full disaggregation.

Two more levers round it out. Cross-request KV reuse and offload went mainstream via LMCache ("prefill-once, reuse-everywhere") — store and evict KV across a tiered hierarchy (GPU → CPU RAM → local disk → Redis/InfiniStore/Mooncake) and reuse it across engine instances, not just within one request; Dynamo adopted it as its caching layer and adds a KV Block Manager with CPU/disk tiers. And low-precision KV is now first-class: FP8 KV roughly doubles cacheable tokens broadly, while Blackwell-only NVFP4 KV halves the bytes again versus FP8 with <1% accuracy loss — though the best Blackwell configs are usually mixed (e.g., NVFP4 weights with FP8/BF16 attention), since pure FP4 hurts quality-sensitive work. Finally, remember the cost lever that sits above all of this: model routing (RouteLLM-style) sends easy queries to a small cheap model and only hard ones to the expensive one — up to ~85% cost reduction while keeping ~95% of frontier quality on MT-Bench, with negligible router overhead. It's complementary to everything here, not a substitute: serving optimizations make each model cheaper to run; routing decides which model runs at all.

★ Review

Summary & self-test

One idea runs through the lesson: the same weights can be served far faster and cheaper by fixing memory movement and repeated work — the KV cache makes decode tractable but becomes the memory ceiling, so page it, keep the GPU saturated with continuous batching, squeeze more tokens per pass with speculative decoding, and skip redundant compute with prefix caching (and, carefully, response caching).
