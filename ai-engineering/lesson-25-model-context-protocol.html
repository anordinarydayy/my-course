The Model Context Protocol

Tool use (Lesson 24) gave one model hands. But every team wiring hands onto every model was solving the same problem in a slightly different way — a bespoke connector for each app crossed with each data source. MCP is the open standard that collapses that mess: build one server, and every MCP-compatible client can use it; build one client, and it reaches every server. It is, in Anthropic's phrase, "USB-C for AI" — and by mid-2026 it is the connective tissue the whole agent ecosystem runs on. This is a Core lesson in Phase 4 — Agents & Tool Use.

00 The failure that starts everything

The N×M integration wall

Start where tool use left off. You have a model that can call functions, and a pile of things you want it to reach: your Postgres, your GitHub, your Slack, a customer database, a filesystem. So you write an integration — the glue that discovers what a tool can do, describes it to the model, authenticates, invokes it, and folds the result back into context. It works. Then a teammate wants the same customer database reachable from a different AI app — the IDE assistant, not the chat app — and discovers your glue is welded to the first app's API. So they write it again. Then a third app appears. Then a fourth data source.

This is the wall, and it is quadratic. With N AI applications and M tools or data sources, the naive world needs N×M bespoke connectors — every app-times-tool pair re-implementing the same plumbing (auth, discovery, schema, error handling), each one a place to drift and rot. Anthropic named the pain precisely when it launched MCP: models were "trapped behind information silos," and every new silo meant another custom bridge. Adding one app forced M new integrations; adding one tool forced N. Nobody could keep up, and no integration was reusable across vendors.

The way out is the oldest trick in networking: stop building bridges and agree on a protocol. Standardize the interface once — a shared message format and a shared, small vocabulary of capabilities — and the arithmetic changes. Any compatible client can now talk to any compatible server without knowing anything specific about it. Build M servers and N clients and you are done: N×M collapses to N+M. That is the entire pitch, and it is why the analogy stuck. Before USB-C you kept a drawer of adapters, one per device-times-port; after it, one connector shape let everything talk to everything. MCP is that connector shape for models and their tools.

Anthropic open-sourced the Model Context Protocol on November 25, 2024, and did the thing that makes a protocol actually spread: it shipped day-one Python and TypeScript SDKs plus pre-built reference servers for Google Drive, Slack, GitHub, Git, Postgres, and Puppeteer. You didn't have to believe in an abstract standard — you could point Claude at your Postgres in an afternoon. The rest of this lesson is what's inside that connector: the roles, the primitives, the transports, the security model, and the reason the whole industry adopted it within a year.

The one-line version

MCP is an open standard so that any MCP client can talk to any MCP server — "USB-C for AI." Its whole value is arithmetic: it turns N×M bespoke integrations into N+M reusable ones. Everything else in this lesson exists to make that one property true and safe.

01 The three roles and the handshake

Host · client · server ▸

A protocol needs an agreed cast. MCP defines exactly three roles. The host is the AI application you actually use — Claude Desktop, VS Code, ChatGPT — and it contains one or more clients. Each client holds a single, stateful, one-to-one connection to exactly one server; if the host talks to five servers, it spins up five clients. The server is a (usually small) program that exposes capabilities — your database lookup, your filesystem, Stripe's API. Every message between a client and its server is JSON-RPC 2.0, UTF-8 encoded: a request names a method and carries params, a response carries a result or an error.

The crucial thing to internalize is that the model never talks to the server directly. The client mediates. The flow is always two beats — discover, then call — and the model only ever sees what the client chooses to surface. Step through one full request, from cold start to answer:

Two properties fall out of that sequence and are worth naming. First, discovery is dynamic: the client asks tools/list at runtime rather than hard-coding what the server offers, which is exactly what lets one client work against a server it has never seen. Second, the connection is stateful and negotiated: in the specs through 2025, the very first exchange is an initialize handshake where both sides declare their protocol version and capabilities, so a client that speaks elicitation and a server that doesn't simply agree not to use it. (Hold that handshake in mind — in §06 you'll see the 2026 revision delete it on purpose to make servers stateless and load-balanceable. The fact that a protocol can evolve that boundary without breaking the primitives is itself the point.)

02 The vocabulary of capability

Three primitives (and three more)

A protocol is only as useful as its shared vocabulary, and MCP keeps that vocabulary deliberately small: six primitives, three exposed by servers and three by clients. Get these six and you can read almost any MCP integration.

On the server side, the defining question for each primitive is who is in control. Tools are model-controlled: actions the model may decide to invoke — a function with a name, a natural-language description, and a JSON-Schema for its arguments, discovered with tools/list and invoked with tools/call. Resources are application-controlled: read-only data attached to the context on demand — a file, a URL, a database snapshot — each identified by a URI and pulled in with resources/read, never executing anything. Prompts are user-controlled: reusable, parameterized templates the user invokes deliberately (think a slash-command with arguments and autocomplete), fetched with prompts/get. Same discovery-then-use shape for all three; different party pulling the trigger.

The three client primitives are the mirror image — capabilities the server can reach back for. Sampling lets a server ask the client to run an LLM completion on its behalf, so the server gets intelligence without ever holding your API keys or dictating your model choice — the client stays in control of both. Roots are base URIs that fence off the filesystem or URL territory a server is allowed to touch, a boundary the host sets. Elicitation — added in the 2025 spec revisions — lets a server pause mid-task to request structured input from the user, like confirming a destructive action or asking for an email address, instead of guessing. Together they invert the usual direction: the server, normally the callee, gets a controlled way to call back.

Why "who controls it" is the load-bearing column

That control axis isn't trivia — it's the seam the security section turns on. A tool is the only primitive the model can trigger on its own, which is exactly why a tool's description is where an attacker aims (§05). A resource can't execute, so it's a smaller blast radius. And sampling exists so a server can be smart without you ever handing it a key. When you design a server, ask of every capability: who pulls this trigger, and what happens if the wrong party does?

03 The wire underneath

stdio vs Streamable HTTP ▸

JSON-RPC says what the messages are; a transport says how they physically travel. MCP standardizes two, and the split maps cleanly onto local versus remote. The naive assumption — "it's all just HTTP" — misses that the most common MCP server runs on your own laptop with no network at all. Watch one client reach both a local tool and a remote service:

A little history explains the second transport's clumsy name. The original 2024-11-05 spec offered an HTTP transport that used two endpoints — one for POSTing requests, a separate long-lived SSE stream for responses — which was awkward to scale and reconnect. The 2025-03-26 revision replaced it with Streamable HTTP: a single endpoint (conventionally /mcp) that handles POST and GET and optionally upgrades to Server-Sent Events only when streaming or server-initiated messages are actually needed. Clients keep a backwards-compatible path — POST an InitializeRequest, and on a 4xx fall back to the legacy flow. HTTP clients also send an MCP-Protocol-Version header (e.g. 2025-11-25); if it's absent, the server assumes 2025-03-26. Optional stateful sessions ride on a cryptographically-secure MCP-Session-Id handed back at initialize.

The practical rule: clients should support stdio wherever they can — it's the zero-config, no-network default for local tools, and it's why your first MCP server (next section) needs no web framework at all. Reach for Streamable HTTP when the server lives somewhere else, and when you do, read §05 first, because a network transport drags a network threat model in with it.

04 The pattern, end to end

A server in twenty lines

Enough theory — build the thing. The point of this worked example is to feel how little you write, because the protocol and the SDK carry the weight. We'll take one realistic task all the way through: expose a single internal lookup so that every AI app you use can call it, without writing a per-app plugin ever again.

Worked exampleexpose one tool · Python · FastMCP
Exposing one internal lookup to every AI app

You keep a read-only customers table. You want Claude — in the desktop app and in your IDE, and tomorrow in whatever tool your teammate prefers — to answer "what tier is customer C-4821?" by actually querying it. The question is not "can the model call a function" (Lesson 24 answered that); it's "how do I expose this once so I never re-integrate it per client?"

The naive path: a bespoke connector per client (N×M)

the world before a protocol — one connector per AI appclaude_plugin.py     # auth + discovery + schema + call, written for Claude
chatgpt_action.py    # auth + discovery + schema + call, written AGAIN
cursor_extension.ts  # auth + discovery + schema + call, a THIRD time
# N apps × M tools = N×M connectors, each re-plumbing identical wiring

Each file re-implements the same four jobs against a different app's API. Add a fourth app and you write a fourth connector; add a second tool and you multiply again. The plumbing isn't hard — it's repeated, and repetition is where drift, security holes, and rot live.

The move: one MCP server, every client (N+M)

server.py — a complete MCP server (official Python SDK, FastMCP)from mcp.server.fastmcp import FastMCP
import sqlite3

mcp = FastMCP("customer-db")

@mcp.tool()
def query_customer(customer_id: str) -> dict:
    """Look up a customer by id; returns their name and tier."""
    con = sqlite3.connect("customers.db")
    row = con.execute(
        "SELECT name, tier FROM customers WHERE id = ?",  # parameterized, read-only
        (customer_id,),
    ).fetchone()
    con.close()
    if row is None:
        return {"error": "not found"}
    return {"name": row[0], "tier": row[1]}

if __name__ == "__main__":
    mcp.run()          # stdio by default; mcp.run(transport="streamable-http") to go remote

That's the whole server. Notice what you did not write: no JSON-RPC parsing, no tools/list handler, no schema by hand. FastMCP reads the type hint customer_id: str and the docstring and auto-generates the tool descriptor — the name, the description, and the JSON-Schema — that a client will fetch. Register this server once in your host's config, and Claude Desktop, VS Code, and Cursor all discover query_customer the same way.

The invariant — the server owns the contract

The docstring becomes the description and the type hints become the input schema. That means the client and the model only ever see the descriptor the server publishes; they never reach into your code. The contract lives on one side, and it's the only thing that crosses the wire — which is precisely why, in §05, that descriptor is also the attack surface.

Discover → call, concretely

Client subprocess-launches the server → initialize → tools/list returns [{name:"query_customer", description:"Look up a customer…", inputSchema:{…}}] → the model, needing a record, emits tools/call {name:"query_customer", arguments:{customer_id:"C-4821"}} → the server runs the query and returns {"name":"Ada Lovelace","tier":"gold"} → the client injects that into context. Exactly Fig. 1, now backed by real code.

Swap the transport, not the tool

To go from "local dev tool" to "remote service the whole company shares," you change one argument: mcp.run(transport="streamable-http"). The tool body is untouched. The same 20 lines run as a stdio subprocess on your laptop or behind an authenticated /mcp endpoint in prod — the protocol decoupled the capability from the wire.

Say it back in one breath

“I exposed one internal lookup as an MCP tool instead of writing a plugin per app. FastMCP turned the docstring into the tool's description and the type hints into its JSON-Schema, so I wrote zero protocol code — and because it's a standard, Claude Desktop, my IDE, and Cursor all discover and call it identically. The server owns the contract; clients only ever see the published descriptor. And when it needs to go remote, I flip one transport argument, not the tool.”

The reusability is the payoff: one server, N clients — N×M became N+M in about twenty lines. The catch, next section, is that the very descriptor that makes this frictionless is text the model trusts, so whoever writes that text writes the model's instructions.

05 Where it bites

The trust boundary is the schema ▸

Here is the uncomfortable consequence of everything above. A tool descriptor — the name, the description, the schema — is merged into the model's context so the model can decide to use it. But once merged, that text is indistinguishable from your own developer instructions. Which means the trust boundary of an MCP integration is not your API key; it's the descriptor itself. Whoever controls the text in that window controls what the model does. Watch the attack, then the defense:

The single most important move a serious engineer makes here is to refuse to lump the threats together, because they have different trust boundaries and therefore different fixes. Keep these four distinct:

  • Tool poisoning (the "MCPoison" class, CVE-2025-54136). A malicious or compromised server hides directives in a tool's description or schema. This is a supply-chain flaw — the attack surface is the descriptor, not your input validation — and no amount of sanitizing user input touches it. Its evil twin is the rug-pull (CVE-2025-54135, "CurXecute"): because servers may re-register their tool list mid-session, an attacker ships a benign tool, passes your one-time audit, then silently swaps in a poisoned schema on the next tools/list refresh. Static, one-time review is worthless against a moving target.
  • Prompt injection via tool results. Untrusted data a tool returns — a web page, a support ticket, a file — can carry instructions the model then obeys. Simon Willison flagged this class in April 2025: an LLM "will trust anything that can send it convincing-sounding tokens." This one is an input-validation problem, and conflating it with tool poisoning gets you the wrong defense.
  • Confused deputy / over-broad permissions. A server acting with your borrowed identity and wide OAuth scopes can be tricked into misusing that authority. In practice this — static client secrets and undifferentiated "borrowed agent identity" — is a bigger day-to-day risk than exotic injection. Fix it with least-privilege scopes, RBAC/ABAC against JWT claims, and per-user credentials.
  • DNS rebinding on local HTTP. Without Origin-header validation and localhost-only binding, a remote website can reach your local MCP server. The spec therefore mandates a 403 on an invalid Origin and says servers should bind only to 127.0.0.1 — the concrete baseline from §03.

The consolidated defense is an MCP gateway at network ingress doing five-stage validation — server trust (registry status + cert pinning), tool eligibility (RBAC/ABAC), shape validation (bound description length, normalize Unicode, strip zero-width characters), sanitization plus LLM-judge detection, and runtime re-validation on every invocation — with post-execution guardrails scanning outputs for secrets and PII. One control point beats client-side fan-out. And it's worth being honest about what's still unsolved in mid-2026: there's no standardized audit trail, no agreed multi-tenancy model, and no protocol-level rate limiting or cost attribution; many deployments still lean on static client secrets. The 2026 auth hardening (next section) is aimed squarely at that list.

The rule to carry out of this section

The trust boundary of a tool is its schema, not its API key. So treat every third-party server as untrusted code, pin to a registry you trust, re-validate descriptors on every refresh (rug-pulls are real), scope credentials to the minimum, and keep "poisoned description" and "poisoned result" in separate mental buckets — because a fix for one does nothing for the other.

06 Why it actually won

Why a protocol won

A standard is only "USB-C for AI" if the industry actually plugs into it, and 2025 is the year everyone did. OpenAI adopted MCP in March 2025 across its Agents SDK, Responses API, and the ChatGPT desktop app, with Sam Altman endorsing it publicly. Google DeepMind's Demis Hassabis confirmed Gemini MCP support in April. Microsoft and GitHub joined the MCP steering committee at Build 2025 and previewed native MCP in Windows 11; Salesforce anchored Agentforce 3 on it in June. A protocol backed by one vendor is a proposal; a protocol backed by every major model provider is infrastructure — and that flip is why network effects took over. By late 2025 there were 10,000+ active public MCP servers, Claude shipped a directory of 75+ connectors, and clients spanned Claude, ChatGPT, Gemini, Copilot, Cursor, Windsurf, VS Code, and Zed, with production servers from Stripe, GitHub, Notion, Linear, Sentry, Figma, Cloudflare, and more.

Two pieces of connective tissue made that scale governable. The MCP registry launched in preview on September 8, 2025 at registry.modelcontextprotocol.io — an open catalog and API of public servers that stores metadata only (never code), exposes an OpenAPI spec so anyone can build public or private sub-registries on top, and grew to ~2,000 entries within months. And the governance handoff: on December 9, 2025 Anthropic donated MCP to the newly formed Agentic AI Foundation (AAIF), a directed fund under the Linux Foundation, co-founded with Block and OpenAI and backed by Google, Microsoft, AWS, Cloudflare, GitHub, and Bloomberg. Handing a standard to a neutral foundation is how you promise the ecosystem it won't be captured — the maintainer-led governance stayed unchanged, but the ownership no longer belongs to a single company.

The spec itself is dated and steadily evolving, which is a feature: 2024-11-05 (original) → 2025-03-26 (Streamable HTTP, OAuth 2.1) → 2025-06-18 (elicitation, OAuth Resource-Server hardening) → 2025-11-25 (current stable: async long-running tasks, statelessness support, an official extensions system, and cache-freshness fields like ttlMs/cacheScope plus Mcp-Method/Mcp-Name headers so gateways can route without reading the body). The 2026-07-28 release candidate goes further: a foundational stateless rework that removes the initialize handshake and the MCP-Session-Id header — so any request can hit any server instance behind a plain round-robin load balancer — moves client info into per-request _meta fields, upgrades tool schemas to full JSON Schema 2020-12, standardizes W3C Trace Context, and tightens auth with issuer validation to close confused-deputy gaps. Official extensions like MCP Apps (Jan 2026, server-provided HTML UIs in sandboxed iframes) and Tasks now ride alongside the core, versioned independently. That the protocol could delete its own handshake without breaking the six primitives is the clearest proof that betting on the interface rather than any one implementation was the right call.

The efficiency footnote worth its own paragraph

Connect hundreds of tools and every descriptor gets stuffed into the prompt — context bloat that wastes tokens and degrades tool selection. At an illustrative mid-2026 Sonnet 4.6 input price of $3 / Mtok, thousands of tool definitions on every call is real money and worse accuracy. Anthropic's "code execution with MCP" and Cloudflare's "Code Mode" answer it by letting the agent call tools via generated code instead of loading every definition into context — Cloudflare reported 98%+ token savings. The protocol scaled; the context window didn't, so the frontier moved to using it sparingly.

★ Review

Summary & self-test

One idea runs through the lesson: MCP is an open protocol that collapses N×M bespoke integrations into N+M reusable ones — three roles (host/client/server) exchanging JSON-RPC, a small vocabulary of six primitives, two transports (stdio & Streamable HTTP), a discover-then-call flow, and a trust boundary that lives in the tool's schema, not its key — and the whole industry adopted it because a shared interface, not any one implementation, is what makes tools interoperable.
