06Concept · AI Engineering

Prompting as a Discipline

Prompting isn't "asking nicely" — it's writing a precise spec for a brilliant, literal, context-free collaborator. It's the first and highest-leverage rung of the adaptation ladder: a huge share of AI features need nothing more than a good prompt and structured output. This lesson is the craft — the techniques that work, and the ones the newest models made obsolete.

Original study notes. A practitioner's guide to prompting. It draws on public guidance — Anthropic's prompting best-practices, OpenAI's reasoning guidance, and the DSPy line of work on prompt optimization — but every explanation, diagram, and example here is our own. 00 The reframe

A prompt is a spec

The fastest way to get worse results from a model is to treat the prompt as a casual request. The fastest way to get better ones is to treat it as a specification — the same discipline you'd bring to writing a ticket, an API contract, or a brief for a contractor. A prompt is production logic; vague input produces vague, inconsistent output, no matter how capable the model.

The mental model that makes this click is Anthropic's: treat the model like a brilliant new hire on their first day. Enormous raw capability, and zero knowledge of your company's norms, your data, or what "done" looks like. You'd never tell a new hire "fix the thing" — you'd point them at the file, show one solved example, and state the acceptance criteria. The model needs the same, and for the same reason: it's smart, but it takes you literally and starts with no context.

The colleague test

Anthropic's golden rule, and the single best habit you can build: show your prompt to a colleague with no context. If they'd be confused about what you want, the model will be too. Most "the model is dumb" moments are really "my prompt was underspecified" moments. Clarity is the lever, not model size.

The good news: the techniques are few, learnable, and mostly free to apply. The rest of this lesson is the toolkit — how to structure a prompt (§1), why showing beats telling (§2), how roles and delimiters work (§3), the big 2025–2026 twist that reasoning models changed the rules (§4), the anti-patterns that quietly hurt you (§5), and a full vague-to-strong rewrite (§6).

01 Structure

The anatomy of a prompt ▸

A strong prompt isn't a paragraph — it's a small set of labeled parts, each doing one job. You won't always need every part, but knowing the full anatomy means you can diagnose a weak prompt by asking "which part is missing?" Build one up, piece by piece:

One placement rule worth memorizing, because it's both counter-intuitive and well-measured: when the data is long, put it at the top of the prompt, above your instructions and question. Anthropic reports that putting the query after a long document can improve response quality by up to ~30%. The model reads the whole context, then sees what you want done with it — and, as we saw in the foundations lessons, it attends best to the start and end, so the question landing last keeps it salient.

02 Examples

Show, don't tell ▸

Of all the techniques, the most reliable is also the simplest: give examples. A couple of input→output demonstrations — few-shot prompting, or in-context learning — steer a model far more dependably than paragraphs of description. Words describe the target; examples show it. Watch a flaky task lock into shape:

A few practical rules from the research. Three to five examples is the usual sweet spot; even two or three beat zero by a lot. Make them relevant and diverse — cover the edge cases (the "neutral" that isn't clearly positive or negative), or the model learns the wrong pattern. Wrap each in a delimiter (<example>) so the model sees where one ends. And two cautions: models show recency bias (they over-weight the last example, so order matters — randomize it), and there's an "over-prompting" cliff past roughly ten examples where more can hurt. More isn't always better; representative is.

03 Roles & structure

Roles & structure

Two structural tools shape how a prompt is read. The first is the role hierarchy. Your instructions don't all carry equal weight — they're layered by trust. The system prompt sets durable rules and persona ("you are a careful financial analyst; never give legal advice"); the user message carries the specific request. Newer model platforms formalize even more levels (a developer message between system and user), with a strict rule: lower levels can't override higher ones. The mental model: the system/developer message is the function definition — the rules and invariants — and the user message is the arguments. Put durable behavior in the system prompt; put the task in the user turn.

The second is delimiters. When a prompt mixes instructions, context, examples, and variable data, the model can blur them together — and, as the next lesson on defensive prompting shows, that blurring is also a security problem. Wrapping each section in a clear boundary fixes it. XML-style tags are the most reliable:

structure a prompt with tags

<role>You are a senior support engineer.</role>
<task>Answer the customer question using ONLY the docs below.</task>
<docs>
{{retrieved_documents}}        # untrusted / variable data, clearly fenced
</docs>
<question>{{user_question}}</question>
<format>Reply in 2-3 sentences; cite the doc section in [brackets].</format>

The model can now tell your instructions from the data it's working on, parse a complex prompt unambiguously, and you can refer to the sections by name ("answer using only <docs>"). It also makes the prompt easier for you to read and version — which matters for the next lesson, where prompts become tracked assets.

04 The 2025 twist

Reasoning models: less is more ▸

Here's the change that trips up everyone who learned prompting before 2025. For years, the headline trick was chain-of-thought: add "let's think step by step" and watch accuracy jump, because you gave the model room to reason before answering. That advice is still right — for classic, non-reasoning models. But for the new reasoning models (the o-series, DeepSeek-R1, Gemini's thinking models, Claude's extended/adaptive thinking), it's redundant at best and harmful at worst. They already reason internally before answering. Toggle between the two:

The vendors are blunt about this. OpenAI's guidance for reasoning models: "prompting them to 'think step by step' or 'explain your reasoning' is unnecessary," and "try to write prompts without examples first" — few-shot can hurt them too. DeepSeek says few-shot "consistently degrades" R1 and recommends zero-shot with an empty system prompt. Anthropic, for adaptive-thinking models, says to prefer general instructions over prescriptive steps and warns about over-prompting. The throughline:

The dividing line

Scaffold the reasoning for classic models; get out of the way for reasoning models. For a classic model, spell out the steps and show examples. For a reasoning model, state the goal and the constraints clearly, then let it think — adding "think step by step," piling on examples, or over-specifying the procedure can make it worse. (Anthropic even notes that with thinking off, some models are so sensitive to the word "think" that you should say "consider" or "evaluate" instead.) Know which kind of model you're calling, and prompt accordingly.

05 What not to do

Anti-patterns

A few habits quietly sabotage prompts — several of them are things that used to be necessary and now backfire on instruction-following modern models:

  • Vagueness. "Make an analytics dashboard" leaves a hundred decisions to chance. The fix is the whole of §1 — specify the parts.
  • Negation that backfires. "Do NOT use markdown" can focus the model on markdown. Tell it what to do instead: "write in flowing prose." Positive framing beats prohibition.
  • Shouting. "CRITICAL: YOU MUST use this tool" was a workaround for older, reluctant models. Newer ones follow instructions closely, so aggressive language causes overtriggering — the model overuses the tool. Plain "use this tool when…" works better.
  • Conflicting or blanket instructions. "If in doubt, use the search tool" makes a modern model search constantly. Replace blanket defaults with targeted conditions.
  • Leading the witness. Models are trained to be helpful and will often comply with a leading or even illogical premise rather than push back. Don't bake your assumption into the question if you want an honest answer.
  • Assuming context. The brilliant-new-hire again: it doesn't know your product, your conventions, or last week's decision unless you tell it (or retrieve it).
Prompts can be optimized, not just hand-tuned

A frontier worth knowing: tools like DSPy flip prompting into "program, not prompt." You define the task as a typed program, supply a handful of examples and a metric, and an optimizer (e.g. MIPROv2) compiles the prompt for you — searching instructions and example selections automatically, the way you'd tune model weights rather than hand-write strings. The payoff is real: when you switch models, you re-run the optimizer instead of re-hand-tuning every prompt. You won't reach for this on day one, but it signals where the craft is heading — from artisanal strings to optimized, testable prompt assets, which is exactly the next lesson.

06 Putting it together

Vague → strong

Everything above, applied to one rewrite. Same model, same task — only the prompt changes.

Rewritebefore → after
“Summarize this feedback”

You're building a feature that turns a pile of raw customer feedback into a triage report for the product team. The first attempt is one line — and the output is different every run, sometimes prose, sometimes bullets, never quite usable.

The before

Summarize this feedback: {{data}}

The after

<role>You are a product analyst.</role>
<task>Turn the customer feedback below into a triage report for the
product team. It will be skimmed in a 5-minute standup.</task>     # context / why
<feedback>{{data}}</feedback>                              # delimited data, at the top
<instructions>
1. Group the feedback into at most 5 themes.
2. For each theme: a one-line description, a severity (high/med/low),
   and how many comments mention it.
3. Rank themes by severity, then by count.
</instructions>
<format>A markdown table: Theme | Severity | Mentions | Example quote.
Then one plain-prose sentence of takeaway.</format>

What changed, and why

Role + context

A role focuses the model; the why ("skimmed in a 5-minute standup") lets it generalize about length and tone without being told each rule.

Delimited data, on top

The feedback is fenced in <feedback> so it can't be mistaken for instructions, and placed first so the model reads it before the task (the long-context placement rule).

Numbered instructions

Sequential steps remove ambiguity about what to produce and in what order — the spec, not a wish.

Explicit output format

Naming the exact format (a table with named columns) is what makes the output consistent run to run and parseable downstream — and it sets up structured output (Lesson 07).

Positive framing

It says what to produce, not a list of "don'ts." If this were a reasoning model, we'd drop any "think step by step" and trust it to reason.

Say it back in one breath

“The first prompt was a wish; the second is a spec. I gave it a role and the reason the report exists, fenced the data and put it on top, numbered exactly what to produce, and named the output format so it's consistent and parseable. Same model — the gain is entirely in the prompt.”

The colleague test passes: hand the second prompt to someone with no context and they know exactly what to make. That's the bar.

★ Review

Summary & self-test

Prompting is writing a spec for a brilliant, literal, context-free collaborator: give it a role and context, fence the data, number the instructions, show a few examples, name the output format — and, for reasoning models, then get out of the way.
