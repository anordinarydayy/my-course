Agent Memory

A one-shot chat lives and dies in a single context window. An agent runs for hundreds of turns, across sessions, and the naive way to keep it coherent — append everything and hope the window is big enough — quietly makes it slower, dumber, and ruinously expensive long before it hits the hard limit. This lesson is about the fix: split memory into a bounded short-term tier and a durable long-term tier, then manage the short one with compaction and context editing while persisting what matters to the long one. Do it well and a long-running agent stays sharp and cheap; do it badly and it forgets the plan on turn 40.

00 The naive default

The context that never stops growing

The simplest possible memory for an agent is no memory system at all: keep the full transcript — system prompt, tool definitions, every user turn, every tool call, every tool result — and re-send the whole thing on every step. For a five-turn chat this is fine. For an agent that reads twelve files, runs forty tool calls, and works a task for an hour, it collapses in three separate, compounding ways.

  • Cost and latency scale with length. The model re-reads the entire history every single turn. At an illustrative Sonnet 4.6 price of $3 per Mtok input (mid-2026), a 150k-token context costs about $0.45 just to read the history — on every step. Prompt caching softens the repeat cost, but a growing window still means more tokens processed, higher latency, and a bigger bill each turn.
  • Quality degrades as the window fills. This is the surprising one, and the reason "just use a bigger model" doesn't save you: accuracy drops as input length grows, even when the answer is sitting right there. More on this in §02.
  • The hard limit ends the run. Eventually the transcript exceeds the context window outright and the agent simply cannot continue — a long task dies mid-flight from context exhaustion.

Anthropic frames the whole problem as an attention budget: every token you add to the window depletes a finite budget. Transformer self-attention has to relate every token to every other token — an n-squared web of pairwise relationships — and models carry fewer specialized parameters for very-long-range dependencies. So context quality is a gradient, not a cliff: it erodes smoothly as you spend the budget, and the discipline of agent memory is spending it wisely.

The way out is to stop treating "the transcript" as "the memory." We split memory into two tiers. Short-term memory is the in-context conversation — system prompt, tools, and message history, bounded by the window and wiped when the session ends. Long-term memory is an external store — files, a vector database, or a structured DB — that the agent reads and writes through tools and that persists across sessions. Everything else in this lesson is techniques for keeping the short tier small and moving what must survive into the long one.

The one-line version

Context is working memory, not storage. Keep the window small and relevant, and externalize anything that must outlive it. A long-running agent isn't "a model with a huge prompt" — it's a model with a disciplined budget plus a store it reads and writes on demand.

01 The split that makes it work

Two tiers of memory ▸

Picture the two tiers as a machine. The context window is short-term memory — fast, always fully visible to the model, and bounded. The external store is long-term memory — slower to reach (it costs a tool call), but effectively unbounded and durable. The agent's job is to keep the window lean and to shuttle durable facts down into the store, then pull them back up when a new session needs them. Step through one lifecycle:

The two tiers have genuinely different physics, and conflating them is the root of most memory bugs:

Short-term — in-context

  • Is the message history inside the window; the model sees all of it, always.
  • Fast and free to "read" (it's already in the prompt), but every token spends the attention budget.
  • Bounded by the window and volatile — it vanishes when the session ends.
  • Managed by shrinking it: context editing and compaction (§03–04).

Long-term — external store

  • Lives outside the window in files, a vector DB, or a structured DB.
  • Costs a tool call to reach, but is effectively unbounded and durable.
  • Keyed per user or project, so it can be recalled in a later session.
  • Managed by writing to it: the memory tool and retrieval (§05).

The invariant that falls out is blunt and worth memorizing: anything not written to the long-term store is lost the moment the window resets. Anthropic bakes this straight into its memory tooling as a rule — "assume interruption" — because a long agent's context can be reset at any time (a crash, a compaction, a new session). If progress lives only in short-term memory, an interruption erases it and the agent repeats work. Externalize first, then continue.

02 The counterintuitive part

Why bigger isn't the fix

The tempting escape hatch is "million-token windows exist — just fit everything in." That fails on the second of our three costs, and it's the least intuitive one: quality degrades as the input grows, even when the relevant evidence is present and the task is trivial. Two well-replicated findings pin this down.

Lost-in-the-middle (Liu et al.) showed that retrieval accuracy is U-shaped by position: a model is best at using information placed at the very start or very end of its context, and worst when the same fact sits in the middle — a drop of more than 30%. It held across GPT-3.5/4, Claude, and later replications across roughly 18 frontier models. The practical rule: never bury a must-follow instruction in the middle of a long prompt — put critical constraints near the top or the bottom.

Context rot (a term popularized by Chroma's 2025 study across ~18 models) went further: accuracy declines as input length grows even when the evidence is fixed, favorably placed, and the task is a trivial lookup. Degradation isn't only about position — it's a property of attention itself, and training reduces but does not remove it. Their sharpest result: on a long-memory benchmark, a focused ~300-token prompt beat the full ~113k-token prompt containing the same answer, and models often did better on a shuffled haystack than a logically structured one. Distractors compound it — one hurts, four hurt more — and lower semantic similarity between the question and the needle makes it worse still.

The lesson under both studies

A token in the window is not free just because it fits. Every irrelevant token you carry dilutes attention and drags accuracy down, on top of the cost and latency it adds. So the goal is not "the most context" — it's the smallest set of high-signal tokens that fully specifies the task. That reframes memory management as a curation problem: continuously prune what no longer earns its place, and page in only what the current step needs.

This is the same instinct as just-in-time context engineering from Lessons 19–20: rather than front-loading everything, the agent holds lightweight identifiers — file paths, record IDs, stored queries — and loads full data into the window only when a step needs it, via tools like grep, head, or a SQL query. Often it's hybrid: load a little up front for speed (a CLAUDE.md, a task brief) and discover the rest on demand, using file names, sizes, and timestamps as cheap signals. Memory management and retrieval are the same discipline pointed at different stores.

03 Lightest touch first

The mitigation ladder

Given a window that's filling up, you have a ladder of techniques, ordered from lightest touch (cheap, low-risk, keeps most context) to heaviest (more machinery, more that can go wrong). Reach for the lightest rung that solves your problem.

  • Context editing / tool-result clearing. The safest first move: prune stale tool outputs from the window while keeping the tool calls and the reasoning. A giant file dump or search result from twenty turns ago is dead weight; swap it for a short placeholder. This is surgical — it clears specific results, not the conversation.
  • Compaction / summarization. When the window nears its limit, condense the old turns into a summary and restart the window with that summary plus the recent tail. Lossy but powerful — the whole of §04.
  • Structured note-taking / scratchpads. The agent writes notes to a file outside the context (a NOTES.md, a to-do list, a progress log) and reads them back on demand. Memory the agent authors itself.
  • Retrieval-based memory. Embed content into a vector store and recall the top-k relevant pieces when needed, instead of carrying it all inline. Long-term memory as search (§05).
  • Sub-agent isolation. Spin off a subagent that explores deeply in its own window and returns only a distilled ~1,000–2,000 token summary — keeping the detailed search context out of the lead agent's window entirely.

Anthropic's rough task-to-technique mapping is a useful default: use tool-result clearing as the low-effort first step; compaction for tasks with extensive back-and-forth; note-taking for iterative development with clear milestones; and multi-agent for research or analysis where parallel exploration pays off. The public numbers are striking: in a 100-turn web-search evaluation, context editing alone cut token consumption by 84% and let agents finish tasks that otherwise failed from context exhaustion; combining the memory tool with context editing scored +39% over a baseline on an agentic-search benchmark (context editing alone: +29%). The lightest rungs pay off first.

The lightest two rungs are one-liners on the Claude Developer Platform — you configure them, the API applies them:

Watch the prompt cache

Clearing tool results rewrites the prompt prefix, which can invalidate the prompt cache and cost you a cache miss. That's what clear_at_least is for: only fire the edit when it reclaims enough tokens to be worth the miss, and use exclude_tools to protect results a later step still needs. Every response returns applied_edits (cleared_tool_uses, cleared_input_tokens) so you can measure what you saved, and /count_tokens previews it beforehand.

04 The workhorse technique

A compaction step, in code ▸

Compaction is the rung you'll reach for most on conversational agents: take a history nearing the window limit, summarize it, and reinitialize a fresh window with the summary plus the most-recent turns. The art is entirely in the summary prompt — write it to maximize recall first (capture architectural decisions, unresolved bugs, key facts, file paths, TODOs), then trim for precision. The 200-turn refactor that kept forgetting the plan

You're building an agent that refactors a codebase over a long session — it reads files, runs tests, edits, re-runs. Around turn 40 the window is packed with stale file dumps and old test output, latency has doubled, cost per step has quadrupled, and it starts contradicting its own earlier decisions because the plan it wrote on turn 6 is now buried in the lost-in-the-middle zone. How do you keep it coherent and affordable for 200 turns?

From the naive default to the rule

Naive — append forever

Keep the entire transcript and re-send it every step. Why it breaks: cost and latency climb linearly, the early plan sinks into the low-attention middle, and by turn ~120 you hit the hard window limit and the run dies. You paid the most tokens for the worst behavior.

The move — compact at a token budget

Track the real input size each turn. When it crosses a BUDGET (say ~80% of the window), split the history into OLD (everything but the last N turns) and RECENT (last N + system, kept verbatim). Summarize OLD with a recall-first prompt, then rebuild the window as summary + recent tail. The thread survives; the token count collapses.

The invariant — recall before brevity

Compaction is lossy. If the summary prompt optimizes for shortness, it silently drops decisions and open bugs and the agent repeats or contradicts itself. So the rule is fixed: maximize recall first (decisions, bugs, key facts, file paths, TODOs), then tighten. And always keep the most-recent turns verbatim — recency is where the active task state lives.

The compaction step, in code

compaction.py — summarize old turns when the window crosses a budgetfrom anthropic import Anthropic

client = Anthropic()
MODEL  = "claude-sonnet-4-6"       # illustrative pricing, mid-2026
BUDGET = 150_000                   # compact once input crosses ~80% of the window
KEEP   = 6                         # newest N turns always survive verbatim

def n_tokens(system, messages):         # never estimate — ask the API
    return client.messages.count_tokens(
        model=MODEL, system=system, messages=messages).input_tokens

def compact(system, messages):
    if n_tokens(system, messages) <= BUDGET:
        return messages              # under budget → leave it untouched
    old, recent = messages[:-KEEP], messages[-KEEP:]
    transcript  = "\n".join(f"{m['role']}: {m['content']}" for m in old)
    summary = client.messages.create(
        model=MODEL, max_tokens=1024,
        system=("Summarize this transcript for an agent that will resume the "
                "task. MAXIMIZE RECALL: keep decisions, unresolved bugs, key "
                "facts, file paths and TODOs; drop redundant tool output."),
        messages=[{"role": "user", "content": transcript}],
    ).content[0].text
    # reinitialize the window: summary as the new floor + the verbatim tail
    return [{"role": "user",
             "content": f"[Earlier context, compacted]\n{summary}"}] + recent
Say it back in one breath

“My agent's window kept growing until it was slow, expensive, and forgetting the plan it wrote 30 turns ago. So before each request I count the real input tokens with the count-tokens endpoint, and when it crosses a budget of about 80% of the window I split off the old turns, summarize them with a prompt tuned for recall — decisions, open bugs, TODOs first — and rebuild the window as that summary plus the last six turns verbatim. The task thread survives, the token count drops by an order of magnitude, and I keep going.”

This is exactly what Anthropic's server-side compaction automates (type compact_20260112, default trigger 150k input tokens) — it summarizes the whole conversation into a compaction block and drops the blocks before it — and what Claude Code does client-side, carrying the summary plus the five most-recently-accessed files. Billing gotcha: the extra summarization pass isn't in the top-level token counts, so you must sum usage.iterations or you'll under-count the cost. As of 2026 Anthropic calls compaction the recommended strategy for long-running conversations.

05 Making memory durable

The memory tool & long-term recall ▸

Compaction keeps a single session alive, but a summary still lives in a window that will eventually vanish. For knowledge that must survive across sessions — a user's preferences, a project's progress log, a startup script the agent keeps rediscovering — you need a real long-term store. Anthropic's memory tool (type memory_20250818, now generally available on the Messages API with no beta header on Claude 4+ models) is the canonical shape, and it's worth understanding because it is entirely client-side: Claude only emits tool calls; your app executes them against storage you control. Enabling it is one line; the responsibility is all yours:

enable the memory tool — GA, no beta header, no input schematools: [ { "type": "memory_20250818", "name": "memory" } ]
# the API auto-injects a "MEMORY PROTOCOL": VIEW /memories before acting,
# and ASSUME INTERRUPTION (context may reset at any time).
# YOU implement the handler for: view · create · str_replace · insert · delete · rename
# and YOU must confine every path to /memories (reject ../ and %2e%2e%2f).
Security is entirely on you

The memory tool executes whatever file operation Claude requests, so a path like /memories/../../secrets.env is a live path-traversal risk. Validate and canonicalize every path, confine strictly to /memories, cap file sizes, strip sensitive data before writing, and expire stale files. This is the same "never trust model-generated arguments" discipline from tool use (Lesson 24) and MCP (Lesson 25) — a memory store is just another tool with a dangerous surface.

The memory tool is file-based long-term memory, but it's one of three shapes. Retrieval-based memory embeds content and recalls the top-k relevant pieces on demand — Mem0's add()/search() API over ~20 vector backends reports LoCoMo 92.5 at only ~6,956 tokens per query, versus ~26,000 tokens for stuffing the full history in context. OS-style paging (Letta/MemGPT) moves data between an in-context "main memory" and external "archival memory" like virtual memory. And a humbling result to keep you honest: Letta's 2025 study found a plain filesystem agent (grep, open, search) hit 74.0% on LoCoMo, beating a specialized graph-memory system's 68.5% — agent capability and tool familiarity often matter more than an elaborate memory framework.

Finally, the axis that decides which store you even need: session vs cross-session. Session memory lives in one window and is fine to lose; cross-session memory must be externalized and keyed per user or project. The open, unglamorous production problem is identity resolution — tying memories to the right user across anonymous sessions and multiple devices. Get it wrong and you get wrong-user memory bleed, still a real failure mode in 2026 (per Mem0's state-of-memory report). OpenAI's ChatGPT memory has been climbing this curve — from a manually-curated saved-memory list to a background "Dreaming" process that synthesizes across years of chats — but programmatic read/write for third parties remains largely anticipated, not shipped.

06 One stack, three jobs

Assembling the memory stack

The single most common mistake is treating these as competing choices — picking "compaction" or "the memory tool" as if one replaces the other. They solve different problems, and a serious long-running agent layers all three:

Context editing

  • Client-side; surgically clears specific stale tool results from the window.
  • Solves: dead weight from old file dumps and search output.
  • Cheapest, lowest-risk — the first rung.

Compaction

  • Server- or client-side; summarizes the whole conversation near the limit.
  • Solves: a long back-and-forth that would otherwise exhaust the window.
  • Recommended default for long chats; lossy, so recall-first.

Memory tool

  • Externalizes durable knowledge to a store keyed per user/project.
  • Solves: what must survive summarization and the session ending.
  • Persists across sessions; you own its security.

The recommended stack combines them: compaction (or context editing) to keep the active window small, plus the memory tool to preserve what must survive being summarized away. A concrete multi-session pattern makes it vivid: an initializer session writes memory files up front (a progress log, a feature checklist, a startup-script reference); each later session opens by reading them to restore state without re-exploring the codebase, works one feature end-to-end, verifies it, and closes by updating the progress log. Because state is externalized, an interruption costs nothing — the next session just reads the log and resumes.

Don't over-engineer — a plain filesystem beat a graph-memory framework on LoCoMo (74.0% vs 68.5%). Start with notes + compaction; add vector recall only when you've measured that you need it.

And the payoff is measurable in dollars. That 150k-token window costing ~$0.45 per turn to read (illustrative Sonnet 4.6 input, mid-2026) drops to ~$0.05 after compaction to ~18k tokens — a 9× cut, on every subsequent step, plus the accuracy you win back by escaping context rot. Memory management isn't a nicety you bolt on at the end; it's what makes a long-running agent viable at all.
