Observability for AI

Phase 7 — Architecture, Deploy & Operations — is about running the system once real users depend on it, and this is its ★Core discipline: you cannot operate, let alone improve, what you cannot see. An ordinary API call is one box you can watch with plain APM. An LLM or agent request is a different animal — a tree of nondeterministic steps: context retrieval, prompt assembly, the model call, each tool, each sub-agent, output parsing — and any node can quietly produce a wrong, slow, or expensive answer. AI observability makes that tree visible, and it stands on three legs: tracing (the span tree), metrics (tokens, cost, latency, quality), and online evals run on real production traffic. Get it right and it closes the loop back into every improvement the rest of the course teaches.

Original study notes. The operations lesson of Phase 7 — how you watch an LLM system once live traffic is hitting it. 00 The gap

You shipped it. Now something's wrong.

A user files a ticket: "the assistant gave me the wrong order status." In an ordinary service you'd open that request in your APM, see the one handler that threw, read the log line, and be done in two minutes. Here you open your dashboard and all you have is average latency 2.9s, error rate 0.4%, requests/min. None of it tells you what happened inside that one request — because inside that one request, a dozen things happened, and the response came back a confident, well-formatted, HTTP-200 lie.

That is the whole reason AI observability is harder than normal APM: an LLM or agent request is not a call, it is a tree. One user turn fans out into context retrieval, prompt assembly, the model call itself, maybe several tool calls, maybe a sub-agent or two, then output parsing — each step nondeterministic, each able to be the thing that broke. Worse, the numbers you do have are actively misleading. A single averaged latency figure hides the entire tree; it is the "averages lie" problem from L35 in a new costume. You need per-span latency plus percentiles (p50/p95/p99), and for streaming you need the TTFT (time to first token — queueing plus prefill) split out from TPOT (time per output token — steady-state decode), because a healthy-looking mean can hide a p99 that times out half your power users.

The naive instinct is to log the final prompt and the final answer to a text file. That captures the two ends and nothing in the middle — which retrieved chunk was junk, which tool timed out, which sub-agent looped, where the tokens and the dollars actually went. To debug the wrong answer and to feed the improvements later phases teach, you have to see the middle.

The one-line version

AI observability = tracing + metrics + online evals, in production. Tracing gives you the tree (which span went wrong). Metrics give you the aggregates over time (tokens, cost, latency, quality). Online evals score real traffic so you learn about failures your offline test set never imagined. Miss any one leg and you are operating blind on that axis — and with a nondeterministic system, blind on one axis is blind.

01 The insight

One request is a tree of spans ▸

The unit that makes the middle visible is the span: one timed step with a start and end, a parent, its inputs and outputs, and a bag of attributes. Wrap every step of a request in a span and the request becomes a trace — a tree you can walk. Give each span its tokens, cost, latency, and status and the tree answers the two questions the dashboard couldn't: which step produced the bad answer, and where did the cost and the seconds actually go. Step through a real trace and watch a vague "it's wrong" collapse onto a single span:

Read the tree top-down and the debugging is almost mechanical. The root span is the whole request. The model span is usually your cost and latency center — here it's 3.9s of the 4.8s and nearly all the tokens. But the wrong answer didn't come from the model at all: the retrieval span returned zero chunks, so the model dutifully answered from nothing. Without the tree you'd have blamed the model, swapped it for a bigger one, and fixed nothing. With it, "the assistant is wrong" becomes "the retrieval span returned junk," which is a bug you can actually go fix. The same tree does attribution for free: sum the spans by feature or by user and you know precisely what each is spending in tokens, dollars, and seconds — the averaged 2.9s meant nothing; the per-span breakdown names the culprit.

The debugging move

When an answer is wrong, don't ask "is the model bad?" — ask "which span went wrong?" A trace turns an unfalsifiable complaint into a falsifiable one. Retrieval span empty → fix retrieval. Tool span returned an error string the model trusted → fix the tool. Parsing span choked on valid output → fix the parser. The model span is the last thing you should suspect, and the tree tells you when it actually is the problem.

02 The wire format

Speak the standard: OpenTelemetry GenAI

A trace is only useful if your tools can read it, so the real question is: what do you call these attributes? The wrong answer is "invent your own" — then your traces work in exactly one dashboard, the one you hand-built, and moving backends means re-instrumenting everything. The converging answer is OpenTelemetry's GenAI semantic conventions: a vendor-neutral vocabulary of gen_ai.* span attributes so the same trace flows, untranslated, into your existing APM and into any LLM-observability backend.

Concretely, a model span carries gen_ai.operation.name, gen_ai.provider.name (the newer attribute that replaces the now-deprecated gen_ai.system), gen_ai.request.model, gen_ai.request.temperature, and gen_ai.request.max_tokens; on the way back it sets gen_ai.response.model, gen_ai.response.id, gen_ai.response.finish_reasons, and the two that drive every cost report — gen_ai.usage.input_tokens and gen_ai.usage.output_tokens. The gen_ai.operation.name is an enum, and its values are exactly the tree from Figure 1: chat, generate_content, text_completion, embeddings, execute_tool, invoke_agent, create_agent, plus retrieval and invoke_workflow. In other words the standard now natively models agents, tool calls, and RAG: a tool span sets gen_ai.tool.name and gen_ai.tool.call.id; an agent span sets gen_ai.agent.name and gen_ai.conversation.id. And it isn't only spans — there are metric instruments too: gen_ai.client.operation.duration (a latency histogram) and gen_ai.client.token.usage (a token histogram), the vendor-neutral names that feed any backend's charts.

Emit gen_ai.* (portable)

  • One instrumentation, any backend — APM + LLM-obs read the same trace.
  • Datadog LLM Observability ingests OTel GenAI semconv natively (v1.37+); LangSmith added full OpenTelemetry support in March 2025; Langfuse, Arize Phoenix, and Braintrust all speak it.
  • "Emit once, route anywhere" is real — switch vendors without touching the app.

Roll your own (locked in)

  • Attribute names only your one dashboard understands.
  • Every new backend = re-instrument; every tool that expects gen_ai.* silently sees nothing.
  • You reinvent the enum the community already agreed on — and get the agent/tool/RAG modelling wrong.
Still experimental — don't call it GA

As of mid-2026 the GenAI conventions are the de-facto standard but officially still in “Development” status — every gen_ai.* attribute is marked experimental, and the spec was moved out of the main repo into a dedicated one (open-telemetry/semantic-conventions-genai). Attributes are still being renamed (that gen_ai.system → gen_ai.provider.name swap is a live example). Existing instrumentations still default to the old attribute shape at the v1.36 baseline; you opt into the new one with OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental. Emit it, pin your instrumentation version, and expect churn — just don't tell your team it's frozen.

One more thing the standard gets right by default, and it's a safety property, not a limitation: OTel captures no prompt or response content unless you turn it on. The content attributes exist — gen_ai.input.messages, gen_ai.output.messages, gen_ai.system_instructions — but they are explicit opt-in, precisely because prompts and completions routinely carry names, emails, and secrets. That default is the seed of the logging rule in Section 04.

03 The three pillars

Traces, metrics, evals — three lenses ▸

Tracing tells you what happened in one request. Operating a system needs two more lenses. Metrics aggregate across requests — token and cost per request, error and refusal rates, TTFT/TPOT, p50/p95/p99 latency, rolling quality scores — so you see trends, not anecdotes. Online evals answer the question metrics can't: was the answer any good? You run an LLM-as-judge or a cheap heuristic scorer on a sampled slice of live traffic and attach the score to the exact trace. Watch all three feed one view:

The three are complementary, not redundant. Traces debug one request; metrics spot a trend; online evals catch quality problems that never surface as errors — the model returns a confident, well-formatted, completely wrong answer with a clean 200, and only a scorer notices. Curated offline eval sets (Phase 2) can't cover this, because production sends inputs you never imagined: distribution shift, weird phrasing, emergent failure modes. That is exactly why sources call online eval a core component of observability rather than a nice-to-have.

Traces

  • Q: what happened in this request?
  • the span tree, per-step I/O, tokens, cost, latency, status.
  • your debugger.

Metrics

  • Q: what's the trend?
  • cost/req, error & refusal rate, TTFT/TPOT, p50/p95/p99.
  • your trendline and alarms.

Online evals

  • Q: was it any good?
  • LLM-judge / heuristics on sampled live traffic, trace-linked.
  • catches the confident-but-wrong 200s.
Trace-linked evals are the glue

The three pillars are only powerful when they share a key. Attach every eval score to the exact trace that produced it, and a dip in the aggregate quality metric is one click away from the individual traces that caused it — and from the exact spans inside them. A quality number with no trace behind it is a mood ring; a quality number linked to its traces is a work queue.

04 The rule

What to log — and what never to

So what actually goes into a trace? The checklist for every request is short and worth memorizing: the full span tree; the prompt and response (scrubbed — see below); the model and its exact snapshot; the params (temperature, top_p, max_tokens); token usage broken out (input / output / cached / reasoning); the dollar cost; per-span latency; the user / session / conversation id; the eval scores; and any user feedback (thumbs up/down, regenerate, edits). That last one is gold — a free label from the person best placed to judge the answer, so capture it as a first-class signal, not a footnote.

The online-eval mechanics deserve a word, because you don't judge everything — the judge is itself an LLM call and it costs money. A typical split is an LLM-judge on ~10–20% of live traffic, 100% of your CI regression cases run offline, and ~2–5% pulled into a human annotation queue for the high-uncertainty and edge cases. And the judge is nondeterministic and can be biased, so treat it like a measuring instrument: pin its exact version, calibrate it against human labels, and use the same rubric online as offline — otherwise your "quality score" drifts for reasons that have nothing to do with your system, and you chase ghosts.

The PII rule

Raw prompts and responses routinely contain names, emails, order numbers, and secrets. Log them verbatim and your observability store becomes a breach waiting to happen — which is exactly why OTel captures no content by default. Scrub or redact before it leaves the process, and treat content capture as a deliberate, governed opt-in — a decision with an owner and a retention policy, not a flag someone flips "to debug real quick" and forgets to flip back.

05 The failure mode

Drift: when the ground moves under you ▸

Here is the production failure with no code change attached to it. You pinned nothing; you pointed production at a model's latest alias; and one morning the provider silently retuned it behind that alias — different output format, different reasoning style, different tool-call ordering. Your tests are green (they test your code). Your error rate is flat (they're all 200s). And your quality quietly falls off a cliff. The only thing that sees it is an online eval score plotted against time. Watch the drift get caught:

The defense is a handful of habits. Pin production to a dated model snapshot (a model-YYYY-MM-DD-style id — illustrative, not a real name) so it cannot change under you; run a canary on the latest alias and diff both against a held-out eval suite so you learn the new behavior on your terms. Alert on all three regression classes, not just one: quality (eval score, malformed-output rate — deterministic pass/fail checks trip earliest), cost (a longer prompt or a chattier model swap quietly inflates spend), and latency (TTFT/TPOT). Watching quality alone while cost and TTFT blow up is a classic and expensive miss.

And treat every model or prompt change as a deployment, because it is one. Shadow the new version on duplicated live traffic and compare offline; then canary a real ramp — 5% → 10% → 25% → 50% → 100% — running online eval at each step with auto-rollback; then A/B distinct alternatives once things are stable. Only promote when the quality delta is statistically real above the noise, because outputs are nondeterministic and a small or noisy sample will happily lie to you.

Shadow

  • duplicate live traffic to the new version.
  • users never see it; compare outputs offline.
  • zero user risk, no promotion decision yet.

Canary

  • ramp real traffic 5→10→25→50→100%.
  • online eval + cost + latency + refusal at each step.
  • auto-rollback on regression.

A/B

  • compare distinct alternatives head-to-head.
  • after stability is confirmed, for a real choice.
  • needs enough samples for significance.

The cheapest place to wire all of this in is an LLM gateway. LiteLLM (open-source, OpenAI-compatible proxy to 100+ providers, self-hosted, zero markup), OpenRouter (one key to 300+ models for a ~5–5.5% credit fee), Portkey (routing + automatic fallback + semantic caching — vendors claim up to ~40% spend reduction on repeated queries — with a real-time cost/latency dashboard by feature/user/model), and Helicone (open-source gateway plus observability) all sit between your app and the providers. The gateway is the natural emission point for gen_ai.* telemetry and, crucially, for per-feature and per-user cost attribution — cost is a first-class observability dimension there, right beside latency and quality; the managed-cloud equivalents are AWS Bedrock and Google Vertex AI.

A prompt edit is a deploy

The most common own-goal is treating a prompt change as "just a config tweak." It isn't — it changes model behavior as surely as a code change, and it deserves the same shadow test, canary ramp, and drift alarm. Ship a prompt the way you'd ship code, and the "we changed one word and quality tanked" incident stops happening.

All of this closes a loop, and the loop is why you observe. Production traces → an online judge flags the low-quality and low-confidence ones → they route to a human annotation queue → annotators turn them into a golden dataset → that dataset fine-tunes a smaller, cheaper model or tightens a prompt → you deploy and observe again. This is the data flywheel: it's how the improvements from the rest of the course actually reach production. NVIDIA's NeMo-microservices flywheel blueprint, paired with observability vendors like Arize, reports smaller fine-tuned models hitting ~94–96% of a large model's accuracy at a fraction of the cost — the payoff of taking the loop seriously rather than observing for its own sake.

06 Put it together

Build the observability you didn't have

Back to the wrong-answer ticket from Section 00 — except this time you build the instrumentation that would have solved it in a minute instead of an afternoon. The worked example wraps each step of the request in a span with OpenTelemetry GenAI attributes, ships the trace to an LLM-obs backend, scores a sampled slice with an online judge, and alarms on the score. Then you re-run the failing request and simply read the tree.

Restate & what you have (nothing)

The ticket says "wrong answer"; your tools give you an average latency and an error rate. That average can't say whether the retrieval, model, tool, or parsing step failed — and the naive fix (log the final prompt + answer) captures the two ends and none of the middle, which is where the bug lives.

The move — a span per step, standard attributes

Wrap every step in a span and set gen_ai.* on each. On the model span, recording gen_ai.response.model means a silent provider swap becomes visible, not invisible; gen_ai.usage.* plus app.cost_usd make token and dollar attribution fall out for free; per-span latency and status localise the slowdown. Ship the trace to any OTel backend and the request is now a walkable tree.

Close the loop — online eval + alarm

Sample ~15% of production and run an LLM-judge with a pinned version and a fixed rubric; write the score onto the trace (trace-linked) so a dip is one click from its cause. Alert when the rolling score drops or the malformed-output rate spikes. Re-run the ticket: the tree shows the retrieval span returned 0 chunks — "it's wrong" is now "retrieval returned junk," a one-line fix.

Say it back in one breath

“I couldn't debug a wrong answer because a request is a tree of nondeterministic steps and all I had was an average. So I wrap each step — retrieval, model, tools, parsing — in a span with OpenTelemetry gen_ai.* attributes carrying tokens, cost, latency, and status; ship the trace to an LLM-obs backend; run an online LLM-judge on a sampled slice of live traffic and attach the score to the trace; and alert when the score drifts. Now ‘the assistant is wrong’ localises to ‘the retrieval span returned nothing,’ and a silent model update shows up as a sagging eval line instead of a mystery.”

Tracing to debug one request, metrics to see the trend, online eval to catch the confident-but-wrong 200s, drift alarms to catch the ground moving. That is what operability means for something nondeterministic.

That's the whole discipline in one file — spans with standard attributes, shipped to a backend, scored online, alarmed on drift. It is also the substrate for the rest of Phase 7: your cost dashboards read the same gen_ai.usage.* attributes, your canary rollout diff reads the same eval scores, and the data flywheel reads the same low-scoring traces. Observability isn't a thing you bolt on at the end — it is the sense organ the rest of operations runs on, and the reason you can honestly say you're improving the system rather than just changing it.

★ Review

Summary & self-test

One idea runs through the lesson: an LLM/agent request is a tree of nondeterministic spans, so you make it visible with tracing (which span went wrong), metrics (the trend in tokens, cost, latency, quality), and online evals on live traffic (was it good?) — emit the OpenTelemetry gen_ai.* standard so it routes anywhere, scrub PII, alarm on quality/cost/latency drift, and feed the low-scoring traces back into the improvement loop.
