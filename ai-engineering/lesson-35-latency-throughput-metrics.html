Latency & Throughput Metrics

Phase 6 — Inference Optimization: Latency & Cost — opens with the one skill every optimization depends on: measurement. You cannot make inference faster or cheaper until you can say, precisely, what “faster” means. This is the ★Core lesson of the phase, because its whole payoff is a single realization: the metrics only start to make sense once you see that autoregressive generation runs in two phases with opposite hardware bottlenecks. Learn to read TTFT, TPOT, throughput and goodput through that split, and every knob in the rest of Phase 6 — batching, quantization, disaggregation, caching — becomes a lever you can aim instead of guess at.

00 The ceiling

You can't optimize what you can't measure

Someone asks how fast your model serves and you say “about 40 tokens per second.” It sounds like an answer. It is barely a question restated. Forty tokens per second where — the burst when the first token lands, or the steady trickle after? Measured for one user alone on the box, or averaged across the sixty who were sharing it? Is that a typical request or your worst one? A single throughput or latency number is the naive instinct, and it fails the same way a thermometer reading the average temperature of a burning house and a freezer fails: the mean is real and completely useless for deciding what to fix.

The reason a lone number can’t carry the weight isn’t sloppiness — it’s that generation is not one activity. It is two, glued end to end, and they stress opposite parts of the GPU. The wait before the first word appears and the pace at which the rest stream out are produced by different work, bounded by different hardware limits, and fixed by different techniques. Collapse them into one average and you have thrown away the exact structure you needed to act. So the discipline for this whole phase is: measure first, and measure in a way that respects the two phases. Get that, and TTFT, TPOT, throughput, and goodput stop being jargon and start being a diagnosis.

The number that lies

  • “40 tokens/sec.” Prefill burst or decode trickle? You can’t tell.
  • “Average latency 800 ms.” An average over a heavy tail — the users who suffered are invisible.
  • One phase assumed. No split means you can’t say which stage to optimize.
  • Load unstated. Solo on the GPU, or batch of 64? The number changes by 10×.

The numbers that help

  • TTFT (p95) — time to first token: does it feel responsive?
  • TPOT (p95) — the gap between streamed tokens: does the text flow?
  • Throughput at a stated batch size — tokens/sec and requests/sec across everyone.
  • Goodput — requests/sec that actually met their SLO. Cost and quality in one figure.
The one-line version

There is no single “speed” of an LLM. There is a first-token latency set by the prefill phase and a per-token cadence set by the decode phase — and a throughput that trades against both. Every metric in this lesson is one of those three, reported at a percentile. Learn the two phases and the metrics arrange themselves.

01 The insight

Generation runs in two phases ▸

An LLM generates autoregressively: to produce token t it must have already produced tokens 1…t−1. That one constraint splits a single request into two phases that could not be more different in what they ask of the GPU. First comes prefill: the model ingests the entire prompt in a single forward pass, computing the attention keys and values for every prompt token at once and caching them. Because all the prompt tokens are known up front, this is a big, dense, embarrassingly parallel matrix multiply — the GPU’s tensor cores run flat out. Prefill is compute-bound, and its cost scales with prompt length: a 200-token prompt prefills quickly, a 20,000-token prompt makes you wait.

Then comes decode. Now the model emits one token per forward pass, appending it and looping, re-using the cached keys and values (the KV cache) so it never re-reads the whole prompt. But here is the twist that governs everything downstream: to generate each single token, the GPU must stream the model’s entire set of weights out of high-bandwidth memory (HBM) and through the compute units — and it does so to produce essentially one column of output. The arithmetic is trivial; the data movement is enormous. Decode is memory-bandwidth-bound: its speed is fixed not by how many FLOPs the chip can do but by how fast it can read weights from HBM. Watch the two phases run:

This is not an abstraction — it is the single most useful fact for reasoning about inference speed. Because decode is bandwidth-bound, the most direct lever on how fast a lone user’s tokens stream is HBM bandwidth, not raw FLOPs. Illustratively, HBM bandwidth climbs from roughly 3.35 TB/s on an H100 (HBM3) to 4.8 TB/s on an H200 (HBM3e) to 8 TB/s on a B200 — and single-request decode speed for a 70B model in FP16 tracks it almost linearly: on the order of ~24 → ~34 → ~57 tokens/sec at batch 1, same model, same everything, just faster memory. Prefill, being compute-bound, responds to a different lever (more FLOPs, more parallelism). One request, two bottlenecks — and every metric below is measuring one side of that split.

Prefill

  • Reads the whole prompt in one parallel forward pass.
  • Compute-bound — tensor cores saturated.
  • Cost scales with prompt length.
  • Produces the first token → defines TTFT.

Decode

  • Emits one token per pass, re-using the KV cache.
  • Memory-bandwidth-bound — all weights streamed from HBM each step.
  • Cost scales with output length.
  • Sets the per-token cadence → defines TPOT.
02 The vocabulary

TTFT, TPOT, throughput, goodput

With the two phases in hand, the metrics practically name themselves — each is a stopwatch on one side of the split. TTFT (time to first token) is the wait from when the request arrives to when the first token appears. It is essentially queue delay plus prefill time: how long the request sat waiting for a slot, plus how long it took the model to chew through the prompt. TTFT is what makes an app feel alive or sluggish — the beat before anything happens — and because prefill scales with prompt length, a bloated system prompt shows up here and nowhere else.

TPOT (time per output token), also called ITL (inter-token latency), is the decode cadence: the average gap between successive streamed tokens once generation is underway. It is the reciprocal of a single user’s token rate, and since decode is bandwidth-bound, TPOT is where memory speed, model size, and batch pressure all register. Put them together and you get the standard decomposition of total latency for a request: total ≈ TTFT + TPOT × (output tokens). Two metrics, one phase each, and the whole end-to-end wait falls out of them — which is exactly what the next section builds up brick by brick.

The last two metrics zoom out from one request to the whole fleet. Throughput is total work per second across all concurrent users — output tokens/sec and requests/sec — and it is fundamentally a utilization and cost measure: high throughput means each expensive GPU-second is serving more people, so your cost per token drops. But throughput alone can look wonderful while individual users suffer, so the metric that has taken over as the real serving target is goodput: the rate of completed requests that actually met their SLO — both the TTFT target and the TPOT target. Goodput folds cost (utilization) and quality (SLO attainment) into one number, which is why the disaggregation work below optimizes for it directly rather than for raw throughput.

03 The budget

Where the milliseconds actually go ▸

The formula total ≈ TTFT + TPOT × output looks tidy, but its whole lesson is in the proportions — and they are wildly lopsided in a way that surprises people. A request’s wall-clock time is a stack: a bit of queueing, then prefill (together, the TTFT), then a long run of decode steps, one per output token. For any answer of real length, that last term dominates so completely that the first two are almost rounding error. Sit with the ending. A perfectly reasonable request — a 2,000-token prompt, a 500-token answer, a snappy 300 ms TTFT and a healthy 20 ms TPOT — still takes ~10.3 seconds end to end, and roughly 97% of that is decode. This immediately tells you where optimization pays and where it doesn’t. Shaving the prompt to make prefill faster nibbles at a term worth a few hundred milliseconds; halving the TPOT or halving the number of output tokens each cuts seconds. When latency hurts and the answers are long, the money is almost always in the decode phase — a conclusion you would never reach from a single “average latency” number, and one we’ll turn into a concrete diagnosis in section 06.

04 The honest number

Averages lie — report percentiles

Suppose your dashboard says the average TTFT is 400 ms. Comforting — and possibly a disaster. Serving latency distributions are heavy-tailed: most requests are fast, but a meaningful slice hit a full queue, an unlucky co-tenant in the batch, or a long prompt, and waited many times longer. A good mean can sit on top of a p99 of four seconds, and it is the tail — the p95 and p99 — that users actually feel and remember. This is why serious inference reporting never quotes a bare average; it quotes p50 / p95 / p99. The p50 tells you the typical experience, the p95 and p99 tell you how bad the bad days are, and the gap between them tells you how consistent your service is.

There is a subtler trap that even percentiles can miss if you compute them on the wrong quantity: “normalized latency,” the practice of averaging TPOT over a whole request. Averaging smooths — and smoothing is exactly what hides an intermittent stall. A decode stream that flows at a crisp 20 ms per token but freezes for a couple of seconds mid-answer can post a perfectly acceptable mean TPOT while delivering a visibly janky experience. This is not hypothetical: work like Etalon documented how a serving engine can suffer a multi-second decode stall — on the order of ~10 seconds — when a long prefill request is onboarded into an ongoing decode batch, a freeze that a mean-TPOT number launders away entirely. The fix is to inspect the distribution of inter-token latencies, not just its average, and to report tail ITL alongside tail TTFT.

Pitfalls that quietly corrupt a benchmark

Don’t report only means — a great average can hide a terrible p99. Don’t trust normalized/averaged TPOT — it masks stalls; look at the ITL distribution. Fix input and output lengths across runs — TTFT depends on prompt length and queue, TPOT and throughput on output length and batch composition; mixing them makes cross-system comparisons meaningless. Warm up first — the initial calls after model load pay for CUDA-graph compilation and cold caches and will inflate your numbers. And don’t benchmark under steady load only — real traffic is bursty, and steady-state figures hide the queueing blowups and SLO violations that only appear under a spike.

The through-line: an average is a summary that assumes the thing being summarized is well-behaved, and inference latency is not. Percentiles — on the right quantity, under realistic load — are the honest reporting unit. Everything downstream, including whether an optimization “worked,” is judged at the tail.

05 The trade-off

Latency versus throughput ▸

Now the central tension of all inference serving, and it falls straight out of section 01. Decode is memory-bandwidth-bound: for a single request the GPU streams every weight out of HBM to produce one token, so the tensor cores mostly sit idle waiting on memory. That waste is an opportunity. If you batch several requests together, each weight loaded from HBM is re-used across all of them in the same pass — you moved the bytes once and did many requests’ worth of math with them. In roofline terms, batching raises arithmetic intensity (FLOPs performed per byte moved), sliding the workload out of the memory-bound region toward the compute-bound region and lifting aggregate throughput. There is the whole trade-off in one dial. Batching is the primary lever you have to buy throughput, and it works — right up to the HBM-bandwidth ceiling, which is a hard physical limit: you cannot move weights faster than the memory allows, so beyond some batch size aggregate throughput flattens no matter how many more requests you cram in. And the cost is paid by the individual: every extra request sharing a pass makes each request’s TTFT and TPOT worse, so per-user speed regresses even as the system-wide tokens/sec looks glorious. This is why “tokens/sec per user” has emerged as a product-facing metric distinct from aggregate throughput, and why the optimization target is goodput — you push the batch up only as far as the SLO still holds, not as far as the throughput graph keeps rising.

Push the batch up

  • Each HBM weight-load serves many requests → throughput ↑.
  • GPU utilization rises → cost per token ↓.
  • Climbs only to the HBM-bandwidth ceiling, then flattens.

…but the user pays

  • More requests per pass → per-user TTFT & TPOT ↑.
  • “Tokens/sec per user” falls as system tokens/sec rises.
  • Tune the batch to goodput (SLO-meeting req/s), not max throughput.
06 The playbook

The 2026 serving playbook — and how to diagnose lag

Once you accept that a request is two phases fighting for the same GPU, the whole 2025–2026 serving stack reads like a set of moves to stop them interfering. The headline shift is prefill/decode disaggregation: run prefill and decode on separate GPU pools, have the prefill nodes process prompts and ship the KV cache over the network to the decode nodes. Now you can size and tune the compute-heavy prefill pool for TTFT independently of the bandwidth-heavy decode pool for TPOT, and neither stalls the other. Pioneered by DistServe (which coined the goodput framing) and Splitwise, it is by mid-2026 the de-facto standard, supported across vLLM, SGLang, TensorRT-LLM, LMDeploy, and NVIDIA Dynamo (with LMCache moving the KV bytes). Its lighter-weight cousin, chunked prefill (from the Sarathi-Serve lineage), keeps a single pool but splits long prompts into 256–512-token chunks and interleaves them with ongoing decode steps — trading a slightly higher average TTFT for much better tail latency, and now the default in mainstream schedulers.

Two cautions keep this honest. First, none of these are free wins measured naively: speculative decoding that clocks 4–6.5× on a single stream at temperature 0 collapses toward ~1.8–2.5× once real batching keeps the GPU busy (and shrinks further as the batch grows), and FP4/NVFP4 gains are hardware-gated to Blackwell and can cost accuracy on sensitive layers. Second, prompt-cache pricing and TTLs differ by vendor and change constantly (Anthropic’s explicit cache charges a 1.25× write premium for a 5-minute TTL and 2× for an hour, with a ~1,024-token minimum; OpenAI’s prefix cache is automatic and won’t guarantee a hit). The point of the whole table is not the numbers — it is that each lever targets a specific phase, so you can only choose the right one after you have diagnosed which phase is hurting.

Reason about itTTFT vs TPOT · a laggy chat app
Is it a first-token problem or a streaming problem?

Users say your chat assistant “feels slow.” Two completely different complaints hide inside that one word: some say “it takes a beat before anything appears,” others say “the text dribbles out.” Same adjective, opposite fixes. Before you touch a single knob — which is it?

Restate → diagnose → compute the budget → the rule

Restate the problem

“Slow” is two metrics wearing one word. A pause before the first word is high TTFT (queue + prefill). A slow trickle after the first word is high TPOT (decode cadence). You cannot fix what you haven’t separated — so measure both, at p95, first.

Diagnose a TTFT problem

First token is late but streaming is fine? That’s prefill. The usual culprit is a long system prompt — thousands of tokens of instructions and few-shot examples that must be prefilled on every turn. Fixes live in the prefill phase: shorten/restructure the prompt, or turn on prompt/prefix caching so the shared prefix’s KV cache is reused (cached reads ~0.1× input, up to ~80–85% latency cut on long prompts).

Diagnose a TPOT problem

First token is quick but the text crawls? That’s decode — bandwidth-bound. Fixes live in the decode phase: a smaller or quantized model (FP8/FP4 moves fewer bytes from HBM), better serving (speculative decoding, ~1.8–2.5× under batching), or a higher-HBM-bandwidth GPU.

The rule / trade-off

Diagnose the phase, then pick the lever for that phase — and let the budget tell you where the leverage is. Cutting output length or TPOT beats shaving the prompt almost every time, because decode dominates the total.

Say it back in one breath

“‘Slow’ was hiding two metrics. A pause before the first token is a TTFT problem — a bloated prompt inflating prefill — so I shorten it or cache the prefix. A slow trickle after is a TPOT problem in decode — so I quantize, speculate, or move to faster HBM. And the budget says the answer’s length and per-token speed dwarf the prompt, so I chase decode before I chase prefill.”

Watch p95 TTFT for “does it feel responsive” and p95 TPOT for “does it flow” — never a single average of the two.
