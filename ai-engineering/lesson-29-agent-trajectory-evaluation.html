Agent & Trajectory Evaluation

Everything in the evaluation phase graded one thing: the final answer. For a single-turn model that is the whole job — the output is the behavior. An agent breaks that assumption. It runs a multi-step trajectory of tool calls that changes the world before it ever speaks, so a right answer reached by a wrong, looping, or ruinously expensive path is a latent failure that end-answer eval waves straight through. This lesson is the discipline of grading not just what the agent said, but the path it took and the state it left behind.

00 The latent failure

The right answer, the wrong path

Everything you built in the evaluation phase graded a single artifact: the output. Feed the system an input, capture what it returns, score that return against a reference or a rubric. For a single-turn task — summarize this, classify that, answer this — that is exactly correct, because the output is the entirety of what the system did. An agent quietly voids that assumption. It doesn't emit an answer so much as run a trajectory: a multi-turn sequence of tool calls, observations, and reasoning steps that mutates real state — a row in a database, a file on disk, a charge on a card — before it says a word to you. Grade only the sentence it ends on and you are scoring the last frame of a movie you never watched.

Make it concrete. Two agents are told to book a flight; both finish with the identical line, "Your flight is booked!" Agent A called search_flights once, called book_flight once, got a confirmation number — three clean steps. Agent B called search_flights nine times, booked the wrong fare, cancelled it, rebooked, and burned forty thousand tokens and thirty seconds getting there. End-answer eval hands both a green check, because the final answers are byte-for-byte the same. Yet one of these is a production incident wearing a passing grade: it happens to work on this input today and will loop, overspend, or strand a user tomorrow. This is the right answer, wrong path failure, and it is measured, not hypothetical — the 2025 TRACE study ("Beyond the Final Answer") found models routinely arrive at a correct final answer through fundamentally flawed intermediate reasoning (and correct reasoning to a wrong answer), so final-answer-only metrics substantially underestimate how unreliable an agent actually is, especially on out-of-distribution problems and answers you can't independently verify.

The way out is to stop conflating three different things. The transcript (also called the trace or trajectory) is the complete record of the run — every tool call, argument, intermediate result, and reasoning step. The outcome is the real state of the world afterward — the reservation row that actually exists — which is different again from what the agent claims ("booked!"). End-answer eval grades the claim alone. Agent evaluation grades the transcript (the path) and the outcome (the state), and treats the claim as the least trustworthy of the three.

The one-line version

For a single-turn model, the output is the behavior. For an agent, the output is only the last line of the behavior. Evaluate the whole trajectory and verify the outcome — or you're grading a movie by its final frame and trusting the narrator that it ended well.

01 Claim, transcript, state

Claim vs outcome ▸

The single most common latent failure in a deployed agent is the confident lie: the model says it did the thing, and the thing did not happen. A final-answer grader — string-match the reply, or even judge the reply with an LLM — reads only what the agent says, so it cannot tell a real booking from a hallucinated one. The fix is an outcome check that ignores the claim and interrogates the environment itself. Watch the same run pass one grader and fail the other:

The outcome check is the non-negotiable floor of agent evaluation: a confident final answer is a hypothesis about the world, not evidence, so you verify the state the task was supposed to produce — the row exists, the file was written, the ticket is genuinely closed. But the outcome alone still can't see the path, and that's the second half of the story. Anthropic's framing splits your scorecard cleanly: outcome metrics are state checks that prove success, and process metrics read the transcript to reveal the inefficiency and fragility a green outcome hides. You need both, because each is blind exactly where the other sees.

02 Process vs outcome

The metrics of the path

Having just insisted on the outcome check, the tempting overcorrection is to lean on it entirely: outcome is objective and close to binary, so why not grade only whether the reservation exists? Because outcome-only has the mirror flaw of answer-only. It tells you that the agent succeeded and never how well — and "succeeded, but in thirty tool calls and at a dollar-fifty a task" is a failure you cannot see in a boolean. A real agent scorecard therefore carries two families of metric side by side:

Outcome metrics — from the state

  • Did the reservation row appear? The file get written? The ticket actually resolve?
  • Task-completion rate and goal accuracy — the agent's real effect on the world.
  • Objective, code-checkable, close to binary. They prove success — but say nothing about cost or fragility.

Process metrics — from the transcript

  • Turn count, number of tool calls, token usage, latency, time-to-first-token.
  • Cost-per-task — the sum of every call the run made.
  • Cheap to compute from the trace. They reveal inefficiency — but a great process to the wrong state is still a failure.

Underneath the process family sit the specific trajectory metrics that make "grade the path" operational. None of these is derivable from the final answer — that is precisely why answer-only eval misses them:

  • Tool-selection accuracy — did the agent pick the right tool at each step? This needs labeled trajectories or a rubric; it is invisible in the output.
  • Tool-call / argument correctness — right tool and right parameters. Score arguments by the fraction correct, not exact all-or-nothing match; a good city="london" vs gold "London" shouldn't zero the call.
  • Redundant / looping steps — duplicate calls (same tool, identical args) and alternating call patterns that spin without progress.
  • Recovery after a failed tool call — when a call errors, did the agent issue a corrective call, or blindly repeat the failing one (a loop) or give up?
  • Number of steps vs a tool budget — a hard cap, commonly ~20–50 calls per task (default around 20), that halts runaway behavior.
  • Cost-per-task and total latency — summed across every tool call, now a first-class dimension rather than an afterthought.

That tool budget is not paranoia. A named 2025–2026 risk class is the overthinking loop: a verbose or adversarial MCP tool can lure an agent into a cyclic pattern that blows up tokens and latency — a denial-of-service vector — and one study found that merely enriching tool descriptions raised average execution steps by roughly 67%. So a budget plus explicit detection of repeated identical or alternating calls is itself a process metric you gate on, not a nice-to-have.

Cost is part of the score now

An agent at 92% task-completion for $0.03/task and one at 94% for $1.50/task are not the same product — reports cite up to 50× cost swings for similar accuracy. Read the accuracy/cost frontier, never accuracy alone. Illustratively (mid-2026): a run that fans out to Opus 4.8 at $5/$25 per Mtok costs an order of magnitude more than one that routes most steps to Haiku 4.5 at $1/$5 for the very same success rate — and cost-per-task is the metric that makes that visible.

03 Reference-trajectory

Diffing the run against a gold path ▸

The cheapest way to score a path is to write the path down first. Reference-trajectory evaluation hard-codes the expected sequence of tool calls — the gold path — and diffs the recorded run against it step by step. It is deterministic, fast, cheap, and adds no LLM call, which makes it the right first tool for any well-defined workflow. Step through a diff where the run drifts from gold in three separate ways:

The open-source agentevals package (also surfaced through LangSmith) packages exactly this diff as create_trajectory_match_evaluator, with four match modes so you diff at precisely the strictness the task deserves: strict (same calls, same order), unordered (all required calls, any order), subset (the agent used only reference tools, no extras), and superset (the agent used at least the reference tools, extras allowed). Argument comparison is a separate knob — tool_args_match_mode of exact / ignore / subset / superset, or a custom comparator (e.g. a case-insensitive city match) — and the evaluator returns a tidy {key, score, comment}.

agentevals — a strict diff with a custom arg comparatorfrom agentevals.trajectory.match import create_trajectory_match_evaluator

evaluator = create_trajectory_match_evaluator(
    trajectory_match_mode="strict",        # or "unordered" / "subset" / "superset"
    tool_args_match_mode="exact",          # or "ignore" / a comparator fn
)
result = evaluator(outputs=run, reference_outputs=gold)
# -> {"key": "trajectory_strict_match", "score": False, "comment": "..."}
Don't pin the whole path

Strict, exact, step-by-step matching is a trap of its own: Anthropic warns it is "overly brittle, as agents regularly find valid approaches eval designers didn't anticipate." Diff what truly matters — the required tools, the few orderings that are genuinely load-bearing, and the outcome — and reach for unordered or superset everywhere order is irrelevant. Pin the constraints, not the choreography.

04 Judges, graders, reliability

No gold path — and reliability

Reference-trajectory matching needs a gold path, and a whole class of tasks doesn't have one. "Research this competitor and write me a brief" has a thousand valid trajectories and no canonical sequence to diff against. That is where reference-free evaluation earns its keep: an LLM-as-judge reads the entire transcript and scores whether the tool choices and reasoning were reasonable, with nothing to compare against. agentevals ships this too — create_trajectory_llm_as_judge with a built-in TRAJECTORY_ACCURACY_PROMPT (reference-free) and a ..._WITH_REFERENCE variant that also diffs against an expected path, plus continuous float scores, discrete choices, and few-shot calibration. It scales to open-ended work — but it is non-deterministic, costs a real LLM call per grade, and drifts from human judgment if left uncalibrated. Tame it with structured single-dimension rubrics, an explicit "Unknown" escape hatch to curb hallucinated verdicts, frequent calibration against domain experts, and multi-judge consensus where the stakes justify it.

Reference-based and reference-free are really two ends of a wider grader taxonomy. You don't choose one; you combine all three:

Code-based

  • String/regex match, static analysis, and outcome/state checks.
  • Fast, cheap, objective, perfectly reproducible.
  • Brittle to valid variation — a code grader once rejected a correct 96.12 against gold 96.124991… on CORE-Bench.

Model-based

  • LLM-as-judge: rubric scoring, natural-language assertions, pairwise preference.
  • Flexible, handles nuance and open-ended paths.
  • Non-deterministic and costs money per grade; must be calibrated.

Human

  • SME review and spot-checks of transcripts.
  • The gold standard — it calibrates the other two.
  • Slow and expensive, so used sparingly and deliberately.

Once you can grade a run, the last question is how often a good run repeats — and here the metric you pick decides whether you ship. pass@k is the probability of at least one success in k attempts; it rises with k and is the right lens when a single working solution suffices (code you can unit-test, keep the one that passes). pass^k is the probability that all k trials succeed; it falls with k and is the standard for a customer-facing agent that has to work every time. The gap is brutal and worth memorizing: a plausibly "good" agent at 75% per-trial success clears pass^3 = 0.75³ ≈ 42% — it succeeds all three times for only about four users in ten. Report pass^k, not pass@k, whenever reliability is the product, so a right answer reached by a lucky path can't masquerade as dependable.

Why this matters for the path

pass^k is where "grade the path" pays off numerically. Two agents can post the same single-trial accuracy, but the one with the shakier trajectory — more redundant calls, weaker recovery, looser tool selection — collapses faster under pass^k, because fragility compounds across trials. The process metrics of §02 are the early-warning signal for the pass^k cliff you'd otherwise discover in production.

05 In code

A trajectory evaluator, end to end ▸

Now assemble the pieces into one scorer. It takes a recorded run — a list of {tool, args, ok} steps — and rolls it into a scorecard: tool-choice, argument correctness, step efficiency, recovery, cost, and latency, combined into partial credit rather than a binary pass. The run that got the right answer the shaky way

A refund agent closes a ticket: the customer got their money, so the outcome check passes — and so does end-answer eval. But the transcript tells a different story: get_customer failed once, was retried, and the run spent five tool calls where four would do. On this ticket it's fine. Deploy a fleet of these and the shaky path becomes a looping incident. How do you make that dent visible without failing an agent that did, in fact, refund the customer?

Score the transcript, not just the outcome

trajectory_eval.py — score one recorded run, no frameworkdef score_trajectory(run, expected_tools, min_steps, arg_ok):
    used = [s["tool"] for s in run]
    # 1 · tool-choice — order-independent coverage of the ideal set
    tool_choice = len(set(used) & set(expected_tools)) / len(expected_tools)
    # 2 · argument correctness — mean fraction of correct params
    checked = [arg_ok[s["tool"]](s["args"]) for s in run
               if s["tool"] in arg_ok]
    arg_score = sum(checked) / len(checked) if checked else 1.0
    # 3 · step efficiency — ideal vs actual; a repeat after a SUCCESS is waste
    loops = sum(1 for a, b in zip(run, run[1:])
                if a["ok"] and a["tool"] == b["tool"] and a["args"] == b["args"])
    efficiency = min_steps / (len(run) + loops)
    # 4 · recovery — every failed call followed by a corrective (ok) call
    recovered = all(run[i + 1]["ok"]
                    for i, s in enumerate(run[:-1]) if not s["ok"])
    # 5 · roll up into PARTIAL CREDIT, and carry cost + latency
    cost = sum(s.get("cost", 0) for s in run)
    ms   = sum(s.get("ms", 0) for s in run)
    score = .4*tool_choice + .2*arg_score + .3*efficiency + .1*recovered
    return {"score": round(score, 3), "tool_choice": tool_choice,
            "arg_score": round(arg_score, 3), "efficiency": round(efficiency, 3),
            "recovered": recovered, "cost": cost, "latency_ms": ms}
Restate — why the naive check is blind

The ticket closed, so an outcome-only check and end-answer eval both flash green. Neither can see the failed get_customer, the retry, or the fifth step that four could have covered. A binary pass launders a shaky path into a clean record.

The move — read the transcript into metrics

Record every step as {tool, args, ok, cost, ms}, then compute: tool-choice as order-independent set overlap, arg-score as the fraction of correct parameters, efficiency as min_steps / actual with a penalty only for repeats after a success, and recovery as "every failed call is followed by an ok call." Here that's tool-choice 1.00, arg 0.92, efficiency 0.80 (4 ideal, 5 actual), recovered ✓ → roughly 0.92 partial credit — a pass with a visible dent, not a silent green.

The invariant — and the reliability caveat

Grade the path and credit it partially, then report reliability as pass^k. A 0.92-ish per-trial agent clears pass^3 ≈ 0.92³ ≈ 78% — and the shaky step you just surfaced is exactly what erodes that number at scale. The rule: a run is only as trustworthy as the worst process metric you were willing to look at.

Say it back in one breath

“The ticket closed, so answer-only and outcome-only both green-lit the run. But I score the transcript: the agent chose all four right tools (1.00), its arguments were 92% correct, it spent five steps where four would do (efficiency 0.80) — and crucially, after get_customer failed it issued a corrective call instead of looping (recovery ✓). That's 0.92 partial credit, not a binary pass, and it's honest about the wasted step. Then I report pass^k, not pass@k, because customer-facing means it has to work every time — 0.92 per trial is only ~78% across three.”

Outcome proves it worked once; process reveals whether it will keep working. Grade both, credit partially, and report the reliability metric that punishes fragility — a right answer via a shaky path is a latent incident, and the trajectory is the only place it's visible.

06 The set, the benchmarks

Building an agent eval set

The instinct, once you can score a run, is to grab a public benchmark and declare the agent "evaluated." Two problems sink that. First, a generic benchmark doesn't test your agent's failure modes — it tests someone else's. Anthropic's guidance is to seed an agent eval set with 20–50 tasks sourced from real user failures and bug reports — the things your agent actually got wrong — before scaling anything. Then hold the set to a few disciplines that separate a real eval from theater:

  • Unambiguous specs. Write each task so two domain experts independently reach the same pass/fail. If they disagree, the spec, not the agent, is broken.
  • Isolated environments. Give every trial a clean, isolated world — shared state causes correlated failures and non-reproducible results.
  • Partial credit. Prefer graded scores over binary pass/fail, so a mostly-right path is distinguishable from a disaster.
  • Negative cases. Include tasks where a tool should not fire, or the answer is "I can't do that" — class imbalance hides false-positive behavior.
  • Read transcripts. Regularly sanity-check that the grader is fair; a 0% pass@100 almost always means a broken task or grader, not an incapable agent, and every task should ship with a reference solution proving it's solvable.

Public benchmarks are still useful as capability signals — if you know their traps:

  • tau2-bench (Sierra, 2025; successor to tau-bench) pioneered a dual-control setup where both the agent and a simulated user can call tools and mutate a shared world — a realistic model of tech support. It grades action correctness against evaluation_criteria.actions, gates on reward_basis, and reports pass^k. Tellingly, its v1.0.0 (Mar 2026) shipped 75+ fixes to its own incorrect gold labels — the reliability of a benchmark's labels is now itself a tracked concern.
  • SWE-bench Verified became partly contaminated: an audit found frontier models could reproduce verbatim gold patches that had leaked into training data, and scores climbed from ~30% to >80% in a year. Two lessons: prefer unseen-task variants (SWE-bench Pro), and treat saturation — an agent nearing ~100% — as the signal to graduate a capability eval into your regression suite.
  • GAIA (general assistants) and WebArena (live-browser tasks) remain the reference suites — and the framing that won 2025 was that a strong GAIA scaffold hit ~65% while being Pareto-optimal on the accuracy/cost frontier. Cost is part of the score, not a footnote to it.
  • Simulated users are imperfect proxies ("Lost in Simulation," 2026): tau-bench-style user-simulation pass rates can diverge from real human behavior, so treat them as directional, never as ground truth.
The bridge to shipping

This is the reflex the Agents phase has been building toward: an agent that can act is only shippable once you can measure the path it took — transcript, outcome, process metrics, reliability. ★ Review

Summary & self-test

One idea runs through the lesson: an agent's output is only the last line of its behavior, so evaluate the whole trajectory — verify the outcome (real state, not the claim), score the path with process metrics (tool selection, arguments, redundant/looping steps, recovery, steps, cost, latency), diff against a gold trajectory where one exists and judge the transcript where it doesn't, build the set from 20–50 real failures, and report pass^k so a right answer via a lucky path can't hide fragility.
