Workflows vs Agents

You have tool use working, and the temptation is immediate: hand the model the tools, wrap it in a framework, and let it “figure it out.” It dazzles in a demo and buckles in production. This lesson draws the single most important line in Phase 4 — a workflow orchestrates LLM calls through a path you write in code; an agent lets the model direct its own process. Five composable workflow patterns, when each fits, why autonomy is expensive, and the golden rule that keeps most production systems on the simple side of the line.

Original study notes. The lesson that opens the Agents phase by drawing its central distinction. It draws on public sources for framing — Anthropic’s “Building Effective Agents,” Chip Huyen’s “AI Engineering,” the 2025–2026 agent-framework wave, and field data on where agents succeed and fail — but every explanation, diagram, and example is our own. 00 The bottleneck

The reflex that fails

Picture the day after you get tool use working. The obvious next move writes itself: give the model a handful of tools, drop it into a loop, maybe reach for a framework you saw on Hacker News, and let it plan its own way to the goal. You demo it and it’s magic — it reads a file, calls an API, corrects itself, lands the answer. So you ship it. And in production it is slow, the bill is three times your forecast, and one run in three ends somewhere strange. Nothing crashed. It just quietly does the wrong thing, expensively, in a way you can’t reproduce.

This is the reflex the whole lesson is built to correct: reaching for an autonomous agent when the task didn’t need one. Most tasks have a knowable shape — classify, then answer; outline, then write; retrieve, then summarize. When the shape is knowable, you should encode it in code and let the model fill the slots. That is a workflow, and it is fast, cheap, and debuggable precisely because the model never gets to choose the route. Autonomy is the expensive tool you save for the genuinely open-ended parts, where you truly can’t predict the steps.

Anthropic groups both ends under one umbrella term, agentic systems, and gives the rule that governs everything below: find the simplest solution possible, and only increase complexity when needed. Agentic systems trade latency and cost for flexibility, so you add that complexity only when the simpler thing has demonstrably fallen short — not by reflex on day one.

The one-line version

A workflow is LLM calls wired together by your code on a predefined path. An agent is an LLM directing its own path through tools in a loop. Workflows are cheaper, faster, and more reliable; agents are more flexible. The engineering skill is knowing which the task actually needs — and defaulting to the simpler one until it fails.

01 Definitions

Two definitions, one umbrella

Anthropic’s definitions are worth learning verbatim, because the whole field’s vocabulary hangs off them. Workflows are “systems where LLMs and tools are orchestrated through predefined code paths.” Agents are “systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.” Both live under the umbrella of agentic systems — the distinction is not whether the model uses tools, but who decides the control flow.

Hold onto that phrase: who decides the control flow. In a workflow, the sequence of steps is fixed in your code before any request arrives — the model is a powerful function you call at known points, and if you sketched the system as a flowchart, you drew every branch. In an agent, the sequence is decided by the model at runtime, one step at a time, based on what it observes — you cannot draw the flowchart in advance because the model draws it as it goes.

Workflow — your code drives

  • Control flow: fixed in code, decided before the request arrives.
  • Model’s role: fills slots at known steps — classify, extract, write.
  • Steps: knowable and countable up front; you can draw the flowchart.
  • Strengths: low latency, budgetable cost, reproducible, easy to test and audit.

Agent — the model drives

  • Control flow: chosen by the model at runtime, step by step.
  • Model’s role: plans, picks tools, decides when it’s done.
  • Steps: unknown in advance; the path emerges from feedback.
  • Strengths: handles open-ended tasks where no fixed path exists.

The trap is treating this as a binary you must pick a side of. The 2025–2026 consensus is the opposite: workflows-versus-agents is a spectrum, not a switch. A production system is usually mostly deterministic code with a small, bounded pocket of autonomy exactly where the input is genuinely open. Where you sit on that spectrum is a design decision you make per task — and the rest of this lesson is how to make it well. We start on the workflow end, because that is where most good systems live.

02 The composable half

The five workflow patterns ▸

Anthropic catalogs five composable workflow patterns — five shapes you can wire from LLM calls plus ordinary code, and freely combine. Four of them have a completely fixed path; only one decomposes tasks at runtime. Start with the simplest and most common, prompt chaining: decompose a task into a sequence of fixed steps, each call processing the previous output, with optional programmatic gates between steps. Step through one:

Prompt chaining fits any task that “can be easily and cleanly decomposed into fixed subtasks” — Anthropic’s own example is writing an outline, checking it against criteria, then writing the document from it. It explicitly trades latency (two or three calls instead of one) for higher accuracy per step, because each call does one small thing well. The other four patterns each solve a different shape:

Routing — classify, then specialize

Routing classifies an input and directs it to a specialized follow-up prompt or model. It fits “complex tasks where there are distinct categories that are better handled separately, and where classification can be handled accurately” — for example, sending general questions, refund requests, and technical support down different downstream processes. Its quiet superpower is cost: route the easy, common query to a cheap small model (Haiku 4.5 at $1/$5 per Mtok, illustrative mid-2026) and reserve the capable, pricey model (Opus 4.8 at $5/$25) for the hard, unusual one. The path is still fixed — you wrote every branch.

Parallelization — sectioning and voting

Parallelization runs LLM calls concurrently and aggregates them, in two flavors. Sectioning splits a task into independent subtasks run at once — a classic is a guardrail where one model instance answers the user while another concurrently screens the same query for policy violations. Voting runs the same task several times to get diverse attempts and picks or aggregates — e.g., several prompts each review a snippet of code for vulnerabilities, and any one flagging a problem raises it. Use it “when the divided subtasks can be parallelized for speed, or when multiple perspectives or attempts are needed for higher confidence.” Again: fixed path.

Orchestrator–workers — decompose at runtime

Orchestrator–workers is the one pattern that bends toward autonomy. A central orchestrator LLM dynamically breaks the task into subtasks, delegates each to a worker LLM, and synthesizes the results. Anthropic’s example is a coding change that touches multiple files: you can’t predict in advance which files or how many, so the orchestrator decides at runtime. This is what separates it from parallelization — in parallelization you predefine the subtasks; here the model does. It is “well-suited for complex tasks where you can’t predict the subtasks needed.”

Evaluator–optimizer — generate, critique, refine

Evaluator–optimizer pairs two roles in a loop: one LLM generates a response, a second evaluates it and returns feedback, and the generator revises — repeating until the evaluation passes. It fits “when we have clear evaluation criteria, and when iterative refinement provides measurable value” — literary translation, say, where the first pass misses a nuance and a critic LLM’s feedback demonstrably improves the next. A good litmus test: would a human’s feedback obviously make the output better? If yes, an automated evaluator often can too.

The distinction that matters

Four of the five patterns — chaining, routing, parallelization, evaluator–optimizer — run on a fixed code path: you can draw the entire flowchart before any request arrives. Only orchestrator–workers decides its subtasks at runtime, which is why it sits right at the boundary with true agents. If you can enumerate the steps in advance, you want a workflow — and one of these five almost always fits.

03 The autonomous half

Agents: the model steers ▸

Cross the boundary and you reach the agent: use one “for open-ended problems where it’s difficult or impossible to predict the required number of steps, and where you can’t hardcode a fixed path.” The surprise for most engineers is how simple agents are architecturally — they are “typically just LLMs using tools based on environmental feedback in a loop.” No exotic machinery; the complexity is all in the loop’s dynamics, not its structure. Step through the loop that is the agent:

Two canonical examples justify that autonomy. A coding agent solving SWE-bench tasks edits many files from a task description, running tests and reading errors in a loop — you genuinely cannot know in advance which files it will touch. And computer-use agents that drive a screen face an environment too open to script. Both are open-ended enough that a fixed path would be a straitjacket. But the same autonomy that makes agents powerful is what makes them dangerous: “the autonomous nature of agents means higher costs, and the potential for compounding errors.” Anthropic’s standing advice is “extensive testing in sandboxed environments, along with the appropriate guardrails” — step limits, a sandbox, human-in-the-loop approval on risky actions.

2025–2026: the loop got cheaper to run

The framework wave was loud — OpenAI’s Agents SDK (Mar 2025), Google’s ADK (Apr 2025), and Anthropic renaming its coding SDK to the Claude Agent SDK to signal it powers general agent loops. Quieter but more consequential were the plumbing wins that attack the agent’s real problem, context bloat: code execution with MCP (Nov 2025) has the agent write code that calls MCP servers as APIs and load only the tools it needs — reportedly cutting token use by up to ~98.7% (a 150K-token workflow down to ~2K). The Tool Search Tool (defer_loading: true) lets the model discover tools on demand instead of front-loading every definition. The lesson: much of “make agents work” is really “control what enters the context window.”

04 The price of flexibility

The cost of autonomy ▸

Here is the math that most argues for staying on the workflow side. Every step in a multi-step system succeeds with some probability p, and independent steps multiply: end-to-end success is pn, not p. A per-step accuracy of 95% feels flawless in a two-or-three-step demo — and quietly collapses as the chain grows. Watch it:

And this is the optimistic version, because it assumes the steps are independent. In a real agent trajectory they are not: a wrong observation at step 3 poisons the reasoning at step 4, so errors don’t just multiply, they cascade. The practical reading is stark — a single 95%-accurate step is a great demo and a poor twenty-step production task, which lands near ~36% end-to-end. That gap between demo and production is exactly why most production systems choose workflows: fewer steps, each gated, keeps n small and the product high.

The reliability cost has a money cost stapled to it. Because an agent reasons repeatedly over environment state and tool outputs, agentic systems in 2025–2026 production writing run roughly 5–25× the cost per task and 5–10× the tokens of a single workflow-style call, and some architectures land ~10× a fixed pipeline. And the number of iterations is unpredictable, so the bill is too — a fixed pipeline with a known call count is far more budgetable. The illustrative shape (mid-2026 prices):

One more correction of instinct: the true blocker is not token price. LangChain’s 2025 State of AI Agents survey of ~1,400 practitioners found quality and reliability — not cost — the top barriers to putting agents into production. Human oversight of a low-success-rate agent runs far more expensive than its tokens, so optimizing token cost while ignoring reliability optimizes the wrong variable. Deloitte’s Q4 2025 reporting described enterprises finding “tens of millions” in monthly inference bills from multi-agent deployments — one healthcare firm burned ~1 trillion tokens (~$6M unplanned) in six months. Autonomy is a real capability; it is also a real invoice.

05 The two rules

The two golden rules

Everything above compresses into two rules that Anthropic states plainly and the field has learned the hard way.

Golden rule #1 — simplicity. “When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed.” Because agentic systems trade latency and cost for performance, you add multi-step autonomy only when simpler solutions fall short. In practice the ladder is: try a single well-prompted call; if that’s not enough, add retrieval or a fixed chain; if the categories diverge, add routing; and reach for an agent only when the path is genuinely unpredictable. Most systems never need the top rung.

Golden rule #2 — don’t reach for a framework first. “We suggest that developers start by using LLM APIs directly: many patterns can be implemented in a few lines of code. If you do use a framework, ensure you understand the underlying code.” Frameworks (LangChain, LangGraph, CrewAI, AutoGen) add abstraction layers that hide the actual prompts and calls, which is precisely what you need to see when debugging — and heavier ones are criticized for verbosity: even a trivial two-agent flow in LangGraph wants a state schema, nodes, edges, and a compile step. The patterns themselves are small. Routing, the workhorse, is a few lines of direct API calls:

routing in ~8 lines of direct API calls — no framework# 1 cheap classify call, then a fixed branch — the whole "framework"
category = classify(query, model="haiku-4.5")   # $1/$5 per Mtok

if category == "refund":
    return refund_flow(query)               # specialized prompt
elif category == "technical":
    return support_agent(query)             # escalate to an agent
else:
    return answer(query, model="haiku-4.5")  # cheap default path

Anthropic pairs these with three design principles for whatever you do build: keep it simple, prioritize transparency (surface the agent’s planning steps so you can see what it’s doing), and invest in your agent-computer interface — thorough tool documentation and testing, treating tool design with the same care you’d give a public API. The community codified the same stance in 2025: “12-Factor Agents” (own your prompts, own your context window, convert inputs to structured tool calls handled by deterministic code, keep agents small and stateless) went viral as an explicit reaction against heavy frameworks. Alongside it, context engineering emerged as a named discipline — deliberately managing exactly what goes into the context window — reinforcing the same direct-API, own-your-context ethos.

Transparency is not a nicety

An agent you can’t see is an agent you can’t debug, test, or certify. Prioritizing transparency — logging every reasoning step, every tool call, every observation — is what turns a mysterious 36%-success loop into something you can actually improve. It is also what regulators and auditors will ask for. Build the glass box before you scale the loop.

06 The decision

Choose the spectrum, not a side

The field data from 2025 is unambiguous about which way engineers over-reach. Gartner found fewer than 5% of enterprise applications had true AI agents in production by late 2025, projected 40% would embed agents by 2026, and separately predicted 40%+ of agentic-AI projects will be cancelled by 2027 over cost overruns, unclear ROI, and risk. A widely cited analysis of ~1,400 automation projects found 62% of the failed ones had applied agentic approaches to tasks that deterministic workflows would have handled more reliably and cheaply. Over-engineering with agents was the single most-cited failure mode of the year.

Meanwhile the boring workhorses run the world: Uber’s Michelangelo serves 5,000+ models at ~10M predictions/sec on deterministic pipelines; Booking.com powers 150+ customer-facing apps via ML pipelines; Airbnb pushes 50+ GB/day through Airflow — all fixed-path, none agentic. And when a well-resourced team does ship an agent, look at the cost of doing it safely: LinkedIn’s Hiring Assistant needed modular agent services over gRPC, a central orchestrator, skill registries, context memory, human-in-the-loop controls, and specialized monitoring. That is the true price of production autonomy.

So the answer is rarely “agent” or “workflow” dogmatically. The recommended production pattern is hybrid: deterministic code and guardrails for the known 80% of the path — where you get reliability, low latency, budgetable cost, and auditability — with bounded LLM reasoning scoped only to the genuinely open-ended pockets. Let’s reason through one concrete case.

Reason about itcustomer support · workflow, agent, or both?
“Automate customer support with AI” — what do you actually build?

A team says: “Let’s put an autonomous agent on our support queue — give it the docs, the refund API, and the account tools, and let it handle tickets end to end.” Tempting, and demo-friendly. Reason it through before you build.

Restate → the naive take → the insight → the rule

Restate the task

The queue is a mix: a flood of general questions (fixed answers from docs), a stream of refund requests (a known, policy-bound procedure), and a thin tail of gnarly technical cases (genuinely open-ended debugging across a user’s data). Three distinct shapes wearing one label, “support.”

The naive take

One autonomous agent for everything. It demos beautifully — but now every trivial “what are your hours?” runs a multi-step reason–act loop at ~10–25× the cost, the refund path can take an unpredictable route through a money-moving API with no fixed guardrail, and per-step errors compound across the whole queue. You’ve applied autonomy to two shapes that never needed it.

The insight

Only the thin technical tail is actually open-ended. The other two are fixed paths. So route first (one cheap classify call), answer general questions with a retrieval + single-call workflow, run refunds as a prompt chain with a hard gate (validate eligibility in deterministic code before the API fires, with human approval above a threshold), and hand only the technical tail to a bounded agent — in a sandbox, with step limits.

The rule / trade-off

This is the hybrid spectrum: deterministic code owns the known 80%, autonomy is scoped to the open 20%. You keep low latency and budgetable cost where volume lives, hard guardrails where money moves, and reserve the expensive, less-reliable loop for the cases that truly can’t be scripted. Start here; escalate a path to more autonomy only when the simpler version measurably falls short.

the hybrid: route once, then the right shape per categorycategory = classify(ticket)                 # cheap Haiku call, fixed branch

if category == "general":
    return rag_answer(ticket)              # retrieval + 1 call (workflow)

if category == "refund":
    outline = draft_refund(ticket)          # LLM call 1
    if not policy_gate(outline):           # deterministic gate — no LLM
        return escalate_to_human(ticket)
    return issue_refund(outline)           # LLM call 2, then API

if category == "technical":
    return support_agent(ticket,            # bounded agent — the only loop
                         max_steps=12, sandbox=True)
Say it back in one breath

“‘Support’ is three shapes, not one. General questions are a retrieval workflow, refunds are a chain with a deterministic policy gate before any money moves, and only the open-ended technical tail earns a bounded agent in a sandbox. I route once up front and give each path the simplest structure that fits — deterministic code for the known majority, autonomy only where the path is genuinely unpredictable. That’s the hybrid, and it’s cheaper, faster, and more reliable than one agent for everything.”

Match the structure to the shape of the task. Reach for an agent when the path is truly unpredictable — not by reflex.
