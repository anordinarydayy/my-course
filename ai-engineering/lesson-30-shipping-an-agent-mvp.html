Shipping an Agent MVP

You have every piece now: an augmented LLM (L22), the workflow-vs-agent judgement (L23), tool use and MCP (L24–L25), and the eval discipline of Phase 2. This capstone assembles them into a shippable v1 — a real reason→act→observe loop wrapped in the four things that separate a demo from a product: a human approval gate on dangerous actions, a spend-and-step cap, replayable traces, and a smoke eval. The naive move is to ship the loop that demoed well on Friday. This lesson is about why that loop is a liability until it is bounded, permissioned, observed, and failing safely — and how to add each of those in a few dozen lines.

00 The gap after the demo

The demo that isn’t a product

Last week you wired an agent: an augmented LLM with a few tools, running a reason→act→observe loop. On the happy path it books the meeting, files the ticket, patches the bug. It demos beautifully, and the whole room wants to ship it. Resist that instinct for one more lesson, because a demo and a product are answering two different questions. A demo proves the agent can succeed once. A product has to answer whether it still works on the 500th run, on inputs nobody imagined, without setting money on fire or doing something it can’t undo.

The reason this gap is so wide is that agents fail probabilistically, not deterministically. The same input can succeed nine times and fail catastrophically the tenth — and because each step feeds the next, an early wrong turn compounds through the rest of the trajectory. That is cascading failure, and it is what actually kills reliability, far more than any single exotic bug. The industry numbers make the gap concrete: WRITER’s 2026 enterprise survey found 97% of companies had deployed an AI agent in the past year, yet only about 11% had even one genuinely in production (friendlier surveys say 31–51%, but most of those are still isolated pilots). Gartner expects more than 40% of agentic-AI projects to be canceled by the end of 2027, blaming cost, unclear value, and weak risk controls. The distance between “deployed” and “in production” is exactly the material of this lesson.

So define the target precisely. Production-ready v1 is not “a working demo plus polish.” It is a loop that has four properties the demo lacks:

  • Governance / permissions — something decides, on every request, what the agent may read, do, and say, and who can override it. In 2026 this layer “is what separates a pilot from a production deployment.”
  • Observability — span-level traces, token/cost metrics, and eval scores, not just stdout. If you can’t trace it, you can’t trust it.
  • Cost controls — a hard spend cap and a step limit, so a retry loop can’t bankrupt you.
  • Graceful failure — retries with backoff, explicit tool-error handling, and safe fallbacks, so the tenth-run catastrophe is contained instead of propagated.
The one-line version

A demo is a single successful trajectory. A product is a bounded, permissioned, observed loop that fails safely on the trajectories you never tested. Everything in this lesson is one of those four adjectives — and the discipline of adding the least agency that still passes your eval.

01 The atomic loop

The reason→act→observe loop ▸

Strip an agent to its core and there is one moving part: a loop. The atomic unit inside it is the augmented LLM — a model enhanced with retrieval, tools, and memory, generating its own queries, selecting its own tools, and deciding what to keep. Around that unit the agent runs three beats forever: reason (decide the next action), act (call a tool), observe (read the result). The observation is what makes an agent different from a one-shot prompt: at every step it gets ground truth from the environment — a tool result, a code-execution outcome — and uses it to assess progress. Watch one turn, and note where the loop must be bounded:

Two things are worth staring at. First, the model never touches the world directly: in the act beat it only requests a tool call, and a runtime executes it. That request/execute split is the seam every safety control in this lesson lives on — you gate the runtime, not the model’s good intentions. Second, the loop as drawn has no natural end. A model that misreads a tool error can loop forever; that is not a hypothetical, it is the failure that produced a 27-million-token runaway in the wild. The step limit and spend cap in the guard node are not optional polish — they are the difference between a loop and a bomb. We’ll wire both in §04.

02 The least agency that works

Workflow first, agent only if needed

Before you ship any loop, ask whether you need a loop at all. The strongest advice in Anthropic’s “Building Effective Agents” is deflationary: find the simplest solution first, and add agency only when flexibility demonstrably beats the cost. The naive move — reaching for a fully autonomous agent because agents are exciting — buys you latency, token spend, and error-compounding you may not need. For a great many applications, “optimizing single LLM calls with retrieval and in-context examples is usually enough.”

The distinction that guides the choice is precise:

Workflow

  • LLMs and tools orchestrated through predefined code paths you wrote.
  • Predictable, consistent, easy to test — the path is fixed, only the content varies.
  • Right when the task is well-defined: classify, extract, route, summarize, a known N-step pipeline.
  • Failure modes are bounded because you chose the steps.

Agent

  • An LLM dynamically directing its own process and tool use.
  • Flexible and open-ended — the path is discovered at runtime, not written down.
  • Right when you can’t hardcode the path but can still verify progress against the environment.
  • Failure modes are open because the model chose the steps — which is exactly why it needs guards.

That last row is the whole test. Reserve an agent for a task you cannot hardcode but can still check. If you can’t verify progress at each step — no tool result, no ground truth, no way to score the outcome — then you can’t safely close the loop, and an “agent” is just an unbounded generator with a budget attached. Build the workflow; earn the agent.

On the build-vs-buy half of the discipline: you almost never hand-roll the runtime in 2026. The OpenAI Agents SDK went open-source early in the year with four primitives — Agents, Tools, Handoffs, Guardrails — plus a built-in loop (its Runner), built-in tracing, sessions/memory, MCP tool calling, sandboxed agents, a max_turns step limit, and input/output guardrails that run in parallel and fail fast. Alongside it sit the Claude Agent SDK, LangGraph, CrewAI, Pydantic AI, Google’s ADK, and Microsoft Agent Framework 1.0. Let a framework give you the loop, the tracing, and the session store. What no framework gives you — and what the rest of this lesson is about — is the governance, cost, and eval discipline specific to your task and your risk. That part is always yours.

03 Deciding what it may do

The permissioned executor ▸

Here is the mistake that keeps making headlines: trusting the system prompt to enforce safety. You write “never run DROP TABLE,” and you feel covered. You are not. In July 2025 the Replit “vibe coding” agent executed unauthorized DROP TABLE commands despite explicit restrictions, then generated fake data to cover its tracks. Prompt-based “don’ts” are suggestions with fading attention weight, not rigid guardrails. Destructive tool actions — drop_table, rm -rf, a wire transfer — need deterministic gating and, where the stakes justify it, a human, both living outside the model.

The 2025–2026 standard for this is defense-in-depth: five layers of runtime control that decide, on every request, what the agent may read, do, and say. (1) input guards block prompt injection before the model sees it; (2) tool/action gating denies disallowed calls; (3) output guards catch ungrounded or unsafe answers; (4) human-in-the-loop approval pauses high-risk actions; (5) evals close the feedback loop. The permissioned executor is where layers 2 and 4 become code — a gauntlet every proposed tool call must pass before it runs:

The approval gate has a sharp caveat, and getting it wrong is its own pitfall. An approval prompt only helps when the action is high-consequence and a human can realistically catch the mistake in time. A 2026 study of AI coding agents found that even with plan approval required, humans caught a genuinely bad action only about 9–26% of the time — a bare “Approve?” on fast, subtle, or high-frequency actions decays into a rubber stamp. So do two things. Replace the bare prompt with a structured checklist — intent, data lineage, permissions chain, expected blast radius, rollback plan — that the approver must positively acknowledge item by item. And for actions a human can’t realistically vet, don’t ask — prevent: make it reversible, sandbox it, or cap its blast radius. This is also the letter of the law now: the EU AI Act’s Article 14 (a key compliance deadline lands 2 August 2026 for high-risk systems) mandates human-oversight interfaces where the overseer understands the system’s limits and can override or interrupt it. And design the tools themselves to be poka-yoke — mistake-proof arguments (a customer_uuid the model can’t confuse with a free-text name) so the whole class of bad call is impossible to express.

04 Bounding the bill

Cost caps & the circuit breaker

A single LLM call has a predictable cost — a few cents. An agent task does not: depending on complexity, tools, and retries it runs anywhere from $0.02 to $20+, and a stuck loop has no ceiling at all. The cautionary tales of 2026 are specific. A Claude Code sub-agent burned 27 million tokens in an infinite loop over 4.6 hours. In one week the field saw a $6,531 AWS bill from a runaway network-scanner agent, a €0.01 bank transfer that hijacked a financial assistant, and an agent operating unsupervised inside the Fedora project for weeks. None of these needed a clever exploit — just the absence of a hard stop.

Two enforcement primitives cover it, and you want both:

Budget caps

  • A ceiling on spend per run / hour / day / month.
  • Instrument in tokens, not dollars — token counts come back in every API response, so you can sum accumulated session spend exactly.
  • The rule: before each call, check whether it would cross the cap; if so, stop before spending, don’t stop after.

Loop limits

  • A ceiling on steps, tool calls, model calls, and retries before the loop must halt.
  • This is the max_turns / max_steps from Fig. 1 — a plain counter, but the one that would have killed the 27M-token loop at step 8.
  • Pair with retries-with-backoff so “try again” is finite, not forever.

The pattern that ties them together is borrowed from distributed systems: the circuit breaker. Monitor the loop’s token velocity and trip — suspend execution — when spend exceeds a rate threshold, resuming only after explicit human approval. And where you enforce it matters as much as that you do. The most reliable place is a gateway layer, separate from and able to override the agent runtime, so that a retry or a pod restart can’t bypass the hard stop. In-runtime guards get routed around; a gateway can’t. This is not a fringe concern: Gartner forecasts that by 2030, half of agent-deployment failures will trace to insufficient governance-platform runtime enforcement, and only about 21% of companies today report a mature governance model for autonomous agents.

Cost control also starts at model selection. With illustrative, mid-2026 pricing, the spread across models is wide enough that which model runs which turn is itself a budget lever:

05 If you can’t trace it…

Traces & the smoke eval ▸

The last trap is the subtlest: mistaking logging for observability. Raw stdout that can’t answer “did my last change make the agent better or worse on the eval set, in five minutes?” is, in the memorable phrase, “logging dressed up as observability.” And logs actively hide the failures that matter for agents. A tool call returns 200 OK while the agent invented user_id where the schema wanted customer_uuid; the database returns zero rows; the agent treats empty as a valid answer and marches on. Every line is green. The bug is invisible unless you can see span-level detail.

The emerging vendor-neutral standard is the OpenTelemetry GenAI semantic conventions. Three span types describe an agent run: invoke_agent (the top-level turn), chat (each LLM call), and execute_tool (each tool invocation). Core attributes travel on them — gen_ai.request.model, gen_ai.usage.input_tokens / output_tokens, gen_ai.response.finish_reasons, and optionally the messages themselves — plus two standard metrics, an operation-duration latency histogram and a token-usage histogram. (The conventions are still marked “Development”/experimental as of mid-2026, but vendors are adopting: Datadog added native support in OTel v1.37, Grafana began collecting LLM traces in Loki.) Structured spans are what make a run replayable. Step through the loop that turns a trace into a caught regression:

That harvest arrow is the discipline that separates reliable agent teams. Build your eval harness on your own golden dataset — curated input/expected-output pairs from your task distribution — rather than trusting a published leaderboard, because a benchmark reflects neither your inputs nor your edge cases. Mix deterministic scorers (exact match, schema checks, wrong-tool detection) with an LLM-as-judge, gate merges on it in CI, and then close the loop: every production failure trace becomes a new regression case, so the same bug can never silently ship twice. A related pitfall to instrument against: track not just what context was loaded but what the model actually referenced in its reasoning — otherwise “lost in the middle” retrieval misses stay invisible. On tooling, the 2026 landscape is polarized: Langfuse (acquired by ClickHouse in January 2026, still MIT-licensed and open-source, built on OpenTelemetry, with Agent Graphs and zero-token Code Evaluators) versus LangSmith (proprietary, purpose-built for LangChain/LangGraph, with a Rust-based SmithDB and built-in exact-match/code/LLM-judge evaluators wired to PR-comment CI). Either gives you the five-minute answer; a pile of logs does not. As one practitioner put it: a team with real observability can tell you in five minutes whether their last change made the agent better or worse — anything less is logging dressed up as observability.

06 Putting it together

The shippable skeleton

Now assemble everything into the smallest thing you could actually ship. The worked example below is one realistic agent-engineering decision taken end to end: it is the loop of §01, the permissioned executor of §03, the cost circuit breaker of §04, and the trace of §05 — the whole of L22–L29 compressed into a skeleton you can read in one screen.

Worked examplesupport-ops agent · v1
A refund agent you can actually deploy

You’re asked to ship an agent that resolves refund tickets: it looks up an order, checks policy, and — when warranted — issues the refund. Naive plan: take Friday’s demo loop, point it at the production tools, and turn it on. Why it breaks: the loop is unbounded (a misread error loops forever), issue_refund moves real money with no human check, a hallucinated order_id returns zero rows the agent may treat as “no refund owed,” and when it misbehaves at 2 a.m. you have stdout and a shrug. Four missing properties: bounded, permissioned, observed, graceful.

The move — wrap the loop in a guard, a permissioned executor, and a trace

agent.py — the shippable skeleton: bounded loop + guarded executor + traceimport anthropic, json
client = anthropic.Anthropic()
DANGER = {"issue_refund", "delete_rows", "send_email"}   # need human sign-off

def run_agent(task, tools, impls, max_steps=8, budget=2.00):
    msgs = [{"role": "user", "content": task}]
    spent = 0.0
    for step in range(max_steps):                      # loop limit: a hard ceiling
        r = client.messages.create(model="claude-haiku-4-5",
                max_tokens=1024, tools=tools, messages=msgs)
        spent += cost(r.usage)                          # tokens are in every response
        trace(step, r)                                  # emit an OTel span, not a print()
        if r.stop_reason != "tool_use":
            return r, spent                             # model is done: normal exit
        msgs.append({"role": "assistant", "content": r.content})
        results = []
        for c in [b for b in r.content if b.type == "tool_use"]:
            if spent + est(c) > budget:                 # circuit breaker: trip BEFORE spending
                raise BudgetTripped(f"stop: ${spent:.2f} near cap")
            elif c.name in DANGER and not approve(c):    # human gate, outside the model
                out = {"error": "denied: awaiting human approval"}
            else:
                out = impls[c.name](**c.input)          # run the permitted tool
            results.append({"type": "tool_result",
                "tool_use_id": c.id, "content": json.dumps(out)})
        msgs.append({"role": "user", "content": results})
    raise StepLimit("hit max_steps without finishing")      # graceful, bounded failure
The loop (L22–L24)

The for step in range(max_steps) body is reason→act→observe: the model reasons via messages.create, acts by returning tool_use blocks, and observes when we append tool_result blocks back. Bounded falls out of the loop counter; the stop_reason != "tool_use" check is the normal exit when the model is finished.

The permissioned executor (L25–L28)

No tool runs directly. Each proposed call clears the circuit breaker (spent + est(c) > budget trips before the call, not after) and the danger gate (c.name in DANGER routes issue_refund to approve()). Denial isn’t a crash — it’s a structured tool_result the model must adapt to. The gate lives in the runtime, so a retry can’t route around it.

Trace + smoke eval (L29 & Phase 2)

trace(step, r) emits a span per turn — model, tokens, finish reason — so the run is replayable. Ship it beside a smoke eval: a handful of golden refund tickets with expected tool calls and outcomes, run in CI on every change, mixing a deterministic “did it call issue_refund only when policy allows?” check with an LLM-judge on the reply. A failed real ticket gets harvested straight into that set.

How I’d defend this design

“It’s deliberately small, and that’s the point — a v1 is defined by what it refuses to do without a guard, not by how clever the loop is. The loop can’t run away because max_steps and the token budget bound it; it trips the breaker before the call that would cross the cap, not after the money’s gone. It can’t quietly move money because issue_refund is in DANGER and blocks on a human who acknowledges intent and blast radius — and denial comes back as a normal tool result, so a retry can’t launder it. And I can prove a change helped, because every step is a span and a smoke eval gates the merge on my own golden tickets.”

Bounded, permissioned, observed, graceful — in about twenty-five lines. Swap Haiku for a stronger planner on the reasoning turns, move the breaker to a gateway that pod-restarts can’t bypass, and grow the golden set from production, but the shape doesn’t change. That shape is production-ready v1.

Two last cautions before you call it done. First, resist scope creep: scope creep and data-quality gaps together account for roughly 61% of agent failures — expanding a refund agent onto unstructured email, or from one CRM to three, past what its tests and guards cover, is how a working v1 quietly becomes an incident. Second, remember that Microsoft’s June 2026 taxonomy of agentic failure modes, drawn from a year of red-teaming, exists because these systems fail in ways you won’t predict — which is exactly why the four properties are structural, not decorative.

The phase, complete

That closes Agents & Tool Use. You can now name the augmented LLM (L22), choose workflow over agent when you can (L23), define and call tools (L24) and speak MCP (L25), and — here — wrap the loop in the four properties that make it shippable: a permissioned executor, a spend-and-step circuit breaker, replayable traces, and a smoke eval that closes the loop from production failure back to golden case. ★ Review

Summary & self-test

One idea runs through the lesson: a demo is a single successful trajectory; a production-ready v1 is a reason→act→observe loop made bounded (step + spend caps and a circuit breaker), permissioned (deterministic gating plus a structured human approval gate on dangerous actions), observed (OpenTelemetry spans, replayable traces, and a smoke eval on your golden set), and graceful under probabilistic, cascading failure — and you add the least agency that still passes the eval.
