The Augmented LLM

Phase 4 opens with the single part every agent and workflow is built from. Not a framework, not a special “agent engine” — just one model given three extra channels it can drive itself: retrieval, tools, and memory. That primitive, the augmented LLM, turns a stateless text generator into something that can look things up, act on the world, and remember — and it runs a tight read → act → observe loop that, composed and repeated, becomes everything you will build in this phase.

Original study notes. The foundational lesson of the Agents phase. It draws on public sources for framing — Anthropic’s “Building Effective Agents” and “Effective context engineering for AI agents,” the Model Context Protocol, Chip Huyen’s “AI Engineering,” and 2025–2026 tooling (MCP registry, the Memory tool, Advanced tool use, OpenAI’s Responses API) — but every explanation, diagram, and example is our own. 00 The ceiling

What a plain model can't do

Start from the simplest possible thing: you call a model with a prompt and read back its text. A plain model — the raw weights behind a single completion call — is astonishingly capable this way. It writes, summarizes, reasons, translates, and explains, often better than you expected. So the naive instinct, when it isn’t good enough, is to reach for a bigger, smarter model and hope the extra IQ closes the gap.

That instinct hits a wall, because a plain model has three limits that no amount of raw capability removes. It is frozen: its knowledge ends at a training cutoff, so it cannot see your private documents, today’s prices, or the ticket the user is asking about — and when it doesn’t know, it tends to invent something fluent and wrong. It is inert: it can only emit text; it cannot run code, query a database, hit an API, book a flight, or change a single byte in the world. And it is forgetful: every call is stateless, so nothing carries from one turn or session to the next beyond whatever you manually paste back into the prompt. A smarter model is still frozen, inert, and forgetful — it just writes more convincingly about things it cannot check, do, or remember.

So the fix isn’t a better brain. It’s giving the brain eyes to look things up, hands to act, and a notebook to remember. Attach those three channels to a single call and you get the augmented LLM: the same model, now able to reach outside its own weights. Every technique in this phase — RAG pipelines, tool-using agents, multi-agent systems — is a way of arranging this one augmented unit. Get the primitive right and the rest is composition.

Plain LLM

  • Frozen — knows only what it trained on; no access to your data or the present moment.
  • Inert — produces text and nothing else; cannot act, compute, or verify.
  • Forgetful — each call is stateless; context vanishes when the window resets.
  • Confident anyway — fills gaps with plausible hallucination.

Augmented LLM

  • Grounded — retrieval lets it pull in fresh, private, or specific facts on demand.
  • Capable — tools let it run code, call APIs, and change the world, then read the result.
  • Persistent — memory lets it carry state across turns and whole sessions.
  • Self-driving — it decides when to use each channel, not you.
The one-line version

The basic building block of every agentic system is not a bigger model — it’s an LLM enhanced with retrieval, tools, and memory. From here on, assume every model call in this course has those three channels available. That single assumption is what turns “a chatbot” into “a system that gets things done.”

01 The primitive

The augmented LLM ▸

Anthropic’s “Building Effective Agents” (December 2024) names this primitive directly: the basic building block of agentic systems is an LLM enhanced with augmentations such as retrieval, tools, and memory. The key word is enhanced — these aren’t passive plumbing bolted on around the model. The model actively drives them: it generates its own search queries, selects which tool to call and with what arguments, and decides what is worth writing down. What that buys you, in a single call, is a tight loop — read → act → observe. The model reads its context, acts by invoking one of its channels, and observes the result fed back in. Watch one turn of the loop:

The reason this deserves a name is that the loop is recursive. One pass through read → act → observe might answer the whole question; or the observed result might prompt another action, and another, until the model decides it’s done. Repeat the loop under the model’s own control and you have an agent; wire a few passes in a fixed order and you have a workflow. That’s section 05. First, meet the three channels the model drives — each one dissolves exactly one of the plain model’s three limits:

Retrieval

  • Cures frozen.
  • The model writes its own search query and reads back the results.
  • Grounds answers in fresh, private, or specific facts.

Tools

  • Cures inert.
  • The model selects a tool, invokes it, and consumes the output.
  • Runs code, calls APIs, acts on real systems.

Memory

  • Cures forgetful.
  • The model decides what to retain and reads it back later.
  • Carries state across turns and whole sessions.
02 The first channel

Retrieval: the model writes its own query

The naive version of retrieval is the classic retrieve-then-generate RAG pipeline: on every request, embed the user’s text, pull the top-k chunks from one vector store, staple them into the prompt, and generate. It works, and for simple lookups it’s often all you need. But notice what it takes away from the model: you decided to retrieve, you decided the query is the user’s raw text, and you decided the source. When the question needs no lookup, it retrieves anyway; when it needs three hops or a different source, it gets one shot at the wrong one.

The augmented view hands those decisions back to the model. Retrieval becomes a channel the model drives: it generates its own search query — reformulating, narrowing, or splitting the user’s question — chooses when a lookup is even warranted, and picks which source to hit. This is agentic RAG: the model routes across a SQL database, a web API, a documentation store, or several vector indexes, and can retrieve again after reading the first result. The bare model answering “What’s our refund policy?” guesses from stale training and may hallucinate; the augmented model issues a query, retrieves the current policy document, and answers grounded in it — the read → act → observe loop, with retrieval as the action.

Two 2025–2026 shifts make this concrete. First, context windows exploded: illustratively, Claude Sonnet 4 gained a 1M-token window in 2025 (a 5× jump from 200K), Claude 4.6 Sonnet shipped 1M by early 2026, and Gemini 2.5 Pro runs 1M on the standard API (2M on some enterprise tiers) — enough to hold an entire 75,000-line codebase or dozens of papers at once. Second, the retrieval market shifted with it: standalone vector databases lost adoption share while hybrid retrieval intent roughly tripled toward a third of teams, and research showed agentic keyword-search tools reaching over 90% of vector-RAG quality without a standing vector database at all. The lesson isn’t “vectors are dead” — it’s that who decides how to retrieve moved from your pipeline into the model.

The invariant

Retrieval, done right, is the model asking a question and reading the answer — not a fixed pre-processing step you run for it. The quality of what comes back, and how cleanly it’s exposed, matters more than how big your context window is. Which is exactly why a 1M-token window is not a license to dump everything in (section 04).

03 The second channel

Tools, and the interface that carries them ▸

A tool is any function you expose to the model: get_weather, run_sql, send_email, search_web. The model doesn’t execute it — it emits a structured request (“call run_sql with these arguments”), your code runs the function, and the result comes back as a tool_result the model reads. That’s the act and observe of the loop again, now with a side effect in the real world.

The naive way to wire this is to hand-roll every integration per model provider: bespoke glue for each tool, re-written for each vendor’s API. With N tools and M providers that’s N×M brittle integrations. The fix, and the reason tool use scaled in 2025, is a standard interface: the Model Context Protocol (MCP), which Anthropic released in late 2024. MCP standardizes tools, resources, and prompts so a model can integrate with a whole ecosystem of third-party servers through one simple client implementation. The payoff is decoupling: swap in a different MCP server — a new database, a SaaS API, a filesystem — and the model side never changes. MCP kept maturing through 2025–2026: an MCP registry (an open catalogue for discovering servers) launched in preview in late 2025, and the specification added OIDC-based auth-server discovery, icon metadata, tool-calling inside sampling, and experimental durable “tasks” with polling. It’s not the only shape, either — OpenAI’s Responses API (2025) bundles augmentations as hosted tools (web_search, file_search over vector stores, computer use) and ships an open-source Agents SDK, and Gemini 2.5 Pro added native MCP support in its API. The through-line across all of them: the tool channel is now a standard socket, not a hand-soldered wire.

A tool definition is deceptively small — a name, a description, and a JSON schema for its arguments — but that little contract is where most tool-use failures are won or lost. Hold that thought; it becomes the whole point in section 06.

04 The third channel

Memory: state that survives the reset

The naive way to give a model “memory” is to keep everything in the context window — paste the whole conversation history back on every turn and trust the big window to hold it. With million-token windows, why not? Because bigger windows don’t recall proportionally better. Anthropic frames the context window as a finite attention budget subject to context rot: as the number of tokens in the context window increases, the model’s ability to accurately recall information decreases — a consequence of the transformer’s n² pairwise attention. Illustratively, Gemini 2.5 Pro reports near-100% recall out to ~530K tokens but degrading single-needle recall deep into 1M. Stuffing the window is not the same as remembering.

That realization is why the field’s vocabulary shifted from prompt engineering to context engineering — a term that crystallized in mid-2025. Anthropic defines it as the set of strategies for curating and maintaining the optimal set of tokens during inference, a strict superset of prompt engineering. Reported surveys put the mood plainly: around 82% of data leaders say prompt wording alone is no longer sufficient, and ~95% call context engineering important to scaling agents. The mindset: your context window is a scarce resource to curate, not a bucket to fill.

The concrete tool that closes the loop is exactly that — a memory channel. Anthropic’s Memory tool (tool type memory_20250818, launched in beta in late 2025, on Claude 4-class models) is a client-side, file-based store in a /memories directory that persists across sessions. The model runs plain file commands — view, create, str_replace, insert, delete, rename — and the API auto-injects a system-prompt “memory protocol” telling the model to check memory first, because your context window might be reset at any moment. It pairs with context editing (auto-clearing stale tool results past a threshold) and server-side compaction, together reported to cut token use by up to ~84% in long workflows. This is just-in-time retrieval applied to state: write learnings out to memory, read them back on demand, instead of dragging the entire history through every call.

memory as a channel the model drives (illustrative)# enable the memory channel — a client-side /memories directory
tools = [ { "type": "memory_20250818", "name": "memory" } ]

tool      → [ "customer_service_guidelines.xml", "escalation_rules.md" ]
assistant → memory( command="view", path="/memories/customer_service_guidelines.xml" )
# ...now it drafts a reply grounded in persisted, cross-session knowledge
Pitfall: the memory channel is yours to secure

Because the Memory tool is client-side, you execute every file operation the model requests — which makes it your attack surface. Add path-traversal protection (reject any path outside /memories; block ../ and its URL-encoded cousin %2e%2e%2f), cap file sizes, and expire stale entries. A model that can write files it later trusts is a model whose file writes you must validate. Treat memory ops like any untrusted input.

The invariant mirrors retrieval’s: memory is the model deciding what to write down and reading it back later — not you holding the entire past in the window. Retrieval reaches out for the world’s facts; memory reaches back for its own. Both are just-in-time, both are model-driven, and both exist because context is finite.

05 Composition

Every agent is augmented LLMs composed ▸

Here is the leap that makes the whole phase click. Once you have the augmented LLM — base model plus retrieval, tools, and memory, running the read → act → observe loop — you do not need a special “agent engine” to build agents. An agent is just an augmented LLM called in a loop that chooses its own next step; a workflow is a few augmented calls wired together in a fixed order you write. Everything ornate downstream is one primitive, composed. Watch the same unit become both:

The distinction that matters is who decides the control flow. In a workflow, you do: the path from call to call is coded ahead of time, which makes it predictable, testable, and cheap to reason about — prompt-chaining, routing, and parallelization are all workflows. In an agent, the model does: it picks each next action at runtime and loops until a goal or stop condition, which is more flexible but harder to bound. Neither is magic, and neither requires a heavyweight library. Anthropic’s explicit advice is to prefer simple, composable patterns rather than complex frameworks: start with the simplest composition that works — very often a single augmented call — and add orchestration only when the task genuinely demands it. Reaching for a multi-agent framework on day one is the classic over-engineering trap.

06 The thesis

The interface beats raw IQ

Everything so far sets up the design thesis that actually separates working agents from flaky ones: the quality of the augmentation interface matters more than the raw capability of the model behind it. Anthropic’s guidance is to tailor these capabilities to your use case and give the model an easy, well-documented interface — and they coin a name for it, the agent-computer interface (ACI): think about how much effort goes into human-computer interfaces, and plan to invest just as much in creating good agent-computer interfaces. A tool with an ambiguous schema and no examples wastes tokens and triggers wrong calls no matter how smart the model is; a tool with a crisp description and a sample call gets used correctly by a cheaper one.

This isn’t a slogan — it’s measured. Anthropic’s “Advanced tool use” (late 2025) shipped three interface upgrades, no new base model required, and each moved the numbers (illustrative, mid-2026):

Read that table as one claim proven three ways: same model, better interface, materially better results. Programmatic Tool Calling in production is why Claude for Excel can read and modify spreadsheets of thousands of rows — it orchestrates the tools in code so the intermediate data never floods the window. And the pitfalls are the mirror image of the wins: loading every tool definition and document up front burns context (a tool surface over ~10K tokens is a red flag — hence Tool Search); the top failure mode across teams is treating the augmentation interface as an afterthought; and agents fail differently than chatbots — their failures are state- and context-management failures, not prompt-wording ones, which is precisely why prompt engineering alone can’t give an agent memory. You reach for context engineering instead.

Reason about itinterface vs IQ · a support agent
The upgrade that wasn’t a model upgrade

Your support agent runs on Sonnet 4.6 (illustratively $3 / $15 per Mtok, mid-2026). It has 40 tools and calls the wrong one, or fills a parameter wrong, on roughly 1 request in 4. The team’s instinct: swap to Opus 4.8 ($5 / $25) for more raw IQ. Is that the fix?

Restate → the naive take → the insight → the rule

Restate the problem

A capable model is picking wrong tools and mis-filling arguments a quarter of the time. That’s a reliability problem in the act step of the loop — not a reasoning problem in the model’s head.

The naive take

“It’s not smart enough — buy more IQ.” So you move to a pricier model. It might nudge accuracy up a point or two, but it costs ~1.7× more per token, and the wrong-tool calls keep happening — because the model was never confused about reasoning; it was confused about the interface.

The insight

These are ACI failures. Forty tool definitions flood the context (a >10K-token tool surface); several schemas are ambiguous; none carry a sample call. The model is guessing at a badly documented API — exactly what a good human-facing API would fix with docs and examples.

The rule / trade-off

Invest in the interface before the model. Turn on Tool Search so only relevant tools load (~85% token cut; 79.5→88.1 on the benchmark), add Tool Use Examples to the ambiguous tools (72→90 on complex parameters), and move multi-tool sequences into Programmatic Tool Calling. Same Sonnet, sharply higher reliability — and cheaper than the Opus upgrade would have been.

the fix is in the tool definition, not the model// BEFORE — a bare schema the model has to guess against

  "description": "Refund an order.",
  "input_schema": { "order_id": "string", "amount": "number" } }

// AFTER — crisp description + a concrete example (the ACI investment)
{ "name": "issue_refund",
  "description": "Refund a *shipped* order. amount is in cents, ≤ order total.",
  "input_schema": { "order_id": "string", "amount_cents": "integer" },
  "examples": [ { "order_id": "A-1042", "amount_cents": 2500 } ] }
Say it back in one breath

“A quarter of my tool calls were wrong, and my instinct was to buy a smarter, pricier model. But the failures were interface failures — 40 tools flooding context, ambiguous schemas, no examples — so raw IQ wouldn’t fix them. I invested in the agent-computer interface instead: tool search to load only what’s relevant, examples on the ambiguous tools, programmatic calling for sequences. Same model, far more reliable, and cheaper than the upgrade.”

Reach for a better interface before a bigger brain. Most agent failures live in the ACI, not the model.

The phase ahead

You now hold the primitive the whole phase is built on: one model, three model-driven channels, one read → act → observe loop, composed into workflows and agents — and an interface you invest in like a product. Next, we compose it deliberately: the workflow patterns (prompt chaining, routing, parallelization, orchestrator–worker), then true agents that plan and act in a loop, and the evaluation and guardrails that keep them honest. Every one of them is this lesson’s augmented LLM, arranged.
