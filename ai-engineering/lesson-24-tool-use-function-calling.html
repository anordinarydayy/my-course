Tool Use & Function Calling

Everything in Phase 4 rests on one move: letting the model do something other than talk. A tool is how you give a language model hands — a typed contract that says "here is a function you may ask me to run." But the model never runs it. It emits a structured request; your code executes; you feed the result back; it decides what to do next. This lesson is that loop — reason → act → observe — and the small pile of rules (schemas, stop conditions, matching IDs, tool_choice, parallel calls, and tool design) that keep it from hanging, looping, or lying.

Original study notes. The build lesson that opens the agents phase in earnest. It draws on public sources — Anthropic's Building Effective Agents and "Writing effective tools for AI agents," the Messages API tool-use documentation, OpenAI's function-calling and Structured Outputs guidance, and the Model Context Protocol spec — but every explanation, diagram, and line of code is our own. 00 The gap

A model that can only talk

You have a strong model. Ask it "what's the weather in Hanoi right now?" and it will either refuse or invent a plausible number, because its weights froze at training time and it has no line to the outside world. Ask it "what is 19.99 × 1.08?" and it will answer confidently and often wrongly, because it is predicting the next digit, not calculating. A language model can reason about actions in astonishing detail — and take exactly none of them. It is a brain in a jar.

The first fixes people reach for are brittle. You can paste today's weather into the prompt — but that doesn't scale to live, unbounded, or private data you can't pre-fetch. Or you can instruct the model to print a line like SEARCH: weather Hanoi and scrape it out with a regex — but now you're re-parsing free text that the model formatted by hope, and one stray comma breaks the pipe. Both hacks are trying to route around the same missing primitive: a reliable, structured hand-off between "the model wants something done" and "your code does it."

That primitive is tool use, also called function calling. You describe a set of tools — each a name, a plain-English description, and a JSON-Schema for its inputs — and pass them alongside the message. The model, trained to recognize when a tool fits the request, replies not with prose but with a structured tool_use request: call get_weather with {location:'Hanoi'}. Your harness — your code — runs the real function and hands the output back. The model proposes; your code disposes. It never executes anything itself; it only ever asks.

The one-line version

A tool call is a request, not an execution. Claude emits {name, input}; you run the function; you return the result as a tool_result. Keep that boundary sacred and everything else in this lesson — schemas, the loop, IDs, tool_choice, parallelism — is just bookkeeping around it. Break it (imagine the model actually running your database delete) and you've built something nobody can reason about or secure.

01 The core cycle

reason → act → observe ▸

Here is the whole mechanism as one turn of a loop. The model reasons about the request, acts by emitting a tool request, then you run it and feed back what happened so it can observe and reason again. Step through a single weather lookup and watch where control actually lives at each moment — and, crucially, where the loop stops:

The shape is worth memorizing because every agent is this loop with more tools and more turns. You send the conversation plus the tools. Claude comes back with stop_reason: "tool_use" and one or more tool_use blocks. Your code runs each one and appends the results as a new user-role message. You send the whole thing again. You keep going while the stop reason is tool_use — and you stop the instant it is anything else (end_turn, max_tokens, stop_sequence, or a refusal). That "while / stop" is the single most important line of the whole pattern, and the next section pins down the four objects that flow around it.

02 The wire format

The contract in four objects

Strip away the SDK sugar and the entire protocol is four JSON shapes. Learn these and you can read any provider's tool-use trace. First, the tool definition — what you pass in. It is a name, a description the model actually reads to decide whether and how to call it, and an input_schema that is plain JSON Schema:

1 · tool definition (you send this in the tools[] array){
  "name": "get_weather",
  "description": "Get current weather for a city. Returns a short summary.",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {"type": "string",
                   "description": "City, e.g. 'Hanoi, Vietnam'"},
      "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
    },
    "required": ["location"]
  }
}

The description is not decoration — it is the model's only clue about when this tool applies, so it does more routing work than the name. Second, when Claude decides to call it, you get back a tool_use block inside the assistant turn: a unique id, the name, and the input the model synthesized to match your schema.

2 · tool_use block (Claude returns this; stop_reason == "tool_use"){"type": "tool_use",
 "id": "toolu_01A09q...",
 "name": "get_weather",
 "input": {"location": "Hanoi, Vietnam", "unit": "celsius"}}

Third, you run the real function and hand the output back as a tool_result block — carried in a user-role message, because from the model's point of view the environment is "speaking" to it. The one non-negotiable field is tool_use_id, which must exactly match the id from the tool_use block. Set is_error: true to tell the model the call failed so it can recover instead of trusting garbage.

3 · tool_result block (you send this back, in a user message){"type": "tool_result",
 "tool_use_id": "toolu_01A09q...",   # MUST match the tool_use id
 "content": "31°C, humid, light rain",
 "is_error": false}                   # true → a recoverable failure

The fourth object isn't a block at all — it's the stop_reason on the response, and it is what drives the loop. When it is tool_use, there is work for you to do; on anything else, the model is done talking. Two rules turn these four objects into a system that doesn't error out:

  • Key the loop on stop_reason, not on "did a tool get called" or a fixed counter. Looping "until a tool call" can hang forever if the model keeps calling tools; capping at a fixed count can cut off the final answer mid-thought. Loop while stop_reason == "tool_use"; exit on end_turn / max_tokens / stop_sequence / refusal.
  • Answer every tool_use, with a matching ID, before the next model call. If a turn contains three tool_use blocks, you must return three tool_result blocks, each with the right tool_use_id. A missing or mismatched ID isn't a soft failure — the next request errors outright.
The two bugs everyone ships first

(1) Forgetting the stop condition — the loop either spins or truncates because it keyed on the wrong signal. (2) Mismatched or missing IDs — you served two of three tool_use blocks, or copied the wrong tool_use_id, and the API rejects the whole turn. Both are invisible in a one-tool demo and both bite the moment real traffic produces a multi-tool turn. Write the loop against stop_reason and iterate over every tool_use block from day one.

03 The steering dial

Steering the call: tool_choice ▸

By default the model decides for itself whether a tool is even needed — which is usually what you want, but not always. Sometimes you need to guarantee a tool fires (you're using the call as structured output); sometimes you need to force one specific tool; sometimes you want the tools visible for context but off-limits this turn. That dial is tool_choice, and it has four settings. Watch the same turn resolve under each:

Two settings deserve extra care. Under any or tool you can add disable_parallel_tool_use: true to force exactly one call — useful when a downstream step expects a single structured result. And when you truly need the model's arguments to conform to your schema no matter what, Anthropic supports strict: true on a tool definition to guarantee the input matches; combine it with tool_choice:{type:'any'} and you've guaranteed both that a tool is called and that its inputs are valid. The other big providers mirror this vocabulary closely, with one sharp incompatibility worth internalizing:

Anthropic (Messages API)

  • auto — Claude decides (default when tools are present).
  • any — must call some tool; no plain-text answer.
  • tool — force one named tool, e.g. {type:'tool',name:'get_weather'}.
  • none — no tools this turn (default when you pass none).
  • disable_parallel_tool_use:true (+ any/tool) → exactly one call; strict:true guarantees schema-valid inputs.

OpenAI (the mirror)

  • auto (default), required (must call ≥1), none, or a named {type:'function',name:'...'}.
  • parallel_tool_calls:false → at most one call per turn.
  • strict:true (Structured Outputs) needs additionalProperties:false and every property in required; model optional fields as nullable.
  • The trap: Structured Outputs is not compatible with parallel calls — to guarantee schema conformance you must also set parallel_tool_calls:false.
  • Keep fewer than ~20 tools live at a turn's start for reliable routing.
04 More than one hand

Two calls at once: parallel tool use ▸

Ask "what's the weather in Hanoi and Tokyo?" and the naive loop does the obvious, slow thing: fetch Hanoi, send it back, fetch Tokyo, send it back — two full round trips for two independent lookups. But the two calls don't depend on each other, so serializing them is pure wasted latency. Claude 4 models handle this well by default: a single assistant turn can emit several tool_use blocks at once. Step through how one turn fans out and comes back together:

The engineering rule from Section 02 does the load-bearing work here: every tool_use block in the turn must get a tool_result, and they all go back in one user message before the next model call. You run them concurrently — asyncio.gather, a thread pool, whatever your stack uses — so wall-clock time is the slowest single call, not the sum. Newer Claude models also ship token-efficient tool use and improved parallel calling by default, so you get fewer output tokens per call and better fan-out without opting into anything. Parallelism is free performance only when the calls are genuinely independent; if Tokyo's query depends on Hanoi's answer, you're back to a sequential chain — and that's fine, that's just the loop turning twice.

05 The interface is the product

Designing good tools: the ACI

Here's the part that separates a demo from a system that works: the model is only as capable as the tools you hand it. Anthropic calls this the agent–computer interface (ACI) — the tool layer is a UI, except the "user" is a model, and like any UI its ergonomics decide whether the thing succeeds. The classic beginner mistake is to wrap your REST API one endpoint to one tool. That floods the model with thin, chatty tools and forces it to orchestrate five calls to do one job. Design for the model's job instead:

  • Consolidate, don't mirror. Build few high-leverage tools, not 1:1 API wrappers. Replace list_users + list_events + create_event with a single schedule_event; replace get_customer_by_id + list_transactions + list_notes with one get_customer_context that returns the recent slice the agent actually needs. Fewer round trips, less context, better accuracy.
  • Name and namespace clearly. Use unambiguous, prefixed names like asana_projects_search so the model routes correctly when many tools are present.
  • Return human-readable fields, drop the noise. Give back names, not raw UUIDs; strip low-signal columns. A tool that returns a wall of internal IDs makes the model burn tokens and mis-reason. Consider a response_format knob (concise vs detailed) — one Slack-thread tool cut a sample from 206 tokens to 72 just by exposing "concise."
  • Write errors as guidance, not tracebacks. The single highest-leverage habit. Instead of Error 400, return "No results found. Try narrowing your search, or run several targeted searches instead of one broad query." A recoverable error message is a second chance for the model to get it right on its own.
  • Enforce a token budget. Paginate, filter, and truncate. A tool that can dump 50,000 tokens into context will, and it will wreck both cost and reasoning.

Good ACI design also has a hard economic edge, because tool definitions cost tokens on every single request. Their names, descriptions, and schemas are billed as input tokens, and the API auto-injects a hidden tool-use system prompt on top. Those numbers are small per tool but compound fast:

Model (illustrative, mid-2026)Price $/Mtok · in / outAuto-injected tool-use overhead / request
Opus 4.8$5 / $25~500 tok (auto/none) · ~590 (any/tool)
Sonnet 4.6$3 / $15~500 · ~590
Haiku 4.5$1 / $5~500 · ~590
Plus each tool definition ~50–100 tokens — a 10-tool agent can add ~2,000–5,000 input tokens to every call, inflating cost and, past a point, degrading routing. tool_choice:none with zero tools adds 0.

That cost is why the 2025–2026 frontier moved toward loading tools lazily. Anthropic's Tool Search Tool (part of "advanced tool use," shipped late 2025) lets Claude discover and load tool definitions on demand instead of front-loading all of them — reported as roughly an 85% token reduction with large libraries, and it lifted accuracy on a tool-heavy eval markedly (for one Opus generation, ~49% → 74%). Reach for it once your tool definitions exceed ~10K tokens or you're wiring in 10+ MCP tools. The companion trick, programmatic tool calling, has Claude orchestrate tools by writing code in a sandbox so intermediate results never hit the context window — a large token cut on multi-step research. Different mechanisms, same lesson: past a handful of tools, which tools are even visible becomes a first-class design decision.

06 Put it together

The loop, end to end

Time to make it real. The worked example below is the entire pattern in one runnable file — two well-designed tools, the reason→act→observe loop, the stop condition, ID matching, and a recoverable error message — using the Anthropic Messages API. It's the honest version of every "agent" you'll build after this: a while loop around one API call.

Worked exampleweather + calculator · full tool loop
A compound question the model can't answer alone

The user asks: "What's the weather in Hanoi, and what is 19.99 × 1.08?" One clause needs a live lookup; the other needs real arithmetic. The model can do neither from its weights — and a naive "call one tool, then answer" shape breaks the instant a second tool is needed. You need a loop that keeps going until the model has everything and decides, on its own, to stop.

The whole pattern, in real Python

weather_calculator_agent.py — the reason→act→observe loopimport anthropic
client = anthropic.Anthropic()

TOOLS = [
  {"name": "get_weather",
   "description": "Current weather for a city, as a short summary.",
   "input_schema": {"type": "object",
     "properties": {"location": {"type": "string",
                    "description": "City, e.g. 'Hanoi, Vietnam'"}},
     "required": ["location"]}},
  {"name": "calculator",
   "description": "Evaluate one arithmetic expression, e.g. '19.99 * 1.08'.",
   "input_schema": {"type": "object",
     "properties": {"expression": {"type": "string"}},
     "required": ["expression"]}},
]

def run_tool(name, args):                        # the ONLY place code runs
    if name == "get_weather":
        return "31°C, humid, light rain"          # ← your real API call
    if name == "calculator":
        try:    return str(eval(args["expression"], {"__builtins__": {}}))
        except Exception as e:                    # recoverable, not a traceback
            return f"Could not evaluate: {e}. Send a plain expression."

msgs = [{"role": "user",
         "content": "Weather in Hanoi, and what is 19.99 * 1.08?"}]

while True:
    r = client.messages.create(model="claude-opus-4-8",
            max_tokens=1024, tools=TOOLS, messages=msgs)
    if r.stop_reason != "tool_use":              # ← the stop condition
        break
    msgs.append({"role": "assistant", "content": r.content})
    results = [{"type": "tool_result", "tool_use_id": b.id,
                "content": run_tool(b.name, b.input)}
               for b in r.content if b.type == "tool_use"]
    msgs.append({"role": "user", "content": results})

print("".join(b.text for b in r.content if b.type == "text"))
Restate & the naive miss

The model can neither fetch weather nor multiply reliably. A one-shot "call a tool, then answer" breaks here because two tools are needed, possibly across turns — and if the model got the calculator input slightly wrong, a one-shot flow has no way to retry. You need iteration keyed on the model's own signal, not on a guess about how many calls it'll make.

The move — loop on stop_reason

The while True re-calls the API and breaks the moment stop_reason isn't "tool_use". Inside the loop, the list comprehension serves every tool_use block in the turn — so parallel calls (Section 04) work with zero extra code — then appends all results in one user message. Claude sees Hanoi's weather and the product 21.59 together and writes the final answer.

The invariant — propose/dispose, IDs match, errors recover

run_tool is the only place code executes; the model just names a tool and inputs. Each result carries b.id as its tool_use_id, so IDs always match. And the except returns guidance, not a stack trace, so a bad expression becomes a retry instead of a dead end. (In production, sandbox the calculator — eval here is illustrative.)

Say it back in one breath

“The model can reason but not act, so I expose get_weather and calculator as tools and run a loop: send the conversation plus the tools, and while stop_reason is tool_use, execute each requested tool in my harness, return a tool_result with the matching tool_use_id, and send it back. It exits the instant Claude answers with end_turn. The model never runs my code — it only asks; I dispose.”

Every agent you meet after this is this file with more tools, a spend cap, and a retry policy. The loop and the stop condition are the load-bearing parts — get those right and the rest is tool design.

That loop is the primitive the whole phase is built on. A model that plans, calls a tool, observes the result, and decides whether to act again is the definition of an agent — and this loop with a single search tool is the agentic retrieval of Lesson 20. Swap that tool for a shell (bash), a database query, or a browser and you have the rest of Phase 4. A few frontier notes to carry forward: server tools like web_search and code_execution run on Anthropic's own infrastructure — you never build their tool_result, and they can return stop_reason:"pause_turn", which means "re-send to continue," not "done." Interleaved thinking lets the model reason between tool calls, but you must pass the complete, unmodified thinking blocks back when you continue, or the request errors. And the whole contract — name, description, JSON-Schema input — is exactly what the Model Context Protocol standardizes so any client can talk to any tool server, which is precisely where Lesson 25 picks up.

★ Review

Summary & self-test

One idea runs through the lesson: a tool call is a request, not an execution — the model proposes a structured tool_use, your harness runs it, you return a tool_result with a matching ID, and you loop on stop_reason until the model answers — then steer with tool_choice, fan out with parallel calls, and design the tools like a UI for a model.

02Four objectsTool definition (name/description/input_schema), tool_use (id/name/input), tool_result (tool_use_id/content/is_error), and stop_reason — which drives everything.
03reason → act → observeSend → tool_use → execute → tool_result → repeat WHILE stop_reason=="tool_use"; exit on end_turn / max_tokens / stop_sequence / refusal.
04Match every IDEach tool_result.tool_use_id must equal a tool_use.id, and ALL tool_use blocks get a result before the next call — or the request errors outright.
05tool_choice dialauto (decide), any (some tool), tool (force one named), none (text only). disable_parallel / parallel_tool_calls:false forces exactly one; strict guarantees the schema.
06Parallel by defaultClaude 4 emits several tool_use blocks in one turn; run them concurrently and return all results in one user message — one round trip, not N.
07Design the ACIFew consolidated tools, namespaced names, human-readable outputs, actionable error messages, token budgets. Definitions cost tokens on every request.
08The agent primitiveThis loop with one tool is an agent. Server tools, interleaved thinking, Tool Search, and MCP scale it — and bridge straight into L25.
