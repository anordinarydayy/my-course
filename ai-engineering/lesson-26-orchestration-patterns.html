Orchestration Patterns

One augmented LLM is a component. A system is many of them wired together so the whole is more reliable than any single call. This lesson is the four workflow patterns that do that wiring — routing, parallelization, orchestrator–workers, and evaluator–optimizer — each one the precise cure for a specific failure of the naive "one big prompt" build. You'll see when each earns its place, what it costs in tokens and coordination, and the single rule that decides whether to reach for it at all.

Original study notes. The composition lesson that opens Phase 4. It draws on public sources — Anthropic's Building Effective Agents and its 2025 write-up of a production multi-agent research system, Chip Huyen's AI Engineering, the MAST failure taxonomy, Cognition's "Don't Build Multi-Agents," and the OpenAI Agents SDK / LangGraph framing — but every explanation, diagram, and example is our own. 00 The composition problem

One prompt to do every job

Picture the obvious first build for a customer-support assistant: one prompt, one model call. Into that single prompt you cram the greeting style, the refund policy, the SQL for the orders table, the escalation rules, the tone guide, and a handful of few-shot examples covering every situation you could think of. It shines in the demo. Then it meets real traffic, and every improvement you make to one behavior quietly dents another — sharpen the refund logic and the technical answers drift; add a safety instruction and the greetings turn robotic. You are tuning one bundle of instructions against a dozen unrelated objectives, and there is no setting that is good at all of them at once.

That is the failure orchestration exists to remove. The cure isn't a bigger prompt or a smarter model — it's structure. Instead of one call that does everything, you compose several small, well-scoped calls and connect them with ordinary code: classify here, fan out there, loop until a check passes. Each call does one thing, is prompted and evaluated in isolation, and can be swapped or made cheaper without disturbing its neighbors. Call the unit an augmented LLM — a model call equipped with retrieval, tools, and memory (Lesson 22). Orchestration is the discipline of combining those augmented calls into a system that is reliable precisely because no single call is asked to carry too much.

Anthropic's Building Effective Agents (Dec 2024) draws the line that organizes this whole phase. Workflows orchestrate LLMs and tools through predefined code paths — you, the engineer, wrote the control flow. Agents let the model dynamically direct its own process and decide its own tool usage at runtime. The four patterns in this lesson are all workflows: the shape is fixed in code, only the contents are generated. (Prompt chaining — decompose into an ordered pipeline of steps — is the fifth, and you have already met it as decomposition.) Agents are the next lesson, and every reliable agent has one of these workflows beating at its center.

The one distinction to keep

Workflows = predefined code paths you control. Agents = paths the model controls. Start with the simplest workflow that works, and only hand the model the wheel when a fixed path genuinely can't express the task — because every bit of control you give up, you pay back in tokens, latency, and new ways to fail. Most production "AI systems" are workflows, not agents, and that is a feature.

01 Classify, then specialize

Routing: classify then specialize ▸

Routing is the cleanest composition of all, and the direct antidote to the mega-prompt of §00. Its move: split the work into two calls — a cheap classifier that reads the input and tags it, and a set of specialist paths, each with its own prompt, tools, and guardrails, one per tag. The classifier answers "what kind of thing is this?"; the specialist answers it. Watch a support ticket flow through:

Routing removes a failure the mega-prompt couldn't escape: a single prompt tuned across unrelated input types is mediocre at all of them, and any fix to one class silently taxes another. Split them, and each lane is optimized in isolation — the refund path never has to also be good at stack traces. The pattern's second gift is cost. Because the classifier already sized the query, you can send the easy 80% to a cheap model and reserve the expensive one for the hard tail: route "what are your hours?" to Haiku 4.5 (~$1/$5 per Mtok, illustrative, mid-2026) and a subtle reasoning bug to Sonnet 4.6 (~$3/$15) or Opus 4.8 (~$5/$25). You stop paying frontier prices for trivial turns. The one rule that makes it safe: the classifier must not answer — it only labels, so the routing decision stays cheap, auditable, and separate from the work.

02 Two ways to run in parallel

Parallelization: section and vote

Some work splits into pieces that don't depend on each other — and running them one after another in a single call wastes both latency and reliability. Parallelization runs independent LLM calls at the same time and aggregates the results in code. It comes in two flavors that fix two different problems, and conflating them is a common mistake.

Sectioning splits a task into independent subtasks that run concurrently, then combines them programmatically. The canonical case: run the answering model and a separate safety guardrail on the same input in parallel. Because the guardrail is its own isolated call, the answer prompt can't argue it into submission — a jailbroken answer still faces an untouched judge. Other sectioning uses: score a draft on three evaluation dimensions with three parallel calls, so each call fully attends to one rubric instead of splitting attention across all of them; or fan a long document out into sections graded independently and merged.

Voting runs the same task multiple times for diverse samples and takes a majority or a threshold. Reviewing a code diff for vulnerabilities, you might fire three-to-five prompts in parallel and flag the diff if any crosses the threshold — deliberately trading tokens for recall on the failure you least want to miss. The reliability gain is real and measured: Wang et al.'s self-consistency — sample several chain-of-thought paths, keep the majority answer — lifted GSM8K by +17.9%, SVAMP by +11.0%, and AQuA by +12.2% over a single greedy pass.

Sectioning — divide the concern

  • Different subtasks, run concurrently, merged in code.
  • Removes attention-splitting: one call juggling unrelated concerns does each worse.
  • Isolation is the point — a guardrail in its own call can't be gamed by the answer prompt.
  • Aggregation is deterministic (AND/OR, sum, concat), not another LLM guess.

Voting — repeat and count

  • Same task, several samples, majority or threshold.
  • Removes single-pass unreliability: one unlucky generation no longer decides.
  • Tune the threshold by risk — "flag if any run trips" maximizes recall.
  • Caveat: naive majority amplifies a wrong answer when the truth is a minority mode; gains plateau as K grows. Prefer confidence/entropy-weighted or critique-based aggregation on hard tasks.

Both variants share one precondition: the pieces must be genuinely independent. Section only what doesn't interlock, and vote only where a count is meaningful. Miss that and you've paid for parallelism that produces contradictions your aggregator can't reconcile — which is exactly the trap the next pattern is built to navigate when the pieces aren't known in advance.

03 Decompose, dispatch, synthesize

Orchestrator–workers ▸

Routing and parallelization both assume you already know the subtasks — the lanes are fixed, the sections predetermined. Many real tasks don't grant you that. You can't know, before reading a research question, how many sources it needs; or, before reading a bug report, how many files a fix touches. Orchestrator–workers handles exactly this: a lead LLM decomposes the task at runtime, delegates the pieces to worker LLMs, and synthesizes their outputs into one answer. The number and nature of the subtasks are decided by the model from the input — that runtime decomposition is the precise difference from parallelization's fixed sections.

Two properties make this pattern earn its cost. The first is compression: each worker explores its slice in a separate context window and returns only condensed findings, so the lead's limited context fills with conclusions instead of raw intermediate tokens — the design scales past a single window. The second is that synthesis is a real step, not concatenation: the lead reconciles overlaps, fills gaps, and decides whether to loop. Anthropic's June 2025 write-up of its multi-agent research system is the canonical build — an Opus-4-class lead spins up three-to-five Sonnet-class subagents in parallel, each running several tools at once, followed by a dedicated CitationAgent — and it cut research time up to 90% on complex queries while outperforming a single Opus agent by 90.2% on their internal eval. The example that separates the two: "identify the board members of every IT company in the S&P 500" is unsolvable in one context but trivial once decomposed per company. It fits work with heavy parallelism, information exceeding one window, and many complex tools — and fits coding far less well, since most coding subtasks interlock (more in §05).

A research task, decomposed and synthesized

The task: "Compare how the three largest cloud providers priced GPU inference in 2025, and where each is heading." No single page holds the answer; it spans provider docs, pricing pages, and analyst notes — and you don't know up front how many sources each provider needs. This is the shape orchestrator–workers is for.

The move: lead decomposes → workers fan out → lead synthesizes

orchestrator-worker in ~30 lines — Anthropic Messages APIimport concurrent.futures as cf
from anthropic import Anthropic

client = Anthropic()
LEAD, WORKER = "claude-opus-4-8", "claude-sonnet-4-6"   # illustrative, mid-2026

def lead_plan(query):                        # 1. DECOMPOSE - count chosen at runtime
    r = client.messages.create(
        model=LEAD, max_tokens=1024,
        tool_choice={"type": "tool", "name": "delegate"},
        tools=[{"name": "delegate",
                "description": "Break the task into independent subtasks",
                "input_schema": {"type": "object", "properties": {"subtasks": {
                    "type": "array", "items": {"type": "object", "properties": {
                        "objective": {"type": "string"},
                        "output_format": {"type": "string"}}}}}}}],
        messages=[{"role": "user", "content": f"Plan research for: {query}"}])
    return r.content[0].input["subtasks"]

def worker(task):                            # runs in its OWN context window
    r = client.messages.create(
        model=WORKER, max_tokens=1024,
        system="You are a focused sub-agent. Return only condensed findings + sources.",
        messages=[{"role": "user", "content": task["objective"]}])
    return r.content[0].text                  # compressed, not raw tokens

def synthesize(query, findings):             # 3. SYNTHESIZE - a distinct step
    joined = "\n\n".join(f"- {f}" for f in findings)
    r = client.messages.create(
        model=LEAD, max_tokens=2048,
        messages=[{"role": "user", "content":
            f"Question: {query}\n\nFindings:\n{joined}\n\n"
            "Merge into ONE cited answer; flag any gaps."}])
    return r.content[0].text

def research(query):
    tasks = lead_plan(query)                             # decompose
    with cf.ThreadPoolExecutor() as pool:                # 2. DISPATCH in parallel
        findings = list(pool.map(worker, tasks))
    return synthesize(query, findings)                   # synthesize
Restate

Answer a question whose number of subtasks isn't knowable in advance — each provider may need one source or five — and stitch the pieces into a single coherent, cited comparison.

Naive — and why it breaks

Hand the whole question to one call and it either overflows its context reading every source serially, or short-cuts to a shallow from-memory summary that's stale and uncited. One context can't both hold all the raw material and reason cleanly over it.

The move

lead_plan forces a delegate tool call so the lead emits a clean list of subtasks — the count decided from the query, not the code. ThreadPoolExecutor runs the workers concurrently, each in its own context returning only condensed findings. synthesize is a separate lead call that merges, reconciles, and flags gaps — the step that turns three summaries into one answer.

The invariant

Workers compress; the lead reconciles. A worker's context never pollutes the lead's, and synthesis is reasoning over results, not string concatenation. Give each worker an explicit objective + output format (the schema does this) so two workers can't silently do the same job.

Say it back in one breath

“A single call can't both hold every source and reason over it, so a lead model decomposes the question into subtasks at runtime — one per provider — dispatches them to workers that each explore in their own context and return only condensed findings, then synthesizes those findings into one cited answer, flagging gaps. Workers compress, the lead reconciles; the number of workers came from the input, not the code.”

The three moving parts are always the same: decompose (runtime, dynamic), dispatch (parallel, isolated contexts), synthesize (a distinct reasoning step). Drop synthesis and you have concatenation; hard-code the count and you have parallelization, not orchestration.

04 Generate, critique, revise

Evaluator–optimizer ▸

The patterns so far fetch or dispatch; none of them looks at the output and asks "is this actually good?" Evaluator–optimizer closes that gap with a loop: one LLM (the optimizer) generates, a second LLM (the evaluator) grades the result against explicit criteria and returns feedback, and the optimizer revises — repeating until the evaluator accepts. It removes the first-draft-is-final failure, and it mirrors human editorial review: the value isn't writing faster, it's the second pair of eyes.

The loop is only worth building when two conditions hold. First, clear evaluation criteria must exist — a rubric the evaluator can apply consistently; a vague "make it better" produces oscillation, not convergence. Second, iteration must measurably improve the output — if v2 isn't reliably better than v1, you're spending tokens to stand still. When both hold, the payoff is real: iterative literary-translation refinement catches idioms a single pass flattens, and multi-cycle search-then-analyze research converges on answers a one-shot can't reach. The OpenAI Agents SDK models it directly as a task agent inside a while-loop with an evaluator that returns pass/fail plus feedback. The mandatory guardrail is a max-iteration cap (three-to-four cycles), because this pattern's sharpest failure mode is non-termination — an evaluator that never quite accepts, or an optimizer that over-edits a good answer into a worse one. That failure has a name in the taxonomy we meet next: no/incorrect verification and premature-or-endless termination.

05 What fan-out costs

The multi-agent tax

Every pattern in this lesson buys capability with tokens, and the bill is not subtle. Anthropic's June 2025 measurements: a single agent uses roughly 4× the tokens of a chat exchange, and a multi-agent system about 15×. On the BrowseComp benchmark, token usage alone explained ~80% of the performance variance (tool-call count and model choice made up the rest) — a blunt way of saying the systems that win are mostly the ones that spend more. An orchestrator that spawns N workers each making M tool calls multiplies cost roughly N×M, so fan-out only pays when the task value is high: legal due diligence, competitive intelligence, biomedical review. Anthropic gates subagent count by complexity for exactly this reason — one agent with 3–10 tool calls for fact-finding, 2–4 subagents for a direct comparison, 10+ only for genuinely complex research. Their early systems spawned 50 subagents for a trivial query. Don't.

The token bill is the predictable cost. The insidious one is coordination. The MAST taxonomy ("Why Do Multi-Agent LLM Systems Fail?", 2025, validated at NeurIPS) studied real multi-agent traces and found failure rates of 41–86.7%, sorted into three buckets: specification & system-design flaws (~41.8%), inter-agent misalignment / coordination (~37%), and task verification & termination (~21%). The finding to internalize: failures come from the interaction between agents, not from weak individual models — architecture and coordination, not capability, are usually the culprit. Agents duplicate each other's work (Anthropic saw two subagents both research 2025 chip supply while a third drifted to 2021), misread their roles, withhold information, or terminate before verifying. The fix lives in the brief: give every worker an explicit objective, output format, tool guidance, and task boundary. Vague instructions are what make workers collide.

There's an influential counter-position worth holding beside all this. Cognition (the Devin team) argued in "Don't Build Multi-Agents" (June 2025) for single-threaded linear agents, on two principles: share context — full agent traces, not just individual messages, and actions carry implicit decisions, so conflicting decisions produce broken results. Their example: parallel subagents built a Mario-style background and a mismatched bird that couldn't be combined, because neither could see the other's choices. For long tasks they compress history with a dedicated model rather than fanning out. Tellingly, they reversed in March 2026 with "Devin can now manage Devins" — a coordinator that scopes work and assigns independent pieces to managed Devins in isolated VMs — conceding fan-out works for genuinely independent subtasks, while still preferring isolation over shared mutable state. The synthesis of both camps: fan out only when subtasks are truly independent and low-coordination; stay single-threaded when they share context or interlock. That's why multi-agent helps research (naturally parallel) far more than coding (mostly interdependent).

The tax, in one line

Agents cost ~4× a chat and multi-agent ~15×, and ~79% of real multi-agent breakdowns trace to specification ambiguity plus coordination — not to weak models. So fan out only for independent, high-value subtasks, and write each worker a brief so explicit that two of them physically cannot do the same job.

06 Handoffs, tools, and the rule

Handoffs, agents-as-tools, and the one rule

When you do compose these systems, two coordination primitives now show up in every framework, and they map exactly onto two patterns from this lesson. A handoff transfers control: a triage agent classifies and hands the conversation to a specialist, which becomes the active agent — that is routing. Agents-as-tools keeps control: a manager agent calls a specialist for a bounded subtask, gets the result, and integrates it — that is orchestrator–workers. Ask "do I want to give up control, or borrow a capability?" and the answer tells you which primitive to reach for.

The OpenAI Agents SDK (March 2025, the production successor to the now-deprecated Swarm) frames the whole space as two strategies: orchestrate via LLM — let the model plan each step, flexible but unpredictable in cost and latency — versus orchestrate via code — deterministic control flow, predictable speed. The code-orchestration patterns map one-to-one onto this lesson: structured outputs, agent chaining, an evaluator while-loop, and parallel execution via asyncio.gather. LangGraph reached 1.0 in October 2025 with durable execution — automatic checkpointing at every node, so hour-long runs survive restarts and resume exactly where they stopped — and its langgraph-supervisor is the orchestrator pattern over shared state. Useful tools, but heed Anthropic's warning: a framework adds abstraction that hides the actual prompts and responses, making debugging harder and tempting you toward complexity you don't need. Start from raw API calls; adopt a framework only once the abstraction demonstrably pays for itself.

Which is the rule the whole lesson has been circling: add complexity only when it demonstrably improves outcomes. Begin with a single optimized LLM call. Reach for a workflow when a fixed structure removes a named failure — routing for straddled input types, parallelization for single-pass unreliability, orchestrator–workers for unpredictable decomposition, evaluator–optimizer for first-draft errors. Reach for a full agent only when even a workflow can't express the task. Every rung up that ladder trades latency and tokens for capability, so measure the trade before you climb.

The phase, opening

This is the composition toolkit for everything that follows. You can now name four workflows, say which failure each cures, and price each in tokens and coordination risk — and, crucially, know when to build none of them. ★ Review

Summary & self-test

One idea runs through the lesson: a reliable AI system is small augmented-LLM calls composed with code — route to specialize, parallelize to de-risk, orchestrate to decompose the unpredictable, and loop to refine — each pattern the cure for one named failure, and each paying in tokens and coordination, so you add it only when it demonstrably beats a single call.

01Compose, don't cramOne mega-prompt tuned for everything is good at nothing — each fix taxes another behavior. Wire small, isolated augmented calls with ordinary code.
02Workflows vs agentsWorkflows = code-defined paths you control; agents = the model controls the path. Start with the simplest workflow; hand over the wheel only when forced.
03RoutingClassify once (the classifier never answers), then dispatch to a specialist path. Removes the straddled-prompt failure and sizes cost — Haiku for easy, Opus for hard.
04ParallelizationSectioning = independent subtasks + a guardrail in its own call. Voting = same task, majority/threshold. Self-consistency lifted GSM8K ~+18%.
05Orchestrator–workersLead decomposes AT RUNTIME, dispatches to isolated-context workers, then SYNTHESIZES. Subtasks aren't predetermined — that's the difference from parallelization.
06Evaluator–optimizerGenerate → critique vs criteria → revise, capped at N cycles. Works only with clear criteria and real per-cycle gains, or it over-edits or never terminates.
07The multi-agent taxAgents ~4× chat tokens, multi-agent ~15×; token use explains ~80% of variance. MAST: 41–87% failure, ~79% from spec + coordination, not weak models.
08The one ruleAdd complexity only when it removes a named failure and beats a single call. Single call → workflow → agent, measuring the trade at each rung.
Self-test before you move on

Cover the notes and answer aloud:
(1) Why does one mega-prompt covering every input type get worse as you tune it, and what's the structural fix?
(2) State the workflows-vs-agents distinction. Why start with the simplest workflow and hand control to the model last?
(3) Routing: what failure does classify-then-specialize remove, how does it also cut cost, and why must the classifier not answer?
(4) Distinguish sectioning from voting. Give a concrete use of each, and name voting's failure mode.
(5) What makes orchestrator–workers different from parallelization? What do "compression" and "synthesis" each contribute, and why isn't synthesis just concatenation?
(6) When does evaluator–optimizer pay off, and what are its two failure modes when those conditions don't hold?
(7) Give the token multipliers for agent and multi-agent vs chat. Per MAST, where do most multi-agent failures come from, and when should you not fan out?
