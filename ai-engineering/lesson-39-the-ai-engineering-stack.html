00 The frame

The model is one box

Here is the trap that every AI project falls into at least once. You wire an app to a model, ask it a hard question, and it answers — fluently, correctly, impressively. The demo lands. Everyone in the room concludes that the model is the product, and that shipping is now a matter of buying enough tokens. Then you put it in front of real users and it falls apart: it’s slow when the vendor is busy, expensive when someone loops it, wrong when it hasn’t seen your data, unsafe when someone pokes it, and completely opaque the first time it gives a bad answer you need to explain. Nothing about the model changed between the demo and the disaster. What changed is that production asked for everything the demo didn’t: freshness, safety, reliability, cost control, and the ability to debug and improve.

That gap is the whole subject of this course, and this lesson is where it snaps into a single picture. AI engineering is systems engineering. A foundation model is a remarkable component — but it is a component, one box on a diagram, and the discipline is in the boxes around it: how you construct the context it sees, how you validate what goes in and comes out, how you route and cap and cache the calls, how you watch what happens, and how you turn what happens into the next version. Chip Huyen frames the build-out of a generative-AI platform as exactly this kind of layering — enhance the context, put in guardrails, then add a model router and gateway — not as “pick a bigger model.” The model is the engine; the stack is the car.

The demo mindset

  • Model = product. The app calls the model and returns the text. That’s the “architecture.”
  • Better = bigger. When it’s not good enough, reach for a smarter, pricier model.
  • One vendor, one call. No fallback, no cap, no cache — whatever the SDK does is what happens.
  • Opaque by default. A bad answer is a shrug; there’s nothing to inspect and nothing to learn from.

The systems mindset

  • Model = one component. Value and reliability live in the scaffolding around it.
  • Better = better system. Context, guardrails, routing, and evals move the numbers — often more than IQ does.
  • A gateway, not a call. Route, cache, cap spend, fail over — the call is a managed operation.
  • Observed & improving. Every request is traced; production failures become tomorrow’s tests.

01 The path

The request path ▸

Start where a user starts: with a single request, and follow it all the way to a response. In a production system that journey is not “app calls model.” It passes through a short, ordered pipeline — each stage a box you built in an earlier phase — and the model sits roughly in the middle of it, not at the front. Before the model: construct the context, then validate the input. At the model: route the call, then run the augmented LLM and its tools. After the model: validate the output, then respond. And running underneath the entire path, touching every hop, are two things that never appear in the demo: observability, which records what happened, and the feedback loop, which turns what happened into the next iteration. Step a single request through it:

Read the picture as an assembly line, not a black box. Every stage has one job, and each is a topic from an earlier phase given a fixed place in the pipeline. The blueprint boxes (context, gateway, model) are the “happy path” of turning a question into an answer; the rust boxes (input and output guardrails) are the checks that keep bad input out and bad output in; the green boxes (observability, feedback) are what make the whole thing operable and improvable over time. The next three sections walk the stages in order, then we zoom out and redraw the same pipeline as a stack of layers.

Why order matters

The sequence is not decorative. You construct context before you validate, so the guardrail sees the real prompt, injected data and all. You validate input before the gateway, so you never pay for a request you were going to reject. You route before the model runs, so cost and failover decisions are made once, centrally. And you validate output before you respond, so a malformed or unsafe generation never reaches the user. Move a box and you break a guarantee.

02 Before the model

Context construction & input guardrails

The first thing that happens to a request is the thing the demo skips entirely: you build the context. A bare call sends the user’s raw words to the model and hopes. A production call assembles a deliberate prompt out of several sources — the system instructions, the retrieved documents that ground the answer in your data (Phase 1’s RAG), the relevant slice of the user’s session and long-term memory (Phase 3), and the current question, formatted and ordered on purpose. Huyen’s framing is exact: context construction for foundation models is like feature engineering for classical machine learning. The model’s answer is only ever as good as the context you feed it, and constructing that context well is engineering, not luck.

Then, before that context reaches a model, it passes an input guardrail. This is a defensive layer, and it earns its place because the input is only partly trusted: it contains user text, and often retrieved or tool-fetched text that an attacker may have planted. The input guard does three jobs. It screens for prompt injection — instructions hidden in a document or a message that try to hijack the model’s behaviour. It strips or masks PII before sensitive data is sent to a third-party model. And it rejects the malformed — empty, oversized, or off-policy requests you shouldn’t process at all. Reject early and you save every downstream cost: no gateway call, no tokens, no risk. In 2025–2026 this is usually a small stack of purpose-built checkers (a PII detector, an injection classifier, a policy filter) rather than one monolith — a layered defence at the very front door.

Naive: raw text → model

  • User’s words go straight to the model.
  • No grounding — the model guesses from stale training data and may hallucinate.
  • No memory — every turn starts from zero.
  • No screen — injected instructions and PII pass through untouched.

Production: construct & validate

  • Retrieve the right facts, pull session + memory, assemble a deliberate prompt.
  • Grounded in your data as of now — not the model’s cutoff.
  • Continuity across turns and sessions via external memory.
  • Guard the input: injection screen, PII mask, reject the malformed — cheaply, up front.
The invariant

Context construction decides what the model gets to see; input guardrails decide what it’s allowed to see. Together they are the reason a production system answers from your policy document instead of a plausible invention — and the reason a hidden “ignore your instructions” buried in a retrieved page gets caught before the model ever reads it. Neither box requires a smarter model; both make an average model behave.

03 The routing layer

The gateway: one door to every model

Once the context is built and validated, the request needs to reach a model — and this is where the demo’s single hard-coded SDK call becomes a liability. A production system does not let application code call model providers directly. It puts a gateway in the middle: one internal endpoint that every request goes through, which owns the messy operational concerns so no individual feature has to. This is Phase 6’s subject, and it is the layer that most cleanly separates a prototype from a platform.

A gateway does a handful of jobs, each of which you’d otherwise reinvent badly in every service. It routes: an intent classifier can send a trivial query to a cheap, fast model and reserve the expensive one for genuinely hard requests. It caches: an exact or semantic cache returns a stored answer for a repeated or near-identical question in milliseconds instead of paying a full round-trip. It enforces rate limits and spend caps: token-aware budgets that stop a runaway agent or a looping prompt chain from quietly burning the monthly budget. It fails over: when the primary provider is down, slow, or throttling, it retries against a backup so a single vendor’s bad afternoon isn’t your outage. And because everything funnels through it, it is the natural place to attach keys, access control, and per-request tracing — one door, instrumented once.

The decoupling win

The gateway does for models what an API gateway does for microservices: it turns “my code is welded to one vendor’s SDK” into “my code asks an internal endpoint for a completion.” Swap providers, add a cheaper model, change the cache policy, tighten a budget — and no feature code changes. Vendor lock-in, uncontrolled spend, and single-provider fragility are not three problems; they are one missing box.

04 Zoom out

The layered architecture ▸

The request path is the horizontal view — a line from user to reply. Now flatten it into the view engineers actually draw when they whiteboard a platform: a stack of layers. Every stage you just walked belongs to one of five layers, and — this is the point of the whole lesson — each layer is a phase of this course. Nothing here is new material; the architecture is just the syllabus stacked. Step through the layers and watch each one name the responsibility it owns and the phase that built it:

Two of these layers deserve a second look because the demo has neither. Observability sits under everything and records it: a trace of each request stitches together the retrieval, the prompt sent, the model call, every tool invocation, the guardrail decisions, latency, and token cost — so that when an answer is wrong you can see why instead of guessing. And the feedback layer closes the circle: online evals score a sample of live traffic, failing traces become new evaluation cases, and those cases drive the next round of prompt, retrieval, and model changes. That is the data flywheel — the loop that lets the system get better in production rather than merely not crash. Together they are what turns “it works today” into “it keeps getting better,” and they are exactly the ops lessons this phase moves into next.

The stack is the syllabus

Context/Data was Phases 1 & 3. Model/Agent was Phase 4. Orchestration and the gateway were Phases 4 & 6. Observability was Phase 2 and continues in L41. Feedback is L42. If a layer feels shaky, you now know exactly which phase to re-read — the architecture doubles as a map back to the material.

05 The state rule

Stateless services, state lives outside

There is one structural rule that quietly governs every box on the diagram, and it is the difference between a system that scales and one that mysteriously corrupts itself under load: the services are stateless; the state lives in external stores. Each service — the context builder, the guardrails, the gateway, the agent runner — should be a stateless worker that takes a request carrying its full context, does its job, and returns a result, holding nothing important in local memory between calls. Everything that must persist — conversation sessions, long-term memory, the response cache, the vector index, evaluation datasets — lives outside the service, in a store built for it.

The reason is practical. Stateless services scale horizontally: because no request depends on which instance handled the last one, you can run twenty copies behind a load balancer, restart any of them, and lose nothing. The moment a service keeps important state in process, you’ve created a machine you can’t safely restart and can’t safely duplicate — and, in agent systems especially, a breeding ground for the classic failure modes: stale state from parallel overwrites, lost context across retries, and race conditions between instances. The fix is to push the state down into stores chosen for the job: a fast in-memory store (Redis-style) for sessions and cache, a durable database for long-term memory and audit logs, a vector index for semantic recall. The augmented LLM from Phase 4 lives in this layer too — the agent runner is a stateless service; the memory it reads and writes is an external store.

This is also where the last stage of the request path earns its keep: the output guardrail. After the model (or agent) produces a result, a stateless validator checks it before it leaves — enforcing the JSON schema a downstream service expects, confirming structured output actually parses, running a safety classifier over free text, and catching the malformed or low-quality response. Like the input guard, it needs no model intelligence and no in-process state; it’s a pure function from output to verdict. Structured, validated output is what lets the rest of your ordinary software trust what the model returned.

Stateful service (the trap)

  • Keeps sessions / memory / cache in local process memory.
  • Can’t scale out — requests are pinned to one instance.
  • Restarting or crashing loses live state.
  • Parallel instances overwrite and race each other.

Stateless + external state

  • Service is a pure worker; state sits in a store built for it.
  • Scale to N instances behind a load balancer, restart freely.
  • Sessions/cache in a fast store; memory/audit in a durable DB; recall in a vector index.
  • Output guardrail is a stateless validator — schema, structure, safety.
Pitfall: “just keep it in memory for now”

The most expensive shortcut in an AI service is caching a session or an agent’s working state in a module-level variable because “it’s simpler.” It works on your laptop and on one instance, then silently breaks the day you scale to two — users see other users’ context, retries lose their place, and the bug is nearly impossible to reproduce. Decide the state boundary early: services compute, stores remember. Externalize state before you scale, not after it corrupts.

06 The threads

Reliability, cost, safety ▸

The diagram so far reads like a pipeline of independent boxes, but three concerns refuse to stay in any single box — they run through all of them, like threads woven across the stack. These are the cross-cutting concerns, and they are what senior AI engineers actually spend their time on: reliability, cost, and safety/quality. None of them is a feature you bolt on at the end; each is a property you either designed into every layer or didn’t. Step through the three and watch each one light up the boxes it touches:

Notice what the figure teaches: each concern touches multiple layers, and each layer serves multiple concerns. Reliability is retries and timeouts at the gateway, fallbacks across providers, and a retry when the output guardrail rejects a malformed schema. Cost is token discipline in how you construct context, caching and spend caps at the gateway, and routing cheap-first. Safety and quality is clean input at the front, grounding in the middle, and schema plus evals at the back. You cannot make a system reliable, cheap, or safe by fixing one box — you make it so by pulling the thread through every box. This is why AI engineering is systems engineering: the properties that matter most are emergent across the stack, not resident in the model.

The demo that died in production

Your prototype answers support questions beautifully on your laptop: an app function calls one vendor’s model with the user’s question and returns the text. You ship it. Within a week it’s flaky (the vendor throttles you at peak and requests just fail), expensive (someone’s retry loop ran up a surprise bill), unsafe (a user pasted text that made it leak its system prompt), and opaque (a customer got a wrong answer and you have no idea why). Restate → walk the stack → what’s missing → the rule

Restate the problem

The model itself is fine — it answered well in the demo. The failures are systemic: availability, spend, safety, and debuggability. Those are properties of the architecture, and the architecture here is a single line: app → model.

Walk the naive path

The request goes from the app straight to one provider’s SDK and back. There is no context construction, no input check, no routing, no output check, and nothing recording what happened. Every cross-cutting concern — reliability, cost, safety — has nowhere to live, because there are no layers for them to thread through.

Name what’s missing

Four boxes. No gateway → no failover, no spend cap, welded to one vendor (that’s the flakiness and the bill). No guardrails → unvalidated input and output (that’s the prompt leak). No observability → a bad answer is undebuggable (that’s the shrug). No feedback loop → the system never learns from the failure, so it recurs.

The rule / trade-off

Don’t reach for a bigger model — reach for the minimal production stack. Wrap the call in a gateway (route, cache, cap, fail over), put a guardrail on each side (input: injection + PII; output: schema + safety), trace every hop, and route failing traces into your evals. Same model, but now reliable, bounded, safe, and improvable. Each box you add is a phase you already studied — you’re assembling, not inventing.

client = Client(api_key=KEY)

def answer(question):
    reply = client.chat(model="big-model", prompt=question)   # one vendor · no fallback
    return reply.text                                    # no guardrail · no trace


# AFTER — the minimal production stack: every box has an owner (and a phase)
def answer(question, user):
    ctx   = build_context(question, user)     # Ph 1 & 3 · retrieve + memory + assemble
    clean = input_guard(ctx)                  # Ph 1 · PII scrub + injection screen
    reply = gateway.call(clean,               # Ph 6 · route · cache · fail over
                        fallbacks=["b", "c"], retries=2, cap_cents=5000)
    safe  = output_guard(reply)              # Ph 6 · schema + safety validation
    trace(question, ctx, reply, safe)        # Ph 2 / L41 · observe every hop
    return safe                              # L42 · a bad answer becomes tomorrow's eval

“My demo was one line — app calls model — and every production failure came from a box that line was missing. So I stopped shopping for a smarter model and assembled the stack I’d already learned: I built context (Phases 1 & 3), guarded the input (Phase 1), routed it through a gateway with failover and a spend cap (Phase 6), ran the augmented LLM (Phase 4), guarded the output, traced every hop (Phase 2 / L41), and fed the failures back into my evals (L42). Same model — reliable, bounded, safe, and improving.”
