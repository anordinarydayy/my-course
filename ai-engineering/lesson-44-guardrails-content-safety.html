Guardrails & Content Safety

This is Phase 8 — Safety, Security & Responsible AI, and it opens with the most practical idea in the phase: a guardrail is the control layer wrapped around the model. Not a nicer prompt, not a better fine-tune — code you own that inspects what goes IN and checks what comes OUT, because you cannot trust a language model to police itself. Its instructions live in the same text channel an attacker controls, so "be safe" is a suggestion, not a boundary. This lesson builds the boundary: input rails, output rails, a layered cheap→expensive stack of checks, PII handling that never leaks the data it protects, and the honest trade-offs — false positives, false negatives, latency, and the fact that no classifier is a silver bullet against prompt injection.

00 The problem

You can't ask a model to be safe

Give a strong model a system prompt that says "never reveal secrets, never produce hateful content, never leak personal data," and it will agree enthusiastically. Then a user pastes a block of text, your retrieval step pulls in a web page, or a tool returns a payload — and every one of those arrives in the same undifferentiated text channel as your careful instructions. The model has no privileged, tamper-proof place to stand: it can't reliably tell "the developer's rule" from "a later sentence that says to ignore the developer's rule," because to a next-token predictor both are just tokens. This is the root cause of prompt injection, which sits at LLM01 — the top entry — on the OWASP LLM Top 10 for 2025, and it is still #1 because mixing instructions and data is baked into how these models read input.

So the safety you actually ship cannot live inside the prompt. It has to live outside the model, as programmatic checks in code you control. That is a guardrail: a deterministic-where-possible layer that runs before the model to decide what may reach it, and after the model to decide what may reach the user. The model proposes; your rails dispose. Same discipline as the tool-use lesson — never let the model be the thing that enforces the boundary.

Why must it be two-sided? Because a request can be poisoned on the way in and a response can be harmful on the way out — different failure modes needing different catches. An input can smuggle a jailbreak, carry personal data you must not forward, or push a disallowed intent. An output can be toxic, echo private data straight back, violate your JSON contract, or confidently hallucinate a policy that doesn't exist. Guard only the input and toxic completions still slip out; guard only the output and you've already fed an injection to a tool-wired model. You need both faces of the sandwich, and the rest of this lesson is what goes in each.

The one-line version

A guardrail is an external, programmatic check around the model — not an instruction inside it — because the prompt is attacker-controllable and the model can't police itself. Input rails decide what reaches the model; output rails decide what reaches the user. Keep them simple: a guardrail you make clever enough to be jailbroken is just a second model to attack.

01 The two rails

The guardrail sandwich: checks before and after ▸

Picture the model as the filling and your checks as the two slices of bread. On the way in, the input rails run first — only what survives them reaches the model. On the way out, the output rails inspect the draft before a single character reaches the user. When a high-risk input trips a rail you fail closed: block rather than forward, because forwarding a suspected injection to a tool-calling model is exactly the mistake the rail exists to prevent. Step through a clean request and then a hostile one, and watch where each is caught:

The two rails hold different responsibilities — input rails are admission control, output rails are safe egress — and neither alone is a system. Keeping the lists concrete matters:

INPUT rails — before the model

  • PII detect + redact — strip or tokenize personal data so it never reaches the model or an upstream provider.
  • Injection / jailbreak classifier — flag attempts to override instructions or smuggle commands (the L43 topic).
  • Topic / policy filter — reject off-topic or disallowed intents your app was never meant to serve.
  • Abuse & rate limits — throttle floods and repeated probing.
  • Fail closed on high-risk input — when in doubt, block rather than forward.

OUTPUT rails — after the model

  • Content-safety classification — toxicity, hate, violence, self-harm, sexual content.
  • PII-leak check — did the model just emit private data it should not have?
  • Format / schema validation — does the structured output actually match the contract?
  • Groundedness / hallucination check — is the answer supported by the retrieved source (the L46 topic)?
  • Brand / competitor + secret deny-lists — keep specific strings from ever leaving the building.

Two design rules keep this honest. First, fail closed on the input side. A false alarm that blocks a legit request is annoying; forwarding a real injection into a tool-wired model is a breach — when the stakes are asymmetric, bias toward blocking. Second, keep the rail itself boring. The moment your input rail is an elaborate judge fed the attacker's text plus your private context, the attacker can jailbreak the judge too — you've just moved the target. Prefer deterministic checks, keep the judge's context minimal, and never hand a rail a secret it doesn't need.

The rail is an attack surface too

Every guard you add is also something that can be attacked, mis-tuned, or fed poisoned context. That's not an argument against guards — it's an argument for simple, least-privilege guards: deterministic filters first, a small classifier next, a judge only where nuance truly demands it, each with the narrowest context it can get away with. Complexity in a guardrail is surface area for the very attacks it's meant to stop.

02 Protect the data

PII: detect, mask, round-trip — and never log raw ▸

Personal data is the guardrail category with the sharpest legal edge and the easiest mistake. The rule of thumb: raw PII should never reach the model, the provider, or your logs — and getting that right is a small pipeline, not a single regex. You detect the sensitive spans, mask or tokenize them, send only masked text upstream, and de-tokenize on the way back so the entitled human still gets a correct answer. Microsoft Presidio is the canonical open-source tool (MIT-licensed, maintained into 2026): an Analyzer fusing spaCy / Hugging Face NER with regex, checksums, and context scoring, and an Anonymizer that masks, replaces, encrypts, or tokenizes — with recognizers for cards, national IDs, names, locations, phones, IBANs, crypto wallets, and more, across text, OCR'd images, and structured data. The move that makes round-trips work is reversible, format-preserving tokenization. Irreversible masking (a card number into ****) is perfect for a log line but useless if a downstream step must actually charge that card. So you replace 4111… with a stable token like <CARD_1>, keep the token↔value mapping in a vault you control, send the tokenized text upstream, and swap the real value back only when composing the reply for the authenticated user. This is exactly the shape AWS Bedrock Guardrails exposes as its sensitive-information filters, and it pairs with L41's log-scrubbing: whatever you write to traces, telemetry, or an eval store gets the scrubbed version, always.

Never log raw PII

The most common way a data-protection guardrail fails is protecting the model call but forgetting the logs. Write the raw request to a trace before redacting and the rail has leaked the very data it guards — often to a wider audience (every engineer with log access) than the model ever had. Redact before anything is persisted or forwarded, practice data minimization, and treat the vault as the only place a real value may rest.

03 Layer the checks

Cheap rules first, the LLM judge last ▸

No single check is both cheap and smart, so you don't pick one — you layer them from nearly free to genuinely expensive, letting each tier catch what the previous one missed. Tier 1 is rules: regex, deny/allow-lists, schema validators, rate limits — microseconds, effectively free, killing the obvious majority. Tier 2 is a small dedicated classifier for the fuzzy middle a regex can't express. Tier 3 is a full LLM-as-judge, reserved for nuance and high stakes. Cheap filters handle the obvious, the classifier the fuzzy, the judge the last, priciest calls. Step a message through the ladder and see where it stops:

What goes in each tier, in mid-2026 terms — illustrative and fast-churning, so design the rail interface to keep the specific model swappable:

The dedicated classifiers earn their place because they encode a taxonomy, not just a pattern. Meta's Llama Guard 4 (12B, natively multimodal, pruned from Llama 4 Scout) classifies both prompts and responses against the MLCommons hazard taxonomy — ~14 categories spanning privacy, hate, self-harm, sexual content, elections, and code-interpreter abuse — and its sibling Llama Prompt Guard 2 ships in 86M multilingual and 22M English sizes (the smaller ~75% cheaper/faster) for jailbreak and injection detection. The point isn't that the judge is best; it's that a judge on every request would blow your budget, so you spend it only where the cheap tiers can't decide.

Each layer catches what the last one missed

A regex can't tell that a politely-phrased request really seeks self-harm instructions; a small classifier can score that but may under-weight a subtle euphemism; a judge with full context reads the intent but costs 100× more per call. Stacked, they cover each other's blind spots — and ordered cheap→expensive with early short-circuiting, they do it without wrecking p95 latency.

04 The dial

False positives vs false negatives — and what to do on a hit

Every guardrail is a hop — another network call or model inference — so guards cost latency and money, and stacking several LLM judges can quietly dominate your p95. But the deeper trade-off is which errors you're willing to make. A false positive blocks legitimate content; a false negative lets harm through. You can't drive both to zero with one threshold: tightening the dial to catch more harm blocks more innocent traffic, and loosening it does the reverse. So you don't chase a universal setting — you calibrate the operating point to the stakes of your specific app, and you monitor both error types in production, not just the one that files complaints.

Tune toward fewer false negatives

  • High-harm domains: medical advice, child safety, self-harm, weapons.
  • You'd rather block a borderline-legit request than let real harm slip out.
  • Accept more friction and more human review; the cost of a miss is catastrophic.
  • Fail closed aggressively; escalate ambiguous cases to a person.

Tune toward fewer false positives

  • Low-harm, high-volume tools: a coding assistant, a writing helper.
  • Over-blocking tanks usefulness and drives users to unguarded tools.
  • Accept a slightly higher miss rate on low-severity content.
  • Prefer redaction or a softer action over a hard block on ambiguity.

When a rail does fire, "block" is only one rung of a ladder. From least to most disruptive: redact the offending span and continue; regenerate with a stricter instruction; escalate to a human for high-impact cases; block outright; or — increasingly the preferred default — emit a safe completion: a constructive partial answer with a safe alternative, not a blunt "I can't help with that." Blunt refusals tank helpfulness on legitimate dual-use requests and, when bypassed, fail all-or-nothing. OpenAI's GPT-5 safety training (the Aug 2025 paper From Hard Refusals to Safe-Completions, arXiv 2508.09224) reframes safety around the output — maximize helpfulness within a safety boundary instead of a binary comply/refuse decision on the input — reporting higher helpfulness on dual-use prompts and softer failures when something slips through.

The outer wrapper: governance

Guardrails are the engineering layer; governance is the wrapper around the whole program. The EU AI Act phases in on dated deadlines — prohibited practices and AI-literacy since Feb 2 2025, general-purpose-model (GPAI) obligations from Aug 2 2025, high-risk Annex III systems from Aug 2 2026, embedded and legacy-GPAI by Aug 2 2027. The NIST AI RMF (Govern/Map/Measure/Manage) and ISO/IEC 42001 (first certifiable AI-management standard, Dec 2023) stay voluntary but are the de-facto baselines, with a NIST crosswalk so one program satisfies all three. The transparency artifacts: model cards and system cards (e.g., the GPT-5 System Card, Aug 2025). Dates illustrative, mid-2026.

05 The hard truth

No classifier is a silver bullet

Here is the sentence that should keep you honest: prompt injection is not solved, and a classifier-based guardrail will not solve it for you. A 2025 joint study from OpenAI, Anthropic, and Google DeepMind — titled The Attacker Moves Second — took twelve published injection and jailbreak defenses and broke all of them with adaptive attacks tuned against each one. The lesson isn't "detection is useless" — detectors still filter huge volumes of low-effort abuse cheaply — it's that against a determined adversary no detector is robust alone, so you must pair detection with architecture: design the system so even a successful injection can't do much damage.

The clearest design lens is Simon Willison's lethal trifecta (Jun 2025): an exfiltration attack needs three ingredients at once — (1) access to private data, (2) exposure to untrusted content, (3) a channel to communicate externally. Remove any one and the chain breaks: an agent that reads your email but can only draft (not send) can't leak it. Meta's Agents Rule of Two (Oct 31 2025) turns this into a budget: within a session an agent should satisfy at most two of {processes untrusted input, accesses sensitive data/systems, changes state or communicates externally} without a human in the loop (Meta softened the label from "safe" to "lower risk" after pushback — these are risk-reduction heuristics, not guarantees). And DeepMind's CaMeL (Apr 2025) shows a real design-level defense: a dual-LLM pattern where a privileged LLM plans and a quarantined LLM only touches untrusted data, with capability/taint tracking enforced in code — blocking ~67% of attacks on AgentDojo. Even that principled design isn't 100%, which is precisely the point.

Detection (necessary, not sufficient)

  • Classifiers, deny-lists, moderation APIs, the layered stack from §03.
  • Cheaply filters the huge volume of low-effort abuse.
  • Falls to adaptive, defense-aware attacks — per Attacker Moves Second.
  • Keep it simple so it isn't itself an easy target.

Architecture (the durable half)

  • Least-privilege tools; break the lethal trifecta; honor the Rule of Two.
  • Human-in-the-loop for high-impact actions (excessive agency = LLM06).
  • Treat retrieved content & tool output as untrusted data, not instructions.
  • Contain blast radius so a successful injection still can't exfiltrate.

Agentic systems add an authorization dimension on top of content safety. The Model Context Protocol's authorization spec (revised June 2025; a label like 2025-11-25 is illustrative) classifies MCP servers as OAuth 2.0 Resource Servers and requires clients to send Resource Indicators (RFC 8707) so a token is scoped to one server and can't be replayed against another — the classic "confused deputy" defense. The threats to internalize: tool poisoning (malicious instructions hidden in a tool's description that the model obeys), rug-pull redefinition after approval, and lookalike-server phishing. The implication is blunt: tool descriptions and outputs are untrusted content that must pass the same input rails, and high-privilege tool calls sit behind explicit human approval.

One more output-side category deserves a name: groundedness. Hallucination isn't a bug you can prompt away — OpenAI's Sept 2025 paper Why Language Models Hallucinate (arXiv 2509.04664) argues it's a near-inevitable consequence of the pretraining objective plus calibration, reinforced by benchmarks that reward confident guessing over "I don't know." That's a systems argument for an external groundedness check rather than trusting the model to self-correct: evaluate faithfulness with corpora like RAGTruth (span-level hallucination labels) and FActScore (decompose an answer into atomic facts, verify each against a source), a check AWS Bedrock Guardrails ships productized as a contextual grounding check.

The attacker moves second

Whatever guardrail you ship, assume a motivated adversary will see it and tune around it. Don't bet safety on a single detector: pair detection with least-privilege architecture, break the lethal trifecta, cap agent capability with the Rule of Two, and keep humans on the high-impact loop. Detection buys you the 95% of low-effort abuse; architecture protects you from the 5% who are actually trying.

06 Build it

Wrapping a support bot in guardrails

Time to build. Two real incidents motivate it. Once, a support bot summarized a ticket and echoed the customer's full card number into the summary field — an output-side data leak. Another time, an angry user typed a slur and the bot, trying to mirror the user's language, repeated it back verbatim — toxic output triggered by unguarded input. Neither is a capability problem you fix with a better prompt; both are missing rails. So we wrap the bot: PII redaction and an injection classifier going in; a content-safety classifier, a PII-leak check, and schema validation coming out; and a graded action on any violation — redact, regenerate, or safe-refuse — logging only the scrubbed trace.

Reason about itsupport bot · input + output rails
A bot that leaked a card number and echoed a slur

The failures live on both faces of the sandwich. The slur got through because nothing screened the input or the output before it was mirrored back. The card leaked because the model was handed raw PII and no output rail checked what it emitted. The fix isn't one guard — it's a pipeline where the request is redacted and injection-checked before the model, and the draft is safety-classified, leak-checked, and schema-validated after, with the raw ticket never written to a log.

The guardrail pipeline, in pseudocode

support_bot_guardrails.py — input rails, model, output railsdef handle(user_msg):
    # ---------- INPUT rails (before the model) ----------
    clean, tokens = pii_redact(user_msg)        # NER → reversible tokens
    if injection_clf(clean).flag:              # jailbreak / injection (L43)
        return safe_refuse("can't help with that request")  # fail closed
    if off_topic(clean):                        # topic / policy filter
        return safe_refuse("that's outside what I can do")

    draft = model.generate(clean)               # model sees REDACTED text only

    # ---------- OUTPUT rails (after the model) ----------
    if content_safety(draft).unsafe:           # toxicity/hate/self-harm/...
        draft = regenerate(draft, stricter=True) or safe_refuse()
    if pii_leak(draft):                        # did it emit private data?
        draft = redact(draft)                   # scrub before it ships
    if not schema_ok(draft):                   # structured-output contract
        draft = repair_or_regenerate(draft)

    log(scrub(user_msg), scrub(draft))          # NEVER log raw PII
    return detokenize(draft, tokens)           # round-trip, for the user only
Restate & the two incidents

Map each failure to a missing rail. The echoed slur is toxic output triggered by unscreened input — now caught by content_safety(draft). The leaked card number is a PII leak — caught by pii_leak(draft), and largely prevented upstream because pii_redact meant the model never saw the raw number at all.

The move — input-side vs output-side coverage

Input rails are admission control: they fail closed, so injection and off-topic requests die before the model runs. Output rails are safe egress: toxicity, leaked PII, and broken schema are caught before a character reaches the user. Some harms can only be caught going in (an injection that would hijack a tool call); others only coming out (the model inventing a policy). That's why you need both — a single-sided guard leaves a whole class of failures uncovered.

The dial & the ladder

Each check has a threshold, and the app's stakes set it: a support bot skews toward fewer false negatives on PII leaks (a leaked card is catastrophic) but fewer false positives on tone (over-blocking normal frustration ruins support). The action escalates with severity: redact a stray entity, regenerate a toxic phrasing, and only safe_refuse when there's no safe partial answer — preferring a safe completion over a blunt "no." Whatever the outcome, only the scrubbed trace is logged.

Say it back in one breath

“I don't trust the model to be safe, so I wrap it. On the way in I redact PII into reversible tokens and run an injection classifier — fail closed. The model sees only redacted text. On the way out I run a content-safety classifier, a PII-leak check, and schema validation; on a hit I redact, regenerate, or emit a safe completion, and I log only the scrubbed trace, then de-tokenize for the user. The slur is an output-safety catch; the card leak is a PII catch prevented upstream by redaction.”

Order the rails cheap→expensive, short-circuit early, tune each threshold to the harm profile. The pipeline is the product — the classifier behind each stage is swappable, the shape of the sandwich is not.

That pipeline is the honest core of every "AI safety layer" you'll build after this: a redact-and-classify wrapper on the input, a classify-and-validate wrapper on the output, a graded action between. Scale it up and the parts get fancier — Presidio behind pii_redact, Llama Prompt Guard behind injection_clf, Llama Guard or a cloud API behind content_safety, an LLM judge behind the hard calls — but the structure doesn't move. And it plugs into the rest of Phase 8: the injection classifier is L43, the groundedness check L46, the log-scrubbing L41, and §04's governance wrapper is the paperwork that says you did it on purpose.
