GraphRAG & Agentic Retrieval

Flat top-k retrieval is a similarity prefilter: it grabs the k chunks that look most like your query and stuffs them in context. That is brilliant for the local "who/what/when/where" lookups of Lessons 15–18 — and it falls apart on two question shapes it can't even represent. This stretch lesson is about the two ways out: GraphRAG builds a smarter index so global and relational structure is precomputed; agentic retrieval installs a smarter policy so the model decides when, what, and how many times to fetch. Both cost far more than top-k — so the real skill is knowing when flat RAG has actually failed.

00 The failure that starts everything

Where flat top-k breaks

Everything in this phase has been one machine: chunk the corpus (Lesson 17), embed each chunk and index it (Lesson 16), and at query time embed the question, pull the k nearest chunks, maybe rerank them (Lesson 18), and hand them to the model. It works because of a quiet assumption — the answer lives inside a chunk that looks like the question. Ask "what is our refund window?" and the chunk containing that sentence is, by construction, the one whose embedding sits nearest the query. Local lookups are exactly this shape, and flat RAG nails them.

That assumption breaks in two structural ways, and "retrieve more chunks" fixes neither — because the problem isn't recall depth, it's that the answer isn't in any single chunk at all.

  • Global questions. "What are the main themes across the whole corpus?" No chunk contains "the themes" — that fact is a property of the entire collection. Top-k just returns k arbitrary fragments that happen to resemble the words "main themes," and the model summarizes those few, not the corpus. Bumping k from 5 to 50 gives you 50 arbitrary fragments.
  • Multi-hop questions. "Which university did the founder of the company that acquired Instagram attend?" The chain Instagram → Facebook → Zuckerberg → Harvard never co-occurs in one chunk. A single similarity lookup on the literal query can't traverse relationships; it can only match surface text, so it surfaces chunks about Instagram and stops.
  • The blind, single-shot policy. A third, quieter failure: flat RAG always fetches exactly k — for a "hi, how are you?" that needs zero lookups, and for a research question that needs many. The retrieval policy is fixed, blind, and one-shot, with no notion of "I have enough" or "I should look again, differently."
Two fixes for two failures

The two failures point at two different layers. Global and multi-hop are failures of the index — the structure you precomputed throws away relationships and corpus-level shape; GraphRAG fixes the index. The blind single-shot fetch is a failure of the policy — who decides to retrieve, and how often; agentic retrieval fixes the policy. They compose, and together they bridge straight into Phase 4.

The graph encodes that relationship explicitly, so the answer is three hops, no guessing. Now: how do you build that graph?

02 A smarter index

GraphRAG: a smarter index ▸

Microsoft's GraphRAG ("From Local to Global," 2024) attacks the global failure and the multi-hop failure with one restructured index. The insight: do the expensive, LLM-heavy structuring work once at build time, so that relationships and corpus-level shape are precomputed rather than reconstructed per query. Step through the build, then watch a global query get answered:

Two pieces deserve a second look. The Leiden hierarchy is what makes "global" tractable: communities nest, from a handful of broad C0 roots down to many fine-grained C3 leaves, so you can answer a sweeping question from a few root summaries instead of reading the whole corpus. In the paper, on million-token podcast and news corpora, GraphRAG beat naive vector RAG on comprehensiveness around 72–83% of the time and on diversity around 62–82% — exactly the axes top-k can't touch. The map-reduce is the query engine: map generates a partial answer plus a 0–100 helpfulness score from each relevant report in parallel; reduce drops the zero-scored ones, ranks the rest, and synthesizes one answer. And the headline efficiency win: answering from C0 root summaries instead of source text cut tokens per global query by roughly 97% (about 9–43× fewer). The multi-hop case from Fig. 1 is the same index used differently — instead of map-reducing summaries, you walk edges. One structure, both failures fixed. The catch is the bill for building it.

03 The cost is the point

LazyGraphRAG & the cost of a graph

Here is the discipline this whole lesson is really teaching: both of these techniques cost dramatically more than top-k, so you must prove flat RAG fails before you reach for them. GraphRAG's indexing pass runs an LLM over the corpus to extract entities, relationships, and claims and to write every community report — practitioners report it landing around 10–40× the cost of a plain vector pipeline (think tens to a couple hundred dollars to index a corpus that vector-indexes for cents). Pay that and you get a reusable, shareable index; pay it on a corpus that only ever needs simple lookups and you have lit money on fire.

The 2025–2026 shift that changed the economics is LazyGraphRAG. Its move is pure laziness in the good sense: defer all LLM work to query time. At build time it uses cheap classical NLP — noun-phrase extraction plus co-occurrence — to assemble a concept graph with no LLM summarization, so indexing costs the same as vector RAG (about 0.1% of full GraphRAG). At query time it runs a best-first (similarity-ranked chunks) plus breadth-first (LLM judges sentence-level relevance across communities) search over a few generated subqueries, governed by a single "relevance test budget" knob. The reported result is striking: it matches full GraphRAG's global-search quality at roughly 700× lower query cost, and at about 4% of global-search query cost it beats competing methods on both local and global queries. A complementary trick, dynamic community selection, prunes irrelevant communities with a cheap model before map-reduce, cutting token cost ~77% at no significant quality loss.

The order of operations

Before a graph: try the cheaper non-graph fixes first — structure-aware parent/child chunking (Lesson 17), the hybrid + rerank stack (Lesson 18), and contextual retrieval (prepend a chunk's surrounding context before embedding it, reported to cut retrieval failures by up to ~67%). Reach for a real graph only once you've shown a question class that genuinely needs relationships or whole-corpus synthesis — and you'll know, because you'll have measured it with the retrieval metrics of Lesson 21.

04 A smarter policy

Agentic retrieval: a smarter policy ▸

GraphRAG fixed the index. The blind, single-shot failure is a different bug entirely — it's about who controls retrieval. Flat RAG bolts a fixed prefilter in front of the model: always embed, always fetch k, always before generating. Agentic retrieval hands that decision to the model and exposes retrieval as a tool. Now the model can PLAN (decompose the question into sub-queries), ISSUE targeted searches, REFLECT on what came back (is it relevant? sufficient? contradictory?), and DECIDE whether to search again, search differently, fall back to the web, or answer. Retrieval stops being a subroutine and becomes a learned, iterative skill.

That loop is a whole family of techniques, and two named ones are worth holding in your head because they show the two places you can put the intelligence — inside the model, or around it:

Self-RAG — reflection inside the model

  • Trains the model to emit reflection tokens mid-generation.
  • Retrieve decides whether a lookup is even needed (so chit-chat fetches nothing).
  • ISREL grades each passage's relevance; ISSUP checks it actually supports the claim; ISUSE rates utility 1–5.
  • Result: retrieve-on-demand and self-critique are learned behavior; reported hallucination rates drop to ~5–6% vs naive RAG.

CRAG — a corrective evaluator around it

  • Adds a lightweight retrieval evaluator that grades docs correct / incorrect / ambiguous.
  • On low confidence it falls back to web search rather than answer on weak context.
  • Refines via decompose-then-recompose — strip retrieved text to key strips, drop the noise, recombine.
  • Model-agnostic: bolts onto any RAG pipeline as a grader + router (the red branch in Fig. 3).

The older cousins — Self-Ask, ReAct, and IRCoT (interleave chain-of-thought with retrieval, forming each follow-up query from the last intermediate answer) — established the pattern; the 2025 RL-trained search agents (Search-R1, DeepResearcher, R3-RAG) reward the model for good fetch-and-stop decisions. On the standard multi-hop benchmarks (HotpotQA, 2WikiMultiHopQA, MuSiQue, built to block shortcuts), iterative beats single-pass by meaningful margins — roughly +10 EM / +7.6 F1 over a single-pass dense baseline (e.g. EM 43→53). Fix A — decompose / retrieve / reflect (policy)

Plan: the model sees a 3-link chain and decomposes. Sub-Q1 "Who acquired Instagram?" → retrieve → Facebook. Reflect: need the acquirer's founder → Sub-Q2 "Who founded Facebook?" → Zuckerberg. Reflect: need his university → Sub-Q3 "Which university did Mark Zuckerberg attend?" → Harvard. Synthesize → "Harvard." Three targeted retrievals, each with an explicit query, beat one blind lookup.

Fix B — a 2–3 hop graph walk (index)

On the knowledge graph there's no re-prompting at all: Instagram --acquired_by--> Facebook --founded_by--> Zuckerberg --attended--> Harvard. The answer is a direct edge walk — precisely Fig. 1's right column.

The payoff — and the caveat to teach beside it

This is the gap the numbers reflect: iterative/agentic reports ~+10 EM / +7.6 F1 over single-pass on multi-hop QA, and a real 4M-doc deployment moved multi-hop correctness 41% → 78% and citation accuracy 87% → 96% by adding traversal — for about +130 ms p95. But Fix A cost 3 LLM+retrieval round-trips and Fix B cost an expensive graph build. Only reach for either once you've shown flat top-k actually fails this question.

Say it back in one breath

“Flat top-k returned Stanford because the query's nearest chunks were all about Instagram and its visible co-founder — the acquirer never co-occurred with the query terms, and more k just piles up more Instagram chunks. Two fixes: decompose into who-acquired / who-founded / which-university and retrieve-reflect each in turn, or walk three edges on a knowledge graph. Both cost far more than one lookup, so I only use them once I've proven the multi-hop class genuinely breaks flat RAG.”

The decompose-retrieve-reflect loop and the edge walk reach the same answer — one fixes the policy, the other fixes the index. In production you often have both, and a router decides which to spend on.

05 Pay only when you must

The router that pays only when flat RAG fails

If GraphRAG and agentic loops both cost multiples of top-k, and most real traffic is simple lookups, the obvious production answer writes itself: don't run the expensive path on every query — classify first. This is Adaptive or Router RAG, the de-facto 2026 architecture. A cheap classifier sizes each incoming query and sends it down the lane that can actually answer it, so you stop paying graph and agent costs on the ~80% of queries that are trivial.

The 2025 Agentic-RAG surveys (Singh et al.) gave the field its vocabulary — single-agent, multi-agent, hierarchical, corrective, adaptive, graph-based — and the framing of "System-1 vs System-2" retrieval is just this router by another name: cheap reflexive lookups for most queries, slow deliberate loops for the few that earn it. The pragmatic enterprise pattern is hybrid graph+vector (vector for broad recall, graph for relationship verification), accepting ~150–200 ms of orchestration overhead for ~15–25% accuracy gains — and reaching for the cheaper non-graph fixes of Lesson 18 first. Microsoft even shipped BenchmarkQED to auto-generate and grade global vs local query sets, because deciding which strategy a query needs is becoming its own discipline. The router is where the trade-off table from §03 becomes a runtime decision instead of an architecture commitment.

06 The bridge

Retrieval is the agent pattern

Step back and look at Fig. 3 again with fresh eyes. A model that plans, calls a tool, observes the result, reflects, and decides whether to act again — that is not a description of retrieval. That is the definition of an agent. Agentic retrieval is simply the agent loop with exactly one tool (search) wired in. Swap that tool for a code interpreter, a database query, or an API call and you have Phase 4. This is why the lesson sits where it does: it is the capstone of retrieval and the on-ramp to agents at the same time.

The same idea shows up in Anthropic's framing of just-in-time context engineering: instead of pre-stuffing everything into the window (the "stuff it all in" failure of Lesson 19), an agent holds lightweight identifiers — file paths, record IDs, queries — and pulls data into context on demand at runtime, exactly as Claude Code writes a targeted query over a database rather than loading the whole thing. "Retrieval-as-a-tool inside a reflective loop" is just-in-time context. And it's why the 2025–2026 "deep research" products — OpenAI, Gemini, Perplexity, Grok, Qwen — are all the same productized loop (plan → search → reflect → iterate) run for minutes over many sources: deep research is agentic retrieval with a generous budget.

Summary & self-test

One idea runs through the lesson: flat top-k is a similarity prefilter that structurally cannot answer global or multi-hop questions and fetches blindly — so fix the index with GraphRAG (entities → communities → map-reduce summaries), fix the policy with agentic loops (plan → retrieve → reflect → decide), and because both cost multiples of top-k, route each query to the cheapest tier that can actually answer it.
