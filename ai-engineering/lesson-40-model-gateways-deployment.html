Model Gateways & Deployment

This is where Phase 7 — Architecture, Deploy & Operations gets concrete. You have a working app; now it has to survive a provider outage, a price change, a model swap, and a Tuesday deploy without a war room. The move that makes all of that boring is one piece of plumbing: a gateway — a single internal API that sits in front of every model provider. Your app never calls a vendor SDK. It asks the gateway for a policy — fast-chat, smart-analysis — and the gateway decides which concrete model answers. Get that seam right and swapping models is a config edit, an outage is a fallback, and a rollout is a dial you turn one percent at a time.

Original study notes. The production-plumbing lesson of the operations phase. It draws on public sources for framing — Chip Huyen's AI Engineering, the LiteLLM and Portkey gateway docs, AWS's Bedrock and multi-provider-gateway solutions, Uber's shadow-deployment practice, and the OpenTelemetry GenAI semantic conventions — but every explanation, diagram, and line of config is our own. 00 The coupling

Married to one vendor

Here is how it happens, and it happens to everyone. You start with one provider because their model was best the week you began. You import their SDK, call client.chat(...) in the one place that needs it, ship, and move on. Six months later that same import lives in forty files — the summarizer, the classifier, three background jobs, the eval harness, a Slack bot someone bolted on. The model name is a string literal in each of them. Your business logic and one company's API are now the same object. You didn't decide that; it accreted.

Then the bill comes due, usually all at once. The provider has a two-hour outage, and because forty call sites talk to them directly, your product has a two-hour outage too — there is no seam to route around, no place to say "use someone else for the next hour." A cheaper model ships that would cut your costs in half, but adopting it means a code change and a redeploy in forty places, so you don't. The provider quietly retrains the model behind the same name and your carefully-tuned prompts start behaving differently, and you can't even tell which day it changed. Every one of these is the same wound: your application is coupled to a specific vendor and a specific model, and coupling is the thing production punishes.

The naive fixes make it worse. You add a try/except around the SDK call that falls back to a second provider — but now that failover logic is copy-pasted into forty files, each subtly different, and the second provider's SDK has a different request shape so half of them are wrong. You add a feature flag to pick the model — but it only covers the three call sites you remembered. You are patching a structural problem one hole at a time. What's missing isn't another patch; it's a seam: a single boundary that everything model-related passes through, so that "which provider, which model, what happens when it fails" becomes one decision made in one place instead of forty decisions scattered across the codebase.

The one-line version

The most valuable piece of production LLM plumbing is not a model — it's the gateway: one internal API in front of all providers that your app calls by policy name, never by vendor SDK. Every hard operational problem in this lesson — swaps, failover, spend caps, caching, canaries, observability — is a thing you can only solve once if there is exactly one place where model traffic flows. Build the seam first; everything else layers onto it.

01 The insight

The gateway seam ▸

Invert the dependency. Instead of forty files that each know "we use vendor X's model Y," you write one internal client that every file calls, and it knows how to reach any provider. Crucially, the app doesn't ask for a model — it asks for a policy. A policy is a name for an intent: fast-chat means "cheap and quick, high volume"; smart-analysis means "throw the expensive brain at this." Configuration — not code — maps each policy to a concrete, pinned model. Step through what changes when you swap the model behind a policy, and watch the one thing that doesn't move:

The payoff is that a whole category of "code change + redeploy + risk" collapses into "edit a value." Want to A/B two models? Point half the traffic for a policy at each — in config. A cheaper model ships? Repoint the policy and watch your dashboards. A model regresses? Flip it back. The app node in the diagram is byte-for-byte identical across every step, and that is the entire thesis: the gateway decouples your application from any specific model or provider. This is why, by mid-2026, the industry stopped debating whether to run a gateway — it is table stakes — and started debating only buy-versus-build and self-host-versus-managed. The next section is what you get to centralize once every request flows through that one seam.

02 The payoff surface

What lives at the gateway

A seam is only worth building if useful things can hang off it. The gateway's real value is that it is the one place every model request passes through, so every cross-cutting concern that used to be scattered (or missing) now has a single home. Six of them earn the gateway its keep:

  • Unified auth & key management. Provider API keys live in the gateway, not in forty services' env files. The app authenticates to the gateway with a virtual key; the gateway holds the real credentials and rotates them without touching app code.
  • Rate limits & hard spend caps. Per-key, per-team, per-user budgets with a hard ceiling — a runaway loop or a leaked key hits a wall at, say, $200/day and gets a 429 instead of a $40,000 invoice. This is only enforceable when spend flows through one meter.
  • Prompt & response caching. Identical (or, carefully, similar) requests can be served from cache, cutting cost and latency — shared across replicas via a store like Redis.
  • Automatic fallback & failover. Primary provider down? The gateway retries a secondary. One policy for the whole fleet (Section 03).
  • A single observability hook. Every request, its tokens, cost, latency, and outcome logged in one place — the seam is also the telemetry tap (Section 05).
  • Built-in A/B & canary. Traffic-splitting by policy makes gradual rollouts a config feature, not a bespoke deploy pipeline (Section 04).

You don't usually build this from scratch — a small ecosystem already solves it, and the three leading tools attack the problem from different angles. The trade-off that actually matters is where the request runs: a self-hosted gateway keeps every byte on your infrastructure and adds only microseconds even at thousands of requests per second; a managed one is less to operate but adds a real network hop and sends your traffic through a third party — a genuine data-sovereignty question if you have residency rules.

LiteLLM

  • Open-source SDK and proxy; the de-facto default.
  • Calls 100+ providers through one OpenAI-format API.
  • Virtual keys, per-key/team budgets with hard caps, ordered fallbacks (order=1 → order=2), instant cooldown on 429.
  • Redis for cross-replica limits + caching; Postgres for keys/spend.
  • AWS's own Multi-Provider Gen-AI Gateway is built on it.

OpenRouter

  • A managed routing network — the "no keys needed" end.
  • Buy credits; it routes across 70+ providers / 300+ models with failover by availability, price, or speed.
  • ~20+ free models for evaluation.
  • Fully managed: least to run, but every request takes the network hop.

Portkey

  • A control plane in front of keys you already hold.
  • Latency/cost-based routing, conditional fallbacks, canaries, circuit breakers, up to 5 retries with backoff.
  • Guardrails, PII redaction, audit trails, observability.
  • Gateway core open-sourced under Apache 2.0 (early 2026) — self-host the routing layer.

The cloud providers now offer first-party versions of this too. Amazon Bedrock Intelligent Prompt Routing (GA since April 2025) is a single serverless endpoint that predicts, per request, how hard the prompt is and routes to the best model within a family — advertised to cut cost up to ~30% without losing accuracy. That is exactly the "auto"/tiered-endpoint pattern the gateway thrives on: your app hits it by policy, and the endpoint (not your code) picks the concrete model. Bedrock and Vertex also underpin cross-provider failover in the gateway solutions above — the same Claude on two clouds, one as the other's backup.

Design rule

App code should contain zero concrete model names and zero vendor SDK imports. It knows policy names (fast-chat, smart-analysis) and nothing else. If you can grep your app for "gpt-" or "claude-" and get hits outside the gateway config, the seam is already leaking — and every leak is a future code-change-and-redeploy that should have been a config edit.

03 When a provider dies

Failover & circuit breakers ▸

The gateway is where reliability gets layered, and the order of the layers matters. A single request that hits a slow or failing provider should not become a user-visible error — nor should ten thousand simultaneous retries turn a struggling provider into a fully dead one. Three mechanisms stack: fallback (try someone else), retries with backoff (try again, politely), and a circuit breaker (stop trying for a while). Step through a primary provider failing over to a secondary:

Two details make this robust rather than fragile. First, retries must use exponential backoff with jitter. Naive fixed-interval retries synchronize — every client that failed at the same instant retries at the same instant, and the thundering herd finishes the provider off. Backoff spreads them out; jitter de-synchronizes them. Second, the circuit breaker is what stops you from hammering a provider that is already down: after a threshold of failures it opens, and the gateway skips the primary entirely for a cooldown window (LiteLLM cools a deployment down the moment it sees a 429; Portkey exposes explicit breakers). A probe later half-opens it to test the waters and closes it when healthy. The secondary is frequently the same model on a different cloud — Claude-on-Bedrock failing over to Claude-on-Vertex — which by mid-2026 is a standard, expected capability rather than a bespoke build.

The pitfall that deepens outages

Retries without backoff and a breaker are worse than no retries. A struggling provider under a retry storm goes from slow to dead, and your "resilience" code caused it. Always pair retries with jitter and a circuit breaker, and never treat the gateway itself as a single point of failure — run it HA with health checks, or the one seam everything flows through becomes the one thing that takes everything down.

04 Rolling out safely

Ship it: canary & shadow ▸

New models and new prompts are deploys, and LLM deploys are dangerous in a way that ordinary code deploys are not: the failure mode isn't a crash you can catch, it's a subtle quality regression that only appears on real traffic. A model that scored fine on your offline benchmark might, in production, emit longer outputs that break your UI, invent a new category of refusal, or hallucinate slightly more on the messy inputs your benchmark didn't cover. So you never big-bang swap a model in front of users. Two techniques — shadow and canary — let you prove a candidate on real traffic before it serves anyone. The precondition for both is that your inference-facing services are stateless: session, memory, cache, and vector state live in external stores (Redis, Postgres, a vector DB), never in the request-serving process, so you can run N identical replicas and roll new versions without draining in-process state. Step through the staged rollout:

The distinction to internalize: shadow serves nobody, canary serves a few. Shadow traffic runs the candidate on real requests but throws its output away, so you compare distributions — latency, output length, refusal rate, quality scores — with zero user risk; Uber's documented practice mandates a 24-hour shadow window with automated checks on distribution divergence and null rates before any canary traffic is allowed. Canary then sends a small real slice — a common ramp is 1% → 5% → 20% → 50% → 100% — while you watch the same signals on users who are actually being served. The non-obvious rule that saves you: make canary assignment sticky. Hash on a stable user key so a given user always lands on the same side; route them to the canary on one turn and the stable model on the next and you get an incoherent conversation. Two honest costs to budget for: shadow doubles your inference spend for its window (you run every request twice), and both techniques only pay off if you gate promotion on the live signals plus online evals — not the offline benchmark that missed the regression in the first place.

Shadow traffic

  • Mirror real requests to the candidate; discard its output.
  • Zero user risk — nobody sees the candidate's answers.
  • Compares distributions: latency, tokens, refusals, eval scores.
  • Costs a second inference per request — cap the sample.
  • Do it first (Uber: 24h) before any live traffic.

Canary traffic

  • Serve a small % of real users the candidate; ramp up.
  • Real user risk, contained — blast radius is the canary %.
  • Catches what only live serving shows (UI breaks, adoption).
  • Sticky assignment per user, or conversations go incoherent.
  • Promote to 100% or roll back — both are one config flip.
05 Pin it, then watch it

Version & observe everything

Two operational habits keep a live system from drifting out from under you. The first is versioning. Providers change models behind the same name — an unpinned latest can silently shift your output quality, latency, and cost overnight, and you'll spend a day debugging a "bug" that was really a vendor's Tuesday. So pin the concrete version explicitly (opus-4.8 → "2026-05-01"), and version your prompts in the same config alongside the model — a prompt is tuned for a specific model, and the two travel together. Because the gateway already maps policy → version, rollback is not a redeploy; it is one config flip back to the pin that worked. Keep that flip one keystroke away and a bad model becomes a five-minute event.

The second habit is observation — and LLM observability is genuinely different from ordinary APM, because your infra monitor has no notion of "semantically wrong." A response can return 200 OK in 400ms and be a confident hallucination; latency and error-rate dashboards will show you nothing. That gap is why a distinct tooling layer exists. The emerging standard for the raw telemetry is the OpenTelemetry GenAI semantic conventions — a namespaced set of gen_ai.* span attributes so traces are portable across tools:

On top of that raw telemetry sits a purpose-built tool layer: by 2026 roughly six platforms own the conversation — LangSmith (deepest LangChain/LangGraph ties, annotation queues), Langfuse (open-source, framework-agnostic, OTel-native), Arize Phoenix (strongest eval rigor: faithfulness, relevance, toxicity, hallucination, RAG retrieval), Braintrust (eval-first, CI/CD eval-gated deploys), and Datadog LLM Observability (ties LLM traces into infra APM). What sets them apart is online evals: LLM-as-judge and heuristic scorers running on live production traffic to catch hallucinations, retrieval failures, and answer-relevance drift that no infra monitor can see. Two cautions carry over from the pitfalls list: capturing full prompt/response content by default logs user text into your traces — a PII and compliance exposure, which is exactly why OTel keeps content off by default and content should be opt-in with redaction, separate retention, and its own IAM.

The data flywheel

Observation isn't just defense — it feeds improvement. Capture user signals (thumbs up/down, edits, whether the answer was adopted), run online evals, curate the failures into eval sets and fine-tuning/RAG-improvement data, redeploy, repeat. The 2025 EMNLP-industry Agent-in-the-Loop work formalized exactly this on a live support system, reporting +11.7% recall@75, +14.8% precision@8, +8.4% helpfulness, and cutting retraining cadence from months to weeks. Production traffic, routed through your gateway and watched by your observability, becomes the training signal for the next version.

06 Build it

Build the gateway

Let's make the seam real. The build below is the whole pattern: rip out the scattered vendor SDK, put one internal client in its place, and move every model decision — which model, what happens on failure, the spend ceiling, the rollout — into a config file the gateway reads. The code that call sites see shrinks to a single function; the operational power moves into config where it belongs.

Build walkthrough40 files → one internal client + config
The vendor SDK is in forty files, and it took you down

Your codebase imports one vendor's SDK in forty places, each with a hard-coded model string. When that vendor had a two-hour outage, all forty call sites failed at once and you went down with them — no seam to route around, no way to switch providers for the hour. The fix is structural: introduce a gateway so model choice becomes a config decision and an outage becomes a fallback, not an incident.

Before — the coupling, copied forty times

scattered across ~40 files — married to one vendorfrom vendor_sdk import Client
client = Client(api_key=VENDOR_KEY)

# every call site looks like this, model name baked in:
resp = client.chat(model="gpt-x-turbo", messages=msgs)   # ← hard-coded

After — one internal client, a policy not a model

the ONLY shape every call site uses, foreverfrom gateway import chat

# the app names an INTENT, never a model or a provider:
resp = chat(policy="smart-analysis", messages=msgs)

The config the gateway reads — where all the power now lives

gateway.yaml — policies, fallbacks, spend caps, rolloutpolicies:                      # app calls THESE names, not models
  fast-chat:       haiku-4.5    # cheap, high-volume
  smart-analysis:  opus-4.8     # the expensive brain

models:
  opus-4.8:
    provider: anthropic
    version: "2026-05-01"          # PIN it — never "latest"
    fallbacks: [claude-on-bedrock, sonnet-4.6, haiku-4.5]
                              # primary → secondary → cheap
  claude-on-bedrock:
    provider: aws.bedrock
    version: "2026-05-01"

budgets:
  smart-analysis:
    daily_usd_cap: 200            # HARD cap → 429 past this, not a $40k bill

retry:  { max: 3, backoff: exponential, jitter: true }
breaker: { trip_after: 5, cooldown_s: 30 }   # stop hammering a down provider

rollout:                       # canary a new model with zero app changes
  smart-analysis:
    canary: { model: opus-4.9-preview, traffic: 0.05, sticky_key: user_id }
    shadow: true                # mirror real traffic, don't serve it
Restate & the naive miss

The vendor SDK is the coupling: forty imports means forty places that must change to swap a model and forty places that die when the provider does. The naive fix — a try/except fallback pasted into each file — multiplies the problem instead of removing it. What's missing is a single seam every request passes through.

The move — one client, config-driven policies

Replace every SDK import with chat(policy=...). The gateway resolves the policy to a pinned model, walks the fallbacks chain on failure, enforces the daily_usd_cap, retries with backoff+jitter, and trips a breaker on a dead provider. Adopting a cheaper model or canarying a new one is now an edit to gateway.yaml — the app never changes.

The invariant & the knobs to sanity-check

App code holds zero model names and zero vendor imports — grep proves it. Sanity-check four knobs before you trust it: is the version pinned (not latest)? does every policy have a fallback? is there a hard spend cap? and does the canary use a sticky key? Miss any one and a silent-drift, single-provider, runaway-cost, or incoherent-conversation bug is waiting.

Say it back in one breath

“The app was married to one vendor's SDK in forty files, so an outage took us down and a swap meant forty edits. I put a gateway in front: every call site becomes chat(policy='smart-analysis'), and a config file maps the policy to a pinned model, a fallback chain, a hard spend cap, and a canary. Now model choice is a config decision, an outage is a fallback, and a rollout is a dial — the app code never changes.”

The gateway is the single most valuable piece of production plumbing precisely because it turns four different incidents — vendor outage, price shock, silent model drift, risky rollout — into four ordinary config edits.

That is the whole discipline of Phase 7 in miniature. Notice what the seam bought: the reliability of Section 03, the safe rollout of Section 04, and the versioning of Section 05 are all just fields in one config file, enforced in one place, observable through one hook. Everything upstream in this course — your prompts, your RAG, your agents, your evals — runs through this seam in production. Build it early, keep app code free of model names and vendor imports, and the operational surprises that sink other teams become, for you, a pull request against a YAML file.
