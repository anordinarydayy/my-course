Structured Output & Function Calling

Prose is for humans; your code needs data it can parse. Structured output makes a model emit JSON that matches a schema you defined — guaranteed — and function calling lets it ask your code to run something. Together they turn a chatty model into a dependable software component, and they're the bridge from "prompting" to building real systems.

Original study notes. A practical guide to making model output reliable. It draws on public provider docs (OpenAI Structured Outputs & function calling, Anthropic structured outputs & tool use, Google Gemini response schemas) but every explanation, diagram, and example here is our own. 00 The problem

Output you can trust

The moment you wire a model into real software, you hit a wall: your code has to parse what the model said. So you write the prompt "respond in JSON," call json.loads(), and it works… most of the time. Then production happens. One in a hundred responses comes back as "Sure! Here's the JSON: ```json {…} ```" — wrapped in prose and a code fence. Another has a trailing comma. Another puts a number where you expected a string, or omits a required field. Each one throws an exception three functions downstream, and a fraction of a million requests is a lot of crashes.

The fix is to stop hoping and start guaranteeing. Structured output makes the model emit JSON that conforms to a schema you define — not as a request, but as a hard constraint enforced during generation. The result is output your code can treat like the return value of a typed function instead of a string to be wrangled. This is the difference between an LLM as a chatbot and an LLM as a reliable component in a pipeline.

The reframe

Stop thinking "I'll ask for JSON and clean it up." Start thinking "I'll define the exact shape and make malformed output impossible." The same shift powers function calling — where the model's job is to emit a valid set of arguments for a tool you defined. Reliable structure is the foundation under both, and under every agent you'll build later.

01 The reliability ladder

Three reliability levels ▸

There are three ways to get JSON out of a model, and they are not equivalent. Same task — "extract the meeting date" — three levels of guarantee. Climb them:

The jump from (b) to (c) is the one that matters, and it's worth understanding how it works — because you met the mechanism in the sampling lesson. Strict structured output uses constrained decoding: your schema is compiled into a grammar, and at every token the engine sets the logits of any schema-violating token to −∞ before sampling. The model literally cannot emit a token that breaks the schema. There's no validation-then-retry, no fallback parser — invalid output simply can't be generated. OpenAI reports 100% schema compliance in their evals; that's not a quality claim, it's a structural guarantee.

What it does and doesn't guarantee

Structured output guarantees the shape, never the truth. The model can still return {"date":"2026-07-01"} when the real date was the 8th — perfectly-formed and wrong. Schema validity removes an entire class of parsing failures; it does nothing for correctness. That's still a job for evaluation (Phase 2). Right shape, possibly wrong value — keep the two apart in your head.

02 The everyday use

Schema → typed object ▸

The most common use of structured output is extraction: messy human text in, a clean typed object out. You define the shape once (in Python with Pydantic, in TypeScript with Zod), and the SDK turns the model's response straight into a typed object — no regex, no parsing, no retry loop. Watch it:

In code, this is shorter than the prompt that describes it. You define a model and let the SDK handle the schema and the parsing:

python · pydantic + a strict parse

class ContactInfo(BaseModel):
    name: str
    email: str
    plan: Literal["Free", "Pro", "Enterprise"]   # an enum → only these 3
    demo_requested: bool

resp = client.messages.parse(                       # or chat.completions.parse(...)
    model=MODEL, messages=[...], output_format=ContactInfo)
contact = resp.parsed_output                          # a typed ContactInfo, guaranteed

Libraries like Instructor (Python) and the Vercel AI SDK with Zod (TypeScript) wrap this across providers, and tools like Outlines bring the same constrained decoding to open-weight models you self-host. The pattern is identical everywhere: schema in, typed object out.

03 Giving the model hands

The tool-call round-trip ▸

Structured output's twin is function calling (also called tool use). Instead of constraining the model's answer, you constrain a set of arguments: you describe a function the model is allowed to call, and it emits a valid call to it. Crucially, the model never runs anything — it produces a request, your code executes it, and you hand the result back. Walk the full round-trip:

Two things make this reliable and important. First, set strict: true on the tool and the arguments are schema-guaranteed — same constrained-decoding mechanism — so you never get a call with a missing parameter or a string where a number belongs. Second, the model can emit several tool calls in one turn (parallel tool use) when the requests are independent. And here's the payoff for the rest of the course: this single round-trip, run repeatedly with planning and memory, is an agent. Everything in the Agents phase is this loop, scaled up.

04 The surfaces

Provider surfaces

Every major provider exposes these features; the names differ but the ideas are identical. The current shapes (mid-2026 — verify against live docs, these churn):

ProviderStrict structured outputTool / function calling
OpenAIresponse_format: {type:"json_schema", strict:true}tools + strict:true on the function
Anthropicoutput_config.format (json_schema)tool use with input schemas + strict
Google GeminiresponseSchema + responseMimeTypefunction declarations

Two gotchas worth knowing before they bite. First, the first request with a new schema is slower — the provider compiles your schema into a grammar (then caches it, so repeats are fast). Second, not all of JSON Schema is supported: strict modes commonly disallow recursion and certain constraints (min/max, regex), and some require every property to be listed as required with optionality expressed as a nullable type. The SDKs often paper over this, but read your provider's limits before designing a deeply nested schema. And watch the non-happy paths: a refusal or a length cutoff means the output may not match the schema — handle those return signals.

05 Using it well

Best practices

The mechanism is easy; using it well is a handful of habits:

  • Make schemas tight. Use enum for fixed value sets (a status, a category), mark fields required, and give explicit types. Every degree of freedom you remove is a way the model can't be wrong about the shape.
  • Reason first, structure last. Forcing the model straight into a rigid object can hurt the quality of what's inside it — there's no room to think. Either give the schema a free-text reasoning field it fills first, or do a reasoning turn and then a structured turn. (This echoes the sampling lesson: don't constrain too early.)
  • Always set strict: true on tools. Guaranteed-valid arguments cost nothing and remove a whole failure mode.
  • Validate at the boundary anyway. The schema guarantees shape, not business rules. Check ranges, allowed values, and invariants where the data enters your system.
  • Design tools least-privilege. A tool the model can call is a capability you've granted it — give it the narrowest one that works (read-only where possible). The defensive-prompting lesson is about exactly why this matters.
Worked round-tripdefine → call → answer
A weather tool, end to end

The user asks "What's the weather in Paris?" Your model has no live data — so you give it a tool, and it asks you to fetch it.

The loop in code

# 1 · define the tool (strict → guaranteed-valid args)
tools = [{"name": "get_weather", "strict": True,
  "input_schema": {"type":"object",
    "properties":{"location":{"type":"string"},
                 "units":{"enum":["celsius","fahrenheit"]}},
    "required":["location","units"]}}]

resp = client.messages.create(model=MODEL, tools=tools, messages=msgs)
# 2 · model emits a tool_use, e.g. {location:"Paris", units:"celsius"} — runs nothing
if resp.stop_reason == "tool_use":
    args = tool_call.input
    result = get_weather(**args)            # 3 · YOUR code runs the real API → {temp:25}
    msgs += [assistant(resp), tool_result(result)]   # 4 · hand the result back
    final = client.messages.create(model=MODEL, tools=tools, messages=msgs)
# final → "It's currently 25°C in Paris."
Say it back in one breath

“I describe a tool with a strict schema; the model decides to call it and emits guaranteed-valid arguments but runs nothing. My code executes the real call and hands the result back, and the model writes the final answer. Structure makes the call safe to dispatch; the round-trip is the atom of every agent.”

Edge cases to handle: a refusal or length cutoff (output may not match the schema); a tool that errors (return the error as the tool_result so the model can recover); and least-privilege on what get_weather — or any tool — is allowed to reach.

Summary & self-test

Reliable structure turns a chatbot into a component: define the exact shape and make malformed output impossible — for the answer (structured output) and for tool arguments (function calling) — then own the execution and the correctness yourself.
