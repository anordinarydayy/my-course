00Concept · Databases & SQL

Why a Database?

Before the SQL syntax and the indexes, the why. A database isn't just "where the data lives" — it's a machine that keeps your data correct while many things change it at once, finds any slice of it without reading all of it, and never loses what it promised to keep. This first lesson builds that mental model: the foundation the whole course, and PostgreSQL, stands on.

00 Orientation

Why not just a pile of files?

Here is the one-sentence definition to anchor everything: a database management system (DBMS) is a program whose entire job is to store data so that it stays correct, stays findable, and stays safe when many users touch it at the same time. That's it. Every feature you will learn — schemas, indexes, transactions, replication — is in service of that job. PostgreSQL is the specific DBMS this course uses, but the job is the same everywhere.

The fastest way to feel why you need one is to imagine building your app without it — storing your data in a folder of JSON or CSV files, or a spreadsheet. It works, right up until it doesn't:

A folder of files

  • No enforced structure — every reader re-guesses the shape.
  • Two writers at once corrupt each other's writes.
  • "Find one record" means reading every file.
  • A crash mid-write leaves a half-written mess.

A spreadsheet

  • Structure and formulas — great at human scale.
  • Really one editor at a time; not app traffic.
  • No rules across sheets (does this id exist?).
  • Filtering still scans the whole sheet.

A DBMS (Postgres)

  • A schema every row must satisfy.
  • Many concurrent readers/writers, held correct.
  • Indexes find rows without scanning.
  • Committed data survives a crash.

Read down that last column and you have the four promises that make something a database rather than a bucket: structure, fast retrieval, safe concurrency, and durability. Hold onto those four words — the rest of this lesson takes them one at a time, and the rest of the course turns each into a phase.

The mental reframe

Stop thinking of a database as a place you put data and start thinking of it as a correctness machine with a query language. You don't reach for it because you need somewhere to keep bytes — you reach for it because you need those four guarantees. If you only needed a bucket, a file would do.

01 The data model

Everything is a table ▸

The first promise — structure — comes from the relational model, the idea (from Edgar Codd in 1970) that you can represent almost any data as relations: plain tables of rows and columns. A relation is a table; each tuple is a row; each attribute is a named, typed column. That's the whole vocabulary, and its power is how little it needs.

Two ideas do the heavy lifting. A primary key is a column (or set of columns) whose value uniquely identifies a row — no two rows share it. A foreign key is a column that stores another table's key value, to express a relationship. Step through how two tables get linked:

The subtle, important part is the very last step. To answer "Ana's orders," the database doesn't follow a pointer in memory — it matches values: every orders row whose user_id equals Ana's id. That matching is a join, and it's the engine of the whole relational model. Because the link is a value, you can follow it in either direction (a user's orders, or an order's user), combine tables that were never designed together, and let the database check the link — refusing an order whose user_id points at no real user.

Links are values, not pointers

orders.user_id just holds a number that happens to match a users.id. Nothing is wired together in memory. That one design choice is why relational databases are so flexible: any table can be joined to any other on matching values, at query time, in any direction — and why referential integrity (the database enforcing that the referenced row exists) is even possible.

02 The query language

Say what you want, not how to get it

Here is the second thing that makes a database special, and it surprises people coming from application code: SQL is declarative. You describe the result you want; you do not write the steps to produce it. Compare the two mindsets on a real question — "how many orders does each user have?"

In file-and-loop code you'd write the how: open the users file, loop over it, for each user open the orders file, loop again counting matches, accumulate into a map, sort. In SQL you write the what:

sql

SELECT u.name, count(*) AS orders
FROM users u
JOIN orders o ON o.user_id = u.id
GROUP BY u.name;

Nowhere does that say how to do the join — nested loops? sort both sides and merge? build a hash table? That decision belongs to the database, not to you. And that is the whole point: because you only stated the what, the engine is free to pick the how, and to change its mind as your data grows. The same four lines run on a thousand rows or a hundred million — the database picks a fast strategy for each, and tuning becomes mostly a matter of giving it better options (an index, fresh statistics) rather than rewriting your logic.

This is also why you have to retrain one instinct. The hardest habit to drop, coming from imperative code, is thinking row by row. SQL wants you to think in sets: "the set of users joined to their orders, grouped by name." Reach for a loop — in app code pulling rows one at a time, or a database cursor — and you turn a single fast set operation into thousands of slow round-trips. Almost every "why is this slow?" story starts with row-by-row thinking.

The shift that pays off all course long

Whenever you're about to loop over rows in your application and do one more query per row, stop — that's the N+1 pattern, the single most common database performance bug. The set-based version (a join, or one query with WHERE id = ANY(...)) asks the database once and lets it do what it's good at. We'll name this trap again in Phase 4 and Phase 8 — it never stops mattering.

03 Under the hood

What actually happens when you run a query ▸

That tidy declarative SELECT hides a small pipeline, and seeing it once demystifies most of what comes later — especially performance. When you send a query, PostgreSQL doesn't just "look something up." It parses, it plans, it executes, and it reads data from memory or disk. Follow a single query down through the engine:

One box on that trip deserves a star: the planner. Because you only said what you wanted, Postgres is free to choose how — and it does so by estimating the cost of competing plans using statistics about your tables (how many rows, how many distinct values). The gap between a query that returns in 10 milliseconds and the very same query taking 10 seconds is, almost always, which plan the planner picked — usually whether it could use an index or had to read the whole table. That's not a side topic; it's the heart of the course's biggest phase.

Where your time will actually go

You will spend more interview minutes and more on-call minutes on that one box — the planner — than on any other. "Why is this query slow?" almost always decodes to "why did the planner choose this plan, and how do I give it a better option?" Learning to read its decisions with EXPLAIN is the entire point of Phase 4. For now, just remember the planner exists and that it chooses.

04 The guarantees

The four guarantees: ACID ▸

We've covered structure, retrieval, and querying. The promise that's left — the one that makes a database trustworthy when many users hit it at once and when the power can fail — comes from the transaction: a group of statements that the database treats as a single, all-or-nothing unit, between BEGIN and COMMIT. Transactions give you four guarantees, abbreviated ACID:

  • Atomicity — all of the statements happen, or none do. A failure halfway rolls the whole thing back.
  • Consistency — every transaction moves the database from one valid state to another; your constraints (keys, checks) always hold.
  • Isolation — concurrent transactions don't see each other's half-finished work; each behaves as if it ran alone.
  • Durability — once it says "committed," the data survives a crash, a kill, or a power cut.

The cleanest way to feel why you need these is to watch what breaks without them. Two people withdraw from the same account at the same instant; with no transaction isolating them, you get a lost update. Step through it:

Map that back to the letters. The race you just watched is an Isolation failure: B was allowed to read a value that A was busy changing. Atomicity is a different protection — it's why a money transfer never debits one account without crediting the other; if the second step fails, the first is undone. And Durability is why, once the screen says "payment confirmed," it stays confirmed even if the server loses power one second later. Together they turn a chaotic interleaving of operations into something you can reason about as if it happened one clean step at a time.

Atomicity vs Isolation — don't mix them up

The two most-confused letters. Atomicity is about failure: the all-or-nothing of a single transaction. Isolation is about concurrency: whether other transactions can see your in-progress work. The lost update above is an isolation problem; a half-applied transfer is an atomicity problem. Phase 5 is devoted to isolation because "how much isolation" turns out to be a real, costly dial — not a free switch.

05 The route ahead

Two workloads, and the road from here

One last distinction shapes nearly every design choice you'll make: what the database is for. Two broad shapes of work pull in opposite directions, and naming yours tells you how to model, index, and scale.

OLTP — transactional

  • Many small, fast reads and writes, each touching a few rows.
  • The user-facing app: place an order, load a profile, post a comment.
  • Optimized for low latency, high concurrency, correctness.
  • What Postgres — and this course — is built around by default.

OLAP — analytical

  • A few big queries scanning millions of rows to aggregate.
  • Dashboards, reports, "revenue by region by month."
  • Optimized for scan throughput, not per-query latency.
  • Often a different tool's job (columnar stores, warehouses).

Postgres handles both adequately, and specialized systems handle one extremely well — but the point for now is just to notice which you're building, because the right index for a profile lookup is the wrong one for a monthly report. With that in mind, here's the road this course walks, in a deliberately dependency-aware order:

  • Phase 0 — Foundations. The model, the engine, and ACID — the vocabulary everything else assumes. You're here.
  • Phase 1 — Schema & modeling. Turning a domain into tables that stay correct: types, keys, and normalization.
  • Phases 2–3 — SQL. Getting data back out, from joins and grouping up to CTEs and window functions.
  • Phase 4 — Indexing & performance. The planner, EXPLAIN, and making slow queries fast — the heart of the course.
  • Phase 5 — Transactions & MVCC. Concurrency done right: isolation, locking, and how Postgres pulls it off.
  • Phases 6–8 — Power features, operations & scale, and applied design. JSONB and partitioning, durability and replication, and a final design-and-interview workshop.

Two principles will serve you the whole way. The first: make it correct first, then make it fast. A sound schema with the right constraints prevents whole classes of bug that no amount of later tuning can fix — bad data is far more expensive than a slow query. The second: measure, don't guess. When something is slow, read the query plan before you change anything, exactly the way you'd profile code before optimizing it. Guessing at database performance is how you spend a day adding an index the planner then ignores.

The posture to carry forward

You already have what this subject leans on: you think in systems and in algorithms. A database is just a system with unusually strong guarantees and an explicit cost model. Treat each new feature — an index type, an isolation level, a JSONB column — the way you'd treat a new data structure: learn what problem it solves, what it costs, and when not to use it. That instinct, more than memorized syntax, is what the interview and the job are really testing.

★ Review

Summary & self-test

The whole lesson compresses to one idea: a database is a correctness machine — it keeps data structured, finds it fast, holds it safe under concurrency, and never loses what it committed — and you drive it by describing what you want, in tables, and letting it decide how. Everything after this is detail on one of those promises.
