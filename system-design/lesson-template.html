05Lesson · System Design Interview

Design a Rate Limiter

A rate limiter decides how many requests a client may make in a given window — and rejects the rest. It's a small component with outsized importance, and the interview lives almost entirely in one place: which counting algorithm you choose, and why.

About these notes: original study material on rate limiting — a standard, widely-documented engineering topic. The explanations, examples, and diagrams here are my own; nothing is copied from any paid course or book. Three diagrams are interactive — press Play (or step with Next) to watch the flow.
00 The problem

Why limit requests?

Left unprotected, a service will happily try to serve every request it receives — including from a client that's broken, abusive, or actively hostile. A single caller hammering an endpoint can exhaust CPU, memory, database connections, and downstream quotas, degrading the experience for everyone and sometimes taking the whole system down. A rate limiter caps the request rate per client so the system stays healthy under pressure.

Concretely, you rate-limit to:

  • Prevent resource starvation from denial-of-service attacks, whether deliberate or accidental (a runaway retry loop).
  • Control cost — especially when each call hits a metered third-party API you pay for.
  • Prevent cascading failure — bounding load on one service keeps an overload from rippling through the others behind it.
  • Enforce fairness so one heavy user can't crowd out everyone else.
  • Blunt abuse like credential-stuffing or brute-force login attempts.
01 Scope it

Pin down the requirements

Before drawing anything, an interviewer wants to see you narrow the problem. Rate limiting has many flavors, and the right design depends on the answers. Ask:

  • What do we limit on? Per IP, per user ID, per API key, per endpoint — or a combination. This is the key the limiter counts against.
  • Client-side or server-side? Client-side checks can be bypassed, so the authoritative limiter must be server-side.
  • What scale? Thousands of requests per second changes the storage and latency story.
  • Distributed? If multiple server instances handle traffic, the count has to be shared across them.
  • How accurate must it be? Exact, or is a small approximation acceptable in exchange for less memory?
  • Do we tell the client? Usually yes — return a clear signal so well-behaved clients back off.

That typically settles on a handful of non-functional goals: low added latency (the limiter sits in the hot path of every request), low memory, distributed counting, clear feedback to throttled clients, and fault tolerance — if the limiter's datastore has a hiccup, it shouldn't bring the whole system down.

Interview tip

Stating these requirements out loud earns easy points and frames everything that follows. The phrase to land is: "the limiter is on the critical path, so latency and fault tolerance matter as much as correctness."

02 Placement

Where the limiter lives

There are three places the logic can sit. Client-side is the weakest — clients can be modified or spoofed, so you can't trust it to enforce anything; at best it's a courtesy to reduce obviously wasteful calls. Server-side puts the check inside the API server itself. The most common production choice is a separate rate-limiter middleware — often inside an API gateway, the component that already handles concerns like authentication, SSL termination, and IP allow-listing. Throttling there stops excess traffic before it ever reaches your services.

How to decide: if you already run a gateway that supports rate limiting, use it. If you need fine-grained control or your stack doesn't offer it, build the middleware yourself. Either way, the limiter intercepts each request, decides allow-or-reject, and only forwards the survivors.

03 Algorithm · bursts allowed

Token bucket

Now the core of the interview: how do you actually count? The token bucket is the most widely used answer. Picture a bucket that holds up to N tokens (its capacity). Tokens are added back at a fixed refill rate — say 5 per second. Every incoming request must take one token: if a token is available it's removed and the request is allowed; if the bucket is empty the request is rejected. You generally keep one bucket per key (per user, per IP, per endpoint).

The clever part is what an empty-vs-full bucket means for traffic shape. When traffic is quiet, tokens accumulate up to the cap, so a client can fire a short burst — up to the bucket size — all at once. Over the long run, though, throughput can't exceed the refill rate. That makes it a great fit for traffic that's spiky but bounded.

Strengths

  • Simple and very memory-efficient (two numbers per key)
  • Allows controlled bursts up to the bucket size
  • Battle-tested — common in real cloud and payment APIs

Watch-outs

  • Two parameters (size + refill) that need careful tuning
  • A poorly chosen bucket size permits large bursts you didn't intend
04 Algorithm · steady outflow

Leaking bucket

The leaking bucket is token bucket's mirror image: instead of allowing bursts, it enforces a steady outflow. Model it as a FIFO queue of fixed size. Incoming requests join the queue if there's room; if the queue is full, they overflow and are dropped. Requests are then pulled off and processed at a constant rate — like water leaking from a bucket through a fixed hole.

The effect is that no matter how bursty the input, the output to your downstream services is smooth and predictable. That's valuable when a stable processing rate matters more than letting clients spike.

Strengths

  • Memory-efficient with a bounded queue
  • Produces a smooth, constant outflow to downstream systems

Watch-outs

  • No burst allowance — spikes just fill the queue
  • A burst can fill the queue with stale requests, delaying fresh ones
  • Two parameters (queue size + leak rate) to tune
05 Algorithm · simplest counting

Fixed window counter

The window family takes a different approach: count requests over slices of time. The fixed window counter chops the timeline into fixed intervals — say one minute — and keeps a counter for each. Every request increments the counter for the interval it falls in; if the counter passes the threshold, the request is dropped. At each boundary the counter resets to zero.

It's wonderfully simple and cheap. But it has one notorious flaw — the boundary burst. Because the count resets sharply at the edge, a client can send a full quota at the very end of one window and another full quota at the very start of the next, squeezing up to twice the limit into a short span straddling the boundary.

06 Algorithm · perfectly accurate

Sliding window log

To kill the boundary burst, stop thinking in fixed slices and track a continuously moving window. The sliding window log stores a timestamp for every request (a sorted set is the natural structure). When a new request arrives, you first drop every timestamp older than now − window, then count what remains. If the count is below the limit, you accept the request and record its timestamp; otherwise you reject it.

Because the window slides smoothly with the clock, the limit holds correctly across any rolling interval — there's no edge to exploit. The price is memory: you keep a timestamp per request, and you still record and evaluate timestamps even for requests you ultimately reject. At high volume, that adds up.

07 Algorithm · the practical default

Sliding window counter

The sliding window counter is the pragmatic hybrid most people reach for. It keeps cheap per-window counters like the fixed window, but smooths the boundary by weighting the previous window according to how much of it still falls inside the rolling window. The estimate is roughly:

count ≈ requests_current + requests_previous × overlap

Say the limit is 7/min, the previous minute saw 5 requests, the current minute has 3 so far, and we're 30% into the current minute — so 70% of the previous window still overlaps the rolling window. The estimate is 3 + 5 × 0.7 = 6.5, which floors to 6. Since 6 < 7, the request is allowed. The boundary spike is gone, and you've stored only two counters instead of a timestamp per request.

Why it works

It assumes requests in the previous window were spread evenly, which isn't strictly true — so it's an approximation. In practice the error is small, and the memory savings over the log are large. For most systems this is the right default; reach for the log only when you need exactness.

08 Scaling out

Going distributed

Everything so far assumed one counter in one place. Real traffic is spread across many server instances behind a load balancer, and an in-memory counter on each instance breaks the global limit immediately — a client routed across ten instances effectively gets ten times the quota. The fix is a centralized datastore all limiter instances share, almost always Redis: it's in-memory (fast), supports per-key expiry (TTL), and offers atomic operations. The limiter nodes themselves stay stateless, so no sticky sessions are needed.

Two hard problems show up the moment you go distributed, and interviewers love both:

  • Race conditions. The naive read-then-increment-then-write is unsafe under concurrency: two requests read the same count, both decide they're under the limit, and both write back — one increment is lost and the limit is breached. Fix it with atomic operations: Redis INCR, or a small Lua script that checks and increments in a single atomic step on the server.
  • Synchronization & latency. A central store is one extra network hop on every request and a potential bottleneck or single point of failure. Mitigate with Redis replication/clustering, and keep each decision to a single round trip.
09 Putting it together

Architecture & the 429

Assemble the pieces. The rate-limiter middleware sits in front of the API servers (or inside the gateway). For each request it builds a key — for example user_id + endpoint — checks and updates the counter in Redis with your chosen algorithm, and decides. Survivors are forwarded to the service. The rest are rejected with HTTP 429 Too Many Requests, accompanied by headers that let well-behaved clients self-throttle:

  • X-RateLimit-Limit — the ceiling allowed in the window.
  • X-RateLimit-Remaining — how many calls are left right now.
  • Retry-After — how many seconds to wait before trying again.

Instead of dropping outright, some systems enqueue excess requests for later processing — useful when delaying is acceptable but losing the request is not.

Deep-dive the interviewer may push for

Fail open vs fail closed: if Redis is unreachable, do you allow requests (fail open, protecting availability) or block them (fail closed, protecting the backend)? Limiters usually fail open so the limiter never becomes the outage. Also be ready to discuss limiting at multiple levels (per-IP and per-user and global) and monitoring — a flood of 429s often means your limit is set too strictly, not that you're under attack.

★ Review

Summary & self-test

The whole topic compresses to one sentence: pick a counting algorithm that matches your tolerance for bursts, memory, and accuracy — then make it work across many servers without races. If you can sketch the five algorithms and name the flaw each one fixes, you can carry the interview.
