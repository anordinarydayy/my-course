Design Search Autocomplete (Typeahead)

Typeahead returns the handful of best completions for whatever prefix the user has typed so far, in well under 100ms, on every single keystroke. The whole design hinges on one move: don't search for the answer at request time — precompute it. Store the top few completions directly at each node of a prefix tree, so answering a query becomes a short walk and a copy, never a scan.

About these notes: original study material on designing a search autocomplete / typeahead service — a standard, widely-documented system-design topic. 00 The problem

What typeahead must do

Open the Shopee app and tap the search box. Type i and a list appears — iphone, iphone case, induction cooker. Add p and the list refines instantly to iphone, iphone case, ipad. You never pressed Enter. That instant, per-keystroke list of completions is search autocomplete, also called typeahead or search-as-you-type.

It feels trivial — it's just suggesting words — but it is one of the most demanding read paths in any consumer product, for three reasons that fight each other:

  • It fires on every keystroke. A user typing iphone generates six queries, not one. Search volume is multiplied by the length of what people type.
  • It has a brutal latency budget. If a suggestion arrives after the next keypress, it is useless — the list flickers and feels broken. The target is sub-100ms end to end, including the network round trip.
  • The answer is a ranking problem, not a lookup. For the prefix ip there are thousands of valid completions; the user wants the few most likely ones, ranked by how popular they are.

Put those together and you get the central tension of this design: you must answer a ranking query, over a huge candidate set, faster than a human can press the next key, billions of times a day. You cannot afford to compute the ranking at request time. The entire lesson is about how to make the answer already exist before the user asks.

01 Scope it

Clarify the requirements

Start the interview by narrowing the problem out loud. Typeahead has many possible shapes, and the design forks depending on the answers.

Functional requirements

  • Return the top k completions for a prefix. Given ip, return the best 5–10 queries that start with ip. We'll take k = 5 as the default.
  • Rank by popularity. The ordering is driven by how often each completion has been searched historically — more searches, higher rank. Not alphabetical.
  • Match on prefix only. We'll scope to prefix matching (ip → iphone), not middle-of-string or fuzzy/typo matching. I'll flag those as extensions, not in the core.
  • Suggestions are reasonably fresh. If a product suddenly trends — say a flash sale on airpods — it should surface within minutes to hours, not the next day. "Real-time to the second" is explicitly not required.

Non-functional requirements

  • Latency. The hard one. p99 under 100ms end to end. The server-side lookup itself should be a small number of milliseconds.
  • Scale. Very high QPS because every keystroke is a request — we'll size this in the next section.
  • Availability. Highly available, but autocomplete is a non-critical feature: if it fails, the user can still type the full query and press Enter. So we can degrade gracefully and serve slightly stale data rather than go down.
  • Cost-efficiency. At this volume, computing rankings per request would be ruinous. The architecture must be cheap to serve.

The single most important clarifying answer is the freshness one. Because we only need "fresh within minutes-to-hours," we are allowed to precompute. That permission is what unlocks the entire design. If the interviewer insisted on per-second freshness with personalized results, this becomes a very different, much harder system.

Interview tip

Land this sentence early: "Because suggestions only need to be fresh within minutes, not seconds, I can precompute the rankings offline and make the read path a pure lookup." That one trade-off — freshness for speed — is the thesis of the whole answer, and stating it up front shows you understand why the design looks the way it does.

02 Estimation

Back-of-the-envelope estimation

Numbers anchor the design. They tell us whether everything fits in memory (it does) and how hard the read path must be (very). Let's size a Shopee-scale search bar.

Query QPS — the keystroke amplifier

Suppose the app has 10 million daily active users, and each does 10 searches per day. That's 100M search sessions per day. But the request count isn't 100M — it's multiplied by keystrokes. If a typical query is ~5 characters and we fire a request per character (we'll debounce later to cut this), that's:

# search requests per day
100M sessions × 5 keystrokes  = 500M typeahead requests / day

# average QPS  (86,400 s in a day)
500,000,000 / 86,400  ≈  ~5,800 QPS

# peak is spikier than average — assume ~3× for lunch/evening rush
peak  ≈  ~17,000 QPS

That keystroke amplifier is the headline. A "modest" 100M searches a day becomes hundreds of millions of requests, and the system must answer each in milliseconds. This is precisely why precomputation, not on-the-fly ranking, is non-negotiable.

How much data is in the trie?

The candidate set is the universe of distinct historical queries. Suppose we track 20 million distinct query strings worth keeping (we'll prune the long tail). Each averages ~20 bytes. The raw strings are ~400MB — small. But the structure that makes lookups fast (the trie, with a precomputed top-k list cached at each node) is bigger. A rough estimate:

# nodes ≈ total characters across distinct queries
20M queries × ~20 chars  ≈  400M characters
# trie de-dupes shared prefixes, so far fewer nodes — say ~100M nodes

# each node caches top-5 completions: ~5 × 20 bytes + pointers/overhead
per-node top-k cache  ≈  ~200 bytes
100M nodes × 200 bytes  ≈  ~20 GB

Twenty gigabytes fits comfortably in the RAM of a single beefy server, and easily across a small sharded cluster with replicas. The whole serving structure lives in memory. That is the second pillar: prefix lookups never touch disk.

Write / aggregation volume

On the write side we're ingesting the stream of completed searches — the 100M sessions/day, ~1,200 writes/sec average. That's a trivial firehose for a log/stream system; the work is in aggregating and ranking it, which we do offline (Step 05), not on the hot path.

Why these numbers matter

Two conclusions fall straight out of the estimate and steer the rest of the design:
(1) read QPS is enormous and latency-critical → the read path must be a precomputed lookup with no computation;
(2) the dataset is ~tens of GB → it fits in RAM, so we shard an in-memory structure rather than hitting a database per keystroke.

03 The contract

API contract & data model

Before architecture, pin down the interface. Typeahead is read-dominated, so the public surface is essentially one endpoint, plus an internal write path that feeds the offline build.

The query endpoint

One GET, designed to be cacheable (idempotent, no side effects, everything in the URL so a CDN can key on it):

GET /v1/suggest?q=ip&limit=5&lang=en&market=sg

// 200 OK — returns in < 100ms
{
  "prefix": "ip",
  "suggestions": [
    { "text": "iphone",      "score": 98213 },
    { "text": "iphone case",  "score": 71044 },
    { "text": "ipad",         "score": 56120 },
    { "text": "ip camera",    "score": 18377 },
    { "text": "ipad mini",    "score": 12090 }
  ]
}

A few deliberate choices in that contract:

  • q is the raw prefix; we normalize it server-side (lowercase, trim, collapse whitespace) so IP and ip  hit the same trie node and the same cache key.
  • limit caps k; we never return more than we precomputed (default 5).
  • lang and market exist because a Singapore super-app serves multiple languages and locales — iphone trends differently in SG vs ID. These become part of the partition key (one trie per (market, lang)), which we'll use again for sharding.
  • The response carries a score only so the client can optionally show "trending" badges; it's not required for ordering, which is already baked in.

The write path (data collection)

The companion endpoint isn't called per keystroke — it's the analytics event fired when a user actually commits a search (presses Enter or taps a suggestion). That signal is what makes a completion "popular":

POST /v1/search-events     // fire-and-forget, async
{
  "query":   "iphone case",
  "market":  "sg",
  "lang":    "en",
  "ts":      1717142400,
  "user_id": "u_88421"          // for abuse/dedup, not stored long-term
}

Data model — what's stored where

There are two very different stores, and conflating them is a classic mistake. Keep them separate:

  • The raw event log — an append-only stream of every committed search. This is the source of truth for popularity. It lives in a log/warehouse, not in the serving path.
  • The aggregated frequency table — the offline pipeline rolls the log up into (query, market, lang) → frequency counts over a recent window:
-- aggregated counts, rebuilt periodically by the pipeline
query_freq(
  query     VARCHAR,
  market    VARCHAR,
  lang      VARCHAR,
  freq      BIGINT,        -- weighted by recency (Step 08)
  updated_at TIMESTAMP,
  PRIMARY KEY (market, lang, query)
)
  • The serving trie — the in-memory structure the query endpoint reads. It is derived from the frequency table, not stored as rows. Each node caches a precomputed top-k list. We design it next.
Follow-up the interviewer will ask

"Why is the read endpoint a GET with everything in the query string?" Because that makes it trivially cacheable at every layer — browser, CDN, and an in-process cache — keyed on the full URL. The most common prefixes (a, i, p) can be answered without the request ever reaching your service. Idempotent GETs are free leverage; don't waste them on a POST.

04 The core idea

The trie & precomputed top-k

Now the heart of the interview. How do we answer "top-5 completions of ip" in milliseconds? The data structure is a trie (pronounced "try," from retrieval), also called a prefix tree.

What a trie is

A trie is a tree where each edge is labeled with a character, and the path from the root down to a node spells out a prefix. The root is the empty string; its children are a, b, c, …; the node you reach by walking i then p represents the prefix ip. Every stored query is a path from the root to some node marked as a complete word. Crucially, shared prefixes are stored once — iphone, iphone case, and ipad all share the i→p spine, branching only where they diverge.

This already gives a fast way to find the prefix node: walking to ip is just 2 hops, O(length of prefix), independent of how many millions of queries are stored. But finding the node is only half the job.

The naive lookup, and why it's too slow

The obvious algorithm: walk to the prefix node, then traverse the entire subtree beneath it to collect every completion, then sort them by frequency and take the top 5. For the prefix i, that subtree could hold hundreds of thousands of queries. Doing a subtree scan plus a sort on every keystroke at 17,000 QPS is exactly the request-time computation our estimate forbids. It would blow the latency budget and melt the CPU.

The key optimization: cache top-k AT each node

So we precompute. At every node in the trie, we store the precomputed top-k completions of the prefix that node represents, already ranked by frequency. The ip node literally holds the list [iphone, iphone case, ipad, ip camera, ipad mini]. The i node holds the top 5 for everything starting with i. The root holds the 5 most popular queries overall.

With that in place, serving a request becomes embarrassingly cheap:

  • Walk from the root following the prefix characters — O(length of prefix), typically 2–10 hops.
  • At the prefix node, return the cached list verbatim. No subtree scan, no sort, no computation. Just a pointer dereference and a copy of ~5 short strings.

That's it. A lookup is a short walk plus a memcpy. This trades space (every node now carries a small list) and build time (we compute those lists offline) for the thing we cannot compromise on: read latency. Step through it:

Compute top-k on the fly

  • Nodes stay tiny — no per-node list to store
  • Always reflects the freshest underlying data
  • But: subtree scan + sort per request → far too slow
  • Cost scales with subtree size; short prefixes are worst

Precompute & cache at nodes

  • Lookup is O(prefix length) + a copy — milliseconds
  • Cost is independent of how popular the prefix is
  • But: more memory (a list per node)
  • But: lists go stale until the next offline rebuild
The follow-up that separates strong answers

"Why precompute the top-k at each node instead of computing it per request?" Because the read path runs at tens of thousands of QPS under a sub-100ms budget, and a subtree scan-plus-sort is unbounded work that grows with prefix popularity. Precomputing converts an expensive ranking query into a constant-time cache read; the cost moves off the hot path and into a periodic offline job, which is exactly where we can afford it.

05 The write path

Building the trie offline

The cached lists don't appear by magic — something has to compute them. That something is an offline data pipeline that runs on a schedule, completely off the hot read path. Its job: turn the raw stream of past searches into a fresh trie, then ship that trie to the serving fleet.

Stage by stage

  • Collect. Every committed search (the POST /v1/search-events from Step 03) lands in a durable log — in practice Apache Kafka. Kafka is the buffer: it absorbs the firehose and lets downstream jobs read at their own pace.
  • Aggregate & rank. A batch or stream job reads the log and counts how often each (query, market, lang) appears over a recent window (say the last 7 days), producing the query_freq table. This is a giant group-by-and-count — the natural fit for Spark or Flink. Recency weighting (Step 08) is applied here.
  • Build the trie. From the ranked frequency table, construct the trie: insert each query along its character path, and at every node compute the top-k of its subtree. A clean way to do this is bottom-up — each node's top-k is merged from its children's top-k lists plus its own terminal frequency, so you never re-scan subtrees.
  • Ship it. Serialize the finished trie to a blob in object storage (S3), then signal the serving nodes to download and atomically swap it into memory. Old trie stays live until the new one is fully loaded, so there's no serving gap.

Because all the heavy lifting — counting, ranking, top-k computation — happens here, the serving side does zero ranking work. The pipeline is where "expensive but infrequent" lives; the read path is where "cheap but constant" lives.

Full rebuild vs incremental merge

Two ways to refresh: tear down and rebuild the whole trie from scratch each cycle, or incrementally merge only the deltas (counts that changed) into the live trie. Full rebuilds are simpler and self-correcting — any drift gets wiped each cycle — but they reprocess everything and are heavier. Incremental updates are cheaper and fresher but trickier: you must mutate live nodes' top-k lists correctly and atomically, which is fiddly under concurrent reads. Most teams start with periodic full rebuilds for simplicity and only move to incremental when the freshness requirement tightens.

Under the hood — atomic swap

How do you replace a 20GB in-memory structure on a live server without a stall? Build the new trie fully in memory beside the old one, then flip a single pointer (the "current trie" reference) to the new root. In-flight reads finish against the old trie; new reads see the new one. The old structure is then garbage-collected. The swap is one atomic pointer write — no lock on the read path, no downtime.

06 Scaling reads

Serving at scale: sharding the trie

One server holding the whole 20GB trie works until traffic or data outgrows it. We scale the read path two ways at once: replicate for QPS and availability, and shard for size and to localize hot prefixes.

Replication first

The trie is read-only between rebuilds, which makes replication trivial: put N identical copies behind a load balancer, and any replica can answer any query. More replicas = more QPS headroom and no single point of failure. Because the data is immutable until the next swap, there's no replication-lag or write-conflict problem — every replica is byte-identical.

Sharding by prefix

When the trie is too big for one box, or when you want to spread load, shard by prefix: split the keyspace so each shard owns a slice of the alphabet. A simple scheme routes by the first character (or first two) — shard A holds prefixes starting a–f, shard B g–m, and so on. A thin router in front inspects the prefix and forwards the request to the owning shard.

The hot-shard problem, and how to fix it

Naive first-character sharding has an ugly flaw: letter frequencies are wildly uneven. Far more queries start with s or a than with x or z, so the s shard becomes a hotspot while the z shard idles. This is the same load-skew problem you see in any naive partitioning.

The fix is to shard by historical load, not by alphabet position. Use the frequency data you already computed offline to estimate the request volume each prefix range will receive, then carve the ranges so every shard gets roughly equal traffic — the busy s prefix might get a shard to itself, while w, x, y, z share one. Because the offline pipeline already knows the distribution, it can recompute these shard boundaries on each rebuild and keep them balanced as trends shift.

Shard by first letter

  • Dead simple to route — just look at char 0
  • No lookup table needed
  • But: badly skewed — "s" shard melts, "z" shard idle
  • But: can't adapt as query trends shift

Shard by historical load

  • Even traffic across shards — no hotspots
  • Recomputed each rebuild from real frequency data
  • But: router needs a prefix→shard mapping table
  • But: rebalancing moves data when boundaries shift
Follow-up the interviewer will ask

"How do you shard the trie, and what goes wrong with the obvious approach?" The obvious approach — split by first character — produces severe load skew because letter frequencies are uneven. Fix it by partitioning the prefix space by measured historical traffic so each shard carries roughly equal QPS, and recompute the boundaries during each offline rebuild so they track changing trends.

07 Cutting latency further

Caching & the CDN edge

Even an in-memory lookup costs a network round trip from the user's device to your data center. For a user in Singapore hitting a server in the same region that's fine, but you can do better — and you can shed a huge fraction of load — with caching at two layers.

The Zipfian shape of prefixes

Prefix popularity is brutally skewed (a Zipf distribution): a tiny number of short prefixes account for an enormous share of requests. The single-character prefixes — a, i, p, s — are hit constantly, because every search passes through its first letter. This skew is a gift: cache the few thousand hottest prefixes and you serve most traffic without ever touching the trie.

In-process and edge caches

  • In-process cache. Each serving node keeps a small LRU map of prefix → top-k list for the hottest prefixes. A hit avoids even the trie walk. Since the underlying data only changes on rebuild, cache invalidation is easy: bump a version/generation number on swap so old entries fall out.
  • CDN / edge cache. Because GET /v1/suggest?q=… is a cacheable idempotent URL, a CDN (CloudFront, Cloudflare, Akamai) can cache the response for the most popular prefixes right at an edge POP near the user. A Singapore user's request for q=i can be answered from a Singapore edge node in single-digit milliseconds, never reaching your origin. Set a short TTL (say 60s) so freshness stays within budget.

Order of preference, fastest to slowest: browser cache → CDN edge → node in-process cache → trie lookup. Each layer absorbs the hottest slice and protects the one behind it.

Client-side: debounce the keystrokes

The cheapest optimization of all is to not send some requests. The client debounces — it waits ~50–100ms after the last keystroke before firing, so a fast typer entering iphone sends maybe 2–3 requests instead of 6. It can also cache results locally and, on a known dead-end prefix that returned nothing, stop querying as the user types more. These client tricks cut the keystroke amplifier we sized in Step 02 directly at the source.

08 Trade-off

Freshness vs rebuild cost

We bought our speed by precomputing, which means suggestions are only as fresh as the last rebuild. How fresh is fresh enough, and what does more freshness cost? This is the central trade-off dial of the whole system.

How recency shapes the ranking

Raw all-time frequency would keep iphone near the top forever and bury a brand-new trending term. So the aggregation step weights recent searches more heavily — for example, a time-decayed count or a sliding window (last 7 days, with the last 24 hours weighted up). During a Shopee 6.6 flash sale, a product that was obscure yesterday can rocket into the top suggestions because its recent-weighted frequency spikes. The decay constant is a tuning knob: faster decay tracks trends but is jumpier; slower decay is stable but stale.

The rebuild-frequency dial

How often you rebuild sets your freshness floor — and your cost ceiling:

Frequent / streaming rebuilds

  • Suggestions track trends within minutes
  • Great for flash sales, breaking events
  • But: heavy, continuous pipeline cost
  • But: incremental merges are complex and bug-prone

Periodic (e.g. hourly/daily) rebuilds

  • Simple, cheap, self-correcting full rebuilds
  • Plenty fresh for most queries
  • But: a sudden trend lags until the next cycle
  • But: a bad rebuild affects everything at once

A common middle path: do a cheap full rebuild on a schedule (say hourly) for the bulk of the catalog, and run a lightweight fast lane that detects sharply trending prefixes from the live stream and merges just those into the live trie within minutes. You get stability for the long tail and freshness where it actually matters.

Follow-up the interviewer will ask

"How do you keep suggestions fresh?" Three layers: weight recent searches more in the offline ranking so trends naturally rise; rebuild and re-ship the trie on a schedule that meets the freshness SLA; and optionally run a real-time fast lane that detects spiking prefixes and incrementally merges them between full rebuilds. Then name the cost: more freshness means a heavier, more complex pipeline, so you tune the rebuild cadence to the business need, not to the technical maximum.

09 Quality & safety

Filtering, personalization, sampling

A raw popularity ranking is naive in three ways that an interviewer will probe. Each is a layer bolted onto the offline pipeline, not the hot path.

Filtering — don't suggest the wrong things

What's popular isn't always what you want to suggest. You need filters:

  • Profanity / safety. Strip offensive, hateful, or unsafe terms against a blocklist during the build, so they never enter the trie. Doing it at build time (not request time) keeps the read path clean and fast.
  • Spam / abuse. A bot can hammer a query to inflate its frequency. Defend with per-user de-duplication and rate caps in the aggregation step, so one actor can't manufacture a trending suggestion.
  • Policy / legal. Some terms must be suppressed for regulatory reasons (region-specific). The blocklist is part of the build config and can differ per market.

Personalization — optional, and expensive

So far every user with the prefix ip gets the same list. Personalization would tailor it to the individual — their past searches, location, language. The catch: personalization breaks the thing that makes typeahead cheap, namely the shared precomputed trie. If every user needs a different ranking, you can't cache one global answer per prefix.

The pragmatic compromise is segment-level personalization: build a small number of tries per coarse segment (per market, per language, maybe per broad category interest) rather than per user. You keep most of the precomputation benefit — a few dozen tries, not millions — while capturing the biggest personalization wins. Full per-user re-ranking, if needed, happens as a light post-step on the small top-k list the trie already returned, never as a fresh ranking query.

Sampling — tame the volume

You don't need to log every search to rank accurately. Popularity is a statistical estimate, and a sample of the stream (say 10%) gives a ranking nearly identical to the full data at a fraction of the storage and processing cost. For the head terms that dominate suggestions, even aggressive sampling preserves the order. Sampling is how you keep the aggregation pipeline affordable at billions of events.

Why filtering belongs in the build, not the request

It's tempting to filter at query time ("skip blocked words before returning"). Don't. At 17,000 QPS, every per-request check is latency and CPU you can't spare, and it would mean the blocked term still sits in the trie taking up a top-k slot. Filtering during the offline build means the served lists are already clean — the read path does nothing but copy. Push work off the hot path whenever the freshness budget allows it; that's the recurring theme of this entire design.

10 Resilience & growth

Failure modes & what breaks at 10×

Wrap up by stress-testing the design: what happens when a piece dies, and what cracks first when traffic grows tenfold?

Failure modes

  • A serving node dies. No problem — the trie is read-only and replicated, so the load balancer simply routes around the dead replica to its peers. Because each replica holds an identical copy, there's nothing to recover; a fresh node just downloads the latest trie blob from S3 and joins the pool.
  • A shard is unavailable. Prefixes owned by that shard temporarily can't be served. Mitigate by replicating each shard (so one node loss doesn't drop the shard) and by failing soft — return an empty suggestion list rather than an error, since autocomplete is non-critical and the user can still type the full query.
  • The pipeline breaks or ships a bad trie. The serving fleet keeps running on the last good trie — stale suggestions are vastly better than none. Guard the build with validation (sanity-check size, top-terms, no empty root) before the atomic swap, and keep the previous blob so you can roll back instantly. This is why decoupling build from serve matters: a broken build degrades freshness, not availability.
  • The raw event log is the recovery anchor. If the aggregated table or a trie is ever corrupted, you can rebuild from the query log — it's the durable source of truth, replayable through the pipeline. The serving structures are all derived and disposable.
  • A trending-term storm / hot prefix. The CDN and in-process caches absorb the spike for the few hot prefixes; the load-based sharding keeps any single shard from being overwhelmed.

What breaks at 10×

Push DAU and catalog up 10×: ~170,000 peak QPS and a ~200GB trie. The pressure points, and the fixes:

  • The trie no longer fits one box (~200GB). This is now mandatory sharding, not optional — split into many load-balanced shards, each holding a slice that fits comfortably in RAM with room for the rebuild's double-buffer.
  • Read QPS dwarfs origin capacity. Lean harder on the CDN and client debouncing; the Zipfian skew means a well-tuned edge cache can still serve the majority of traffic, so origin QPS grows far slower than total QPS.
  • Rebuild time balloons. A full rebuild over 10× the data may exceed your freshness window. Parallelize the build (per-shard, per-segment builds run independently) and lean more on the incremental fast lane for trending terms.
  • Aggregation cost explodes. Sample more aggressively and pre-aggregate closer to the edge so the central pipeline ingests counts, not raw events.
The closing follow-up

"What's the single point of failure, and how do you survive it?" There isn't one on the read path — the trie is replicated and read-only, and the CDN fronts it. The pipeline is a soft dependency: if it stops, suggestions go stale but keep serving from the last good trie. The true source of truth is the durable query log, from which every derived structure can be rebuilt. The design fails open and stale, never closed and down — exactly right for a non-critical, latency-sensitive feature.

★ Review

Summary & self-test

The whole topic compresses to one sentence: trade a little freshness for enormous speed by precomputing the top-k completions at every trie node offline, then make the read path a pure in-memory lookup — replicated, sharded by load, and fronted by caches. If you can draw the trie with lists at its nodes and the offline pipeline that fills them, and explain why each piece exists, you can carry this interview.
