18Lesson · System Design Interview

Design a Rate Limiter

A rate limiter caps how many requests a client may make inside a time window and rejects the rest. It is a tiny component with outsized importance — and the interview lives almost entirely in one place: which counting algorithm you pick, and the reasons you can defend it.

About these notes: original study material on rate limiting, a standard and widely-documented engineering topic. The prose, numbers, examples, and diagrams here are my own; nothing is copied from any paid course or book. Three diagrams are interactive — press Play (or step with Next) to watch the flow. 00 The problem

Why limit requests?

Left unguarded, a service will dutifully try to answer every request that reaches it — including requests from a client that is broken, accidentally abusive, or actively hostile. A single caller hammering one endpoint can drain CPU, saturate memory, exhaust the database connection pool, and burn through a paid third-party quota, degrading the experience for everyone and sometimes toppling the whole system. A rate limiter caps the request rate per client so the service stays healthy under pressure.

Make this concrete with a Singapore super-app. Imagine GrabPay's OTP-send endpoint. A buggy mobile build ships a retry loop that fires the "send OTP" call every 200 ms when the network flaps. Ten thousand phones on a flaky 4G cell now generate 50,000 SMS sends per second. Without a limiter you pay the SMS gateway for every one, the SMS provider throttles you anyway, and legitimate users can no longer log in. A limiter that says "one OTP send per phone number per 60 seconds" turns a six-figure incident into a non-event.

Concretely, you rate-limit to:

  • Prevent resource starvation from denial-of-service attacks, whether deliberate or the accidental kind (a runaway retry storm).
  • Control cost — especially when each call hits a metered downstream you pay for (SMS, an LLM API, a maps geocoder, a KYC vendor).
  • Prevent cascading failure — bounding load on one service keeps an overload from rippling into the others behind it.
  • Enforce fairness so one heavy merchant integration cannot crowd out every other tenant.
  • Blunt abuse like credential stuffing, brute-force login, scraping, and coupon-farming bots during a Shopee flash sale.
The interview frame

This is a "design a system" drill, so treat it like a real mock interview: clarify scope, estimate, sketch the API and data model, draw the architecture, then go deep on the bottleneck. For a rate limiter the headline bottleneck is always the counting algorithm and how it behaves across many servers — budget most of your time there.

01 Clarify + scope

Clarify the requirements

Before drawing anything, narrow the problem. Rate limiting has many flavors and the right design depends on the answers. Open with the question that decides everything: what is the key we count against?

Functional requirements

  • Limit per key. The key is whatever identity the counter is scoped to: per user_id, per source IP, per api_key, per endpoint, or a composite like user_id:endpoint. Most production rules are composite — "100 search calls per minute per user per endpoint" — because a global per-user number is too blunt.
  • Authoritative server-side. The enforcing decision must live on infrastructure we control. A client-side check is a courtesy, never a guarantee.
  • Distributed. Traffic is spread across many instances behind a load balancer; the count has to be global, not per instance.
  • Clear client feedback. When we reject, tell the caller why and when to retry, so well-behaved clients self-throttle instead of hammering harder.
  • Configurable rules. Limits change per route and per plan tier (a free Shopee Open API key gets less than a partner key). Rules should be data, not a redeploy.

Non-functional requirements

  • Low added latency. The limiter is on the hot path of every request. A few milliseconds of overhead per call is a tax on the entire platform, so the check should add well under ~5 ms at p99.
  • Low memory. At millions of distinct keys, per-key bookkeeping must be tiny.
  • High availability & fault tolerance. If the limiter's datastore hiccups, the limiter must not become the outage. This forces an explicit fail-open vs fail-closed decision (we return to it in §11).
  • Accuracy tolerance. Ask whether the limit must be exact or whether a small approximation is acceptable in exchange for far less memory. The answer steers the algorithm choice.
Interview tip

Saying these out loud earns easy points and frames the whole answer. The sentence to land is: "the limiter sits on the critical path, so latency and fault tolerance matter as much as correctness — and the headline trade-off is the counting algorithm's accuracy versus its memory and burst behavior."

02 Estimation

Back-of-envelope estimate

A quick estimate proves the design is feasible and sizes the datastore. Assume a mid-size super-app surface: peak 1,000,000 requests/sec across all endpoints at a regional rush hour (Friday 6 pm Grab demand spike). Every request crosses the limiter, so the limiter must sustain 1M decisions/sec.

Storage per key

For the algorithm we will end up recommending (a sliding-window counter), each key needs roughly two integer counters plus a small key string — call it ~50 bytes, generously ~100 bytes with overhead. Suppose 20 million distinct active keys in any window (users × a handful of throttled endpoints). Then:

# active keys in memory
20,000,000 keys × 100 bytes  =  2 GB

# a single modern Redis node holds tens of GB — comfortable.
# the sliding-window LOG instead stores one timestamp (~16B) per request:
# a noisy key doing 1000 req/min = 1000 entries ≈ 16 KB for ONE key.

That contrast is the whole memory argument in one snippet: counters are flat per key; the log grows with traffic.

Throughput & latency budget

A single Redis instance handles on the order of 100k operations/sec comfortably (often more for simple ops). To absorb 1M decisions/sec we need roughly 10–20 shards of Redis, partitioned by key, plus replicas for failover. Each decision is one round trip; on a same-AZ network that is sub-millisecond, leaving headroom inside the ~5 ms p99 budget. The limiter compute nodes are stateless and scale horizontally behind the load balancer — so the datastore, not the compute, is the thing to size carefully.

Why estimate at all

The estimate flips a vague "use Redis" into a defensible plan: ~2 GB and ~10–20 shards at 1M QPS, single-round-trip decisions. It also exposes the failure surface you will be asked about — those shards are the single point of failure that motivates the fail-open discussion later.

03 Contract

API contract & data model

Going past the book: an interviewer is impressed when you can sketch the actual interfaces. A rate limiter has two surfaces — the control plane (how operators define rules) and the data plane (the per-request decision).

Rule definition (control plane)

Rules are configuration, stored in a small relational table or a config service and pushed to the limiter nodes. A rule binds a key pattern to a limit and a window.

-- rate_limit_rules
rule_id        BIGINT PK
key_pattern    TEXT      -- e.g. "user:{uid}:POST:/v1/payments"
scope          TEXT      -- user | ip | api_key | endpoint | composite
algorithm      TEXT      -- sliding_window_counter | token_bucket | ...
limit_count    INT       -- e.g. 100
window_secs    INT       -- e.g. 60
burst          INT       -- bucket capacity, for token bucket only
action         TEXT      -- reject | enqueue | shadow (log only)
priority       INT       -- which rule wins when several match

The decision call (data plane)

Conceptually the limiter exposes one internal operation, checkAndConsume(key, cost), returning allow/deny plus the metadata the client needs. The cost lets you charge an expensive endpoint more than one "token" per call.

// request
POST /internal/ratelimit/check
{ "key": "user:8842:POST:/v1/payments", "cost": 1 }

// allowed
200 OK
{ "allowed": true,  "limit": 100, "remaining": 73, "reset_secs": 18 }

// rejected
200 OK
{ "allowed": false, "limit": 100, "remaining": 0,  "retry_after": 18 }

Note the internal check itself returns 200 with an allowed flag — it is the outer request that becomes a 429 when allowed is false. We wire the public-facing response in §11.

Counter storage (state)

The hot per-key state lives in Redis, not in SQL. A typical layout for the sliding-window counter keeps two integer fields per rolling window plus a TTL so abandoned keys evict themselves automatically:

# Redis keyspace (one hash per limiter key)
HSET rl:user:8842:payments  cur 73  prev 41  win_start 1717142400
EXPIRE rl:user:8842:payments 120   # 2× window, self-cleaning

The EXPIRE is doing quiet but important work: without per-key TTL, a key per user per endpoint would accumulate forever and the 2 GB estimate would creep toward infinity. TTL bounds memory to active keys only.

04 Placement

Where the limiter lives

There are three places the logic can sit, and the choice is partly a trust question and partly an operational one.

Client-side — untrusted

Client-side rate limiting runs inside the mobile app or browser. It is the weakest option because clients can be patched, decompiled, or replayed with a script — you cannot trust them to enforce anything. Its only legitimate use is courtesy: a debounced button or a local cooldown that stops obviously wasteful calls and saves the user's battery and data. Treat it as a UX nicety, never as a security control.

Server-side, in the application

Server-side puts the check inside the API service itself, as a middleware in the request pipeline. This is authoritative and gives you the richest context (you already know the authenticated user), but it couples limiting logic into every service and spends the service's own CPU on traffic you are about to reject.

Rate-limiter middleware in an API gateway — the common choice

The most common production answer is a dedicated rate-limiter middleware, usually living inside an API gateway — the component that already terminates TLS, authenticates, and routes. Throttling there stops excess traffic before it touches your services, so a flood costs you a cheap gateway rejection instead of a full request through the stack. Real tools that do this: NGINX (limit_req with the leaky-bucket-style burst and nodelay knobs), Envoy (local plus a global ratelimit gRPC service), HAProxy (stick tables), Kong and AWS API Gateway (usage plans + throttling). Grab-style platforms typically run Envoy at the edge with a global limit service backed by Redis.

Gateway middleware

  • Rejects flood before it hits services — protects CPU
  • One place to configure and observe limits
  • Off-the-shelf in NGINX / Envoy / Kong
  • Less app context (may not know business-level identity)

In-application

  • Full business context (tenant, plan tier, entitlement)
  • Per-feature precision without gateway plumbing
  • Spends your service CPU on doomed requests
  • Logic duplicated across every service
05 Algorithm · bursts allowed

Token bucket

Now the core of the interview: how do you actually count? The token bucket is the most widely used answer. Picture a bucket that holds up to N tokens — its capacity, also called the burst size. Tokens are added back at a fixed refill rate — say 5 per second. Every incoming request must take one token: if a token is available it is removed and the request passes; if the bucket is empty the request is rejected. You keep one bucket per key.

How it works under the hood

The elegant trick is that you do not run a background timer that drips tokens in. That would mean millions of timers. Instead each bucket is just two numbers in Redis — tokens and last_refill_ts — and refill is computed lazily at read time. When a request arrives:

  • Compute elapsed time since last_refill_ts.
  • Add elapsed × refill_rate tokens, capped at capacity N (so idle time never overflows the bucket).
  • If tokens ≥ cost, subtract the cost and allow; else reject.
  • Write back the new tokens and last_refill_ts.

Two numbers per key, no timers — that is why it is so memory-efficient.

Worked example — Grab driver location pings

A Grab driver app pushes a GPS ping. The rule: bucket capacity 10, refill 2 tokens/sec per driver. A driver who was offline for a while has a full bucket of 10, so when the app reconnects it can flush 10 backlogged pings in one burst — good, that catches the map up instantly. After that, sustained throughput settles to 2 pings/sec because that is the refill rate. The long-run rate is bounded by refill; the bucket size bounds the burst. That is exactly the shape you want for spiky-but-bounded traffic.

Tools & failure mode

Token bucket is everywhere: AWS API Gateway usage plans, Stripe-style payment APIs, and NGINX's limit_req with burst= behave bucket-like. The failure mode to name is clock skew: because refill is "tokens accrued since last_refill_ts," if you compute elapsed using the app server's wall clock instead of a single authoritative clock, two limiter nodes with drifting clocks will grant slightly different token counts. Fix it by deriving time from the Redis server (TIME command) so there is one clock, and by doing the read-refill-write as one atomic step (§10).

Strengths

  • Tiny state — two numbers per key
  • Allows controlled bursts up to the bucket size
  • Battle-tested in real cloud and payment APIs

Watch-outs

  • Two parameters (capacity + refill) that need tuning
  • A generous bucket permits bursts you may not have intended
  • Needs one authoritative clock to avoid skew
06 Algorithm · steady outflow

Leaking bucket

The leaking bucket is token bucket's mirror image: instead of permitting bursts, it enforces a steady outflow. Model it as a FIFO queue of fixed size. Incoming requests join the queue if there is room; if the queue is full they overflow and are dropped. A worker then pulls requests off and processes them at a constant rate — like water leaking from a bucket through a fixed-size hole.

How it works under the hood

The queue is the state. Two parameters define it: queue size (how much burst it can absorb) and leak rate (the constant processing rate). No matter how bursty the input, the output is perfectly smooth at the leak rate, because the leak is decoupled from the arrivals. NGINX's limit_req without nodelay is essentially this: excess requests are delayed in a queue and released at the configured rate rather than rejected outright.

Worked example — Shopee payment settlement

Suppose Shopee's settlement service can safely write to a fragile downstream ledger at exactly 500 writes/sec and no faster. A flash-sale checkout spike produces 5,000 settlement events in a burst. A leaking bucket with leak rate 500/sec smooths that into a steady 500/sec stream the ledger can survive; events beyond the queue's capacity overflow and are retried later. Here a stable processing rate matters more than letting the producer spike — the opposite priority from the Grab ping example.

Strengths

  • Memory-bounded by the queue
  • Produces a smooth, constant outflow downstream
  • Natural fit when you can delay rather than reject

Watch-outs

  • No burst allowance — spikes just fill the queue
  • A burst can fill the queue with stale requests, delaying fresh ones
  • Two parameters (queue size + leak rate) to tune
07 Algorithm · simplest counting

Fixed window counter

The window family takes a different tack: count requests over slices of time. The fixed window counter chops the timeline into fixed intervals — say one minute — and keeps one counter per interval. Each request increments the counter for the interval it falls in; if the counter passes the threshold, the request is rejected. At each boundary the counter resets to zero.

How it works under the hood

It is gorgeously cheap: a single counter per key plus the current window id. In Redis it is literally INCR rl:{key}:{window} followed by EXPIRE — one atomic increment, no read-modify-write, no race (we will see in §10 why that matters). The window id is just floor(now / window_secs).

The boundary-burst flaw

It has one notorious defect — the boundary burst. Because the count resets sharply at the edge, a client can send a full quota at the very end of one window and another full quota at the very start of the next, squeezing up to twice the limit into a short span straddling the boundary. With a limit of 5/min, a caller fires 5 at 0:59 and 5 more at 1:00 — that is 10 requests in about a second, double the intended rate, and every counter individually looks legal.

When it is still fine

For coarse, non-adversarial limits — "send at most 5 marketing pushes per user per hour" — the boundary burst is harmless and the simplicity is a gift. The flaw only bites when a precise short-window ceiling matters or an attacker is deliberately timing the edge. That nuance is exactly what the next two algorithms fix.

08 Algorithm · perfectly accurate

Sliding window log

To kill the boundary burst, stop thinking in fixed slices and track a continuously moving window. The sliding window log stores a timestamp for every request — a sorted set is the natural structure. When a new request arrives you first drop every timestamp older than now − window, then count what remains. If the count is below the limit you accept and record the new timestamp; otherwise you reject.

How it works under the hood

Redis makes this almost a one-liner with a sorted set whose score is the timestamp:

# prune expired, count, conditionally add — inside one Lua script
ZREMRANGEBYSCORE rl:{key}  0  (now - window)
ZCARD            rl:{key}                # how many remain in window
# if ZCARD < limit:
ZADD             rl:{key}  now  <uniq>   # record this request
EXPIRE           rl:{key}  window

Because the window slides smoothly with the clock, the limit holds correctly across any rolling interval — there is no edge to exploit, so it is exact.

The cost

The price is memory and write amplification. You keep a timestamp per request, and — subtly — you still record and prune timestamps even for some requests you ultimately reject (a common variant logs the attempt regardless). For a key doing 1,000 requests/min that is ~1,000 entries living in memory for that one key, versus two integers for a counter. At Grab/Shopee scale across millions of keys that difference is the gap between a 2 GB datastore and a 200 GB one.

When the interviewer pushes "where would you actually use the log?"

Reach for it only where exactness is worth the memory: a security-sensitive, low-volume limit such as "max 5 failed login attempts per account per 15 min" on SeaMoney, or a hard contractual API quota you must never exceed. For everyday throttling at scale, the next algorithm wins.

09 Algorithm · the practical default

Sliding window counter

The sliding window counter is the pragmatic hybrid most teams reach for. It keeps cheap per-window counters like the fixed window, but smooths the boundary by weighting the previous window by how much of it still overlaps the rolling window. The estimate is:

count ≈ requests_current + requests_previous × overlap

Worked example

Say the limit is 7/min, the previous minute saw 5 requests, the current minute has 3 so far, and we are 30% into the current minute — so 70% of the previous window still overlaps the rolling window. The estimate is 3 + 5 × 0.7 = 6.5, which floors to 6. Since 6 < 7, the request is allowed. The boundary spike is gone, and you have stored only two counters instead of a timestamp per request.

Why it works (and where it is wrong)

It assumes the previous window's requests were spread evenly, which is not strictly true — so it is an approximation. If the previous minute's traffic was all clustered at its end, the estimate slightly understates the true rolling count and may let a sliver through; if clustered at its start, it slightly overstates and may reject a hair early. In practice the error is a fraction of a percent and the memory savings over the log are enormous. This is the right default; reach for the log only when you need provable exactness.

vs Fixed window

  • Removes the 2× boundary burst
  • Same tiny storage (two counters)
  • One extra multiply per check — negligible

vs Sliding log

  • Vastly less memory (counters, not timestamps)
  • Approximate, not exact
  • Good enough for ~99% of throttling needs
10 Scaling out

Going distributed

Everything so far assumed one counter in one place. Real traffic is spread across many limiter instances behind a load balancer, and an in-memory counter on each instance breaks the global limit immediately — a client routed across ten instances effectively gets ten times the quota. The fix is a centralized datastore all limiter instances share, almost always Redis: it is in-memory (fast), supports per-key expiry (TTL), and offers atomic operations. The limiter nodes themselves stay stateless, so no sticky sessions are needed and any node can serve any request.

The race condition in read-then-increment

The first hard problem is concurrency. The naive sequence — read count, check it, write count+1 — is unsafe when two requests for the same key land at once:

# Request A and Request B, same key, limit = 100, current = 100? no, = 99
A: GET count   -> 99
B: GET count   -> 99      # B read before A wrote
A: 99 < 100 ✓  SET 100
B: 99 < 100 ✓  SET 100     # lost update — should be 101, limit breached

Both requests saw 99, both decided they were under the limit, both wrote 100 — one increment vanished and the limit silently leaks. This is a classic lost-update race.

The fix: atomic operations

Make the read-check-write happen as a single indivisible step on the server:

  • Redis INCR is atomic by itself, which is why the fixed-window counter is race-free for free: INCR returns the new value, and you compare that to the limit. No read-then-write gap exists.
  • A Lua script handles the algorithms that need a read, a branch, and a write together (token bucket's refill math, the sliding-window weighting). Redis runs a Lua script atomically — nothing else executes on that key mid-script — so the whole decision is one safe unit. This is the standard production technique.
  • Single round trip. The Lua script also collapses three would-be round trips into one, which matters for the latency budget.
-- sliding-window-counter, atomic on the Redis server
local cur  = redis.call('GET', KEYS[1]) or 0
if (estimate(cur, prev, overlap) + 1) > limit then
  return 0                 -- reject, no write
end
redis.call('INCR', KEYS[1])
redis.call('EXPIRE', KEYS[1], window)
return 1                   -- allow

The cost of centralizing: an extra hop and a SPOF

A central store is one extra network round trip on every request and a potential bottleneck or single point of failure. Two mitigations:

  • Shard and replicate. Partition keys across a Redis Cluster so no single node carries 1M QPS, and run replicas so a primary failure promotes a replica rather than losing the counters.
  • Local pre-check / batching. Some high-scale designs keep an approximate local counter and only sync to the central store periodically (Envoy's local + global model), trading a little accuracy for far fewer central calls. You accept that the global count can briefly overshoot.
The follow-up you will get

"How do you make the counter safe under concurrency?" — answer in one breath: "never read-then-write; use Redis INCR for the simple counter, and a Lua script for token-bucket or sliding-window so the whole decision is atomic and one round trip."

11 Putting it together

Request lifecycle & the 429

Assemble the pieces. The rate-limiter middleware sits in front of the API servers (or inside the gateway). For each request it builds a key — for example user_id:endpoint — runs the atomic check against Redis with your chosen algorithm, and decides. Survivors are forwarded to the service. The rest are rejected with HTTP 429 Too Many Requests, carrying headers that let well-behaved clients self-throttle:

  • X-RateLimit-Limit — the ceiling allowed in the window.
  • X-RateLimit-Remaining — how many calls are left right now.
  • Retry-After — seconds to wait before trying again (also valid as an HTTP date).
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
Retry-After: 18
Content-Type: application/json

{ "error": "rate_limited",
  "message": "Too many requests. Retry in 18s." }

Instead of dropping outright, some systems enqueue excess requests for later processing (the leaking-bucket action) — useful when delaying is acceptable but losing the request is not, such as a payment webhook you must eventually process.

Fail open vs fail closed

The failure mode you will absolutely be asked about: what happens when Redis is down? The limiter cannot get a count, so it must choose a default.

Fail open

  • On store error, allow the request
  • Protects availability — the limiter never causes an outage
  • Risk: a real flood gets through unthrottled while Redis is down
  • Usual default for general API throttling

Fail closed

  • On store error, reject the request
  • Protects the backend / honors a hard quota
  • Risk: the limiter's own outage takes down all traffic
  • Right for security-critical or strict-quota limits

Most general-purpose limiters fail open so a Redis blip never becomes a platform-wide outage — but a payment or login limit may justify failing closed. A good hybrid is to fail open and fall back to a coarse local in-memory limit, so you are degraded but not defenceless.

Deep-dive the interviewer may push for

Be ready to discuss layering — per-IP and per-user and global limits at once (catch both a single abuser and a botnet). And monitoring: a sudden flood of 429s usually means your limit is set too strictly for legitimate traffic, not that you are under attack — so emit metrics on rejection rate per rule and alert on it.

12 Scale & failure

Scaling & what breaks at 10x

The design holds at 1M QPS. Push it to 10M QPS and three things break in order — naming them unprompted is what separates a strong answer.

1. The single Redis becomes the bottleneck

One node cannot serve 10M ops/sec. Move to a sharded Redis Cluster, partitioning by limiter key so each shard owns a disjoint key range — and because each key's counter lives entirely on one shard, an atomic single-key operation stays atomic (no cross-shard transaction needed). Add replicas per shard; on a primary failure the replica is promoted. The one window of weakness: counters written to the primary but not yet replicated are lost on failover, so the limit briefly under-counts. For a rate limiter that is acceptable — you would rather lose a little accounting than block traffic.

2. The hot key

A single celebrity key — a viral Shopee live-stream seller, or a global limit shared by all anonymous traffic — concentrates millions of ops on one shard while the others idle. Sharding by key does not help because it is one key. Fixes: split the global counter into N sub-counters (global:shard0..shardN), increment a random one, and sum them periodically — trading a little real-time precision for spread; or push that specific limit down to per-node local counters with periodic reconciliation.

3. The cross-region problem

At 10x you are multi-region (Singapore, Jakarta, Bangkok). A strict global limit would require every region to consult one central Redis — adding 100+ ms of cross-region latency to the hot path, which is unacceptable. The practical answer is per-region limits with the global ceiling split across regions, or eventual reconciliation between regional stores. You explicitly give up a perfectly global instantaneous count to keep latency local. State this trade-off out loud: "global accuracy or low latency — at multi-region scale you cannot have both, so I shard the limit per region."

Other failure modes worth a sentence

  • Limiter node dies: harmless — nodes are stateless, the load balancer routes around it, and the count lives in Redis.
  • Clock skew: derive time from one authoritative source (the Redis TIME command) so windows and bucket refills agree across nodes.
  • Misconfigured rule: ship a shadow / dry-run mode that logs what it would reject without rejecting, so you can validate a new limit against real traffic before enforcing it.
The one-line scaling story

Shard Redis by key for throughput, split or localize hot keys, and shard the limit itself per region once a single central store would add cross-region latency. Each step trades a little global accuracy for the latency and availability that actually matter on the hot path.

★ Review

Summary & self-test

The whole topic compresses to one sentence: pick a counting algorithm that matches your tolerance for bursts, memory, and accuracy — then make it work across many servers without races, and decide what happens when the store is down. If you can sketch the five algorithms, name the flaw each one fixes, and defend the distributed and fail-open choices, you can carry the interview.
