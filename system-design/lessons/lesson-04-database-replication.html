Database Replication

A single database is two problems wearing one trench coat: it is the one machine whose death takes the whole product down, and it is the hard ceiling on how many reads you can serve. Replication copies your data onto more machines so reads scale and the system survives a lost node — but it hands you back lag, stale reads, and the genuinely hard question of what to do the instant the leader stops answering.

About these notes: original study material on database replication — a standard, widely-documented engineering topic. The explanations, examples, numbers, and diagrams here are my own; nothing is copied from any paid course or book. 00 The problem

One database is a cliff edge

Start with the simplest possible setup: one application server talking to one database. It works beautifully until two forces collide with it. The first is a single point of failure (SPOF) — a component whose failure brings down the entire system because nothing else can take over its job. If that one database loses power, corrupts a disk, or its host gets reaped by a cloud provider, every read and every write stops cold. Your GrabPay wallet balances, your Shopee order history, your Agoda bookings — all unreachable until someone restores from backup, which can take minutes to hours.

The second force is the read ceiling. A single machine has a fixed number of CPU cores, a fixed amount of RAM for the buffer pool, and a fixed disk throughput. Past some queries-per-second it saturates, and latency climbs from single-digit milliseconds into the hundreds. You can buy a bigger box — vertical scaling — but that runs out fast and gives you exactly zero redundancy: a bigger SPOF is still a SPOF.

Here is the lever that makes the fix obvious: most real workloads are read-heavy. A user browsing the Shopee catalogue, opening a chat thread, refreshing a feed, or checking a wallet balance issues many reads for every write. A read:write ratio of 9:1 is common; for content and catalogue systems it is often 50:1 or higher. So if you could spread reads across several machines, you would relieve the bottleneck for the bulk of your traffic and gain redundancy at the same time. That is exactly what replication does: keep multiple copies of the same data on different nodes.

Why it matters

Replication is the cheapest first win in scaling a stateful system. It targets the read side and availability at once. It does not by itself help write throughput — every copy still has to absorb every write — which is the bottleneck that eventually forces sharding (covered in Lesson 10). Keep that boundary crisp: replication scales reads and survivability; sharding scales writes and storage.

01 The core topology

Leader and followers

The standard answer is the leader–follower topology, also called primary–replica (older docs say master–slave). The rule is short and you should be able to recite it: one node is designated the leader, and all writes go to the leader. Every other node is a follower (replica) that holds a copy of the same data. Reads can be served by any node — usually you offload them to the followers so the leader is freed up for writes.

The leader keeps the followers in step by continuously sending them a stream of every change it makes — its replication log (we unpack exactly what flows in the next step). A follower receives that stream and applies the same changes in the same order to its own copy, so it converges on the leader's state.

Why this shape and not something else

Putting writes through one node is a deliberate simplification: it gives you a single, unambiguous order of writes. There is exactly one place that decides "first this update happened, then that one," so there are no write-write conflicts to resolve. Followers just replay that order. That single-writer discipline is what makes leader–follower the easiest replication model to reason about, and it is why it is the default in PostgreSQL, MySQL, SQL Server, MongoDB, and managed services like Amazon RDS and Aurora.

Adding a follower buys you two things at once. It is one more node that can answer reads, so your read capacity grows roughly linearly with replica count. And it is a hot copy of the data, so if the leader dies a follower can be promoted to take its place (Step 06). Walk the flow below — note especially the dashed failover branch at the end.

Interview tip

Open with the rule in one breath: "single leader for writes, fan reads out to followers, leader ships its change log to keep them in sync." Then immediately name the two payoffs — read scaling and redundancy — and the one thing it does not fix — write throughput. That framing tells the interviewer you know exactly what problem this tool is and isn't for.

02 The mechanism

How the change log streams

"The leader ships its changes" is the line that gets you partial credit. To get full credit, you should be able to say what actually travels over the wire, because the choice has real consequences. Under the hood, a leader writes every committed change into an append-only log on disk, and followers pull from that log and replay it. There are three common things that log can contain.

Statement-based replication

The simplest idea: ship the literal SQL statements (UPDATE wallets SET balance = balance - 50 WHERE user_id = 7) and have each follower re-run them. It is compact, but it is dangerous because non-deterministic functions diverge: NOW(), RAND(), or an auto-increment that depends on timing will produce different results on each replica. MySQL historically defaulted to this and accumulated a long list of caveats. Mostly abandoned for general use.

Write-ahead log (WAL) shipping

Ship the low-level, physical log the storage engine already writes for crash recovery — the write-ahead log, the byte-level record of which pages and bytes changed on disk. PostgreSQL's streaming replication works this way: the follower replays the exact same physical changes. It is precise and cheap to produce, but it is tightly coupled to the storage format, so the leader and follower generally must run the same engine and often the same major version, which complicates zero-downtime upgrades.

Logical (row-based) replication

The pragmatic middle ground: ship a logical description of the row change — "in table wallets, the row with primary key 7 changed column balance from 200 to 150." This is decoupled from the physical storage format, so it survives version differences and can even feed a different system entirely. MySQL's row-based binlog and PostgreSQL's logical replication use this. It is also the foundation of change data capture (CDC): tools like Debezium tail the binlog/WAL and publish each row change onto Kafka, so a SeaMoney transactions table can fan out to a search index, an analytics warehouse, and a fraud-detection stream — all from the same replication log.

// A logical replication event, conceptually
{
  "op": "UPDATE",
  "table": "wallets",
  "key":   { "user_id": 7 },
  "before": { "balance": 200 },
  "after":  { "balance": 150 },
  "lsn": "0/16B3748"   // log sequence number = position in the stream
}

Two details matter for later sections. First, every entry has a monotonic position — PostgreSQL calls it the log sequence number (LSN), MySQL the binlog coordinates, MongoDB the oplog timestamp. A follower's position in this stream is exactly how you measure how far behind it is. Second, replication can be a pull model (the follower asks the leader for everything after position X) which is robust because a follower that disconnects can reconnect and catch up from where it left off, as long as the leader still retains that segment of the log.

Under the hood — catch-up after a disconnect

Say Follower 2 in a Grab cluster reboots for a kernel patch and is offline for 90 seconds. On restart it tells the leader its last applied LSN and streams everything since. This works only while the leader still has those log segments; if the follower was down long enough that the log was recycled (PostgreSQL wal_keep_size / replication slots, MySQL binlog_expire_logs_seconds), the follower can't catch up incrementally and must be re-seeded from a fresh base snapshot. Sizing log retention is therefore an availability decision, not a housekeeping one.

03 The durability dial

Synchronous vs asynchronous

The instant you have a leader streaming to followers, a timing question appears: when a client sends a write and the leader commits it, does the leader wait for the followers to confirm they received it before telling the client "done"? The answer is a dial, not a switch, and it trades durability against latency.

Synchronous replication

In synchronous replication, the leader does not acknowledge the write to the client until at least one designated follower confirms it has the change. The payoff is strong durability: the moment the client sees success, the data exists on two machines, so even if the leader's disk evaporates a microsecond later, the write is not lost. The cost is latency and fragility — every write now pays a network round trip to the follower, and if that follower is slow, GC-pausing, or unreachable, the write stalls. A naive fully-synchronous setup means one sick follower can freeze all writes for the whole system.

Asynchronous replication

In asynchronous replication, the leader writes locally, acknowledges the client immediately, and ships the change to followers in the background. Writes are fast and a slow follower never blocks the leader — you can add many followers cheaply. The catch is a durability hole: if the leader crashes after acknowledging a write but before any follower received it, that write is lost. This is the model behind most read-scaling deployments precisely because it keeps write latency low.

Semi-synchronous — the middle ground

Semi-synchronous replication is the setting most production systems actually use. You make one follower synchronous and the rest asynchronous. Now every acknowledged write is guaranteed on at least two nodes (no data loss if the leader alone dies), but you only ever wait on the single fastest sync follower, and the async followers scale reads without dragging on write latency. PostgreSQL exposes this directly via synchronous_commit and synchronous_standby_names; MySQL has a semi-sync plugin with a configurable timeout that automatically falls back to async if the sync follower goes quiet for too long.

Synchronous

  • + No data loss — acked writes survive a leader crash
  • + Followers are guaranteed current at ack time
  • − Every write pays a follower round trip → higher latency
  • − A slow or dead sync follower can stall all writes

Asynchronous

  • + Low write latency — leader acks immediately
  • + A slow follower never blocks writes; scales to many replicas
  • − Recent acked writes can be lost if the leader dies first
  • − Followers lag, so reads can be stale (Step 04)

For a SeaMoney funds transfer you lean synchronous (or semi-sync) — losing a confirmed payment is unacceptable. For a Shopee product-view counter or an Agoda "recently viewed" list you happily run async — a lost increment is invisible and speed wins. The interview move is to tie the dial to the business cost of a lost write, not to pick one globally.

Follow-up the interviewer will ask

"With one synchronous follower, what happens if that follower itself dies?" Most systems automatically demote the requirement to async (or promote another follower into the sync slot) so writes keep flowing — otherwise the sync follower becomes its own SPOF. The honest answer names that trade: while running degraded, you have briefly given up the no-data-loss guarantee in exchange for staying available.

04 The cost of async

Replication lag and stale reads

Asynchronous replication's free lunch comes with a bill: replication lag — the delay between a write committing on the leader and that same write being applied on a follower. Under healthy conditions lag is a few milliseconds. Under load — a follower replaying a heavy batch, a network blip, or a long-running query holding things up — it can spike to seconds or, in pathological cases, minutes.

Lag produces stale reads: a read served by a follower that hasn't yet caught up returns an older version of the data. Because the system is correct eventually but not instantly, this is eventual consistency — read a key on two different followers a moment apart and you may get two different answers, both of which were once true on the leader.

A worked example: the lagging follower

A Shopee seller updates a listing price from S$25 to S$20 at t=0. The write commits on the leader instantly. Follower A is 50ms behind and Follower B is, right now, 4 seconds behind because it's chewing through a big replay queue.

  • A buyer's app reads the price at t=100ms and the load balancer routes it to Follower A → it sees S$20. Correct.
  • Another buyer reads at t=200ms and gets routed to Follower B → it still sees S$25. Stale.
  • At t=4s Follower B finishes catching up and now also serves S$20.

For a price display that flicker is usually tolerable. For a "you have S$0 balance" message shown to a user who just topped up, it is alarming. The size of the problem is entirely about which data is being read stale.

Measuring and bounding lag

You measure lag with the log positions from Step 02. PostgreSQL exposes pg_stat_replication with each follower's replay_lsn versus the leader's current LSN; the byte gap (or a time estimate) is your lag. MySQL reports Seconds_Behind_Source. The operational rule: monitor lag and eject over-lagged followers from the read pool. Many routing layers (ProxySQL, Vitess, AWS RDS Proxy) will stop sending reads to a replica whose lag exceeds a threshold, so a sick follower degrades read capacity rather than serving wrong answers.

05 Taming staleness

Read-your-own-writes

The most jarring form of stale read is when a user can't see their own action. A GrabPay user tops up S$50, the success screen appears, the app immediately re-fetches the balance from a lagging follower, and it shows the old balance. To the user the top-up looks like it failed. The guarantee you need here has a name: read-your-own-writes consistency (also called read-after-write) — a user is always guaranteed to see updates they themselves just made, even though they may still see other users' updates with some lag.

This is a weaker, cheaper promise than full strong consistency, and there are several standard ways to deliver it.

Route the writer to the leader for a window

The bluntest fix: for a short period after a user performs a write (say a few seconds), route that user's reads of the affected data to the leader instead of a follower. The leader is always current, so they always see their own change. After the window, assume followers have caught up and resume reading from them. Simple and effective; the downside is it pushes read load back onto the leader, so you keep the window tight and scope it to the data the user just touched.

Track a write version and wait for it

The more surgical fix uses the log positions again. When a user performs a write, the app remembers the leader's log position at that moment (an LSN / token). On the next read, it only accepts a follower whose replay_lsn has reached or passed that token — otherwise it waits briefly, retries another follower, or falls back to the leader. This delivers read-your-own-writes without permanently pinning the user to the leader.

// pseudo-flow for read-after-write via a version token
token = leader.write(topUp)        // returns LSN, e.g. "0/16B3748"
session.lastWriteLSN = token

// on the next read for this user:
f = pickFollower()
if (f.replayLSN >= session.lastWriteLSN) read(f)   // caught up → safe
else                                       read(leader) // fall back

The cross-device wrinkle

"Their own writes" gets harder when a user has two devices, or when the write and the read arrive at different datacentres. A user who tops up on their phone and immediately checks the balance on the web should still see the new value — so the version token must travel with the user (in their session / a centralised store), not just the single TCP connection. This is exactly the kind of nuance an interviewer probes to separate "I memorised the term" from "I've built it."

Classic interview question

"A user just posted a comment and reloads the page — how do you make sure they see it?" Walk the menu: pin their reads to the leader briefly, or carry a write-version token and only read from a caught-up follower. Then note the harder variant — monotonic reads (a user should never see time go backwards by hopping between followers at different lag) — which you fix by routing a given user consistently to the same follower (e.g. hash the user id to a replica).

06 The failure mode

When the leader dies

Everything above assumes a healthy leader. The whole reason we replicated, though, was to survive losing it — so what actually happens the moment the leader stops responding? This process is failover: detecting the leader is gone, choosing a follower to take over, and rewiring the system around the new leader. It is the single hardest and most-probed part of replication, because every sub-step has a way to go wrong.

The four moves of failover

  • Detect. Decide the leader is actually dead, not merely slow. Done with timeouts and heartbeats — a monitor (or the followers themselves) pings the leader, and after it misses responses for some window (say 10–30 seconds) it's declared down. Too short a timeout causes false alarms during a GC pause or a network hiccup; too long extends the outage. There is no perfect value.
  • Choose. Pick which follower becomes the new leader. The best candidate is the one with the most up-to-date data — the highest applied log position — to minimise lost writes. Picking the new leader requires agreement among the surviving nodes, which is a consensus problem (Raft/Paxos under the hood in modern systems).
  • Promote & redirect. Promote the chosen follower to leader and point all writes at it. Clients and the other followers must learn the new leader's address — via a coordinator, a virtual IP, DNS, or a service-discovery layer. Until they do, writes fail.
  • Reconfigure & recover. The remaining followers start replicating from the new leader. When the old leader eventually comes back, it must not resume as leader; it rejoins as a follower and catches up.

Step through the full lifecycle — the first three steps are the happy path, and the last three are the failover branch.

RPO and RTO — the two numbers to quote

Frame failover quality with two metrics. RPO (Recovery Point Objective) is how much data you can afford to lose, measured in time — with async replication, RPO equals the replication lag at the moment the leader died (the un-replicated tail of writes is gone). RTO (Recovery Time Objective) is how long you can be down, measured from failure to a working new leader — dominated by detection timeout plus promotion time. Synchronous replication pushes RPO toward zero at the cost of write latency; faster detection lowers RTO at the cost of more false failovers. Saying "with semi-sync our RPO is near zero and our RTO is about 30 seconds" is exactly the language interviewers want.

Failure within the failure — lost writes

Under async, the dead leader may have acknowledged writes that no follower received. Those writes are lost on promotion (that's your RPO). Worse, if the old leader rejoins and naively replays its un-replicated changes onto a timeline the new leader has already advanced, you get conflicting data. The standard guard: when the old node returns, it discards any local writes ahead of the new leader's branch point (PostgreSQL's pg_rewind does exactly this) and re-syncs as a follower.

07 The dangerous failure

Split-brain and fencing

The nastiest failover bug isn't losing a leader — it's accidentally ending up with two. Split-brain is the situation where two nodes both believe they are the leader and both accept writes. It usually happens when detection is wrong: a network partition cuts the original leader off from the monitor, so the monitor declares it dead and promotes a follower — but the original leader is alive and well on the other side of the partition, still taking writes from clients that can reach it. Now you have two divergent write histories, and merging them later means real, possibly unresolvable, data corruption — duplicate payment records, conflicting balances.

Quorum: require a majority to act

The first defence is to never let a minority promote. Require a quorum — a majority of nodes must agree before a new leader is elected. In a 3-node cluster you need 2 votes; in 5 you need 3. During a partition only the side holding the majority can elect a leader, so the minority side (which may include the old leader) can't keep operating as a leader. This is why clusters are sized to odd numbers: it guarantees exactly one side can ever have a majority.

Fencing: forcibly silence the old leader

Quorum stops a new bad leader, but you also have to be sure the old one stops. Fencing (memorably, STONITH — "shoot the other node in the head") forcibly prevents the deposed leader from accepting any more writes — by revoking its access, cutting its network, or refusing its writes at the storage layer. A common software version is a monotonic fencing token: every leadership term gets an increasing number, the shared resource records the highest token it has seen, and it rejects any write stamped with an older token. So even if a zombie old leader wakes up and tries to write, its stale token is bounced.

This is why real systems lean on a dedicated coordination service rather than home-grown heartbeats: ZooKeeper, etcd, or Consul provide consensus, leader election, and fencing tokens as primitives. Patroni (for PostgreSQL) builds HA on top of etcd/ZooKeeper; MongoDB's replica sets run their own Raft-based election internally; Redis uses Sentinel or Cluster for the same job.

The question that separates levels

"You set the heartbeat timeout to 5 seconds to fail over fast. What goes wrong?" Strong answer: a transient network blip or a stop-the-world GC pause longer than 5s triggers a needless failover, and during the chaos you risk split-brain if the old leader is only briefly unreachable. The real fix isn't a magic timeout — it's quorum plus fencing so that even a wrong detection can't cause two active leaders. Detection makes failover faster; fencing makes it safe.

08 The road not taken

Why not multiple leaders?

A natural question: if one leader is a write bottleneck and a failover risk, why not just have several leaders, each accepting writes? This is multi-leader replication, and it is genuinely useful in two cases — multi-datacentre setups where you want a local leader in each region so writes don't pay a cross-ocean round trip (a Grab leader in Singapore and another in Jakarta), and offline-capable clients where each device is effectively its own leader that syncs later.

The price: write conflicts

Multi-leader's fatal complication is the write conflict. Two leaders can accept conflicting writes to the same record concurrently, and because each accepted locally before learning about the other, there is no single authority to say which one "wins." Imagine the Singapore leader sets a user's display name to "Toan" while, at the same instant, the Jakarta leader sets it to "TVT." Both commit. When the streams meet, the system has two equally valid truths and must resolve the conflict — last-write-wins (which silently discards data), application-defined merge logic, or conflict-free data types (CRDTs). All of these are extra complexity that single-leader simply doesn't have.

So the honest framing in an interview: single-leader is the default precisely because it sidesteps write conflicts by funnelling all writes through one ordering authority. Reach for multi-leader only when geographic write latency or offline operation forces your hand — and the moment you do, conflict resolution becomes the central design problem.

Single-leader

  • + One write order — no conflicts to resolve
  • + Simple to reason about and operate
  • − Write throughput capped by one node
  • − Cross-region writers pay a round trip to the leader

Multi-leader

  • + Local writes per region → low write latency
  • + Tolerates a region or link going down
  • − Write conflicts are unavoidable and hard to resolve
  • − Much more operational and reasoning complexity

There is a third branch worth naming so the interviewer sees you know the map: leaderless replication (the Dynamo style used by Cassandra and DynamoDB), where clients write to several nodes directly and reads use quorums (R + W > N) to paper over staleness. That deserves its own lesson; here, just place it as "the other end of the spectrum from single-leader."

09 The concrete contract

The read/write API split

Bring it down to code. The clean way to use a replicated database is to make the read path and write path explicit in your data-access layer: writes always target the leader's connection; reads target a replica pool. This split is what actually delivers the read scaling — if your code accidentally sends reads to the leader, you've gained redundancy but not capacity.

// Data layer routes by operation type
class WalletStore {
  read(userId)  { return replicaPool.query(SELECT_BALANCE, userId); }
  write(userId, delta) { return leader.exec(UPDATE_BALANCE, userId, delta); }

  // read-your-own-writes: caller can force the leader
  readFresh(userId) { return leader.query(SELECT_BALANCE, userId); }
}

How is this wired in practice? Three layers do it. Application-level: many ORMs and drivers support a primary plus read-replica config and route by the SQL verb. A proxy: ProxySQL, MaxScale, PgBouncer/Pgpool, or AWS RDS Proxy sit between app and database, sending SELECT to replicas and writes to the primary, and transparently failing over the primary endpoint. A connection-string convention: managed services give you a single writer endpoint and a separate reader endpoint (Aurora's reader endpoint load-balances across all replicas) so the app just picks the right URL.

The trap: read-after-write through a naive split

The split reintroduces the staleness problem from Step 05 in a sneaky way: a request that writes then immediately reads will send the write to the leader and the read to a (lagging) replica, and read its own stale data. That's why the readFresh escape hatch exists — for the narrow set of reads that must reflect the caller's just-made write, force them to the leader (or use the version-token check). Designing that escape hatch deliberately, rather than discovering it as a bug in production, is the senior move.

Failure mode of the split itself

What if the entire replica pool is lagging or down? You need a policy: do reads fail open by falling back to the leader (protecting correctness and availability, but risking overloading the leader), or do they fail closed? Most systems fall back to the leader for reads when no healthy replica exists — accepting temporary extra leader load as cheaper than an outage. Decide it on purpose and document the threshold.

10 The next ceiling

What breaks at 10x

Replication scaled your reads and made you resilient. Now imagine the product 10x's its write volume — a Shopee 9.9 sale, a payday surge of SeaMoney transfers, a viral moment on a feed. Where does the design break?

  • The single leader saturates. Every write still goes through exactly one node, and adding followers does nothing for write throughput — in fact each new follower adds replication work to the leader. The leader's CPU, write IOPS, and the fsync on its commit path become the wall.
  • Replication lag worsens. Followers must replay the same write stream the leader produces. As write rate climbs, followers fall behind, stale-read windows widen, and read-your-own-writes gets harder. The very thing scaling your reads now struggles to keep them correct.
  • Failover gets scarier. A higher write rate under async means a larger un-replicated tail at any instant — your RPO grows precisely when an incident is most likely.

The fix is the next tool, not more of this one: sharding (horizontal partitioning) — split the data across multiple independent leader sets by a partition key (user_id, wallet_id, region), so each shard absorbs only its slice of writes. Sharding scales writes and storage; replication scales reads and survivability — and you use them together: each shard is itself a small leader-follower cluster. That combination is the backbone of essentially every large-scale stateful system, and it's the subject of Lesson 10.

Where the interviewer takes it next

Expect the bridge question: "Replication didn't help your writes — so now what?" The answer is sharding, and the moment you say it you've opened the next set of trade-offs: choosing a partition key that spreads load evenly (avoiding hot shards), handling cross-shard queries and transactions, and rebalancing when you add shards (which is exactly where consistent hashing earns its keep).

★ Review

Summary & self-test

The whole topic compresses to one chain: one database is a SPOF and a read ceiling → copy the data onto followers → choose how synchronously to copy it → accept that async copies lag and serve stale reads → patch the user-visible cases (read-your-own-writes) → and design carefully for the moment the leader dies, where consensus, quorum, and fencing keep you from losing or duplicating data. If you can draw the leader-follower diagram, name what flows in the log, and reason about lag and failover out loud, you can carry this question.
