25Lesson · System Design Interview

Design a Chat / Messaging System

A chat system must deliver a message in near real time, know who is online, keep a conversation in order, and lose nothing when the recipient is offline. The crux is unglamorous but brutal: holding millions of long-lived connections open at once, and routing one message to the exact server that holds the recipient's socket.

About these notes: original study material on designing a real-time messaging system — a standard, widely-documented interview topic. The explanations, numbers, schemas, and diagrams here are my own; nothing is copied from any paid course or book. Three diagrams are interactive — press Play (or step with Next) to watch the flow. Examples lean toward Singapore super-apps and fintech (Grab, Shopee, GrabPay, SeaMoney) but the mechanics are general.
00 The problem

What makes chat hard

On the surface a chat app looks trivial: A types a message, B sees it. Under the surface it is one of the hardest "design a system" prompts, because the comfortable request/response model you have built your whole career on does not fit. A web server is stateless and pull-based: the client asks, the server answers, the connection closes. Chat is stateful and push-based: the server must shove a message down to a client that never asked, the instant it arrives, while that client sits idle.

That single inversion creates every bottleneck in this lesson. To push to B, the server must keep an open channel to B. Multiply by a real user base and you are holding millions of idle connections open simultaneously — most doing nothing most of the time, all consuming file descriptors and memory. And when A's message lands on some server, that server has no idea which of its hundreds of sibling servers is holding B's channel. Finding B is the central puzzle.

Concretely, a chat system has to solve four problems at once, and they fight each other:

  • Real-time delivery — latency budgets in the tens of milliseconds, not seconds.
  • Presence — accurately showing who is online without melting under the fan-out.
  • Ordering — messages in a conversation must appear in a consistent order for everyone.
  • Durability — if B is offline (or B's phone is in a tunnel), the message must survive and arrive later, exactly once, in order.

The whole lesson is the chain of fixes that this inversion forces. Keep that lens: every box we add exists because the previous design could not push a message to an offline-or-elsewhere recipient cheaply, in order, without losing it.

01 Scope it

Clarify the requirements

Chat is huge — narrow it before drawing. The interviewer wants to see you carve the scope down and name your assumptions out loud. Separate the functional from the non-functional.

Functional requirements

  • 1:1 chat and group chat (cap groups at, say, 500 members — fan-out behaves very differently at 5 vs 50,000).
  • Real-time delivery when both parties are online; a message lands in well under a second.
  • Offline delivery — store the message and deliver on reconnect, plus a push notification to wake the app.
  • Message ordering — within a conversation, everyone sees the same order.
  • Presence / online status and last seen.
  • Receipts — delivered (reached the device) and read (the user opened it).
  • History — a user can scroll back and sync messages across multiple devices.

Out of scope (state it)

To keep the hour focused, explicitly defer: audio/video calls, end-to-end encryption (mention you would layer it on top with per-device keys), stickers/media transcoding, and spam/abuse detection. Naming what you are not building is as valuable as naming what you are.

Non-functional requirements

  • Low latency — the delivery path is the product; aim for sub-200 ms end-to-end for online users.
  • High availability — a single chat server dying must not lose messages or drop the user permanently.
  • Durability / reliability — once the server acknowledges a send, the message must never vanish.
  • Scale — tens of millions of concurrent connections; this drives nearly every later decision.
  • Consistency — per-conversation ordering is required; cross-conversation global order is not.
Interview tip

The line that frames the rest of your answer: "The hard part isn't storing messages — it's holding millions of persistent connections and routing a message to the one server that owns the recipient's socket." Say that early and you have set the agenda the interviewer wants to explore.

02 Estimation

Back-of-the-envelope estimation

Numbers turn vague design into engineering. Assume a regional super-app's messaging feature — think Grab's in-app chat between rider and driver, plus social DMs. Pick round numbers and reason from them.

Connections — the dominant cost

  • 50 M daily active users (DAU).
  • Assume 20% are connected at peak → 10 M concurrent persistent connections.
  • If one connection ("chat") server holds ~100 K sockets — an aggressive but achievable number with tuned epoll and lean per-connection memory — you need ~100 connection servers just for the live channels, before any redundancy.

That last line is the headline. The fleet is sized by open connections, not by request rate. A server holding 100 K idle sockets at, say, 10 KB of kernel + app state each burns roughly 1 GB of memory doing nothing — and that is the cheap part; the expensive part is the routing table that tracks where all 10 M sockets live.

Message throughput

  • Say each DAU sends 40 messages/day → 2 billion messages/day.
  • 2 B / 86,400 s ≈ 23 K messages/sec average; assume a 5× peak → ~115 K writes/sec.
  • Reads (history sync, receipts) are higher — call it 5–10× the writes.

Storage

  • Average stored message ~ 300 bytes (text + metadata, media stored separately in object storage).
  • 2 B × 300 B = ~600 GB/day of message data → ~210 TB/year before replication.
  • With 3× replication that is well into the petabyte range over a few years — this rules out a single relational box and points straight at a partitioned, write-optimized store.
Why this matters

Two numbers should drive your whole design: 10 M concurrent connections (so you need a horizontally scaled connection tier and a way to find any given socket) and ~115 K writes/sec into hundreds of TB (so you need a write-optimized, conversation-partitioned store, not a single SQL instance). Everything downstream is a consequence of these two.

03 Contract

API & data model

Before architecture, sketch the contract. Chat has a split personality, and naming it scores points: a real-time channel for live send/receive, and a plain HTTP/REST tier for everything that is a normal request — login, fetching history, creating a group, uploading media.

The two surfaces

Over the persistent WebSocket channel, the client and server exchange small typed frames rather than calling endpoints. A minimal frame protocol:

// client → server (over the open WebSocket)
{ "op":"send", "client_msg_id":"a91f...", "conversation_id":"c_77",
  "text":"on my way, 5 min", "ts":1717050000123 }

// server → client (acks the send, assigns the real id + sequence)
{ "op":"ack", "client_msg_id":"a91f...",
  "message_id":"7283461902837", "seq":4412 }

// server → client (a message from someone else, pushed down)
{ "op":"deliver", "message_id":"7283461902901", "conversation_id":"c_77",
  "sender_id":"u_12", "seq":4413, "text":"thanks!", "ts":1717050001500 }

// client → server (receipts) and server → others
{ "op":"receipt", "conversation_id":"c_77", "up_to_seq":4413, "kind":"read" }

The stateless REST tier handles the non-real-time work and can be scaled like any normal web service:

POST   /v1/conversations                 // create 1:1 or group
GET    /v1/conversations?limit=20        // the user's chat list (inbox)
GET    /v1/conversations/{id}/messages?before_seq=4413&limit=50  // history page
POST   /v1/media   (returns an upload URL) // large files go to object storage
GET    /v1/ws-endpoint                     // returns which gateway to dial

Data model sketch

Three logical stores, each shaped by how it is read. Going past the textbook: notice the message table is keyed so that all rows for one conversation live together and are sorted by sequence — that is what makes "load the last 50 messages" a single, cheap range scan.

-- conversations (who is in which chat)
conversation:  conversation_id (PK), type {dm|group}, created_at,
               last_message_seq, member_count

-- membership (a row per user per conversation)
member:        conversation_id, user_id, joined_at, last_read_seq, role

-- messages: wide-column, partition by conversation, clustered by seq
message:       PARTITION KEY = conversation_id
               CLUSTERING KEY = seq DESC      // newest first
               message_id, sender_id, type, body, created_at, edited_at

The last_read_seq on each membership row is the trick that makes read receipts and unread counts cheap: a receipt is just an update to one number, and "unread count" is last_message_seq − last_read_seq, computed without scanning any messages.

Interviewer follow-up

Expect: "why a separate WebSocket protocol instead of REST for everything?" Answer: REST is request/response and cannot push; opening a fresh TCP+TLS handshake per message would add 100+ ms and crush the server. The persistent channel pays the handshake once and then streams tiny frames in both directions.

04 Transport

Transport: WebSocket vs the alternatives

The first real decision is how the client and server talk. We need a channel the server can push down at any moment. HTTP request/response cannot do that natively, so the industry has tried several hacks; understanding why each fails leads you to the answer.

How we got to WebSocket

  • Short polling — the client asks "anything new?" every few seconds. Simple, but wasteful: mostly empty responses, and latency equals the poll interval. At our scale this is millions of pointless requests per second.
  • Long polling — the client sends a request and the server holds it open until a message arrives or a timeout fires, then the client immediately re-asks. Better latency, works through any HTTP proxy, but each delivery still costs a full HTTP round trip, and a held-open request ties up a server worker. It is a reasonable fallback when WebSocket is blocked.
  • Server-Sent Events (SSE) — a one-way server→client stream over HTTP. Great for push, but unidirectional, so you still need a second channel for the client to send. Awkward for chat.
  • WebSocket — the answer. After a one-time HTTP upgrade handshake, the same TCP connection becomes a full-duplex, persistent, low-overhead channel: either side sends framed messages at will, with just a few bytes of framing per message.

How WebSocket works under the hood

The client opens a normal HTTP/1.1 request with Upgrade: websocket and a Sec-WebSocket-Key header. The server replies 101 Switching Protocols, and from that point the bytes on the socket are WebSocket frames, not HTTP. The connection stays open for hours. Because it begins as HTTP on port 443, it traverses firewalls and load balancers that already speak HTTP/TLS. To keep middleboxes from killing an idle socket, both sides exchange periodic ping/pong frames (also our presence heartbeat — see §10).

WebSocket

  • True bidirectional push, lowest per-message overhead
  • One handshake, then the channel is free
  • Doubles as the presence heartbeat
  • Needs stateful connection servers + reconnect logic

Long polling (fallback)

  • Works anywhere plain HTTP works; trivial to proxy
  • No special server tier required
  • A full HTTP round trip per delivered message
  • Higher latency; ties up a worker while held open

Tools & knobs

The connection servers are typically built on event-driven runtimes that handle many sockets per thread: Netty (JVM), Go goroutines, or Node.js with ws. The OS must be tuned: raise ulimit -n (file descriptors) into the millions, enlarge the ephemeral port range, and lift net.core.somaxconn. A real production chat tier (WhatsApp famously did this on FreeBSD/Erlang) pushes a single box to millions of connections by keeping per-connection state tiny.

Failure mode

Sockets die constantly — mobile users switch from Wi-Fi to 4G, walk into the MRT, lock the phone. The design must treat disconnection as the normal case, not the exception: the client auto-reconnects with backoff, redials the gateway, re-authenticates, and asks "what did I miss since seq N?" Never assume a connection that was alive a second ago still is.

05 The blueprint

High-level architecture

Now assemble the tiers. The single most important design move is to split stateful from stateless: the connection-holding servers are special and precious; everything else is ordinary and replaceable.

The tiers

  • Connection (chat) servers — the stateful tier. Each holds tens of thousands of live WebSocket sockets and does one job: keep the channel open and shuttle frames between the socket and the rest of the system. Green = app server.
  • Stateless API servers — handle REST: login, history, group management, media. Scale them like any web tier.
  • Session / presence registry — the map user_id → connection_server (plus presence state). Almost always Redis. This is what makes routing possible. Amber = cache/registry.
  • Message router + queue — moves a message from the sender's server to the recipient's server, and buffers for offline users. Often a pub/sub or queue like Kafka or Redis pub/sub. Violet = queue.
  • Message store — durable history, a wide-column DB like Cassandra partitioned by conversation. Blue = database.
  • Push service — hands off to APNs/FCM when the recipient is offline (ties to the notification-system lesson).

Notice the dependency: the chat servers are dumb pipes that read/write the registry, publish to the router, and persist to the store. The intelligence of "where is B, and how do I reach them" lives in the registry + router. That separation is what we attack next, because it is the part that breaks first.

06 The hard part

Routing: how do you find B's socket?

This is the heart of the interview. A is connected to chat-7; B is connected to chat-42. When A sends a message, it arrives on chat-7, which holds no socket for B. How does chat-7 push the message down B's channel, which lives on a different machine it has never heard of?

The session registry

The answer is a shared lookup table: the session registry (a.k.a. presence/session store). When B's WebSocket connects to chat-42, that server writes a record into the registry:

// in Redis, on B's connect
SET   session:u_B   "chat-42"   EX 60      // short TTL, refreshed by heartbeat
SADD  conn:chat-42  u_B                    // reverse index for cleanup on crash

Now routing is a two-step lookup-then-forward. When chat-7 needs to reach B, it does GET session:u_B, learns "B is on chat-42", and forwards the message there. chat-42 finds B's socket in its local in-memory table and writes the frame down the wire.

Direct forward vs pub/sub

Two ways to do the "forward to chat-42" hop, and the interviewer will want the trade-off:

Direct server-to-server

  • Look up the server, then make a direct gRPC/internal call to it
  • Lowest latency — one hop, no broker
  • Servers must discover and connect to each other (a mesh)
  • A stale registry entry sends the message to the wrong/dead box

Pub/sub via a broker

  • Each chat server subscribes to its own topic/channel
  • Forwarding = publish to topic:chat-42; the broker delivers
  • Decouples senders from receivers; easy fan-out for groups
  • One extra hop through the broker (Redis pub/sub or Kafka)
Interviewer follow-up

"What happens when the registry entry is stale?" — B's server crashed but the session:u_B key has not expired yet. The forward goes to a dead box. The fix is twofold: short TTLs refreshed by heartbeat so stale entries self-clean in seconds, and treating a delivery failure as a signal to re-resolve and fall back to offline storage. Never trust the registry blindly; always have the offline path ready.

07 End-to-end

The send / deliver path

Stitch the pieces into the full lifecycle of one message — the flow the interviewer most wants to hear you narrate cleanly. There is a critical ordering decision baked in: persist before you acknowledge. The server must write the message durably before it tells the sender "got it," otherwise a crash between ack and write silently loses a message the sender believes was sent.

Online path

  • A sends a send frame to its chat server.
  • The server assigns a message_id and a per-conversation seq, then persists the message to the store.
  • It returns an ack to A (carrying the assigned id + seq) — A's spinner turns to a single tick.
  • It looks B up in the session registry, forwards the message to B's chat server, which pushes it down B's socket.
  • B's client returns a delivered receipt (double tick); when B opens the chat, a read receipt (blue ticks).

Offline branch

If the registry shows B has no live connection (or the forward fails), the message is already safely persisted. The server writes it to B's inbox / offline queue and fires a push notification via APNs/FCM. When B reopens the app and reconnects, the client asks "what did I miss since my last seq?" and the server replays the backlog in order.

Exactly-once feel, at-least-once reality

Networks drop acks, so the client retries sends with the same client_msg_id. The server deduplicates on that id (a unique constraint or a short-lived seen-set), so a retried send maps to the same stored message rather than a duplicate. This gives users an exactly-once experience on top of at-least-once delivery — a point worth saying out loud.

Interviewer follow-up

"What happens when the recipient is offline?" — the message is already persisted (you wrote before you acked), so nothing is lost. You enqueue it to B's inbox and trigger a push notification. On reconnect, B's client syncs from its last-seen seq and receives the backlog in order. The push notification is a wake-up, not the message itself.

08 Ordering

Message IDs & per-conversation ordering

Two people on two phones in two cities cannot agree on a clock, yet everyone in a conversation must see messages in the same order. We need an ordering scheme that does not rely on synchronized client clocks.

Why client timestamps fail

If you ordered by the sender's device clock, a phone with a skewed clock would inject a message "into the past," and two messages sent in the same millisecond would tie. Clock skew across devices can be seconds. Ordering must be decided server-side, where you control the source of truth.

The fix: a per-conversation sequence number

Assign each message a monotonically increasing seq within its conversation. The chat server (or a small per-conversation coordinator) increments a counter and stamps the message. Because all messages for one conversation funnel through the same partition, ordering is total and gap-detectable: a client that has seq 100 and receives 102 knows it missed 101 and can request it. You do not need a global order across all conversations — only within each one — which keeps the counter local and cheap.

Globally unique message IDs

Separately, every message needs a globally unique message_id for storage, dedup, and cross-device references. Reach for a Snowflake-style ID (from the unique-ID lesson): 64 bits of timestamp ‖ machine_id ‖ sequence. These are roughly time-sortable, generated without coordination, and never collide. Note the division of labor: message_id is globally unique but only roughly ordered; seq is the strict per-conversation order. Use the right one for the right job.

Per-conversation seq

  • Strict, total order within a chat; gaps are detectable
  • Cheap — a single counter per conversation
  • Needs one writer per conversation to stay monotonic

Snowflake message_id

  • Globally unique, no coordination, time-sortable
  • Perfect for dedup and primary keys
  • Only approximately ordered — not safe as the display order
Failure mode

If two chat servers both think they own a conversation's counter, you get duplicate seq values and a corrupted order. Avoid it by routing all writes for a conversation through one place (a per-conversation partition leader, or letting the store assign the seq atomically via a conditional write / lightweight transaction). When that owner fails over, a new one resumes the counter from the store's last_message_seq.

09 Storage

Message storage & the sync model

We estimated hundreds of TB/year at ~115 K writes/sec. That workload is write-heavy, append-mostly, and always read by conversation in time order. That shape points away from a single relational database and toward a wide-column store like Cassandra (or DynamoDB / HBase).

Partition by conversation, cluster by seq

The decisive schema choice: partition key = conversation_id, clustering key = seq (descending). All messages of one chat live on the same partition (replica set), physically sorted newest-first. That makes the two hot queries trivial: "give me the latest 50" is a bounded range scan from the top of the partition; "load older" is a scan starting before a cursor. Writes are appends to the tail of a partition — exactly what an LSM-tree store like Cassandra is fastest at.

The hot-partition risk

A huge, very active group chat becomes one enormous, scorching partition — the classic hot partition. Mitigate by bucketing the partition key with time (conversation_id ‖ month) so an old conversation's history spreads across partitions and no single one grows unbounded.

Inbox / sync model

History (the shared message store, keyed by conversation) is separate from the per-user inbox (what each device still needs to receive). A device tracks its last synced seq per conversation; on reconnect it sends those cursors and the server streams the deltas. This is what makes multi-device work: your laptop and phone each hold their own cursor and independently catch up to the same shared history. Media never goes in the message row — large files live in object storage (S3), and the message carries only a URL + thumbnail.

Interviewer follow-up

"How do you guarantee ordering on the read side?" — because the store sorts by seq within the partition, a history page is already ordered; the client merges live-pushed messages by seq, detects gaps, and back-fills missing seqs from history. Display order is always by seq, never by arrival time at the device.

10 Presence

Presence & heartbeats

Presence — the green dot and last seen — is deceptively expensive. The naive idea "set online on connect, offline on disconnect" fails because disconnects are often silent: a phone that loses signal never sends a clean close. So you cannot trust the absence of a disconnect to mean "still online."

How heartbeats work

The client sends a small heartbeat (a WebSocket ping, or an app-level frame) every N seconds — say 30. On each heartbeat the chat server refreshes the presence key with a TTL a bit longer than the interval: SET presence:u_B online EX 90. If three heartbeats are missed, the key expires on its own and the user is automatically considered offline — no explicit disconnect event needed. This is the same registry record from §6 doing double duty: its TTL is the presence signal.

The presence fan-out trap

The real danger is broadcasting presence changes. If a popular user with 5,000 contacts flips online, and you push that event to all 5,000, one toggle becomes 5,000 messages. A flaky connection that flaps online/offline every few seconds turns into a presence storm that can saturate the system — a genuine outage cause in real chat products. Mitigations: only compute presence on demand when someone actually opens a chat or contact list; debounce flapping (don't broadcast a blip shorter than a few seconds); and for very large groups, drop per-member presence entirely.

Interviewer follow-up

"How do you keep presence from overwhelming the system for users with huge follower counts?" — pull, don't push: resolve presence lazily when a viewer opens the relevant screen, cache it briefly, debounce flaps, and never fan a single toggle out to thousands of subscribers. The 80/20 answer: presence is best-effort and eventually consistent, so optimize for cost, not perfection.

11 Scale & failure

Group chat, offline delivery & what breaks at 10×

Group fan-out: write vs read

A 1:1 message has one recipient; a group message has M. The core decision mirrors the news-feed lesson — fan-out-on-write vs fan-out-on-read:

Fan-out on write

  • On send, push to all M online members and enqueue for offline ones
  • Reads are instant — each member already has it
  • Cost scales with M; a 50 K-member group is a write storm
  • Great for small/medium groups (our 500 cap)

Fan-out on read

  • Store once in the conversation; members pull on open/sync
  • Write is cheap regardless of group size
  • Read does more work; needs good caching of recent messages
  • Better for very large or broadcast-style groups

The pragmatic hybrid: fan-out-on-write for normal groups, fan-out-on-read (members poll the conversation) past a size threshold. Either way the message is stored once in the conversation partition; fan-out only governs delivery, not storage.

Offline delivery & push

When members are offline, the message sits in the store and a push notification goes out via APNs (iOS) / FCM (Android). The notification is just a nudge carrying enough to render a preview and the conversation id; the real message syncs when the app reconnects. Coalesce notifications per conversation so a 200-message group burst becomes "12 new messages," not 200 buzzes.

Failure mode: a connection server dies

This is the marquee failure question. A chat server holding 100 K sockets crashes. Every one of those clients' TCP connections drops at once. The recovery chain:

  • Clients detect the dead socket (failed heartbeat) and reconnect with exponential backoff and jitter, so 100 K clients don't stampede the same replacement box simultaneously.
  • The load balancer routes each reconnect to a healthy chat server.
  • The new server writes fresh session:user → new_server entries; the dead server's stale entries expire by TTL within ~60 s, and its reverse-index set (conn:chat-dead) lets a janitor purge them faster.
  • On reconnect, each client syncs from its last seq — anything sent during the gap was persisted and is replayed. No messages are lost because we persisted before acking.

What breaks at 10×

Push to 100 M concurrent connections and the pressure points shift:

  • The session registry becomes the bottleneck. A single Redis can't hold 100 M keys with heartbeat write pressure. Shard the registry by user_id (Redis Cluster), and consider co-locating presence with the routing map to halve the lookups.
  • Cross-server forwarding floods the network. The mesh of N×N server connections explodes. Move to a pub/sub broker (Kafka, or a dedicated routing tier) so each server has O(1) broker connections instead of O(N) peers.
  • Geography. A Singapore user messaging another Singapore user shouldn't traverse a US region. Deploy connection servers and registries per region, route users to the nearest, and bridge cross-region conversations through replicated topics.
  • The store needs more partitions. Add Cassandra nodes; because we partition by conversation, the data redistributes by consistent hashing without re-architecting.
Interviewer follow-up

The three questions you must be able to answer crisply, because they are this design:
(1) how does the system find the recipient's connection? (session registry: user → server).
(2) how do you guarantee ordering? (server-assigned per-conversation seq, store sorted by it).
(3) what happens when the recipient is offline? (persist-before-ack, enqueue to inbox, push notification, replay on reconnect).
If you can whiteboard those three, you have the lesson.

★ Review

Summary & self-test

The whole topic compresses to one sentence: hold the recipient's connection somewhere, keep a map of where every connection lives, and persist every message before you acknowledge it so it survives the gap until the recipient reconnects — in order. If you can draw the connection tier, the session registry, and the send/deliver path with the offline branch, you can carry the interview.
