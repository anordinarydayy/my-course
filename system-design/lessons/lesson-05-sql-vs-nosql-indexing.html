05Lesson · System Design Interview

SQL vs NoSQL & Indexing

The right datastore falls out of your access pattern, not out of the hype cycle. This lesson gives you a defensible way to choose between relational and NoSQL stores — then opens the engine itself: B-tree versus LSM-tree, what an index actually costs you on every write, and the moment when duplicating data quietly beats a clean join.

About these notes: original study material on a standard, widely-documented engineering topic — relational versus non-relational stores and how their indexes work. Every explanation, number, example, and diagram here is my own; nothing is copied from any paid course or book. One diagram is interactive — press Play (or step with Next) to watch the LSM write path.
00 The decision

Choose by access pattern, not by trend

The fastest way to get a database question wrong in an interview is to answer "SQL or NoSQL?" with a preference instead of a reason. Both families are mature, both scale to enormous workloads, and both back trillion-dollar companies. The honest answer is that neither is "better" — they make opposite bets, and the right bet depends entirely on how your data is read and written.

So lead with the decision, then justify it. The single most useful question to ask out loud is: "What is the dominant access pattern?" Concretely:

  • Shape of the queries. Do you fetch one entity by its key, or join five tables and aggregate across them?
  • Read/write ratio. A read-heavy product catalogue and a write-heavy event log want very different engines underneath.
  • Consistency need. Can two users briefly see different balances, or must a wallet debit be correct to the cent, always?
  • Scale and growth. Will one machine hold this in three years, or are you already planning to spread it across a hundred nodes?
  • Schema volatility. Is the shape of a record fixed, or does every new feature add fields?

Hold those five questions in mind for the rest of the lesson — every trade-off we hit maps back to one of them. We'll build up the relational story first, watch it strain, introduce NoSQL as the response, and then drop a level into the indexes that make either choice fast. The index is where the real performance story lives.

01 The baseline

What relational gives you

Start with the default, because for most products it is the correct default. A relational database (an RDBMS) stores data in tables of rows and columns, with a fixed schema declared up front: each column has a type, and rows must conform. PostgreSQL, MySQL, and cloud-managed flavours like Amazon Aurora and Google Cloud SQL are the workhorses here. You query with SQL, a declarative language where you describe what you want and the engine's query planner decides how to get it.

Joins: model relationships once, query them many ways

The defining strength is the join. You store each fact exactly once — users in a users table, orders in an orders table linked by a user_id foreign key — a design called normalization. There is one source of truth for a user's name; change it in one place and every query sees the new value. The join then stitches normalized tables back together at read time:

-- Shopee SG: list a seller's recent orders with the buyer's city
SELECT o.id, o.total_cents, u.city, o.created_at
FROM orders o
JOIN users u ON u.id = o.buyer_id
WHERE o.seller_id = 'S-99214'
  AND o.created_at > now() - interval '7 days'
ORDER BY o.created_at DESC;

You never anticipated this exact query when you designed the schema, and it doesn't matter — the relational model lets you ask new questions of existing data without reshaping it. That flexibility at query time is the trade you make for rigidity at schema time.

ACID transactions: correctness under concurrency

The second pillar is ACID transactions — the property an interviewer most wants to hear named for anything touching money. ACID stands for:

  • Atomicity — a transaction is all-or-nothing. Debit a GrabPay wallet and credit a merchant either both happen or neither does; a crash mid-way leaves no half-transfer.
  • Consistency — every committed transaction moves the database from one valid state to another, respecting constraints (a balance can't go negative if you declared a CHECK).
  • Isolation — concurrent transactions don't step on each other; the result is as if they ran one after another (the strength is tunable via isolation levels).
  • Durability — once committed, the write survives a power loss, because it was flushed to a write-ahead log on disk first.

This is exactly the property a payment ledger lives or dies by. If a SeaMoney transfer can ever leave money in neither account, you don't have a payments product — you have a lawsuit. Relational engines have spent forty years making ACID correct and fast, which is why correctness-critical data overwhelmingly lives in SQL.

How it scales: up, then out via replicas

A relational database scales vertically first — give the single primary node more CPU, RAM, and faster disks. This is simple and gets you remarkably far; a well-tuned Postgres on a large instance handles many thousands of transactions per second. When reads outgrow the primary, you add read replicas: copies that asynchronously follow the primary and absorb read traffic, while all writes still funnel to the one primary.

Under the hood

Replicas stay current by streaming the primary's write-ahead log (the WAL in Postgres, the binlog in MySQL) and replaying it. Because replay is asynchronous, a replica lags the primary by milliseconds to seconds — so a user who writes then immediately reads from a replica can see stale data (the read-your-own-writes problem). This is the first crack we widen in the next section.

02 The bottleneck

Where SQL hits a wall

Read replicas solve read scaling but do nothing for write scaling — every write still hits the one primary. So the question becomes: what happens when the write volume, or the raw data size, outgrows the biggest single machine you can buy? You have to shard: split the data across many independent database servers, each holding a slice keyed by some shard key (say user_id mod 64).

Sharding a relational database is where the pain begins, and naming that pain precisely earns interview points:

  • Joins across shards break down. A join is cheap when both tables sit on one machine. Once users live on shard 3 and their orders on shard 17, the join becomes a cross-network scatter-gather — slow, fragile, and something most SQL engines won't do for you automatically.
  • Transactions across shards get expensive. ACID across machines needs a distributed-commit protocol (two-phase commit), which adds latency and a coordinator that can stall the whole transaction if it dies mid-flight.
  • Rebalancing is operationally heavy. Adding a shard means migrating live data and updating routing while traffic flows. It's doable, but it's a project, not a config change.

The deeper issue is that the relational model's best feature — cross-entity joins and multi-row transactions — is exactly the feature that doesn't survive being spread across a network. The fixed schema also bites: an ALTER TABLE to add a column on a billion-row table can lock or rewrite the table, a genuinely dangerous operation at scale.

Interview tip

Don't say "SQL doesn't scale" — that's wrong and the interviewer will pounce. Say "relational stores scale reads cheaply with replicas and scale up vertically, but scaling writes means sharding, and sharding is where joins and cross-shard transactions get hard." That sentence shows you know exactly where the boundary is.

03 The response

The four NoSQL families

NoSQL ("not only SQL") isn't one thing — it's a label for stores that drop some relational guarantees in exchange for horizontal scalability and flexible schemas. The unifying move is the same: design the storage around the access pattern so that the common query is a single, cheap lookup that lands on exactly one node — which is precisely what makes spreading across many nodes painless. There are four families, and you should be able to name a use case for each.

Key-value: the dictionary at scale

A key-value store is a giant distributed hash map: you put a value under a key and get it back by that key, with no querying inside the value. Redis (in-memory) and Amazon DynamoDB are the canonical examples. Perfect for session tokens, feature flags, shopping carts, and caches — anything where you always know the exact key. A Grab driver's live location keyed by driver_id is a textbook fit: millions of tiny writes per second, every read by key.

Document: JSON objects you can query inside

A document store like MongoDB (or DynamoDB used document-style) holds self-describing JSON-like documents. Unlike key-value, you can query and index fields inside the document. Each document carries its own shape, so adding a field to new records needs no migration — ideal for product catalogues where a phone has different attributes than a sofa, or a user profile that grows feature by feature.

Wide-column: rows with billions of dynamic columns

A wide-column store like Apache Cassandra or Google Bigtable organizes data by a row key, and within a row you can have an enormous, sparse set of columns. It's built for massive write throughput and time-series-shaped data: every GrabPay transaction event, or every "driver moved" ping, partitioned by key and time. You design the table for one query and it serves that query at any scale.

Graph: when the relationships are the data

A graph database like Neo4j stores nodes and edges as first-class citizens and is optimized for traversing relationships — "friends of friends who like this restaurant," fraud rings, recommendation paths. Ironically this is the case where deep relationship queries are faster than in a relational database, because the edges are stored as direct pointers rather than rediscovered by joining on keys each time.

04 The guarantee trade

ACID vs BASE

The reason NoSQL stores scale horizontally so easily is that many of them relax ACID in favour of a softer contract called BASE: Basically Available, Soft state, Eventually consistent. Instead of guaranteeing every reader sees the latest write instantly, the system promises only that, absent new writes, all replicas will converge on the same value given a little time. This is eventual consistency, and it's the price of partition tolerance and high availability at scale (the deep version is the CAP theorem, a lesson of its own).

How eventual consistency plays out

Picture a Shopee product's review count stored in Cassandra, replicated across three data centres. A new review increments the count on one replica and the update propagates to the others over the next few hundred milliseconds. For a moment, a user in Singapore might see "1,204 reviews" while a user in Jakarta sees "1,203." Nobody is harmed; the numbers reconcile shortly. That tiny, acceptable staleness is what buys you the ability to keep serving reads even when a replica is briefly unreachable.

Now run the same logic on a wallet balance. Two concurrent withdrawals each read "S$50 available," each approve, and the account goes to −S$30. That is why money doesn't live in an eventually-consistent store without extra machinery. The rule of thumb: eventual consistency is fine when a stale read is merely cosmetic; it's unacceptable when a stale read lets you violate an invariant.

ACID (typical SQL)

  • Strong consistency: every read sees the latest committed write
  • Atomic multi-row transactions across entities
  • Correctness-first — right for payments, orders, inventory
  • Harder to spread writes across many nodes

BASE (typical NoSQL)

  • Eventual consistency: replicas converge after a short delay
  • Stays available and fast even under partition or node loss
  • Scales writes horizontally with little coordination
  • App must tolerate (or repair) brief staleness and conflicts
Nuance worth stating

The ACID/BASE split is a spectrum, not a wall. Many NoSQL stores offer tunable consistency — DynamoDB lets you request a strongly consistent read per call, and Cassandra lets you pick QUORUM (a majority of replicas must agree) for stronger guarantees at higher latency. Meanwhile "NewSQL" stores like CockroachDB and Google Spanner deliver ACID transactions across a horizontally sharded cluster. So the modern honest line is "pick the consistency you need per operation," not "SQL is consistent, NoSQL isn't."

05 The verdict

The decision, distilled

Now collapse everything above into a decision you can deliver in fifteen seconds. The table is the version to memorize; the guide below it is the version to say out loud.

Reach for SQL when…

  • Data is highly relational and you query it many different ways
  • Correctness is non-negotiable — payments, ledgers, orders, inventory
  • You need multi-row ACID transactions and strong consistency
  • The schema is well-understood and stable
  • Current scale fits one primary plus read replicas

Reach for NoSQL when…

  • The access pattern is simple and known — lookups by a key
  • You need massive horizontal scale or very high write throughput
  • The schema is flexible or evolving (catalogues, profiles, events)
  • Brief staleness is acceptable for the data in question
  • Data is naturally denormalized or self-contained per record

The polyglot reality

Real super-apps don't choose one — they use polyglot persistence, the right store per workload. A single Grab request might touch PostgreSQL for the payment ledger, Redis for the driver's session and surge cache, Cassandra for the stream of location pings, and a graph store for matching heuristics. The interview-grade insight is that "which database" is the wrong framing; "which database for this access pattern" is the right one. Name the workload, name the store, justify with the access pattern.

The follow-up that's always coming

Whatever store you pick, the interviewer's reflex question is "what's your shard key / partition key?" Have an answer ready: it should spread load evenly (avoid hot keys) and match your most common query so that query hits a single partition. Picking user_id when you mostly query by user is good; picking country when 80% of users are in one country is a hot-shard disaster.

06 The engine half

What an index really is

We've chosen a store. Now the harder, more impressive half of the topic: why is a lookup fast, and what makes one store read-fast versus write-fast? The answer is the index, and it's the same idea in SQL and NoSQL.

Without an index, finding the rows where email = 'tan@example.sg' means a full table scan: read every row and check. On a 50-million-row users table that's tens of millions of reads for one answer — hopelessly slow for a login endpoint. An index is a separate, sorted data structure that maps a column's values to the locations of the matching rows, so the engine can jump straight to the answer instead of scanning.

The mental model: an index is the alphabetical index at the back of a textbook. To find every mention of "Cassandra" you don't read all 600 pages — you flip to the index, find the word (it's sorted, so that's fast), and it tells you the page numbers. The table is the book; the index is the back-matter that points into it. Crucially, the index is extra — it's maintained alongside the data, which is the cost we'll quantify in step 10.

Two structures dominate how indexes are built, and they sit at opposite ends of a trade-off. The B-tree optimizes for reads and is the relational default. The LSM-tree optimizes for writes and powers Cassandra, RocksDB, and friends. Understanding both — and the single trade-off that separates them — is the heart of this lesson.

07 Read-optimized

B-tree: the read-optimized default

A B-tree (more precisely a B+tree, the variant real databases use) is a balanced tree that keeps keys sorted and guarantees every lookup touches the same shallow number of levels. It's the index structure behind PostgreSQL, MySQL's InnoDB, Oracle, and SQL Server — essentially the relational world.

How it works under the hood

The tree is built from fixed-size pages (commonly 4 KB or 8 KB), each holding a sorted batch of keys. To find a key you start at the root page, which divides the whole key range into sub-ranges and points to child pages; you pick the child whose range contains your key, descend, and repeat until you reach a leaf page holding the key and a pointer to the row (or the row itself). Because each page fans out to hundreds of children, the tree stays extremely shallow: a B-tree over a billion keys is typically only three or four levels deep. That means a point lookup is three or four page reads — effectively constant time no matter how big the table gets.

Why it's great for ranges and sorts

Because the leaf pages are kept in sorted order and linked together, a B-tree shines at range scans and ordered queries — "all orders between two dates," or "next 20 listings sorted by price." You seek to the start of the range once, then walk the linked leaves sequentially. An ORDER BY on an indexed column comes nearly for free, since the data is already in order.

The cost: in-place random writes

The catch is on the write side. To insert a key, the engine must place it in the correct leaf page to preserve sort order — an in-place update at a specific spot on disk, which is a random write. If the target page is full it must split: allocate a new page, move half the keys over, and update the parent — occasionally cascading splits up the tree. So writes scatter across the disk and occasionally do extra page-rewriting work. On spinning disks this was brutal; on SSDs it's tolerable but still the structure's weak spot.

Under the hood

To make a B-tree crash-safe, engines write changes to a write-ahead log first, then update the pages. So a single logical insert can mean: append to the WAL, modify a leaf page, and (on a split) modify a parent page — several physical writes for one logical write. Hold that thought; it's the seed of "write amplification" in step 09.

08 Write-optimized

LSM-tree: trade read work for write speed

What if your workload is write-dominated — a firehose of GrabPay transaction events or Cassandra location pings, where you ingest far more than you read back? The B-tree's random in-place writes become the bottleneck. The Log-Structured Merge-tree (LSM-tree) is the answer, and it powers Cassandra, RocksDB, LevelDB, and the storage layer of many modern stores.

How it works under the hood

The core insight: sequential writes are dramatically faster than random writes, so never write randomly. Instead:

  • Incoming writes first land in an in-memory sorted structure called the memtable (and are appended to a small on-disk commit log for durability). This makes a write a cheap memory operation — that's the speed.
  • When the memtable fills, it's flushed to disk in one big sequential write as an immutable, sorted file called an SSTable (Sorted String Table). It's never modified afterward.
  • Over time, many SSTables accumulate. A background process called compaction merges them — combining sorted files, dropping superseded versions and deletions — to keep read costs and disk usage in check.

Updates and deletes never overwrite. An update is just a newer write of the same key in a later SSTable; a delete writes a tombstone marker. The newest version always wins, and compaction eventually garbage-collects the old ones.

The cost: reads must check several places

The price of cheap writes is paid on reads. To read a key, the engine must check the memtable first, then potentially several SSTables (newest to oldest), because the key could live in any of them. To keep this from being slow, each SSTable carries a Bloom filter — a tiny probabilistic structure that answers "is this key definitely not here?" so most SSTables are skipped without a disk read. A sparse index per SSTable then narrows the search within the ones that might match. Even so, a read can touch multiple files, which is why LSM trades read effort for write throughput.

The follow-up to rehearse

Expect: "What happens during compaction, and what's the downside?" Compaction is essential but consumes CPU, disk I/O, and bandwidth in the background, and a poorly-timed compaction storm can spike read latency. Also flag the tombstone trap: in Cassandra, a workload that deletes a lot can leave the store wading through tombstones until compaction clears them, hurting read latency — a real, frequently-asked failure mode.

09 The core trade-off

Read vs write amplification

Here is the single idea that ties B-tree and LSM-tree together, and the one to state crisply in an interview. Every storage engine pays a tax measured in two currencies:

  • Read amplification — how many physical reads (pages, files) you must do to answer one logical read.
  • Write amplification — how many physical bytes you actually write to disk for one logical write (counting WAL, page rewrites, and compaction re-writes).

You cannot minimize both at once; the two structures pick opposite corners. The B-tree minimizes read amplification (a lookup is a handful of page reads) at the cost of higher write amplification (in-place writes plus the WAL, plus page splits). The LSM-tree minimizes write amplification at ingest (writes are buffered and flushed sequentially) at the cost of higher read amplification (a read may scan memtable plus several SSTables) — though note that compaction quietly re-writes data, so LSM has its own write-amplification over the data's lifetime, just deferred and sequential.

B-tree (read-optimized)

  • Fast point reads and range scans — data is sorted in place
  • Predictable reads: ~3–4 page hits regardless of size
  • Writes are random, in-place; page splits add work
  • Default for relational stores: PostgreSQL, MySQL/InnoDB
  • Best when reads dominate or you need ordered queries

LSM-tree (write-optimized)

  • Very high write throughput — sequential flushes, no random I/O
  • Great for write-heavy, append-style workloads
  • Reads may touch many SSTables; Bloom filters mitigate
  • Background compaction costs CPU and I/O; tombstones can bite
  • Used by Cassandra, RocksDB, LevelDB, ScyllaDB
10 The hidden tax

What an index actually costs

Indexes feel free because they make the slow query fast and you see the win immediately. The cost is invisible and arrives on the write path. Here's the rule to internalize: an index speeds reads but slows every write, because every insert, update, or delete must also update every index on that table.

The worked cost

Suppose the Shopee orders table has five indexes — on buyer_id, seller_id, status, created_at, and a composite one. A single INSERT of one order now does six writes: one to the table and one to each of the five index structures (each potentially triggering a B-tree page split). At a peak of, say, 8,000 orders per second during a flash sale, that's roughly 48,000 physical writes per second — 8,000 to the table plus 40,000 across the five indexes — all generated by 8,000 logical inserts. Add the storage: each index is a full extra copy of its column(s) plus row pointers, so heavily-indexed tables can hold more bytes in indexes than in the data itself.

So indexing is a deliberate trade, not a default-on optimization. The discipline:

  • Index only what you query. An index that no query uses is pure write-and-storage tax with zero benefit. Audit and drop unused indexes.
  • Composite-index column order matters. An index on (seller_id, created_at) serves "this seller's orders, newest first" perfectly and also serves "this seller's orders" (the leftmost prefix). It does not efficiently serve a query on created_at alone — you can't skip the leading column. Order the columns by how you filter.
  • Use a covering index when it pays. A covering index includes every column a query needs, so the engine answers entirely from the index and never touches the table — an "index-only scan." It costs more storage but eliminates the second lookup.
The follow-up to expect

"This table is write-heavy and has eight indexes — what would you do?" The strong answer: question every index against an actual query, drop the unused ones, consider whether a write-optimized engine (LSM) fits the workload better than a B-tree, and remember that on a write-heavy table each extra index is a recurring tax paid on every single insert — not a one-time setup cost.

11 The scale move

Denormalize vs join

The last bottleneck connects indexing back to the SQL/NoSQL choice. We saw that joins are SQL's superpower — but joins across huge tables, especially across shards, are expensive: the engine reads from multiple places and combines on the fly, for every read. When a read happens millions of times per second, paying that join cost each time is wasteful.

The move: precompute and duplicate

Denormalization is the deliberate opposite of normalization: instead of storing each fact once and joining at read time, you duplicate the data into the shape your read needs, so the read becomes a single lookup with no join. You trade write-time work and storage for read-time speed.

Concrete example. To render a Shopee order-history screen, the normalized way joins orders to users, products, and sellers every time the screen loads. The denormalized way stores a ready-made order_view document that already embeds the buyer's name, each product's title and thumbnail, and the seller's shop name — so loading the screen is one GET by key. This is exactly why document and wide-column stores are popular for read-heavy product surfaces: the access pattern is "fetch this one assembled view," and that's a single-partition lookup.

The cost: keeping the copies in sync

Denormalization isn't free — you've now stored the buyer's name in many order_view documents. When the buyer renames their shop you must update every copy, or accept that they drift apart. That synchronization is the real cost, and it's where bugs live. Practical patterns to manage it:

  • Write-time fan-out: on a change, push updates to all derived copies (often asynchronously via a message queue), accepting brief eventual consistency.
  • Periodic rebuild: recompute the denormalized views in the background, tolerating staleness between rebuilds.
  • Pick fields that rarely change to duplicate; avoid duplicating volatile data unless the read win is worth the sync pain.

This is precisely the trade-off behind news-feed precomputation: rather than assembling each user's feed by joining "who I follow" with "their recent posts" on every page load (a read-time join), high-scale feeds precompute and fan out each post into followers' feed lists at write time, so opening the feed is a single read. Same principle, same trade — write-time work and duplicated storage bought you a cheap read. We'll build that fan-out in full in the news-feed lesson.

When to denormalize

The honest default is to start normalized in a relational store — it keeps data correct and is easy to reason about — and denormalize only where a measured, hot read path demands it. Premature denormalization spreads the same fact across the system before you know which reads matter, and then every write becomes a consistency hazard. Normalize for correctness; denormalize for a specific, proven read bottleneck.

★ Review

Summary & self-test

The whole topic compresses to two sentences. Choose the store from the access pattern: relational for relationships and correctness, NoSQL for scale, simple lookups, and flexible schemas. And underneath: every index trades write cost for read speed, B-trees and LSM-trees trade read-amp against write-amp, and at scale you duplicate data so the hot read is a single lookup. If you can defend a store choice and explain why a lookup is fast, you can carry the question.
