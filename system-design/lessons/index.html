How to use this course

Do L00–L01 first — framework and estimation are meta-skills every later answer reuses. With two years in, treat Phases 1–3 as a fast gap-fill, not a course. If you're time-boxed, run the ★ Core 12 end-to-end before touching anything else; they cover the question shapes a product/fintech loop actually throws.

The trap with ByteByteGo is that interviewers have read it too. For your top 3–4 systems, go past the book: write the real data model and API, name the failure modes, and answer "what breaks at 10×". Each lesson is built to do exactly that.

▸ The single biggest lever

The Mock Protocol

Coverage doesn't pass interviews — reps under time do. The interviewer scores how you drive the room. Turn each ★ Core drill into a timed 45-minute mock and self-grade it. This is what closes the gap between "I know news feed" and "I can run news feed cold."

45-minute time budget
0–5 min
RequirementsFunctional + non-functional; state your scope cut out loud.
5–10 min
EstimationQPS, read/write ratio, storage/yr, bandwidth, cache size.
10–15 min
API + data modelEndpoints/contract and the core tables or entities.
15–30 min
High-level designDraw the boxes and the request/data flow end to end.
30–40 min
Deep divePick the bottleneck (or take the interviewer's pick) and go deep.
40–45 min
Wrap-upFailure modes, trade-offs, and what you'd change at 10×.
Self-grade rubric (1–5 each, after every mock)
  • Scoping — did you pin requirements and cut scope before designing?
  • Estimation sanity — are the QPS/storage numbers in the right order of magnitude?
  • Clear HLD — could someone rebuild your diagram from your words alone?
  • Justified data choices — store, schema, shard key, and consistency level each defended by the access pattern.
  • Bottleneck & failure modes — did you name what breaks and how it recovers (failover, retries, idempotency)?
  • Communication & trade-offs — did you surface alternatives instead of asserting one answer?

Score below 3 on any line → that's your next rep. Re-run the same drill until every line clears 4.

Before any system: how to run the 45 minutes. Pure technique — the scaffolding every score sheet starts from.

L00
The 4-Step Interview Framework

Drive every round the same way: scope functional + non-functional requirements → high-level design → deep dive → wrap-up. The structure is half the score and stops you designing a database before you know the read/write ratio.

Scale from Zero to Millions of Users

The master narrative: single box → split the database → load balancer → read replicas → cache → CDN → stateless app tier → multi-DC → message queue → sharding. Every later lesson is a deep dive on one move.

Database Replication

Leader–follower topology, synchronous vs asynchronous replication, replication lag and read-your-writes, and what happens on failover when the leader dies.

SQL vs NoSQL & Indexing

Choose a store from the access pattern, not the hype. B-tree vs LSM-tree engines, what an index really costs on writes, and when denormalisation beats a join.

Caching Strategies

Cache-aside, read-through, write-through, write-back; LRU vs LFU eviction; and the parts interviewers probe — invalidation, cache stampede, and hot keys.

CDN & Content Delivery

Push vs pull edge models, TTL and Cache-Control, and serving static assets and media close to the user to cut latency and origin load.

API & Gateway Design

REST vs gRPC, an API gateway / BFF in front of services, versioning and pagination, auth (OAuth2 / JWT), and idempotency keys so a retried write never double-charges — a constant probe in fintech rounds.

Microservices, Service Discovery & Mesh

When to split a monolith, how services find each other (registry / DNS), and what a service mesh handles — retries, mTLS, traffic shifting — versus your app code. Grab and Sea run thousands of services; expect this.

P2 Phase Two

Data at Scale & Consistency

When one database is no longer enough. Distribution forces hard trade-offs — the topics that separate a junior answer from a senior one. L14 covers SG's data-residency reality.

L10
Partitioning & Sharding

Range vs hash vs directory partitioning, the hot-shard problem and picking a shard key, and the operational pain of resharding a live, growing system.

Hardopen ▸
L11
Consistent Hashing

Why naive hash % N reshuffles everything on scale-out, and how a hash ring with virtual nodes moves only ~K/N keys when a node joins or leaves.

MediumBBG V1·C5open ▸
L12
CAP, PACELC & Consistency Models

What you actually surrender under a partition; strong vs eventual vs causal consistency; and how quorum reads/writes (W + R > N) let you dial the trade-off per request.

Hardopen ▸
L13
Leaderless Replication & Conflict Resolution

Dynamo-style quorums, read-repair and anti-entropy, and resolving concurrent writes with version vectors instead of a single source of truth.

Hardopen ▸
L14
Multi-Region & Data Residency

Active-active vs active-passive across regions, geo-routing, cross-region replication lag and conflict handling, and keeping data in-country for PDPA / SEA residency rules — a real constraint for SG products.

P3 Phase Three

Asynchronicity, Transactions & Reliability

Decoupling in time, and keeping data correct across services. L16 (saga) is the backbone of every payment and order flow — promoted to its own lesson for a fintech target.

L15
Message Queues & Async Processing

Decouple producers from consumers; at-least-once vs at-most-once vs exactly-once delivery; idempotency, retries with backoff, and dead-letter queues.

Mediumopen ▸
L16
Distributed Transactions & the Saga Pattern

Why two-phase commit stalls, the saga pattern (choreography vs orchestration), the transactional outbox for reliable events, and idempotency keys. The correctness backbone behind payments, wallets, and orders.

Beyond BBGHardSG · fintechopen ▸

Design a Distributed Message Queue

Design a Rate Limiter

Token bucket, leaking bucket, fixed window, sliding window log/counter — the algorithm choice is the whole interview. Then make it distributed with an atomic counter in Redis.
