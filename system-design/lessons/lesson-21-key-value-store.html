21Lesson · System Design Interview

Design a Key-Value Store

This is the fundamentals capstone. A single drill that bolts together everything from the earlier lessons — consistent hashing, replication, quorum consistency, version-based conflict resolution, and Merkle-tree anti-entropy — into one coherent box. What you'll build is, in effect, a miniature Dynamo: a store that survives nodes dying and networks splitting without ever going dark.

About these notes: original study material on building a distributed key-value store — a standard, widely-documented engineering topic. The explanations, examples, and diagrams are my own; nothing is copied from any paid course or book. Two diagrams are interactive — press Play (or step with Next) to watch the data flow.
00 The problem

Why a key-value store?

A key-value store is the simplest possible database: every record is just a unique key mapped to an opaque value, and the whole interface is two calls — put(key, value) to write and get(key) to read. There are no joins, no secondary indexes, no schema to migrate. That radical simplicity is exactly what lets it scale to volumes a relational database can never reach.

You meet these stores everywhere in a super-app stack. GrabPay holds a user's wallet-balance snapshot keyed by user_id. Shopee caches a product's price-and-stock blob keyed by sku_id. A ride-hailing service stores each driver's last-known location keyed by driver_id. Agoda keeps a session's search filters keyed by session_token. None of these need SQL; they need a single value, fetched in single-digit milliseconds, that never goes offline.

The interview question is not "can you write a hash map" — anyone can. The question is: what happens when one machine isn't enough, and machines start dying? A hash map on one box dies with the box. So this lesson is really about distributing that hash map across hundreds of machines while keeping it fast, durable, and always available. Every section that follows exists to repair a hole the previous one opened.

01 Scope it

Clarify the requirements

Open by narrowing the problem out loud. A key-value store can be a single-box cache or a globally-replicated database, and the design diverges wildly depending on the answers. Distinguish the functional requirements (what it does) from the non-functional ones (the properties it must hold under load and failure).

Functional requirements

  • get(key) — return the value for a key, or "not found."
  • put(key, value) — write or overwrite a value for a key.
  • Keys are short strings; values are opaque blobs we treat as bytes — say up to ~10 KB. We do not interpret value contents (no querying inside them).

Non-functional requirements

This is where the real design lives. State each as a target, because each one forces a structural decision later:

  • Horizontally scalable — data must outgrow any single machine, so we add capacity by adding nodes, not by buying a bigger box.
  • Highly available — a GrabPay wallet read cannot fail just because one server rebooted. Target "always writable, always readable," even during a partition.
  • Tunable consistency — the operator should be able to trade freshness for speed per workload, rather than being locked into one global setting.
  • Low latency — single-digit-millisecond reads and writes at the 99th percentile.
  • Large data — petabyte-class total, far beyond one disk.

The CAP fork in the road

The moment you say "highly available even during a network partition," you have made the central decision. The CAP theorem says that when the network splits, a system can preserve either consistency (every read sees the latest write) or availability (every request gets an answer) — not both. We choose availability. That makes this an AP, eventually-consistent design: nodes always answer, and any divergence between replicas is reconciled shortly after. If the question were "design a bank ledger," you would pick CP instead and the rest of the lesson would look very different.

Interview tip

Naming the CAP trade-off early — and justifying it from the use case — is the single highest-leverage sentence in this interview. Say it plainly: "a wallet-balance cache must always answer, so I'll build an AP store and reconcile replicas afterward; if it were the authoritative ledger I'd flip to CP." Everything downstream (quorums, versioning, anti-entropy) follows from that one choice.

02 Estimate

Back-of-the-envelope

Before drawing boxes, size the system so the numbers justify them. Take a concrete workload: a regional super-app storing user wallet and session snapshots.

Storage

  • Assume 200 million users, each with a key whose value averages 2 KB.
  • Raw data: 200M × 2 KB ≈ 400 GB. Small — but that is the logical size.
  • We will replicate every key N = 3 times for durability, so physical data ≈ 1.2 TB.
  • Add room for the commit log, SSTable overhead, and headroom: budget roughly 3–4 TB of raw disk, spread across the cluster.

Throughput

  • Suppose 5 million daily active users generate, on average, 20 key-value operations each per day → 100M ops/day.
  • 100M ÷ 86,400 s ≈ ~1,160 ops/sec average. Peak is spikier — say 10× at lunch-time order rush → ~12,000 ops/sec.
  • Reads dominate writes here, roughly 5:1 (lots of balance checks, fewer top-ups), so ~10,000 reads/sec and ~2,000 writes/sec at peak.

What the numbers tell us

One modern server could almost hold 1.2 TB and serve 12K ops/sec. So why distribute? Because the requirements — not the raw size — force it. A single node cannot be highly available (it has a single failure domain) and cannot grow smoothly when the next region launches and the dataset 10×'s overnight. We distribute for availability and elastic growth, and the storage math just confirms a handful of nodes is plenty to start. Sizing also tells us replication factor 3 roughly triples disk — a cost worth naming aloud.

Why estimate at all

The estimate is a sanity check, not the design. Its real value is exposing the binding constraint. Here the binding constraint is availability and growth, not bytes — which is why the rest of the lesson spends its time on replication and reconciliation rather than on storage capacity.

03 Contract

API & data model

Pin down the contract before the architecture, so the diagrams have something concrete to serve. The external API is deliberately tiny.

# Write a value. Returns the version that was stored.
PUT  /v1/kv/{key}
  body: { "value": <bytes> }
  # client echoes back the version vector it last read,
  # so the server can detect concurrent writes (see step 09)
  header: X-Version-Vector: "A:3,B:1"
  -> 200 { "version": "A:4,B:1" }

# Read a value. Returns value(s) + the version vector.
GET  /v1/kv/{key}
  -> 200 { "value": <bytes>, "version": "A:4,B:1" }
  -> 200 { "siblings": [ ... ], "version": "A:4,B:2" }  # conflict!
  -> 404 if absent

# Tunable knobs the caller may set per request:
  header: X-Consistency: "quorum" | "one" | "all"

Two design choices in that contract are worth defending in the room. First, the value is opaque bytes — the store never parses it, which is what keeps the engine generic and fast. Second, the API surfaces a version vector (covered in step 09) on both read and write; this is how an AP store admits, rather than hides, the possibility of conflicting concurrent writes.

The on-node data model

Internally each node is not storing a plain map. Each key maps to a small record carrying the value plus the metadata needed to reconcile replicas:

Record {
  key:        string        // e.g. "wallet:u_88421"
  value:      bytes         // the opaque blob, ~2 KB
  version:    VersionVector // {"A":4,"B":1} — who wrote, how often
  timestamp:  int64         // last-write-wins tiebreaker (ms)
  tombstone:  bool          // true = logically deleted
}

The tombstone deserves a note: in a distributed store you cannot simply erase a deleted key, because a replica that was offline during the delete would later "resurrect" the value via anti-entropy. Instead a delete writes a tombstone marker that propagates like any other write, and is garbage-collected only after enough time has passed that every replica has surely seen it.

04 Partitioning

Partition the ring

First bottleneck: 1.2 TB and 12K ops/sec must be split across many nodes, and we need a rule for which node owns which key. The naive rule — node = hash(key) % N — is a trap. The instant you add or remove a node, N changes and almost every key remaps to a different node, triggering a near-total data reshuffle. At our scale that is hours of downtime.

Consistent hashing, recapped

The fix is consistent hashing (built in Lesson 11). Imagine a circular hash space — a ring — running from 0 up to a large maximum (say 2128) and wrapping back to 0. Hash each node's id onto a position on the ring, and hash each key onto the ring too. A key is owned by the first node found walking clockwise from the key's position. That owner is the key's coordinator for storage purposes.

The magic property: when a node joins or leaves, only the keys in the arc it is responsible for move — roughly K/N keys for K keys and N nodes — not the whole dataset. Adding a 7th node to a 6-node ring relocates about 1/7 of the keys, and only between two neighbors. Everything else stays put.

Virtual nodes fix the lumpiness

Placing one point per physical node makes the ring lumpy: by chance, one node may own a huge arc and get overloaded while another owns a sliver. The standard remedy is virtual nodes (vnodes). Each physical node is hashed onto the ring many times — say 128 or 256 tokens — so its responsibility is the sum of many small arcs scattered around the circle. The law of large numbers smooths the load, and when a node dies its arcs are absorbed by many different neighbors rather than dumped onto one. Real systems such as Cassandra (the num_tokens setting) and Amazon DynamoDB / Riak use exactly this.

Follow-up the interviewer will ask

"How does a new node learn the ring, and how do others learn about it?" There is no central registrar in an AP design — that would be a single point of failure. Nodes exchange membership and token assignments through a gossip protocol: each node periodically picks a random peer and swaps its view of who is alive and what they own. Within a few rounds the whole cluster converges on the same ring. We return to gossip in step 12.

05 Replication

Replicate for durability

Next bottleneck: if each key lives on exactly one node, that node is a single point of failure. When it dies — and at petabyte scale a disk dies every week — its keys are gone or unreadable. So no key may live on only one machine.

The preference list

The fix layers directly onto the ring. Instead of storing a key only on its owner, store it on the owner plus the next N−1 nodes walking clockwise. That ordered set of N nodes is the key's preference list (also called its replica set). With N = 3, a key landing on node B is replicated on B, C, and D. Any read or write can be served by any node on the list, so losing one or even two replicas does not lose the data.

One subtlety: when virtual nodes are in play, the preference list is built from distinct physical nodes. As you walk clockwise you skip vnodes that map back to a physical node already on the list, so the three replicas are genuinely three different machines — otherwise a single machine failure could take out two "copies."

Rack and region awareness

Three replicas in the same rack all die when the rack's top-of-rack switch fails. Production systems therefore make the preference list topology-aware: place the N replicas across distinct racks, and ideally distinct availability zones. For a Singapore deployment you might pin replicas across three AZs in ap-southeast-1 so an entire datacenter can burn down and the wallet store keeps answering. Cassandra encodes this with its NetworkTopologyStrategy and a per-datacenter replication factor.

Higher N (e.g. 5)

  • More copies → survives more simultaneous failures
  • More read/write fan-out to reach a quorum
  • More disk and cross-node bandwidth

Lower N (e.g. 3)

  • Cheaper storage and less coordination traffic
  • The common default — survives 1–2 node losses
  • Less margin during a multi-node or AZ outage
06 Consistency

Quorum consistency

Replication created a new bottleneck of its own: with N copies of a key, how many must we touch on each operation? Write to all N and one slow or dead replica stalls every write. Read from one and you might read a stale copy that missed the latest write. We want a tunable middle ground.

The N / W / R quorum

The answer is the quorum (the core of Lesson 12), governed by three numbers:

  • N — replicas per key (the preference-list size).
  • W — the write quorum: a write is acknowledged to the client once W replicas confirm it.
  • R — the read quorum: a read gathers responses from R replicas and returns the freshest.

The whole scheme hinges on one inequality:

W + R > N  ⟹  strong consistency guarantee

Why does that work? If the write set (any W replicas) and the read set (any R replicas) must together exceed N, then by the pigeonhole principle the two sets always overlap by at least one replica. That overlapping replica has the latest write, so a read is guaranteed to see it. With N = 3, choosing W = 2 and R = 2 gives 2 + 2 = 4 > 3 — every read overlaps every recent write.

Tuning for the workload

The point of exposing W and R is that you tune them per use case:

  • W=1, R=1 (sum 2, not > N): fastest possible, but reads can be stale. Good for a non-critical cache like recent search terms.
  • W=2, R=2: the balanced default — strong-ish consistency with one replica of slack on each side. This is what you would pick for a GrabPay balance snapshot.
  • W=3, R=1: reads are blazing and always fresh, but every write needs all three replicas up — slow, fragile writes. Suits write-rarely, read-constantly data.
  • W=1, R=3: writes are instant and durable-enough, reads do the work of finding the latest. Good when writes must never block.
Follow-up: "Set N/W/R for high availability"

For maximum availability you want both reads and writes to succeed even when replicas are down, so keep W and R small — e.g. N=3, W=1, R=1. Now a single live replica can serve any operation, but you have given up the overlap guarantee, so reads may be stale and you lean on read-repair and anti-entropy (steps 08 and 10) to converge afterward. The honest one-liner: "smaller W and R buy availability and latency; their sum exceeding N buys consistency — you cannot maximize all three at once."

07 Write path

The write path through a coordinator

Now wire the pieces into an end-to-end write. A client may contact any node — there is no master. The node it hits becomes the coordinator for that request: it hashes the key to the ring, finds the preference list, forwards the write to those N replicas, and waits for W acknowledgements before replying success to the client. The remaining replicas are written too, just not waited on.

Walk the interactive figure below. A put("wallet:u_88421", …) arrives, the coordinator routes it to replicas B, C, D, and with W = 2 it answers the client the moment B and C confirm — D's slower disk does not hold up the user.

The commit log makes the ack safe

When a replica "confirms," it does not mean the value is sorted into its main data file. It means the replica has appended the write to a durable, append-only commit log (write-ahead log) on disk and updated its in-memory table. If the replica crashes a millisecond later, it replays the commit log on restart and recovers the write. We unpack that machinery in step 11 — but it is why a W = 2 ack is a real durability promise, not just a memory poke.

Follow-up: "What if the coordinator dies mid-write?"

If it dies before sending to replicas, the client times out and retries against another node — at-least-once semantics, and the version vector lets duplicates de-duplicate. If it dies after some replicas got the write but before W acks, the client retries; the write may already be partially landed, but because writes are idempotent under their version, replaying is safe and anti-entropy heals the rest. The system never has a "the one machine that knew about this write is gone forever" moment — that is the entire point of N replicas.

08 Read path

Read & read-repair

Reads mirror writes. The coordinator asks R replicas for their copy, compares the versions they return, and serves the freshest. But this is where the AP design's chickens come home to roost: because writes only waited for W acks, a lagging replica may answer with a stale value. The read must detect and, ideally, fix that on the spot.

How read-repair works

Read-repair is the cheap, opportunistic reconciliation that happens during a normal read. The mechanism, step by step:

  • The coordinator collects values and versions from R replicas.
  • It compares version vectors. If they all agree, done — return the value.
  • If one replica's version is strictly older (it is missing a write the others have), the coordinator returns the newest value to the client and writes that newest value back to the stale replica in the background.
  • The next read of that key will find all replicas in agreement.

The interactive figure traces it: a read for the wallet key gathers C and D, notices D is one version behind, serves C's fresh value, and quietly pushes the fix to D.

Read-repair is elegant but only fixes keys that happen to be read. A key that is written, then never read for a month, can stay divergent on a replica the whole time. Hot keys self-heal; cold keys do not — which is precisely the gap that anti-entropy (step 10) closes.

09 Versioning

Versioning & conflict resolution

The deepest bottleneck in any AP store: two clients write the same key concurrently on different replicas. With no single master to serialize them, both writes are "valid." A wall-clock timestamp cannot reliably decide which came first — clocks drift between machines, and the later write may carry stale data. We need a way to tell "B happened after A" apart from "B and A happened concurrently."

Version vectors

The tool is the version vector (a vector clock, from Lesson 13). Each value carries a small map of {node_id → counter} recording how many times each coordinating node has updated it. When a node coordinates a write, it increments its own entry. Comparing two version vectors yields one of three verdicts:

  • A descends from B (every counter in B ≤ the matching counter in A, and A is larger somewhere) → A is strictly newer; keep A, discard B.
  • B descends from A → the mirror case; keep B.
  • Neither descends from the other → the writes are concurrent; they conflict and both must be preserved as siblings.

A worked GrabPay scenario

Suppose a user's wallet value is replicated on B and C with version {A:1}. The network partitions. The user's phone tops up S$10 via a coordinator that writes to B, producing {A:1, B:1}. Simultaneously a merchant refund of S$5 is processed via a coordinator that writes to C, producing {A:1, C:1}. When the partition heals, a read fetches both. Neither vector descends from the other — they are concurrent. The store cannot silently pick one without losing money, so it returns both siblings and lets the application decide.

Who resolves the conflict?

Two strategies, and the interviewer wants you to weigh them:

Last-write-wins (LWW)

  • Pick the value with the highest timestamp; discard the rest
  • Dead simple, no app logic, no siblings to manage
  • Silently loses data — fatal for a wallet (the refund vanishes)
  • Used by Cassandra by default; fine for caches and idempotent overwrites

Client-side merge

  • Store returns siblings; the app combines them by domain rules
  • No data lost — a shopping cart unions items; a wallet sums deltas
  • Pushes complexity onto every reader of that key
  • The DynamoDB / Riak "semantic reconciliation" approach
Follow-up: "Two concurrent writes to one key — what happens?"

The crisp answer: "version vectors let the store detect that the two writes are causally concurrent rather than one superseding the other. It cannot safely auto-pick a winner, so it keeps both as siblings and surfaces them on the next read. The application merges them with domain logic — sum the wallet deltas, union the cart — or, if loss is acceptable, the store falls back to last-write-wins by timestamp." The trap to avoid is claiming a timestamp alone resolves it; clock skew makes that lossy.

10 Failure handling

Failure handling & anti-entropy

So far we have handled the happy path and stale reads. Now the hard cases: a replica is down when a write arrives, or two replicas silently drift apart over time. AP means the system must keep accepting writes through both.

Hinted handoff for temporary failures

Suppose a write should go to B, C, D but D is rebooting. We refuse to block the write — that would violate availability. Instead the coordinator sends D's copy to a healthy substitute node, say E, tagged with a hint: "this really belongs to D; hold it." E stores it in a separate hinted area and keeps trying to deliver it. When D comes back, E hands the data off and deletes its hint. This is hinted handoff, and it is what lets writes succeed at full N-durability even while a replica is briefly offline. It handles transient failures — a reboot, a 30-second network blip.

Merkle trees for permanent reconciliation

Hinted handoff covers brief outages, but what about a replica that was down for hours, or two replicas that drifted apart because some hints were lost or read-repair never touched a cold key? Comparing two replicas' entire datasets key-by-key would move terabytes across the network — absurd. The efficient solution is anti-entropy via Merkle trees.

A Merkle tree is a tree of hashes. Each leaf hashes a small range of keys; each parent hashes its children; the single root hash summarizes the whole dataset. To compare two replicas, the nodes exchange just their root hashes. If the roots match, the replicas are identical and zero data moves. If they differ, the nodes descend the tree, comparing child hashes, and recurse only into the subtrees that disagree. The cost is proportional to the amount of divergence, not the size of the data — you walk down to exactly the leaf ranges that differ and sync only those keys.

Follow-up: "How do replicas reconcile after a partition?"

Lead with the layered answer: "Three mechanisms, increasingly heavyweight. During the partition, hinted handoff keeps writes durable on stand-ins. Right after, ordinary reads trigger read-repair on hot keys. For everything else — cold keys, long outages — a periodic Merkle-tree anti-entropy pass compares replicas and syncs only the diverged ranges. Together they drive the cluster back to convergence without ever shipping the whole dataset." Naming all three, in that order, is what separates a strong answer from a vague "they sync eventually."

11 Storage engine

The per-node storage engine

Zoom into a single node. Each one is itself a small, write-optimized database, and the interviewer often probes here to see if you understand what happens on disk. The dominant design — used by Cassandra, RocksDB, LevelDB, ScyllaDB, and HBase — is the LSM-tree (Log-Structured Merge-tree, from Lesson 05). It exists to solve one bottleneck: random disk writes are slow, but our store must absorb thousands of writes per second.

How a write flows through the LSM-tree

An LSM-tree turns random writes into sequential ones. The path of a single put:

  • 1. Commit log. The write is appended to an on-disk commit log (write-ahead log). Appends are sequential and therefore fast, and this is the durability anchor — if the node crashes, the log is replayed on restart.
  • 2. Memtable. The write also lands in the memtable, a sorted in-memory structure (a balanced tree or skip list). Reads and writes here are RAM-fast. Because we wrote the log first, losing the memtable to a crash is recoverable.
  • 3. Flush to SSTable. When the memtable fills past a threshold, it is flushed to disk as an immutable, sorted file called an SSTable (Sorted String Table) in one big sequential write. The memtable is then cleared and the corresponding commit-log segment can be discarded.
  • 4. Compaction. Over time many SSTables accumulate. A background compaction process merges them, discarding superseded values and tombstones, keeping read cost bounded.

How a read finds the value

A read checks the memtable first, then the SSTables newest-to-oldest. To avoid touching every file, each SSTable has a Bloom filter — a compact probabilistic structure that answers "this key is definitely not here" cheaply, so the reader skips files that cannot contain the key. A sparse index points near the key's offset within a file. This is the classic LSM trade: writes are cheap (sequential), reads pay a little more (possibly several files), and the Bloom filter keeps that "little more" small.

LSM-tree (write-optimized)

  • Sequential writes → very high write throughput
  • Cheap appends; compaction runs in the background
  • Reads may hit several SSTables (Bloom filters mitigate)
  • Best for write-heavy KV stores: Cassandra, RocksDB

B-tree (read-optimized)

  • Updates in place → fast point reads
  • Random writes and page splits are slower
  • Write amplification under heavy insert load
  • Best for read-heavy relational engines: PostgreSQL, MySQL InnoDB
Under the hood: why immutability helps

SSTables are never modified after they are written. That immutability is what makes the engine safe under concurrency and crashes: there is no in-place update to corrupt, no lock to hold during a flush, and a half-written SSTable from a crash is simply ignored on restart in favor of replaying the commit log. The cost — stale, superseded copies piling up — is paid lazily by compaction, off the hot path.

12 Scale & failure

Membership, scaling & what breaks at 10x

The final piece: how the cluster grows, shrinks, and keeps a shared view of itself — and where this design strains when traffic 10×'s.

Gossip-based membership

An AP store cannot lean on a central coordinator to track who is alive — that node would be a single point of failure and a CP-style bottleneck. Instead nodes run a gossip protocol: every second or so, each node picks a random peer and exchanges its view of the cluster — which nodes exist, their ring tokens, and a per-node heartbeat counter. Information spreads epidemically; a fact known to one node reaches the whole cluster in O(log N) rounds. Failure detection rides on the same channel: if a node's heartbeat stops advancing in everyone's gossip state, its peers mark it down and route around it.

Adding and removing nodes

Because of consistent hashing, scaling is cheap. When a new node joins, it claims a set of token ranges and only the keys in those ranges stream over from their current owners — roughly K/N keys, not the whole dataset. Neighbors hand off the relevant arcs; everything else is untouched, and the cluster keeps serving throughout. Removing a node is the reverse: its ranges are absorbed by the next nodes clockwise, which already held replicas of much of that data thanks to the preference list. This is the elastic-growth property we promised back in step 02.

What breaks at 10x

  • Hot keys / hot partitions. A single viral key — say a flash-sale SKU during a Shopee 12.12 campaign — funnels all its traffic to one preference list of N nodes, which consistent hashing alone cannot spread. Fixes: add a caching layer in front, or split the hot key (append a shard suffix) so it spans multiple preference lists.
  • Compaction storms. 10× the write rate means 10× the SSTables and far more background compaction, which competes with live traffic for disk I/O and can spike read latency. You tune compaction strategy (size-tiered vs leveled) and throttle it.
  • Anti-entropy cost. Bigger datasets make Merkle trees deeper and repair passes longer; you schedule them off-peak and bound their bandwidth.
  • Gossip overhead. Gossip traffic grows with cluster size; at thousands of nodes you partition the gossip groups or move to a more hierarchical membership scheme.
  • Wide rows / large values. If values creep past the ~10 KB assumption, single-node memory and network buffers strain; you cap value size and push large blobs to an object store like S3, keeping only a pointer in the KV store.
Real systems that are this design

You have essentially re-derived Amazon Dynamo — consistent hashing, preference lists, N/W/R quorums, vector clocks, hinted handoff, Merkle anti-entropy, and gossip. Apache Cassandra and Riak are open-source descendants; DynamoDB is the managed evolution (it leans more toward configurable strong consistency). Cassandra's knobs map one-to-one: replication_factor = N, consistency_level = the W/R you pick per query, num_tokens = virtual nodes.

★ Review

Summary & self-test

The whole design compresses to one sentence: spread keys around a hash ring, copy each onto the next N nodes, let the operator tune how many of those N a read and write must touch, detect concurrent writes with version vectors, and heal divergence with read-repair, hinted handoff, and Merkle anti-entropy. If you can draw the ring, the preference list, and the W + R > N inequality from memory and explain why each failure mechanism exists, you can carry this interview.
