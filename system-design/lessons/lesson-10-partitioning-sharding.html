10Lesson · System Design Interview

Partitioning & Sharding

A single database primary eventually hits a wall — too many writes, too much data to fit on one machine. Partitioning splits the data across many nodes so the system scales horizontally. The interview is almost entirely about two decisions: which shard key you pick, and how you survive a live reshard without taking the system down.

00 The bottleneck

The single-primary wall

You scaled reads first — that is the natural order. When read traffic grew, you added read replicas and pointed queries at them (that is Lesson 04). But a leader-follower setup has a structural ceiling: every write still goes through one primary, and every replica must replay that same write stream. Adding replicas multiplies read capacity; it does nothing for writes. So when an app becomes write-heavy — ledgers, order books, message stores, event logs — the single primary becomes the bottleneck.

Two separate limits collide here, and it helps to name them apart:

  • Write throughput. One machine has a finite number of CPU cores, a finite disk write bandwidth (IOPS), and a finite fsync rate for the write-ahead log. At GrabPay scale, a few hundred thousand payment writes per second simply will not commit through one node, no matter how large the instance.
  • Dataset size. Even if writes were slow, the data itself outgrows the box. A Shopee order history spanning years of transactions across Southeast Asia is tens of terabytes. Indexes that no longer fit in RAM turn every query into a disk seek, and a single volume caps out long before the logical dataset does.

You have already pulled the obvious lever — vertical scaling, buying a bigger machine. That buys time, not a future: the largest instance is still one instance, the price curve is brutally non-linear, and you eventually hit the biggest box the cloud sells. The structural fix is to stop putting all the data on one node. Partitioning (also called sharding) splits the dataset into independent pieces spread across many machines, so the system scales horizontally — you add capacity by adding nodes, not by enlarging one.

A word on vocabulary

People use partitioning and sharding loosely, and so will your interviewer. The common convention: partitioning is the general act of splitting a dataset into pieces (a partition can even live on the same node); sharding specifically means partitioning across separate machines for horizontal scale. Each piece is a shard. We focus on sharding throughout.

01 Two axes to cut

Vertical vs horizontal split

There are two fundamentally different ways to cut a table apart, and they solve different problems. Picture a table as a grid: rows are records, columns are fields. You can slice it vertically (by column) or horizontally (by row).

Vertical partitioning — split by column or table

Vertical partitioning moves different columns or whole tables onto different stores. A GrabPay users table might carry a hot, tiny set of fields read on every request — user_id, display_name, kyc_status — alongside a cold, fat blob: the scanned ID document, the full address history, a base64 selfie. Keeping the fat columns in the same row bloats every page the database reads, so hot queries drag cold bytes off disk for nothing. Vertical partitioning peels the cold columns into a separate table or store (often blob storage like S3, with only a pointer left behind), so the hot path stays lean.

The classic real-world version is decomposing a monolithic schema along service boundaries: the payments tables move to a payments database, messaging tables to a messaging database. This is how a monolith becomes microservices at the data layer. The catch: a query that once joined across those tables now spans two stores and cannot use a SQL JOIN — you stitch the data in the application. Vertical partitioning relieves the bloat problem but not the fundamental one: if a single table's rows are too many for one machine, splitting its columns does not help. For that you need the other axis.

Horizontal partitioning — split by row (this is sharding)

Horizontal partitioning, the proper meaning of sharding, keeps the schema identical but splits the rows across nodes. Shard 1 holds users A–H, shard 2 holds I–P, shard 3 holds Q–Z — same columns everywhere, different rows. Now both limits are relieved at once: each shard owns a fraction of the writes and a fraction of the bytes. Ten shards means roughly one-tenth the write load and one-tenth the data per machine, and you scale by adding an eleventh.

Vertical partitioning

  • Separates hot from cold fields; keeps the hot path small
  • Natural fit for decomposing a monolith into per-service DBs
  • Does not help when one table has too many rows
  • Breaks SQL joins across the split; app must stitch

Horizontal partitioning (sharding)

  • Relieves both write throughput and dataset size at once
  • Scales out: add a node to add capacity
  • Introduces a routing problem and cross-shard query pain
  • The shard key choice now dominates everything
Interview tip

When asked "how would you scale the database," do not jump straight to sharding. Walk the ladder out loud: add read replicas → vertical scaling for the primary → vertical partitioning to peel off cold data → and only then horizontal sharding. Sharding is the most powerful and the most painful, so reach for it last.

02 How to assign rows

Sharding strategies

Once you decide to shard horizontally, the immediate question is: given a row, which shard owns it? The function from a key to a shard is the heart of the system. There are three classic answers, and an interviewer wants you to compare them honestly.

Range-based sharding

Range-based sharding assigns a contiguous range of key values to each shard: users A–H on shard 1, I–P on shard 2, Q–Z on shard 3; or orders from Jan–Mar on one shard, Apr–Jun on the next. Under the hood the router keeps a small sorted table of range boundaries and does a binary search to find the owning shard. The big win is range scans: "all Agoda bookings between two dates" lands on one or a few adjacent shards, so the query is cheap and local. The big risk is imbalance. Ranges are rarely uniform — if you shard orders by date, the newest shard takes all the write traffic while older shards sit idle, and any range with a popular prefix (every user whose ID starts with a common value) becomes a hot range.

Hash-based sharding

Hash-based sharding runs the key through a hash function and uses the result to pick a shard — classically shard = hash(key) % N. Because a good hash scatters keys uniformly, load spreads evenly across shards almost by definition, which kills the imbalance problem. The cost is the inverse of range sharding: hashing destroys order. Two consecutive order IDs land on different shards, so "all orders in March" becomes a scatter-gather across every shard — there is no locality to exploit. Hashing is the default when your access is point lookups by key (give me this user, give me this wallet) and you do not need range scans.

Directory-based (lookup) sharding

Directory-based sharding adds a level of indirection: a lookup service holds an explicit mapping from key (or key-range) to shard, and you consult it to route. Because the mapping is data rather than a formula, it is the most flexible — you can move one noisy tenant to a dedicated shard, split a hot range, or migrate keys, all by editing the directory, without rehashing the world. The price is an extra network hop on the lookup and, more seriously, a single point of failure: if the directory is down or wrong, every query is mis-routed. You make it highly available (replicate it) and cache it aggressively on the clients. Many large systems converge on a directory-like coordinator precisely because operational flexibility beats elegance at scale.

Range vs Hash

  • Range: fast range scans, simple to grow at the high end — but prone to hot ranges and uneven load
  • Hash: near-uniform distribution and predictable per-shard load — but no efficient range queries, and naive % N is a nightmare to resize

Directory adds

  • Decouples placement from any formula — rebalance by editing data
  • Lets you isolate a hot tenant onto its own shard
  • Must be made HA and cached, or it is the bottleneck and the SPOF
Under the hood — who runs which

HBase and MongoDB (ranged sharding) use range partitioning with split/merge of ranges. Cassandra and DynamoDB hash the partition key onto a token ring (a refinement of hash sharding — that ring is consistent hashing, Lesson 11). MongoDB also offers hashed sharding via a hashed shard-key index. Vitess, the sharding layer behind YouTube's MySQL, keeps a directory-like map (the VSchema) that a routing proxy consults.

03 The one decision that matters

Choosing the shard key

The shard key (also called the partition key) is the column whose value decides which shard a row lands on. Almost every problem with a sharded system traces back to a bad shard key, and almost every interview probes it directly. A good shard key has three properties at once:

  • High cardinality. The key must have many distinct values so the data can spread thinly. Sharding a payments table by country_code (a handful of values) means you can never have more shards than countries, and Indonesia would dwarf the rest. Sharding by currency is worse still.
  • Even access distribution. Not just many values, but values that are touched evenly. A key can be high-cardinality and still hot if a few values get the lion's share of traffic (a celebrity's user_id, a flash-sale product_id). Even spread of load matters more than even spread of rows.
  • Alignment with the dominant query. Pick the key the most important queries already filter on, so those queries hit a single shard. If 90% of GrabPay reads are "show me this wallet's recent transactions," shard the ledger by wallet_id and every one of those reads is single-shard.

user_id vs timestamp vs composite

Three candidates illustrate the trade-off space. Sharding a transactions table by user_id gives high cardinality and good per-user locality — one user's whole history lives on one shard — but it cannot answer "all transactions in the last hour" without a scatter-gather, and a whale user can overload their shard. Sharding by timestamp makes time-range reports trivial but is the textbook anti-pattern for writes: all new writes target the single newest shard while the rest go cold — a self-inflicted hotspot. A composite key like (merchant_id, created_at) blends the two: it co-locates a merchant's data (good for the dominant "this merchant's sales today" query) while spreading writes across merchants. The art is matching the key to your real access pattern, not to what is convenient.

-- SeaMoney wallet ledger, sharded by wallet_id (point lookups dominate)
CREATE TABLE ledger_entry (
  wallet_id   BIGINT,      -- <-- shard key: high cardinality, even, matches query
  entry_id    BIGINT,
  amount      DECIMAL(18,2),
  currency    CHAR(3),
  created_at  TIMESTAMP,
  PRIMARY KEY (wallet_id, entry_id)
);
-- route: shard = hash(wallet_id) % N   ->  one wallet's entries stay together
Interview tip — the question you will be asked

"What is your shard key, and why?" is the single most common follow-up in this topic. Answer with all three properties in one breath: "I'd shard by wallet_id — it's high-cardinality so data spreads, access is roughly even because no single wallet dominates, and it matches our dominant query, per-wallet balance and history, so those reads stay single-shard. I'd accept that cross-user reporting becomes a scatter-gather and handle it offline." Naming what you give up is what makes the answer senior.

04 The write path

Routing the write

With a shard key chosen, follow a single write through the system. The application does not (and should not) know which physical shard owns a row. Instead a routing layer computes the shard from the key. For hash sharding the math is one line — shard = hash(key) % N — but the conceptual flow is the same for every strategy: extract the key, map it to a shard, talk only to that shard. Notice what the router does not do: it never touches shards 0 or 2 for this write. That single-shard property is the whole point — a sharded write is exactly as cheap as an unsharded one, because each shard is just a smaller database. The cost is entirely on the design side: the router must stay in lock-step with which shard owns which keys, and the moment that mapping changes (next section), life gets hard.

05 When one shard melts

The hot-shard problem

Even with a high-cardinality key, real-world access is rarely uniform — and an interviewer will press on this immediately. The hot shard (or celebrity / hotspot) problem is when a disproportionate share of traffic concentrates on one shard, so it saturates while its siblings idle. Sharding gave you N machines of capacity, but a hotspot means your effective ceiling is one machine again.

Three named scenarios cause it. First, the celebrity: shard a social graph by user_id and a creator with twenty million followers makes their shard a furnace on every post. Second, the flash sale: shard an inventory table by product_id and a single Shopee 11.11 doorbuster receives a million reads and decrements in minutes, all on one shard. Third, the monotonic key: shard by an auto-incrementing ID or by timestamp and every new write piles onto the newest shard while the rest go cold.

Mitigations

  • Compound (composite) keys. Append a second dimension to spread a hot value. Sharding a chat system by (channel_id, bucket) — where bucket is a number from 0–9 derived from the message id — splits one blazing channel across ten shards instead of one. You trade single-shard reads for a small, bounded scatter (read all ten buckets).
  • Splitting a hot shard. If one shard is hot because it simply holds too much, split its key range in two and move half elsewhere. Directory-based sharding makes this a directory edit; range sharding supports explicit split operations (HBase region splits, MongoDB chunk splits).
  • Salting. Prefix or suffix the key with a small random or hashed value — salt:user_id — so a single logical key fans out across many physical partitions. This is exactly how you defeat the monotonic-key write hotspot: a tiny salt scatters sequential writes. The cost, again, is that reading the logical key now means reading every salt bucket.

The thread running through all three: you defeat a hotspot by trading read locality for write spread. There is no free lunch — every fix that spreads a hot key across shards makes reading that key a multi-shard gather.

Interview tip

If asked "what if one user is a celebrity," do not just say "use consistent hashing" — that spreads shards evenly, not the load on a single key. The honest answer is: detect the hot key, then spread that key across partitions with a salt or compound key (accepting a fan-out read), and for read-heavy hotspots, put a cache in front so the hot shard is shielded. Hotspots are an access-pattern problem, not a hashing problem.

06 The cost of splitting

Cross-shard queries, joins & transactions

Sharding's bill comes due whenever an operation needs data from more than one shard. On a single database, a JOIN or a multi-row transaction is trivial. Across shards, the database can no longer help you — the rows live on different machines that know nothing of each other. Three things get hard, in rising order of pain.

Cross-shard reads — scatter-gather

A query that does not filter on the shard key cannot be routed to one shard, so it must scatter to all shards and gather the results. "Top 10 Agoda hotels by bookings this week," when sharded by hotel_id, fans out to every shard, each returns its local top 10, and the coordinator merges them. Scatter-gather works but scales poorly: latency is governed by the slowest shard (the tail-latency tax), and load multiplies — one logical query becomes N physical queries. The mitigation is to design these away: precompute the answer with an offline job and store it, or maintain a secondary index in a system built for it (Elasticsearch, a separate analytics store).

Cross-shard joins

A SQL JOIN across shards is not supported by the engine — you must do the join in the application (fetch from shard A, fetch from shard B, stitch in memory) or, far better, denormalize so the data you read together lives together. If you always read an order with its line items, store the line items in the order's shard, keyed by the same order_id, so the read is single-shard. This is why NoSQL stores push you toward denormalized, query-shaped data: the schema is bent so the common read never crosses a shard.

Cross-shard transactions

The hardest. A GrabPay transfer debiting wallet A on shard 1 and crediting wallet B on shard 2 needs both writes to commit atomically — but they are on different machines. A local BEGIN…COMMIT cannot span shards. Options, none free: two-phase commit (2PC) gives atomicity but is slow and blocks if the coordinator dies mid-commit, locking rows on every participant. The modern preference is a Saga — break the transfer into local transactions per shard plus compensating actions to undo on failure (debit A; if credit B fails, refund A) — which trades strict atomicity for availability and eventual consistency. The senior move in an interview is to design the transaction away: co-locate accounts that transact together on the same shard when you can, so the common case stays single-shard.

Two-phase commit (2PC)

  • Strong atomicity across shards — all commit or none
  • Synchronous, multiple round trips: high latency
  • Coordinator failure mid-commit leaves rows locked (blocking)
  • Reduces availability; rarely used on the hot path at scale

Saga (compensation)

  • Local commits per shard, no global lock — stays available
  • Failure handled by compensating actions (undo)
  • Only eventual consistency; intermediate states are visible
  • You must design and test every compensation path
What breaks at 10x

A scatter-gather that touches 4 shards at 1k QPS is fine. At 10x scale across 40 shards, the same query now hits 40 machines per request and your tail latency is set by the worst of 40 — a p99 that was 20ms becomes 200ms. This is why teams move reporting and search off the sharded OLTP store entirely and onto a purpose-built secondary index, fed asynchronously. Plan for it before it bites.

07 The operational nightmare

Resharding without downtime

Everything above assumed a fixed number of shards. The moment you need to add one — and you always eventually do — the naive scheme detonates. With shard = hash(key) % N, changing N from 4 to 5 changes the modulus for almost every key: a key that mapped to hash % 4 == 2 now maps to hash % 5 == something_else. In the worst case roughly (N−1)/N of all keys move — about 80% of the data reshuffles on a single scale-out. That is a multi-terabyte data migration triggered by adding one box, and during it the mapping is in flux.

This pain is exactly what consistent hashing (Lesson 11) was invented to fix: it arranges shards on a ring so that adding a node moves only the keys in one adjacent arc — roughly 1/N of the data, not (N−1)/N. Keep that as the structural answer to "how do you avoid reshuffling everything." But even with consistent hashing, the data still has to move while the system is live, and that operational dance is what the next interview question is really about.

Online resharding: double-write, backfill, cutover

The safe recipe for moving a key range to a new shard without downtime has four phases:

  • 1. Double-write. Start writing every affected key to both the old and the new shard. New writes are now safe on the destination; the old shard stays authoritative for reads.
  • 2. Backfill. A background job copies the existing (historical) rows for that range from old to new. Because double-writes are already flowing, you only need to backfill data written before double-writing began.
  • 3. Verify. Compare old and new for the range — row counts, checksums, sampled reads — until they agree. This catches bugs before they become data loss.
  • 4. Cutover. Flip reads to the new shard (a directory edit or routing config change), confirm health, then stop double-writing and reclaim space on the old shard.

The step-through shows the structural win: when a new shard is added under consistent hashing, only the keys in one arc move — most of the data does not budge.

Interview tip — the second-most-common question

"How do you reshard without downtime?" Land the four phases — double-write → backfill → verify → cutover — and mention the rollback story (if verify fails, you simply stop double-writing and never cut over, losing nothing). Bonus points for noting that consistent hashing or pre-splitting into many virtual shards upfront (say 1024 logical shards mapped onto 8 physical nodes) makes future growth a cheap remapping instead of a rehash.

08 Who computes the route

Where routing lives

Something has to turn a shard key into a shard address. There are three architectures for where that logic runs, and the choice is a recurring design-review question.

Smart client (routing in the app)

The application library knows the sharding map and connects directly to the owning shard — no middle hop, so it is the lowest latency. Cassandra's drivers work this way: they are token-aware and route straight to the right node. The downside is operational: every client must carry and refresh the topology, and changing the sharding scheme means redeploying every service that uses the database.

Proxy / router (routing in a middle tier)

A dedicated proxy sits between app and shards; the app talks to the proxy as if it were one database, and the proxy routes. This centralizes the sharding logic — one place to update the map, clients stay dumb. Vitess (vtgate) and ProxySQL do this for MySQL. The cost is an extra network hop and one more tier to make highly available, since the proxy is now on the critical path.

Coordinator / cluster-aware nodes

The data nodes themselves know the topology and forward a misrouted request to the right peer. DynamoDB and the request-coordinator role in Dynamo-style systems work like this — you can hit any node and it routes internally. Simplest for clients, at the cost of intra-cluster forwarding hops.

Rebalancing

Wherever routing lives, the topology is not static — shards fill unevenly, machines are added, hotspots emerge. Rebalancing is the ongoing process of redistributing data to keep load even. The cardinal rule: rebalancing should move the minimum necessary data (this is why hash % N is so bad and consistent hashing or fixed virtual-shard counts are good), and it should run as a throttled background task so it never starves live traffic of disk and network. Systems like Cassandra and DynamoDB rebalance automatically; classic sharded MySQL setups rebalance with operator-run migration tooling.

09 When a shard dies

A shard down: combine with replication

Sharding has a sharp failure mode that is easy to miss. On an unsharded database, the primary dying takes everything down — total but uniform. On a sharded system, a single shard dying takes down only its key range: if shard 2 of 4 is lost, roughly a quarter of your users cannot read or write while three-quarters are completely fine. That partial-outage property is a double-edged sword — the blast radius is smaller, but you now have N times as many machines that can fail, so the probability that something is down at any moment goes up with N.

The resolution is the key insight of this whole topic: sharding and replication are orthogonal and you use both together. Sharding splits data for scale; replication copies each piece for durability and availability (Lesson 04). In production, each shard is itself a small replica set — one primary plus one or more followers. If a shard's primary dies, a follower is promoted (failover) and that key range keeps serving; you have lost a replica, not the data.

Under the hood — the full topology

Put the lessons together and a production sharded store is a grid: N shards × R replicas. A write routes to shard k, commits on shard k's primary, and replicates to shard k's followers. MongoDB calls each group a replica set and the whole thing a sharded cluster (with mongos routers and config servers as the directory). Cassandra spreads each shard's replicas around the token ring with a replication factor. The shard key controls scale; the replication factor controls survivability.

Interview tip

If asked "what happens when a shard goes down," resist saying "the system is down." The precise answer: "Only that shard's key range is affected, and because each shard is a replica set, a follower is promoted and the range keeps serving — so it's a brief, partial degradation, not a full outage. The trade-off is that more shards means more machines that can fail, which is why each shard needs replication and good failure detection."

★ Review

Summary & self-test

The topic compresses to one sentence: when one primary can't hold the writes or the bytes, split the rows across nodes — and live or die by your shard key and your ability to reshard while serving traffic. If you can defend a shard-key choice, name the hot-shard fixes, and walk the double-write reshard, you can carry this interview.
