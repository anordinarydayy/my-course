Consistent Hashing

The instant you add a server, lose one to a crash, or scale your cluster in for the night, the obvious way of mapping keys to machines reshuffles almost every key at once — wiping caches cold or triggering a brutal data migration. Consistent hashing makes that cost proportional to the change: add one node out of ten, and you move roughly a tenth of the keys, not all of them.

00 The problem

Spreading keys across many nodes

Picture GrabPay's wallet-balance cache. It is far too large for one Redis box, so you run a fleet of, say, ten cache servers and split the data between them. Every read and write carries a key — wallet:SG:8841273, txn:SG:99a1f — and the one decision your client has to make on every single call is: which of the ten servers owns this key?

This is the routing problem at the heart of every sharded cache, every distributed key-value store, and every partitioned database. You need a function that takes a key and returns a node, and it has to satisfy three things at once:

  • Deterministic. The same key must always map to the same node, or a write goes to server 3 and the read goes looking on server 7 and finds nothing.
  • Balanced. Keys should spread roughly evenly, so no single node holds 40% of the wallets while another sits idle.
  • Stable under change. Nodes come and go — you scale out on payday, a box crashes at 3 a.m. — and the mapping must survive those events without re-routing the entire keyspace.

The first two are easy. A plain hash function handles determinism and balance beautifully. It is the third requirement — stability when N, the number of nodes, changes — that quietly destroys the naive approach and forces us toward consistent hashing. The whole lesson is the story of that one requirement.

01 The bottleneck

Why hash(key) % N is a trap

The textbook first attempt is modulo hashing: take a hash of the key, divide by the number of nodes, and use the remainder as the node index.

# pick a node for a key, N = number of nodes
def route(key, N):
    return hash(key) % N

route("wallet:SG:8841273", 10)   # → 4  (lives on node 4)
route("txn:SG:99a1f",       10)   # → 7  (lives on node 7)

With a good hash function this is perfectly deterministic and beautifully balanced — each of the ten nodes gets about a tenth of the keys. For a fixed cluster it is genuinely fine, and you should not over-engineer past it if N truly never changes.

The trap is the % N. The node index depends on N, so the moment N changes — you add an eleventh cache server, or one crashes and leaves nine — the divisor changes for every key at once. The remainder for almost every key lands somewhere new.

Walking through the damage

Take the key wallet:SG:8841273 whose hash is, say, the integer 1,000,004. With ten nodes it maps to 1000004 % 10 = 4. Add one node so N becomes 11, and now it maps to 1000004 % 11 = 4 — fine, that one happened to stay. But its neighbour at hash 1,000,005 goes from 5 to 1000005 % 11 = 6. And 1,000,007 goes from 7 to 1000007 % 11 = 8. Run the arithmetic across the whole keyspace and you find that going from N to N+1 keeps only about 1/(N+1) of the keys in place — every other key moves. Going from 10 nodes to 11 relocates roughly 90% of all keys.

For a cache, "relocates" means the key is now looked up on a node that has never seen it. Every one of those lookups is a miss. The cache effectively goes cold all at once, and all that traffic stampedes through to the database behind it.

Why this is genuinely dangerous

For a sharded database the same arithmetic means a massive data migration: 90% of your rows have to physically copy to a different shard before the new layout is correct, while the cluster is still serving live traffic. For a cache it means a cache stampede — a wave of misses all hitting the origin database at once, which is exactly the moment your database is least able to cope. Either way, a routine "add a node" turns into an incident.

02 The fix

The hash ring

The root cause was that the node index was computed from N. Consistent hashing breaks that dependency. The trick is to stop hashing keys into "N buckets" and instead hash both keys and nodes into the same large, fixed space — and crucially, that space does not depend on how many nodes there are.

How it works under the hood

Take the full output range of a hash function — for a 32-bit hash, the integers 0 to 2³² − 1 — and bend it into a circle so that the largest value wraps around to meet 0. This is the hash ring. Then:

  • Place the nodes. Hash each node's identifier (its hostname or IP, e.g. hash("cache-3.sg")) and drop the node at that point on the ring.
  • Place the keys. Hash each key the same way and drop it at its point on the ring.
  • Assign by walking clockwise. A key belongs to the first node found walking clockwise from the key's position. That node is the key's owner; if you walk past the top of the ring, you wrap around to the first node after 0.

Each node is therefore responsible for the arc of the ring that ends at it — every key landing between the previous node (counter-clockwise) and this node. In implementation terms, the nodes' ring positions are kept in a sorted structure (a balanced tree or a sorted array), and routing a key is a O(log N) binary search for "the smallest node position ≥ hash(key)", wrapping to the first node if none is larger.

This already buys us determinism and balance, just like modulo hashing. The payoff comes in the next section, when N changes — because nothing about a key's position on the ring depends on the node count.

03 The payoff

Adding and removing nodes

Here is the property the whole technique exists for. When you add a node, you hash its name and drop it at one point on the ring. The only keys whose owner changes are those sitting in the arc immediately counter-clockwise of the new node — the slice it now intercepts before they reach the node that used to own them. Every other key on the ring is completely untouched: its clockwise walk still lands on the same node it always did.

How many keys actually move

If keys are spread evenly around a ring with N nodes, each node owns roughly 1/N of the ring. A newly added node carves its slice out of one neighbour's arc, so it captures on the order of K/N keys — where K is the total number of keys. Going from 10 nodes to 11, you move about 1/11 of the keys (~9%), not 90%. That is the difference between "proportional to the change" and "proportional to everything."

Removal is the mirror image, and it is exactly what happens when a node crashes. The failed node vanishes from the ring; every key it owned now walks clockwise to the next surviving node. Only that one node inherits the orphaned arc — the rest of the cluster never notices. Recovery is automatic: re-add the node (or a replacement at the same ring position) and it reclaims its arc.

A worked example: Shopee's session cache

Suppose Shopee runs 8 cache nodes holding 80 million active shopping-session entries — about 10M per node. A Black-Friday surge forces you to add a 9th node:

  • Modulo hashing: N goes 8 → 9, divisor changes, ~8/9 of keys (~71M) are now looked up on the wrong node. The session cache goes cold and 71M lookups stampede the database.
  • Consistent hashing: the 9th node lands on the ring and steals one arc. Roughly 80M / 9 ≈ 8.9M keys re-home to it; the other ~71M never move and stay warm. Only the 8.9M that moved miss once, then re-populate.

Modulo · hash(key) % N

  • Dead simple, one line of code
  • Perfectly balanced for a fixed N
  • Changing N remaps ~all keys → stampede / mass migration
  • Hopeless for autoscaling or node failure

Consistent hashing

  • Changing N moves only ~K/N keys — proportional cost
  • Node failure is absorbed by one neighbour, not the whole ring
  • More moving parts (ring structure, vnodes)
  • Plain ring can be uneven; needs virtual nodes (next)
The interviewer will ask this directly

"What happens to the keys when a node dies?" The crisp answer: only the keys that the dead node owned are affected, and they all shift clockwise to the single next node on the ring; every other key is untouched, so there is no cluster-wide reshuffle. If you can also say the inherited node now carries roughly double load until you rebalance, you have nailed it.

04 A new problem

The imbalance problem

The basic ring fixed stability, but it quietly introduced a fairness problem. We assumed keys and nodes scatter evenly around the ring. With only a handful of nodes, they do not. Hashing three node names gives you three essentially random points on the circle, and random points are rarely equidistant — you might get nodes clustered at the top with a huge empty gap at the bottom.

Whoever owns the arc before that huge gap owns all the keys in it. With three nodes you can easily end up with one node holding 60% of the keyspace and another holding 15%. That node is now a hotspot: more memory pressure, more CPU, more requests, and it will fall over first.

It gets worse when a node dies

Now combine imbalance with failure. When a node crashes, its entire arc is dumped onto the single next node clockwise. That neighbour's load can instantly double. If it was already near capacity, the inherited load tips it over too — and its arc then lands on the next node, which can fall over in turn. This is a cascading failure rolling around the ring, and it is the failure mode the basic ring is most vulnerable to.

The mechanism behind the imbalance

With M random points on a circle, the gaps between them are not uniform — the largest gap tends to be several times the average. The standard deviation of arc sizes shrinks only as you add more points. So the fix is not "use a better hash"; it is "put far more points on the ring." That is exactly what virtual nodes do.

05 The smoothing fix

Virtual nodes

The cure for clustering is to stop giving each physical machine a single point on the ring. Instead, give it many. A virtual node (or vnode) is one of the several ring positions that all map back to the same physical server.

How it works under the hood

For each physical node you generate V positions by hashing decorated identifiers — hash("cache-3.sg#0"), hash("cache-3.sg#1"), … up to #V-1. Each of those V hashes is a separate point on the ring, and all of them route to the same physical box. With V = 150 vnodes per machine and 10 machines, the ring now has 1,500 points instead of 10. Keys still route by walking clockwise to the next point; that point is a vnode, and a small lookup table maps the vnode back to its physical server.

Because each machine now owns 150 small, scattered arcs instead of one big one, the law of large numbers takes over: the sum of 150 random slivers is far closer to the fair share of 1/N than any single random slice could be. Load variance drops sharply — going from one point to ~100+ points per node typically pulls the imbalance down from tens of percent to a few percent.

A second, free benefit: smoother failure

Vnodes also fix the cascade from Section 04. When a physical machine dies, its 150 vnodes disappear from 150 different places on the ring. The orphaned keys are inherited not by one unlucky neighbour but spread across many different surviving machines — so the failed node's load is shared roughly evenly, and no single survivor sees its load double. The cascade risk largely evaporates.

More vnodes per machine

  • Tighter load balance, smaller hot arcs
  • Failure load spreads across many survivors
  • Larger ring structure → more memory, slower rebuild
  • Routing table grows; lookups still O(log) but constants rise

Fewer vnodes per machine

  • Smaller, cheaper ring metadata
  • Faster membership changes to propagate
  • Worse balance; bigger hotspots and lumpier failure
The follow-up: "How do you avoid imbalance?"

Answer: virtual nodes — give each physical machine many points on the ring so its share of the keyspace is the sum of many small arcs, which is far more even than one random arc. Mention the knob: the vnode count (often ~100–256 per node) trades balance against the size of the ring metadata. Cassandra historically defaulted to 256 num_tokens per node and later moved toward smaller values like 16 for faster operations.

06 Real clusters aren't uniform

Weighting & heterogeneous nodes

Real fleets are rarely uniform. You add newer servers with double the RAM, or you are mid-migration with a mix of instance types. If every machine gets the same number of vnodes, the beefy box and the tiny box each own ~1/N of the keys — and the small box drowns while the big one coasts.

Vnodes give you a clean lever for this: weighting. Allocate vnodes in proportion to capacity. A machine with twice the memory gets twice as many ring points, so it naturally owns about twice the keyspace. No special case in the routing logic — the clockwise walk is unchanged; the heavy node simply appears more often.

# build the ring with capacity-weighted vnodes
ring = SortedMap()
for node in nodes:
    vnodes = base_vnodes * node.capacity_weight   # e.g. 150 * 2 for a big box
    for i in range(vnodes):
        pos = hash(f"{node.id}#{i}")
        ring[pos] = node                          # many positions → one machine

Routing then stays a single binary search; only the density of each machine's points differs.

# route a key: first vnode clockwise, wrapping at the top
def owner(key):
    pos = hash(key)
    vnode_pos = ring.ceiling_key(pos)             # smallest ring pos ≥ hash(key)
    if vnode_pos is None:
        vnode_pos = ring.first_key()              # wrapped past the top → first node
    return ring[vnode_pos]
Under the hood: how nodes agree on the ring

Every client and node must share the same view of who sits where, or routing diverges. Two common approaches: a central coordinator (e.g. ZooKeeper or etcd) holds the authoritative membership and pushes updates; or a gossip protocol (used by Cassandra and DynamoDB-style systems) where nodes continuously exchange membership state peer-to-peer until everyone converges. Gossip avoids a central dependency but means the ring view is eventually consistent — for a brief window during a membership change, two nodes may disagree on an arc's owner, which the system must tolerate.

07 Surviving real failures

Replication on the ring

So far each key lives on exactly one node. If that node dies, the data is gone (a cache can tolerate that — it just refetches; a database cannot). For durability, distributed stores keep N replicas of each key, and the ring gives a beautifully simple rule for choosing where they go.

The preference list

To replicate a key R times, walk clockwise from the key's position and pick the next R distinct physical nodes. The first is the primary owner (the coordinator for that key); the next R−1 are the replicas. This ordered set is called the key's preference list. Because it is derived purely from ring position, every node and client can compute it independently — no central registry of "where are the replicas" is needed.

One subtlety the interviewer loves: with vnodes, the next few vnodes clockwise might all belong to the same physical machine. That would put several "replicas" on one box — useless for fault tolerance. So the preference list must skip vnodes whose physical node is already in the list, keeping walking until it has R distinct machines. Production systems extend this further by skipping until they have nodes in distinct racks or availability zones, so losing a whole AZ never takes out every copy.

What happens when a replica dies

If N2 fails, reads and writes for the key still succeed against N1 and N3, and the system can temporarily route the missing copy's writes to the next node down the list — a technique called hinted handoff — which holds the data until N2 returns, then forwards it back. When N2 comes back (or is replaced) it re-derives its arc from the ring and the missing copies stream back to it via an anti-entropy repair. The ring never needs a global reshuffle; only the affected arcs heal.

The follow-up: "How does replication work on the ring?"

Say: each key's replicas are the next R distinct physical nodes clockwise — its preference list — skipping vnodes that resolve to a machine already chosen, and ideally spanning multiple availability zones. This is the bridge to the key-value store lesson (Cassandra/DynamoDB), where the same ring + preference list underpins quorum reads and writes (R + W > N for strong-ish consistency). It also connects back to the sharding lesson: consistent hashing is simply the partitioning function that decides which shard owns a row.

08 In the wild

Where it is used in production

Consistent hashing is not a textbook curiosity — it is load-bearing infrastructure in systems you use every day. Naming the real implementations is worth easy points in an interview.

  • Distributed caches. The original motivation was memcached client-side sharding: the client library hashes the key onto a ring of cache servers, so adding or removing a memcached box only re-homes a slice of keys instead of cold-starting the whole tier. Many Redis client libraries and proxies (e.g. twemproxy) offer a consistent-hashing distribution mode for the same reason.
  • Amazon DynamoDB & Apache Cassandra. Both partition data with a hash ring and vnodes, and place replicas via a clockwise preference list. In Cassandra the knob is num_tokens (vnodes per node); the partition key's hash decides the owning vnode.
  • Riak, ScyllaDB, Voldemort and other Dynamo-lineage key-value stores use the same ring-and-preference-list design.
  • Load balancers. Envoy and NGINX support ring-hash (consistent-hashing) and Maglev load-balancing policies so a given client or session key sticks to the same upstream even as the backend pool changes — useful for cache affinity and sticky sessions without a shared session store.
  • Sharded data layers. Discord famously used consistent hashing to route which service instance owns a given guild; ride-hailing and super-app backends use it to pin a city or a user partition to a consistent set of workers.
Beyond the basic ring: bounded loads & jump hash

Two refinements come up at senior level. Consistent hashing with bounded loads (used by some CDNs and Google's research) adds a cap: if a key's natural owner is already over a load threshold, the key overflows to the next node clockwise — capping any hotspot. Jump consistent hashing is a tiny, allocation-free algorithm that maps a key to one of N buckets with minimal movement when N grows — extremely fast and memory-light, but it cannot remove an arbitrary node from the middle (only shrink/grow the count), so it fits stateless sharding better than a churny node membership.

09 Honest limits

Limits, hot keys & what breaks at 10×

Consistent hashing solves node churn elegantly, but it does not solve every distribution problem, and a strong candidate names the gaps before the interviewer does.

The single hot key

The ring balances keyspace, not traffic. If one key is white-hot — the txn:flash-sale:11.11 counter, or a celebrity's user:profile during a livestream — that key hashes to exactly one node, and vnodes do not help, because a single key sits at a single ring position. That one node gets hammered regardless of how evenly the keyspace is divided. Fixes live outside consistent hashing: replicate the hot key to multiple nodes and read from any, add a local in-process cache layer in front of the ring, or shard the value itself (e.g. split a hot counter into N sub-counters and sum them).

Rebalancing still costs I/O

"Only K/N keys move" is a big improvement, but for a database those K/N keys are still real bytes that must stream across the network to the new node, competing with live traffic. At 10× the data, K/N of a much larger K is itself large — adding a node to a 100-node, petabyte cluster can still move terabytes. Production systems throttle this streaming (Cassandra's stream_throughput), do it gradually, and serve from old and new locations during the transition.

Ring metadata and coordination at scale

With 256 vnodes per node and thousands of nodes, the ring has hundreds of thousands of tokens. Every node gossiping that state, and recomputing ownership on each membership change, becomes non-trivial. This is precisely why Cassandra moved its default vnode count down over time, and why very large fleets sometimes prefer fewer vnodes with smarter placement (token allocation algorithms) over brute-force counts.

The data model still has to cooperate

Consistent hashing routes by a single key's hash, which scatters related rows everywhere. That makes point lookups fast but range scans expensive (the range is smeared across the whole ring). If your access pattern is "all of one user's transactions in time order," you may want range partitioning instead, or a composite key whose partition component groups related rows on one node. Choosing the partition key is the real design decision; consistent hashing is just the mechanism that places it.

The deep follow-up to rehearse

"Your keyspace is balanced but one node is still melting — why, and what do you do?" The expected answer separates two ideas: consistent hashing balances how much of the keyspace each node owns, but a hot key is a traffic skew on a single ring point that no amount of vnodes can spread. Then you reach for hot-key replication, a front cache, or value-sharding — and you mention bounded-load consistent hashing as the in-protocol mitigation. Showing you know the technique's boundary is what separates a senior answer from a memorized one.

★ Review

Summary & self-test

The whole topic compresses to one sentence: map keys and nodes onto a shared ring so a key belongs to the next node clockwise — then adding or losing a node moves only its neighbouring slice of keys, not the whole keyspace. Layer virtual nodes on top for balance, and a clockwise preference list on top for replicas, and you have the backbone of every Dynamo-style store.
