02Lesson · System Design Interview

Scale from Zero to Millions of Users

This is the master narrative the whole course hangs on. You start with a single box, then add exactly one component each time the previous one becomes the bottleneck. Internalise this chain and you can rebuild almost any architecture on a whiteboard from first principles — every later lesson is just a deep dive into one move in it.

About these notes: original study material on a standard, widely-documented engineering topic — evolving a system from one server to a multi-region fleet. The explanations, numbers, examples, and diagrams here are my own; nothing is copied from any paid course or book. One diagram (the evolving architecture in Step 03) is interactive — press Play (or step with Next) to watch the system grow one tier at a time. Reference: BBG V1·C1.
00 The starting point

One box does everything

Every large system began as something small enough to fit in your head. Before you reach for queues and shards, you should be able to say exactly what the simplest possible deployment looks like — because every later decision is a reaction to a limit you hit on this one. The starting point is a single server: one machine running the web application, the database, and an in-process cache, all together. The only thing living outside that box is DNS, which we lean on rather than run ourselves.

How a request actually flows

Walk it through end to end, because this flow never really changes — we just keep inserting components into the middle of it. Imagine a user opening the Shopee app to view a product page:

  • The app needs to reach api.shopee.sg, but it only knows the name, not the address. It asks DNS — the internet's phone book — which returns an IP such as 52.74.10.5. DNS is a paid third-party service (Route 53, Cloudflare, your registrar) that we don't host; that single fact will matter again when we add multiple regions in Step 08.
  • The app opens a connection to that IP and sends an HTTP request: GET /products/123.
  • The web server runs the application code, which queries the database (SELECT * FROM products WHERE id = 123), renders a JSON response, and returns it.
  • The browser or app paints the page.

That is a complete, working product. It can serve real users. The reason we don't stop here is not that it's wrong — it's that it has a ceiling, and the rest of the lesson is the story of repeatedly raising that ceiling.

Why start this small on purpose

In an interview, opening with the single-box design is a feature, not a weakness. It proves you scale in response to evidence rather than cargo-culting microservices on day one. Premature distribution is its own failure mode: you pay coordination cost long before you have the traffic to justify it.

01 Bottleneck → first split

Split the data tier out

The bottleneck: the single box is a single point of failure (SPOF) — if that machine dies, the whole product is down — and the web app and database now fight over the same CPU, RAM, and disk. A burst of expensive queries starves the web server of memory; a spike in web traffic steals cycles from the database. Worse, the two tiers have opposite hardware appetites: web servers want lots of cheap CPU and can be thrown away and recreated freely, while databases want fast disks, large RAM, and stable, long-lived storage. Cramming both onto one box means you can never tune the machine for either.

The fix: give the database its own host

The first move is to pull the data tier onto its own server. The web server now connects to the database over the network instead of over localhost. This buys three things at once: each tier can be sized for its own workload, you can scale the two independently, and a crash in one no longer automatically takes the other with it.

This is also the moment you make a foundational data-model decision — relational (SQL) versus non-relational (NoSQL). For a fintech product like GrabPay, where a wallet balance must never silently drift, the strong consistency and transactions of a relational store like PostgreSQL or MySQL are usually the right default. NoSQL stores such as DynamoDB, Cassandra, or MongoDB earn their place when you have enormous write volume, a simple access pattern, or data that doesn't fit neatly into rows and joins — that's a whole lesson of its own.

Interview follow-up

"You split web and DB — which would you pick, SQL or NoSQL, and why?" Answer with the access pattern, not the brand. "Wallet balances need ACID transactions, so relational; the activity feed is append-heavy and read-mostly, so a NoSQL store works there." Showing you'd use both, chosen per workload, beats picking a side.

02 The scaling fork

Vertical vs horizontal scaling

Now that you have separate tiers, the obvious next question is: when one of them runs hot, do you make the box bigger or add more boxes? These are the two axes of scaling, and naming the trade-off cleanly is one of the most reliable points you can score in an interview.

Vertical scaling — a bigger box

Vertical scaling (scaling up) means moving to a more powerful machine: more vCPUs, more RAM, faster NVMe disks. Its great virtue is simplicity — there is nothing to coordinate, no code to change, you just resize the instance. For the database tier specifically, vertical scaling is often the first thing you do, because keeping all the data on one node sidesteps the hard distributed-data problems entirely.

But it has a hard ceiling: there is a largest instance money can buy, and you will eventually hit it. It gets disproportionately expensive near the top end, and — critically — a single big box is still a SPOF. One machine, however mighty, can still fail, and while it's being resized it's typically offline.

Horizontal scaling — more boxes

Horizontal scaling (scaling out) means adding more machines and spreading load across them. There's no ceiling in principle — you keep adding commodity nodes — and redundancy comes for free, because losing one node out of twenty barely dents capacity. This is the only path to true high availability and is what large consumer products run on. The catch is that the moment you have more than one node, you inherit coordination problems: how do you spread traffic across them, how do they share state, and how do you keep their data consistent? Steps 03 through 10 are essentially the bill for choosing this path.

Vertical (scale up)

  • Dead simple — resize the instance, change nothing in code
  • No coordination, no data distribution to reason about
  • Hard ceiling: a biggest-possible machine exists
  • Cost climbs steeply at the high end
  • Still a single point of failure — no redundancy

Horizontal (scale out)

  • Effectively unlimited — keep adding commodity nodes
  • Built-in redundancy and high availability
  • Needs a load balancer to spread traffic
  • Forces you to handle shared state and consistency
  • More moving parts to operate and monitor
The pattern to remember

Vertical scaling is the cheap early win; horizontal scaling is the long-run answer. Most real systems do both: scale the web tier out (it's stateless and easy) and scale the database up first, then reach for replication and sharding only when up runs out of room. The rest of this lesson chooses horizontal — and pays its coordination bill one component at a time.

03 Bottleneck → spread the web tier

Load balancer + a fleet of web servers

The bottleneck: one web server is, again, a SPOF, and when it saturates — too many concurrent connections, CPU pinned at 100% — every user sees timeouts at once. We chose horizontal scaling, so the answer is to run several identical web servers. But that immediately raises the coordination question: clients only know one address, so who decides which server handles each request?

The fix: a load balancer in front of a server fleet

A load balancer is a component that accepts all incoming traffic on a single public address and distributes it across a pool of backend servers. The public address it exposes is a virtual IP (VIP) — the one thing DNS now points at. The web servers themselves move onto private IPs reachable only inside your network, so they're never addressed directly from the internet. That's a security win on its own: attackers can't reach the app servers, only the load balancer.

How it works under the hood

For each new connection the load balancer picks a backend using a balancing algorithm — round-robin (rotate through the list), least-connections (send to whoever is least busy), or a weighted variant when machines have different sizes. It continuously runs health checks, pinging something like GET /healthz on each backend every few seconds. A server that fails its checks is pulled out of rotation automatically and added back when it recovers. This is exactly how a dead node is handled: if web server 2 crashes, the next health check fails, the load balancer stops routing to it, and users never notice because servers 1 and 3 absorb the traffic. Add capacity by registering a new private IP; remove it by deregistering — the basis of autoscaling.

Real implementations: NGINX and HAProxy for software load balancers, Envoy as a modern L7 proxy, and managed cloud options like AWS ALB/NLB or GCP Cloud Load Balancing. They split roughly into L4 (route by IP/port, blazingly fast, protocol-agnostic) and L7 (read the HTTP request and route by path or host, e.g. send /pay/* to the payments fleet).

Interview follow-up

"A web server dies mid-request — what does the user see?" The clean answer: the load balancer's health check removes it within a few seconds; in-flight requests on that node fail and the client (or the LB, with retries) reissues them to a healthy server. Because the tier is replicated, capacity dips but the service stays up. If you can't yet say why the web tier survives a node death but a naive session wouldn't, hold that thought — it's exactly Step 07.

04 Bottleneck → the DB can't keep up

Database replication

The bottleneck: we scaled the web tier out, but every one of those web servers still hammers the same single database. The database is now both the busiest node and the last remaining SPOF — if it dies, no amount of web servers helps. Most products are also heavily read-skewed: a Shopee shopper browses dozens of product pages for every one purchase, so reads outnumber writes by perhaps 10:1 or more.

The fix: leader–follower replication

Database replication runs multiple copies of the database. One node is the leader (also called primary or master): it is the only node that accepts writes — INSERT, UPDATE, DELETE. The others are followers (replicas), which receive a continuous stream of the leader's changes and serve read-only queries. The application routes writes to the leader and spreads reads across the followers.

How replication works under the hood

The leader records every change to an ordered write-ahead / replication log (MySQL calls it the binlog; PostgreSQL streams the WAL). Each follower keeps a long-lived connection to the leader, pulls new log entries, and replays them in order against its own copy. By default this is asynchronous: the leader confirms a write to the client without waiting for followers, so there is a small replication lag — usually milliseconds. That lag is the price, and it creates a classic trap: a user updates their GrabPay display name (write → leader), the page immediately re-reads it (read → a follower that hasn't caught up), and they briefly see the old name. The fix is read-your-writes consistency — route a user's reads to the leader for a short window after they write, or read from the leader for that one query.

What this buys you, and the failure modes

  • Read scaling: need more read throughput? Add followers. Reads scale almost linearly.
  • Availability & redundancy: data lives on several machines, so a single disk failure isn't data loss.
  • A follower dies: the load balancer (or the app's connection logic) simply routes reads to the surviving followers — no outage.
  • The leader dies: the serious case. The system runs a failover — promote a follower to be the new leader and repoint writes at it. Tools like Patroni (PostgreSQL) or Orchestrator (MySQL) automate this. The risk: if the old leader had un-replicated writes when it died, those can be lost (a window of data loss tunable down with semi-synchronous replication).
Interview follow-up

"What happens the instant the leader fails, and can you lose data?" Name the steps: detect failure, promote the most up-to-date follower, repoint writes, possibly fence the old leader to avoid split-brain (two nodes both thinking they're leader). Then concede the honest risk — asynchronous replication means a tiny tail of un-replicated writes can be lost, which is why payment systems often run semi-synchronous replication.

05 Bottleneck → repeated reads still cost

A cache tier

The bottleneck: followers scale reads, but every read still costs a database round trip, query parse, and disk/buffer lookup. The same handful of rows get read over and over — the homepage banner config, a hot product, today's exchange rate. Re-fetching identical data thousands of times per second is wasted work, and it caps how far replication alone can take you.

The fix: an in-memory cache in front of the database

A cache is a fast, in-memory key-value store that holds the results of expensive lookups so repeat reads skip the database entirely. Because it lives in RAM, a hit returns in well under a millisecond versus several for a database query. The dominant tools are Redis and Memcached; they run as their own tier so every web server shares the same cache.

Cache-aside: the workhorse pattern

The most common strategy is cache-aside (lazy loading). The application owns the logic:

# read path — cache-aside
val = cache.get("product:123")
if val is None:                       # cache miss
    val = db.query("SELECT ... WHERE id=123")
    cache.set("product:123", val, ttl=300)  # populate, expire in 5 min
return val

On a hit the database is never touched. On a miss you read the DB once, write it into the cache, and serve it; the next reader hits the cache. A close cousin is read-through, where the cache library itself fetches from the DB on a miss, so the app just calls cache.get() and stays simpler.

The two hard parts: expiry and invalidation

Caching trades freshness for speed, so you must manage staleness. Two levers:

  • TTL (time-to-live): every entry expires after a set window (the ttl=300 above). Too short and your hit rate collapses; too long and users see stale data. It's a tuning dial, not a constant.
  • Invalidation: when the underlying row changes, delete or update the cached copy so the next read repopulates it. "There are only two hard things in computer science: cache invalidation and naming things" is a joke because invalidation genuinely is hard — miss an invalidation and you serve wrong data indefinitely.

Failure modes to name

  • Cache node dies: reads fall through to the database. Fine in a trickle, dangerous in a flood (see below). Run the cache replicated/clustered (Redis Cluster, Redis Sentinel) so a node loss doesn't wipe the whole cache.
  • Thundering herd / stampede: a hot key expires and thousands of concurrent requests all miss and slam the DB at once. Mitigate with a short lock so only one request repopulates, slightly randomized TTLs, or serving stale-while-revalidate.
  • Cold cache: right after a restart everything misses and the DB sees a spike — sometimes worth warming the cache with known-hot keys on startup.
Interview follow-up

"When would you not cache something?" Good signals: data that changes on every read, data you must never serve stale (a live wallet balance unless you can invalidate on write), or data so rarely read that the cache never warms up. Also be ready for write strategies — write-through (write cache + DB together, consistent but slower writes) vs write-back (write cache now, DB later — fast but risks loss if the cache dies).

06 Bottleneck → static bytes travel too far

CDN for static assets

The bottleneck: the cache and replicas handle dynamic data, but a huge share of bytes is static — product images, app bundles, CSS, fonts, videos. Serving an Agoda hotel photo from a Singapore data center to a user in London means the bytes cross the planet on every request: high latency, wasted bandwidth on your origin, and a heavier load on your web tier for work that has nothing to do with your business logic.

The fix: a Content Delivery Network

A CDN is a geographically distributed network of edge servers (points of presence, or PoPs) that cache your static content close to users. When a user requests an image, DNS or the CDN routes them to the nearest edge. If that edge has the file cached (a cache hit), it serves it directly — a short hop instead of a transcontinental one. On a miss, the edge fetches it once from your origin (your server or an object store like Amazon S3), caches it, and serves all future nearby requests from the edge.

How it works under the hood

Each cached object obeys HTTP cache headers: Cache-Control: max-age=86400 tells the edge how long to keep it, and ETag lets the edge cheaply revalidate whether its copy is still current. Because cached assets are immutable in practice, the universal trick is cache-busting via the filename: ship app.4f9a2c.js with a content hash, set a one-year TTL, and to "update" the file you just deploy a new hashed name. Old clients keep their old file; new HTML points at the new one. Major CDNs: Cloudflare, Amazon CloudFront, Fastly, Akamai.

Costs and failure modes

  • You pay for CDN traffic, so cache only what's worth it and tune TTLs to keep the hit rate high.
  • Stale content: if you must update something before its TTL, you issue an invalidation/purge — but purges are slow and sometimes metered, which is exactly why filename hashing is preferred.
  • An edge or the whole CDN fails: well-configured CDNs fail over to another PoP or fall back to origin; just make sure your origin can survive a cold-cache burst if the CDN drops out entirely.
07 Bottleneck → servers remember things

Make the web tier stateless

The bottleneck: back in Step 03 we waved at a problem. If a web server stores session state in its own memory — "user 42 is logged in, here's their cart" — then a user is tied to the one server that holds their session. We call this sticky sessions, and they quietly poison horizontal scaling: if that server dies, the user is logged out; the load balancer can't freely rebalance traffic; and autoscaling can't kill a node without dropping everyone on it.

The fix: move state out of the servers

A stateless web server keeps no per-user data locally. Any request can land on any server and be handled identically, because the shared state lives in an external session store — typically Redis or a NoSQL store like DynamoDB. Each request carries an identifier (a session ID cookie, or a self-contained JWT); the server looks the session up in the shared store, does its work, and forgets everything.

Why this unlocks real autoscaling

Once the tier is stateless, servers become interchangeable and disposable. The load balancer can use plain round-robin with no stickiness. Autoscaling can spin up ten extra servers when GrabFood lunch traffic spikes and tear them down at 3pm with zero user impact, because no server is holding anything irreplaceable. A crashed node costs nothing: its in-flight requests retry on a peer, and the sessions were never on it in the first place. This is the property that makes every earlier step pay off — it's the difference between "more servers" and "elastic capacity".

Interview follow-up

"Server-side session store vs stateless JWT — which and why?" Both make the tier stateless. A central store lets you revoke a session instantly (log a user out everywhere) at the cost of a lookup per request; a JWT is self-contained and needs no lookup but is hard to revoke before it expires. Fintech often prefers revocable server-side sessions for exactly that reason.

08 Bottleneck → one region, one fate

Multiple data centers

The bottleneck: everything so far lives in one data center. Two problems remain. First, users far from that region pay a latency tax on every dynamic request (the CDN only helps with static bytes). Second — and this is the big one — a single region is a single point of failure at the largest scale: a power outage, a network partition, or a cloud-provider zone failure takes the entire product offline. For a payments platform like SeaMoney, that's catastrophic.

The fix: run in multiple regions and geo-route

Deploy the full stack in two or more data centers — say ap-southeast-1 (Singapore) and another region — and use geoDNS (geo-routing DNS) to direct each user to the nearest healthy one. This is where the "DNS is external" fact from Step 00 finally pays off: because DNS sits in front of everything, it's the natural place to make the region decision. A user in Jakarta resolves api.seamoney.com to the Singapore region; a user in another market resolves to theirs.

Failover: how a region death is survived

Each region is health-checked. If the Singapore region goes dark, the DNS/traffic-management layer (e.g. AWS Route 53 health checks, or a global load balancer) stops handing out its IPs and routes all users to a surviving region. Users see a brief blip — bounded by DNS TTL — then service resumes. This is active-active (all regions serve live traffic) or active-passive (one serves, another is a warm standby).

The genuinely hard part: cross-region data

Stateless web servers replicate trivially — you just run more of them. Data is the problem. If both regions accept writes (active-active), you must synchronize data across regions, and now you're staring straight at the CAP theorem: across a slow, sometimes-broken inter-region link, you cannot have both strong consistency and full availability during a partition. The two honest options:

  • Asynchronous cross-region replication — fast and available, but a write in one region is briefly invisible in the other (eventual consistency), and a failover can lose the un-replicated tail. Fine for a social feed; scary for a balance transfer.
  • Pin each user's data to a home region (data residency / sharding by geography) so writes are always local and strongly consistent, accepting that cross-region access is slower. Often also required by regulation — Singapore and Indonesian financial data frequently must stay in-country, which makes this less a choice than a constraint.
Interview follow-up

"Active-active across regions — how do you handle a balance write in both regions at once?" This is a CAP question in disguise. Strong answers either funnel all writes for a given account to one home region (no cross-region conflict) or accept eventual consistency with conflict resolution and reconcile asynchronously. Saying "I'd just replicate both ways synchronously" is the trap — the inter-region link is too slow and too flaky for that.

09 Bottleneck → tight coupling & spikes

A message queue

The bottleneck: as features grow, the web tier ends up doing slow work inline on the request — sending the payment-receipt email, generating a thumbnail, updating a search index, calling a fraud-scoring service. Each makes the user wait, and tightly couples the web server to a downstream that might be slow or down. When a flash sale on Shopee 10x's the order rate, that synchronous work melts the web tier even though the actual order write is cheap.

The fix: decouple with an async message queue

A message queue is a durable buffer that sits between a producer (the web tier, which publishes a message describing work to do) and one or more consumers (separate worker services that process those messages later). The web server publishes "order 9981 placed" and returns to the user immediately; workers pick the message up and send the email, score fraud, and update the index on their own time.

What decoupling buys you

  • Spike smoothing (buffering): the queue absorbs a burst that the workers couldn't handle in real time. During a flash sale, messages pile up briefly and drain as workers catch up — the producer never blocks. This is back-pressure done right.
  • Independent scaling: the producers and consumers scale separately. Queue getting deep? Add more workers. The web tier is untouched.
  • Loose coupling & resilience: if the email service is down, messages wait safely in the queue instead of failing user requests. The web tier doesn't even know the email service exists.

Under the hood, and the failure modes

Tools: Apache Kafka (a high-throughput partitioned log, great when you want to replay history), RabbitMQ (a classic broker with rich routing), and managed options like Amazon SQS/SNS. Delivery is usually at-least-once, which means a message can be delivered more than once after a retry — so consumers must be idempotent (processing "send receipt for order 9981" twice must not send two emails or, worse, double-charge). A poisoned message that always fails is routed to a dead-letter queue for inspection rather than blocking the line. And the new failure mode to name: an unbounded queue can grow forever if consumers can't keep up — monitor queue depth and scale workers (or shed load) before it does.

Interview follow-up

"Your queue guarantees at-least-once delivery — how do you avoid double-charging a card?" The expected word is idempotency: attach a unique key to each operation and have the consumer record completed keys, so a redelivered "charge order 9981" is recognized and skipped. Mentioning dead-letter queues and ordering guarantees (per-partition in Kafka) earns extra credit.

10 Bottleneck → writes outgrow one leader

Shard the database

The bottleneck — the 10x at the data layer: replication scaled reads, but all writes still funnel through one leader. At enough scale — millions of users, every order, payment, and message a write — that single leader runs out of write throughput, RAM to hold the working set, and disk to hold the data. You can scale it vertically for a while, but you've already met that ceiling in Step 02. There is exactly one move left.

The fix: horizontal partitioning (sharding)

Sharding splits one logical database into many independent databases called shards, each holding a disjoint subset of the rows and each accepting its own writes. Ten shards give you roughly ten times the write capacity and a tenth of the data per node. A shard key — say user_id — decides which shard a row lives on, typically shard = hash(user_id) % N. To find user 7's wallet you hash 7, land on (say) shard 2, and that shard alone serves the read and the write.

Choosing the shard key well

The shard key is the single most consequential decision here. A good key spreads data and load evenly. A bad one creates hot shards: shard by country and Indonesia overwhelms one node while Brunei's sits idle; shard by an auto-incrementing ID and all new writes pile onto the highest shard. Hashing a high-cardinality, uniformly-accessed key like user_id usually distributes best.

The complexity sharding introduces — and the 10x failure

  • Cross-shard queries get hard. A query that spans users now has to fan out to every shard and merge results. Joins across shards effectively disappear — you denormalize or query per-shard. "Top sellers across all of Shopee" is no longer a single GROUP BY.
  • Resharding is painful. Plain hash % N means adding a shard remaps almost every key — a massive, risky data migration. The standard remedy is consistent hashing (its own lesson), which moves only a small fraction of keys when you add or remove a node.
  • Cross-shard transactions. A transfer from a user on shard 2 to one on shard 5 can't use a single local transaction; you need a distributed pattern (sagas, two-phase commit) — a real headache for fintech, often avoided by keeping related accounts co-located.
  • Failure mode: a shard going down takes only its slice of users offline, not everyone — partial outage instead of total. Each shard is therefore itself replicated (leader + followers), so sharding and replication stack: many shards, each redundant.
Interview follow-up

"You sharded on user_id — now you need to add shards next quarter. What breaks?" The expected answer: naive mod N remaps nearly everything, so you'd reach for consistent hashing or pre-split virtual shards to migrate only a slice. Then be ready for "how do you run a transaction across two shards?" — the cross-shard transaction problem that pushes you toward sagas or keeping related data co-located.

11 Operating the machine

Logging, metrics, automation

The bottleneck: you now have load balancers, dozens of stateless servers, replicas, caches, edges, queues, workers, and shards across regions. No human can reason about that by SSH-ing into boxes. The system has outgrown your ability to see it, and an outage you can't observe is an outage you can't fix.

Observability: logs, metrics, traces

  • Logs — append-only records of discrete events. With many servers you must centralize them (the ELK stack: Elasticsearch, Logstash, Kibana; or Loki/Grafana) so you can search one place instead of a hundred. Tag each log line with a request/trace ID so you can follow one request across every tier.
  • Metrics — numeric time series: request rate, error rate, p50/p95/p99 latency, CPU, queue depth, cache hit ratio, replication lag. Prometheus scrapes and stores them; Grafana dashboards and alerts on them. Watch the tail (p99), not just the average — averages hide the users having a bad time.
  • Distributed tracing — tools like Jaeger or OpenTelemetry stitch a single request's path across services so you can see which hop is slow.

Automation: the system that runs itself

At this scale, manual operations are too slow and too error-prone. You lean on CI/CD for automated build, test, and deploy; infrastructure as code (Terraform) so environments are reproducible; autoscaling policies that add and remove servers on the metrics above; and automated alerting/on-call (PagerDuty) so a breached threshold pages a human before users complain. The recurring theme of the whole lesson — identify the bottleneck, add a tier, manage the new complexity — applies to operations too: monitoring is how you find the next bottleneck.

The pattern, one last time

Every step in this lesson is the same loop: a component became the bottleneck → you replicated it, partitioned it, or added a tier in front of it → that fix introduced new complexity, which you then managed. The architecture isn't memorized; it's derived by applying this loop until the traffic stops hurting. That's the skill the interview is really testing.

★ Review

Summary & self-test

The whole lesson compresses to one move repeated: find the component that's the bottleneck, then replicate it, partition it, or insert a tier in front of it — and manage the new complexity that creates. If you can narrate the chain from a single box to multi-region shards, naming the bottleneck that forces each step and the trade-off it introduces, you can derive almost any architecture on a whiteboard.
