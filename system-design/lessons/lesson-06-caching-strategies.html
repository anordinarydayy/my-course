06Lesson · System Design Interview

Caching Strategies

Reads almost always buckle the database before writes do, and the awkward truth is that most of those reads ask for the same handful of hot rows over and over. A cache parks that data in memory in front of the store. Adding one is the easy half-hour; the interview lives in invalidation, the stampede, and the hot key — the three problems the cache quietly creates.

About these notes: original study material on caching — a standard, widely-documented engineering topic. The explanations, examples, numbers, and diagrams here are my own; nothing is copied from any paid course or book. Two diagrams are interactive — press Play (or step with Next) to watch the flow.
00 The problem

Why reads break the DB first

Picture the GrabFood home screen at 12:00 on a weekday in Singapore. Hundreds of thousands of people open the app at lunch and all see roughly the same thing: the list of restaurants near the CBD, the banner promos, the menu of the three or four chains everyone orders from. Each of those screens fans out into dozens of read queries. The writes — someone places an order, a merchant flips an item to "sold out" — are a tiny trickle by comparison. This read-heavy, write-light shape is the norm for almost every product system.

A relational database can serve those reads, but each one costs something real: parse the query, plan it, walk an index, read pages off disk (or out of the buffer pool), serialize the result, and hold a connection open the whole time. A primary that comfortably handles 5,000 reads per second falls over at 50,000. And the cruel part is the access skew: the traffic isn't spread evenly across millions of rows. It clusters. A small slice of the data absorbs the overwhelming majority of reads.

This is the 80/20 rule (more pessimistically, 90/10): roughly 80% of read traffic targets roughly 20% of the data — the popular menu, the trending merchant, the logged-in user's own profile fetched on every screen. Repeatedly asking the database for the exact same answer is wasteful. If a row hasn't changed since the last 10,000 people asked for it, computing the answer 10,000 times is pure overhead.

The fix is a cache: a fast, in-memory key-value store that sits in front of the database and holds the answers to recent or popular reads. RAM lookups are on the order of microseconds; a disk-backed database query is often milliseconds — a 100x to 1000x gap. Serve the hot 20% from memory and the database's load collapses, while p99 latency for those reads drops by an order of magnitude.

Why it matters

A cache doesn't make the database faster — it makes the database do less. The win is twofold: lower latency for the user on a hit, and dramatically reduced load on the store, which means you defer the expensive day when you have to shard or add read replicas. That deferral alone is often worth the entire exercise.

01 The vocabulary

Hit, miss & hit ratio

Two words anchor every caching conversation. A cache hit is when the data you want is already in the cache, so you return it immediately without touching the database. A cache miss is when it isn't there, so you fall back to the database (the origin or source of truth), pay the slow path, and usually store the result for next time.

The single number that tells you whether your cache is earning its keep is the hit ratio — the fraction of lookups that are hits:

hit ratio = hits / (hits + misses)

A 95% hit ratio means only 1 in 20 reads reaches the database. Note how nonlinear the relief is: going from a 90% to a 99% hit ratio doesn't shave 9% off database load — it cuts the misses that reach the DB from 10% to 1%, a 10x reduction in origin traffic. That is why interviewers care about hit ratio specifically, not just "do you have a cache."

The economics of a worked example

Say SeaMoney's wallet-balance read endpoint takes 250,000 requests per minute at peak. The 80/20 skew applies: active users check their balance repeatedly, so a small set of keys dominates.

  • No cache: all 250k/min (about 4,170/sec) hit Postgres. That alone could saturate a sizeable primary.
  • 90% hit ratio: only 25k/min (about 417/sec) reach the DB — a tenth of the load.
  • 99% hit ratio: only 2,500/min (about 42/sec) reach the DB. The same primary is now basically idle on this workload.

The factors that drive hit ratio up are: a workload with real skew (most product reads have it), a cache big enough to hold the working set, and a sensible eviction and expiry policy so you don't throw out things you're about to need. We'll get to each.

The real-world tools are Redis and Memcached, both in-memory key-value stores reachable over the network in well under a millisecond. Memcached is a lean multithreaded string cache; Redis adds rich data types (hashes, sorted sets, lists), persistence options, replication, and Lua scripting, which is why it's the more common default for product caches today.

Interview tip

When asked "would a cache help here?", don't answer yes reflexively. The honest test is: is this workload read-heavy, and is there access skew? A cache in front of a uniformly-random-access, write-heavy table can have a near-zero hit ratio and only add a hop. Say that out loud — it shows judgment.

02 The default read pattern

Cache-aside (lazy loading)

So you've added a cache. How does the application use it? The most common pattern by far is cache-aside, also called lazy loading: the application code is in charge, and the cache is a dumb store off to the side that knows nothing about the database.

The mechanism, walked step by step, is the same on every read:

  • The app builds a key — say user:8842:wallet — and asks the cache for it (a GET).
  • Hit: the cache returns the value. The app returns it to the caller and the database is never touched. Done in microseconds.
  • Miss: the cache returns nothing. The app falls back, runs the query against the database, gets the value, writes it into the cache (a SET with a TTL) so the next reader hits, and returns it.

The name "lazy loading" captures the key property: data only enters the cache when someone actually asks for it. You never pre-load; the cache fills itself with exactly the working set the traffic demands, and cold data never wastes a slot.

Why cache-aside is the workhorse

It has two properties that make it the safe default. First, it's resilient: because the application talks to the cache and the database separately, if the cache goes down every request simply becomes a miss and falls through to the database. You lose performance, not correctness — the system degrades instead of breaking. Second, it only ever caches data someone read, so it doesn't fill up with rows nobody wants.

Its two weaknesses

  • The first read always misses. Every key pays a cold-start penalty: the first reader after a key is added or evicted eats the slow database path. For a brand-new viral item, that "first reader" can be thousands of simultaneous requests — the stampede problem in section 07.
  • Data can go stale. The cache copy is a snapshot. If the database changes and nothing updates the cache, readers keep getting the old value until the entry expires or is invalidated. Cache-aside on its own does nothing about this — keeping the two in sync is the topic of section 06.

A compact sketch of the read path in pseudocode makes the contract concrete:

def get_wallet(user_id):
    key = f"user:{user_id}:wallet"
    val = cache.get(key)            # 1. ask the cache
    if val is not None:
        return val                  # 2. HIT
    val = db.query_wallet(user_id)  # 3. MISS -> origin
    cache.set(key, val, ttl=60)     # 4. populate, expire in 60s
    return val                      # 5. return
The follow-up to rehearse

"What's wrong with caching here?" The strong answer names both weaknesses unprompted: cold-start misses (and the stampede they invite) and staleness (and how you'll bound it with a TTL or invalidate on write). If you only mention speed, you've shown you haven't run a cache in production.

03 Read-through & the write side

Read-through and the write patterns

Cache-aside puts the orchestration in your application code. Read-through moves it into the caching layer instead: the application asks the cache for the key and walks away. If it's a miss, the cache itself (via a provider or library that knows how to reach the origin) loads from the database, stores the result, and returns it. To the app, a read is always just "ask the cache."

The trade is convenience for coupling. Read-through keeps your application code clean and centralizes the loading logic — handy when many services read the same data and you don't want each one re-implementing the miss path. But the cache now needs a loader that understands your data model, which is why read-through usually shows up with a caching library or a managed cache (Amazon DynamoDB Accelerator / DAX in front of DynamoDB is a clean example) rather than a bare Redis you wired up yourself. The cold-start and staleness problems are identical to cache-aside.

The four write patterns

Reads are only half the story. The moment data changes, you have to decide how a write touches the cache and the database — and this is where consistency is won or lost. There are four canonical patterns.

  • Write-through. On every write, the app (or cache) updates the cache and the database synchronously, in the same operation, before acknowledging the client. The cache is never behind the database. Reads after a write are always fresh.
  • Write-back (write-behind). The write goes to the cache only and is acknowledged immediately; the cache flushes to the database asynchronously, often batched. Writes feel instant, but the database lags the cache, and an unflushed crash loses data.
  • Write-around. The write goes straight to the database and skips the cache entirely; the cache entry is just invalidated (deleted). The data only re-enters the cache lazily, on the next read miss. Good for write-once-read-rarely data you don't want polluting the cache.
  • Cache-aside on write. The most common pairing with cache-aside reads: write to the database, then delete (not update) the cache key, so the next read repopulates it from the fresh source. Deleting rather than updating sidesteps a subtle race we'll meet in section 06.
Why "delete, don't update" on write

When data changes, you can either rewrite the cache with the new value or just delete the key. Deletion is usually safer: it's idempotent, it avoids caching a value computed from a write that later rolls back, and it dodges a concurrency race where a slow reader's old value overwrites a fast writer's new value in the cache. The cost is one guaranteed miss on the next read — almost always worth it.

04 The write trade-off

Write-through vs write-back

The sharpest decision among the write patterns is write-through vs write-back, because they sit at opposite ends of the consistency-versus-speed spectrum. It's worth walking the mechanism of each carefully.

Write-through, step by step

A write arrives. The system updates the cache, then writes the database (or both), and only after the database confirms does it acknowledge the client. Because the durable store is updated on the critical path, the cache and database are always in agreement, and any read that follows a write sees the new value. The price is latency: every write now pays for the slow database operation, and if the database is briefly slow, your writes are slow too. Write-through also tends to fill the cache with data that may never be read again, which you can offset by pairing it with a TTL.

Write-back, step by step

A write arrives. The system updates the cache and acknowledges the client immediately, then schedules an asynchronous flush to the database — often coalescing many writes to the same key into one and batching them. Writes feel instant and the database sees far fewer, larger operations, which is fantastic for write-heavy spikes. But the cache is now the temporary source of truth, and if a cache node dies before its dirty entries flush, those writes are gone — the database never saw them.

Write-through

  • Strong cache↔DB consistency; reads after writes are fresh
  • No data loss on cache crash — the DB already has it
  • Slower writes: every write waits on the database
  • Caches data that may never be read (mitigate with TTL)

Write-back

  • Very fast writes; absorbs write bursts beautifully
  • Batches/coalesces writes → far less DB load
  • Risk of data loss if a node dies before the flush
  • More complex: needs a flush queue, retries, ordering

Worked example: choosing per workload

For a GrabPay transaction ledger, the answer is unambiguous: you do not use write-back for money. A lost flush is a lost payment, which is a financial and regulatory incident. Writes go durably to the database (write-through or, more realistically, write directly to the DB and invalidate the cache), and you accept the extra write latency. For something like a "likes" or view-counter on a TikTok-style feed, write-back shines: losing a handful of counter increments in a rare crash is acceptable, and coalescing thousands of increments into periodic batched writes saves the database from being hammered.

The failure-mode question

"You picked write-back — what happens when that cache node dies?" Have the answer ready: unflushed dirty entries are lost, so write-back is only acceptable when losing the most recent writes is tolerable. To narrow the window you replicate the cache (so a replica holds the dirty data), shorten the flush interval, or persist a write-ahead log. The honest framing is: write-back trades durability for write throughput, and you only spend that trade where the data can afford it.

05 Making room

Eviction: LRU, LFU, FIFO, TTL

A cache lives in RAM, which is finite and expensive. A Redis node with 64 GB cannot hold a 2 TB table. So when the cache fills up and a new entry needs space, something already in the cache has to be thrown out. The rule that decides what goes is the eviction policy. Choosing it well is what keeps the hot working set resident and the hit ratio high.

How eviction actually works

The cache tracks metadata alongside each entry — last-access time, an access counter, or insertion order — and when memory pressure hits a threshold, it consults that metadata to pick a victim. The policies differ only in which metadata they trust.

  • LRU — Least Recently Used. Evict the entry that hasn't been accessed for the longest time. The bet is recency: if you haven't touched something lately, you probably won't soon. Classically implemented as a hash map plus a doubly-linked list — every access moves the entry to the head, so the tail is always the least-recently-used victim, and eviction is O(1).
  • LFU — Least Frequently Used. Evict the entry with the fewest accesses over time. The bet is popularity, not recency. Better for stable hot sets, but it can cling to an item that was popular last week and is now cold, and it needs counters that decay or it never forgets.
  • FIFO — First In, First Out. Evict the oldest-inserted entry regardless of use. Simple, but it will happily evict your hottest key just because it was added first. Rarely the right choice for a read cache.
  • TTL — Time To Live. Not strictly an eviction policy but an expiry: each entry is stamped with a lifetime, and once it elapses the entry is considered gone. TTL is your primary lever for bounding staleness, and it works alongside an eviction policy, not instead of it.

Why LRU usually wins

Product traffic is overwhelmingly recency-skewed: the merchant trending right now, the post going viral this hour, the user active this minute. LRU naturally keeps exactly that resident and quietly discards yesterday's hot items as interest moves on. It's cheap to implement, adapts instantly to shifting popularity, and matches the access pattern of nearly every product workload — which is why it's the common default.

Real systems refine it. Redis exposes a maxmemory ceiling plus a maxmemory-policy with options like allkeys-lru, allkeys-lfu, volatile-lru (evict only keys that have a TTL set), and noeviction (reject writes when full). Redis's LRU is actually an approximation — it samples a few random keys and evicts the least-recently-used among the sample rather than maintaining a perfect global order, which trades a tiny bit of accuracy for a lot of speed and memory. Redis 4+ also ships a proper LFU mode with access-count decay.

LRU (recency)

  • Tracks the current working set as popularity shifts
  • O(1) with map + linked list; cheap to run
  • A one-off scan of cold data can evict your hot set

LFU (frequency)

  • Protects stable, repeatedly-hot keys from a scan
  • Better when popularity is long-lived, not bursty
  • Can cling to formerly-hot keys unless counts decay
Failure mode: the scan that wrecks your hit ratio

A nightly analytics job or a crawler that reads every row once is poison for LRU: it streams cold data through the cache, evicting the genuinely hot keys to make room for entries it will never touch again. Your hit ratio craters until the working set re-warms. Mitigations: keep the scan on a read replica that bypasses the cache, use a policy variant that resists scans, or give scanned keys a tiny TTL so they evict themselves quickly.

06 The hardest problem

Invalidation: keeping cache and DB in sync

There's an old engineering joke that there are only two hard problems in computer science: naming things, cache invalidation, and off-by-one errors. The middle one is the part of caching interviewers probe hardest, because it has no clean answer. The moment you copy data into a cache, you have two copies, and the database can change underneath the cache. A reader who hits the cache then gets a stale value — correct yesterday, wrong now. Invalidation is the act of getting the cache to stop serving the old value. There are three core strategies, and real systems blend them.

1 — TTL expiry (let it lapse)

The simplest approach: stamp every entry with a TTL and let it expire. After the lifetime elapses, the next read misses and reloads fresh from the database. You never explicitly invalidate; you just accept that the cache can be stale for up to the TTL. The mechanism is trivial and self-healing — even if your invalidation logic has a bug, a TTL guarantees the wrong value can't live forever. The cost is a built-in staleness window: a 60-second TTL means a price change can be invisible for up to a minute. You tune the TTL to how much staleness the data can tolerate.

2 — Explicit invalidation on write

When the application writes to the database, it also tells the cache the old value is dead — by deleting the key (preferred, per section 03) or overwriting it. The next read repopulates from fresh data. This is precise and keeps staleness near zero when it works. The catch is that it's only as reliable as your code: every write path that touches that data must remember to invalidate, including the ones added six months later by someone who didn't know the cache existed. Miss one, and you have a silent stale bug that's miserable to track down.

3 — Write-through (never let it drift)

As in section 04, writing through the cache keeps it in lockstep with the database by construction, so there's nothing to invalidate. It buys consistency at the cost of write latency and a cache that may hold rarely-read data.

The consistency reality you must state

A cache is, by definition, eventually consistent with the database unless you go to expensive lengths. The real design question is never "how do I make it perfectly consistent" — it's "how stale is acceptable here?" A Shopee product description can be a few minutes stale and nobody notices. A flash-sale stock count being stale oversells the item and creates refunds. A bank balance being stale is unacceptable. You set the TTL and invalidation aggressiveness per piece of data based on the business cost of staleness, not on a single global rule.

At scale, teams stop relying on every write path remembering to invalidate and instead drive invalidation off the database's change stream: a process tails the write-ahead log via change data capture (Debezium reading the Postgres WAL, the MySQL binlog, or DynamoDB Streams) and emits cache-delete events for every changed row. That makes invalidation a property of the data layer rather than scattered through application code, so a new write path can't forget.

The question you will be asked

"How do you invalidate the cache?" is nearly guaranteed. Don't pick one mechanism — show the layered answer: a baseline TTL so nothing is ever stale forever, explicit delete-on-write (or CDC-driven invalidation) for fast convergence, and an explicit statement of how much staleness this specific data tolerates. Then name the failure mode: a missed invalidation leaves a stale entry until its TTL saves you, which is exactly why you never set TTL to infinite.

07 Thundering herd

Cache stampede

Here's a failure that looks fine in testing and detonates in production. Recall from cache-aside that the first reader of a missing key pays the slow database path. Now make that key extremely popular and let it expire at one instant. A cache stampede (also thundering herd or dogpile) is what happens next: the entry vanishes, and in the moment before anyone repopulates it, thousands of concurrent requests all miss simultaneously and all stampede to the database for the same value at once.

The mechanism, concretely

Suppose Shopee caches the homepage of a flash sale under one key with a 60-second TTL, and 20,000 requests per second are reading it. The cache served all 20,000/sec happily. At second 60 the key expires. Between that instant and the moment the first slow query finishes and writes the value back — say 200 ms — about 4,000 requests arrive, every one of them misses, and every one fires the same expensive query at the database. A query the DB ran comfortably once per minute is now run 4,000 times in 200 ms. The database's CPU and connection pool saturate, queries queue, latency spikes, and the slowdown can cascade outward and take the service down — all triggered by a single key expiring at a bad moment.

The fixes (use several together)

  • Per-key lock / single-flight. On a miss, the first request acquires a short lock for that key (e.g. SET key:lock 1 NX EX 5 in Redis) and is the only one allowed to query the database and repopulate. The others either wait briefly and re-read the cache, or serve the previous stale value. One DB query satisfies the whole herd. This is the most direct fix and the one to name first.
  • Jittered TTLs. Stampedes get worse when many related keys expire at the exact same second (e.g. you warmed 10,000 keys at startup with identical TTLs). Add randomness — ttl = base + random(0, jitter) — so expirations spread out instead of detonating together.
  • Early / background recompute. Don't wait for expiry. A background job (or a probabilistic check that fires more eagerly as the entry nears expiry) refreshes hot keys before they expire, so the cached value is replaced rather than briefly absent. No window of simultaneous misses ever opens.
  • Stale-while-revalidate. When an entry expires, keep serving the stale value to readers while exactly one background request recomputes the fresh one. Readers never see a miss; the data is briefly stale instead. This is the same pattern CDNs and HTTP use via the stale-while-revalidate cache-control directive.
The question you will be asked

"What happens when a hot key expires?" is a direct stampede probe. The answer that lands: name the thundering herd, then give the layered defense — single-flight lock so one request repopulates while the rest wait, jittered TTLs so keys don't all expire together, and stale-while-revalidate so readers see slightly-old data instead of a miss storm. Mention that "never expire the hottest keys; refresh them in the background" is the cleanest option when you can identify them.

08 The hot key

The hot key

The stampede is about time — too many misses in one instant. The hot key problem is about space — too much steady traffic to one key, all of it landing on the single cache node that owns that key. In a distributed cache, keys are spread across many nodes (by consistent hashing, covered in its own lesson). That balances load only if traffic is spread across keys. When one key is wildly more popular than the rest, the node holding it gets a hugely disproportionate share of requests while its peers idle. That node's CPU and network saturate; it becomes a hotspot and a single point of failure for that key — even though the data fits in memory just fine.

This is common in product systems precisely because of the skew that made caching worthwhile. The TikTok video that's exploding right now, the one celebrity's profile, the single flash-sale SKU on Shopee, the surge-pricing zone over Orchard Road during a downpour — all map to one key, and that key's traffic can be orders of magnitude above the average.

The fixes

  • Replicate the hot key. Store copies of the key on several nodes (e.g. write video:42#0 … video:42#9) and have readers pick a random replica. Traffic to the logical key now fans out across N nodes instead of hammering one. The cost is that writes must update all N replicas, so this suits read-mostly hot keys.
  • A small local (in-process) cache. Cache the very hottest keys in the application server's own memory for a few seconds, in front of the distributed cache. The hot key is then served from local RAM — zero network hops — and the distributed node barely sees the traffic. The trade is a second staleness layer (each app instance can hold a slightly different copy for the local TTL), so keep the local TTL short. This two-tier shape is sometimes called a near cache.
  • Split the key. If the hot key is an aggregate (a counter, a leaderboard), shard it into sub-keys across nodes and combine on read — e.g. keep likes:42:shard0..9 and sum them. Spreads both reads and writes.
Detecting it before it hurts

You can't fix a hot key you can't see. Redis offers redis-cli --hotkeys (it leans on the LFU counters) and the MONITOR / slowlog tools to surface skewed access. In practice you instrument per-key request rates and alert when one key crosses a threshold, then apply replication or a local cache to that specific key — the hottest keys are usually a tiny, identifiable set.

09 Misses for nothing

Cache penetration & negative caching

One more failure rounds out the set interviewers like. So far every miss eventually found data in the database and populated the cache, so the next request hit. Cache penetration is when requests repeatedly ask for keys that don't exist anywhere — not in the cache, not in the database. Each one misses the cache (nothing to find), falls through to the database, the database also returns nothing, and because there's no value, nothing gets cached. So the next identical request does it all again. Every such request "penetrates" straight through the cache to the database, and the cache provides zero protection.

This is sometimes accidental (a buggy client requesting a deleted item ID in a loop) but is frequently an attack: a malicious actor hammers your API with random, non-existent user IDs or product IDs precisely because they know those keys can't be cached, weaponizing your database as the bottleneck. For a public Agoda or Shopee API exposed to the internet, this is a real DoS vector.

Fix 1 — negative caching

Cache the absence of data. When the database returns "not found", store a sentinel value (a null marker, an empty object) under that key with a short TTL — say 30 seconds. The next request for the same missing key now hits the cache, sees the negative marker, and returns "not found" immediately without touching the database. The short TTL matters: if the key later starts existing (the item gets created), you don't want to deny it for long.

Fix 2 — a Bloom filter

When the space of possible-but-nonexistent keys is huge (random IDs), negative caching every one of them would itself blow up memory. A Bloom filter is a compact, probabilistic set-membership structure: it can tell you with certainty that a key is not in the set, while occasionally giving a false positive on "maybe present." You load it with all the keys that do exist. On a request, you check the filter first: if it says "definitely not present," you reject immediately without touching cache or database. Only keys that might exist proceed. A few megabytes of Bloom filter can guard against billions of bogus lookups, which is why it's the go-to defense against deliberate penetration.

Three failures, three distinct fixes — don't confuse them

Interviewers love to check you can tell these apart: penetration = keys that don't exist (fix: negative caching / Bloom filter); stampede / breakdown = a hot key that did exist just expired (fix: single-flight lock, jittered TTL, stale-while-revalidate); avalanche = many keys expiring at once and the cache layer briefly collapsing (fix: jittered TTLs plus a circuit breaker / fail-open to the DB). Same symptom — a flood of DB load — three different root causes.

10 The cache hierarchy

Where caches live

We've been talking about one cache tier — a Redis cluster in front of the database. In a real request, data passes through several caches stacked from the user inward, and a good answer names the layers and what each is for. The guiding principle: cache as close to the user as the data's freshness allows, because the closest cache is the cheapest and fastest, and every layer it satisfies is load the layers behind it never see.

  • Client / browser cache. The data lives on the user's own device — HTTP response caching, an in-app store of the user's profile. Zero network cost and instant, but you can't invalidate it (you can't reach into a phone), so it's only for data that can be stale or is governed by short cache-control lifetimes.
  • CDN edge cache. A content delivery network (Cloudflare, Akamai, CloudFront) caches responses at points of presence physically near the user. Built for static assets — images, JS, the Agoda hotel photos — and increasingly for cacheable API responses. Cuts both latency and origin traffic dramatically. (This is a lesson of its own.)
  • In-process / local cache. A cache inside the application server's own memory (a Caffeine map in a JVM, a process-local dict). Fastest possible server-side lookup, no network hop — but each instance has its own copy, so it's small, short-lived, and best for the very hottest keys (the hot-key local cache from section 08).
  • Distributed cache tier. The shared Redis/Memcached cluster every app instance talks to — the main event of this lesson. One coherent view of the hot data, survives app restarts, scales independently. One network hop, but still vastly faster than the database.
Connecting the course

This hierarchy ties directly to the load-balancing and CDN lessons: the load balancer fronts the stateless app tier, the CDN sits at the edge, and the distributed cache sits between the app and the database. Caching is what makes the stateless-services pattern affordable — when servers hold no session state, every instance can share the same cache tier, and you scale the app horizontally without each new instance pounding the database.

★ Review

Summary & self-test

The whole topic compresses to one sentence: a cache trades a little staleness for a lot of speed and a lot less database load — and every interview point is about controlling that staleness and surviving the failure modes a cache creates. If you can draw the cache-aside read path, name the four write patterns, and explain invalidation, stampede, hot key, and penetration with a fix for each, you can carry this round.
