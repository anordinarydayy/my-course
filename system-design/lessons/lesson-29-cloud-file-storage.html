Design Cloud File Storage

A sync service like Google Drive keeps the same files byte-for-byte identical across every device a user owns — while uploading as little as physically possible. The entire design turns on one idea: stop treating a file as one blob, split it into content-addressed blocks, deduplicate them, and sync only the blocks that actually changed.

About these notes: original study material on designing a cloud file-sync service — a standard, widely-documented system-design topic. 00 The problem

Why sync is hard

Imagine you keep a 2 GB folder of design assets on your laptop, your phone, and a desktop at the office. You rename one file, fix a typo in a contract, and add a new photo. A naive sync service would re-upload everything that looks "dirty" — and on a typo fix in a 40 MB document, that means pushing 40 MB up the wire to change three characters. Multiply that across millions of users editing all day and you have melted your bandwidth bill and made the product feel slow.

A cloud file storage service (Drive, Dropbox, OneDrive, iCloud Drive) is a system that stores a user's files in the cloud and keeps local copies on each of their devices continuously consistent. The hard part is never the storing — object storage solved that. The hard part is the sync loop: detecting what changed, transferring the minimum bytes, and reconciling the result on every other device without corrupting anyone's data.

The bottlenecks we will chase, in order, are:

  • Bandwidth. Re-uploading whole files on tiny edits is the cardinal sin. We fix it with chunking and delta sync.
  • Storage cost. Storing the same byte ranges over and over — across versions and across users — wastes petabytes. We fix it with content-addressed deduplication.
  • Freshness. A device must learn that a file changed quickly, without melting the server by polling. We fix it with a long-poll notification service.
  • Correctness. Two offline devices edit the same file. We must not silently lose one of them. We fix it with versioning and conflict copies.
01 Scope it

Pin down the requirements

An interviewer wants to watch you carve the problem down to something buildable in 40 minutes. Cloud storage is enormous; name what is in scope and what you are deferring.

Functional requirements

  • Upload & download files of any size, including very large ones (multi-gigabyte video).
  • Sync across devices. A change on one device appears on all the user's other devices automatically and quickly.
  • Save bandwidth. Editing part of a file should transfer only the changed part, not the whole file.
  • Versioning. Keep a history of previous versions so a user can roll back.
  • Sharing & permissions. Share a file or folder with another user as viewer or editor.
  • Offline edits. Work disconnected; sync (and resolve conflicts) on reconnect.

Non-functional requirements

  • Durability above all. A user's files must never be lost — target 99.999999999% ("eleven nines") durability, the object-storage standard.
  • Availability. The service should be highly available; a brief metadata outage is more tolerable than data loss.
  • Bandwidth efficiency & low sync latency — the product's whole reason to exist.
  • Scale to hundreds of millions of users and exabytes of data.
  • Consistency model: a single file should converge to one agreed state; we will accept brief eventual consistency during propagation but never silent loss.

Assumptions to state aloud

Lock down a few numbers so the estimate has something to chew on: assume roughly 50 million daily active users, an average of 2 connected devices each, an average stored footprint around 10 GB per user, and a working file population dominated by documents and photos with a long tail of large media. I will tilt examples toward a Singapore context — say Sea/Shopee teams syncing campaign creative, or Grab ops folks sharing rider-incident PDFs across phone and laptop.

Interview tip

The single highest-leverage sentence you can say up front is: "the design is fundamentally about moving the fewest bytes, so I'll split files into content-addressed blocks and sync only the deltas." If you lead with that, every later decision (dedup, metadata/block split, delta sync) reads as a consequence rather than a surprise.

02 Estimation

Back-of-the-envelope estimate

Rough numbers tell you where the system will hurt and which component must scale hardest. Do them out loud; precision matters less than the order of magnitude and what it implies.

Storage

50M users × 10 GB = 500 PB of logical data. But deduplication and delta sync are exactly the tools that shrink the physical footprint. In practice cross-user and cross-version dedup commonly reclaims 30–50%; assume a conservative 40%, leaving roughly 300 PB physical, before counting the redundant copies object storage keeps internally for durability.

Write / upload throughput

Suppose each active user changes about 100 MB of new or modified content per day. That is 50M × 100 MB = 5 PB ingested per day. Spread over 86,400 seconds that is about 58 GB/s of sustained upload — and real traffic is peaky, so provision for several times that. This is why we must never upload bytes that already exist.

Metadata QPS

Metadata operations (list a folder, fetch a file's block list, commit a new version, poll for changes) dwarf byte transfers in count. If each of 50M DAU triggers ~200 metadata calls/day, that is 10 billion/day ≈ 115,000 QPS average, with peaks easily 5–10× higher. The metadata service, not the byte store, is the throughput-critical tier — a crucial realization that justifies the architecture split in step 04.

Block sizing trade-off

If we chunk files into 4 MB blocks, then 300 PB physical / 4 MB ≈ 75 billion blocks. Each needs a metadata row (hash, size, location). At ~100 bytes/row that is ~7.5 TB of block metadata alone — small relative to the data, but enough that the metadata DB must be sharded.

Why block size matters

Smaller blocks = finer-grained dedup and cheaper deltas, but far more metadata rows and per-block overhead. Larger blocks = less metadata but you re-upload more on every edit. 4 MB is a common sweet spot; the number itself is less important than being able to reason about the trade-off when the interviewer pokes it.

03 Contract

The API and a data-model sketch

Sketching the API and tables early forces the abstractions to be concrete and gives you a vocabulary for the rest of the interview. The defining choice you will reveal here is that file content and file metadata travel on separate paths.

Client-facing API

The client never blindly PUTs a file. It first computes the file's blocks and their hashes, asks the server which blocks are missing, uploads only those, then commits a new version pointing at the full block list.

// 1. Negotiate: which blocks does the server already have?
POST /v1/files/{fileId}/blocks:check
  body: { "blockHashes": ["a91f…", "7c30…", "d4e1…"] }
  resp: { "missing": ["7c30…"] }          // server already has the rest

// 2. Upload only the missing blocks (often straight to object storage
//    via a short-lived presigned URL, bypassing the app tier)
PUT  /v1/blocks/{blockHash}                 // body = raw 4 MB block bytes

// 3. Commit a new version referencing the ordered block list
POST /v1/files/{fileId}/versions
  body: { "blockList": ["a91f…","7c30…","d4e1…"],
          "baseVersion": 41, "mtime": 1717000000 }
  resp: { "version": 42 }                   // 409 if baseVersion is stale

// 4. Long-poll for changes since a cursor
GET  /v1/changes?since=cursor_8842          // blocks until something changes
  resp: { "changes": [ {fileId, version, op:"updated"} ], "cursor": "cursor_8907" }

// 5. Download blocks the client is missing
GET  /v1/blocks/{blockHash}                 // presigned read from object storage

Data model

Three entity groups: the namespace (folder tree), the file/version records, and the block catalog. Note that a version is just an ordered list of block hashes — that indirection is what makes dedup and delta sync possible.

files        (fileId PK, ownerId, name, parentFolderId,
              currentVersion, isFolder, deleted)
versions     (fileId, version, blockList[], size, mtime, createdBy,
              PRIMARY KEY (fileId, version))
blocks       (blockHash PK, sizeBytes, objectStoreKey, refCount)
shares       (fileId, granteeUserId, role)        // role: viewer | editor
devices      (deviceId, userId, lastSeenCursor)   // sync watermark per device

The blockList in versions stores references (hashes), never bytes. The bytes live once in the blocks catalog, which points into object storage. refCount tracks how many versions/files still reference a block so we know when it is safe to garbage-collect.

Follow-up they'll ask

"Why does the client check which blocks are missing before uploading?" Because that single round trip is what turns an edit into a delta. If the server already has 9 of a file's 10 blocks, the client uploads 1. Skip the check and you have rebuilt the naive whole-file uploader from step 00.

04 The core split

Split metadata from blocks

This is the architectural keystone. Two kinds of data have completely different access patterns, so we store them in completely different systems.

The metadata service

The metadata service owns the structured truth: the folder tree, each file's version history, the ordered block list per version, permissions, and per-device sync cursors. Access is small, frequent, transactional, and relational — list a folder, commit a version, check a permission. That maps to a strongly-consistent, sharded relational store: PostgreSQL or MySQL sharded by userId, or a managed NewSQL store like Spanner/CockroachDB when you want transactions across shards. It must be ACID enough that "commit version 42" either fully happens or doesn't — a half-written block list is a corrupt file.

The block store

The block store owns the bytes: opaque, immutable, content-addressed 4 MB blocks. Access is large-payload, write-once-read-many, and needs eleven-nines durability but not transactions. That is exactly what object storage is for: Amazon S3, Google Cloud Storage, or an on-prem equivalent. The object key is literally the block's content hash, so the store is a giant content-addressed key-value map of blockHash → bytes.

Why split at all?

Because forcing both into one database is the classic mistake. Put 4 MB blobs in PostgreSQL and you destroy its cache and bloat the WAL; put the folder tree in S3 and you cannot do a transactional "rename folder" or an efficient "list children." Splitting lets each side scale on its own axis: object storage scales bytes near-infinitely and cheaply; the metadata DB scales QPS via sharding. They are even on different cost curves — cold object storage is pennies per GB-month.

Metadata service (RDBMS)

  • Strong consistency & transactions for "commit version"
  • Rich queries: list folder, version history, permission checks
  • Small rows, very high QPS → shard by userId
  • Costlier per byte; bad at holding large blobs

Block store (object storage)

  • Eleven-nines durability, near-infinite byte scale
  • Cheap per GB; tiers to cold storage for old versions
  • Immutable, content-addressed, write-once
  • No transactions, no "list children" semantics
Follow-up they'll ask

"Metadata vs block storage — why split? Why not just stuff everything in one database?" Answer with access patterns: metadata is tiny, transactional, query-rich, and QPS-bound; blocks are huge, immutable, durability-bound, and byte-bound. One store cannot be optimal for both. The split also lets clients upload/download bytes directly to object storage via presigned URLs, keeping your app servers out of the multi-gigabyte hot path.

05 Chunk & dedup

Chunking and deduplication

Now the mechanism that makes everything else possible. We never store a file as one object. We split it into blocks and address each block by a hash of its own contents.

How chunking works

Chunking is splitting a file into blocks. The simplest scheme is fixed-size chunking: cut every 4 MB. For each block, compute a cryptographic hash of its bytes — say SHA-256 — and use that hash as the block's identity. This is content addressing: the name of a block is its content. Two blocks with identical bytes get identical hashes, anywhere in the system, forever.

A file then becomes an ordered list of hashes — its block list. A 40 MB document is 10 hashes; a 1 GB video is ~256 hashes. The file's content is fully reconstructable by fetching those blocks in order and concatenating them.

How block-level deduplication works

Deduplication means storing each unique block exactly once. Because the block's key is its content hash, dedup is automatic: before uploading a block, the client (or server) checks whether blocks[blockHash] already exists. If it does, nobody uploads or stores it again — the new version simply references the existing block and bumps its refCount.

This collapses redundancy on two axes at once:

  • Across versions: editing one block of a 100-block file means 99 blocks are unchanged, so v43 shares 99 blocks with v42. The history of a file costs barely more than the file itself.
  • Across users: if a Shopee marketing team of 30 people all save the same 200 MB brand-guidelines PDF to their Drives, the bytes are stored once. The other 29 copies are pure metadata references.

Worked example

A GrabPay finance analyst stores a 16 MB quarterly report → 4 blocks [A,B,C,D]. A teammate downloads it, appends a paragraph that lands inside block D, and re-saves. New block list: [A,B,C,D']. Blocks A, B, C already exist, so only D' (4 MB) is uploaded and stored. Net new storage: 4 MB for a 16 MB "new" file, and 4 MB on the wire instead of 16.

Fixed vs variable-size chunking

Fixed-size chunking has an Achilles heel: insert a single byte at the front of a file and every subsequent block shifts, so every hash changes and dedup collapses to zero — you re-upload the whole file. Variable-size (content-defined) chunking fixes this by choosing block boundaries based on the content itself (a rolling hash like Rabin fingerprinting cuts a boundary wherever the rolling checksum hits a pattern). An inserted byte then only disturbs the one block around it; the rest keep their boundaries and their hashes.

Fixed-size blocks

  • Dead simple, trivially parallel to compute
  • Great for append-only or block-replace edits
  • Breaks on byte insertion: all later blocks shift, dedup lost

Variable-size (CDC)

  • Boundaries follow content → robust to inserts/shifts
  • Best real-world dedup ratios
  • More CPU (rolling hash) and more complex to implement
Under the hood — hash collisions

Content addressing trusts that "same hash ⇒ same bytes." With SHA-256 the odds of an accidental collision are astronomically smaller than a disk silently corrupting data, so it is treated as safe. Paranoid systems verify a full byte comparison on a hash match, or salt hashes per namespace to blunt deliberate collision attacks.

06 The payoff

Delta sync

Chunking and dedup set the table; delta sync is the meal. Delta sync means that when a file changes, the client uploads only the blocks that differ from the previous version — never the whole file.

How the client computes the delta

The crucial point is that the diff happens on the client, so the network only ever sees the minimum. When a watched file changes on disk, the local sync agent:

  • Re-chunks the file and computes the hash of each block.
  • Compares the new block list to the previous version's block list it has cached locally. Blocks whose hashes match are unchanged and skipped entirely.
  • Sends the new hash list to the metadata service's blocks:check endpoint, which replies with the subset the block store does not already have (catching cross-user dedup too).
  • Uploads only that final missing subset, then commits a new version referencing the full block list.

So a one-line edit to a 40 MB document re-uploads a single 4 MB block (or less, with variable chunking) — a 90%+ bandwidth saving, and that is the conservative case. The other device, in step 07, downloads the same single block to catch up.

Follow-up they'll ask

"How do you avoid re-uploading the whole file on a one-line edit?" This section is the answer: chunk locally, hash each block, compare against the cached previous block list, ask the server which of the changed blocks it still lacks, and upload only those. The diff is computed client-side precisely so the wire only ever carries the delta. Add variable-size chunking if they push on "what about inserting text at the top of the file?"

07 Propagation

Notify and reconcile

Uploading the delta is half the loop. The other devices have to learn a file changed and pull the new blocks — fast, and without hammering the server.

Why not just poll?

If every device polled "anything new?" every few seconds, 100M devices would generate a flood of mostly-empty requests. The fix is a dedicated notification service that holds a request open until there is actually something to report.

Long-polling, step by step

Long-polling works like this: a device opens a GET /changes?since=cursor request. The server does not answer immediately if nothing has changed — it holds the connection open (parked for, say, up to 30 seconds). The instant a relevant file commits a new version, the notification service responds to all parked connections that care about that file/namespace, handing back the list of changed file IDs and a new cursor. The device immediately re-opens a new long-poll, so it always has one ear to the ground. This gives near-real-time push using only plain HTTP, which sails through corporate firewalls. A persistent WebSocket is an alternative when you want true server push and can keep millions of sockets open.

Reconcile on the receiver

When Device B is notified that file F advanced to v43, it does not download v43 wholesale. It fetches v43's block list from the metadata service, diffs it against the blocks it already has on disk, and downloads only the missing blocks from the block store — the mirror image of the upload-side delta. It then reassembles the file locally and updates its cursor. The same dedup logic runs in reverse, so freshness costs the network only the delta, both ways.

What the cursor buys you

The cursor (a monotonic change watermark per user/namespace) is what makes sync resumable. A device that was offline for a week reconnects with its last cursor and receives exactly the changes it missed, in order — no full re-scan. Store it in the devices table as lastSeenCursor.

Failure mode — a notification is lost

Push is an optimization, never the source of truth. If the notification service drops or a device misses an event, no data is lost: the device's cursor is still behind, so its next long-poll (or a periodic safety-net poll every few minutes) returns the missed changes. The cursor makes the system self-healing — pushes make it fast, cursors make it correct.

08 Correctness

Conflicts and versioning

The nastiest bug in any sync system is silent data loss. It happens when two devices edit the same file while disconnected from each other, then both try to commit. Whatever we do, we must never let one person's work vanish.

How conflicts are detected

Every commit carries the baseVersion it was derived from (recall the API in step 03). The metadata service accepts a commit only if baseVersion equals the file's current version — classic optimistic concurrency control. If Device A and Device B both edited v41 offline: A reconnects first and commits v42 (base 41 → matches → accepted). B then reconnects and commits with base 41, but the current version is now 42 → mismatch, so the server rejects it with 409 Conflict.

How conflicts are resolved

On a 409, the client does not overwrite and does not throw away the user's edit. The standard, user-safe resolution is a conflict copy: B's version is saved as a sibling file, e.g. report (conflicted copy — Bao's MacBook 2026-05-31).docx. Both edits now exist; a human decides which to keep or how to merge. This is deliberately dumb-but-safe — the system refuses to guess and refuses to lose data.

Richer schemes exist for collaborative editors (Google Docs uses Operational Transformation; many modern apps use CRDTs to auto-merge concurrent edits). But for a general file-sync service holding arbitrary binary files — a PSD, a video, a ZIP — automatic merge is impossible, so conflict copies are the honest answer.

Versioning as the safety net

Because every commit creates a new immutable version (an ordered block list), the full version history is free — old versions just reference their old blocks, which dedup keeps cheap. A user can roll back to any prior version; an accidental "select-all, delete, save" is undoable. Versions also make the whole sync loop idempotent: re-applying a known version is a no-op.

Follow-up they'll ask

"How do you handle two devices editing at once?" Optimistic concurrency on baseVersion: first commit wins, the second gets a 409 and becomes a conflict copy. Stress that the design prioritizes never losing data over automatic merging — for arbitrary binary files, auto-merge is impossible, so you preserve both and let a human resolve. Mention OT/CRDTs only as the path for live collaborative text editing.

09 The 10x question

Scale, failure, and what breaks at 10x

The closing move is to push your own design until it cracks, then say how you would shore it up. From 50M to 500M users, here is where it strains.

Sharding the metadata service

The metadata DB is the QPS bottleneck (step 02). Shard by userId so that all of one user's files, versions, and permissions live on one shard — that keeps "list my folder" and "commit my version" as single-shard transactions. A shared file is read via the owner's shard plus a shares lookup. Hot shards (a viral shared folder) get split or fronted by a read cache. Avoid sharding by fileId alone, or every folder listing fans out across shards.

Resumable transfers for large files

A 5 GB video over a flaky mobile connection will drop mid-upload; restarting from zero is unacceptable. Block-based design makes transfers naturally resumable: each 4 MB block is uploaded independently and idempotently (its key is its hash, so re-PUTting it is harmless). On reconnect the client re-runs blocks:check, sees which blocks landed, and uploads only the rest. This is exactly how S3 multipart upload and the tus protocol behave. The version isn't committed until all blocks are present, so a partial upload is never visible.

Cold storage for old versions

Version history is cheap thanks to dedup, but blocks referenced only by old versions are rarely read. Tier them down: keep current-version blocks in standard object storage, and migrate blocks that no live version touches to a cheaper cold tier (S3 Glacier / GCS Coldline), accepting higher retrieval latency for a big cost cut. A background garbage collector walks the blocks catalog and deletes any block whose refCount has hit zero — the only safe way to reclaim space without orphaning a referenced block.

Failure modes

  • Block store node dies: object storage replicates each block across devices/AZs, so a node loss is invisible — the eleven-nines durability is doing its job.
  • Metadata shard dies: run each shard as a replicated cluster (primary + replicas). On primary failure, a replica is promoted; the window of unavailability affects only that shard's users, and clients retry. Because object storage holds the actual bytes, no file content is at risk during a metadata outage.
  • Client crashes mid-sync: nothing is committed until all blocks exist and the version commit succeeds, so a crash leaves a partial upload that the next blocks:check simply resumes. No corrupt half-file ever becomes visible.
  • Garbage-collection race: a block being deleted (refCount 0) just as a new version references it. Guard with a transactional refCount bump or a grace period before reclaiming, so a block is never deleted out from under a live reference.

What breaks at 10x

The notification service becomes the new pressure point: holding 500M+ open long-polls or sockets demands a fleet of connection-handling nodes with sticky routing and a fan-out layer (often Kafka or Redis pub/sub) carrying change events to whichever node holds each device's connection. The blocks:check path also gets hot, so cache "does this hash exist?" lookups in Redis in front of the metadata DB. And cross-user dedup raises a privacy nuance — a global dedup namespace can leak existence of content; isolate dedup per-user (or per-org) when that matters, trading some savings for safety.

Deep-dive the interviewer may push for

Be ready for: encryption (encrypt blocks at rest; note that client-side end-to-end encryption defeats cross-user dedup, since identical plaintext encrypts to different ciphertext — a real trade-off); consistency of shared folders (a member should see a coherent view, so propagate share changes through the same versioned metadata path); and mobile constraints (only sync on Wi-Fi, throttle background bandwidth, sync metadata first so the file tree appears before bytes arrive).

★ Review

Summary & self-test

The whole system compresses to one sentence: split files into content-addressed blocks, store each unique block once, sync only the blocks that changed, and keep structured metadata separate from opaque bytes. If you can draw the metadata/block split and walk an edit through delta sync to another device, you can carry this interview.
