08Lesson · System Design Interview

API & Gateway Design

Before any service does real work, clients reach it through a contract — and in a world of dozens of services, through a single front door. This lesson covers choosing REST against gRPC, what an API gateway absorbs, how to version and paginate without breaking callers, and the idempotency keys that stop a retried payment from charging a rider twice.

About these notes: original study material on API and gateway design — a standard, widely-documented engineering topic. The prose, examples, numbers, and diagrams here are my own; nothing is copied from any paid course or book. Three diagrams are interactive — press Play (or step with Next) to watch the flow. Examples lean toward Singapore product and fintech systems (Grab, Shopee, GrabPay, SeaMoney, Agoda), but the concepts are general.
00 The problem

Why a front door?

Picture a super-app like Grab in its early days: one mobile client, one backend, one contract. Life is simple. Then the company grows. Now there is an iOS app, an Android app, a web booking site, a partner-merchant portal, and a stable of third-party developers building on a public API. Behind them sits not one backend but a dozen microservices — rides, payments, food, wallet, identity, promos, notifications. Suddenly every client must know the address of every service it talks to, and every service must independently solve the same boring problems.

Let each client call each service directly and you pay a tax that compounds:

  • Duplicated cross-cutting work. Authentication, rate limiting, TLS termination, logging, and request validation get re-implemented in every service — inconsistently, so a bug fixed in one is still live in five others.
  • Chatty round trips. To render one ride-history screen, the mobile app fires separate calls to the trips service, the fare service, and the driver-profile service. On a flaky 4G connection in a Jakarta tunnel, three sequential round trips is a visibly slow screen.
  • A brittle, leaky contract. Clients now depend on the internal shape and network location of every service. Move a service, rename a field, or split it in two, and you break shipped mobile apps you can no longer update.
  • An exposed attack surface. Every internal service is reachable from the public internet, each one a door you must independently harden.

The fix is to put one component in front of everything — a single entry point that owns the cross-cutting concerns once and hides the service topology behind it. That component is the API gateway, and it is the spine of this lesson.

01 The single entry point

The API gateway

An API gateway is a server that sits at the edge of your system and is the one address every client talks to. It receives every inbound request, applies a pipeline of cross-cutting policies, then routes the survivor to whichever internal service owns that path — and relays the response back. The clients never learn that ten services exist; they see one coherent API at one hostname, say api.grab.com.

What the gateway absorbs

The gateway's value is that it owns, in one place, the work that would otherwise be scattered and duplicated:

  • Authentication & authorization. Validate the caller's token once at the edge, reject anonymous traffic, and pass a trusted identity inward so downstream services don't each re-verify credentials.
  • Rate limiting & quotas. Throttle abusive callers before they touch a single service (this is the limiter from Lesson 18, deployed at the door).
  • TLS termination. Decrypt HTTPS once at the edge so internal hops can run cheaper.
  • Routing. Map a public path like /v1/rides to the rides service and /v1/wallet to the wallet service.
  • Request aggregation / fan-out. Combine several internal calls into one client response, cutting round trips.
  • Response shaping, logging, metrics, and tracing. Tag every request with a correlation ID and emit uniform logs and latency metrics.

How a request flows through it

Mechanically, a request is processed as an ordered chain of filters (sometimes called middleware or handlers). Each filter inspects or mutates the request, then either passes it to the next filter or short-circuits with a response. A typical inbound order is: terminate TLS → parse and normalize the request → authenticate → rate-limit → route-match → forward upstream. On the way back the chain runs in reverse: the upstream response is shaped, compressed, logged, and returned. Short-circuiting matters — if authentication fails at filter three, the request never reaches routing or the service, which is exactly the protection you wanted.

What implements it

In practice you rarely write a gateway from scratch. NGINX and Envoy are the workhorse reverse proxies; Envoy underpins service meshes like Istio. Cloud-managed options include AWS API Gateway, Kong, and Apigee. The knobs you tune are timeouts (connect, read, upstream), retry and circuit-breaker policy, max request body size, and the route table mapping paths to upstream clusters.

Failure mode

The gateway is now a single point of failure and a latency tax on every request — if it dies, everything is unreachable. So you never run one: you run a fleet of stateless gateway instances behind a load balancer, across availability zones, and you keep the gateway logic thin so a slow plugin can't stall the whole edge. The blast radius is real, which is precisely why the gateway must be boringly reliable.

Interviewer follow-up

"Isn't the gateway a bottleneck and a single point of failure?" Answer with the deployment: a horizontally-scaled, stateless gateway fleet behind a load balancer, health-checked, multi-AZ, with conservative per-route timeouts so one slow upstream can't exhaust connections for the rest. The gateway adds a hop, but it removes far more duplicated work than it costs.

02 Protocol choice

REST vs gRPC vs GraphQL

Before the gateway can route anything, you must decide what shape the conversation takes. The protocol you pick at each boundary is a real interview decision, and the right answer differs between the public edge and the internal mesh.

REST — the lingua franca of the edge

REST (Representational State Transfer) is a resource-oriented style over plain HTTP. You model the domain as nouns — a ride, an order, a wallet — addressed by URLs, and act on them with HTTP verbs: GET to read, POST to create, PUT/PATCH to update, DELETE to remove. Outcomes ride on standard status codes (200, 201, 404, 409, 429), and bodies are usually JSON. Its superpowers are ubiquity (every language, every tool, every browser speaks it), cacheability (a GET is safely cacheable by CDNs and proxies), and human readability. That makes REST the default for public and edge APIs — the contract you hand to third-party developers and your own mobile apps.

gRPC — the internal speed layer

gRPC is a binary RPC framework: you define your service and message types in a Protocol Buffers (.proto) schema, a code generator produces strongly-typed client and server stubs, and calls travel as compact binary over HTTP/2. The mechanism is what makes it fast: Protobuf serializes to a fraction of the bytes of JSON, HTTP/2 multiplexes many calls over one connection (no head-of-line blocking at the connection level), and the generated stubs make a remote call look like a local function. gRPC also supports streaming — server-streaming, client-streaming, and bidirectional — over a single long-lived connection. The cost is that it is not browser-native and not human-readable on the wire, so it shines internally, service-to-service: the rides service calling the pricing service ten thousand times a second wants gRPC's low latency and strict contract, not JSON over HTTP/1.1.

REST over HTTP/JSON

  • Universal: every client, browser, and tool speaks it
  • GETs are cacheable at the CDN and proxy layer
  • Human-readable; trivial to debug with curl
  • Looser contract — easy to evolve, easy to drift
  • Heavier on the wire; one request per round trip on HTTP/1.1

gRPC over HTTP/2 + Protobuf

  • Compact binary payloads, low latency, high throughput
  • Strict, generated, typed contract — fewer integration bugs
  • First-class streaming and multiplexed connections
  • Not browser-native (needs a proxy/grpc-web); opaque on the wire
  • Schema must be shared and versioned between both sides

GraphQL — let the client shape the response

REST has a chronic annoyance: a fixed endpoint returns a fixed shape, so a screen often over-fetches (gets fields it ignores) or under-fetches (must make follow-up calls). GraphQL answers this with a single endpoint where the client sends a query describing exactly the fields it wants, possibly spanning several resources, and gets back precisely that shape. For a data-rich screen — a Shopee product page pulling listing, seller rating, shipping options, and recommendations — one GraphQL query can replace four REST calls.

The trap is the N+1 problem: a query that fetches a list of 50 orders and, for each, the buyer's name can naively trigger 1 list query plus 50 buyer lookups. The fix is a batching loader (the DataLoader pattern) that collects all 50 buyer IDs within a tick and resolves them in a single batched query. GraphQL also complicates caching (it is mostly POST, so HTTP caching doesn't apply for free) and rate limiting (one query can be arbitrarily expensive, so you score query complexity instead of counting requests).

Interviewer follow-up

"REST or gRPC between these two services, and why?" The crisp answer: REST at the public edge for ubiquity and cacheability; gRPC inside the mesh for latency, throughput, and a strict typed contract; GraphQL when one client must assemble many resources and you want to kill over- and under-fetching — while guarding the N+1 with a batching loader. Naming where each lives, not just what each is, is what scores.

03 Under the hood

The gateway as a reverse proxy

A gateway is, at its core, a reverse proxy with policy. It is worth slowing down on what "reverse proxy" actually means and the work it quietly does, because interviewers probe the mechanism, not just the name.

Forward vs reverse proxy

A forward proxy sits in front of clients and makes requests on their behalf to the wider internet — think a corporate proxy that filters employee web traffic. A reverse proxy sits in front of servers: clients believe they are talking to the origin, but the proxy intercepts every request and decides how to fulfill it. The gateway is a reverse proxy because it fronts your services — clients address it, and it speaks for the backend.

TLS termination

TLS termination means the proxy holds the certificate and private key, decrypts inbound HTTPS, and hands plaintext to the backend over the trusted internal network. This centralizes certificate management (renew in one place, not in every service) and saves each service the CPU cost of the handshake. If the internal network is not trusted — common in fintech — you re-encrypt for the internal hop (TLS re-origination) or run mTLS between services.

Caching, compression, and buffering

Because the proxy sees every response, it can cache safe GETs keyed by URL and headers, returning a stored copy without touching the origin — a static promo catalog can be served straight from the edge. It can compress responses (gzip or Brotli) so a fat JSON payload travels small to a mobile client. And it buffers: it can read a slow client's full request before opening a backend connection, so a trickling upload doesn't tie up a precious backend worker for seconds.

Header rewriting and origin hiding

The proxy rewrites headers as traffic crosses the boundary. It strips internal headers, injects X-Forwarded-For (the real client IP), adds a correlation ID, and removes Server banners that would reveal your stack. By presenting one hostname and never exposing internal addresses, it performs origin hiding — an attacker who probes api.grab.com cannot tell whether one service or fifty sit behind it, and cannot reach any of them directly.

Under the hood

Reverse proxies maintain a pool of keep-alive connections to each upstream. Instead of opening a fresh TCP (and TLS) connection per request, the proxy reuses warm connections, which slashes tail latency. Watch the knob: if the pool is too small under load, requests queue at the proxy; too large, and you can overwhelm a fragile backend. NGINX exposes this as keepalive in an upstream block; Envoy as connection-pool circuit breakers.

04 Tailored doors · aggregation

BFF & request aggregation

A single generic gateway eventually strains under a new tension: an iOS app, a smartwatch, and a partner web dashboard want very different response shapes from the same services. Bloat the one gateway with per-client logic and it becomes a tangle owned by no one. The pattern that resolves this is the Backend-for-Frontend (BFF): a thin, dedicated gateway per client type, each shaping responses for exactly that experience.

Grab might run a mobile BFF, a web BFF, and a partner BFF. The mobile BFF trims payloads to the minimum a phone needs; the partner BFF exposes a stable, conservative contract for external integrators. Each is owned by the team that owns that client, so they evolve independently.

The headline trick a BFF performs is request aggregation (fan-out / fan-in). Instead of the client making three sequential calls to render a screen, it makes one call to the BFF; the BFF fires the three internal calls in parallel, waits for all to return, stitches the results into one response, and sends it back. Three round trips over a slow mobile link collapse into one.

Failure mode & the 10x question

Aggregation couples the response to its slowest dependency: if the driver service is slow, the whole screen waits. The defenses are per-call timeouts, partial responses (return trips and fare, mark the driver block as degraded), and a circuit breaker that stops calling a failing dependency and serves a fallback. At 10x traffic the BFF's own fan-out can hammer a downstream into the ground — so add concurrency limits and caching at the BFF, or the convenience layer becomes the outage amplifier.

05 Who are you?

Authentication & tokens

The gateway must answer "who is this caller, and may they do this?" before forwarding anything. Several mechanisms coexist, each fitting a different caller.

OAuth2 — access & refresh tokens

OAuth2 is the framework that lets a user authenticate once and the system issue two tokens: a short-lived access token the client attaches to every API call, and a longer-lived refresh token used only to obtain new access tokens when they expire. The split is deliberate. The access token is sent constantly, so if it leaks the damage is bounded by its short lifetime (often 5 to 15 minutes). The refresh token is sent rarely and can be revoked server-side, so it can live for weeks without the same exposure.

JWT — stateless signed claims

The access token is commonly a JWT (JSON Web Token): a base64-encoded header and payload of claims (user id, roles, expiry exp) plus a cryptographic signature. The gateway verifies the signature with the issuer's public key and trusts the claims without any database lookup — that is what "stateless" buys you: no round trip to a session store on every request. The catch is the flip side of statelessness: a JWT is valid until it expires, so you cannot easily revoke it mid-life. If a SeaMoney user reports a stolen phone, you can't instantly kill an already-issued token. The mitigations are a short TTL plus refresh (the token self-expires fast) or a denylist of revoked token IDs the gateway checks — which reintroduces a lookup, trading some of the statelessness back.

API keys & mTLS

For machine callers — a partner merchant's server calling the GrabPay API — an API key (a long random secret tied to a client account) is simpler than a user login flow; you scope and rate-limit per key. For internal service-to-service calls inside the mesh, mTLS (mutual TLS) has both sides present certificates, so the pricing service can cryptographically prove it really is the pricing service. mTLS authenticates the service identity, while OAuth/JWT authenticates the user identity — production systems use both at different layers.

Interviewer follow-up

"Your JWTs can't be revoked — how do you log out a stolen device immediately?" Strong answers combine a short access-token TTL (so it dies on its own within minutes) with refresh-token revocation in a store, optionally backed by a small denylist of token IDs the gateway checks for the rare immediate-kill case. Acknowledge that this trades a slice of statelessness for control — that honesty is the point.

06 Don't break callers

Versioning a contract

Once a third party or a shipped mobile app depends on your API, you cannot change it freely — there are clients in the wild you cannot force-update. Yet the product must evolve. Versioning is how you change the contract without breaking yesterday's clients.

Three places to put the version

  • URI versioning: /v1/payments then /v2/payments. Explicit, trivially visible in logs and curl, and the most common choice for public APIs.
  • Header versioning: a custom header like Api-Version: 2. Keeps URLs clean but is invisible in a browser address bar and easy to forget.
  • Content negotiation: an Accept: application/vnd.grab.v2+json media type. The most RESTfully "pure" but the least obvious to most developers.

Backward compatibility and the additive rule

The deeper skill is avoiding new versions at all. Most evolution can be backward compatible if you follow one discipline: only add, never break. Adding a new optional field, a new endpoint, or a new enum value an old client can ignore is safe. Removing a field, renaming one, changing its type, or tightening validation is breaking — an old client that still reads fare_amount will crash or misbehave the moment you rename it to total_fare. So you never break an existing field; you add the new one alongside it and let old clients keep reading the old one.

When a breaking change is truly unavoidable, you cut a new major version and run both in parallel through a deprecation path: announce the timeline, emit a Deprecation / Sunset header on v1 responses, watch traffic migrate off it, then retire v1 only when usage approaches zero. gRPC encodes the additive rule structurally — Protobuf field numbers are append-only, so a new field gets a new number and old readers skip what they don't recognize.

Interviewer follow-up

"You need to rename a field thousands of mobile clients still read — what do you do?" The wrong answer is "rename it." The right one: add the new field, keep populating the old one in parallel, migrate clients over releases, mark the old field deprecated with a sunset date, and remove it only once telemetry shows old clients are effectively gone.

07 Paging a moving list

Pagination that scales

No endpoint should return ten thousand orders in one response — it blows up memory, latency, and the client. Pagination returns results in pages. There are two strategies, and choosing wrong is a classic feed bug.

Offset / limit — simple, but it drifts

Offset pagination is the intuitive one: ?limit=20&offset=40 means "skip 40, give me the next 20", which the database runs as LIMIT 20 OFFSET 40. It is dead simple and lets you jump to an arbitrary page. Two problems sink it at scale. First, drift: on a list that changes, if a new order is inserted at the top between page 1 and page 2, every row shifts down by one — the reader sees a duplicate at the page boundary, or misses a row entirely. Second, deep-page cost: OFFSET 100000 forces the database to scan and discard 100,000 rows before returning 20, so latency grows linearly with how deep you page.

Cursor / keyset — stable and fast

Cursor pagination (also called keyset pagination) fixes both. Instead of "skip N rows", you say "give me rows after this specific point". The server returns an opaque cursor — typically an encoded pointer to the last item's stable sort key, like (created_at, id) — and the client passes it back to fetch the next page. The query becomes WHERE (created_at, id) < (:cursor) ORDER BY created_at DESC, id DESC LIMIT 20, which an index serves in roughly constant time no matter how deep you are. And because it anchors on a stable key rather than a position, inserts at the top don't shift the window — no duplicates, no gaps. That stability is exactly why feeds and timelines (a Shopee order list, a Grab ride history that keeps growing) use cursors by default.

Offset / limit

  • Trivial to implement and reason about
  • Can jump directly to any page number
  • Drifts: inserts shift rows → duplicates or gaps
  • Slow on deep pages: OFFSET scans and discards
  • Fine for small, stable, admin-style tables

Cursor / keyset

  • Stable under inserts — no duplicates or skips
  • Constant-time deep paging via an indexed key
  • The default for feeds, timelines, infinite scroll
  • No arbitrary "jump to page 50"; only next / prev
  • Needs a stable, unique sort key to anchor the cursor
# Cursor request and response shape
GET /v1/orders?limit=20&cursor=eyJjcmVhdGVkX2F0IjoiMjAyNi0wNS0zMCJ9

{
  "data": [ /* up to 20 orders, newest first */ ],
  "page_info": {
    "next_cursor": "eyJjcmVhdGVkX2F0IjoiMjAyNi0wNS0yOSJ9",
    "has_next": true
  }
}
Interviewer follow-up

"How do you paginate a feed that keeps changing as the user scrolls?" Cursor pagination on a stable, unique sort key such as (created_at, id). Explain why offset fails here: a new item at the top shifts every offset, so page 2 repeats a row from page 1. The cursor anchors to content, not position, so the window stays consistent even as new rows arrive.

08 The fintech centerpiece

Idempotency keys

Here is the scenario that defines a fintech round. A rider taps "Pay" in the Grab app. The phone sends POST /v1/payments, the server charges the card — and then the response is lost in a tunnel. The client times out, sees nothing, and does the natural thing: it retries. Without protection, the second request charges the card again. The rider is billed twice for one ride. Retries are unavoidable on unreliable networks, so the system must make an unsafe POST safe to retry. The tool is the idempotency key.

How it works under the hood

An operation is idempotent if performing it many times has the same effect as performing it once. GET and PUT are naturally idempotent; POST is not — each call creates a new resource. We make POST idempotent by attaching a client-generated unique Idempotency-Key header (a UUID the client mints once for one logical payment and reuses on every retry of it). The server keeps a key store mapping each key to the result it produced:

  • On arrival, the server looks up the key. First time seen: it processes the charge, stores key → {status, response} atomically, and returns the result.
  • Key already present: it skips processing entirely and returns the stored response — same body, same status — so the retry observes the original outcome and the card is never charged twice.
  • Key present but still processing (the first request hasn't finished): the server returns 409 Conflict or makes the retry wait, so two concurrent retries can't both slip through.

The store is typically Redis (fast, with a natural TTL so keys expire after, say, 24 hours) or a PostgreSQL / DynamoDB table with a unique constraint on the key. The unique constraint is the safety net: even under a race, the database rejects the second insert of the same key, collapsing two requests into one charge.

Where it connects

Idempotency is the safety mechanism that makes higher-level patterns trustworthy. A saga (Lesson 16) that retries a failed step relies on each step being idempotent, or a retry would corrupt state. A real payment system (Lesson 39) layers idempotency keys with a ledger and reconciliation. The key is the smallest unit of "make this retryable", and interviewers treat it as a litmus test for whether you've actually built money-moving systems.

Failure mode & the 10x question

What if the first request charges the card but crashes before storing the key? The retry, finding no key, charges again — the exact bug you were avoiding. So the result must be recorded in the same atomic transaction as the side effect (or the key reserved before calling the payment provider, then finalized after). At 10x volume the key store must be highly available and the TTL long enough to outlive a client's retry budget but short enough to bound storage — a 24-hour window is a common balance.

Interviewer follow-up

"How do you make this POST safe to retry?" Idempotency key on the request, a key→result store checked before acting, the result persisted atomically with the side effect, and a unique constraint to win races. Bonus points for naming the crash-between-charge-and-store failure and fixing it with one transaction. This is the single most-tested API design question in fintech rounds.

09 Past the book

The payment contract, concretely

Interviewers reward a candidate who can write an actual contract, not just talk about one. Here is a create-payment endpoint with everything this lesson assembled — a version prefix, an idempotency header, a correlation ID for tracing, and a clear status code:

# Request
POST /v1/payments  HTTP/1.1
Host: api.grabpay.com
Authorization: Bearer <access_token>        # OAuth2 access token (JWT)
Idempotency-Key: 9f2c1b7a-4e2d-4a01-bf33   # client-minted, reused on retry
X-Request-Id: req-7731-aa19                 # correlation id for tracing
Content-Type: application/json

{
  "amount": 1850,                  # minor units: 18.50 SGD
  "currency": "SGD",
  "source": "wallet_8812",
  "description": "Ride GR-44213"
}
# Response — first time the key is seen
HTTP/1.1 201 Created
X-Request-Id: req-7731-aa19

{
  "payment_id": "pay_01HF8...",
  "status": "succeeded",
  "amount": 1850,
  "currency": "SGD",
  "created_at": "2026-05-30T09:12:44Z"
}

# A retry with the SAME Idempotency-Key returns the SAME 201 body —
# the card is not charged a second time.

And the key store behind it, as a relational sketch — the unique constraint is what enforces "one key, one charge":

CREATE TABLE idempotency_keys (
  idem_key      TEXT PRIMARY KEY,   # the client's Idempotency-Key
  request_hash  TEXT NOT NULL,      # guards against key reuse on a different body
  status        TEXT NOT NULL,      # in_progress | completed
  response_code INT,
  response_body JSONB,
  created_at    TIMESTAMPTZ DEFAULT now(),
  expires_at    TIMESTAMPTZ            # TTL: purge after 24h
);

Note request_hash: if a client reuses a key for a different request body (a bug), the server can detect the mismatch and reject it rather than silently returning the wrong stored result. That detail separates a textbook answer from one that has shipped.

10 The finishing touches

API hygiene & errors

The last 10% of an API contract is what makes it pleasant — or miserable — to integrate against. Three habits matter.

A consistent error model

Every error should have the same shape, a machine-readable code, and the right status. Don't return 200 with {"error": "..."} in the body — that lies to caches and clients. Map outcomes to the standard families: 4xx for client mistakes (400 bad input, 401 unauthenticated, 403 forbidden, 404 missing, 409 conflict, 422 unprocessable, 429 rate-limited) and 5xx for server faults. A uniform body lets clients branch on a stable code rather than parsing English:

{
  "error": {
    "code": "insufficient_balance",
    "message": "Wallet balance is below the charge amount.",
    "request_id": "req-7731-aa19"
  }
}

Correlation IDs for tracing

Give every request a correlation ID (X-Request-Id) at the gateway and propagate it through every internal hop. When a payment fails, one ID lets you stitch the gateway log, the payments-service log, and the wallet-service log into a single trace — the difference between a five-minute fix and a five-hour hunt. This is the foundation distributed tracing (OpenTelemetry, Jaeger) builds on.

Sensible timeouts

Every call needs a timeout — a missing timeout is a latent outage. Without one, a single stuck downstream call holds a connection open until threads exhaust and the whole gateway hangs. Set per-route timeouts shorter as you go deeper (the client waits longer than the gateway, which waits longer than the service) so failures surface cleanly instead of cascading, and pair them with bounded retries and a circuit breaker so a struggling dependency isn't hammered while it tries to recover.

Interviewer follow-up

"A downstream call hangs — what happens to your gateway?" The answer is the timeout discipline: bounded per-route timeouts, capped retries with backoff and jitter (so retries don't synchronize into a thundering herd), and a circuit breaker that trips after repeated failures and serves a fast fallback. No timeout means one slow dependency can take the entire edge down.

★ Review

Summary & self-test

The whole topic compresses to one arc: put one front door in front of many services so cross-cutting work is solved once; choose REST or gRPC by where the call lives; version and paginate so you never break a client; and make unsafe writes retryable with idempotency keys. If you can draw the gateway, justify the protocol, and explain why a retried payment doesn't double-charge, you can carry the round.
