03Lesson · System Design Interview

Load Balancing & Reverse Proxies

The moment you run more than one server, something has to spread traffic across them, notice the instant one dies, and hide the whole fleet behind a single address. That something is the load balancer — and standing in the same doorway is its close cousin, the reverse proxy, the edge component that terminates TLS, caches, compresses, and shields your backends. The interview lives in the algorithm you choose, the health checks you wire up, the reason sticky sessions quietly sabotage horizontal scaling, and your ability to keep "reverse proxy", "load balancer", and "API gateway" cleanly apart.

About these notes: original study material on load balancing and reverse proxies — standard, widely-documented infrastructure fundamentals. 00 The problem

The single-server ceiling

Start with the simplest possible deployment: one web server answering every request. It works beautifully right up until it doesn't. There are two failure modes baked into that one box, and both are dead ends.

First, it's a single point of failure (SPOF) — the one node whose death takes the entire service offline. A bad deploy, a kernel panic, a full disk, an out-of-memory kill, an availability-zone outage: any of them turns into total downtime because there is nowhere else for traffic to go. Availability is usually quoted in "nines" — 99.9% uptime is roughly 8.7 hours of allowed downtime a year, and a single box that reboots for ten minutes during a bad patch has already blown a tighter budget. A lone server simply cannot offer a high-availability number, because its expected failures have no fallback.

Second, it's a capacity ceiling. You can buy a bigger machine — that's vertical scaling (scaling up) — but CPU cores, RAM, network cards, and the operating system's open-file-descriptor and TCP-connection limits all top out, and the price curve goes vertical long before the performance does. A single box can also only be in one rack, in one data center, in one region: a user in Jakarta hitting a server in Frankfurt eats hundreds of milliseconds of unavoidable round-trip latency no matter how fast the CPU is. The answer is horizontal scaling (scaling out): run many ordinary servers instead of one heroic one.

But running many servers immediately creates a new problem. Clients can't be told to pick a server themselves — you'd be leaking your private fleet topology to the world, you'd have no clean way to skip a dead node, and every client would need to re-learn the fleet every time it changed. You need a single front door that accepts all traffic, spreads it evenly, and routes around failures. That front door is the load balancer.

Why it matters

Almost every later decision in this lesson exists to make horizontal scaling actually safe: a smart algorithm so load lands evenly, health checks so a dead node is skipped, statelessness so any node can serve any request, and redundancy so the load balancer itself isn't the new SPOF. Keep that thread in mind — each section removes one obstacle to running many cheap boxes instead of one expensive one.

01 One entry point

One entry point: the VIP

The load balancer sits in front of the server fleet and is the only thing the outside world ever talks to. Clients resolve your domain — say api.superpay.sg — to a single public address called a virtual IP (VIP). The word "virtual" is the whole point: the VIP is not the hardware address of any one machine. It is an address the load balancer answers on behalf of the fleet, which is exactly why it can be moved to a different load balancer during a failover (we'll use that in step 06). Behind it, each application server lives on a private IP inside your virtual network — something like 10.0.0.11 — that is unroutable from the public internet and invisible to clients.

For every incoming connection the load balancer picks one healthy server and forwards the request, then relays the response back. The fleet's size, IP addresses, and which nodes are currently up are all hidden behind that single VIP. You can add servers during a traffic surge or pull a sick one out, and clients never notice — they keep hitting the same address. This indirection is what makes a fleet elastic: capacity becomes a dial you turn, not a migration you announce.

How does a client even find the VIP in the first place? Through DNS. Your authoritative DNS returns the VIP as the A/AAAA record for the hostname. Note the division of labor: DNS gets the client to the front door (and, as we'll see in step 09, can choose which regional front door), while the load balancer decides which server behind that door actually handles the request. DNS is coarse and slow to change because answers are cached; the load balancer is fine-grained and reacts in milliseconds. Keeping those two jobs distinct is a frequent source of interview clarity.

Walk the routing once, step by step. This is the canonical "draw the request path" the interviewer wants to see.

Why it matters

The VIP is the abstraction that makes everything else in this lesson possible. Because clients bind to the VIP and never to a server, the fleet becomes elastic — you can scale it up and down and replace dead nodes without touching DNS, app config, or the client at all. Every "what if a server dies / we deploy / we autoscale" follow-up has the same root answer: the client only ever knew the VIP, so the change is invisible to it.

02 Which layer?

L4 vs L7 balancing

Load balancers come in two flavors, named after the OSI layer they operate on. The first decision an interviewer probes is which one you'd pick and why — and a strong answer explains how each one actually forwards a packet, not just what it can route on.

How an L4 balancer forwards

A Layer 4 (L4) load balancer works at the transport layer. It routes by IP and port only and never reads the request body — it doesn't even know the bytes flowing through are HTTP. Mechanically, the common mode is NAT-style connection forwarding: a client opens a TCP connection to the VIP, the balancer picks a backend, rewrites the destination address (and tracks the flow in a connection table so every subsequent packet of that connection goes to the same backend), and forwards the packets. The balancer relays packets; it does not terminate the client's TCP connection and open a fresh one. Because it parses almost nothing, it is extremely fast, cheap on CPU, and works for any protocol — TCP, UDP, a game socket, a database wire protocol, MQTT. AWS Network Load Balancer (NLB) is the canonical managed L4; it advertises handling millions of requests per second at ultra-low latency precisely because it does so little per packet.

Under the hood — Direct Server Return

An advanced L4 trick is Direct Server Return (DSR). Normally the response retraces its path back through the balancer. With DSR, the request goes in through the balancer but each backend replies directly to the client, bypassing the balancer on the way out. Since responses are usually far larger than requests (think a 2 KB request fetching a 2 MB video chunk), removing the balancer from the egress path massively raises throughput. The cost is operational complexity — the backends must be configured to answer for the VIP — which is why DSR shows up in high-throughput, hand-tuned L4 setups rather than everyday web tiers.

How an L7 balancer forwards

A Layer 7 (L7) load balancer works at the application layer. It does something fundamentally different from L4: it terminates the client's TCP (and TLS) connection itself, fully reads and parses the HTTP request, then opens a separate connection to the chosen backend and forwards a possibly-rewritten request. There are effectively two connections — client↔balancer and balancer↔backend — which is exactly what lets the balancer pool and reuse backend connections, buffer slow clients, and inspect everything in between. Because it understands HTTP, it can read the URL path, headers, and cookies and route on them: send /api/* to the order-service pool and /static/* to a different pool, or split by Host header so shop.example.com and pay.example.com reach different fleets from one VIP. It can also terminate TLS, rewrite headers, cache, and compress. All that inspection costs a little latency and CPU. AWS Application Load Balancer (ALB), NGINX, HAProxy, and Envoy are the everyday L7 workhorses.

Layer 4 — transport

  • Routes by IP / port; no payload inspection
  • Forwards/NATs packets — does not terminate the connection
  • Very fast, low CPU, low latency; DSR can skip the return path
  • Protocol-agnostic — TCP, UDP, anything
  • Can't route by path, header, or cookie
  • Can't terminate TLS, cache, or compress

Layer 7 — application

  • Understands HTTP: path, headers, cookies, Host
  • Terminates the connection, opens a fresh one to the backend
  • Content routing (/api vs /static, host-based)
  • Can terminate TLS, rewrite headers, cache, compress
  • Enables auth, buffering, and observability at the edge
  • Slightly more overhead; HTTP-aware only

The practical rule: reach for L7 for normal web and API traffic, where smart routing and TLS termination are worth the small overhead — this is what fronts a typical Shopee or Grab API tier, where one VIP needs to fan /orders, /payments, and /static to different services. Reach for L4 when you need raw throughput, a non-HTTP protocol (a real-time game socket, a Postgres proxy, a gRPC stream you don't want to terminate), or the lowest possible latency. A very common real-world pattern stacks both: an L4 NLB takes the internet-facing punishment and spreads connections across a fleet of L7 proxies, each doing the HTTP-aware work.

Interviewer follow-up

"If the L7 balancer terminates TLS, what does the backend see as the client's IP?" The backend sees the balancer's IP, because the balancer opened the connection. The fix is header injection — the balancer adds X-Forwarded-For (the real client IP) and X-Forwarded-Proto (was it HTTPS?) so the app can recover the truth. We unpack that mechanism in the reverse-proxy section, but flagging it here shows you understand the consequence of terminating the connection.

03 How to pick a server

Distribution algorithms

Once the balancer has a pool of healthy servers, it needs a rule for choosing one per request. The choice trades simplicity against how evenly real load actually lands, and each rule has a characteristic way of going wrong. Walk them with the mechanism and the failure mode in mind.

  • Round-robin — hand requests out in a rotating cycle: server 1, 2, 3, 1, 2, 3… The balancer keeps a single rotating index; that's the entire mechanism, which is why it's the default almost everywhere. It's even by request count. Where it goes wrong: it's blind to how heavy each request is, so if one server lands on three slow report-generation calls while peers get cheap key lookups, round-robin keeps feeding the busy one — the queue on that node grows while others idle.
  • Weighted round-robin — give beefier machines a higher weight so they receive proportionally more traffic (a weight-3 box gets three turns in the cycle to a weight-1 box's one). Useful when the fleet is heterogeneous — some 16-core boxes, some 4-core — or during a canary deploy where you send 5% of traffic to the new version. Where it goes wrong: weights are static, so they don't react if a "big" box is momentarily slow.
  • Least-connections — send the next request to the server with the fewest open connections right now. The balancer tracks a live connection count per backend and picks the minimum. This adapts to real load and is a great default when request durations vary a lot (long-lived connections, slow queries, streaming responses). Where it goes wrong: "fewest connections" isn't "least loaded" if connections do wildly different amounts of work — but it's far better than counting requests.
  • Least-response-time — pick the server with the best blend of active connections and recent latency. The most load-aware option; the balancer must measure per-backend timings. Where it goes wrong: a freshly added (cold-cache) node can briefly look fast and get flooded before it warms up.
  • IP-hash / consistent hashing — hash a key (the client IP, or a user ID) to deterministically map it to a server, so the same key keeps landing on the same node. Plain modulo hashing (hash(key) % N) gives stickiness but reshuffles almost every key when N changes — add or remove one server and nearly all keys move, which is catastrophic for caches. Consistent hashing places servers and keys on a ring so that adding or removing a node moves only a small slice of keys; it's the standard way to get cache affinity (route a user to the node that already cached their data) without a mass reshuffle. That property earns it its own lesson later — here, just know it's the right tool when you want stickiness by key with minimal churn, and that naive IP-hash is the seed of the sticky-session trap in step 05.

Round-robin family

  • Trivial to implement and reason about
  • Even by request count; weights handle uneven hardware
  • Blind to request cost and current server load
  • Good when requests are short and uniform

Least-connections / -response-time

  • Adapts to real-time load, not just count
  • Best when request durations vary widely
  • Needs the LB to track per-server state
  • The safer default for mixed, bursty API traffic
Interviewer follow-up

"When would round-robin distribute load unevenly?" Answer: whenever request cost varies — one server lands on three slow report-generation calls while the others get cheap reads, yet round-robin keeps feeding the busy one. That's the cue to mention least-connections as the load-aware upgrade. A sharper follow-up: "and when would you choose consistent hashing?" — when you need cache affinity (sticky-by-key) but can't afford a full reshuffle every time the fleet size changes.

04 Detecting failure

Health checks & draining

Spreading load is only half the job. The whole point of a fleet is that one node can die without taking the service down — but that only works if the balancer knows a node is dead and stops sending it traffic. That's what health checks are for, and the interview reward is in describing the mechanism and its dials precisely.

Active vs passive probing

There are two styles, usually combined. An active health check has the balancer probe each server on a schedule — typically an HTTP GET to a lightweight /health (or /healthz) endpoint every few seconds. The probe is governed by three dials worth naming explicitly: the interval (how often to probe — say every 5s), the unhealthy threshold (how many consecutive failures before ejecting — say 3), and the healthy threshold (how many consecutive successes before re-admitting — say 2). Miss enough probes and the node is marked unhealthy and ejected from the pool; pass enough and it's restored. A passive health check watches real live traffic instead: if a server starts returning connection errors, resets, or 5xx responses, the balancer infers it's sick and pulls it out without waiting for the next scheduled probe. Passive is faster to react (it uses traffic you're already sending) but can't tell you a freshly-ejected node has recovered — only an active probe can safely re-admit it. That's why mature setups run both.

A subtlety the best candidates raise: a shallow health check (does the process answer at all?) versus a deep one (can the server actually reach its database and dependencies?). A deep check catches a server that's up but cut off from its datastore, but if every node shares one sick dependency, a deep check can eject the entire fleet at once. The pragmatic middle ground is a shallow liveness check for ejection and richer readiness signals for routing decisions.

Outlier detection & circuit breaking

The modern, more granular form of passive checking is outlier detection (Envoy's term) or circuit breaking. Instead of a single up/down verdict, the proxy watches each backend's live error and latency rates and temporarily ejects a node that is statistically misbehaving relative to its peers — then probes it back in after a cooldown. A circuit breaker generalizes this: once errors to a backend cross a threshold, the proxy "opens the circuit" and fails fast (returning an error immediately) instead of piling more doomed requests onto a struggling dependency, giving it room to recover. This is the failure-containment mechanism that keeps one sick service from dragging down everything calling it.

Connection draining (graceful shutdown)

Failure isn't the only reason to remove a node — deploys and scale-downs do it on purpose. Yanking a server instantly would kill every request it's mid-way through. The fix is connection draining (a.k.a. graceful shutdown). The mechanism: the balancer marks the node "draining", immediately stops sending it new requests, but lets existing in-flight ones finish — up to a configured drain timeout (commonly 30–300 seconds), after which any stragglers are forcibly cut. In Kubernetes this is the same dance the preStop hook plus terminationGracePeriodSeconds orchestrate. Get it right and you can deploy a new version of the Grab order service in the middle of peak dinner-time traffic with zero dropped requests; get it wrong (drain timeout of zero) and every user mid-checkout on the retiring node sees an error.

Interviewer follow-up

"How fast do you detect a dead node, and what's the trade-off?" Detection time is roughly interval × unhealthy-threshold — tight intervals and low thresholds detect failure faster but risk flapping: ejecting a node over a transient blip (a single GC pause), then re-admitting it, then ejecting it again, thrashing the pool. Looser settings are stable but leave a dead node taking traffic longer. Naming this tension (detection speed vs false positives) — and adding that outlier detection plus a healthy-threshold hysteresis dampens flapping — is the point.

05 The stickiness trap

Sticky sessions & the fix

Here's a problem the fleet creates for itself. Suppose each server keeps a user's login session in its own local memory. Now request #1 from a user lands on Server 1 and creates a session there; request #2 round-robins to Server 2, which has never heard of this user — and the user is suddenly logged out. The very load-spreading you wanted has broken correctness.

The patch: session affinity, and its two mechanisms

The quick patch is a sticky session (session affinity): the balancer pins each client to one server so every request from that client returns to the node that holds its session. There are two ways to implement it, and the difference matters. Cookie-based affinity has the L7 balancer set its own cookie (NGINX's sticky cookie, ALB's AWSALB cookie) encoding which backend to use; the client echoes the cookie on every request and the balancer reads it. It's precise and survives the client changing networks. IP-hash affinity instead hashes the client's source IP to a backend — no cookie needed, works for non-HTTP traffic, but breaks when many users share one NAT/proxy IP (they all pin to one node) or when a mobile user's IP changes mid-session. Either way, stickiness fights everything else you've built:

  • Uneven load. Once clients are pinned, the balancer can no longer freely spread work — a few "whale" users on one node can hot-spot it while others idle, and least-connections is powerless to correct it.
  • Breaks on node death. If the pinned server dies, every session it held is gone; those users are logged out and must start over — the exact downtime the fleet was supposed to prevent.
  • Fights autoscaling. Add a new server during a spike and existing clients stay glued to old nodes, so the new capacity barely helps the hot ones. Scale down and you drop live sessions.

The real fix: externalize state, make servers stateless

The real fix is to stop storing session state on the server at all. Push it into a shared store — a distributed cache like Redis, or a database — that every server can read. The mechanism: on login the server writes session:<id> → {user, expiry} to Redis and hands the client a session ID (in a cookie or a signed JWT); on every later request any server looks the ID up in Redis and gets the same answer. Now the servers are stateless — they hold no per-user data that can't be reconstructed from a shared source — so any node can serve any request. Stickiness becomes unnecessary, load spreads freely, a dying server takes no sessions with it, and autoscaling works cleanly. (This is exactly the stateless-services principle from Lesson 02 — externalizing session state is what makes horizontal scaling actually work. The cost is one extra Redis lookup per request, which is microseconds and well worth it.)

Why it matters

"Make your servers stateless" is one of the highest-leverage sentences in a system-design interview. Statelessness is precisely what lets the load balancer use a smart algorithm, autoscale freely, and survive a node death — three things sticky sessions quietly take away. When an interviewer pushes "but we have sessions", the senior answer is "I'll keep the server stateless and move the session to Redis, not pin clients with stickiness."

06 Who balances the balancer?

The LB is a SPOF too

We added the load balancer to eliminate the single point of failure — but if you stop here, you've just moved the SPOF. Now every request funnels through one box, and when it dies, the whole fleet behind it becomes unreachable even though every server is perfectly healthy. The balancer must be made redundant too.

Active-passive with a floating IP

The classic answer is an active-passive pair. Two load balancers run side by side; the active one serves all traffic while the passive one stands by. They share a floating IP — the VIP isn't nailed to either box's network card but assigned to whichever is currently active. The mechanism that moves it is VRRP (Virtual Router Redundancy Protocol), implemented by tools like keepalived: the two nodes exchange a constant heartbeat (multicast advertisements every second or so). While the passive keeps hearing the active's heartbeat, it stays quiet. The instant the heartbeat stops, the passive concludes the active is dead, promotes itself, and claims the floating IP — at the link layer it broadcasts a gratuitous ARP so the local network updates its ARP cache and starts delivering the VIP's packets to the new box. That's a failover, and it typically completes in a few seconds. Clients keep hitting the same address and barely notice.

The cost of active-passive is honest: the standby sits idle, so you're paying for capacity you only use during failures. And you must guard against split-brain — if the heartbeat link (not the active node) is what failed, both nodes could think they're active and fight over the VIP; a separate heartbeat path and quorum logic prevent that.

Active-active

The richer option is active-active: multiple balancers all serve traffic at once. They're fronted either by DNS handing out several VIPs (round-robin or weighted), or by an upstream L4 layer / anycast, where the same IP is advertised from multiple locations and the network's ECMP (equal-cost multi-path) routing naturally spreads flows across them. You get more aggregate capacity and use both machines instead of paying for an idle standby — at the cost of more coordination, and the need for every balancer to share or independently derive the same view of backend health.

What managed cloud LBs do for you

In practice you rarely hand-roll keepalived anymore. A managed load balancer — AWS ALB/NLB, GCP Cloud Load Balancing, Cloudflare — is a horizontally-scaled, multi-AZ, self-healing fleet behind a single endpoint; the provider runs the redundancy for you. You still must understand the active-passive model, because it explains why a managed LB can survive a node loss and what "the LB is highly available" actually means under the hood.

Interviewer follow-up

"What happens during failover — do clients drop?" In-flight TCP connections on the dead balancer are lost and those clients reconnect, but the floating IP moves in seconds and new connections succeed immediately. Add the nuance that this is why managed LBs (AWS ELB/ALB, GCP LB) are attractive — they hide failover entirely behind a redundant, horizontally-scaled fleet — so you rarely build active-passive by hand, but you must be able to explain VRRP, the heartbeat, and the gratuitous ARP if pressed.

07 The edge component

The reverse proxy, in depth

So far the box in front of the fleet has worn one hat — distributing traffic. But that same box almost always does a lot more, and the term for the more-general role is reverse proxy. This is the part interviewers find candidates rush, so we'll slow right down.

Forward proxy vs reverse proxy: who is being hidden?

Both are intermediaries that relay requests, but they sit on opposite ends and hide opposite parties. A forward proxy sits in front of the clients and represents them to the internet — the servers it talks to see the proxy's IP, not the real client's. A reverse proxy sits in front of the servers and represents them to clients — the clients it talks to see the proxy as if it were the origin, never the real backends. One sentence to memorize: a forward proxy hides the clients; a reverse proxy hides the servers.

  • Forward-proxy example. A corporate office routes all employee web traffic through a company proxy that enforces policy, filters malware, and caches popular downloads. To shopee.sg, every employee looks like one IP — the proxy. The proxy serves the clients.
  • Reverse-proxy example. api.superpay.sg resolves to an NGINX box; that box terminates TLS, checks a cache, and forwards surviving requests to a private fleet of app servers. The client believes it is talking to SuperPay's server; it's actually talking to the proxy, which serves the servers.

The jobs a reverse proxy does — each with its mechanism

Because the reverse proxy terminates the client's connection and stands between the client and your backends, it's the natural place to put cross-cutting concerns once, instead of in every app server. The full set of jobs:

  • TLS / SSL termination. The proxy holds the certificate and private key, decrypts inbound HTTPS once at the edge, and talks plain HTTP to the backends. Mechanism: it does the expensive TLS handshake and symmetric decryption so each app server doesn't have to; certificates are managed (and rotated) in one place instead of on every box. The trade-off is real: traffic on the internal hop is now plaintext, so anyone who breaches your network can read it. The hardened answer is to re-encrypt to the backend (TLS again, or full mTLS where proxy and backend mutually authenticate) — the classic "TLS termination vs TLS passthrough vs re-encryption" decision.
  • Response caching. The proxy keeps recent responses keyed by URL (plus relevant headers) and serves a repeated request straight from its cache without touching the origin. Mechanism: it honors Cache-Control / ETag or your explicit rules; a cache hit never reaches the app server, so a popular Shopee category page that's expensive to render once is then served thousands of times for free. This is also the foundation of a CDN — a globally distributed fleet of caching reverse proxies.
  • Compression. The proxy compresses responses with gzip or brotli at the edge based on the client's Accept-Encoding, shrinking a JSON or HTML payload by 60–80% so it crosses the wire faster — without every backend re-implementing compression.
  • Request buffering / absorbing slow clients. The proxy reads the entire slow request from the client into its own buffer before bothering the backend, and drip-feeds the response to a slow reader the same way. Mechanism: an event-driven proxy like NGINX handles tens of thousands of idle/slow connections cheaply, whereas a backend app worker tied up waiting on a trickling client is one fewer worker for real work. This is the direct defense against a Slowloris attack, where an adversary opens many connections and sends headers one byte at a time to exhaust your worker pool — the proxy soaks that up so the backend never sees it.
  • Static-content serving. The proxy can serve files (images, CSS, JS, the SPA bundle) directly off disk or cache, so the application servers only ever handle dynamic requests. One config line offloads a whole class of traffic.
  • Header rewriting & X-Forwarded-*. Because the proxy terminated the connection, the backend's socket shows the proxy's IP, not the client's. The proxy fixes this by injecting headers the backend can trust: X-Forwarded-For carries the real client IP (appended as the chain grows), X-Forwarded-Proto records whether the original request was https, and X-Forwarded-Host the original host. The modern standardized equivalent is the Forwarded header. This is how a backend behind a TLS-terminating proxy still logs the true client IP and still knows to issue secure cookies. (Security note: only trust these headers from your own proxy, or an attacker can spoof them.)
  • Origin hiding & security. A single hardened ingress means there's exactly one place to patch, one place to run a WAF (web application firewall) that blocks SQL-injection and known exploit patterns, one place to rate-limit and absorb DDoS. The backends' private IPs and topology never appear on the public internet, shrinking the attack surface to the proxy you've hardened.
  • HTTP/2, multiplexing & keep-alive pooling. The proxy can speak modern HTTP/2 (or HTTP/3) to the client — multiplexing many streams over one connection — while maintaining a warm pool of reused keep-alive connections to the backends. Mechanism: TCP and TLS setup is expensive, so reusing a handful of long-lived proxy↔backend connections for thousands of client requests slashes connection-setup overhead and protects backends from connection storms.

The tools, and what a config conceptually says

The dominant reverse proxies: NGINX (the default workhorse — fast event loop, great at static files, TLS, and caching), HAProxy (extremely strong L4/L7 load balancing and health checking), Envoy (the modern, dynamically-configurable proxy that powers service meshes and supports outlier detection and rich observability), and Caddy (developer-friendly with automatic HTTPS via Let's Encrypt). Managed equivalents — Cloudflare and AWS ALB — give you the same edge jobs without running the box.

Conceptually, an NGINX reverse-proxy config expresses two ideas: a named upstream (the pool of backends, with an algorithm and health checks) and a server block that terminates TLS and forwards matching requests to that upstream while injecting the forwarding headers. An original, illustrative sketch:

# the fleet behind the proxy — chosen algorithm + passive ejection
upstream superpay_api {
    least_conn;
    server 10.0.0.11:8080 max_fails=3 fail_timeout=10s;
    server 10.0.0.12:8080 max_fails=3 fail_timeout=10s;
    keepalive 32;          # warm pooled connections to backends
}

server {
    listen 443 ssl http2;                 # TLS terminates here, HTTP/2 to client
    ssl_certificate     /etc/ssl/superpay.crt;
    gzip on;                              # compress at the edge

    location / {
        proxy_pass http://superpay_api;   # forward to the upstream (plain HTTP)
        proxy_set_header X-Forwarded-For   $remote_addr;   # real client IP
        proxy_set_header X-Forwarded-Proto $scheme;        # was https?
    }
}
Interviewer follow-up

"You terminate TLS at the proxy — isn't the internal hop now insecure?" Yes, and the answer separates juniors from seniors: inside a trusted VPC many teams accept plaintext on the internal hop for simplicity and performance; in a zero-trust or regulated (payments) environment you re-encrypt to the backend or run mTLS so the proxy↔backend link is authenticated and encrypted too. State the trade-off explicitly — "TLS termination buys central cert management and cheap backends; re-encryption / mTLS buys defense-in-depth at some CPU cost" — and you've nailed it.

08 Keeping the terms straight

Proxy vs LB vs API gateway

These three terms overlap, and interviewers test whether you can hold them apart cleanly. The trick is to see them as layers of added responsibility on the same base component, not three unrelated boxes.

Start from the base. A reverse proxy is a server that fronts your application servers, forwards client requests to them, and returns the responses — so to the client it looks like the origin. Its defining job is to be the front door and intermediary; distribution across many servers is not its essence, which is why a reverse proxy can sit usefully in front of a single server purely to terminate TLS, cache, or compress.

A load balancer's defining job is narrower and additive: distribute incoming traffic across multiple backends and route around unhealthy ones. The cleanest way to relate it to the proxy: an L7 load balancer IS a reverse proxy that also distributes. It terminates the connection and forwards it (reverse-proxy behavior) and on top of that picks among many backends with an algorithm and health checks (load-balancer behavior). An L4 load balancer, by contrast, is purer distribution — it forwards packets without terminating, so it isn't really acting as a reverse proxy at all. That's the precise relationship: at L7 the two roles merge in one box.

An API gateway is the next layer out: a reverse proxy (usually also load-balancing) that adds API-specific concerns. On top of fronting and distributing, it handles authentication / authorization (validate the JWT or API key once at the edge), rate limiting (the subject of a later lesson), request aggregation / composition (fan one client call out to several microservices and stitch the results), API versioning and routing (/v1 vs /v2, path-to-service mapping), request/response transformation, and per-consumer metering. Tools: Kong, AWS API Gateway, and Envoy-based gateways. The mental model is a strict nesting:

Reverse proxy vs Load balancer

  • Proxy: front door; useful with one backend
  • LB: distribution; pointless with one backend
  • An L7 LB is a reverse proxy that distributes
  • Same product (NGINX/Envoy) usually does both

Load balancer vs API gateway

  • LB: spread traffic, route around dead nodes
  • Gateway: + auth, rate limit, aggregation, versioning
  • Gateway is API-aware; an LB needn't be
  • Gateway often sits in front of, or wraps, an LB
Land this line

"They're concentric, not separate: a reverse proxy is the front-door intermediary; an L7 load balancer is a reverse proxy that also distributes across backends; an API gateway is that plus API-specific concerns like auth, rate limiting, and request aggregation. In practice one product — NGINX, Envoy, an ALB, or Kong — plays whichever roles you turn on." Saying it as nesting, with one example product, is the answer interviewers reward.

09 The hard follow-ups

Dies mid-request & geo-routing

Two questions come up almost every time, and a crisp answer to each sets you apart.

"What happens when a server dies mid-request?"

The honest answer has two halves. For the request already in flight on the dead node, the connection breaks and that one request fails — there's no magic recovery of work the dead box was doing. But the system contains the blast radius. The health check ejects the node within seconds (step 04) so no new requests go to it, and the failed request is often recoverable in two ways. First, if it was idempotent — a GET, or a write keyed by an idempotency token so a replay can't double-apply — the client or the LB can simply retry on another server. Second, because the server was stateless (step 05), any healthy node can serve that retry without needing whatever the dead node held in memory.

The subtlety worth surfacing: safe retries require idempotency. Auto-retrying a non-idempotent POST /charge can double-charge a customer. For a payments app like GrabPay, the client sends an idempotency key (say a UUID) with the charge; the server records "this key already succeeded" so a retry of the same key returns the original result instead of charging again. That single mechanism is what makes the retry safe — which is exactly why statelessness, health checks, and idempotency keep showing up together. (Beware naive blanket retries, too: retrying everything during an overload can amplify the load and cause a retry storm; production clients add exponential backoff and jitter.)

"How do you balance across regions?"

Everything so far balances within one data center. To send a user in Singapore to the Singapore region and a user in Jakarta to the Jakarta region, you balance one level up with global server load balancing (GSLB), usually via DNS. Mechanism: when a client resolves api.superpay.sg, a geo-aware DNS service (AWS Route 53, Cloudflare, NS1) inspects the resolver's location and the health of each region, and returns the VIP of the nearest healthy region — cutting latency and giving region-level failover (if a whole region goes dark, DNS stops handing out its address). Each region then runs its own local load balancer doing the in-region distribution from this lesson — GSLB is a balancer of balancers.

The catch to name explicitly: DNS answers are cached by resolvers and clients for the record's TTL, so DNS-based failover is slower than the second-scale floating-IP failover inside a region. A 60-second TTL means a client can keep hitting a dead region for up to a minute after you've pulled it from DNS. Lower TTLs speed failover but increase DNS query load and can defeat caching; this is the central trade-off of the GSLB tier. (The latency-zero alternative is anycast — advertising the same IP from every region and letting the network route to the nearest — which sidesteps TTL caching but needs control of the routing layer, which is why CDNs and big clouds use it.)

Land this line

"Within a region I use a redundant L7 load balancer with health checks, least-connections, and stateless servers; mid-request failures are contained by ejection plus idempotent retry. Across regions I use DNS-based GSLB for latency and region failover, knowing DNS TTL caching makes that tier slower to fail over than the in-region floating-IP path." Saying both tiers, the failure containment, and the trade-off of each is exactly the structured answer interviewers reward.

★ Review

Summary & self-test

The whole topic compresses to one chain: one server is a SPOF and a ceiling, so put many behind a single VIP; spread traffic with the right algorithm; health-check to route around the dead; keep servers stateless so any node can serve any request; make the balancer itself redundant; and recognize that the box doing all this is a reverse proxy that also terminates TLS, caches, and shields the fleet. If you can draw that chain and name what each link fixes, you can carry the interview.
