17Lesson · System Design Interview

Design a Distributed Message Queue

Build a Kafka-like distributed commit log: a durable, partitioned, replayable stream of records that many producers write into and many consumer groups read out of, each at its own pace. Almost the entire design follows from one idea — an append-only log split into partitions, with each reader tracking its own offset.

About these notes: original study material on distributed message queues and commit logs — a standard, widely-documented engineering topic. The explanations, examples, numbers, and diagrams here are my own; nothing is copied from any paid course or book. One diagram is interactive — press Play (or step with Next) to watch a record travel from producer to consumer.
00 The problem

Why a durable log?

Start with a concrete pain. At a super-app like Grab, a single tap of "confirm ride" fans out into a dozen reactions: charge the wallet, notify the driver, update the ETA service, write to the fraud model, append to the data-warehouse pipeline, refresh the loyalty points. If the ride service called all of those synchronously, one slow downstream — say the analytics warehouse is busy — would stall the user's ride confirmation. And if a downstream is down, the event is simply lost.

The fix is to put a buffer in the middle: the ride service writes one event ("ride 88421 confirmed") to a queue and returns immediately; every interested system reads that event on its own schedule. This decouples producers from consumers in time, in throughput, and in failure domain. That buffer is a message queue.

But a naive in-memory queue that deletes a message the instant it is read is too weak for the jobs above. We need more:

  • Durability — if a broker restarts, the ride-confirmed event must still be there. Money and audit trails cannot evaporate.
  • Replay — when the fraud team ships a new model, they want to re-process the last 7 days of events from the beginning. A delete-on-read queue makes that impossible.
  • Independent consumers — the warehouse pipeline reading slowly must not block the driver-notification service reading fast.

The data structure that satisfies all three is a distributed commit log: an append-only, ordered, replicated sequence of records that is kept on disk for a configurable window and read by position rather than consumed-and-deleted. This is the model behind Apache Kafka, AWS Kinesis, and Apache Pulsar, and it is what we will design.

Queue vs. log — the distinction to land

A classic message queue (RabbitMQ, AWS SQS) deletes a message once a consumer acknowledges it, and competing consumers each take different messages. A log keeps every record for a retention window and lets each consumer read the whole stream by tracking a position. Interviewers test whether you know which one the question asks for. "Kafka-like, replayable, multiple independent readers" means log.

01 Scope it

Clarify the requirements

Before drawing a single box, narrow the problem out loud. A distributed log can be tuned for very different points, and the interviewer wants to see you state assumptions rather than guess.

Functional requirements

  • Publish — a producer appends a record (an opaque byte payload, optionally with a key) to a named topic.
  • Subscribe & consume — a consumer reads records from a topic in order, starting from a position it chooses (earliest, latest, or a specific offset).
  • Consumer groups — many independent applications (a consumer group each) read the same topic without interfering. Within one group, work is split across members.
  • Retention — records persist for a configurable window (by time, e.g. 7 days, or by size, e.g. 500 GB per partition), then are discarded.
  • Replay — a group can reset its position backwards and re-read history that is still within retention.

Non-functional requirements

  • High throughput — millions of writes and reads per second, with payloads from tens of bytes to a few KB.
  • Durability — an acknowledged record survives the loss of a machine. We replicate.
  • Ordering — guaranteed within a partition, not across the whole topic (we will justify this hard limit in Step 04).
  • At-least-once delivery by default — no record is silently dropped; duplicates are possible and consumers must tolerate them. We will discuss exactly-once later.
  • Horizontal scalability — add brokers to add capacity; no single machine is the ceiling.
  • Low end-to-end latency — single-digit-to-tens of milliseconds from publish to available-for-consumption.

Explicit non-goals

Stating what you will not build is as valuable as stating what you will. We will not promise global total ordering across a topic, we will not do per-message priority or selective deletion of an individual record, and we will not implement complex routing logic (topic-exchange-binding rules) the way RabbitMQ does — the log model deliberately keeps the broker dumb and pushes logic to clients.

Interview tip

The single sentence that frames this whole design: "This is a partitioned, append-only commit log; ordering and parallelism both come from the partition, and each consumer group tracks its own offset." Say it early — everything that follows is an elaboration of it.

02 Estimate

Back-of-envelope sizing

A quick estimate justifies why we partition and replicate at all, and roughly how many machines we need. Take a plausible Shopee-style event firehose: order events, click events, inventory updates, and payment events flowing into one analytics platform.

Throughput

  • Assume 1 million records/second at peak across all topics — a reasonable regional super-app firehose.
  • Average record size 1 KB (a JSON-ish event with some context).
  • Write bandwidth in: 1,000,000 × 1 KB = 1 GB/s.

Replication multiplies the write

With a replication factor of 3 (one leader, two followers — the standard for durability), every byte is written three times across the cluster: 1 GB/s × 3 = 3 GB/s of disk write traffic, and 2 GB/s of inter-broker replication network traffic. This alone tells the interviewer we cannot live on one disk.

Read amplification

Reads multiply by the number of consumer groups. If 5 independent teams each consume the full firehose (analytics, fraud, search indexer, billing reconciliation, ML features), read bandwidth out is 1 GB/s × 5 = 5 GB/s. Plus catch-up readers replaying history spike this further. Reads dominate, which is exactly why the log keeps data on disk and serves it sequentially (Step 07).

Storage from retention

  • Retain 7 days. Seconds in a week: 7 × 86,400 ≈ 604,800 s.
  • Raw data at rest: 1 GB/s × 604,800 s ≈ 605 TB.
  • With replication factor 3: 605 TB × 3 ≈ 1.8 PB.

Roughly 1.8 petabytes of disk for one week of one firehose. At, say, 10 TB usable per broker after overhead, that is around 180 brokers just for storage — before considering CPU or network headroom. The numbers scream partition the data and spread it across many machines, which is the heart of the design.

What the estimate buys you

You do not need these figures exact. You need them to motivate the architecture: write amplification → replication is expensive, so tune the ack level; storage in PB → you must partition and shard; reads dominate → sequential disk + page cache + zero-copy matter. Each number points at a later design decision.

03 Contract

The API & data model

Pin down the interface next. The log's surface is deliberately tiny: append, fetch, and commit-offset. Going past the book, here is a concrete contract.

Producer API

# Append one record to a topic. The broker picks the partition
# from the key (hash), or round-robins if no key is given.
POST /topics/{topic}/records
{
  "key": "ride_88421",        // optional; controls partition + ordering
  "value": "<base64 payload>",
  "headers": { "trace_id": "abc" }
}
// 200 OK
{ "partition": 3, "offset": 90412, "timestamp": 1717142400123 }

The response is the heart of the contract: the broker returns the partition the record landed in and the offset — the record's permanent, monotonically increasing position within that partition. That pair, (partition, offset), uniquely and forever names the record. Producers may also batch many records into one request to amortize network and disk cost; this is the single biggest throughput lever in practice.

Consumer API

# Fetch a batch starting at an offset, blocking up to max_wait_ms.
GET /topics/{topic}/partitions/{p}/records?offset=90412&max_bytes=1048576

# Commit the position this group has finished processing.
POST /groups/{group}/offsets
{ "topic": "rides", "partition": 3, "offset": 90460 }

Note what is not here: there is no "delete this message" and no "acknowledge message X." A consumer simply moves a cursor. Committing offset 90460 means "this group has durably processed everything up to and including 90459; on restart, resume at 90460."

Core entities

  • Topic — a named stream, e.g. rides, payments. Purely logical.
  • Partition — a topic is split into N partitions; each is an independent ordered log. The unit of parallelism and ordering.
  • Record — {partition, offset, key, value, headers, timestamp}. Immutable once written.
  • Broker — a server that hosts a set of partition replicas on its disks.
  • Consumer group — a set of consumers sharing a group id; the cluster stores one committed offset per (group, topic, partition).

Where are committed offsets stored? Cleverly, in the log itself — a special internal compacted topic (Kafka calls it __consumer_offsets) keyed by (group, topic, partition). The system reuses its own durable, replicated primitive to store consumer state, so offsets survive broker failure for free.

04 The core idea

The log abstraction

Everything rests on one structure. A log is an append-only, totally-ordered sequence of records. You can only do two things to it: append a record to the end, and read records from a given position forward. There is no random insert, no update, no delete-in-the-middle. That radical simplicity is what makes it fast and replicable.

Offsets: a record's permanent address

Each record appended to a partition is assigned the next integer in sequence — its offset. Offsets are monotonic and never reused within a partition. The offset is not stored inside the record; it is implied by the record's position, which (as we will see in Step 07) lets the broker locate any record by arithmetic. Reading "from offset 90412" means "give me records 90412, 90413, 90414, …" — a sequential scan, the friendliest possible access pattern for a disk.

Partitions: parallelism and the ordering trade-off

A single log on a single machine has a hard throughput ceiling — one disk, one CPU. So we split a topic into N partitions, each an independent log living (potentially) on a different broker. Producers spread writes across partitions; consumers in a group spread reads across partitions. Throughput now scales with partition count.

But this is where the central trade-off bites: ordering is guaranteed only within a single partition, never across partitions of a topic. Records in partition 0 and partition 1 have independent offset sequences and no defined relative order. There is no cheap way to maintain a global order across machines without a global serialization bottleneck — which would destroy the parallelism we just bought.

Keys: buying per-entity ordering back

So how does GrabPay keep all events for one wallet in order while still spreading load? With the partition key. The producer hashes the record key and maps it to a partition: partition = hash(key) % N. Use wallet_id as the key and every event for that wallet lands in the same partition, hence in strict order — while different wallets scatter across partitions for parallelism. You get per-entity ordering exactly where it matters, and global parallelism everywhere else.

Interviewer follow-up: "How do you guarantee ordering?"

The crisp answer: ordering is per-partition only, and you route related records to the same partition by choosing a key (wallet_id, user_id). If a question demands strict global ordering, you must use a single partition — and then honestly admit you have capped that topic's throughput at one machine. Naming that trade-off out loud is the point.

05 The roles & the flow

Producers, brokers, consumer groups

Three roles cooperate around the log. Producers append records to partitions. Brokers are the servers that store partitions on disk and serve reads and writes. Consumers belong to a consumer group; within a group, each partition is owned by exactly one consumer, and that consumer tracks the group's committed offset for the partitions it owns.

Why the consumer pulls (and stores its own offset)

A subtle but crucial design choice: consumers pull batches from the broker rather than the broker pushing to consumers. Pull means a slow consumer simply asks for data less often — it cannot be overwhelmed, and there is no need for the broker to track per-consumer flow control. It also makes batching natural: a consumer fetches as much as it can handle in one round trip.

Because the consumer owns its offset, the broker is stateless about consumption. The broker does not know or care who has read what; it just stores bytes. This is why one topic can feed five consumer groups with zero coordination between them — each group's offset is independent. It is also why replay is trivial: rewind your committed offset to an earlier value and re-read.

The interactive diagram below walks one record end to end: producer appends → it lands at the next offset → it replicates to follower replicas → a consumer in a group reads it and commits its offset.

Pull-based consumers

  • Slow consumers self-throttle; no broker-side flow control
  • Natural batching for high throughput
  • Broker stays stateless about who read what
  • Cost: extra latency when idle (mitigated by long-poll max_wait_ms)

Push-based brokers

  • Lower latency when data is flowing steadily
  • Broker must track and respect each consumer's rate
  • A burst can overwhelm a slow consumer unless backpressure is added
  • Replay is harder — the broker owns delivery state
06 Durability

Replication & the in-sync replica set

One disk can fail; one broker can die. To make an acknowledged record survive that, each partition is stored as several copies — a replication factor of typically 3. One replica is elected leader; the others are followers. All writes and reads for a partition go through its leader; followers exist only to stay caught up so one of them can take over.

How replication works under the hood

The mechanism is pull-based, just like consumers: each follower continuously fetches new records from the leader and appends them to its own copy of the log, in the same offset order. A follower that is keeping up — within a configurable lag bound — is a member of the in-sync replica set (ISR). The ISR is the set of replicas that currently hold (or are about to hold) all the leader's committed data.

A record is considered committed — and only then visible to consumers — once all members of the ISR have it. The leader advances a marker called the high-water mark to the highest offset that the entire ISR has replicated. Consumers can never read past the high-water mark, so they never see a record that could vanish if the leader died.

The acks knob: latency vs. durability

The producer chooses how much confirmation to wait for via the acks setting. This is the single most important durability lever, and a guaranteed interview question.

  • acks=0 — fire and forget. The producer does not wait at all. Lowest latency, highest throughput, but a record can be lost before it ever lands. Fine for non-critical telemetry (a Grab heat-map ping).
  • acks=1 — wait for the leader to write it. Survives a consumer crash but not a leader crash that happens before followers caught up. A middle ground.
  • acks=all — wait for the whole ISR to have it. Strongest durability; a committed record survives any single (and, with care, multiple) broker failures. The right choice for a GrabPay wallet debit. Higher latency.

Pair acks=all with min.insync.replicas=2: the producer's write fails if fewer than 2 replicas are in sync, so you never accept a write that only one machine holds.

Interviewer follow-up: "What happens when the leader dies?"

A controller (in Kafka, the cluster's KRaft/controller quorum; historically ZooKeeper) detects the leader is gone and promotes one of the in-sync followers to leader for that partition. Because every ISR member holds all committed records up to the high-water mark, no acknowledged data is lost. Producers and consumers discover the new leader via metadata refresh and reconnect. The danger to name: unclean leader election — if you allow an out-of-sync replica to become leader (to restore availability faster), you trade away durability and can lose committed records. Default is to stay unavailable rather than lose data.

07 Storage

On-disk storage: why a log is fast

A common objection: "disk is slow, so won't a disk-backed queue be slow?" The answer is the most elegant part of the design, and it hinges on how the log writes and reads.

Sequential writes beat random writes by orders of magnitude

Because a partition is append-only, every write goes to the end of the current file — the disk head (or the SSD controller) never seeks to a random location. Sequential disk throughput is enormous; on commodity disks it rivals memory bandwidth, while random I/O is hundreds of times slower. The log's append-only constraint is not just for simplicity — it is precisely what turns "disk" into "fast." This is why the system can sustain gigabytes per second on spinning rust, let alone SSDs.

Segment files and how a read finds a record

A partition is not one giant file; it is a series of segment files, each capped at, say, 1 GB or 1 hour. The newest segment is the active segment being appended to; older segments are sealed and immutable. Alongside each segment sits an index file mapping offset → byte position. To serve "read from offset 90412," the broker finds the right segment (segment file names encode their base offset, so this is a binary search over file names), looks up the byte position in that segment's index, and starts a sequential read from there. No scanning from the beginning.

Splitting into segments is also what makes retention cheap: to drop data older than 7 days, the broker just deletes whole sealed segment files whose newest record is past the retention bound. No per-record bookkeeping, no expensive compaction of a single huge file.

Page cache + zero-copy: the read fast path

Two operating-system tricks make reads nearly free. First, the broker does not maintain its own data cache; it relies on the OS page cache. Records just written sit in RAM as dirty pages; a consumer reading the tail of the log (the common case) is served straight from page cache, never touching the disk. Second, zero-copy: to send bytes from a segment file to a consumer's socket, the broker uses a sendfile-style system call that moves data from page cache to the network card without copying it through user-space application buffers. Fewer copies, fewer context switches, far higher throughput. The broker is, in effect, a thin shim that points the kernel at the right bytes.

Under the hood: log compaction

Time/size retention is not the only mode. Log compaction keeps only the latest record per key and discards older ones, turning the log into a durable changelog of "current value per key." This is how the internal __consumer_offsets topic stays bounded, and how teams ship a compacted topic of "latest known wallet balance per user" that a new service can read from offset 0 to rebuild full state. The interviewer who asks "how would a brand-new service bootstrap current state?" is fishing for compaction.

08 Elasticity

Consumer-group rebalancing

Within a consumer group, every partition is owned by exactly one member, so the group's reading is parallelized without two consumers double-processing the same partition. But membership changes: a consumer instance is deployed, crashes, or autoscaling adds capacity for a Shopee flash sale. When that happens, partition ownership must be re-divided. That process is a rebalance.

How a rebalance works under the hood

One broker is designated the group coordinator for a given group. Each consumer sends periodic heartbeats to it. If a consumer joins, or stops heartbeating (crash, or a long GC pause exceeding session.timeout.ms), the coordinator triggers a rebalance: all members pause, re-register, and the coordinator (or an elected group leader) assigns partitions across the new membership. Each consumer then resumes from the committed offset of whatever partitions it now owns — which is why offsets live in the durable log, not in any single consumer's memory.

Worked example: GrabFood's order-events topic has 12 partitions and the kitchen-routing service runs 4 consumer instances, so each owns 3 partitions. A dinner-rush autoscale adds 2 more instances → rebalance → each of the 6 now owns 2 partitions. One instance later OOM-crashes → its 2 partitions are reassigned across the surviving 5. Note the ceiling: with 12 partitions, a 13th consumer instance would sit idle — partition count caps useful consumer parallelism, so you choose it generously up front.

The cost: stop-the-world pauses

The naive rebalance is "stop the world": every consumer stops processing while assignments are recomputed. During a flapping deploy this causes repeated pauses and latency spikes. Real systems mitigate with incremental cooperative rebalancing (only the partitions that actually need to move are revoked, the rest keep flowing) and static membership (a restarting instance reclaims its old partitions without triggering a full rebalance). Knowing these two mitigations by name signals production experience.

Interviewer follow-up: "How does a consumer resume after a crash?"

It does not resume in its own memory — its memory is gone. The coordinator reassigns the dead consumer's partitions to a surviving member, which reads the last committed offset from the durable offsets topic and continues from there. The nuance: any records the dead consumer processed but had not yet committed get re-delivered to the new owner. That is precisely the source of at-least-once duplicates — and the cue to discuss idempotency next.

09 Guarantees

What you can and cannot promise

Delivery semantics are where interviews get sharp, because the honest answer is full of trade-offs. There are three possible promises, and they come from when a consumer commits its offset relative to when it does its work.

At-most-once

Commit the offset before processing. If the consumer crashes mid-work, the offset already moved past the record, so it is never retried — the record is effectively lost. Zero duplicates, possible loss. Acceptable only for data you can afford to drop (a metrics sample).

At-least-once (the default)

Commit the offset after processing succeeds. If the consumer crashes after processing but before committing, the new owner re-reads from the old committed offset and re-processes those records. No loss, but duplicates are possible. This is the practical default, and it pushes the burden onto consumers: make processing idempotent. For a GrabPay debit, that means deduplicating on a stable event_id so replaying "debit ride 88421" twice still charges the wallet once.

Exactly-once — really?

Yes, but with asterisks, and saying it carefully is the win. True exactly-once across arbitrary external systems is impossible (the consumer might write to a database and crash before committing its offset; those two writes cannot be made atomic in general). What systems like Kafka actually provide is exactly-once within the platform, via two mechanisms: an idempotent producer (each producer gets an id and per-partition sequence numbers, so the broker silently drops a retried duplicate append) and transactions (a consume-process-produce cycle that atomically commits the output records and the input offsets together, so it is all-or-nothing). The moment your consumer's side effect lands in an external store, you are back to needing idempotency there. The honest interview answer: "exactly-once is real inside the log's own read-process-write loop; for external side effects, you still design idempotent consumers."

At-least-once

  • Commit after processing
  • No data loss
  • Duplicates possible → consumers must be idempotent
  • Simple, the right default for most pipelines

Exactly-once (in-platform)

  • Idempotent producer + transactions
  • No loss, no duplicates within the log read-process-write loop
  • Higher latency and complexity; transactional overhead
  • Breaks down at non-transactional external side effects
Ordering × delivery, together

Combine the guarantees you can actually offer: per-partition ordering plus at-least-once delivery, with optional in-platform exactly-once. You cannot offer global ordering across partitions, and you cannot offer exactly-once over arbitrary external systems. Stating both the promises and their precise limits is what separates a strong answer from a hand-wave.

10 Scale & failure

Scaling & what breaks at 10x

Assemble the full picture and stress it. Producers and consumers discover, via cluster metadata, which broker leads each partition and connect directly to it. Partitions are spread across brokers so that load and storage are balanced, and a controller quorum tracks broker liveness and partition leadership. Now push the system 10x and name what cracks first.

The hot partition

The most common real failure is skew. If you key payment events by merchant_id and one mega-merchant (a national e-wallet top-up partner during a Shopee 12.12 sale) generates 40% of traffic, its single partition becomes a bottleneck — one leader, one disk, one consumer must absorb 40% of the firehose while the others idle. The fix is a better key (compound key like merchant_id:hour, or salting the hottest keys across sub-partitions), or simply increasing partition count to dilute. Naming the hot-partition problem unprompted is a strong signal.

Backpressure and lag

When consumers fall behind producers, consumer lag (high-water mark minus committed offset) grows. The log absorbs the gap up to the retention limit — this is its superpower — but if lag exceeds retention, the consumer's next read finds its offset has been deleted and it skips forward, silently losing data. So you monitor lag as a first-class metric and alert before it approaches the retention window. Mitigations: scale out consumers (up to the partition ceiling), increase retention to buy time, or shed load.

Failure modes to name

  • Broker dies → controller promotes an in-sync follower to leader for each partition the dead broker led; clients refresh metadata and reconnect. Under-replicated partitions trigger re-replication to restore the replication factor.
  • Whole AZ/datacenter dies → if replicas are placed across availability zones (rack-aware placement), surviving AZs hold complete copies and serve through. This is why you spread the 3 replicas across 3 zones.
  • Disk fills → retention deletion frees space, but a runaway producer can outrun it; quota throttling on producers protects the cluster.
  • Network partition splits the controller quorum → the side without quorum stops electing leaders; you favor consistency over availability for metadata.

Adding a broker is the scaling move: it can host new partition replicas, and a partition reassignment migrates some existing replicas onto it to rebalance storage and load — a background copy that throttles itself so it does not starve live traffic.

The 10x question, in one breath

"At 10x, the first thing to crack is a hot partition from key skew — fix it with a better/compound key or more partitions. The second is consumer lag exceeding retention, which silently loses data — monitor lag and scale consumers up to the partition ceiling. Storage and write-amplification grow linearly, so add brokers and reassign partitions across AZs for both capacity and fault tolerance."

★ Review

Summary & self-test

The entire system compresses to one sentence: a topic is a set of append-only partitioned logs; the partition gives you both ordering and parallelism, replication gives you durability, and each consumer group reads independently by tracking its own offset. If you can derive every other piece — keys, ISR, segments, rebalancing, delivery semantics — from that sentence, you can carry the interview.
