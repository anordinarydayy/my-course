Design Full-Text Search

Type "red running shoes" into Shopee or Carousell and you get back relevance-ranked listings in tens of milliseconds — across hundreds of millions of items. One data structure makes that possible: the inverted index, a map from each word to the list of documents that contain it. Master it and you can whiteboard Elasticsearch, Lucene, and every product-search box.

About these notes: original study material on full-text search and the inverted index — a standard, widely-documented engineering topic. The explanations, examples, numbers, and diagrams here are my own; nothing is copied from any paid course or book. Two diagrams are interactive — press Play (or step with Next) to watch the flow.
00 The problem

Why scanning every row fails

Imagine Carousell holds 200 million listings, and a user searches for "vintage leather jacket". The naive approach is to scan every listing's title and description, check whether all three words appear, and return the matches. With a relational database this is a WHERE description LIKE '%leather%' query — and it is catastrophically slow, because a leading-wildcard LIKE cannot use a B-tree index. The database must read every row, every time. At 200M rows that is a full table scan per keystroke in the search box.

The deeper issue is that a B-tree indexes whole column values in sorted order, which is perfect for email = 'a@b.com' or range queries, but useless when the thing you want is a word buried inside a longer text field. Search needs an index keyed by the words themselves, not by the documents. That inversion — from "document → its words" to "word → the documents containing it" — is the entire idea, and it gives the structure its name.

Beyond speed, plain matching also has no notion of relevance. A user searching "leather jacket" does not want results in random order; they want the best matches first. So full-text search is really two problems welded together: find the matching documents fast, then rank them well. We will build up to both.

01 Scope it

Clarify and scope the requirements

Before drawing anything, narrow the problem out loud. "Search" can mean a Google-scale web crawler, a log-analytics system, or a product-search box — wildly different designs. For this lesson we will design product/document search in the spirit of Shopee or Carousell. State your assumptions and confirm them.

Functional requirements

  • Keyword search over a corpus of documents (here, product listings: title, description, brand, category).
  • Relevance ranking — return the best matches first, not an arbitrary order.
  • Freshness — when a seller posts or edits a listing, it should become searchable within seconds, not hours.
  • Filters and facets — narrow by price range, category, location, in-stock; show counts per facet ("Shoes (1,204)").
  • Pagination — return the top-k and allow paging deeper.

Non-functional requirements

  • Low latency — p99 search under ~100 ms; the box should feel instant.
  • High throughput — search is read-heavy and bursty (lunchtime, payday, 9.9 / 11.11 sales).
  • Horizontal scalability — the corpus grows; a single machine's RAM cannot hold the whole index.
  • High availability — a node dying must not take search down or lose data.
  • Eventual consistency is fine. A listing appearing a second or two late is acceptable; search is not a bank ledger.

Explicitly out of scope

Calling out non-goals is as valuable as listing goals — it shows judgement. We will not build the crawler/ingestion source of documents, autocomplete/typeahead (its own lesson), spell-correction, or semantic/vector search. We assume documents arrive from an upstream system (a catalog service emitting change events).

Interview tip

The single sentence that frames the whole design: "Search is a read-heavy problem where we trade a moderately expensive write path — building the index — for an extremely cheap read path." Everything that follows is a consequence of optimizing reads at the cost of writes.

02 Estimation

A quick back-of-the-envelope

Numbers turn a vague problem into a concrete one and tell you whether a single box can hold the index (it cannot). Assume the Carousell-style corpus:

  • Corpus: 200M listings, each averaging ~120 words of indexable text (title + description + attributes).
  • Search traffic: 50M searches/day. Spread evenly that is ~580 QPS; with a 5× peak factor, budget for ~3,000 QPS.
  • Write traffic: ~5M new/edited listings per day ≈ 60 writes/sec average. Reads outnumber writes ~10:1 even before peaks — confirming the read-heavy framing.

How big is the index?

The dominant cost is the posting lists — the document IDs stored under each term. A useful rule of thumb: total postings ≈ corpus size × average distinct terms per document. Say each listing has ~80 distinct terms after analysis: 200M × 80 = 16 billion postings. A doc ID is ~4–8 bytes raw, but posting lists compress hard (delta + variable-byte encoding) down to roughly 1–2 bytes per posting. Call it ~24 GB of postings, plus positions and the term dictionary — realistically 40–80 GB of index.

That comfortably exceeds the RAM you want on one node, and you also need replicas for availability. Conclusion: the index must be sharded. That single estimate justifies the entire distributed architecture in step 09 — which is exactly why interviewers want to see it.

Why store the document twice

The inverted index answers "which docs contain this word" but does not store the listing to show the user. You also keep the source documents — title, price, image URL — in a forward store (object storage or a doc store) keyed by doc ID. Search returns ranked IDs; a second fetch hydrates them into result cards. Budget the forward store separately: 200M × ~2 KB ≈ 400 GB.

03 Contract

The API and a data-model sketch

Pin down the interface before the internals. A search service has two surfaces: a query API the storefront calls, and an indexing API the catalog service feeds. Sketching real endpoints (going past the book) keeps the rest of the design honest.

Field mappings — the schema of search

Each field gets a mapping declaring how it is treated. This is the equivalent of a table schema and it drives everything downstream.

  • Analyzed text (title, description): run through the analyzer and put into the inverted index. Used for relevance.
  • Keyword (brand, category): stored verbatim, not tokenized. Used for exact filtering and facets.
  • Numeric / range (price): stored in a structure that supports range filters (a BKD-tree in Lucene), not the inverted index.

The lesson interviewers love: the same field is often indexed two ways — brand as analyzed text (so "nike" matches) and as a keyword (so the facet says exactly "Nike (612)"). Conflating the two is a classic mistake.

04 The core structure

The inverted index

The inverted index is a map from each term (a normalized word) to its posting list: the set of documents that contain the term. It is "inverted" relative to the natural forward layout (document → words). Reading it is the inverse direction: word → documents.

What lives in a posting list

A posting is one entry for one (term, document) pair. At minimum it holds the doc_id. To support ranking and phrase queries it usually also carries:

  • Term frequency (tf) — how many times the term appears in that document. More mentions of "leather" suggests the doc is more about leather. This feeds ranking.
  • Positions — the offsets where the term occurs (word 3, word 17, ...). Needed for phrase queries ("running shoes" as an exact phrase) and proximity scoring.
  • Field — which field the term came from, so a title match can outrank a description match.

Posting lists are stored sorted by doc ID. That ordering is not cosmetic — it is what makes the intersection in the next step fast, because two sorted lists can be merged in a single linear pass (a galloping/skip-list merge), the way you would zip two sorted arrays.

The term dictionary

To find a term's posting list you first look the term up in the term dictionary — a sorted, on-disk structure mapping each term to the byte offset of its postings. Lucene stores this as an FST (finite-state transducer), a compact trie-like structure that shares prefixes and suffixes, so a dictionary of tens of millions of terms fits in a few hundred MB and can stay memory-resident. The lookup is the search's first hop.

Under the hood — why posting lists compress so well

Because doc IDs in a list are sorted and ascending, you store deltas (gaps) instead of absolute IDs: [8, 51, 8841] becomes [8, +43, +8790]. Small gaps need few bits, and variable-byte or Frame-of-Reference encoding packs them tightly. A common term like "the" has billions of postings but tiny, near-constant gaps, so it compresses dramatically. This is why a 200M-doc index fits in tens of GB, not hundreds.

05 Building terms

The analyzer pipeline

How do you decide what a "term" is? Raw text like "Nike's Running SHOES!!" must be turned into clean terms before indexing. The analyzer is the pipeline that does this, and the cardinal rule is: the exact same analyzer must run at index time and at query time. If you lowercase documents but not queries, a search for "Nike" will never match the indexed term "nike". This symmetry is the most common source of "why does my search return nothing" bugs.

The stages, in order

  • Character filtering — strip HTML, normalize punctuation, fold accents (café → cafe).
  • Tokenization — split text into tokens on whitespace and punctuation: "red running shoes" → [red, running, shoes].
  • Lowercasing — fold case so "Shoes" and "shoes" are the same term.
  • Stop-word removal — optionally drop ultra-common words (the, a, of, dan, yang) that carry little meaning and bloat posting lists. Modern systems often keep them, because BM25 already down-weights them and dropping them breaks phrases like "to be or not to be".
  • Stemming / lemmatization — collapse word variants to a root. A stemmer (Porter, Snowball) chops mechanically: running, runs, ran → run. A lemmatizer uses a dictionary to find the true base form (better → good). Stemming is cheaper and usually enough.
  • Synonyms — expand or fold equivalents: sneakers → shoes, handphone → mobile phone (very relevant for SG/SEA vernacular).

A worked example

The listing title "Nike's Running SHOES — barely used!" flows through the pipeline to the term set {nike, run, shoe, bare, use} (after lowercasing, stemming, stop-word and punctuation removal). The query "running shoe" analyzed the same way becomes {run, shoe} — and now it matches, even though not a single surface form was identical. That is the whole payoff of analysis: matching by meaning of the word form, not by exact bytes.

Tools and knobs

In Elasticsearch / OpenSearch this is configured per field as an analyzer built from a char_filter, a tokenizer, and a chain of token_filters (e.g. lowercase, stop, porter_stem, synonym). The underlying engine is Apache Lucene. For CJK or Thai text — common in this region — you swap in a language-specific tokenizer (icu_tokenizer, kuromoji for Japanese) because whitespace splitting fails when words are not space-separated.

Interviewer follow-up

Expect: "You changed the analyzer — added a synonym. Do existing documents pick it up?" Answer: no. Analysis happens at index time, so the old documents hold the old terms. You must reindex the corpus (or at least the affected fields) for an analyzer change to take effect. This is why analyzer changes are treated like schema migrations.

06 Reading the index

Query execution, step by step

Now the read path — the part that must be fast. A query for "red running shoes" is executed in four moves: analyze the query into terms (using the same analyzer), fetch each term's posting list, combine the lists with a boolean operation, then score and rank the survivors and return the top-k.

Boolean combination: AND vs OR

How you combine posting lists defines recall. An AND (intersection) keeps only documents present in every term's list — precise but can return zero results if one rare word knocks everything out. An OR (union) keeps documents in any list — high recall but noisy. Most search engines use a smart default: OR the terms but boost documents that match more of them, so "red running shoes" still surfaces a perfect 3-term match at the top while a "red shoes" listing ranks below it. The intersection of sorted posting lists is the hot loop, and it runs in roughly linear time in the size of the shorter list thanks to skip lists embedded in the postings.

Notice the work is proportional to the size of the matching posting lists, not the size of the corpus. Whether you have 2M or 200M listings, "red running shoes" touches only the postings for those three terms. That is why search latency stays flat as the catalog grows — the defining win over the table scan in step 00.

07 Ranking

Relevance: TF-IDF and BM25

Finding matches is half the job; ordering them is the other half, and where good search is won or lost. The classic scoring model is TF-IDF, and its modern refinement BM25 is the default in Lucene/Elasticsearch. Both blend three intuitions.

The three ingredients

  • Term frequency (TF) — a document that mentions "leather" five times is probably more about leather than one that mentions it once. More occurrences → higher score.
  • Inverse document frequency (IDF) — a term that appears in almost every document ("shoes" in a shoe catalog) is barely informative, while a rare term ("Pegasus") is highly discriminating. IDF up-weights rare terms and down-weights common ones. It is computed as roughly log(N / df), where N is the total docs and df is how many contain the term.
  • Length normalization — without it, a 2,000-word listing would score high on everything just by being long. Normalizing by document length prevents verbose docs from dominating.

What BM25 fixes over plain TF-IDF

Plain TF-IDF makes term frequency grow linearly — the 20th occurrence of "leather" counts as much as the 2nd, which over-rewards keyword stuffing (a real spam problem for marketplace sellers). BM25 introduces saturation: TF contributes with diminishing returns, controlled by a knob k1 (typically ~1.2), so going from 1→2 mentions matters a lot but 19→20 barely moves the needle. A second knob b (~0.75) tunes how aggressively length normalization is applied. These two parameters are the main relevance dials you tune in practice.

A worked example

Searching "Pegasus" across 200M listings: only ~5,000 contain it, so its IDF is high — a match is meaningful. Searching "shoes" matches 40M listings, so its IDF is near zero — matching "shoes" alone tells you almost nothing. For the query "Pegasus shoes", BM25 therefore lets the "Pegasus" match dominate the score, which is exactly what a shopper hunting that model wants: the Pegasus listings float to the top, generic shoes sink.

TF-IDF

  • Simple, intuitive, easy to explain in an interview
  • Linear TF over-rewards repetition / keyword stuffing
  • Cruder length handling

BM25

  • TF saturates (k1) — resists stuffing
  • Tunable length normalization (b)
  • The industry default; better relevance out of the box

Going past lexical scoring

Real product search rarely ranks on BM25 alone. The text score is one signal blended with business signals: recency, seller rating, conversion rate, price competitiveness, in-stock, and boosts ("sponsored"). A common pattern is two-phase: BM25 cheaply retrieves a top-few-hundred candidates, then a heavier learning-to-rank model re-scores just those. Mentioning this layering shows senior judgement.

Interviewer follow-up

"How does BM25 rank?" — answer in one breath: "TF rewards repetition but saturates so stuffing doesn't help, IDF rewards rare/discriminating terms, and length normalization stops long docs from winning by sheer size; k1 and b are the tuning knobs." Then add that in production BM25 is one input to a broader ranking model.

08 Writing the index

Segments, merging, and freshness

So far the index has been read-only in our heads. But listings change constantly, and here is the catch: an inverted index is terrible at in-place updates. Inserting one doc ID into a sorted, compressed posting list means rewriting that list. Doing this for every write would make the index a thrashing mess. The solution is the key architectural idea of Lucene, and the reason Elasticsearch behaves the way it does.

Immutable segments

The index is not one big file. It is a set of segments — each a small, complete, immutable mini-index. New documents are buffered in memory and periodically flushed as a brand-new segment; existing segments are never modified. A search runs across all segments and merges their results. Immutability is a gift: segments need no locks for readers, can be cached aggressively, and compress optimally because nothing ever changes inside them.

Near-real-time indexing and the refresh interval

A new doc is searchable only once its segment is visible. The refresh interval (default ~1 second in Elasticsearch) controls how often the in-memory buffer is turned into a searchable segment. This is the freshness knob: shorter refresh = fresher results but more tiny segments and more CPU; longer refresh = cheaper indexing but staler search. For a flash sale you might lengthen it to survive the write storm; for live inventory you keep it short. Durability is handled separately by a translog (write-ahead log) that survives a crash before the segment is fsync'd.

Updates and deletes via tombstones

Since segments are immutable, you cannot delete a document in place. Instead you write a tombstone — a marker in a .del file saying "doc 8841 in segment 3 is dead." Queries skip tombstoned docs. An update is just a delete of the old version plus an insert of the new one as a fresh segment. The dead docs linger, taking space, until...

Segment merging

...a background merge runs. It combines several small segments into one larger segment, and crucially drops tombstoned documents while rewriting — reclaiming their space and keeping the segment count low (fewer segments = faster search, since each query touches every segment). Merging is I/O-heavy and competes with indexing for disk, so it is throttled. This is the classic LSM-tree pattern (also used by Cassandra and RocksDB): cheap appends, deferred compaction.

Short refresh interval

  • Fresher results — new listings appear in ~1s
  • Many small segments → slower queries
  • Higher CPU/IO from frequent flush + merge

Long refresh interval

  • Cheap, high-throughput indexing (bulk loads, sales)
  • Fewer, bigger segments → faster queries
  • Staler search — edits visible only after the interval
Interviewer follow-up

"How do you keep the index fresh as data changes?" — answer: changes flow as events; documents buffer in memory and become searchable on each refresh (near-real-time, ~1s); deletes/updates use tombstones; background merges compact segments and reclaim dead docs. The freshness-vs-cost trade-off is the refresh interval.

09 Scaling out

Sharding and scatter-gather

Step 02 proved the index is too big for one node. The fix is to shard: split the corpus into N partitions and put each shard's inverted index on a different node. The standard scheme is document-based partitioning — each document (and all its terms) lives entirely on one shard, chosen by hash(doc_id) % N. Every shard is a self-contained mini search engine over its slice of documents.

Scatter-gather

A query now cannot be answered by one shard, because matching documents are spread across all of them. So a coordinator node runs scatter-gather: it broadcasts the query to every shard (scatter), each shard searches its local index and returns its own local top-k, and the coordinator merges those partial lists into the global top-k (gather). To return the global top 20, each shard must return its top 20 — because a document ranked #18 on one shard could be the global #1. The coordinator then fetches the full documents for the winning IDs from the forward store and returns result cards.

Document- vs term-based partitioning

The alternative, term-based partitioning (each shard owns a slice of the vocabulary, so all postings for "shoes" live on one shard), sounds appealing — a single-term query hits one shard. But multi-term queries must ship whole posting lists between shards to intersect them, which is brutal at scale, and one hot term ("free") creates a hot shard. Document-based partitioning wins in practice precisely because it keeps each query self-contained per shard and spreads load evenly.

Document-based (the default)

  • Each shard fully answers a query locally
  • Even load; trivially add shards as the corpus grows
  • Cost: every query fans out to all shards (more network)
  • IDF is per-shard, a slight ranking approximation

Term-based

  • Single-term query touches one shard
  • Multi-term queries ship posting lists across the network
  • Hot terms create hot shards (skew)
  • Rarely used at scale

Replicas, availability, and failure

Each shard has one or more replicas on other nodes. Replicas serve reads (multiplying query throughput) and provide failover. What happens when a shard node dies? The coordinator routes that shard's queries to a surviving replica; results keep flowing. If a primary dies, a replica is promoted to primary and a new replica is rebuilt on another node from the existing data. The corpus stays fully searchable as long as one copy of each shard survives. This is exactly how Elasticsearch's primary/replica model behaves.

Interviewer follow-up

"How do you search across shards?" — scatter-gather: coordinator broadcasts, each shard returns its local top-k, coordinator merges to global top-k, then hydrates documents. The subtle trap: IDF is computed per shard, so scores aren't perfectly comparable across shards. For most corpora the skew is negligible; when it matters, a two-phase "dfs_query_then_fetch" first gathers global term statistics, then scores — at the cost of an extra round trip.

10 Wrap-up

Filters, facets, and what breaks at 10x

Filters vs the full-text query

A search like "red shoes under $80 in footwear" mixes a relevance query with hard filters. The crucial distinction: the text part is scored (BM25), but filters are pure yes/no and should be cached, not scored. Engines keep a filter cache (a bitset per filter, e.g. "category:footwear" → a bitmap of matching doc IDs) that is reused across queries and ANDed cheaply against the scored results. Putting price or category in the scoring clause instead of the filter clause is a common performance bug — it wastes scoring work and pollutes the relevance signal.

Facets and their cost

Facets (aggregations) are the counts beside each filter: "Nike (612), Adidas (488)". Computing them means, for the matching result set, grouping by a field and counting. This needs the field values laid out column-wise — the inverse of the inverted index — so engines maintain doc values: a per-field, doc-ID-ordered columnar store optimized for aggregation and sorting. Facets over high-cardinality fields (every seller ID) are expensive and memory-hungry; cap them or precompute.

What breaks at 10x

  • Deep pagination. Asking for from=100000&size=20 forces every shard to compute and ship 100,020 results just to discard most. The fix is search-after (cursor-based paging on a sort key), not offset paging.
  • Hot shards / skew. If a celebrity seller or one mega-category lands disproportionate traffic, document-hashing usually spreads it, but a bad shard count can concentrate load. Tune shard count to data and query volume; over-sharding adds coordinator overhead, under-sharding bottlenecks.
  • Indexing storms. A bulk catalog import at 10x write rate floods refresh and merge. Mitigate by lengthening the refresh interval during bulk loads, using bulk APIs, and routing indexing to dedicated nodes so it doesn't starve queries.
  • Thundering reads during a sale. 11.11 traffic spikes reads 10x. Add replica shards (reads scale horizontally with replicas), cache the hottest queries' results, and shed load gracefully.
  • The IDF-per-shard skew grows visible with very uneven shard contents; rebalance or use global term stats.
★ Review

Summary & self-test

The whole topic compresses to one sentence: invert the data so words point to documents, analyze text consistently at index and query time, combine and rank posting lists with BM25, write into immutable segments for fresh near-real-time updates, and shard with scatter-gather so it scales. If you can sketch the inverted index, the analyzer, the query flow, segments, and scatter-gather, you can carry the interview.
