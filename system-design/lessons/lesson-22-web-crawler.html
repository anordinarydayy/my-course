Design a Web Crawler

A web crawler walks billions of pages: fetch a URL, parse the HTML, extract the links, repeat. The art is doing this politely — without hammering any one host, without re-fetching the same page forever, and without falling into a calendar that generates new URLs to infinity. Every design decision orbits one data structure: the URL frontier.

00 The problem

What a crawler is for

A web crawler (also called a spider or bot) is a program that systematically discovers and downloads pages from the web. It starts from a small set of seed URLs, fetches each one, extracts the hyperlinks inside, and feeds those new URLs back in to be fetched later. Run that loop long enough and you traverse an enormous chunk of the reachable web.

Why would you build one? The motivating use cases shape every later decision, so name them up front:

  • Search-engine indexing — the classic case. Download pages so a separate indexer can build the inverted index that answers queries.
  • Archival — snapshot pages over time (think a national web archive or a compliance record).
  • Data mining & monitoring — a price-comparison engine watching merchant listings, or a fintech compliance team watching for leaked credentials and brand-abuse sites.
  • Product use in a super-app context — at a marketplace like Shopee you might crawl partner catalogues to enrich listings; at Agoda you might re-crawl hotel description pages to keep content fresh; a GrabPay risk team might crawl phishing-suspect domains that impersonate the brand.

The deceptively simple loop hides three hard problems that the rest of this lesson is about: scale (billions of pages means you cannot keep everything in memory), politeness (you must not look like a denial-of-service attack to the sites you visit), and termination (the web has cycles and infinite link generators, so a naive crawl never stops).

01 Scope it

Clarify the requirements

"Design a web crawler" is wide open. The interviewer wants to see you fence it in before you draw anything. Ask, then state the assumptions you'll proceed with.

Functional requirements

  • Input: a set of seed URLs. Output: downloaded page content (HTML) stored for a downstream consumer, plus the discovered link graph.
  • Scope of content: HTML pages only, or also images, PDFs, video? We'll assume HTML text pages, with a note on how to extend.
  • Freshness: one-shot snapshot, or a continuous crawl that re-visits pages as they change? We'll design for a continuous crawl.
  • Politeness: respect robots.txt and per-host rate limits — non-negotiable for a real crawler.

Non-functional requirements

  • Scalability: billions of pages, so the design must shard cleanly across many machines.
  • Robustness: the web is hostile — malformed HTML, redirects, timeouts, servers that hang. The crawler must tolerate all of it and never crash on a bad page.
  • Politeness (worth repeating as a non-functional goal): never overload a host.
  • Extensibility: adding a new content type or a new processing step should be a plug-in, not a rewrite.
  • Freshness: popular and fast-changing pages should be re-crawled more often than static ones.

Two cross-cutting design qualities fall out of this list and will recur throughout: we need a single canonical "seen" set so we never store the same URL twice, and we need per-host fairness so a site with a million pages doesn't get pummelled by a hundred workers at once.

Interview tip

The phrase that scores points here: "a crawler is fundamentally a graph traversal, but the graph is adversarial and unbounded — so the interesting design is in the frontier, the dedup, and the politeness layer, not in the fetch itself." Say that and the interviewer knows you've built one before.

02 Estimate

Back-of-envelope estimation

Numbers turn the vague "billions of pages" into concrete capacity targets, and they justify every later choice (why we shard, why we need a Bloom filter, why storage goes to object stores not a SQL row per page). Let's size a crawl that fetches one billion pages a month.

Crawl rate (QPS)

A month is roughly 2.5 million seconds (30 × 24 × 3600 ≈ 2.59M). So:

1,000,000,000 pages / 2,590,000 sec ≈ 386 pages/sec  (average)
peak ≈ 2× average ≈ ~800 pages/sec

That's modest per second but relentless. The constraint is rarely raw QPS — it's the politeness rule that you may only hit any single host every few seconds, so to sustain ~400 pages/sec you must be fetching from thousands of distinct hosts concurrently. Concurrency, not throughput, is the real number.

Storage

Assume an average HTML page is ~100 KB after we keep the raw markup (compresses well, but estimate raw):

1B pages × 100 KB = 100 TB / month of raw page content
with 2× freshness re-crawls retained ≈ 200 TB/month
over 1 year ≈ ~1.2 PB (before compression; ~3-5× less compressed)

That number alone tells you page bodies live in an object store (S3-style blob storage), not in a relational row. Metadata — URL, fetch time, status, content hash — is far smaller: maybe ~500 bytes/page, so 1B × 500B = 500 GB/month, which a wide-column store handles comfortably.

Bandwidth

386 pages/sec × 100 KB ≈ 38 MB/sec ≈ 310 Mbps sustained download
peak ≈ ~620 Mbps

The "seen" set

We will discover far more URLs than we fetch — say we encounter 10 billion unique URLs over the year. Storing each URL (avg ~80 bytes) in a hash set would cost 10B × 80B = 800 GB of RAM just to remember what we've seen. That single line is the entire justification for the Bloom filter we reach for in step 08.

Why these numbers matter

Estimation isn't ritual — each figure forces an architectural choice. 100 TB/month ⇒ blob storage. 800 GB of URLs ⇒ a probabilistic membership filter. ~400 pages/sec spread over thousands of hosts ⇒ a frontier organized by host, not a single global queue. Derive the design from the math and it writes itself.

03 Contracts

API & data model

A crawler is mostly an internal pipeline, but it still has interfaces worth sketching: an admin/control surface to seed and steer it, and the data shapes that flow between stages. Showing these makes the design concrete and goes past hand-waving.

Control API

A small internal API to manage crawls. These are not public endpoints — they're the operator's steering wheel.

POST /crawls                      # start a crawl job
  body: { seeds:["https://shopee.sg/"],
          maxDepth:8, maxPagesPerHost:50000,
          allowDomains:["shopee.sg"], priority:"high" }
  -> 201 { crawlId:"cr_8f2a" }

POST /crawls/{id}/seeds           # inject more seeds mid-flight
GET  /crawls/{id}/stats           # pages fetched, frontier size, error rate
POST /crawls/{id}/pause           # stop popping new URLs (drain in flight)
DELETE /crawls/{id}               # cancel

Data model sketch

Three logical stores, each chosen for its access pattern.

1. URL metadata — one row per known URL, the system of record for crawl state. High write volume, key-value access by URL hash. A wide-column store like Cassandra or DynamoDB fits.

url_hash      (PK)   # 64-bit hash of the normalized URL
url                  # the normalized URL string
host                 # parsed host, used for sharding & politeness
status               # NEW | FETCHING | FETCHED | FAILED | BLOCKED
content_hash         # hash of page body, for content dedup
first_seen_at
last_fetched_at
next_recrawl_at      # drives the freshness scheduler
fetch_count
error_count

2. Page content — the raw bytes, written once, read rarely, huge. Object storage (S3/GCS). The metadata row holds the object key.

key:  pages/{url_hash}/{fetch_ts}.html.gz   # gzip'd raw HTML
plus a small sidecar:  { contentType, httpStatus, headers, fetchedAt }

3. Per-host policy — robots rules and timing, cached so we don't re-fetch robots.txt for every URL.

host          (PK)
robots_rules         # parsed allow/disallow paths
crawl_delay          # seconds between requests to this host
last_access_at       # when we last fetched from this host
robots_fetched_at    # TTL: refresh robots.txt daily-ish
Follow-up they'll ask

"Why not one big SQL table?" Because the access patterns diverge: metadata is point-lookup-by-key at billions of rows (NoSQL's sweet spot), page bodies are large blobs (object storage), and the frontier is a queue (Kafka/Redis). Forcing all three into Postgres makes each one slow. Right tool per access pattern is the answer they want.

04 Big picture

High-level architecture

Now assemble the components into one pipeline. Read it left to right; the dashed line back from the link extractor to the frontier is the loop that makes it a crawler rather than a one-shot downloader.

  • URL frontier — the heart. A queue of URLs waiting to be fetched, organized so that priority and politeness can both be honored. Everything else feeds it or drains it.
  • Fetcher (downloader) — pops a URL, does DNS resolution (with a cache), opens the HTTP connection, downloads the body. Many fetcher workers run in parallel.
  • DNS resolver + cache — resolving a hostname can take tens of milliseconds; at hundreds of fetches/sec, uncached DNS becomes the bottleneck, so we cache resolutions.
  • Parser / link extractor — parses the HTML, validates it, and pulls out hyperlinks (and optionally the text for downstream indexing).
  • Content storage — writes the page body to the blob store and metadata to the wide-column store.
  • Dedup stages — a URL "seen" filter (so we don't enqueue the same URL twice) and a content dedup (so we don't store the same body twice).

The end-to-end flow for one URL: the frontier hands a URL to a fetcher; the fetcher resolves DNS (cache first), downloads the body, and passes it to the parser; the parser stores the content and extracts links; each extracted link is checked against the "seen" filter; brand-new URLs are normalized and pushed back into the frontier. Then the loop turns again. The next five sections each take one box and ask "what breaks here, and how do we fix it?"

05 The loop, step by step

The crawl loop

Before zooming into any single component, watch the full cycle for one URL. This is the sequence you'll narrate on the whiteboard, and it's the mental model that every later optimization plugs into. Press Play, or step through with Next and predict each transition before it happens.

Note the order of the checks. We verify robots.txt and politeness before we fetch — there's no point downloading a page we're not allowed to take, and no point hammering a host that asked us to wait. And we dedup the extracted links against the seen filter before enqueueing, so the frontier never fills with URLs we've already handled.

Under the hood — why DNS deserves its own cache

Inside the "fetch" box, the very first thing a worker does is turn shopee.sg into an IP. A cold DNS lookup is a network round trip of 20–120 ms. At 400 fetches/sec that's a wall of DNS traffic, and many fetches to the same host repeat the same lookup. A local DNS cache (respecting the record's TTL) collapses thousands of repeats into one resolution. Without it, DNS — not bandwidth — silently becomes your throughput ceiling.

06 The core data structure

The URL frontier

Everything hinges here, so it gets the most attention. The URL frontier is the queue of URLs discovered but not yet fetched. A first instinct is "it's just a FIFO queue — pop the next URL, fetch it." That naive version breaks in two ways at once, and watching it break tells us exactly what to build.

Bottleneck 1 — a plain FIFO has no priority

Not all URLs are equal. The Shopee homepage matters more than a buried filter-combination page; a frequently-updated news section deserves attention before a static "about us" page. A flat queue treats them identically, so important pages wait behind junk. We want priority: rank URLs (by site importance, page rank, or update frequency) and pull high-priority ones sooner.

Bottleneck 2 — a plain FIFO destroys politeness

Worse: imagine the frontier currently holds 10,000 URLs all from agoda.com (a big site fans out fast). A flat queue feeds those to your 100 fetcher workers, and suddenly 100 workers are downloading from Agoda simultaneously. To Agoda that's indistinguishable from an attack, and they'll block your IP range. We need the opposite guarantee: at most one in-flight request per host at a time, with a delay between requests to the same host.

The two-layer design that satisfies both

The standard answer is a front-queue / back-queue structure — two banks of queues with a router between them:

  • Front queues (prioritizer). Several FIFO queues, one per priority level (say 1–5). A prioritizer assigns each incoming URL a priority and drops it in the matching front queue. This layer is purely about what to fetch next.
  • Back queues (politeness). Many FIFO queues, where each back queue maps to exactly one host. A router moves URLs from front to back queues such that all URLs for a given host land in the same back queue. This layer is purely about not overloading anyone.
  • Queue selector + host table. Worker threads pull from back queues, but a mapping table enforces that a given host's queue is served by only one worker thread, and a per-host timestamp enforces the crawl delay. A worker that finishes a host waits out the delay before the next pull.

This split is the elegant part: priority and politeness are orthogonal concerns, and the two-layer frontier lets each be tuned independently. The front layer answers "what's most worth fetching?"; the back layer answers "who can I safely fetch from right now?".

Where the frontier physically lives

At billions of URLs the frontier can't sit in one process's memory. In practice the queues are backed by durable infrastructure: Kafka topics (partitioned by host, which conveniently gives you per-host ordering for free) or Redis lists/sorted sets for the hottest queues, with the long tail spilled to disk. The host-to-worker assignment table lives in Redis or ZooKeeper so all machines agree on who owns which host.

Follow-up they'll ask

"How big does the frontier get, and what if it doesn't fit in RAM?" It grows faster than you fetch (each page yields many links), so it must spill to disk. Keep a bounded in-memory buffer per queue for speed and page the rest from disk/Kafka. Also bound it: cap URLs-per-host and total depth (step 09) or the frontier grows without limit.

07 Be a good citizen

Politeness at scale

Politeness is the requirement most candidates under-weight and most interviewers care about, because getting it wrong gets your crawler banned. It has three distinct mechanisms.

Obey robots.txt

robots.txt is a file at the root of a site (https://host/robots.txt) where the owner declares which paths bots may and may not fetch, and sometimes a suggested crawl delay. Before fetching any URL on a host, the crawler fetches and parses that host's robots.txt, then honors its Disallow rules. You cache the parsed rules in the per-host policy store (with a TTL of, say, a day) so you don't re-download it for every URL — re-fetching robots.txt a million times would itself be impolite.

Honor a per-host crawl delay

Even on allowed paths, you space out requests. A typical rule is "no more than one request to a host every N seconds" (e.g. 1–5s). The host table stores last_access_at per host; a worker that wants to fetch from a host computes now − last_access_at and, if it's less than the delay, sleeps the difference. This is per-host, not global — you can fetch from a thousand different hosts in that same second.

Distribute politeness so one host isn't hit by many workers

This is the subtle one and the reason for the back-queue design. In a distributed crawler with hundreds of worker machines, the danger is two machines independently deciding to fetch from agoda.com at the same instant — each thinks it's polite, but together they double the load. The fix is the host-to-queue mapping: a given host is owned by exactly one back queue, and that queue is served by exactly one worker thread. Consistent hashing on the hostname assigns each host to one machine, so all of a host's traffic funnels through a single point that can enforce the delay. The unit of work is the host, not the URL.

Shard the frontier by host

  • Politeness is automatic — one host, one owner, one timeline
  • Per-host crawl delay is trivial to enforce locally
  • Maps naturally onto Kafka partitions / consistent hashing

Shard by URL hash instead

  • Perfectly even load balancing across workers
  • But a host's URLs scatter across all workers → no single point to throttle
  • You'd need cross-worker coordination on every fetch — slow and fragile
Follow-up they'll ask

"What about a giant host that dominates? You said one worker per host." A site with 50M pages crawled one-request-per-2s would take years on a single worker. Real systems relax this: split a huge host across a few queues by sub-path while still bounding total concurrency to that host, or negotiate a higher rate. Name the tension — politeness vs. coverage of mega-sites — and the interviewer is satisfied.

08 Don't do the same work twice

Dedup & the Bloom filter

The web is densely cross-linked: the same URL is reachable from thousands of pages. If we enqueue a URL every time we see it, the frontier explodes with duplicates and we fetch the same page over and over. We need a "seen" set — a record of every URL we've already discovered — and we check every extracted link against it before enqueueing. There are two layers of dedup.

URL dedup — and why a hash set won't fit

Recall the estimate: ~10 billion unique URLs at ~80 bytes each is ~800 GB just to remember them. A plain in-memory hash set is out of the question, and a database lookup on every extracted link adds a network round trip to the hottest path in the system. We need something tiny and fast.

Enter the Bloom filter: a probabilistic set membership structure. It's a bit array of m bits plus k independent hash functions. To add a URL, hash it with all k functions and set those k bits to 1. To test a URL, hash it the same way and check those k bits: if any bit is 0, the URL is definitely new (no false negatives); if all bits are 1, the URL is probably seen — but might be a false positive where other URLs happened to set those same bits.

The crucial asymmetry: a Bloom filter never returns a false negative. So if it says "new," it's truly new and we crawl it. If it says "seen," we skip it — and occasionally (the false-positive rate) we wrongly skip a genuinely new URL. For a crawler that's an acceptable trade: missing a tiny fraction of pages is fine; the prize is fitting the entire seen-set in a sliver of the memory a hash set would need.

# 10 billion URLs, target 1% false-positive rate
m ≈ 1.44 × k × n,   k ≈ 7   (optimal for 1% FP)
m ≈ 10B × ~9.6 bits ≈ 96 Gbit ≈ 12 GB

# vs a hash set of the raw URLs:  ~800 GB
# ~65× smaller — and a lookup is just k bit-tests, no network

Content dedup — different URL, same page

URL dedup catches the same address; it doesn't catch two different URLs serving the same content (e.g. ?ref=email vs ?ref=fb, or a page mirrored on www and non-www). For that we hash the page body: compute a checksum (MD5/SHA) of the content and keep a set of seen content hashes. If a freshly fetched page's hash already exists, we skip storing and indexing it — we've seen this content before.

Exact hashing misses near-duplicates: pages that differ only in a timestamp, a session ID, or an ad. For those, a SimHash (a locality-sensitive hash) produces fingerprints whose Hamming distance is small when the content is similar, so you can detect "this is 98% the same as a page we already have" and skip it. That's how you avoid storing a million almost-identical product pages that differ only by a tracking parameter.

Under the hood — where the filter lives when distributed

A 12 GB Bloom filter fits on one machine, but in a sharded crawler each worker owns a slice of the URL space (by host hash), so each shard keeps its own Bloom filter for the hosts it owns. That keeps the seen-check local and lock-free. A scaled implementation often uses a scalable/partitioned Bloom filter backed by Redis, or RocksDB on disk for the authoritative seen-set with the Bloom filter as the fast front-line check.

Follow-up they'll ask

"What's the downside of the Bloom filter, and how do you recover from a false positive?" A false positive silently drops a real URL — you never crawl that page. You can't detect it directly (that's the point of the structure). Mitigations: size the filter for a low FP rate, and back it with an authoritative on-disk seen-set you consult only when the filter says "seen," turning a probabilistic skip into a confirmed one at the cost of a disk read on hits.

09 Make it terminate & stay fresh

Traps, loops & freshness

The web is not a tidy DAG. It has cycles, and it has pages that generate infinite new links. A naive crawler walks straight into these and never comes back. Two defenses plus a freshness policy keep the crawl finite and useful.

Crawler traps — infinite URL generators

A crawler trap (or spider trap) is a region of a site that produces an unbounded stream of distinct URLs. The classic is an online calendar: /cal?date=2026-06-01 links to 2026-06-02, which links to the next day, forever — there's always a "next month" arrow. Faceted-search pages are another: every combination of filters (?color=red&size=l&sort=price&page=2…) is a new URL, and the combinations are astronomically many. Each is technically a "new" URL, so URL dedup doesn't save you — the URLs really are distinct.

Defenses, layered:

  • URL normalization. Before hashing, canonicalize: lowercase the host, strip default ports, remove fragments (#…), sort or drop tracking query params (utm_*, ref, fbclid), resolve ../ path segments. This collapses many trap variants into one URL the seen-filter already knows.
  • Depth / budget limits per domain. Cap how deep links chain from a seed (maxDepth) and cap total pages per host (maxPagesPerHost). A calendar can generate infinite URLs, but the budget cuts it off after, say, 50,000 pages — coverage you can live with.
  • URL-length and pattern heuristics. Absurdly long URLs or ones with deeply repeated path segments (/a/a/a/a/…) are trap signatures; drop them.

Loops — cycle avoidance

Pure cycles (page A links to B links back to A) are handled by the same seen-filter that does dedup: once a URL is marked seen, we never enqueue it again, so a cycle can't spin. Cycles are the easy case; the hard case is the trap above, where every URL is genuinely new and only budgets save you.

Freshness — when to re-crawl

A crawl is never "done" — pages change. But re-crawling everything uniformly wastes effort on static pages while letting fast-moving ones go stale. The policy is re-crawl by observed change frequency: track how often a page's content hash actually changes between fetches, and set next_recrawl_at adaptively. A news front page that changes hourly gets re-crawled hourly; a "Terms of Service" page that hasn't changed in a year gets pushed out to monthly. A scheduler scans next_recrawl_at and re-injects due URLs into the frontier — turning the one-shot crawl into a perpetual, self-tuning one.

Follow-up they'll ask

"How would you escape a crawler trap you didn't anticipate?" You can't enumerate every trap, so make budgets the backstop: hard caps on depth and pages-per-host bound the damage of any trap. Layer on monitoring — if one host's frontier share spikes abnormally, alert and auto-throttle or blacklist that host. The honest answer is "defense in depth: normalize, budget, and monitor — no single rule catches them all."

10 Scale & survive

Distribute & survive failure

A single machine can't fetch a billion pages a month politely. We scale out — and the moment we do, we inherit the standard distributed-systems failure questions. Here's how the crawler answers them.

Shard by host

We already chose the sharding key in step 07: consistent hashing on the hostname assigns each host to one worker machine. This is the right key for two reasons — it keeps all of a host's politeness state (last-access time, robots rules, crawl delay) local to one machine, and consistent hashing means adding or removing a machine only re-maps a fraction of hosts rather than reshuffling everything.

Failure mode: a fetcher node dies

If a worker crashes, the hosts it owned are now unserved and any URLs it had popped-but-not-finished are in limbo. Recovery rests on two ideas:

  • Checkpoint the frontier. The frontier is durable (Kafka/Redis with persistence, or a backing DB), not just in worker RAM. A URL is only removed from the queue after it's been fully processed and its links enqueued — like a message-queue ack. If a worker dies mid-fetch, the un-acked URL becomes visible again and another worker re-processes it.
  • Re-balance ownership. A coordinator (ZooKeeper/etcd) detects the dead node via heartbeat and re-assigns its hosts to surviving nodes via the consistent-hash ring. The new owner loads that host's policy from the per-host store and resumes.

Failure mode: a fetch fails

Individual fetches fail constantly — timeouts, 5xx errors, connection resets. The policy is retry with exponential backoff, capped at a few attempts; persistent failures mark the URL FAILED in metadata and schedule a re-try far in the future rather than blocking the queue. Crucially, one bad page must never crash a worker — wrap parsing in defensive error handling, because malformed HTML and hostile pages are guaranteed, not exceptional.

What breaks at 10×

Push from 1B to 10B pages/month (~4,000 pages/sec) and the pressure points shift:

  • DNS becomes a first-class service. A simple per-worker cache no longer cuts it; you run a dedicated, replicated DNS-resolution layer with prefetching, or you'll be DNS-bound long before you're bandwidth-bound.
  • The Bloom filter grows past one machine's RAM (~120 GB at 10B URLs / 1% FP), forcing a partitioned/sharded filter aligned with the host-shards.
  • Storage write throughput dominates: ~400 MB/sec of page bodies streaming into the blob store needs the writes batched and the metadata store partitioned by url_hash to spread the load.
  • The frontier can't be one Kafka cluster; partition it and accept that global priority becomes approximate per-shard priority.
  • Hot hosts (a handful of mega-sites with billions of pages) become stragglers — one polite worker can't drain them — so they get special handling (sub-path sharding, negotiated higher rates, or dedicated capacity).
Extensibility, as promised in the requirements

Because the pipeline is stage-based, new behavior is a new stage, not a rewrite. Want to also crawl PDFs and images? Add a content-type router after the fetcher that forwards each body to the right parser. Want to detect spam or malware (a fintech compliance use case)? Insert a classification stage between parse and store. The frontier, fetcher, and dedup layers don't change — that's the extensibility goal made real.

★ Review

Summary & self-test

The whole topic compresses to one sentence: a crawler is a graph traversal over an adversarial, unbounded web — so the design is a polite, dedup'd, fault-tolerant pipeline built around one priority-plus-politeness data structure, the URL frontier. If you can draw the pipeline, explain why the frontier has two layers, and name the trap-and-dedup defenses, you can carry the interview.
