CAP, PACELC & Consistency Models

When the network splits, a distributed datastore faces a brutal choice: keep answering with possibly-stale data, or refuse to answer so it never lies. CAP names that forced choice. PACELC and the consistency spectrum turn it from a one-time verdict into a per-request dial you can actually tune.

00 The problem

Why one machine isn't enough

On a single database server, consistency is free. There is one copy of the data, every write lands on it, and every read sees the last write. Ask for a GrabPay wallet balance and you get the true balance, full stop. The trouble is that one machine can't survive its own death, can't serve a region on the other side of the planet with low latency, and can't hold the data of a hundred million users. So we replicate: we keep multiple copies of each piece of data on multiple machines, often in multiple data centres and multiple countries.

The instant you have more than one copy, a new question appears that simply did not exist before: are the copies the same right now? A write has to travel from wherever it landed to all the other replicas, and that travel takes time — milliseconds within a data centre, tens to hundreds of milliseconds between Singapore and Frankfurt. During that gap, different replicas hold different values. If a read hits a replica that hasn't caught up yet, it returns stale data — a value that was true a moment ago but isn't anymore.

This lesson is about the rules that govern that gap. They are not optional engineering preferences; they are mathematical limits on what any replicated system can promise. Understand them and you can answer the two questions every distributed-systems interviewer eventually asks: "is your store CP or AP?" and "what consistency does this particular piece of data actually need?"

01 The fact of life

What a partition really is

A network partition is when the machines in your cluster can't all talk to each other, even though each one is alive and serving. A switch fails, a fibre link is cut, a security group rule goes wrong, a data centre's uplink saturates — and suddenly the cluster splits into two or more groups that can each see themselves but not the others. Crucially, from inside one group you cannot tell whether the machines on the other side are dead or merely unreachable. The packets just stop arriving.

The single most common misreading of CAP is "you give up Partition tolerance." You don't, and you can't. In a system that spans more than one machine connected by a network, partitions will happen — they are a property of the universe, not a checkbox. So partition tolerance is not negotiable for any real distributed datastore. The genuine choice CAP forces is what to do during a partition, and that choice is between consistency and availability. We'll make that precise in the next section.

Partition vs a dead node — why it matters

A dead node is easy: the others notice the missing heartbeats, route around it, and carry on. A partition is harder precisely because both sides are still up and still receiving client traffic. Imagine Shopee's order service replicated across two availability zones in Singapore. The link between the zones drops for 40 seconds. Zone 1 still serves shoppers; Zone 2 still serves shoppers. Each zone has live customers writing data. The question — should each zone keep accepting writes it can't share with the other? — has no free answer.

Why "C" here is not the ACID "C"

The C in CAP means linearizability — every read returns the most recent completed write, as if there were a single copy. It is not the C in ACID (which is about preserving database invariants like foreign keys). Conflating the two is a classic interview slip. When we say "strong consistency" in this lesson, we mean CAP's C: the system behaves as though there is exactly one up-to-date copy.

02 The theorem

CAP: the forced choice

The CAP theorem states that a distributed datastore can offer at most two of three properties at the same time:

  • Consistency (C) — every read sees the most recent write, or an error. There is one logical truth and reads never contradict it.
  • Availability (A) — every request to a non-failing node gets a non-error response, even if that response might be stale.
  • Partition tolerance (P) — the system keeps operating even when the network drops or delays messages between nodes.

The "pick two" framing is famous but a little misleading. The honest way to read it: since P is mandatory, your real decision is C vs A, and you only have to make that decision while a partition is actually happening. When the network is healthy, a well-built system can give you both C and A. CAP is a statement about the bad moments, not the good ones.

The mechanism — what actually forces the choice

Walk through it concretely. A network partition splits replicas into two groups, G1 and G2, that can't reach each other. A client sends a write to G1: set a SeaMoney wallet balance to S$120. A moment later another client reads the same balance from G2. G2 has not heard about the new write — the partition blocks the propagation. Now the system must do one of exactly two things:

  • Answer with the old value (S$100). That keeps it available but breaks consistency — the read is stale.
  • Refuse to answer (return an error or block) until the partition heals. That keeps it consistent but breaks availability — the client got no answer.

There is no third door. G2 physically cannot know about a write it never received. So during a partition you are choosing, for each request, between a possibly-wrong answer and no answer. That is the entire content of CAP.

Interview tip

If you say "we sacrifice partition tolerance for consistency," you've already lost points. The phrase that lands is: "partitions are unavoidable, so the only question is whether, during a partition, this data should fail closed for consistency or stay up and accept staleness."

03 Two camps

CP vs AP, with real systems

Systems are loosely sorted into two camps by which side of the C–A edge they default to during a partition.

CP systems — consistency first, refuse if unsure

A CP system keeps data consistent even if that means becoming unavailable to part of the cluster during a partition. The usual mechanism is a majority quorum run by a consensus protocol such as Raft or Paxos. The cluster elects a leader; writes must be acknowledged by a majority of nodes before they commit. When a partition splits the cluster, only the side that still holds a majority can elect or keep a leader and serve writes. The minority side, knowing it might be stale and can't form a quorum, steps down and refuses rather than risk lying.

Real examples: ZooKeeper, etcd (the brain behind Kubernetes), HBase, and Google Spanner. These are the systems you reach for when being wrong is worse than being briefly unavailable — service discovery, leader election, configuration, distributed locks, financial ledgers. If etcd loses quorum, the minority nodes return errors; Kubernetes would rather pause than schedule pods against stale state.

AP systems — availability first, reconcile later

An AP system keeps answering on every reachable node during a partition, accepting that different sides may temporarily diverge, then reconciles once the network heals. The mechanism is usually leaderless replication: any replica can take a write, writes spread gossip-style, and conflicts are resolved afterward with version metadata (vector clocks, last-write-wins timestamps) — the subject of the next lesson.

Real examples: Amazon DynamoDB (in its eventually-consistent mode), Apache Cassandra, Riak, and Amazon S3. These power shopping carts, social feeds, product catalogues, view counts — data where serving a slightly-old answer is far better than serving no answer. During a partition, both sides of a Cassandra cluster keep taking writes; afterward, the divergent copies are merged.

CP — choose consistency

  • During a partition the minority side rejects requests (errors / timeouts)
  • Reads never return stale data — there is one truth
  • Built on majority quorum + consensus (Raft/Paxos)
  • Best for: locks, config, leader election, ledgers, balances
  • Cost: lower availability; a quorum loss takes the service down
  • Systems: ZooKeeper, etcd, HBase, Spanner

AP — choose availability

  • During a partition every reachable node still answers
  • Reads may be stale; sides diverge and reconcile after healing
  • Built on leaderless replication + conflict resolution
  • Best for: carts, feeds, catalogues, counters, sessions
  • Cost: you must design for conflicts and stale reads
  • Systems: Dynamo, Cassandra, Riak, S3
The follow-up they always ask

"Is your system CP or AP?" The strong answer refuses the binary: "It depends on the data. We run the wallet ledger on a CP store so balances never lie, and the activity feed on an AP store so it never goes dark. CAP applies per datastore, and increasingly per request." That last clause is PACELC and tunable consistency — exactly where we're headed.

04 The fuller rule

PACELC: the fuller rule

CAP only talks about the rare moments when the network is broken. But most of a system's life is spent with a perfectly healthy network — and even then, consistency isn't free. To make a read see the absolute latest write, you have to contact enough replicas to be sure, and contacting more replicas (especially across regions) costs latency. PACELC captures both halves of the story in one line:

if (P)artition → trade (A)vailability vs (C)onsistency · (E)lse → trade (L)atency vs (C)onsistency

Read it as two clauses. The P clause is just CAP: under a partition, pick A or C. The new and arguably more practical clause is E (else): in normal operation, you still pick — between low latency and strong consistency. A store that always waits for all replicas to agree is consistent but slow; a store that answers from the nearest replica is fast but may be stale. Every replicated system sits somewhere on this dial even when nothing is broken.

How real systems classify

PACELC gives a two-letter tag of the form PA/EL, PC/EC, etc. A few honest labels:

  • Cassandra / DynamoDB → PA/EL. Available under partition, and tuned for low latency in normal times — both clauses lean away from strong consistency (though both are tunable).
  • etcd / ZooKeeper / HBase → PC/EC. Consistent under partition, and consistent in normal times too — they always pay the latency to be correct.
  • Google Spanner → PC/EC. Consistent on both clauses; it pays for it with TrueTime (GPS + atomic clocks) and commit-wait latency.
  • MongoDB → PA/EC (roughly). The defaults and write/read concerns let you slide this around.

The big upgrade PACELC gives you over CAP: it explains why two "AP" systems can feel completely different day to day. The partition behaviour is identical, but their normal-operation latency-vs-consistency stance is what you actually experience 99.99% of the time.

05 Not binary

The consistency spectrum

"Consistent vs eventual" is a useful headline, but consistency is really a spectrum of guarantees, each stronger than the last and each more expensive. Knowing the rungs lets you ask for the weakest guarantee that still keeps your feature correct — which is the cheapest one. From strongest to weakest:

Strong / linearizable

Linearizability is the gold standard: the system behaves as if there is a single copy of the data and every operation takes effect at one instant between its call and its return. Once a write completes, every subsequent read — from any client, anywhere — sees it. This is what a single-machine database feels like, recreated across many machines. It requires coordination (a leader or a quorum) on the critical path, so it is the slowest. Use it for money and locks.

Sequential consistency

Sequential consistency is slightly weaker: all clients agree on one global order of operations, and each client's own operations appear in the order it issued them — but that global order need not match real-time (wall-clock) order. Two clients can see the same sequence even if it lags reality. It's rarely the explicit target in product systems but is a useful theoretical rung between linearizable and causal.

Causal consistency

Causal consistency preserves cause-and-effect: if operation B could have been influenced by operation A (you read A, then wrote B), everyone sees A before B. Operations with no causal link can be seen in any order. This is exactly what you want for a comment thread — a reply must never appear before the comment it answers — while unrelated posts can arrive in any order. It's much cheaper than linearizability because only causally-related operations need ordering.

Read-your-writes

Read-your-writes consistency guarantees that after you write something, you will see it in your own subsequent reads — even if other users don't yet. A shopper who edits their Shopee delivery address must see the new address immediately on refresh, or they'll think the save failed. A common cheap implementation: route a user's reads to the same replica (or the primary) for a short sticky window after they write.

Monotonic reads

Monotonic reads guarantee you never move backward in time: if you've seen a value, later reads return that value or a newer one, never an older one. Without it, two reads that hit different replicas can show a like count of 50, then 48 — data appearing to "rewind," which feels broken. Sticky routing to one replica per session gives you this too.

Eventual consistency

Eventual consistency is the weakest useful guarantee: if writes stop, all replicas eventually converge to the same value. It says nothing about when, or about the order reads see in the meantime — only that you won't diverge forever. It's cheap and highly available, and perfectly fine for a view counter or a trending list, where being a few seconds behind harms nobody.

Client-centric vs data-centric

Read-your-writes and monotonic reads are client-centric guarantees — they're promises to one session, not about the global state of the data. That's why they're cheap: you often get them just by pinning a session to one replica, without any cluster-wide coordination. A great move in an interview is to reach for these instead of full linearizability when a feature only needs to look right to the user who acted.

06 The tuning dial

Quorums: how you actually tune it

So far the spectrum is abstract. Quorums are the concrete knob that lets a single AP-style store (Cassandra, DynamoDB, Riak) be dialed from eventual all the way up to strong, per request. Three numbers define it:

  • N — the replication factor: how many copies of each key exist. Say N = 3.
  • W — the write quorum: how many replicas must acknowledge a write before the client is told "OK."
  • R — the read quorum: how many replicas must respond to a read before the client gets an answer.

The whole trick is one inequality:

W + R > N  ⟹  the read set overlaps the latest write set

Here's the mechanism. A write is durably recorded on at least W replicas before it's acknowledged. A read collects responses from at least R replicas. If W + R > N, then by the pigeonhole principle the R replicas you read from and the W replicas that hold the newest write must share at least one node — they can't fit into N slots without overlapping. That shared node holds the latest value, the read includes it, and the client picks the freshest version (using a version number or timestamp attached to each value). That overlap is what turns "eventual" into "strong" on demand. Step through it:

Notice what W + R > N buys you: strong (linearizable-flavoured) reads on a store that's otherwise eventually consistent, chosen per request. Want speed instead? Drop below the threshold and accept staleness. The same cluster serves both.

Under the hood — sloppy quorums and hinted handoff

During a partition, Dynamo-style systems can use a sloppy quorum: if the "home" replicas for a key are unreachable, the write still goes to the first W reachable nodes, which stash it as a hint and replay it to the rightful owners once they return (hinted handoff). This keeps writes available during a partition — but it also means W + R > N no longer strictly guarantees overlap, so you can still read stale data. It's the AP camp choosing availability even while pretending to honour a quorum.

07 Setting the knobs

Tuning W and R in practice

Once you accept that W and R are dials, the design question becomes: which way do I turn them? The trade is between write latency, read latency, and consistency, and it depends on your read/write mix.

Read-heavy: push the cost onto writes

Most product systems read far more than they write — a Shopee product page might be read thousands of times for every price update. Set W = N and R = 1. Every write must reach all replicas (slow, but rare), while reads hit just one (fast, and frequent). Because the write touched every replica, any single replica you read is current, so you still get fresh reads. The failure cost: if even one replica is down, writes block, so availability of writes suffers.

Write-heavy: push the cost onto reads

For a write-heavy workload — say an event-ingestion or metrics pipeline — flip it: W = 1 and R = N. Writes ack from a single replica (fast), reads must consult all of them and pick the newest (slower, rarer). Now a single down replica blocks reads instead of writes.

Balanced: the quorum sweet spot

When you don't know the mix, the standard default is W = R = ⌊N/2⌋ + 1 — a strict majority on both. With N = 3 that's W = R = 2; with N = 5 it's W = R = 3. This keeps W + R > N (so reads are fresh) while tolerating the failure of one node (N=3) or two nodes (N=5) without losing the ability to reach quorum. It's the same majority idea CP systems use for consensus, applied as a per-request setting.

R = 1, W = N (read-optimized)

  • + Reads are fast and always fresh (every replica is current)
  • + Great when reads vastly outnumber writes
  • − Writes are slow; one dead replica blocks all writes
  • − Poor write availability during any node failure

R = N, W = 1 (write-optimized)

  • + Writes are fast and highly available
  • + Great for ingestion / append-heavy workloads
  • − Reads are slow and one dead replica blocks all reads
  • − Easy to read stale data if you accidentally drop R

A concrete config — Cassandra

In Cassandra these dials are the per-statement consistency level (CL). With a keyspace at replication factor 3:

-- read-heavy catalogue: fast, fresh reads
SELECT price FROM products WHERE id = ?
  USING CONSISTENCY ONE;        -- R = 1

UPDATE products SET price = ? WHERE id = ?
  USING CONSISTENCY ALL;        -- W = N = 3

-- a balance check that must be correct: majority on both
SELECT balance FROM wallets WHERE user_id = ?
  USING CONSISTENCY QUORUM;     -- R = 2 of 3

UPDATE wallets SET balance = ? WHERE user_id = ?
  USING CONSISTENCY QUORUM;     -- W = 2 of 3  →  W+R = 4 > 3

DynamoDB exposes the same idea more coarsely: a flag ConsistentRead=true forces a strongly-consistent read (it reads from enough replicas to be current) at extra cost and latency, while the default eventually-consistent read is cheaper and faster.

The follow-up they always ask

"This store is read-heavy — set W and R." The crisp answer: "N=3, W=3, R=1 so reads are single-replica-fast and always fresh, accepting that writes block if a replica is down. If write availability matters more than always-fresh reads, I'd move to W=R=2 — majority quorum — which still satisfies W+R>N but survives one node failure on both paths."

08 Matching data to model

Picking consistency per use case

The senior move is to refuse to pick one consistency level for the whole product. Different data has different tolerance for being wrong, and each tolerance maps to a rung on the spectrum. Walk a real super-app:

  • Wallet balance / payment ledger (GrabPay, SeaMoney) → strong / linearizable. A double-spend or a balance that briefly shows the wrong number is a financial and regulatory incident. Use a CP store or W+R>N quorum, and accept the latency. This is non-negotiable.
  • Order placement & inventory decrement → strong for the "is this item still in stock" check, so you don't oversell the last unit during a flash sale. Often backed by a transactional store or conditional writes.
  • Your own profile / address edits → read-your-writes. You must see your change immediately; other users seeing it a second late is harmless.
  • Comment threads / chat ordering → causal. Replies must follow what they reply to; unrelated messages can arrive in any order.
  • Like counts, view counts, "X people are looking at this" → eventual. Off by a few for a few seconds? Nobody cares, and the availability and speed are worth it.
  • Activity feed / recommendations → eventual. Freshness is nice, correctness-to-the-millisecond is irrelevant.

The reasoning pattern to verbalize: "What goes wrong if a reader sees a value that's a few seconds stale?" If the answer is "money moves incorrectly" or "we sell something twice," you need strong. If it's "a number is briefly a little off," eventual is not just acceptable — it's the right choice, because paying for strong consistency you don't need just buys latency and fragility.

09 When it breaks

Failure modes & conflicts

Weak consistency isn't free — it shifts work onto you. Here are the three failures it introduces, and how systems cope. Each is a likely interview probe.

Stale reads

The mildest: a read returns a value that was true a moment ago. With W + R ≤ N (or a sloppy quorum during a partition) the read set may miss the newest write. Mitigation: raise R, raise W, or for one session use read-your-writes / monotonic-reads stickiness. Detection: many stores tag values with versions, and read repair notices when replicas disagree on a read and quietly pushes the newest version to the laggards.

Lost updates

The dangerous one for counters and balances: two clients both read balance = 100, both add 50, both write 150. One increment vanishes; the balance should be 200. This is the same race a single-machine database prevents with a transaction, now multiplied across replicas. Fixes: atomic read-modify-write server-side (Cassandra/DynamoDB counters, conditional/compare-and-set writes), optimistic concurrency with a version check (write only if the version still matches what you read), or routing these operations to a CP store. Never do read-then-write of a balance over an eventually-consistent path.

Conflicting concurrent writes

The hardest: during a partition, two sides each accept a write to the same key with no causal link between them. When the partition heals, the system holds two equally-valid versions and must reconcile. The blunt option is last-write-wins (LWW) by timestamp — simple, but it silently throws away one user's write and is sensitive to clock skew. Smarter options track causality with vector clocks (or version vectors) so the system can tell a true conflict from a stale overwrite, and either auto-merge or hand both versions ("siblings") to the application to resolve. CRDTs — data types mathematically designed to merge without conflict — are the cleanest answer where they fit.

The bridge to Lesson 13

Conflict resolution — vector clocks, last-write-wins, sibling values, CRDTs, and read repair / anti-entropy — is a whole topic of its own, and the natural next question after "what happens to the writes that diverged during a partition?" If an interviewer pushes there, name vector clocks for detecting concurrency and CRDTs for conflict-free merging, and note that the right answer is usually to keep the data that can conflict (balances) on a strongly-consistent path in the first place, so you never have to merge it.

★ Review

Summary & self-test

The whole topic compresses to one sentence: partitions are inevitable, so during one you trade consistency for availability; and even when the network is healthy you trade latency for consistency — quorums and consistency levels let you make that trade per request, per data type. If you can place each data type on the spectrum and set W and R to match, you can carry the interview.
