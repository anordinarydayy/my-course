Leaderless Replication & Conflict Resolution

Leader-based replication funnels every write through one node, so the leader is a throughput ceiling and its death opens a failover window where nobody can write. Dynamo-style leaderless replication deletes the leader: any replica accepts any write. You buy availability and write throughput — and you pay for it with conflicts you can no longer prevent, only detect and reconcile.

About these notes: original study material on leaderless replication and conflict resolution — standard, widely-documented distributed-systems fundamentals popularized by Amazon's Dynamo paper and its descendants. The explanations, numbers, pseudocode, and diagrams here are my own; nothing is copied from any paid course or book. One diagram is interactive — press Play (or step with Next) to watch the flow.
00 The problem

The leader bottleneck

In Lesson 04 we built leader-based (single-leader) replication: one node is the leader, it accepts all writes, and it streams a change log to a set of read-only followers. That model is correct, simple to reason about, and powers most relational deployments. But it has two structural weaknesses that get sharper the more you scale and the more you care about staying up.

First, the leader is a write bottleneck. Every mutation in the system has to land on one machine before it can be acknowledged. You can add followers all day and your read capacity grows, but write capacity is pinned to a single box's CPU, disk, and network. For a write-heavy workload — say GrabPay ingesting a flood of payment-ledger appends during a 11.11 sale, or a wallet that must record every micro-transaction — that one node becomes the thing the whole system waits on.

Second, and worse, is the failover window. When the leader dies, no writes can be accepted until a new leader is elected and promoted. Detecting the failure (a few seconds of missed heartbeats so you don't promote on a transient blip), agreeing on a new leader, and redirecting clients all take time — typically seconds, sometimes tens of seconds. During that window the system is read-only or fully down for writes. For a payments product where a declined top-up at checkout is a lost transaction and an angry customer, a recurring multi-second write outage on every leader failure is unacceptable.

You can soften both with multi-leader replication, but that just moves the conflict problem around rather than solving it — and it still keeps the notion of designated leaders. The radical move is to ask: what if there is no leader at all?

01 The pivot

Going leaderless

In leaderless replication — the model Amazon described in its 2007 Dynamo paper and that Cassandra, Riak, and Voldemort later implemented — there is no special node. Every replica is equal and every replica accepts writes directly. The data for a given key is stored on N replicas (the replication factor), and the client (or a coordinator node acting on the client's behalf) sends each write to all N and each read to all N rather than to one designated owner.

This immediately fixes both leader weaknesses. There is no single box every write funnels through, so write throughput scales with the cluster. And there is no failover: if one replica is down, the client simply talks to the others — there is no election to wait on, because nothing was special about the dead node. Availability stays high precisely because no node is load-bearing on its own.

How a write and a read actually flow

The mechanics are deliberately simple, which is part of the appeal. On a write, the coordinator fans the new value out to all N replicas in parallel and waits for a configurable number of them to acknowledge. On a read, the coordinator queries all N (or enough of them) in parallel and gathers whatever versions come back. The coordinator does not serialize writes through a primary — two clients can write the same key at essentially the same instant, and both writes are accepted by whatever replicas they reach. That is the source of the system's power and, as we will see, the source of its conflicts.

Who coordinates — client or node?

There are two flavors of "the coordinator." In the client-driven style the client library itself knows the cluster topology, computes which N nodes own the key (via consistent hashing, Lesson 11), and fans the requests out directly. In the more common server-coordinated style the client sends one request to any node, and that node becomes the coordinator for that operation — it forwards to the N replicas, gathers responses, and replies. The choice is a trade between a fat, topology-aware client (one fewer network hop, but the client must track the ring) and a thin client (any node will do the work, at the cost of an extra internal hop). Cassandra uses server-side coordination; a node receiving a request acts as coordinator regardless of whether it owns the key.

The obvious worry: if a replica was down during a write, it now holds stale data; if a client reads only the replicas that missed the write, it sees an old value. Leaderless systems do not prevent this — they accept it and add machinery to repair it. The first and most important piece of that machinery is the quorum.

The core trade

Leader-based replication prevents conflicting writes by routing them through one ordering point. Leaderless replication allows conflicts and resolves them after the fact. Every other idea in this lesson — quorums, hinted handoff, read-repair, anti-entropy, version vectors — exists to make "allow then repair" safe enough to bet a database on.

02 The consistency knob

Quorums: why W + R > N works

Writing to "some replicas" and reading from "some replicas" only gives you a useful guarantee if the two sets are forced to overlap. That is exactly what a quorum does. Pick three numbers:

  • N — the replication factor: how many replicas store each key.
  • W — the write quorum: how many replicas must acknowledge a write before the coordinator calls it a success.
  • R — the read quorum: how many replicas the coordinator must hear back from on a read.

The magic condition is W + R > N. When it holds, the set of replicas that accepted the latest write and the set of replicas a read consults are guaranteed to share at least one node — by the pigeonhole principle, two subsets of an N-element set whose sizes add up to more than N must intersect. That one overlapping replica is holding the freshest write, so as long as the read collects versions from all R replicas and keeps the newest, it will see the latest write. This is how a leaderless system delivers strong-ish consistency on top of nodes that individually accept writes with no coordination.

Under the hood: how the read picks the winner

The overlap guarantees that the latest value is present in the read set, but it does not magically tell the coordinator which returned value is newest. Each stored value carries a version stamp (a timestamp, or a version vector — see Section 08). The coordinator gathers all R responses, compares their versions, and returns the most recent. The stale replicas it noticed along the way get fixed by read-repair (Section 05). So the quorum is two cooperating mechanisms: an overlap rule that guarantees the fresh value is somewhere in the read set, and a versioning scheme that lets the reader recognize it.

A worked example

Take the common default N = 3, W = 2, R = 2. A write must be acknowledged by 2 of the 3 replicas to succeed; a read must hear from 2 of the 3. Suppose a write of balance = v2 lands on R1 and R2 (that is 2 acks, so the write succeeds) while R3 was briefly unreachable and still holds v1. Now a read consults R2 and R3. R2 returns v2, R3 returns the stale v1. Because W + R = 4 > 3, the read set had to include at least one node from the write set — here it is R2 — so the newest value is in hand. The coordinator compares versions, returns v2, and (via read-repair) pushes v2 back to R3.

Under the hood: the partial-write problem

A subtlety interviewers love to corner you with: there is no rollback in a leaderless write. Suppose W = 2 but only 1 of the 3 replicas acknowledges before the coordinator gives up — the write is reported as failed to the client, yet that one replica already durably stored the new value and the system does not undo it. A later read can therefore surface a value from a write that "failed." Worse, if you read that key right after, the answer depends on which replicas you happen to consult. This is why quorum reads alone aren't linearizability: a failed-but-partially-applied write blurs the line. Real systems live with it because the repair machinery (read-repair, anti-entropy) eventually propagates that orphaned value to all replicas anyway — the write becomes durable in retrospect rather than being cleaned up.

Interviewer follow-up

"What does W + R > N actually guarantee — and what does it NOT?" It guarantees that a read overlaps the most recent successful write, so you won't read a value older than the last completed write. It does not guarantee linearizability, it does not stop two concurrent writes from conflicting, and the guarantee gets fuzzy under sloppy quorums (Section 04) and partial write failures (a write that got 1 ack out of a needed 2 leaves some replicas updated and some not, with no rollback). Say that out loud and you have shown you understand the limits, which is what the question is really probing.

03 Tuning the knob

Tuning W and R for your workload

The beauty of separating N, W, and R is that they are tunable per operation. As long as you respect W + R > N for strong reads, you can slide the cost between writes and reads to match traffic. The intuition: a larger quorum is more durable and consistent but slower and less available (you must wait for more nodes, and you fail if too many are down); a smaller quorum is faster and more available but weaker.

Read-heavy vs write-heavy

If your workload is read-heavy — a product catalog, a user-profile service hit on every Shopee page load — you want cheap reads. Set R = 1 and W = N. Reads touch a single replica (fast, highly available), and the cost is moved onto writes, which must reach every replica. With W = N = 3 and R = 1, the overlap still holds (3 + 1 > 3). The downside: writes now fail if even one replica is down, so this trades write availability for cheap reads.

If your workload is write-heavy — an event log, a metrics ingest pipeline, a wallet append stream — flip it: W = 1 (well, usually W = 2 to keep a little overlap room) and a larger R. Writes are accepted as soon as one or two replicas ack, so ingest stays fast and available even when nodes are flaky, and reads pay the price of consulting more replicas.

Larger quorum (e.g. W or R = N)

  • Stronger consistency, more durable
  • Higher latency — wait for more nodes
  • Lower availability — fails if a few nodes are down
  • Good when correctness > speed for that op

Smaller quorum (e.g. W or R = 1)

  • Lower latency, higher availability
  • Weaker freshness — may miss the newest write
  • Survives more node failures
  • Good when speed/uptime > strict freshness

The latency story is subtle and worth saying explicitly: the coordinator sends to all N replicas regardless, and the operation's latency is governed by the W-th (or R-th) fastest response, not the slowest. A larger quorum waits for slower nodes, so a single straggling replica drags up your tail latency more as W and R grow. This is why a read-heavy system often sets R = 1 even when it could afford R = 2 — it shaves the tail.

Interviewer follow-up

If you set W + R ≤ N on purpose — say N = 3, W = 1, R = 1 — you have chosen eventual consistency: lower latency and higher availability, but a read may return stale data because the read and write sets can miss each other entirely. That is a legitimate, common choice (think a "last seen online" timestamp). The interviewer wants to hear that you know it is a deliberate trade, not a bug. Connect it back to Lesson 12: quorums are the dial that moves you along the CAP/PACELC spectrum.

04 Availability machinery

Sloppy quorums & hinted handoff

A strict quorum has a brittle edge. Suppose N = 3 and W = 2, but a network partition cuts the client off from two of the three home replicas (the nodes that actually own this key by the partitioning scheme — see Lesson 11). The client can reach plenty of other nodes in the cluster, just not the two it needs. Under a strict quorum the write fails: not enough home replicas are reachable. But that feels wasteful — there are healthy nodes right there willing to accept the data.

Sloppy quorum: accept writes on stand-ins

A sloppy quorum relaxes the rule: as long as the coordinator can reach any W nodes in the cluster, the write succeeds — even if some of those nodes are not among the key's home replicas. The reachable-but-wrong nodes accept the write to keep the system available. This raises write availability dramatically during partitions, which is exactly what a Dynamo-style store is built to optimize for.

Hinted handoff: replay the data home later

The stand-in node knows it is not the rightful owner, so it stores the value along with a hint — a note recording which home replica this data really belongs to. This is hinted handoff. The stand-in holds the data temporarily and, the moment it detects that the proper home replica has recovered, it forwards (hands off) the value to it and deletes its local hinted copy. The data finds its way home asynchronously; the client never had to wait.

The catch: a sloppy quorum is not a real quorum

While the data is sitting on a stand-in, the W + R > N overlap guarantee is suspended. A read that consults only the home replicas can completely miss a write that is currently parked on a stand-in, because the stand-in is not in the read set. So a sloppy quorum buys write availability at the cost of a weaker freshness guarantee during failures — it is "we'll durably accept your write somewhere" rather than "your write is on the canonical replicas." In Cassandra this exact behavior is configurable per-keyspace; DynamoDB and Riak implement hinted handoff as a default availability feature.

Interviewer follow-up

"How do replicas catch up after a partition heals?" Three mechanisms stack up: hinted handoff replays writes that were parked on stand-ins; read-repair (next section) fixes stale replicas opportunistically whenever they are read; and anti-entropy (Section 06) is the background sweep that catches everything the first two missed. Name all three and explain that hinted handoff handles the recent stuff while anti-entropy is the safety net for cold, rarely-read keys.

05 Repair on the read path

Read-repair

Quorums guarantee a read sees the latest value, but on their own they leave stale replicas stale. If R3 missed a write, it keeps serving old data until something corrects it — and the next read that happens to pick a different pair of replicas could get unlucky. Read-repair is the cheap, opportunistic fix: whenever a read discovers that the replicas disagree, it writes the freshest value back to the stale ones inline, as part of serving that read.

The mechanism is elegant because it piggybacks on work you are already doing. The coordinator was going to read R replicas anyway; it compares their versions, returns the newest to the client, and — noticing R3 is behind — fires an extra write to R3 to bring it up to date. No separate process, no schedule. Frequently-read keys effectively self-heal: the more a key is read, the faster its stale replicas converge.

Read-repair has one important limitation, which sets up the next section. It only fixes replicas that get read. A key that is written once and rarely read — cold data — can have a stale or missing replica that read-repair never touches, because no read ever consults the laggard. For those, you need a background process that does not wait for reads.

06 Repair in the background

Anti-entropy & Merkle trees

Anti-entropy is the background repair process that continuously compares the data held by different replicas and copies over anything that is missing or stale — the safety net that catches cold keys read-repair never sees. The naive version is hopeless: comparing every key on two replicas means transferring or hashing the entire dataset on each, which is enormous and constant network and disk cost for a database that might hold billions of keys. You need a way to find the differences cheaply without shipping everything.

How a Merkle tree localizes the diff

The standard trick is a Merkle tree (a hash tree). Each replica partitions its key space into ranges, and for each range computes a hash of all the data in it — these are the leaves. Every parent node stores the hash of its children's hashes, all the way up to a single root hash that summarizes the entire dataset. Two replicas compare trees top-down:

  • Compare root hashes. If they match, the two datasets are identical — done, with a single hash exchanged. This is the common, cheap case.
  • If the roots differ, descend into the children and compare their hashes. Identical subtrees are skipped entirely; you only follow branches whose hashes disagree.
  • Keep descending only the differing branches until you reach the leaves — the specific key ranges that actually diverge. Sync only those ranges.

Because matching subtrees are pruned at every level, the work is proportional to the size of the difference (logarithmic in the dataset for a localized diff), not the size of the data. If two replicas differ in one key range out of thousands, you exchange a handful of hashes to pinpoint it instead of streaming the whole store.

Cassandra runs this during nodetool repair and builds per-range Merkle trees to decide what to stream between replicas; Riak and DynamoDB use the same family of technique for background convergence. The trade-off is cost timing: anti-entropy is heavier than read-repair (it scans and hashes data even for keys nobody is touching) so it is run periodically or during quiet hours, while read-repair handles the hot path for free.

Why two repair mechanisms

Read-repair is reactive and free but only covers read keys. Anti-entropy is proactive and complete but costs background I/O. Together they give you the property leaderless systems promise: eventual consistency — given enough time and no new writes, all replicas converge to the same value, whether the key is hot or stone cold.

07 The hard problem

Concurrent writes & the conflict problem

Repair handles staleness — one replica simply being behind another. But leaderless replication has a deeper problem that repair cannot solve by itself: concurrent writes. Because any replica accepts writes with no global ordering, two clients can write the same key at the same time, each reaching different replicas first. Now different replicas hold genuinely different values, and neither is "stale" — they are concurrent. Which one wins? There is no leader to have decided.

Last-write-wins (LWW), and why it loses data

The simplest answer is last-write-wins: attach a timestamp to every write and keep the one with the highest timestamp, discarding the rest. It is trivial to implement and Cassandra uses it by default. But it has two teeth. First, it is lossy — if two writes are genuinely concurrent, LWW silently throws one away. Imagine two of a customer's devices both topping up a wallet at the same instant: LWW keeps one top-up and drops the other, and money quietly vanishes. Second, it is clock-dependent — "highest timestamp" relies on synchronized wall clocks across machines, and clock skew means the write that physically happened later might carry an earlier timestamp and lose. LWW is fine for data where loss is acceptable (a cache entry, a "last seen" marker) and dangerous everywhere else.

Happens-before vs concurrent

To do better than blindly trusting clocks, you need to distinguish two relationships between writes. Write A happens-before write B if B was made by a client that had already seen A (B is a deliberate update of A — A is safely superseded). Two writes are concurrent if neither client had seen the other's write when it wrote — there is no causal order, and the system genuinely cannot say which "should" win. The crucial insight from the Dynamo paper: the database cannot resolve a true concurrency on its own, but it can reliably detect it and hand both versions to the application — these competing versions are called siblings. Detecting concurrency correctly, without trusting wall clocks, is the job of version vectors.

Interviewer follow-up

"Two clients write the same key concurrently — what happens?" The model answer: with LWW one write is silently dropped (call out the data-loss and clock-skew risks); with version vectors the system detects that the writes are concurrent rather than ordered, keeps both as siblings, and defers resolution to read time — either the application merges them or a CRDT merges them automatically. The phrase that scores points: "a leaderless store can't prevent the conflict, so it makes the conflict visible and reconcilable instead of pretending it didn't happen."

08 Detecting concurrency

Version vectors

A version vector (the generalization of a vector clock to replicated data) is the bookkeeping that lets the system decide whether one write supersedes another or whether the two are concurrent — without trusting wall clocks. The idea: each replica keeps a counter, and a value carries a vector of the counters it has seen — essentially a fingerprint of "what this value knows about." Comparing two vectors tells you their causal relationship.

How it works under the hood

Each value stored for a key carries a version vector — a map from replica id to a counter. When a replica processes a write, it increments its own entry and merges in (takes the element-wise maximum of) the vector the client presented. To compare two vectors:

  • If every entry in vector X is ≥ the matching entry in Y (and at least one is strictly greater), then Y happens-before X — X is the newer, superseding value, and Y can be discarded.
  • If neither dominates — X is ahead on some replica's counter and Y is ahead on another's — the writes are concurrent. Keep both as siblings.

Crucially, when a client reads, the server hands back the value and its version vector (often opaquely, as a "context" token); the client must present that vector on its next write to that key. That is how the server learns the client "had seen" the prior version — it is how causality is tracked without any clock.

A worked example: two concurrent cart writes

Picture a Shopee shopping cart on key cart:42, replicated across replicas A and B. Two of the customer's devices act at once, each with an empty starting context:

start:    cart:42 = { }        vv = {}

Device 1 → replica A:  add "milk"
   A increments its counter:
   cart:42 = {milk}            vv = {A:1}

Device 2 → replica B (concurrently, hasn't seen Device 1):  add "eggs"
   B increments its counter:
   cart:42 = {eggs}            vv = {B:1}

Now compare the two versions:
   {A:1}  vs  {B:1}
   A:1 > A:0  but  B:0 < B:1   → neither dominates
   ⇒ CONCURRENT  → keep both as siblings

A read of cart:42 now returns TWO values (siblings):
   sibling 1: {milk}  vv={A:1}
   sibling 2: {eggs}  vv={B:1}

Notice the database did not pick a winner and did not lose data. It reported the truth: "these two writes happened concurrently, here are both, you decide." Had Device 2 instead first read {milk} vv={A:1} and then added eggs, its write would carry vv={A:1,B:1}, which dominates {A:1} — that is a clean happens-before, no sibling, the cart becomes {milk, eggs}. The vector is what tells those two scenarios apart.

09 Reconciling

Resolving conflicts

Detection is half the battle; something must eventually collapse siblings back into a single value. There are three families of strategy, in increasing order of how much the database does for you.

Last-write-wins (database picks)

The database silently keeps the highest-timestamp version and drops the rest. Zero application effort, but lossy and clock-dependent as covered above. Acceptable only when losing a concurrent write does not matter.

Application / client-side merge (the Dynamo shopping-cart way)

The database returns the siblings; the application merges them using domain knowledge the database doesn't have. The classic Dynamo example is the shopping cart: a cart is a set of items, and the natural merge of two concurrent carts is their union. So siblings {milk} and {eggs} merge to {milk, eggs} — the customer keeps everything, which is the customer-friendly default. The application writes the merged value back with a version vector that dominates both siblings, retiring the conflict. The famous caveat: union-merge means a removed item can resurrect if it was deleted on one side while added on the other — which is why deletes in such systems often use a tombstone (a "deleted" marker that itself participates in the merge) rather than just dropping the item.

CRDTs (automatic, provably convergent merge)

A CRDT (Conflict-free Replicated Data Type) is a data structure designed so that its merge function is commutative, associative, and idempotent — meaning concurrent versions can always be merged automatically into the same result regardless of order, with no application logic and no lost updates. A grow-only counter, an OR-set (observed-remove set), a last-writer-wins register: each defines its own deterministic merge. Riak ships CRDT types (counters, sets, maps) so the database resolves siblings for you; Redis offers CRDT-based active-active geo-replication on the enterprise side. The trade is expressiveness: not every domain maps cleanly onto a CRDT, and when it doesn't you fall back to application merge.

Last-write-wins

  • Trivial; database handles it
  • Lossy — drops concurrent writes
  • Depends on synchronized clocks
  • OK for caches / "last seen", risky for money

Siblings + merge (app or CRDT)

  • No silent data loss
  • App merge needs domain logic; CRDTs automate it
  • Must handle deletes via tombstones
  • Right choice when losing a write is unacceptable
Failure mode & recovery

The dangerous failure here is silent: choose LWW for data that can't tolerate loss, and concurrent writes vanish with no error and no log. Recovery is a design decision, not a runtime fix — pick siblings + a merge strategy before you ship, model your data as a set/counter/CRDT where you can, and use tombstones so deletes don't get clobbered by concurrent adds. In an interview, naming the wallet/cart loss scenario and the tombstone caveat shows you have actually thought past the happy path.

10 In the wild

Where this lives

Leaderless replication is not a theory exhibit — it is the engine room of several databases you will name in interviews, and the knobs we have discussed are real configuration:

  • Amazon Dynamo / DynamoDB. The 2007 paper that started it all: N/W/R quorums, sloppy quorums, hinted handoff, vector clocks, and application-side reconciliation, all motivated by Amazon's "always writable" shopping cart. DynamoDB the managed service hides much of this but inherits the lineage.
  • Apache Cassandra. The most common interview reference. Per-query tunable consistency: you literally pass a consistency level like ONE, QUORUM, or ALL per read and per write, which sets R and W. It uses LWW by default (timestamp-based), read-repair, and Merkle-tree anti-entropy via nodetool repair.
  • Riak. Closest to the Dynamo design: explicit N/R/W, vector clocks (later dotted version vectors), sibling values surfaced to the app, and built-in CRDT data types for automatic merge.
  • Voldemort. LinkedIn's open-source Dynamo clone, vector-clock based — historically important even if less used now.

Tie the threads together: quorums are the concrete dial that moves a leaderless store along the consistency–availability spectrum from Lesson 12 (CAP / PACELC) — small quorums lean AP, W + R > N leans toward CP-ish reads. And everything in this lesson is a building block for the key-value store capstone in Lesson 21, where you assemble consistent hashing for partitioning (Lesson 11), leaderless quorum replication for availability, and version vectors plus read-repair and anti-entropy for convergence into one coherent design.

Interview tip

When an interviewer asks you to design a highly-available key-value store or a shopping cart that's "always writable," reach for this stack explicitly: "leaderless replication with N=3, tunable W/R quorums, hinted handoff for partition tolerance, read-repair plus Merkle-tree anti-entropy for convergence, and version vectors with application-side merge — like Dynamo/Cassandra." That single sentence demonstrates the whole lesson.

★ Review

Summary & self-test

The whole lesson compresses to one trade: delete the leader to win write throughput and availability, then pay for it with conflicts you detect and reconcile instead of prevent. Quorums give you a freshness dial, hinted handoff and the two repair mechanisms drive convergence, and version vectors plus a merge strategy turn concurrent writes from silent data loss into a visible, reconcilable event. If you can draw the quorum overlap, explain Merkle-tree pruning, and walk the cart-sibling example, you can carry this interview.
