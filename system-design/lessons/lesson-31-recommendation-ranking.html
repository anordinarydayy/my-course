31Lesson · System Design Interview

Design a Recommendation / Ranking System

"Design the For You feed" is not the news-feed question. The feed is reverse-chronological; the For You page is personalized and ranked. The expected answer is a two-stage recommender: cheaply retrieve a few hundred candidates from billions of items, then score them with a machine-learning model and return the best handful — all inside a tight latency budget. This lesson walks that funnel and the serving infrastructure around it.

About these notes: original study material on recommendation and ranking systems — a standard, widely-documented engineering topic. Examples lean toward Singapore product and super-app companies (TikTok, Shopee, Sea, Grab, Agoda) but the concepts are general. The explanations, numbers, and diagrams are my own; nothing is copied from any paid course or book. Three diagrams are interactive — press Play (or step with Next) to watch the flow.
00 The problem

A feed sorts by time; a recommender sorts by you

If you have already done the news-feed lesson, file this one under a sharply different category. A news feed (think a friends-and-follows timeline) answers "what's new from the accounts I follow?" Its hard problem is fan-out — copying a new post into the timelines of millions of followers — and its ordering is essentially reverse-chronological with light tweaks. The candidate set is bounded: it's the union of who you follow.

A recommendation / ranking system answers a much harder question: "out of every item in the catalog — most of which you've never interacted with and don't follow — which ones will you most want to engage with right now?" This is the engine behind TikTok's For You page, Shopee's product recommendations, Agoda's "properties you might like," and the merchant-discovery rail in the GrabFood home screen. There is no follow graph to bound the candidates, so the system must personalize (different output per user) and rank (order by predicted engagement), choosing from a corpus of millions to billions of items.

That difference is the whole interview. The naive idea — "score every item for this user and sort" — is a non-starter: scoring a billion items per request with a real ML model would take minutes and melt your GPU fleet. The discipline the interviewer is testing is whether you reach for the two-stage funnel: a cheap retrieval stage that throws away 99.99% of the corpus, followed by an expensive ranking stage that runs only on what survives.

Interview tip

Open by naming the distinction out loud: "This isn't a fan-out problem like a timeline; it's a retrieval-and-ranking problem over an unbounded candidate set, so I'll design a two-stage funnel." That single sentence signals you know the right shape before you draw a box.

01 Scope it

Clarify the requirements

Recommenders vary enormously — a video For You page, a product grid, a "jobs you may like" rail, and a dating-style match queue all share the funnel but differ in freshness, latency, and what "engagement" means. Nail the scope before drawing.

Functional requirements

  • Personalized, ranked items. Given a user (and optional context like the current screen, location, time of day), return an ordered list of item IDs the user is most likely to engage with.
  • The surface. Let's anchor on a TikTok-style For You video feed — an infinite stream where each "page" is ~10 videos. The same design generalizes to a Shopee product rail or Agoda hotel list.
  • Feedback loop. The system must log what it showed (impressions) and how the user reacted (clicks, watch time, likes, purchases) so the model can keep learning.
  • Pagination / "more". When the user scrolls, return the next page without repeating recent items.

Non-functional requirements

  • Low latency. The feed sits on the hot path of every app open. A reasonable budget is p99 < 200 ms end-to-end for a page of recommendations.
  • Scalable. Tens of millions of daily active users, a corpus of hundreds of millions to billions of items.
  • Fresh. A video uploaded an hour ago, or a flash-sale SKU on Shopee, should be eligible to surface — the system can't only know about yesterday's catalog.
  • Available & graceful. If the ranking model or feature store is down, the user must still get something reasonable (popular / trending) rather than an empty screen.
  • Quality, measured. Success isn't "it returns items" — it's a lift in business metrics (watch time, CTR, conversion) proven by A/B test.

Assumptions to state

Make the implicit explicit: this is a read-heavy system (far more "get me a feed" than "ingest a new item"); the corpus and user base are large but the per-request candidate set is small; eventual consistency is fine (it's okay if a brand-new video takes a minute to become recommendable); and we optimize for a single primary objective (say, expected watch time) while constraining others (diversity, freshness).

Interview tip

The phrase that frames the rest of the design: "It's read-heavy and latency-bound, so I'll precompute everything I can offline and keep the online path to retrieval + a fast model inference." Online-vs-offline split is the load-bearing idea of the whole lesson.

02 Estimation

Back-of-the-envelope numbers

Quick sizing tells you which parts must be precomputed and which can be live. Assume a Shopee/TikTok-scale Southeast-Asian product: 100M DAU, each opening the app and scrolling a few feed pages.

Request rate (QPS)

  • Say each user requests 10 feed pages/day on average → 1B feed requests/day.
  • 1B / 86,400 s ≈ ~12,000 QPS average. Peak (evening commute, lunch break in SG) is spiky — assume 5× → ~60,000 QPS peak.
  • Each feed request triggers: 1 candidate-retrieval call, hundreds of feature lookups, and 1 batched model-inference call. So the feature store and inference fleet are the real load: at ~500 candidates ranked per request, that's 60k × 500 ≈ 30M item-scorings/second at peak. That number is why ranking must be a cheap, batched model — not why retrieval matters; retrieval is what keeps that 500 from being 1 billion.

Storage

  • Item embeddings: 500M items × a 256-dim float32 vector = 500M × 256 × 4 bytes ≈ 512 GB. Fits in a sharded in-memory ANN index across a handful of machines.
  • User embeddings: 100M users × 256 × 4 B ≈ 102 GB — easily a single Redis/key-value tier.
  • Interaction log: 1B feed requests/day × ~10 impressions each × ~200 bytes/event ≈ ~2 TB/day of impression+engagement logs streaming into the data lake. This is the fuel for training.

The takeaway from the numbers

Three facts fall out, and they pre-justify the architecture:
(1) you cannot score the whole corpus per request — retrieval is mandatory;
(2) embeddings are small enough to precompute and serve from memory;
(3) the engagement log is huge and append-only — it belongs in a streaming pipeline plus a data lake, not your serving database.

Why state numbers at all

The estimate is not arithmetic for its own sake — each line removes a design option. "30M scorings/sec" kills the idea of a heavyweight model on the full corpus. "512 GB of vectors" tells you the ANN index is RAM-resident and shardable. Interviewers reward the inference, not the digits.

03 Contracts

The API and data model

Pin down the request/response shape before architecture — it forces clarity on what the client sends and what each internal service must produce.

Public API

One read endpoint for the feed and one write path for feedback. The feed endpoint takes a user, context, and a pagination cursor; it returns ranked item IDs plus a token to fetch more.

200 OK

  "next_cursor": "eyJzZWVuIjpbInZfNTUyMSJdfQ",
  "model_version": "rank_v37"     # carried for A/B attribution

Two details earn points. The response carries a model_version so every downstream engagement event can be attributed back to the model that produced it — essential for A/B analysis. And the next_cursor encodes what was already shown (e.g. seen item IDs) so the next page de-duplicates without server-side session state.

Data model sketch

Four distinct stores, each chosen for its access pattern — going past "just put it in Postgres":

  • Item catalog (source of truth, document store like MongoDB / DynamoDB): item_id, author, language, category, creation time, status (active/removed), raw stats.
  • Item & user embeddings (the vectors): item vectors live in the ANN index; user vectors live in a low-latency KV store keyed by user_id.
  • Feature store (online: Redis-class KV keyed by user_id / item_id; offline: columnar tables in the lake) — covered in depth in step 06.
  • Interaction events (append-only): streamed via Kafka, landed in S3/data lake, queryable for training. Never your serving DB.
-- a precomputed feature row served online, keyed by user
user_features:u_8841 = {
  emb: [0.04, -0.11, ...256d],
  top_categories: ["dance","food_sg","comedy"],
  avg_watch_ms_7d: 11800,
  active_hour_mode: 19,        # engages most at 7pm
  device: "ios", country: "SG"
}
-- a precomputed item-side feature row, keyed by item
item_features:v_5521 = {

  ctr_1h: 0.07, completes_1h: 4210, age_min: 52, author_tier: 3
}
Interviewer follow-up

"Why not just store everything in one relational DB?" Because the access patterns are incompatible: the catalog wants document lookups, embeddings want vector similarity search, features want sub-millisecond key lookups, and events want append-only streaming. Forcing one engine to do all four makes each one slow.

04 The architecture

The two-stage funnel

Here is the spine of the whole design. A single feed request flows through a funnel that narrows the corpus in stages, each stage cheaper-per-item but seeing more items than the next:

  • Stage 1 — Candidate generation (retrieval). Cheaply narrow billions of items to a few hundred plausible candidates. Cheap per item, runs over everything.
  • Stage 2 — Ranking. Run a richer ML model over those few hundred, scoring each with many features, and return the top-k (say top 10). Expensive per item, runs only on survivors.

Why split it instead of one big model? Cost and latency. Retrieval is allowed to be approximate and feature-light because its only job is "don't miss the good stuff" (high recall); it can be a vector lookup answered in single-digit milliseconds. Ranking is allowed to be expensive and feature-rich because it only sees ~500 items, not a billion. You spend your compute where it changes the answer. This is exactly the discipline behind a database query planner pushing a cheap filter before an expensive join.

Production systems often add a third micro-stage, re-ranking, after scoring: a cheap pass that enforces business rules — inject a fresh video, ensure language/category diversity, drop items the user just saw, respect "don't show competitor ads back-to-back." Keep it in your back pocket; mention it as the place policy lives.

The end-to-end request flow, then, is: the client hits the recommendation service (the orchestrator); it calls candidate generation to get ~500 IDs; it asks the feature store for the user's and those items' features; it sends the batch to the inference service, which returns a score per candidate; it re-ranks for policy and returns the top-10. All of it must fit in the p99 latency budget, so each hop has a sub-budget (see step 07).

Interviewer follow-up: "Why two stages?"

The crisp answer: "To put expensive compute only where it matters. Retrieval optimizes recall over a billion items at ~1ms/item-equivalent using vector search; ranking optimizes precision over a few hundred with a heavy model. One model trying to do both would be either too slow (heavy on everything) or too dumb (light on everything)."

05 Deep dive · retrieval

Candidate generation & approximate nearest neighbor

The first bottleneck: how do you pull a few hundred relevant items out of a billion without scoring them? The trick is to not score them at all in this stage — instead, represent both users and items as points in the same vector space, and pull the items closest to the user.

Embeddings: turning users and items into vectors

An embedding is a dense vector (say 256 floats) that encodes the "meaning" of an item or user, learned by a model so that things that get engaged with together end up near each other in the space. A user who watches a lot of SG food content gets a vector close to the vectors of SG food videos. This is the modern form of collaborative filtering — "users like you liked these" — but compressed into geometry: similarity is just distance (cosine or dot product) between vectors.

The serving consequence is beautiful: retrieval becomes "find the items whose vectors are nearest to this user's vector." That is a pure geometry query — no per-item model call.

Why approximate: ANN, not exact KNN

Finding the true nearest neighbors among a billion vectors exactly (k-nearest-neighbors, KNN) means computing the distance to every one of them — a billion dot products per request. Far too slow. So we use approximate nearest neighbor (ANN) search: accept a tiny chance of missing a true neighbor in exchange for a ~1000× speedup. Since retrieval only needs good recall, not perfection (ranking sorts the survivors anyway), the approximation is essentially free in quality terms.

The dominant ANN method is HNSW (Hierarchical Navigable Small World), a graph index. Under the hood it builds a multi-layer graph of vectors: sparse "express lanes" at the top for big jumps across the space, dense layers at the bottom for fine-grained local search. A query enters at the top, greedily hops to the nearest node it can see, descends a layer, and repeats — converging on the neighborhood of the query in roughly logarithmic hops instead of scanning everything. Libraries: FAISS (Meta), HNSWlib, and serving systems built on them — Vespa, Milvus, Elasticsearch/OpenSearch vector search, Pinecone. Tunable knobs: M (graph connectivity), efConstruction (build-time effort), and efSearch (query-time effort — higher = better recall, slower).

Multiple candidate sources, then merge

ANN over embeddings is the workhorse, but a good system blends several candidate sources and merges them, because each covers a blind spot:

  • Embedding ANN — "more like what you engaged with." The personalization backbone.
  • Trending / popular — globally or per-geo (e.g. "trending in SG"). Covers cold-start and gives everyone a baseline of quality.
  • Recent / fresh — items uploaded in the last hour, so new content has a chance to be seen before it has any stats.
  • Follow graph — items from creators/sellers the user follows, when such a relationship exists.

The service queries these in parallel, deduplicates, and passes the merged ~500 to ranking. A worked example: on the GrabFood home rail, retrieval might pull 200 merchants near you by location, 150 by cuisine-embedding similarity to your past orders, 100 currently-promoted, and 50 newly onboarded — merge, drop closed/out-of-range, send ~400 to ranking.

Exact KNN

  • Always returns the true nearest items
  • O(N) per query — scans the whole corpus
  • Infeasible at a billion vectors and 60k QPS

Approximate NN (HNSW)

  • ~log(N) hops via a navigable graph — milliseconds
  • Tiny, tunable recall loss (efSearch trades speed vs recall)
  • Good enough because ranking re-sorts anyway
Failure mode

The ANN index is RAM-resident and sharded — if a shard dies, you lose the candidates it held. Mitigate with replicas per shard and by always merging in the (separately served) trending/recent sources, so a degraded ANN tier still yields a non-empty, reasonable candidate set. New items won't be in the index until the next build/upsert, which is exactly the freshness problem step 09 addresses.

06 Deep dive · features

The feature store

Ranking needs features — the numbers the model reads to score a (user, item) pair: the user's watch history summary, the item's recent click-through rate, the time of day, whether the languages match, and so on. The bottleneck: the model runs on the online request path with a few-millisecond budget, but many features are expensive to compute (a 7-day average watch time is an aggregation over billions of events). You cannot compute those live per request. Enter the feature store.

What it is and the two halves

A feature store is the system that computes, stores, and serves features consistently for both training and inference. It has two faces:

  • Offline store — large columnar tables in the data lake (Parquet on S3, queried via Spark). Holds the full history of feature values; used to build training datasets. Throughput matters, latency doesn't.
  • Online store — a low-latency KV store (Redis, DynamoDB, Cassandra) holding only the latest value of each feature, keyed by user_id or item_id. Latency matters intensely; it's read on every request.

Under the hood: streaming and batch jobs materialize features into both stores. A Flink job consuming the Kafka event stream updates item_features:v_5521.ctr_1h in near-real-time; a nightly Spark job recomputes user_features.avg_watch_ms_7d. At inference, the ranking service does one (batched) read against the online store to fetch the user row once and all ~500 item rows, then feeds them to the model. Tools: Feast (open source), Tecton, and the in-house stores at large shops (e.g. Michelangelo-style platforms). Knobs: per-feature TTL and freshness SLA, batch-read fan-out limits.

The online/offline consistency problem

This is the feature store's signature interview trap. The model is trained on feature values computed by the offline pipeline, but served values come from the online pipeline. If the two compute a feature even slightly differently — different time windows, different rounding, a bug in one path — the model sees one distribution in training and another at serving. That mismatch is training/serving skew, and it silently tanks model quality: the model is, in effect, being asked questions in a slightly different language than it studied.

The defenses:
(1) compute each feature with shared transformation code used by both paths, so there is one definition;
(2) for the strictest correctness, log the exact feature values used at serving time and train directly on those logged features ("log-and-wait"), which makes training data identical to what serving produced by construction;
(3) monitor feature distributions in production and alert when online drifts from offline.

Precomputed embeddings live here too

The user and item embeddings from step 05 are themselves features — precomputed offline (or in near-real-time for items) and pushed to the online store, so the request path never has to run the heavy embedding model. The user vector for ANN retrieval is read from exactly this online store.

Interviewer follow-up: "What's a feature store for?"

Answer: "To serve the same features the model trained on, but fast enough for online inference — and to guarantee the training and serving values match. It splits into an offline store for building datasets and an online store for sub-millisecond reads; its job is killing training/serving skew while hitting the latency budget."

07 Deep dive · serving

Ranking and serving a model in milliseconds

Now the second stage. We have ~500 candidates and a feature store. Ranking scores each candidate with a model and sorts. The bottleneck is the one the interviewer will press hardest: how do you run an ML model over 500 items inside a few-millisecond budget, 60,000 times a second?

The model and what it predicts

The ranking model predicts the probability the user will perform the target action on each candidate — for a video feed, something like P(watch > X seconds) or expected watch time; for Shopee, P(click) × P(purchase | click). It reads dozens-to-hundreds of features per (user, item) pair: user features (read once), item features (read per candidate), and cross/context features (language match, hour, geo). Architectures range from gradient-boosted trees (XGBoost / LightGBM — cheap, strong tabular baseline) to deep neural nets (the typical choice at scale, e.g. a wide-and-deep or two-tower-plus-DNN). The output is one score per candidate; sort descending, apply re-ranking policy, take top-10.

How you make inference fast

The few-millisecond budget is met by a stack of techniques, not one trick:

  • Batch the candidates. Score all ~500 in a single forward pass (one big matrix), not 500 calls. This is the single biggest win — it amortizes overhead and saturates the hardware.
  • Precompute the heavy half. The item-side embeddings/features are already computed offline; the online model only combines them with the user/context — far less work than computing features live.
  • A dedicated inference service. Serve the model behind TensorFlow Serving, NVIDIA Triton, or TorchServe, on CPU for trees / GPU for big nets, separate from the orchestrator so it scales independently.
  • Optimize the model. Quantization (float32 → int8), distillation to a smaller student model, and compilation (ONNX Runtime, TensorRT) cut latency several-fold.
  • Cache where safe. Cache a user's full ranked page for a few seconds so rapid re-requests (scroll-up, retries) don't re-rank.

The latency budget, allocated

Make the p99 < 200 ms concrete by handing each hop a slice: candidate retrieval ~20 ms, feature fetch ~15 ms, model inference ~30 ms, re-ranking + assembly ~10 ms, with the rest as network and headroom. If any stage threatens its slice, it returns what it has — partial candidates, stale features, a fallback score — rather than blowing the budget. Latency is a hard contract, not a best-effort target.

Interviewer follow-up: "How do you serve a model in a few ms?"

Hit four words: batch, precompute, optimize, isolate. Batch all candidates into one forward pass; precompute embeddings/features offline so the online model does minimal work; optimize the model (quantize/distill/compile); isolate it behind a dedicated, independently-scaled inference service on the right hardware.

08 The loop

The training loop, skew, and A/B testing

So far the model is a magic box that scores well. Where does it come from, and how do you know it's actually good? This is the offline-train / online-serve loop, and it's where the system improves over time.

Generating training data from logs

Every page you serve emits impression events (what was shown, in what position, with which model version) and, moments later, engagement events (watched, clicked, purchased, or ignored). Joining impressions to engagements produces labeled examples: this (user, item, features-at-serving-time) pair → label = watched/not-watched. An item shown and ignored is a negative; one engaged with is a positive. This labeled stream — gigabytes to terabytes a day — flows through Kafka into the data lake and becomes the next training set. The model never stops getting fresh ground truth, because the act of serving generates it.

Train, then deploy

Offline, a training job (Spark for feature joins, then PyTorch/TensorFlow or XGBoost) fits a new model on recent logged data, validates it against held-out data, and — if it beats the incumbent on offline metrics like AUC — promotes it to a candidate for online testing. The model is versioned (rank_v37) and pushed to the inference service. Cadence ranges from daily full retrains to continuous/online updates for fast-moving surfaces.

Online vs offline skew (again, but bigger)

Beyond feature-value skew (step 06), there's a subtler trap: the model affects its own training data. Users can only engage with items the current model chose to show — so the logs are biased toward the incumbent's taste. A genuinely great item the old model never surfaced has no positive examples, so the new model never learns to like it either. This is feedback-loop / selection bias. Mitigations: inject a small fraction of randomized or exploratory recommendations (explore/exploit) so the logs contain unbiased signal, and weight examples by inverse-propensity to correct for it.

A/B testing: the only real verdict

Offline AUC going up does not prove the new model is better for the business — offline metrics and online behavior diverge constantly. So you A/B test: route, say, 1% of users to the new model (treatment), keep the rest on the old (control), and compare on online metrics — CTR, average watch time, session length, retention, and for Shopee/GrabPay, conversion and GMV. The model_version carried in every event (step 03) is what lets you attribute each engagement to the right arm. Only a statistically significant lift, with no regression in guardrail metrics, earns a full rollout.

Interviewer follow-up

"Your offline metric improved but the A/B test was flat or negative — what happened?" Strong answers: offline/online metric mismatch, feedback-loop bias (the model can't be rewarded for items it never showed), a feature available offline but not online (leakage), or a novelty effect that fades. Naming feedback-loop bias unprompted is a senior signal.

09 Real-world edges

Cold start, diversity, and freshness

A funnel that only knows about well-established users and items produces a stale, repetitive, homogeneous feed. Three classic problems, each with a concrete fix, separate a textbook answer from a real one.

Cold start

The cold-start problem: a brand-new user has no history, so there's no embedding to query ANN with; a brand-new item has no engagement stats, so the model has nothing to score and ANN doesn't index it yet. Fixes:

  • New user: fall back to popular/trending in their geo (trending in SG, in their language), optionally seeded by a quick onboarding interest pick or signup-time signals (device, locale, referral). As they interact, an embedding forms and personalization kicks in within a few sessions.
  • New item: bootstrap with content-based features — derive an initial embedding from the item's content (caption, audio, category, seller) rather than behavior, and give it a deliberate exploration budget (force some impressions) so it can earn real engagement data and graduate into the behavioral model.

Diversity (the filter bubble)

Pure score-maximization collapses into a monotonous feed — ten dance videos in a row, or the same three sellers. That actually hurts long-term engagement. Inject diversity in the re-ranking stage: penalize candidates too similar to ones already placed (e.g. maximal-marginal-relevance), cap items per author/category per page, and reserve slots for exploration. This is a deliberate, small sacrifice of immediate predicted score for healthier sessions.

Freshness

For TikTok, Shopee flash sales, or breaking content, a video or SKU from minutes ago must be eligible. Two mechanisms: keep a dedicated recent candidate source backed by a near-real-time index that ingests new items in seconds; and stream real-time item features (last-hour CTR, completion rate) into the online feature store via Flink so the ranker can react to a video going viral right now, not tomorrow.

Graceful degradation & fallbacks

The availability requirement made concrete: at every stage, define what happens when it's slow or down. If the ANN tier is degraded, still return trending. If the feature store times out, score with default/last-known features. If the inference service is unavailable, fall back to a non-personalized popular/trending feed — a static list refreshed every few minutes. The cardinal rule: a recommender must never show an empty screen. A slightly worse feed is infinitely better than a spinner.

Worked scenario · Shopee 9.9 sale

During a SG mega-sale, traffic spikes 5×, thousands of flash-sale SKUs appear and sell out within minutes, and the ranking model is overwhelmed. The system holds up because: freshness streams the new SKUs into the recent source instantly; real-time features mark sold-out items so they drop out within seconds; and if inference saturates, the home rail degrades to a curated trending-deals list rather than failing. Users still see a sensible page.

10 Wrap-up

Scaling, failure modes, and what breaks at 10x

Pull it together and pressure-test. The serving path is stateless and horizontally scalable behind a load balancer: the orchestrator, candidate-generation, feature, and inference services each scale out independently because they hold no per-user state — all state lives in the ANN index, the feature store, and the model registry. That separation is what lets you 10x one bottleneck without touching the others.

Where the caches sit

  • Feed cache — a user's freshly-ranked page, cached for a few seconds to absorb scroll-up and retries.
  • Feature cache — hot item features (a viral video's stats) read by everyone; cache them at the edge of the feature tier.
  • Candidate cache — for users querying repeatedly within seconds, reuse the retrieved candidate set and just re-rank.

What breaks at 10x

  • The ANN index outgrows RAM. At 5B items the vectors no longer fit on a small shard set. Fix: shard the index by item partition and fan-out-query in parallel, or compress vectors (product quantization in FAISS) trading a little recall for a lot of memory.
  • The inference fleet becomes the cost center. 10x QPS × the same per-request candidate count is the dominant compute bill. Fix: cap candidates harder (rank 200 not 500), distill to a cheaper model, and add the cheap pre-ranking stage between retrieval and full ranking to cut what the heavy model sees.
  • The feature store hot-keys. A viral video's feature row is read by millions/second, hot-spotting one shard. Fix: replicate hot keys, add a local in-process cache, and use consistent hashing with hot-key spreading.
  • The training data outpaces the pipeline. 10x logs can stall the daily retrain. Fix: sample negatives aggressively, move to incremental/online training, and pre-aggregate features in the stream.

Failure modes, named

What happens when a node dies? An inference node dies → the LB drops it; in-flight requests retry on a peer, and if the whole fleet is unhealthy, fall back to trending. An ANN shard dies → a replica serves it; if none, that partition's candidates are simply missing and the other sources fill in. The feature store hiccups → serve with stale/default features. The training pipeline fails → no new model deploys, but the current one keeps serving — staleness, not an outage. The theme: every dependency has a degraded mode that keeps the feed alive.

★ Review

Summary & self-test

The whole topic compresses to one sentence: cheaply retrieve a few hundred candidates from billions using vector search, score them with a feature-rich model inside a tight latency budget, then keep the loop honest with logged feedback and A/B tests. If you can draw the funnel, name what each stage optimizes, and explain training/serving skew and the cold-start fallback, you can carry this interview.

05Blend candidate sourcesANN + trending + recent + follow graph, merged and deduped — covers each other's blind spots.
06Feature storeOffline for training, online for serving; one definition of each feature kills training/serving skew.
07Fast inferenceBatch all candidates, precompute, optimize the model, isolate behind a dedicated service.
08Serve → log → trainLogs become labels; only an A/B-tested online lift proves a new model is actually better.
09Cold start & diversityFall back to popular for new users/items; inject diversity and freshness in re-ranking.
10Never show emptyEvery dependency has a degraded mode; the worst feed beats a spinner.
